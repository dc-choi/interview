# DB 운영 자동화 (DBA 플랫폼 엔지니어링)

DB 운영의 근본 문제는 **수작업이 서버 수에 비례해 늘고, 그에 따라 운영 인력도 비례해 늘어나는 구조**다. DB는 손상되면 서비스가 무너지는 핵심 자산이라 작업이 보수적으로(사람이 직접 확인하고 실행) 운영되기 쉽고, 그럴수록 휴먼 에러와 인력 선형 증가 문제가 커진다.

해법은 반복 운영 업무를 도구화해 **서버가 늘어도 DBA가 비례해서 늘지 않는 구조**를 만드는 것이다. 목표는 DBA를 없애는 게 아니라, 사람을 반복 작업에서 떼어내 장애 원인 분석, 구조 설계, 자동화 판단 기준 같은 고부가가치 업무에 집중시키는 것. DBA의 역할이 "운영자"에서 "DB를 이해하는 자동화 엔지니어"로 확장되는 흐름이다.

## 성장 선순환 구조

자동화의 효과는 일회성 절감이 아니라 누적되는 선순환이다.

수작업 감소 → 연구/개발 시간 확보 → 더 나은 자동화 도구와 운영 품질 → 다시 시간 확보.

이 루프가 자리 잡으면 서버 수가 늘어도 운영 비용과 인력 부담이 함께 늘지 않는다. 반대로 수작업에 매몰되면 개선할 시간이 없어 수작업이 더 쌓이는 악순환에 빠진다. 이 toil(반복적, 수동적, 자동화 가능한 운영 부하) 감소는 SRE 원칙과 같은 결이다.

## 자동화 대상 도메인

### 설치, 구성 표준화 (1순위 기반 작업)

DB 생성은 버튼 몇 번처럼 보이지만, 실제 운영에서는 네트워크, 권한, 파라미터, 모니터링, 백업, 알림, 표준 태그까지 챙길 항목이 많다. 사람이 매번 설정하면 누락이 생긴다. 표준화된 생성과 삭제(프로비저닝)가 가능해야 운영 품질이 안정되므로 가장 먼저 자동화할 기반 작업이다. 이 도메인의 구체적 구현(이벤트 체인 + 보상 롤백, 파라미터 템플릿, 백업 태그, 후처리)은 [[DB-Provisioning-Pipeline|DB 프로비저닝 파이프라인]] 참고.

### 스키마 변경 자동화

`ALTER`는 겉보기엔 단순하지만 DBMS마다 내부 처리가 다르고, 테이블 크기와 변경 유형에 따라 위험도가 크게 달라진다. 좋은 도구는 변경 내용을 보고 **즉시 실행해도 되는지 / 트래픽 적은 시간에 해야 하는지 / 별도 마이그레이션 방식이 필요한지**를 판정해야 한다. DB 지식과 개발 역량이 모두 필요한 대표 영역. 변경 전략(Online DDL, pt-osc, gh-ost 등) 자체는 [[Schema-Migration-Large-Table|대용량 테이블 스키마 변경]] 참고 — 자동화는 그 위에 "어느 전략을 언제 쓸지" 판정 로직을 얹는 일이다.

### DB 오토스케일링

애플리케이션 서버처럼 트래픽 변화에 따라 DB 노드를 자동으로 늘리고 줄이는 방식. 단, DB는 노드만 추가한다고 바로 안전해지지 않는다.

- **추가 시**: 새 노드는 캐시(버퍼 풀 등)가 비어 있어 워밍업 전까지 성능이 불안정 — 콜드 캐시 문제
- **제거 시**: 사용 중인 노드를 갑자기 빼면 진행 중인 커넥션/쿼리가 끊겨 서비스 오류

따라서 핵심은 "노드 수 조절"이 아니라 **서비스 영향 없이 추가/제거하는 절차**(워밍업 대기, 커넥션 드레이닝, 헬스 기반 편입)다. 비용 측면은 [[Autoscaling-Cost|오토스케일링 비용]] 참고.

### Kubernetes Operator (셀프 매니지드 DB)

모든 DB를 Aurora 같은 관리형 서비스로만 운영할 수는 없다(직접 운영하는 MongoDB 등이 남는다). 셀프 매니지드 DBMS를 Kubernetes 위에서 Operator로 관리하면 배포, 복구, 확장, 설정 관리를 선언적으로 자동화할 여지가 커진다.

Operator는 **CRD(사용자 정의 리소스) + 컨트롤 루프** 패턴으로, "원하는 상태"를 선언하면 컨트롤러가 실제 상태를 그쪽으로 수렴시킨다 — DB 페일오버, 백업, 스케일 같은 운영 지식을 코드로 박아 넣는 방식. 그래서 앞으로 DBA에게 Kubernetes는 선택이 아니라 운영 역량이 될 가능성이 크다. 실행 기반은 [[EKS|Amazon EKS]] 참고. MongoDB(PSMDB Operator)의 구체적 운영 — 선언적 멤버 추가, 외부 접속(Split Horizon + SNI), EBS Volume Clone 빠른 프로비저닝 — 은 [[MongoDB-Kubernetes-Operator|MongoDB Kubernetes Operator]] 참고.

## 셀프서비스 진단 도구

모든 DB 이슈가 DBA에게만 몰리면 사람이 병목이 된다. 개발자나 비전문가도 문제의 **위치**를 파악할 수 있는 도구(쿼리가 느린지, 커넥션이 부족한지, 특정 노드에 부하가 몰렸는지를 화면에서 확인)가 있으면, DBA 호출 전에도 1차 판단이 가능해진다. DBA 부담을 줄이고 개발 조직 전체의 문제 해결 속도를 높인다. 진단 지표 자체는 [[Container-Monitoring|모니터링]], 슬로우 쿼리 진단은 [[MySQL-Slow-Query-Diagnosis|Slow Query 진단]]과 연결된다. 이 도구의 구체적 구현(정보 통합 화면, 시점 비교, Slack 문의, AI/MCP 반자동 분석)은 [[Self-Service-DB-Diagnostics|셀프서비스 DB 진단 플랫폼]], 그 위에서 도는 분석 방법론은 [[DB-Incident-Triage|DB 장애 분석 방법론]] 참고.

## 운영 방식: 오너십 비고정

특정 사람이 특정 서비스/기술을 독점 담당하지 않는 운영 방식.

- 팀원 모두가 다양한 기술과 중요 서비스를 경험 → 팀 역량이 고르게 성장, 신입도 고난도 업무 기회
- 누가 자리를 비워도 업무가 멈추지 않음 — **버스 팩터(특정 인물에게 지식이 몰리는 병목)** 를 줄이는 구조

자동화 도구가 운영 지식을 코드/도구로 외부화하면, 이 비고정 오너십이 더 쉬워진다(지식이 사람 머릿속이 아니라 도구에 있으므로).

## 미래의 DBA 역량

과거엔 SQL, 튜닝, 장애 대응이면 충분한 경우가 많았다. 앞으로는 여기에 개발/자동화 역량이 더해진다. 단, DB 관리 도구를 만들려면 단순 프로그래밍만으로 부족하고 **DBMS 내부 동작, DDL 처리 방식, 인덱스와 락, 복제 구조** 같은 지식이 함께 필요하다 — DB 깊이와 개발 폭이 둘 다 요구된다. AI 도구가 개발 진입 장벽을 낮추면서 DBA가 직접 자동화 도구를 만들기 쉬운 환경이 오고 있다.

실전 학습 방향: SQL/튜닝에 더해 Python, Go, Kubernetes, AWS, 모니터링, 자동화 설계 중 하나씩 도구로 익히기.

## 면접 체크포인트

- "수작업 ∝ 서버 수 ∝ 인력" 구조와 선순환으로 끊는 논리 (toil 감소)
- DB 오토스케일링이 앱 서버와 다른 이유 (콜드 캐시, 커넥션 드레이닝)
- 스키마 변경 자동화에서 "위험도 판정"이 핵심인 이유
- Kubernetes Operator(CRD + 컨트롤 루프)로 셀프 매니지드 DB를 관리하는 의미
- 셀프서비스 진단 도구가 DBA 병목을 줄이는 메커니즘
- 오너십 비고정과 버스 팩터의 관계

## 사례
- AWS 기반 DB 운영 팀이 PostgreSQL/MySQL은 Aurora, MongoDB는 셀프 매니지드로 운영하며, 설치/구성/스키마 변경/오토스케일링/K8s Operator 자동화로 toil을 줄이는 방향을 추진한 사례가 있다.
