# Aggregate경계와데이터접근

> 출처: [[Elegant-OOP-Design|우아한 객체지향]]

## 객체참조의3가지문제

| 문제 | 설명 |
|------|------|
| 결합도 상승 | 참조로 연결된 모든 객체를 따라가서 기능 구현 가능 → 경계가 사라짐 |
| 성능 문제 | 어디까지 조회할 것인가? (Lazy Loading, N+1) |
| 트랜잭션 경합 | 변경 빈도가 다른 도메인이 하나의 트랜잭션으로 묶임 → Lock 경합 |

## 묶는기준

- 함께 생성되고 함께 삭제되는 객체 → 함께 묶기
- 도메인 제약사항을 공유하는 객체 → 함께 묶기
- 그 외에는 가능하면 분리
- 모든 것은 **비즈니스 룰**에 따라 결정

## 접근방식

| 경계 | 접근방법 | 트랜잭션 |
|------|---------|---------|
| 경계 안 | 객체 참조 | 하나의 트랜잭션 |
| 경계 밖 | **ID로 접근** + Repository를 통한 탐색 | 별도 트랜잭션 |

- 그룹 간 객체 참조를 제거하고 ID 기반 연관관계로 전환
- 조회의 경계도 Aggregate 경계를 따름
- **경계를 잘 나누면 영속성 저장소 변경까지 가능** (MySQL → MongoDB 등)

## 객체참조를끊은후의전략

### 검증객체로이동
- 직접 참조하던 로직을 별도 Validator 등으로 이동
- 절차지향적이지만 응집도가 올라감

### 도메인이벤트
- Aggregate 간 통신은 이벤트로 느슨하게 연결
- 시스템 단위로 분리되면 메시지 브로커를 통한 이벤트 기반 통신

## 면접포인트
- "Aggregate란?" → 함께 변경되는 객체의 묶음, 트랜잭션의 최소 단위
- "왜 객체 참조 대신 ID를 쓰나?" → 결합도, 성능, 트랜잭션 경합 해결
- "MSA에서 데이터 분리 기준?" → Aggregate 경계가 서비스 분리의 기초
