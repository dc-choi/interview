# ORM과임피던스불일치

> 출처: [[App-Architecture-OOP|애플리케이션 아키텍처와 객체지향]]

## 패러다임충돌

도메인 모델을 사용하면 객체 지향 패러다임과 관계형 패러다임이 충돌한다.

| 관점 | 객체지향 | 관계형DB |
|------|---------|---------|
| 초점 | **행위** (메서드, 협력) | **데이터** (테이블, 컬럼) |
| 관계 | 참조(레퍼런스) | 외래키(FK) |
| 상속 | 클래스 계층 | 테이블에 상속 없음 |
| 정체성 | 객체 동일성(==) | PK 동일성 |

이 불일치를 **임피던스 불일치(Impedance Mismatch)** 라고 부른다.

## DataMapper패턴(ORM)

- 객체 모델과 DB 스키마 사이에 **데이터 매퍼**를 두어 독립성 유지
- Java에서는 **JPA**가 이 역할을 수행
- 단, JPA의 제약사항으로 완전한 객체 지향 설계는 어려울 수 있음
  - 값 타입 컬렉션 제약
  - 복잡한 상속 매핑
  - 양방향 연관관계 관리 비용

## 트랜잭션스크립트에서의데이터접근

트랜잭션 스크립트 패턴에서는 ORM이 필요하지 않음:
- 데이터 객체와 테이블을 **1:1로 매칭** (아토믹 도메인 모델)
- **테이블 데이터 게이트웨이**를 만들어 데이터를 조회
- 구조가 단순하지만, 비즈니스 로직이 복잡해지면 유지보수가 어려움

## 면접포인트
- "JPA를 왜 사용하나?" → 임피던스 불일치 해결, 객체 중심 도메인 모델 유지
- "MyBatis vs JPA?" → 트랜잭션 스크립트에는 MyBatis, 도메인 모델에는 JPA가 적합
- "JPA의 한계?" → 제약사항으로 인해 도메인 모델을 100% 반영하기 어려움
