# 운영 레벨 지표 — 여러 메트릭을 하나의 단계로 압축

대시보드에 지표가 10개·20개 넘어가면 **운영자가 동시에 이해할 수 없다**. "지금 상황이 괜찮은가?"라는 질문에 즉각 답하려면 여러 원시 지표를 **하나의 단계(Level)** 로 압축해야 한다. SRE의 SLI·SLO와 유사하지만, 비운영자(PM·CS·경영진)도 공유할 수 있도록 **언어**를 맞추는 쪽에 초점.

## 왜 필요한가

- **대시보드 과부하**: 동시에 여러 지표를 관찰하면 놓치는 신호 발생
- **주관적 판단**: "느려 보인다" vs "평소보다 조금 높다" — 사람마다 기준이 다름
- **시간대별 기준 차이**: 평일 오전·점심·주말 피크의 "정상"은 다름. 고정 임계값으로는 오탐 폭증
- **부서 간 소통 단절**: 개발팀의 "p99 3초 초과"를 CS·마케팅이 이해 못함

하나의 정규화된 레벨은 **공통 언어**가 된다.

## 레벨 설계의 기본 구조

### 단계 나누기

보통 3~9단계. 너무 적으면 정보량이 부족, 너무 많으면 해석이 어렵다.

| 단계 | 라벨 | 의미 |
|---|---|---|
| 1~3 | 안정 (Stable) | 평소 상태, 자동화 운영 |
| 4~5 | 주의 (Caution) | 임계에 근접, 관찰 강화 |
| 6~7 | 경고 (Warning) | 운영자 개입 필요, 알림 발송 |
| 8~9 | 위험 (Critical) | 장애 또는 임박, 온콜 호출 |

숫자는 조직·도메인에 맞게 조정. 중요한 건 **단계별 대응 플레이북**이 함께 존재할 것.

### 입력 지표 선택

원시 지표를 어떻게 합성할지 결정.

- **수요·공급 비율**: 주문/라이더·요청/처리자·트래픽/용량
- **지연 상위 백분위**: p95·p99 응답 시간
- **오류율**: 5xx·4xx 비율
- **큐 대기 시간**: 처리되지 않고 쌓인 작업의 최고 나이

도메인별로 가중치 다르게 결합. 수요·공급 비율 단일 지표만으로도 충분한 경우가 많음.

### 정규화·시간대 보정

단순 임계값은 피크 시간대에 과민, 새벽에 둔감해진다. 해결책:

- **시간대별 기준값**을 저장 (예: 요일·시각별 평균)
- **Z-score** / **예측 대비 차이**로 정규화
- 절대값이 아니라 **"평소 대비 얼마나 벗어났는가"**로 레벨 매핑

## 설계 프로세스

### 1. 이해관계자 정렬
운영·개발·PM·데이터 분석가가 모여 "단계별로 우리가 실제로 해야 할 일"을 먼저 정의. 기술이 아니라 **의사결정 언어**가 먼저.

### 2. 지표 후보 선정
원시 지표 중 강한 상관관계가 있는 것을 후보로. 상관관계 분석·도메인 전문가 의견 결합.

### 3. 검증 루프
과거 데이터로 백테스트. "이 공식이면 지난 장애에서 몇 분 일찍 경보가 울렸을까?" 여러 주기 반복.

### 4. 운영 플레이북
단계별로 **누가 무엇을 하는가**를 명시.
- Level 3: 자동화, 당직 확인
- Level 6: Slack 알림, 지역 담당자 수동 조정
- Level 8: 온콜 호출, 전사 공지

### 5. 점진 배포
한 지역·일부 트래픽에 먼저 적용 → 지표 신뢰도 검증 → 전체 확산.

## 성과 지표

레벨을 도입하면 어떤 개선을 측정할지 미리 정하는 게 좋다.

- **모니터링 인원 축소** (예: 5명 → 3명)
- **의사결정 시간 단축** (10분 → 3분)
- **수동 개입 시간 감소** (주 5시간 → 없음)
- **장애 감지 속도** — 사고 직전 레벨 상승이 포착되는지

측정 없이 도입하면 "그래서 좋아졌나?"를 증명하지 못한다.

## SLI·SLO와의 관계

SRE의 SLI(Service Level Indicator)·SLO(Service Level Objective)와 **겹치지만 다르다.**

| 항목 | SLI·SLO | 운영 레벨 지표 |
|---|---|---|
| 주 청중 | SRE·엔지니어 | 전사(개발·운영·PM·경영) |
| 초점 | 가용성·지연 정량 목표 | **지금 해야 할 대응**을 단계로 |
| 시간 단위 | 분기·월 에러 버짓 | 실시간·단기 |
| 출력 | 소수점 비율 (99.95%) | 단순 단계 (1~9) |

대형 조직은 둘 다 운영. SLI·SLO로 장기 목표를, 운영 레벨로 실시간 대응을.

## 흔한 실수

- **단계만 만들고 플레이북 없음** — "Level 6이 떴다, 그래서?"
- **과도한 입력 지표 결합** — 공식이 복잡해지면 해석 불가능·디버깅 어려움
- **고정 임계값** — 시간대·계절성 무시로 오탐 폭발
- **부서 간 합의 없이 설계** — 운영자는 "너무 민감", 개발자는 "너무 둔감"
- **도입 전 기준선 미측정** — 효과 증명 불가능

## 면접 체크포인트

- **단일 레벨 지표**가 해결하는 대시보드 과부하 문제
- 수요·공급 비율 같은 **합성 지표의 예**
- **시간대별 정규화**가 왜 필요한가
- **단계별 플레이북**이 지표 자체보다 중요한 이유
- SLI·SLO와 운영 레벨 지표의 **청중·시간 단위 차이**
- 도입 전후 **측정 가능한 성과 지표**
