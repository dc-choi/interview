# S3 보안 설계 패턴 (사용 사례별)

S3 보안은 옵션 하나를 켜는 문제가 아니라 **어떤 목적의 버킷인가를 먼저 정의하는 문제**다. 버킷을 만들고 파일을 올리면 바로 쓸 수 있을 만큼 진입장벽이 낮아, "일단 되게 만들고 방치"하는 위험한 버킷이 쌓이기 쉽다. S3를 단순 저장소가 아니라 외부 노출 가능성이 있는 데이터 보관 시스템으로 보고, 사용 사례별로 다른 보안 패턴을 적용한다. 메커니즘 수준(Block Public Access, Bucket Policy 문법, 암호화 4종, 비용)은 [[S3-Security-Cost]] 참조.

## 공통 원칙

- **목적별 버킷 분리**: 정적 사이트용, CDN 리소스용, 로그 업로드용, 민감정보 보관용으로 나눈다. 한 버킷에 `/team-a`, `/static`, `/private`처럼 디렉터리만 나눠 멀티팀이 공유하면, 한 팀의 정책 변경이 다른 팀의 장애나 보안 사고로 번진다. 분리하면 정책이 단순해지고 책임 범위가 명확해진다.
- **기본값은 퍼블릭 차단**: 공개가 필요해도 S3를 직접 공개하지 말고 CloudFront 같은 앞단을 통해 제한적으로 공개한다.
- **단순하고 일관된 가이드**: 선택지(버킷 정책, ACL, 퍼블릭 차단, 암호화, CDN 연동)가 많을수록 단순 규칙으로 일관성을 유지한다.

## 사용 사례별 패턴 요약

| 사용 사례 | 핵심 패턴 |
|---|---|
| 정적 공개 사이트 | CloudFront + OAC, S3는 비공개 |
| 제한 공개 사이트 | CloudFront + AWS WAF IP Set |
| 정적 리소스 서빙 | 별도 CDN 도메인 (`cdn.example.com`) |
| 원격 파일 업로드 | 커스텀 헤더 + WAF 검사 (+ 토큰/서명) |
| 임시 파일 공유 | Presigned URL (+ Source IP 제한) |
| 민감정보 보관 | 별도 보안 계정에 분리 |

## 정적 웹사이트 호스팅

S3 단독 호스팅은 웹사이트 엔드포인트를 직접 노출하고 버킷을 퍼블릭으로 열어야 하며 오리진 정보가 그대로 드러나므로 지양한다.

권장 구성: **CloudFront 앞단** + ACM 인증서로 HTTPS + Route 53 도메인 연결 + **OAC(또는 구형 OAI)** 로 CloudFront만 S3에 접근하도록 제한. 사용자는 CloudFront/서비스 도메인으로 접근하고 S3 버킷은 외부에서 직접 접근할 수 없다.

## 제한 공개 사이트

내부 API 문서, 개발 명세, 파트너용 가이드처럼 특정 네트워크에서만 접근해야 하는 정적 사이트는 모두에게 공개하지 않는다.

- S3 버킷 정책으로 IP를 제한할 수도 있지만, CloudFront를 쓴다면 **CloudFront에 AWS WAF를 붙이는 방식**이 더 적합하다.
- WAF의 **IP Set**으로 사내 VPN, 오피스 IP, 파트너사 IP만 허용한다.
- 보안팀이 WAF 규칙을 중앙 관리하면 임의 변경을 줄이고 접근 통제 기준을 일관되게 유지한다.

## 정적 리소스 파일 서빙

이미지, CSS, JS를 S3에서 직접 불러오면 프론트엔드 코드에 S3 객체 URL이 박혀 버킷 이름과 오리진이 노출된다.

- 동적 서비스는 ALB/EC2/API 서버로, 정적 리소스는 **별도 CDN 도메인**으로 제공한다. 예: `service.example.com`은 애플리케이션 서버, `cdn.example.com`은 CloudFront + S3.
- 성능, 보안, 관리 모두 유리하다.

## 원격 파일 업로드

고객 PC나 모바일 앱에서 S3로 로그를 업로드하는 경우, 고객 네트워크를 통제할 수 없어 IP 제한만으로는 부족하다.

- 차선책: **Referer나 커스텀 헤더**를 클라이언트가 포함하도록 만들고 CloudFront와 WAF에서 그 값을 검사한다.
- 단 헤더 검사는 완벽한 인증이 아니다. 민감한 데이터 업로드라면 인증 토큰, 서명된 요청, 서버 중계 방식 같은 더 강한 통제를 함께 쓴다.

## Presigned URL

private S3 객체에 일정 시간 동안 접근하게 해주는 서명된 URL. 버킷 전체를 공개하지 않고도 특정 파일만 임시 공유할 수 있다.

- **서버 비용 절감**: 서버가 S3에서 받아 사용자에게 전달하면 서버 트래픽과 비용이 는다. 대신 서버가 Presigned URL을 발급하면 사용자가 S3에서 직접 다운로드한다.
- **주의**: 만료 전까지는 반복 호출이 가능하므로 URL이 공유되면 의도치 않은 사용자도 접근한다. TTL을 짧게 둔다.
- **Source IP 제한**: 버킷 정책에서 특정 Source IP에서만 `GetObject`를 허용하면, URL을 알아도 허용 네트워크 밖에서는 접근할 수 없다.
- **1회용 다운로드**: S3 Presigned URL 자체엔 "1회 사용 후 만료"가 없다. 패턴은 **원본을 보호하고 임시 복제본을 만든 뒤 그 복제본에 Presigned URL을 발급**, 다운로드 이벤트를 CloudTrail → EventBridge → Lambda로 감지해 복제본을 삭제하는 것. 같은 URL은 더 이상 유효하지 않게 된다.

## 민감정보 버킷: 별도 보안 계정 분리

개인정보, 기업 기밀처럼 사고 시 피해가 큰 데이터는 일반 서비스 계정 버킷에 섞지 않는다.

- 멀티 계정 환경이라면 **민감정보 전용 보안 계정**을 만들어 민감정보 버킷을 집중시킨다.
- 보안 계정에는 일반 웹 서비스나 퍼블릭 서브넷을 두지 않아 **공격 표면을 최소화**한다.
- 보안팀이나 제한된 관리자만 생성/관리하게 해 이름 규칙, 태그, 보관 주기, 문서화 기준을 일관 적용한다.

### Cross-account 접근 (양쪽 모두 제한)

서비스 계정 서버가 보안 계정 버킷에 접근해야 할 때는 최소 권한으로 설계한다.

- **서비스 계정의 IAM Role**에 필요한 S3 권한만 부여하고, **보안 계정의 Bucket Policy**에서도 그 Role만 허용한다. 접근하는 쪽 IAM 정책과 받는 쪽 버킷 정책이 **둘 다 맞아야** 접근된다.
- 여러 서버가 공유하는 Role에 권한을 추가하지 말고, **민감정보 접근 전용 Role**을 새로 만들어 꼭 필요한 서버에만 연결한다.

### 침해 사고 시 피해 범위 축소 (blast radius)

같은 계정에 일반 버킷과 민감정보 버킷이 섞이면, 서버 탈취나 Access Key 유출 시 공격자가 S3 목록을 훑어 중요한 데이터를 찾는다. 민감정보 버킷이 별도 보안 계정에 있고 서비스 계정에 목록 조회 권한이 없으면, 공격자가 서비스 계정 권한을 얻어도 버킷 이름과 경로를 알기 어렵고 권한도 제한돼 공격 난도가 오른다. 보안의 핵심은 완벽한 차단뿐 아니라 **사고가 나도 피해 범위를 줄이는 구조**다.

## 모니터링 (CloudTrail 기반)

- **설정 변경 이벤트 감시**: `CreateBucket`, `PutBucketPolicy`, 퍼블릭 접근 변경, ACL 변경. 버킷 정책이 변경되면 정책을 조회해 Principal에 외부 계정이 포함됐는지 검사한다.
- **GetObject 이상 징후**: 비정상 시간대, 대량 다운로드, 해외 IP 접근, `AccessDenied` 급증은 침해 시도 신호일 수 있다.
- GuardDuty, CloudTrail, SIEM, 자체 모니터링을 함께 써 탐지 품질을 높인다.

## Access Key 수명 주기

- Access Key는 유출되면 큰 사고로 이어지므로 **IAM Role 기반 접근을 우선**하고 발급은 예외로 제한한다.
- 발급이 필요하면 공개 티켓으로 요청을 남겨 책임성과 투명성을 확보하고 최소 권한으로 검토한다.
- 발급된 키는 **주기적으로 교체**한다. 오래된 키일수록 어디에 복사됐고 누가 쓰는지 추적이 어렵다.
- 궁극적으로 Role, 임시 자격 증명, 비밀 관리 시스템으로 전환한다.

## 운영 실천

- 새 버킷 생성 시 **이름, 태그, 목적, 공개 여부, 보관 주기, 담당자**를 반드시 기록하게 강제한다.
- CloudTrail 기반으로 위험한 변경을 자동 탐지해 빠르게 가이드하면 개발 속도와 보안을 함께 지킨다.

## 면접 체크포인트

- 정적 사이트를 S3 단독이 아닌 CloudFront + OAC로 숨기는 이유 (오리진 노출, HTTPS, 직접 접근 차단)
- Presigned URL의 한계(만료 전 반복 호출)와 Source IP 제한, 일회용 복제본 패턴
- 민감정보 버킷을 별도 보안 계정에 두는 이유 (blast radius 축소)
- Cross-account 접근에서 IAM Role과 Bucket Policy를 양쪽 다 맞춰야 하는 이유
- 위험 변경 탐지를 위해 감시할 CloudTrail 이벤트 (`PutBucketPolicy`, 퍼블릭 접근 변경 등)
