# 프로그래밍에 필요한 수학과 논리 기초

개발에 필요한 수학은 선형대수, 미적분 같은 고등수학이 아니라 **중고등 과정의 논리/이산수학** — 명제, 집합, 수학적 귀납법, n진법 수준이다. 컴퓨터가 0과 1로 계산하는 기계인 이상 프로그래밍에는 수학이 녹아 있지만, 실무에 쓰이는 대부분은 숫자 계산이 아니라 **논리적 사고**다. 핵심은 깊은 이론을 외우는 게 아니라 수식 기호에 위축되지 않는 것.

## 수식은 코드로 번역하면 단순해진다

낯선 수식이 어려워 보이는 이유는 의미가 복잡해서가 아니라 **기호가 낯설기 때문**이다. 대부분의 수식에서 알파벳은 변수, 기호는 규칙(명령문)이나 특별한 함수를 뜻한다. 하나씩 코드로 옮기면 단순해진다.

- `∑` (Sigma, 합) = 뒤에 오는 값을 반복하며 더하라. `∑_{i=1}^{N}`은 `for (let i = 1; i <= N; i++) acc += ...`와 같다. for문 관용 변수 `i`도 여기서 왔다.
- `x_i` = 어떤 집합의 i번째 원소 (`arr[i]`)
- `1/N × ∑` = N번 더한 값을 N으로 나눔 = 평균

예를 들어 평균제곱오차 `Ĉ = (1/N) ∑_{i=1}^{N} (f(x_i) − y_i)²`는 다음 한 줄짜리 루프다.

```js
function meanSquaredError(xs, ys, f) {
  let acc = 0;
  for (let i = 0; i < xs.length; i++) acc += (f(xs[i]) - ys[i]) ** 2;
  return acc / xs.length;
}
```

모르는 기호를 만나면 의미를 찾아 코드로 옮겨 본다. `∫`(적분)처럼 의미 자체가 무거운 기호도 있지만, 대부분은 변수와 규칙의 조합이라 풀어 보면 별것 아니다.

## 명제 (Proposition)

**명제 = 참, 거짓의 진릿값을 갖는 문장.** 참/거짓을 가릴 수 없는 문장(주관이 들어간 "그는 키가 크다")은 명제가 아니다.

명제의 참/거짓은 프로그래밍의 `true`/`false`, `1`/`0`과 같은 개념이고, **명제 = 조건식(condition)**이다. `if (array.includes('a'))`에서 "array에 'a'가 포함되어 있다"가 명제이고, 참이면 then 분기가 실행된다.

조건문에 들어가는 모든 조건은 반드시 명제여야 한다. 그래서 명제에 익숙한 개발자는 요구사항을 들었을 때 **그 요구를 충족하는 명제를 빠르게 세운다.** 명제는 집합, 귀납법을 포함한 모든 논리의 기초이고, 수학이 모호한 학문이 아니라 정확한 질문과 정확한 답을 다루는 학문인 이유다.

## 집합 (Set)과 논리식

논리식은 명제를 논리 연산자로 묶어 `true`/`false` 하나로 평가되는 식이다. 논리 연산자는 집합 연산과 정확히 대응한다.

| 논리 연산 | 집합 | 벤다이어그램 |
|---|---|---|
| `&&` (AND) | 교집합 (∩) | 겹치는 부분 |
| `\|\|` (OR) | 합집합 (∪) | 둘을 합친 영역 |
| `!` (NOT) | 여집합 (ᶜ) | 바깥 영역 |

그래서 복잡한 논리식은 **벤다이어그램으로 그려 볼 수 있고**, SQL의 JOIN(INNER = 교집합, OUTER = 합집합 방향)도 같은 그림으로 이해된다. 복잡한 조건식이나 JOIN을 보고 벤다이어그램이 떠오르면 코드나 자연어로 따라 읽는 것보다 빠르고 직관적으로 파악된다.

### 드 모르간의 법칙

부정이 괄호 안으로 분배될 때 AND와 OR가 뒤집힌다 (집합보다는 더 포괄적인 논리학의 법칙).

- `!(A && B)` ≡ `!A || !B`
- `!(A || B)` ≡ `!A && !B`

진가는 **자연어를 논리식으로 옮기고, 누적된 조건식을 정리할 때** 드러난다. 기획이 기능에 조건을 덧붙일 때 기존 조건까지 함께 고려해 말해 주지 않으므로, 시간에 쫓겨 짜다 보면 조건식이 괴물처럼 불어난다. 이때 벤다이어그램으로 펼치면 **같은 명제인데 서로 역(부정)의 꼴인 항**을 찾아 합치고, 드 모르간으로 부정을 정리해 더 알아보기 쉬운 단위로 조건을 쪼갤 수 있다. 복잡한 분기의 인지 부하를 줄이는 실질적 도구다 ([[Readable-Code-Cognition|읽기 좋은 코드와 인지]]).

## 귀납논증과 연역논증

수학적 귀납법을 이해하려면 두 논증법을 먼저 구분해야 한다.

| | 귀납논증 (Inductive) | 연역논증 (Deductive) |
|---|---|---|
| 형태 | 지금까지 그래왔으니 앞으로도 그럴 것이다 | 전제가 맞으면 결론도 반드시 맞다 |
| 예 | 매년 여름은 더웠다 → 여름은 항상 덥다 | 맥북은 애플이 만든다, 내 PC는 맥북이다 → 내 PC는 애플이 만들었다 (삼단논법) |
| 보장 | 전제가 다 참이어도 결론이 틀릴 수 있음 | 결론이 거짓이면 전제 중 하나가 반드시 거짓 |
| 쓰임 | 새 지식 탐구(현대 과학의 가설 제시) | 이미 전제에 담긴 것의 증명 |

프로그래밍의 목표는 새 지식 탐구가 아니라 **내 코드가 오류 없이 성립하는가의 증명**이므로 연역논증이 알맞다.

## 수학적 귀납법 (Mathematical Induction)

이름과 달리 수학적 귀납법은 **귀납이 아니라 연역논증**이다. 어떤 명제 P에 대해 다음 두 가지를 보이면 모든 자연수에서 P가 참임이 보장된다.

1. **기저 단계**: P(1)이 참이다.
2. **귀납 단계**: P(n)이 참이면 P(n+1)도 참이다.
3. 따라서 P는 모든 자연수에서 참이다.

도미노로 비유하면, 첫 도미노가 쓰러지고(P(1)) 임의의 n번째가 쓰러질 때 항상 n+1번째도 쓰러진다면(P(n)→P(n+1)), 첫 도미노를 밀면 전부 쓰러진다. 전제의 참을 먼저 보인 뒤 그로부터 보편 결론을 끌어내는 전형적 연역이다.

### 알고리즘 정당성 검증

알고리즘은 보편 규칙이고 어떤 형태로든 반복(또는 재귀) 요소를 가지므로 수학적 귀납법으로 정당성을 검증하기 좋다. 팩토리얼 재귀가 대표 예다.

```js
function factorial(n) {
  if (n < 1) return 1;        // 기저: 0! = 1
  return n * factorial(n - 1); // 귀납: n! = n × (n-1)!
}
```

- 기저: `n = 0`일 때 `0! = 1`로 올바르다.
- 귀납: `(n-1)!`이 올바르면 `n! = n × (n-1)!`도 올바르다.
- 따라서 모든 자연수 n에서 올바르다.

재귀 함수의 기저 조건과 재귀 호출이 곧 귀납법의 기저 단계와 귀납 단계에 대응한다 ([[tech/computer-science/algorithm/알고리즘(Algorithm)|재귀와 알고리즘]]). 당장의 코딩보다, 복잡한 문제에서 **일반화된 해결법을 찾는 사고력**을 길러 준다.

## 실무 적용

- 요구사항을 들으면 그것을 충족하는 **명제(조건식)부터** 세운다.
- 분기 조건이 불어나면 벤다이어그램으로 펼치고 **드 모르간으로 부정을 정리**해 중복, 역항을 합친다.
- 반복/재귀 로직의 정당성은 **기저 + 귀납 단계**로 점검한다.
- 낯선 수식, 라이브러리 알고리즘은 기호를 변수/규칙/함수로 보고 **코드로 옮겨** 이해한다.

## 면접 체크포인트

- 명제와 조건식이 같은 개념임을 설명할 수 있는가
- `&&`/`\|\|`와 교집합/합집합의 대응, 드 모르간 법칙을 조건식 정리에 적용해 봤는가
- 귀납논증과 연역논증의 차이, 그리고 수학적 귀납법이 사실 연역인 이유를 말할 수 있는가
- 재귀 함수의 정당성을 기저/귀납 단계로 검증할 수 있는가
- 타입 시스템에서 명제/증명 대응(커리-하워드)으로 확장해 설명할 수 있는가 ([[Types-As-Proofs|타입은 증명이다]])
