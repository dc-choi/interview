# 코드 가독성의 인지과학 — "왜 어떤 코드는 읽기 쉬운가"

가독성을 **취향이 아닌 인지 비용**의 문제로 본다. 작업 기억, 청킹, 듀얼 프로세스 이론, 게슈탈트 원리, 인지 부하 이론, 예측 부호화의 관점에서 "읽기 쉬운 코드"가 무엇인지 정의한다. 구체적 안티패턴 사례는 [[Code-Readability-Dark-Patterns]]를 함께 본다.

## 한 줄 요약

> 가독성은 **외재적 인지 부하를 줄여서**, 작업 기억 슬롯을 본질적 문제 해결에 남기는 일이다.

## 1. 작업 기억의 한계 — 슬롯은 3~4개

조지 밀러의 후속 연구 기준 작업 기억은 **약 3~4개의 청크**만 동시에 다룬다. 코드를 읽을 때 슬롯에 들어가는 항목들:

- 변수의 현재 값
- 제어 흐름의 위치
- 함수 호출의 컨텍스트
- 현재 스코프의 상태

한 줄짜리 표현식이라도 **삼항·옵셔널 체이닝·체크 + 계산**을 한 줄에 모으면 슬롯 4개를 즉시 채운다. 그 다음 줄을 읽을 자원이 없어 "이거 뭐가 복잡한데" 라는 감각이 생긴다.

**대응**: 의미 단위로 변수 추출, early return으로 분기 깊이 축소, 함수 분리.

## 2. 청킹 — 전문가는 패턴 단위로 본다

장기 기억에 저장된 패턴(청크)을 불러와 작업 기억의 부담을 줄이는 메커니즘.

- **전화번호 예시**: `010-1234-5678`이 `01012345678`보다 쉬운 이유 — 익숙한 `010`이 1청크로 처리됨
- **체스 마스터 실험**: 의미 있는 포진은 빠르게 외우지만, 무작위 배치는 초보자와 차이 없음. **패턴 인식이 능력의 본체**

**코드에서**: `users.filter(...).map(...)`을 초보자는 메서드 하나하나 해석하지만 숙련자는 **"필터 후 변환"** 한 청크로 인식. 관용적인 코드가 읽기 쉬운 진짜 이유는 "올바른 코드라서"가 아니라 **이미 장기 기억에 청크로 저장돼 있기 때문**이다.

**대응**: 팀이 공유하는 관용 패턴(linter 룰, 컨벤션 문서, 라이브러리 선택)을 의도적으로 좁힌다. 매번 새 패턴을 도입하면 청크가 늘어나지 않고 매번 시스템 2가 호출된다.

## 3. 시스템 1 vs 시스템 2 (Kahneman)

| 시스템 | 특성 | 코드에서 |
|---|---|---|
| **시스템 1** | 빠르고 자동 | 익숙한 `for`, `if`, `map/filter`, 잘 알려진 패턴 |
| **시스템 2** | 느리고 의식적 | 이중 부정, 드모르간, 비트 트릭, 중첩 삼항 |

**"읽기 어렵다"의 정체**: 시스템 1로 흘러가야 할 코드가 시스템 2를 호출하는 순간 **인지적 긴장(cognitive tension)** 이 발생한다. 이 긴장이 누적되면 리뷰 속도와 정확도가 동시에 떨어진다.

```ts
// 시스템 1
const canBuy = user.age >= 18 && user.isVerified;

// 시스템 2 호출
const canBuy = !(user.age < 18 || !user.isVerified) && user.age !== undefined;
```

## 4. 게슈탈트 원리 — 시각이 의미를 만든다

뇌는 시각적 그루핑으로 의미를 추론한다.

- **근접성(Proximity)**: 빈 줄 하나로 관련 라인을 묶는다. "사용자 정보"와 "권한 처리"를 분리하면 자동으로 두 그룹으로 인식
- **유사성(Similarity)**: `fetchUser`/`getOrders`/`pmtHistory` 처럼 형식이 깨지면 "같은 종류"로 묶이지 않는다. 일관된 prefix(`fetchXxx`)가 핵심
- **연속성(Continuity)**: 깊은 중첩은 시선의 흐름을 끊는다. **early return / guard clause**로 위에서 아래로 흐르는 흐름을 만든다

## 5. 인지 부하 이론 (Sweller) — 세 가지 부하

| 부하 | 무엇 | 다룰 수 있는가 |
|---|---|---|
| **내재적(Intrinsic)** | 문제 자체의 본질적 복잡도 | 줄이기 어려움 |
| **외재적(Extraneous)** | 표현 방식이 만드는 불필요한 복잡도 | **줄여야 함** |
| **본유적(Germane)** | 스키마(청크) 형성을 돕는 부하 | 유익함 |

**가독성 개선 = 외재적 부하 최소화**. 내재적 복잡도는 도메인이 정한다(분산 시스템 동기화는 본질적으로 복잡). 그러나 같은 문제를 일관된 네이밍·평탄한 구조·명시적 타입으로 표현하면 외재적 부하는 0에 가까워진다.

(주의: 2023년 체계적 문헌 리뷰는 **Cyclomatic Complexity 같은 기계적 메트릭과 실제 인지 부하의 상관관계가 약함**을 보고. 메트릭 숫자에 매몰되지 말고 실제 독해 경험을 본다.)

## 6. 예측 부호화 — 뇌는 다음을 예측한다

뇌는 수동적 수신 장치가 아니라 **다음 토큰을 끊임없이 예측**한다. 예측이 맞으면 시스템 1로 빠르게 흐르고, 틀리면 시스템 2가 깨워진다.

```ts
async function fetchUserProfile(userId: string) {
  const res = await api.get(`/users/${userId}`);
  globalEventBus.emit('user-fetched', res.data);   // 예측 위반 — fetch가 발행한다고?
  localStorage.setItem('lastUser', JSON.stringify(res.data));  // 예측 위반 — fetch가 저장한다고?
  return res.data;
}
```

**함수명이 약속한 것 외의 일을 하면 예측이 깨진다.** 이것이 SRP가 "취향"이 아닌 **인지 비용 절감**인 이유. 모든 수준(함수명·변수명·파일 구조·API 응답 형식·에러 모양)에서 예측 가능성이 곧 가독성.

## 가독성 좋은 코드의 6가지 조건 (종합)

1. **작업 기억 초과 방지**: 동시 추적할 컨텍스트 ≤ 3~4개
2. **기존 청크와 매칭**: 팀이 공유하는 관용 패턴 사용
3. **시스템 1로 처리 가능**: 트릭·이중 부정·비트 연산 회피
4. **시각적 구조 일치**: 게슈탈트 원리(근접·유사·연속) 준수
5. **예측 가능성**: 이름이 약속한 행동만 한다 (SRP)
6. **외재적 부하 최소화**: 형식·표현이 본질을 가리지 않게

## 익숙함 편향 주의

이 모델의 **함정**: "내가 읽기 어렵다 = 객관적으로 나쁜 코드"가 아닐 수 있다. 익숙하지 않은 패턴(예: 함수형 파이프라인, 모나드, 옵셔널 체이닝)은 처음에는 시스템 2를 부른다. 이때 거부하지 말고 **본유적 부하**로 받아들여 청크를 만드는 게 장기적으로 이득. **외재적 부하(트릭)** 와 **본유적 부하(새 청크)** 를 구별하는 감각이 시니어 시그널.

## 면접 체크포인트

- "읽기 쉬운 코드"를 **취향이 아닌 인지 비용**으로 설명할 수 있는가
- **작업 기억 한계와 청크**의 관점에서 함수 분리를 정당화할 수 있는가
- **시스템 1/2** 관점에서 비트 트릭이나 중첩 삼항이 왜 위험한가
- **예측 부호화 = SRP** 의 인지 과학적 근거를 설명할 수 있는가
- 코드 리뷰 피드백이 **"전환 비용"의 신호**라는 시각을 제시할 수 있는가
- **익숙함 편향**을 인정하고 새로운 패턴 학습과 구분할 수 있는가
