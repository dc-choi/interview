# 컴파일과런타임

## 프로그래밍이란
- 일의 진행 순서를 정하는 것 = 절차적 글쓰기
- 글로 작성한 순서대로 수행하는 것 = 실행
- 모든 실행은 고유한 흐름을 가지며, 프로그램의 근본 목적은 **자동화**

## 고급어vs저급어

| 구분 | 설명 | 예시 |
|------|------|------|
| 저급어 | 기계어, 인간이 이해하기 어려움 | 어셈블리, 기계어(0/1) |
| 고급어 | 인간이 이해하기 쉬운 언어 | C, Java, Python, JS |

- 고급어의 코드는 보통 함수 단위로 묶어 표현
- 고급어 → 기계어로 변환하는 것이 **컴파일**

## 실행방식3가지

### 네이티브(Native)
- 대표: **C, C++, Rust**
- 컴파일하면 OS와 CPU에 의존적인 실행 파일 생성
- 장점: 실행 속도가 가장 빠름
- 단점: 플랫폼마다 별도 컴파일 필요

### 매니지드(Managed)
- 대표: **Java, C#, Kotlin**
- VM(가상 머신)이 CPU를 가상으로 구현
- Java → 컴파일 → `.class`(바이트코드) → JVM에서 실행
- 장점: 플랫폼 독립적 ("Write Once, Run Anywhere")
- 단점: VM 오버헤드로 네이티브보다 느릴 수 있음

### 인터프리터(Interpreter)
- 대표: **JavaScript, Python, Ruby**
- 소스코드를 직접 한 줄씩 실행
- 장점: 변경 시 빠르게 적용, 유연성 뛰어남
- 단점: 실행 속도가 컴파일 방식보다 느림

### JIT컴파일(보강)
- **Just-In-Time Compilation**: 인터프리터와 컴파일의 하이브리드
- 자주 실행되는 코드(Hot Code)를 런타임에 기계어로 컴파일하여 캐싱
- **V8**(Chrome, Node.js): JS를 JIT 컴파일
- **HotSpot**(Java): 바이트코드를 JIT 컴파일
- 초기 실행은 느리지만, 반복 실행 시 네이티브에 가까운 성능

## API와SDK

| 구분 | 비유 | 설명 |
|------|------|------|
| API | 공구 하나 | 함수, 특정 기능을 호출할 수 있는 인터페이스 |
| SDK | 공구 세트 | 함수 세트 + 도구 + 문서, 개발에 필요한 도구 모음 |
