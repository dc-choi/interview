# SOLID원칙실전적용

> 출처: [[App-Architecture-OOP|애플리케이션 아키텍처와 객체지향]]

## 변경에강한설계가필요한이유

> "변하지 않는 것은 모든 것이 변한다는 사실 뿐이다" — 헤라클레이토스

- 초반에 요구사항이 어떻게 변경될지 아는 사람은 없음
- 대부분의 코드는 요구사항 변경에 부적합
- 변경이 발생할 때마다 **리팩토링이 필수**

## OCP(개방-폐쇄원칙)

**기존 코드를 수정하지 않으면서 새로운 확장이 가능해야 한다.**

### 예시:할인정책변경
기존: 할인 정책이 1개 → 할인 중복 허용 요구사항 추가

**트랜잭션 스크립트에서는:**
- 기존 코드를 직접 수정해야 함
- 절차 안에 개념이 암묵적으로 숨겨져 있어 변경 포인트 찾기 어려움
- 개발자는 기존 코드 수정에 대한 두려움이 있음

**도메인 모델에서는:**
- **Composite 패턴**으로 기존 할인 정책을 조합하는 통합 할인 객체 생성
- 기존 코드를 건들지 않으면서 새로운 확장만 추가
- 도메인 개념이 클래스로 명확하게 표현됨

## DIP(의존성역전원칙)

**고수준 모듈이 저수준 모듈에 의존하지 않고, 둘 다 추상화에 의존해야 한다.**

- 할인 정책 확장 시 추상화(인터페이스)에 의존하게 됨
- 구체적인 할인 정책(고정금액, 퍼센트)은 추상화를 구현
- 새로운 할인 정책을 추가해도 기존 코드에 영향 없음

## 추상화의본질

추상화에 대한 선입견: 반드시 추상 클래스나 인터페이스를 사용해야 한다?

**본질적으로 추상화 = 잘 변하지 않는 것**

- 두 객체 사이에 잘 변하지 않는 중간 객체를 넣으면 그것이 추상화
- 인터페이스는 추상화의 한 방법일 뿐, 전부가 아님

## 면접포인트
- "OCP를 실무에서 어떻게 적용하나?" → Composite, Strategy 패턴으로 확장 지점 설계
- "SOLID 중 가장 중요한 것?" → OCP와 DIP가 변경에 강한 설계의 핵심
- "추상화란?" → 잘 변하지 않는 것, 반드시 인터페이스일 필요 없음
