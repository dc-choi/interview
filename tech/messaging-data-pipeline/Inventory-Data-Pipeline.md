# 재고 데이터 파이프라인 (Inventory Data Pipeline)

오프라인 매장 재고를 온라인 구매 여정(상품 전시, 검색, 장바구니, 주문서)에 제공하는 데이터 파이프라인. 옴니채널의 핵심 난제는 재고 정합성과 처리량, 지연을 동시에 지키는 것이다 — 재고는 변동 이벤트가 끊임없이 발생하고, 하나의 변동이 여러 파생 모델(품절 여부, 시계열, 이력)로 퍼지며, 사용처마다 요구하는 정합성 수준이 다르다. 이 특성이 만드는 전형적 병목 3가지와, 이벤트 기반 아키텍처로 푸는 설계.

## 전형적인 병목 3가지

1. **기초재고 배치 리드타임** — 매일 전 매장의 기초재고 스냅샷을 만드는 배치가 매출, 상품 수, 변동 이벤트 증가로 계속 느려진다. 기초재고가 늦거나 틀리면 하루의 시작부터 재고가 부정합.
2. **단일 동기 파이프라인의 연쇄 지연** — 변동 이벤트 하나를 받아 매장재고, 시계열 재고, 품절 여부 등 서로 다른 데이터 모델을 한 파이프라인 안에서 동기적으로 모두 가공해 적재. 결합도가 높아 한 모델의 지연이 전체로 전파된다.
3. **일률적인 Pull API** — 모든 사용처가 단일 API를 주기적으로 폴링. 검색, 전자라벨처럼 자체 데이터 스토어를 가진 서비스까지 변경되지 않은 데이터를 통째로 재조회해 불필요한 Read 부하와 오버헤드가 쌓인다.

## 개편 축 1 — 기초재고 배치: 파티셔닝과 코드화

- **자연스러운 비즈니스 단위를 파티션 키로**: 매장을 그룹으로 나눠 Spring Batch Partitioning의 Worker Step들이 독립 병렬 처리한다. gridSize로 분할 수를 조정하고 taskExecutor로 멀티스레딩. 인위적 분할 기준을 발명하기보다 도메인에 이미 있는 단위(매장)를 쓰는 것이 분배와 확장 모두 단순하다.
- **프로시저 제거**: DB 프로시저에 갇힌 로직을 애플리케이션 코드 기반 배치로 전환. 테스트와 버전 관리가 가능해지고, 파티셔닝 같은 구조적 최적화의 전제가 된다.
- **오케스트레이션**: Airflow(관리형은 Amazon MWAA)로 파이프라인 전체를 코드로 관리하고 스케줄링과 재시도를 표준화.

## 개편 축 2 — 적재: 임계 경로 최소화 + Kafka 팬아웃

- **동기 구간 최소화**: 부정합 비용이 가장 큰 핵심 모델(매장재고)만 동기 적재하고, 즉시 변동 재고 토픽에 이벤트를 발행한 뒤 트랜잭션을 끝낸다. 쓰기 임계 경로가 핵심 모델 1개 적재 + 이벤트 발행으로 줄어든다.
- **파생 모델은 독립 구독**: 품절 여부, 시계열 재고, 재고 이력은 같은 토픽을 각자의 컨슈머 그룹으로 구독한다. 오프셋이 그룹별로 분리 관리되므로 한 모델의 처리 지연이 다른 모델로 전파되지 않는다 ([[Fan-Out-Architecture|팬아웃]], [[Consumer-Group|컨슈머 그룹]]).
- 파생 모델들은 각자의 속도로 따라오는 최종 일관성이 된다 — 어떤 모델이 얼마나 뒤처지는지 컨슈머 lag 모니터링이 필수가 되는 이유.

## 개편 축 3 — 전달: Push와 Pull 분리

| 방식 | 대상 사용처 | 동작 | 일관성 |
|---|---|---|---|
| Push (이벤트 구독) | 자체 데이터 스토어 보유 (검색, 전자라벨) | 변동 재고 또는 품절 여부 토픽을 직접 구독해 자율 동기화 | 최종 일관성 |
| Pull (실시간 API) | 구매 결정 순간 (상품 상세, 장바구니, 주문서) | 매장재고 실시간 조회 API | 강한 정합성 |

- **분리 기준은 부정합의 비용**: 구매 결정 순간의 재고 오류는 주문 실패와 고객 경험 훼손으로 직결되므로 Pull로 실시간 정합성을 지키고, 파생 사용처는 변경 이벤트만 받아도 충분하므로 Push로 전환한다. 모든 사용처에 일률적 API를 주는 대신 사용처 성격별로 계약을 나누는 것.
- Push 사용처가 자체 스토어를 이벤트로 미리 동기화해 두는 구조는 CQRS의 읽기 모델 분리와 같은 축이다.

## 트레이드오프

- 단순 동기 구조 대비 토픽, 컨슈머 그룹, 배치 파티션, 오케스트레이터가 늘어나 운영 복잡도가 커진다. 컨슈머 lag, DLQ 같은 모니터링이 전제 조건.
- Push 사용처는 완벽한 실시간성을 포기하고 최종 일관성을 수용한다 — 허용 가능한 사용처에만 적용해야 하는 이유.
- 전면 전환보다 재고 유형별 점진 마이그레이션이 현실적이다.

## 사례

전국 1,400여 개 매장의 재고를 온라인몰에 제공하는 옴니채널 리테일 사례. 매일 1,200만여 건의 기초재고 생성이 평균 1~2시간까지 늘어난 상태에서, 약 2만 라인의 DB 프로시저를 전면 제거하고 매장 그룹 단위 Spring Batch Partitioning과 Amazon MWAA로 개편해 리드타임을 평균 50분에서 25분으로 50% 단축했다. 적재는 매장재고만 동기 처리한 뒤 Kafka 변동 재고 토픽으로 팬아웃하고, 전달은 Push와 Pull 하이브리드로 분리해 Push 전환한 특정 사용처의 API 호출을 약 90% 줄였다.

## 면접 체크포인트

- "하나의 이벤트로 여러 데이터 모델을 갱신해야 한다면?" → 동기 구간에는 부정합 비용이 가장 큰 핵심 모델만 두고, 나머지는 컨슈머 그룹 독립 구독으로 지연을 격리.
- "대용량 일 배치가 계속 느려진다면?" → 도메인의 자연스러운 단위를 파티션 키로 삼은 Partitioning으로 수평 분할. Reader의 OFFSET 병목 같은 단일 스텝 최적화와는 별개 축 ([[Spring-Batch-Essentials-Performance|Spring Batch 성능]]).
- "모든 사용처가 같은 API를 폴링하고 있다면?" → 사용처를 정합성 요구로 분류해 Push와 Pull을 분리. 판단 기준은 부정합의 비용.
- 프로시저 로직을 코드로 옮기는 가치 → 테스트, 버전 관리, 병렬화 같은 구조적 최적화의 전제 확보.
