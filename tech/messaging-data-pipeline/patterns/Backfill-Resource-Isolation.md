# 데이터 백필과 자원 격리 전략

백필(backfill)은 **장애, 네트워크 단절, 다운스트림 DB 마비 등으로 유실/누락된 데이터를 다시 채워 넣는 재전송 프로세스**다. IoT, 금융, 실시간 모니터링의 시계열 파이프라인처럼 초당 수만 건 처리량을 다루는 시스템에서, 백필은 평상시 트래픽 위에 **수백만 건이 한꺼번에 얹히는 이례적 부하**라서 운영 트래픽과 자원을 공유하면 그 자체가 2차 장애가 된다.

## 격리 없는 백필의 3가지 실패 모드

1. **자원 고갈로 실시간 서비스 마비** — 실시간 요청과 백필이 공용 HTTP 스레드 풀을 공유하면, 재전송 워커가 CPU/메모리/스레드를 독점해 정작 핵심 실시간 요청이 스레드를 할당받지 못하고 Connection Timeout이 난다.
2. **시간 기반 스케줄링의 부하 스파이크** — "N초마다 스케줄러를 돌려 큐를 전부 비운다" 방식은 쌓인 양에 비례해 한 번에 투하한다. 특정 시간대에 데이터가 폭증하면 다음 스케줄 타임에 수만 건 쿼리가 동시 투하 → DB 커넥션 풀 마비, CPU 100%.
3. **동기 블로킹 대기의 리소스 낭비** — sleep, latch 같은 블로킹 대기로 완료를 기다리면 대기 스레드가 메모리를 계속 점유하고 컨텍스트 스위칭만 늘린다.

## 격리 4계층

### 1. 메시징 토픽과 버퍼 큐 분리 (인프라, 자료구조 레이어)

실시간 데이터 토픽과 **백필 전용 토픽을 브로커 레벨에서 분리**하고, 백필 수용용 **독립 인메모리 버퍼 큐**를 둔다. 버퍼 큐는 락 기반 큐 대신 **CAS 기반 lock-free 논블로킹 큐**(Java의 ConcurrentLinkedQueue류)를 쓰면 초당 수만 건 유입도 동시성 경쟁 없이 받는다. 조건: 극단적 동시성에서는 CAS 스핀 오버헤드가 커질 수 있다.

### 2. 전용 워커 풀 분리 (스레드 레이어) — 벌크헤드의 적용

백필 처리를 **전용 스레드 풀**로 돌려 실시간 요청 처리 스레드를 오염시키지 않는다. [[External-Service-Resilience|벌크헤드 패턴]]을 백필에 적용한 것.

- 예시 설정 (Spring ThreadPoolTaskExecutor): core 5, max 10, 대기 큐 500.
- 포화 정책을 **CallerRunsPolicy**로 두면 풀이 가득 찼을 때 제출자 스레드가 직접 실행하게 되어 **제출 속도가 자연히 느려지는 백프레셔**로 작동한다.
- 효과: 백필 수백만 건 유입 중에도 실시간 유저 응답을 독립적으로 유지 (사례 기준 0.1초 미만).

### 3. 크기 기반 청크 흐름 제어 (백엔드 흐름)

시간 기준(스케줄마다 전부 처리) 대신 **데이터 개수 기준**으로 꺼낸다 — 큐에서 poll하며 청크가 기준 수(예: 1000건)에 차면 벌크 인서트, 남은 잔여분은 마지막에 한 번. 유입이 수십만 건으로 튀어도 **DB에 들어가는 단위는 항상 일정**해 처리량이 평탄해진다. 청크 크기 최적값은 DB마다 다르다 (1000건은 PostgreSQL 기준 예시 — 벤치마크로 조정).

### 4. 논블로킹 완료 검증 (비동기 리소스)

"백필이 다 끝났는가"를 블로킹 대기 없이 감지한다:

- 스트림 종료 신호(End-of-Stream 플래그)가 오면 상태를 COMPLETING으로 전환
- 워커는 잠들지 않고 큐의 `isEmpty()`를 논블로킹으로 확인, 비는 즉시 최종 집계를 비동기 트리거
- 네트워크 지연으로 종료 신호가 데이터보다 먼저 오는 레이스는, 정상 응답 이벤트를 인메모리 화이트리스트에 먼저 등록해 두고 교차 검증하는 식으로 방어

## 교훈 — 병목은 자리만 옮긴다

고성능 분산 브로커(NATS 등)를 도입해도, 받아내는 애플리케이션 내부의 자료구조, 스레드 모델, 흐름 제어가 부실하면 병목은 브로커에서 앱으로 자리만 옮겨 다시 발생한다. 토픽 분리 + 전용 풀 + lock-free 버퍼 + 크기 기반 청크가 **유기적으로 결합**해야 파이프라인이 완성된다.

## 트레이드오프

- 독립 버퍼 큐만큼 힙 메모리 소비 증가, 전용 스레드만큼 스택 오버헤드
- 인프라 토픽, 자료구조, 스레드 모델을 모두 관리해야 하는 복잡성
- 스레드 풀 상태, 큐 크기, DB 커넥션 풀 모니터링이 전제 조건

Node.js 환경 매핑: 단일 스레드 이벤트 루프라 스레드 풀 오염 대신 **이벤트 루프 점유와 커넥션 풀 공유**가 같은 문제 축이다 — 큐 컨슈머의 동시성 제한과 별도 프로세스 워커([[NestJS-Queues|BullMQ separate process]]), 백필 전용 DB 커넥션 풀 분리로 같은 격리를 구현한다.

## 면접 체크포인트

- 백필/재처리가 운영 트래픽과 자원을 공유할 때의 3가지 실패 모드
- 시간 기반 vs 크기 기반 흐름 제어 — 스파이크 유무가 갈리는 이유
- 전용 풀의 포화 정책(CallerRunsPolicy)이 백프레셔로 작동하는 원리
- 벌크헤드 패턴과의 관계 (자원 격리의 파이프라인 적용)
- 완료 감지를 블로킹 대기 없이 하는 법과 종료 신호 레이스 방어
