# Modular Monolith

단일 배포 단위 안에서 도메인별로 강한 경계를 가진 모듈들로 시스템을 구성하는 아키텍처. 모노리스의 단순한 운영과 마이크로서비스의 독립성 사이의 현실적인 중간 지점이다. 좋은 아키텍처의 기준은 멋진 구조가 아니라 변경을 빠르고 안전하게 수용해 비즈니스 가치를 전달하는 것(→ [[Why-Architecture-Matters]])이며, 모듈러 모노리스는 분산 시스템의 복잡도 없이 그 경계를 미리 연습하게 해준다.

## 왜 나오는가 — 성급한 MSA의 역설

마이크로서비스가 항상 정답은 아니다. 조직 규모, 운영 역량, 시스템 복잡도에 안 맞으면 오히려 복잡도와 운영 부담을 키운다. MSA에서 다시 모노리스(모듈러)로 합치는 선택이 합리적일 수 있는데, MSA가 나빠서가 아니라 상황에 안 맞아서다.

- **소수 인원 다수 서비스 → 경계 붕괴**: 개발자가 여러 서비스를 동시에 맡으면 경계를 지키기보다 일단 동작하게 만드는 선택이 반복되고 기술 부채가 분산된 채 쌓인다.
- **공유 데이터 모델은 숨은 폭탄**: 서비스는 나뉘었는데 여러 서비스가 같은 테이블, 같은 모델을 직접 수정하면 진짜 독립성이 없다. 한쪽 변경이 다른 쪽 장애로 번지는 분산된 스파게티가 된다. MSA의 전제는 서비스별 자기 DB 소유다 (→ [[Monolith-vs-Microservice]] Fowler 특성 6번).
- **서비스 분리는 커뮤니케이션 비용을 만든다**: 여러 서비스가 함께 바뀌어야 하는 요구가 잦은데 팀이 다르면 배포 일정, API 변경, 테스트 범위, 장애 책임을 모두 조율해야 한다. 한 팀 안 대화로 끝날 일이 팀 간 일정 조율 문제가 된다. 서비스를 나누는 것은 코드만이 아니라 소통 비용을 나누는 것이다.

## 구조

- 배포는 하나, 내부는 도메인별 모듈(예: 상품, 주문, 배송). 각 모듈은 자기 책임을 갖고 외부에는 공개 인터페이스만 노출하며 내부 구현은 숨긴다.
- 모듈 간 협력은 공개 API를 통해서만. 다른 모듈의 내부 클래스나 테이블을 직접 참조하지 않는다.
- 원칙은 응집도는 높이고 결합도는 낮추는 것 — 같이 바뀌는 것은 한 모듈에, 따로 바뀌는 것은 다른 모듈에.

## 도메인 기준 모듈 (레이어 기준 아님)

controller, service, repository 같은 기술 계층으로만 나누면 기능 하나를 바꿀 때 모든 계층을 건드린다. 상품, 주문, 배송처럼 비즈니스 기능으로 나누면 변경 범위가 명확해지고, 각 모듈이 스스로 동작하고 스스로 테스트 가능하며, 나중에 독립 서비스로 떼기 쉽다. 같은 원리가 패키지 구조에도 적용된다 (→ [[Elegant-OOP-Design|레이어 중심 vs 도메인 중심 패키지]]).

## 경계는 의지가 아니라 도구로 강제한다

개발자가 늘 규칙을 지킬 것이라 기대하면 경계는 쉽게 무너진다. 컴파일러, 빌드 도구, 프레임워크의 힘을 빌린다.

- **접근 제어자**: Java의 public, protected, package-private로 공개할 것과 숨길 것을 구분한다. 모듈의 공개 API만 public, 내부 구현은 package-private으로 막는다.
- **빌드 도구**: Gradle 멀티 프로젝트로 모듈을 별도 프로젝트로 두고 의존성 방향을 명시적으로 통제한다. 허용되지 않은 의존은 컴파일에서 걸린다.
- **프레임워크**: Spring에서 모든 빈이 한 컨테이너에 있으면 내부 구현까지 서로 참조할 수 있다. 모듈별로 별도 ApplicationContext를 두고 공개할 빈만 부모 컨텍스트로 노출하면, 같은 애플리케이션 안에서도 모듈 경계가 강해진다. (Spring Modulith, ArchUnit으로 경계 위반을 테스트로 잡을 수도 있다.)

핵심은 내부 구현은 숨기고 공개 API만 쓰게 만드는 것이다.

## 헥사고날과의 관계

모듈 내부를 외부 기술로부터 보호하는 것은 [[DDD&Hexagonal|헥사고날 아키텍처]]의 원리와 같다. 도메인 로직을 중앙에, 데이터베이스, 메시징, 웹 API 같은 기술을 바깥에 두고 인터페이스로만 소통하면, 자주 바뀌는 바깥을 교체 가능한 부품으로 다룰 수 있어 DB나 외부 호출 방식을 바꿔도 핵심 로직은 덜 흔들린다. 각 모듈을 하나의 작은 헥사곤으로 구성한다고 보면 된다.

## 점진적 서비스 추출

모듈러 모노리스의 실익은 필요할 때 특정 모듈을 독립 서비스로 떼어낼 수 있다는 점이다. 경계가 이미 잡혀 있으면 바뀌는 것은 주로 통합 방식이다 — 내부 인터페이스 호출을 HTTP API 호출로 교체하는 정도. 처음부터 경계를 잘 잡아두면 단일 시스템과 분산 시스템을 양방향으로 오갈 여지가 생긴다. 레거시를 점진적으로 떼어내는 전략은 [[Legacy-Modernization-Strategies|Strangler Fig]]와 연결된다.

## 콘웨이 법칙과 함께 설계한다

시스템 구조는 조직의 의사소통 구조를 닮는다(콘웨이 법칙). 독립 팀이 명확한 비즈니스 책임을 가지면 서비스 분할이 효과적이지만, 작은 팀이 많은 서비스를 억지로 운영하면 혼란만 커진다. 조직이 아직 하나의 팀이라면 모듈러 모노리스가 조직 구조에 더 잘 맞는다. 아키텍처는 조직 구조와 함께 설계해야 한다.

## 언제 무엇을

- **모듈러 모노리스**: 작은~중간 팀, 도메인 경계를 학습, 검증하는 중, 분산 운영 역량이 아직 없을 때. MSA 전환 판단을 미루면서도 경계를 미리 연습한다.
- **MSA로 승격**: 팀이 커져 독립 배포가 필요하고, 모듈별 부하나 기술 요구가 크게 다르며, 인프라 자동화와 관측이 준비됐을 때 (→ [[Monolith-vs-Microservice]]).
- 먼저 모노리스를 잘 다루지 못하면 MSA에서도 같은 문제가 분산되어 반복된다. 질문은 무엇이 더 멋진가가 아니라, 지금 우리 조직과 시스템이 변경을 가장 안전하고 빠르게 수용하는 방법은 무엇인가이다.

## 면접 체크포인트

- 모듈러 모노리스가 모노리스, MSA와 다른 점 (단일 배포 + 강한 내부 경계)
- 공유 DB가 MSA의 독립성을 깨는 이유 (분산된 스파게티)
- 모듈 경계를 의지가 아니라 도구로 강제하는 방법 (접근 제어자, Gradle, ApplicationContext)
- 도메인 기준 모듈과 레이어 기준 모듈의 변경 범위 차이
- 모듈에서 서비스로 추출할 때 실제로 바뀌는 것 (통합 방식, 내부 호출 → HTTP)
- 성급한 MSA의 함정과 Monolith First
