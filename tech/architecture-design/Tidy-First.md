# TidyFirst:코드정리의기술

> Kent Beck 저, "Tidy First?" 번역자 세미나 정리

## 핵심철학

### 아기발걸음으로학습하기
- Kent Beck은 bottom-up 방식을 선호
- 작은 단계부터 시작하여 점진적으로 확장

### ManInTheMirror
- 먼저 나 자신부터 변화시키기
- 남을 통제하는 건 어렵지만, 나를 통제하는 건 상대적으로 쉬움
- 레거시 정리도 마찬가지 — 나부터 실천해야 팀이 따라옴

## 코드정리vs리팩터링

| 구분 | 코드정리(Tidying) | 리팩터링(Refactoring) |
|------|------------------|---------------------|
| 모양/행위 | 동일 | 동일 |
| 규모 | 작음 | 더 큼 |
| 영향 범위 | 국소적 | 변경 영향이 큰 코드 구조 변경 |
| 의미 | 코드를 유기체처럼 다루는 연습 | 구조적 개선 |

## 1부:코드정리법

코드를 **유기체**로 다루는 법을 익히는 것이 목적:
- 갈래를 명료하게
- 줄이고
- 따로 또 같이 정리하면서 유기체처럼 짜기

## 2부:코드정리관리

### B와S의구분
- **B(Behavior)**: 기능적 요구사항, 경제적 가치를 만드는 일
- **S(Structure)**: 개발자로서 해야 하는 코드 구조 개선
- 이 둘을 명확히 구분하여 커밋/배포해야 함

### 성장단계
```
닥치는대로 코딩 → 작업 종류 인지(B/S 구분) → 한번에 담기(배포&통합) → 나누어 담기
```

### 연쇄적인정리
- 자꾸 손이 가게 하려면 더 작게 해서 더 자주 즐기기
- 작은 단계로 두어 다음 수를 내다보기
- 짧게 작성하면서 효능감을 느끼도록 하기

### 일괄처리량
- 배포도 자주, 짧게, 빠르게
- 단, 짧은 배포 주기는 팀원과 갈등이 생길 수 있음 → 합의 필요

### 리듬
- 소프트웨어 설계는 **작고 빈번하게** 할수록 유리
- 길을 닦는 일처럼 쓰임새를 지켜보며 지속해야 하는 일

### 얽힘풀기
- 실타래를 풀려면 먼저 엉켜있다는 사실을 알아차려야 함
- 레거시 정리는 이것이 문제라고 **서로 판단하고 합의**해야 시작 가능
- 기술 부채 청산은 상당히 힘든 일

### 코드정리시점
스스로 판단해야 하는 4가지 선택지:

| 시점 | 설명 |
|------|------|
| 아예 안 함 | 정리할 가치가 없는 경우 |
| 나중에 정리 | 재미로, 여유가 있을 때 |
| 동작 변경 후 정리 | 기능을 먼저 구현하고 정리 |
| 정리 후 동작 변경 | 정리를 먼저 하여 기능 구현을 수월하게 |

## 3부:설계이론

### 설계의정의
- **요소들을 유익하게 관계 맺는 일**
- 관심사의 분리가 설계의 본질

### 경제이론:시간가치와선택가능성
- **오늘의 1달러 > 내일의 1달러** (시간 가치)
- 아직 정해지지 않은 일은 나중에 하도록 두는 것이 더 가치 있음 (옵션의 가치)
- 즉, 불확실한 설계 결정은 미루는 것이 경제적으로 합리적

### 가치>가격>비용
- 내가 만드는 **가치**가 나에게 투자되는 **가격**(급여)보다 커야 함
- **비용** = 가치를 만들어내는 데 필요한 시간
- 사회적으로 인정받으려면 가치 > 가격 관계를 유지해야 함

## 좋은설계를하는방법

### 기술적관점
- 피벗 시점을 잘 정하기
- 리스크를 줄이기 위해 당장 할 수 있는 것부터 하나씩

### 도메인적관점
- 현실을 알고 그것을 코드에 녹여내는 능력 = **소프트 스킬**
