# Prototype 패턴이란?
원본 객체를 복제(clone)해서 새 객체를 만드는 생성 패턴. 객체 생성 비용이 클 때 매번 새로 만들지 않고 한 번 만든 원형을 베껴 파생한다.

## 왜 쓸까?

### 생성 비용이 큰 객체 회피
객체를 만들 때마다 무거운 초기화(내부에 다른 객체들을 함께 생성, 외부 자원 조회 등)가 따른다면, 같은 형태가 반복될 때 비용이 누적된다. 초기 상태가 동일하다는 전제가 있으면 원본을 한 번만 만들고 그 다음부터는 `clone()`으로 베껴 비용을 줄인다.

### 구체 클래스에 의존하지 않는 생성
새 인스턴스를 만들 때 어떤 클래스인지 몰라도 원본의 복제 인터페이스만 알면 된다. 생성 로직과 사용 코드를 분리한다.

## 핵심 개념

### 복제 인터페이스
복제 대상은 자신을 베끼는 메서드를 공개한다. Java에서는 `Cloneable`을 구현하고 `clone()`을 오버라이드하는 식이다. 호출 측은 `new` 대신 `original.clone()`으로 새 객체를 얻는다.

### 얕은 복사 vs 깊은 복사 (핵심 함정)
기본 복제는 보통 **얕은 복사**라 원본과 복제본이 내부 참조 타입 멤버(객체, 배열)를 **공유**한다. 메모리를 아끼는 이점이 있지만, 복제본의 내부 객체를 수정하면 원본도 함께 바뀌어 추적하기 어려운 버그가 된다. 내부까지 독립시키려면 깊은 복사를 따로 구현해야 한다. 원시 타입은 값으로 복사되고 참조 타입은 주소가 복사되는 [[JS-Value-vs-Reference|Call by value/reference]] 원리와 같은 맥락이다.

## JavaScript와의 관계
JS는 이 패턴을 언어 차원으로 내재화했다. 모든 객체가 생성 시 어떤 원본(프로토타입)에 위임 연결되며, 객체 전체가 하나의 복제, 위임 관계망(프로토타입 체인)으로 얽힌다. 단일 객체 쌍의 복제를 넘어선 구조라 더 복잡하지만 뿌리는 같은 발상이다. JS의 구현 메커니즘은 [[Prototype-Mechanism|프로토타입 동작 원리]].

## 실 사용 사례
1. 설정, 템플릿 객체를 원형으로 두고 복제해 변형 (기본값 묶음 복제)
2. JS 객체 복제 — `structuredClone()`(깊은 복사), `{ ...obj }`(얕은 복사)
3. 게임 엔티티 — 동일 초기 스펙의 개체를 원형에서 대량 복제
