# 우아한객체지향

> 조영호님의 "우아한 객체지향" 발표 내용 정리

## 설계란

- 코드를 **어떻게 배치할 것인가**에 대한 의사결정
- 배치 기준은 **변경**에 따라 결정됨
  - 같이 변경되는 코드 → 같이 배치
  - 같이 변경되지 않는 코드 → 분리

## 의존성의종류

### 클래스의존성

| 유형 | 설명 | 예시 |
|------|------|------|
| 연관관계(Association) | A에서 B로 탐색 가능 | A가 B를 필드로 참조 |
| 의존관계(Dependency) | A가 B를 일시적으로 사용 | 파라미터, 리턴타입, 지역변수 |
| 상속관계(Inheritance) | B가 변경되면 A도 변경 | A extends B |
| 실체화관계(Realization) | A가 B 인터페이스를 구현 | A implements B |

### 패키지의존성
- 패키지에 포함된 클래스 사이의 의존성으로 결정됨

## 의존성관리원칙

1. **양방향 의존성을 피하라** — A↔B 양방향이면 하나가 바뀔 때 둘 다 영향
2. **다중성이 적은 방향을 선택하라** — OneToMany보다 ManyToOne을 지향
3. **불필요한 의존성은 제거하라**
4. **패키지 사이의 의존성 사이클을 제거하라** — 양방향과 같은 문제

## 연관관계vs의존관계:언제사용하는가

| 구분 | 연관관계 | 의존관계 |
|------|---------|---------|
| 성격 | 영구적 탐색 구조 | 일시적 의존성 |
| 구현 | 필드로 객체 참조 | 파라미터, 리턴타입, 지역변수 |
| 사용시점 | 두 객체의 협력이 영구적일 때 | 협력이 일시적일 때 |

**연관관계든 의존관계든 유형보다 방향성이 더 중요.** 방향은 런타임에 객체가 어떻게 협력하는지에 따라 결정.

## 객체참조의문제점

객체 참조(연관관계)는 모든 것을 연결시켜 **가장 높은 결합도**를 만든다.

### 세가지문제
1. **결합도 상승** — 참조로 연결된 모든 객체를 따라가서 기능 구현 가능 → 경계가 사라짐
2. **성능 문제** — 어디까지 조회할 것인가? (Lazy Loading, N+1)
3. **트랜잭션 경계 문제** — 연관된 도메인을 하나의 트랜잭션으로 묶으면, 변경 빈도가 다른 도메인 간 Lock 경합으로 성능 저하

예: 배달 완료 시 가게·주문·배달이 하나의 트랜잭션 → 각각 변경 빈도가 달라 Lock 경합 발생

## 객체그룹의경계설정

### 묶는기준
- 함께 생성되고 함께 삭제되는 객체
- 도메인 제약사항을 공유하는 객체
- 가능하면 분리 (비즈니스 룰에 따라 결정)

### 접근방식
| 경계 | 접근방법 | 트랜잭션 |
|------|---------|---------|
| 경계 안 | 객체 참조로 접근 | 하나의 트랜잭션 |
| 경계 밖 | ID로 접근 + Repository를 통한 탐색 | 별도 트랜잭션 |

- 그룹 간 객체 참조 → ID 기반 연관관계로 전환
- 조회의 경계도 이 경계를 따름
- **경계를 잘 나누면 영속성 저장소 변경까지 가능**해짐

## 패키지의존성사이클해결

### 중간객체를통한추상화
- 추상화 = 반드시 인터페이스/추상클래스는 아님
- **본질적으로 추상화는 잘 변하지 않는 것**
- 두 객체 사이에 잘 변하지 않는 중간 객체를 넣어 의존성 사이클을 끊음

## 객체참조를끊은후의설계전략

### 방법1:검증객체로로직이동
- 객체를 직접 참조하던 로직을 별도의 검증 객체(Validator 등)로 이동
- 절차지향적이지만 **응집도가 올라감**

### 방법2:도메인이벤트
- 주문 완료 시 → 주문에서 도메인 이벤트를 발행
- 가게, 배달 등 다른 도메인은 이벤트 리스너로 처리
- 도메인 간 직접 의존을 제거하고 이벤트로 느슨하게 연결

### 근본적해결:도메인을잘게쪼개기
- 도메인별로 패키지를 잘게 쪼개는 것이 의존성 사이클을 줄이는 핵심
- 의존성을 잘 관리하면 시스템 단위 분리가 가능해짐
- 시스템 간에는 메시지 브로커를 통한 이벤트 기반 통신으로 진화

## 핵심정리

```
의존성 관리 → 객체 경계 설정 → 패키지 분리 → 시스템 분리 → 메시지 브로커
```

의존성을 기준으로 시스템을 점진적으로 진화시키는 것이 핵심.

1. 코드를 먼저 작성한 후 의존성 관점에서 검토
2. 객체 참조가 꼭 필요한지 의문을 가질 것
3. 경계 안은 참조, 경계 밖은 ID + Repository
4. 이벤트로 도메인 간 결합을 느슨하게
5. 패키지를 잘게 쪼개서 사이클 제거
