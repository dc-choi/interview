# 애플리케이션아키텍처와객체지향

## 아키텍처란

- 개발자들이 설계에 대해 공유하는 이해를 반영하는 **주관적인 개념**
- 아키텍처에 들어가는 요소:
  - 개발자들이 알아야 하는 중요한 것
  - 변경하기 어려운 것
  - 일찍 올바르게 결정하고 싶은 것

## 레이어드아키텍처

관심사의 분리(Separation of Concerns)를 위한 가장 보편적인 방법

| 레이어 | 역할 |
|--------|------|
| Controller | 사용자 입력을 받고 제어 |
| Domain | 순수한 비즈니스 로직 |
| Repository | 데이터 조작 |

**가장 중요한 것은 도메인 레이어.** 도메인 레이어의 설계 방식이 전체 아키텍처 구성을 주도한다.

## 도메인레이어설계:두가지패턴

### 트랜잭션스크립트패턴(절차지향)
- **중앙 집중식 제어 스타일**
- 하나의 서비스에서 모든 로직을 절차적으로 처리
- 트랜잭션 관리 + 애플리케이션 로직 + 도메인 로직을 한 곳에서 관리
- 별도의 서비스 레이어가 필수는 아님
- 데이터 객체와 테이블을 1:1로 매칭 (아토믹 도메인 모델 + 테이블 데이터 게이트웨이)

**문제점:**
- 코드의 어디를 변경해야 하는지 찾기 어려움
- 도메인 개념이 절차 안에 암묵적으로 숨겨짐 (What이 아닌 How로 봐야 유추 가능)
- 요구사항 변경 시 기존 코드 수정이 두렵고 어려움

### 도메인모델패턴(객체지향)
- **분산식 제어 스타일**
- 각 객체가 책임에 맞게 로직을 위임받아 협력
- 객체지향 설계 = 주어진 책임을 수행하는 객체들의 협력

**책임 할당 과정 (영화 예매 시스템 예시):**

1. **예매 생성** → 예매에 필요한 데이터를 가장 많이 아는 객체 = **상영** (Creator 패턴, 구현이 어려우면 Factory 패턴 사용)
2. **가격 계산** → 가격 데이터를 가장 많이 아는 객체 = **영화**
3. **할인율 계산** → 할인율 적용 책임을 가진 별도 객체 추가
4. **할인 여부 판단** → 할인 적용 여부를 판단하는 책임을 가진 별도 객체 추가

## 서비스레이어의역할

도메인 모델을 사용하면, 도메인 로직의 전후 작업(DB, 네트워크 등)을 처리하는 레이어가 필요함.

이것을 도메인에 직접 넣으면:
- 외부 의존성이 생김
- 테스트가 어려워짐
- 결합도가 높아짐

따라서 **서비스 레이어**에서 애플리케이션 로직을 수행:
- 애플리케이션 로직의 경계를 정의
- 도메인 로직의 재사용성을 촉진
- 트랜잭션 관리 (커밋/롤백)
- 도메인 모델을 캡슐화

## 데이터소스레이어와ORM

- 도메인 모델 사용 시 객체 지향 패러다임과 관계형 패러다임이 충돌 (임피던스 불일치)
  - 객체: **행위** 관점
  - 관계형 DB: **데이터** 관점
- **데이터 매퍼**(ORM)를 중간에 두어 객체 모델과 DB 스키마 간 독립성 유지
- Java에서는 JPA를 사용하지만, JPA 제약사항으로 완전한 객체 지향 설계는 어려울 수 있음

## 변경에강한설계

> "변하지 않는 것은 모든 것이 변한다는 사실 뿐이다" — 헤라클레이토스

### 트랜잭션스크립트의한계
- 예: 할인 중복 허용 요구사항 추가 시 기존 코드 수정이 어려움
- 절차 안에 개념이 숨겨져 있어 변경 포인트를 찾기 힘듦

### 도메인모델의해결
- **Composite 패턴**으로 기존 할인 정책을 조합하는 통합 할인 객체를 생성
- 기존 코드를 건들지 않으면서 새로운 확장이 용이 → **OCP(개방-폐쇄 원칙)**
- 추상화에 의존하게 되면서 → **DIP(의존성 역전 원칙)**
- 클래스를 확장하면 도메인 개념을 코드에 명확하게 녹여낼 수 있음

## 도메인모델이적합한경우

- 복잡성을 알고리즘이 아닌 객체 간 관계로 분리할 수 있을 때
- 유효성 검사, 계산, 파생 등이 포함된 복잡하고 끊임없이 변하는 비즈니스 규칙이 있을 때
- 변경의 방향과 속도를 판단할 수 있다면, 객체 지향 패러다임이 변화를 수용하는 설계를 지원

## 핵심정리

| 구분 | 트랜잭션스크립트 | 도메인모델 |
|------|-----------------|-----------|
| 제어방식 | 중앙 집중식 | 분산식 (객체 협력) |
| 로직위치 | 서비스에 집중 | 각 도메인 객체에 분산 |
| 변경용이성 | 낮음 (코드 수정 어려움) | 높음 (OCP, DIP 활용) |
| 데이터접근 | 테이블 데이터 게이트웨이 | 데이터 매퍼 (ORM) |
| 서비스레이어 | 필수 아님 | 필수 (애플리케이션 로직 분리) |
| 적합한경우 | 단순한 비즈니스 로직 | 복잡하고 변화가 잦은 비즈니스 규칙 |
