# 공통 코드 관리

상태값·카테고리·유형 같은 **상수성 데이터**를 여러 시스템(DB·서버·프론트엔드)에 걸쳐 어디에 두고 어떻게 동기화할지 결정하는 문제. 단순해 보이지만 팀 규모·배포 주기가 늘어나면 **가장 자주 버그를 만드는** 영역 중 하나.

## 왜 어려운가

- **저장 위치 선택지**: DB 테이블·서버 Enum·설정 파일·외부 설정 서비스·프론트 하드코딩
- **동기화 타이밍**: 한 쪽이 바뀌면 다른 쪽도 즉시 반영돼야 하지만, 일반적으로 **사람이 수동 동기화**
- **타입 안전성**: 프론트엔드에 문자열로 뿌리면 오타·누락·대소문자 버그가 런타임까지 숨어듦
- **문서화 비용**: Notion·Wiki에 공통 코드를 또 적으면 관리 포인트가 3곳(코드·DB·문서)으로 분산

## 접근 방식 3가지 진화

### 1단계 — DB 중심 (Code Table + Generator)

가장 흔한 초기 방식. MySQL 등에 `code_table` 을 두고, 코드 생성기로 서버 Enum을 만든다.

- **저장**: DB 테이블이 Source of Truth
- **서버**: `codeGenerator` 유틸이 DB를 읽어 Kotlin/Java Enum 파일 생성
- **프론트**: 서버 API 호출로 코드 목록 수신
- **장점**: 비개발자가 DB에서 코드 조회·수정 가능
- **단점**: Enum과 DB가 다른 환경에서 **불일치** 빈발. 개발자가 `git pull` 후 DB 업데이트 전이면 런타임 오류. 동기화 타이밍을 사람이 관리

### 2단계 — DB 제거, Reflection으로 API 응답

DB의 중복을 제거하고 **Kotlin Enum 자체**를 Source of Truth로 승격.

- **저장**: 서버 코드의 Enum이 SoT
- **서버**: Reflection으로 Enum 클래스를 읽어 API 응답 JSON을 동적 생성
- **프론트**: 서버 API를 통해 조회 (이전과 동일)
- **장점**: DB 테이블·Notion 문서·동기화 절차 모두 제거. 배포 한 번에 변경 전파
- **단점**: 프론트엔드는 여전히 **런타임 문자열** 사용 → 오타·대소문자 버그 잔존

### 3단계 — 코드 생성으로 타입 안전성 확장

프론트엔드까지 **타입으로 보호**. Gradle 플러그인이 Kotlin 소스를 파싱해 TypeScript 코드를 자동 생성.

- **서버**: Kotlin PSI (Program Structure Interface)로 Enum 파일을 파싱
- **빌드 시점**: `enum.ts` 파일을 자동 생성, 프론트 리포지토리로 커밋/배포
- **프론트**: 생성된 TS `const`·`type`을 import → 컴파일 시점에 오타·누락 차단
- **장점**: 백·프론트 모두 **컴파일 타임에 일치 검증**. 서버 한 곳만 고치면 프론트까지 전파
- **단점**: 코드 생성 파이프라인 구축·유지 비용. 생성물 커밋 정책·CI 흐름 설계 필요

## 저장 위치 선택 기준

| 상황 | 권장 |
|---|---|
| 비개발자가 직접 수정·조회 필요 | **DB + 관리자 UI** (단, 동기화 문제 감수) |
| 코드 빈도의 변화가 거의 없음 | **서버 Enum + 코드 생성** |
| 런타임 동적 변경이 필수 (A/B 테스트·feature flag 성격) | **외부 설정 서비스** (Unleash·LaunchDarkly) |
| 사용자에게 노출되는 정적 레이블 | **i18n 리소스 파일** |

## 타입 안전성 확보 방법

### 서버 언어
- **Enum·Sealed Class**: 컴파일 타임 보장
- **문자열 상수 `const`**: 타입 도움 없음, 오타 취약 — 지양
- **도메인 VO**: 검증·변환 로직을 타입에 싣기

### 프론트엔드
- **서버에서 자동 생성된 TS 타입**: 위 3단계 방식
- **union literal type**: `type OrderStatus = 'PAID' | 'CANCELED' | 'REFUNDED'`
- **zod·io-ts 런타임 검증**: API 응답을 실제로 검증해 잘못된 값이 들어오면 즉시 실패

## 진화 판단 — 정답은 없다

세 단계는 **점진적 개선**이지 유일 해법이 아니다. 조직·도메인에 맞춰 선택:

- 작은 팀·초기 스타트업: **1단계**가 빠르고 실용적
- 개발자 중심 팀·안정된 코드: **2단계**로 관리 포인트 축소
- 프론트-백 분리·다수 앱: **3단계**로 타입 안전성까지 확장

**언제 다음 단계로 가야 할까**: 같은 동기화 버그가 **반복**되면 이미 전환 시점. 매번 수동으로 해결하는 비용이 전환 비용을 넘어선다.

## 흔한 실수

- **Notion·Wiki와 코드를 모두 Source of Truth로** — 어느 쪽이 맞는지 몰라짐
- **프론트에 `'PAID'` 문자열 하드코딩** — 서버가 `PAID_IN_CASH`로 바뀌어도 안 터지고 조용히 버그
- **DB 테이블과 Enum 동기화 없이 배포** — 개발·스테이지·프로덕션 환경마다 불일치
- **코드 생성물을 gitignore** — 소스 없이 빌드 불가, 리뷰어가 변경을 못 봄
- **코드 변경을 개발자만 할 수 있게** → 운영·CS 팀의 작은 변경도 개발자 큐에 적체

## 면접 체크포인트

- 공통 코드가 **동기화 버그의 원천**이 되는 구조적 이유
- DB 중심 vs 서버 코드 중심 저장의 **장단점**
- **코드 생성(PSI·Reflection)** 으로 타입 안전성을 확장하는 아이디어
- 런타임 변경 요구(A/B 테스트)가 있을 때는 **외부 설정 서비스**가 더 적합한 이유
- "정답이 없다"를 전제로 **팀 상황에 맞춰 선택**하는 판단 근거
