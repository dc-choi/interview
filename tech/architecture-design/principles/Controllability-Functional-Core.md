# 제어 가능성과 Functional Core

좋은 설계의 한 축은 **제어 가능한 영역을 넓히고 제어 불가능한 영역을 격리**하는 것이다. 시간, 외부 API, 브라우저 기능, 외부에서 주어진 값처럼 내가 통제할 수 없는 것에 핵심 설계를 의존시키면 외부 변화가 내부를 무너뜨리고 테스트도 어려워진다. 제어 불가능한 것은 경계 바깥으로 밀어내고 순수한 비즈니스 로직을 중심에 둔다 — Functional Core, Imperative Shell.

## 원칙은 의사결정 비용을 줄인다

개발은 매 순간 선택의 연속이다. 클래스로 만들지 함수로 끝낼지, 외부 라이브러리를 쓸지 직접 구현할지, 테스트를 어떻게 작성할지 결정해야 한다.

- 원칙이 없으면 매번 흔들린다. 원칙이 있으면 "이 상황에서는 이렇게 한다"는 기준이 생겨 중요하지 않은 선택은 빠르게 결정하고, 정말 설계가 필요한 부분에 사고력을 집중한다.
- 결과적으로 일정 안에서 일정 수준 이상의 품질을 유지한다. 제품 개발자는 연구자가 아니라 사용자에게 가치를 전달하는 사람이므로, 완벽한 코드를 끝없이 추구하기보다 **제때 전달되는 80점**을 안정적으로 만든다.

## 제어 가능 vs 제어 불가능

| 제어 가능 (코어로 모은다) | 제어 불가능 (경계 밖으로 민다) |
|---|---|
| 금액 계산, 조건 필터링, 할인 판단 같은 순수 로직 | 현재 시간, 외부 API 호출, DB 접근, 모달/UI |
| 입력과 출력이 명확한 함수 | 외부에서 주어진 값(식별자, 정책) |

## 제어할 수 없는 값에 핵심 설계를 의존하지 않는다

외부에서 주어진 값은 편해 보여도 위험하다. 주민등록번호는 한때 유일한 식별자처럼 보였지만 정책이 바뀌면 더 이상 마음대로 수집하거나 저장할 수 없다. 이런 값을 데이터베이스의 핵심 키로 삼으면 **외부 변화가 내부 설계를 무너뜨린다**.

핵심 식별자, 핵심 규칙, 핵심 흐름은 가능하면 내가 통제할 수 있는 방식으로 설계한다. 예를 들어 외부 값 대신 내부에서 발급하는 surrogate key를 기본 키로 두고, 외부 값은 속성으로만 보관한다.

## 시간과 외부 효과는 주입한다

현재 시간에 따라 결과가 달라지는 함수는 테스트하기 어렵다. "오늘이 일요일이면 할인한다"는 로직이 함수 내부에서 직접 현재 시간을 가져오면, 테스트 실행 날짜에 따라 결과가 달라져 검증이 불가능해진다.

- 해결책은 현재 시간을 함수 내부에서 직접 가져오지 않고 **외부에서 주입**하는 것이다. 테스트에서는 원하는 날짜를 넣고, 실제 실행에서는 기본값으로 현재 시간을 쓴다.
- 같은 원리를 외부 API, 랜덤 값, ID 생성 같은 비결정적 요소에도 적용한다. 비결정성을 주입 경계로 밀어내면 코어는 결정적(deterministic)이 된다.

## Functional Core, Imperative Shell

- **순수 코어(Functional Core)**: 입력과 출력이 명확하고 부수 효과가 없는 순수 함수. 대부분의 비즈니스 로직이 여기 속한다.
- **명령형 셸(Imperative Shell)**: 외부 API, DB, UI 호출 같은 부수 효과. 코어를 감싸는 얇은 바깥층에 둔다.
- 효과: 대부분의 로직은 빠른 **단위 테스트로 광범위하게 검증**하고, 외부 의존이 있는 부분만 제한적으로 모킹한다. 테스트 불가능한 영역(Black Box)이 의존 구간으로 전이되는 것을 차단하는 [[Mock-Testing-Strategy|Mock 격리]]와 같은 원리다.

## 같은 원리의 응용: 통제 가능한 것에 집중

제어 가능성은 코드뿐 아니라 일하는 방식에도 적용된다. 통제할 수 없는 것(회사의 투자 상황, 개발자 연봉 시장, 경쟁사의 채용 조건)에 매달리기보다 **통제 가능한 것**에 에너지를 쓴다.

예를 들어 좋은 사람을 당장 데려오지 못하더라도, 외부 시니어를 초청해 사내 강연을 열고 정적 분석, 테스트 코드, 린트, 코드 리뷰 자동화를 구축하는 것은 통제 가능하다. 지금 있는 사람이 더 좋은 개발자가 되게 만드는 일이 그렇다.

## 면접 체크포인트

- 핵심 식별자나 규칙이 외부에 의해 바뀔 수 있는 값에 묶여 있지 않은가
- 시간, 랜덤, 외부 호출이 로직 내부에 박혀 테스트를 막고 있지 않은가
- 순수 로직과 부수 효과가 분리되어 코어를 단위 테스트로 덮을 수 있는가
- 비결정적 요소를 주입으로 격리해 테스트를 결정적으로 만들 수 있는가
