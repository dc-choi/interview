# 레거시 시스템 현대화 전략

10년 이상 된 시스템을 현대화하는 **4가지 전략**과 선택 기준. Strangler Fig는 기본이고, **인프라 단절·Recode·사양 경량화** 3가지가 더 과격한 대안. 어떤 전략을 택할지는 도메인 특성·팀 규모·경영진 지원·리스크 감수 능력의 함수.

## 4가지 전략 개관

| 전략 | 핵심 | 성공 확률 | 기간 | 위험도 | 주요 대상 |
|---|---|---|---|---|---|
| **Strangler Fig** | 기능 단위로 점진적 교체, 구/신 공존 | 높음 | 매우 길음 | 낮음 | 핵심 도메인 (결제·주문) |
| **인프라 단절** | 레거시와 신규를 API로만 연결, 지역/세그먼트별 롤아웃 | 중~높음 | 중~긴 | 중간 | 독립 도메인 전체 교체 |
| **Recode** | 사양은 유지, 언어/스택 완전 교체 | 중간 | 중간 | 중간 | 클라이언트 앱, 단일 모듈 |
| **사양 경량화** | 저가치 고부담 기능 과감히 제거 | 낮음 (정치적) | 매우 길음 | 중간 | 부수 기능, 무거운 통합 |

네 전략은 **배타적이지 않다** — 하나의 현대화 프로젝트에서 도메인별로 다른 전략을 조합하는 경우가 많다.

## 1. Strangler Fig Pattern

Martin Fowler가 이름 붙인 고전 패턴. 레거시 외부에 **파사드(façade)** 를 두고, 기능 단위로 신규 구현으로 트래픽을 점진 이동. 옛 나무를 옥죄어 죽이는 교살 무화과(strangler fig)에서 유래.

- **장점**: 리스크 분산, 언제든 롤백, 일상 배포 지속 가능
- **단점**: 매우 긴 기간 (수년), 두 시스템 운영 비용 중복, 중간 상태가 영구화될 위험
- **언제**: 핵심 도메인 + 매출 중단 허용치가 낮을 때

## 2. 인프라 단절 (Infrastructure Disconnection)

Strangler Fig의 한 단계 더 과격한 버전. 레거시와 신규를 **서로 다른 인프라에 둔 뒤 API(WebAPI/gRPC/JSON) 로만 연결**. 더 나아가 **지역/사용자 세그먼트/점포 단위**로 트래픽을 끊어내며 이동.

### 핵심 메커니즘
- 신규 시스템은 **새 데이터 스키마**로 재설계 (레거시 DB 공유 금지)
- 인증은 레거시가 담당, 신규 시스템은 토큰 교환 방식으로 수용
- 롤아웃은 **지역 단위 4–6 단계** — 소규모 파일럿 → 도시 일부 → 전국
- 신뢰 확보 후 **레거시 전체 삭제** (Strangler Fig는 삭제 전환 어려움, 이건 명시적 끝점)

### 장점
- 데이터 구조의 깊은 재설계 가능 (레거시 스키마에 묶이지 않음)
- 롤아웃 완료 후 **운영 비용이 즉각 절감** (두 시스템 동시 운영이 끝남)
- 새 시스템이 자기 완결성을 가져 이후 리팩토링 용이

### 단점
- 초기 리스크 큼 — 지역별 장애가 사용자 신뢰에 직접 영향
- **팀 간 중복 제거** 선행 필요 (신규 팀이 자율적으로 일할 수 있어야)
- 예측 불가능한 일정 지연 — 지역별 특수 요구가 드러날 때마다 재설계
- "사람도 레거시가 된다" — 기존 패턴을 고수하는 인력 교육/재배치 문제

### 언제
- 독립적 도메인 전체를 교체할 때
- 다지역·다점포 같은 **세그먼트 단위 롤아웃이 가능**한 도메인
- 장기 투자를 감당할 수 있는 조직

## 3. Recode (재작성)

**사양은 동일하게 유지하되 언어·기술 스택을 통째로 교체**. "같은 기능, 새 코드". 기존 코드를 읽거나 참고하지 않고 사양서만 보고 다시 짠다.

### 필요한 상황
- 사양서가 유실되어 코드만이 유일한 스펙 (코드 고고학 비용 > 재작성 비용)
- 팀 교체로 기존 코드를 아는 사람이 0명
- 기술 스택이 지원 중단 (Xamarin → Flutter, IE11 전용 SPA → 현대 웹)
- 플랫폼 파편화 심각 (iOS native + Android native + React Native + Xamarin → Flutter 단일화)

### 프로세스
1. **기획 동결 선언** — 신규 기능 개발 중단 (3–6개월)
2. 사양서 역추출 — 현재 시스템 행동을 문서화
3. 새 스택으로 구현
4. **병렬 QA** — 기존 시스템과 동일 입력에 대해 출력이 같은지 비교
5. Feature flag + 트래픽 점진 전환

### 장점
- 기술 스택을 현대화하면서 내부 품질까지 상승
- 유실된 사양서를 이 기회에 복구
- 팀이 새 언어·프레임워크 학습 → 장기 생산성 상승

### 단점
- **기획 동결 기간**을 조직이 버텨내야 함 (영업·마케팅 저항)
- 저실력 팀이 하면 레거시보다 더 나쁜 결과
- **버그가 사양화된 경우** 혼란 — 기존 동작을 "사양"으로 볼지 "버그로 수정"할지 결정 필요
- "사용자가 변경을 인지 못하면 성공" 기준이 역설적으로 어려움

### 언제
- 클라이언트 앱 (상대적으로 독립, 사양이 visual 중심)
- 작은~중간 규모 모듈
- 기술 스택 지원 중단이 임박해 어쩔 수 없을 때

## 4. 사양 경량화 (Specification Simplification)

**가장 급진적이지만 동시에 가장 가성비 높은 전략** — 낮은 비즈니스 가치 + 높은 기술 부담의 기능을 **제거**. "고치지 않고 지운다".

### 사례 유형
- 팩스·SMS 같은 레거시 통신 채널 (이중화 물리 서버, SaaS 전환 불가)
- 쓰이지 않는 옛 결제 수단
- 구 API 버전
- 복잡한 분기 로직의 일회성 마케팅 기능

### 추진 프로세스
1. **대상 기능 식별** — 사용량 로그·수익 기여도·운영 비용
2. **손실 임계 설정** — "매출의 X% 이하 손실까지 감내"를 경영진과 합의
3. **영향 사용자 소통** — 사전 공지·대체 경로 제공
4. **단계적 제거** — 신규 사용 차단 → 기존 사용자 전환 → 완전 제거
5. **인프라 정리** — 제거와 함께 관련 서버·데이터·모니터링 모두 제거

### 장점
- 즉각적 운영 비용 절감 (서버·라이선스·모니터링·팀 공수)
- 시스템 복잡도 근본 감소 → 이후 리팩토링 쉬워짐
- 지속가능성 확보

### 단점
- **정치적 어려움**이 가장 큼 — 영업·운영·CS팀이 "한 명이라도 쓰고 있다"며 반대
- **경영진 설득**이 핵심 — 손실 기준·대안·커뮤니케이션 플랜 필요
- 대체 경로 설계에 예상보다 긴 시간
- 법적·규제 영향 사전 검토 필수

## 전략 선택 가이드

| 상황 | 추천 전략 |
|---|---|
| 결제·주문처럼 매출 중단 허용 불가 | **Strangler Fig** |
| 배송·물류처럼 지역 단위 트래픽 있음 | **인프라 단절** + 지역 롤아웃 |
| 클라이언트 앱 + 기술 스택 파편화 | **Recode** (단일 스택으로) |
| 팩스·SMS 등 저가치 고부담 기능 | **사양 경량화** |
| 대규모 모놀리스 전체 | 여러 전략 조합 (핵심=Strangler, 주변=Recode/경량화) |

## 점진적 롤아웃 공통 전술

현대화 중 사용자 영향을 최소화하는 4가지 축:

1. **지역 단위** — 파일럿 지역 → 소도시 → 대도시 → 전국 (Demaecan 방식)
2. **사용자 세그먼트** — 내부 직원 → 베타 사용자 → 전체
3. **기능 단위** — 읽기 먼저 → 쓰기 나중 (CQRS 스타일)
4. **시간 단위** — 비피크 시간부터 → 피크 시간
각 단계 사이에 **관측 기간**을 두어 지표 이상을 조기 감지.

## 팀·조직 고려사항

| 요소 | 중요성 |
|---|---|
| **팀 업무 중복 제거** | 신규 팀이 레거시 유지보수를 겸하면 집중력 분산 |
| **장기 팀 구성 안정성** | 2–3년 프로젝트에서 핵심 인력 교체는 치명적 |
| **기획 동결 합의** | Recode는 3–6개월 기획 중단을 버틸 조직 체력 필요 |
| **경영진 C-level 후원** | 사양 경량화는 반드시 C-level 결정 |
| **"사람도 레거시가 된다" 인식** | 기존 패턴 고수 인력 교육·재배치 계획 |
| **QA 중복 예산** | 병렬 운영 기간 QA 공수는 2배 |

## 흔한 실패 패턴

- **데드라인 없는 Strangler Fig** — 중간 상태가 5년째 지속, 두 시스템 영구 병존
- **사양서 없는 Recode** — 기존 코드를 "사양"으로 삼아 버그까지 포팅
- **경영진 지원 없는 사양 경량화** — 영업팀 반대에 무기한 지연
- **팀 교체와 동시에 현대화** — 도메인 지식 손실로 요구사항 재발견 반복
- **빅뱅 커버 완료 후 QA** — 지역·사용자별 실제 트래픽 고려 안 해 롤아웃 초반 장애 폭증
- **신규 시스템이 레거시 DB 공유** — "단절"이 깨져 결국 Strangler도 인프라 단절도 아님

## 면접 체크포인트

- **Strangler Fig의 정의**와 고전적 한계 (중간 상태 영구화)
- **인프라 단절 vs Strangler Fig** 차이 (명시적 끝점·새 스키마·지역 롤아웃)
- **Recode**가 필요한 조건 (사양서 유실·스택 파편화)
- **사양 경량화**의 거버넌스 (손실률 기준·경영진 설득)
- 대규모 모놀리스 현대화에서 **전략 조합** 사용하는 이유
- **점진적 롤아웃 4축** (지역·세그먼트·기능·시간)
- "사람도 레거시가 될 수 있다"의 조직적 의미
