# 소프트웨어아키텍처의중요성

## 아키텍처란무엇인가

### IEEE정의
> 구성요소들 간의 관계, 환경, 설계와 발전을 관리하는 원칙으로 이루어진 시스템의 근본적인 구조

**이 정의의 한계:**
- 너무 거시적이고 포괄적인 컴포넌트 개념
- 어떤 컴포넌트를 선택하고, 어떤 연관관계를 가져갈지에 대한 구체적 고민이 빠져 있음
- 실제로는 무수히 많은 컴포넌트와 연관관계가 존재

### MartinFowler의정의
- 아키텍처 = **전문 개발자들이 공유하는 시스템 디자인에 대한 이해**
- 다이어그램, 문서는 이 지식을 공유하는 불완전한 표현일 뿐
- 프로젝트가 잘 성장하고 있을 때 = 팀원들 간에 프로젝트에 대한 이해도가 잘 공유될 때

### RalphJohnson의정의
> 아키텍처란 "뭔가 중요한 것(the important stuff, whatever that is)"

소프트웨어 시스템을 설계할 때 무엇이 중요한지, 그 핵심 가치에 대한 판단이 곧 아키텍처

## 아키텍처의두가지핵심

### 1.지식의공유
- 프로젝트 내부 구조를 잘 아는 개발자들의 이해가 아키텍처에 큰 영향을 미침
- 이 이해는 사회적 요소 — 팀 내 커뮤니케이션과 문화에 좌우됨

### 2.결정의비가역성
- 아키텍처 디자인은 우선적으로 진행되어야 함
- 올바른 결정은 빨리 내려지길 원하지만, 정보 없이는 빨리 할 수 없음
- **한번 결정하면 변경하기 매우 어려움** — 이것이 아키텍처가 중요한 이유

## 왜아키텍처가중요한가:경제적관점

### 장인정신vs경제
- 소프트웨어 장인 정신으로 임하지만, 실무에서는 경제적 관점이 항상 이김
- 따라서 아키텍처의 중요성을 경제적 논리로 설명할 수 있어야 함

### 외부품질vs내부품질
| 구분 | 설명 |
|------|------|
| 외부품질 | 사용자가 직접 경험하는 품질 (기능, UX, 성능) |
| 내부품질 | 코드 구조, 아키텍처, 테스트 가능성 등 외부에서 보이지 않는 품질 |

- 아키텍처는 **내부 품질**에 해당
- 외부 이해관계자는 내부 품질을 볼 수 없고, 평가할 수도 없음
- 하지만 내부 품질은 장기적으로 가장 중요한 요소

### DesignStaminaHypothesis(설계체력가설)

Martin Fowler가 제시한 개념으로, 아키텍처 투자와 개발 속도의 관계를 설명:

```
기능 추가 속도
  ^
  |        ╱ 좋은 아키텍처
  |      ╱
  |    ╱───────── 아키텍처 무시
  |  ╱  ╲
  | ╱    ╲
  |╱      ╲
  +──────────────→ 시간
```

- **아키텍처를 무시하면**: 초기에는 빠르지만 시간이 지날수록 기능 추가가 급격히 느려짐
- **아키텍처에 투자하면**: 초기에는 느리지만 시간이 지날수록 기존 코드가 플랫폼처럼 동작하여 오히려 더 빨라짐
- 이것이 CI/CD가 보편화되는 이유이기도 함 — 기능을 잘, 자주 추가할 수 있어야 하므로

### 리팩토링의역할
- 아키텍처를 신경쓰지 않으면 시간이 지나며 새로운 기능 추가가 매우 어려워짐
- 지속적인 리팩토링으로 내부 품질을 유지해야 함
- 기능별 컴포넌트화를 잘 해놓으면 기존 코드가 플랫폼이 되어 생산성이 올라감

## 면접에서활용하기

- **"좋은 아키텍처란?"** → 팀의 공유된 이해 + 변경하기 어려운 결정을 올바르게 내리는 것
- **"리팩토링을 왜 하나?"** → 내부 품질 유지 → 장기적 기능 추가 속도 확보 (경제적 논리)
- **"기술 부채란?"** → 내부 품질 저하로 인한 개발 속도 감소, Design Stamina Hypothesis로 설명
