# 서비스레이어와테스트경계

> 출처: [[App-Architecture-OOP|애플리케이션 아키텍처와 객체지향]]

## 왜서비스레이어가필요한가

도메인 모델을 사용하면 도메인 로직의 전후 작업(DB, 네트워크 등)을 처리하는 레이어가 필요하다.

도메인에 외부 의존성을 직접 넣으면:
- **테스트가 어려워짐** (DB, 네트워크 연결 필요)
- **결합도가 높아짐** (도메인이 인프라에 의존)
- **재사용성이 떨어짐**

## 서비스레이어의역할

| 역할 | 설명 |
|------|------|
| 애플리케이션 로직 경계 | 도메인 로직의 전후 작업을 정의 |
| 트랜잭션 관리 | 커밋/롤백 제어 |
| 도메인 캡슐화 | 순수 도메인 로직의 재사용성 촉진 |

## 테스트전략

### 도메인모델패턴
| 레이어 | 테스트 방식 | 특징 |
|--------|-----------|------|
| 도메인 | **단위 테스트** | 외부 의존성 없이 순수 로직 검증, Mock 불필요 |
| 서비스 | **통합 테스트** | 트랜잭션, DB 연동 등 전체 흐름 검증 |
| 컨트롤러 | **통합 테스트** | API 요청/응답 검증 |

- 도메인이 순수하므로 단위 테스트가 빠르고 안정적
- 변경해야 할 위치를 쉽게 찾을 수 있음

### 트랜잭션스크립트패턴
| 레이어 | 테스트 방식 | 특징 |
|--------|-----------|------|
| 서비스 | **통합 테스트** | 모든 로직이 서비스에 집중, Mock이 많아짐 |

- 서비스에 트랜잭션 + 애플리케이션 + 도메인 로직이 혼재
- 테스트 시 Mock이 많아지고, 변경 포인트를 찾기 어려움

## 핵심정리

| 비교 | 트랜잭션스크립트 | 도메인모델 |
|------|-----------------|-----------|
| 도메인 단위테스트 | 어려움 (외부 의존성 혼재) | 쉬움 (순수 로직) |
| Mock 사용량 | 많음 | 적음 |
| 변경 위치 파악 | 어려움 | 쉬움 |
| 서비스 레이어 | 필수 아님 | 필수 |
