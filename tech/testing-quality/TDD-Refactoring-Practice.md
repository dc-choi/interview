# TDD 리팩토링 연습법 — 의식적인 연습과 작은 변화

TDD와 리팩토링은 지식이 아니라 **기술**이다. 글쓰기나 운동처럼 상황마다 정답이 달라지고, 반복하며 감각을 키워야 한다. 중요한 것은 중요성을 아는 것이 아니라 **꾸준히 연습하는 방식** — 작은 목표와 제약을 두고 피드백을 받으며 반복하는 것.

## 반복만으로는 늘지 않는다 — 의식적인 연습

이를 매일 닦아도 양치 실력이 늘지 않는 이유는 무의식적으로 하기 때문이다. 기존 방식대로만 코드를 쓰면 경험은 쌓여도 실력은 도약하지 않는다. 성장은 편한 영역을 살짝 벗어나는 연습에서 나온다.

의식적인 연습(deliberate practice)의 조건:

1. **현재 실력보다 약간 높은 난이도** — 너무 쉬우면 변화가 없고, 너무 어려우면 포기한다
2. **구체적인 목표** — 클린 코드를 짜자(X) → 메서드 길이를 10줄 이하로(O)
3. **피드백 장치** — 코드 리뷰, 페어 프로그래밍, 테스트 결과, 정량적 제약
4. **한 번에 한 가지에 집중** — 한 주는 메서드 분리만, 다음 주는 else 제거만

## 좋은 연습은 정량적 제약에서 시작된다

메서드는 한 가지 일만 하라는 말은 초보자에게 추상적이다. 바로 확인 가능한 제약이 스스로 피드백을 만든다 (객체지향 생활체조 계열의 규칙들):

- **메서드 15줄 이하** — 넘으면 분리를 고민하게 됨
- **들여쓰기 1단계까지만** — 중첩 조건, 반복문이 메서드 추출을 강제
- **else 금지** — 조기 반환(early return), 조건 분리를 고민하게 됨
- **원시값 포장** — int 대신 의미 있는 타입
- **인스턴스 변수 3개 이하** — 자연스럽게 새로운 객체를 찾게 됨

이런 제약은 처음엔 불편하지만, 기존 방식으로 풀 수 없는 문제를 만나게 해 새 설계 감각을 만든다.

## 연습 경로 — 쉬운 것부터 단계적으로

### 1. 장난감 프로젝트로 시작

회사 프로젝트에는 일정, 레거시, DB, UI, 외부 API 같은 변수가 많아 첫 시도가 실패하기 쉽다. 문자열 계산기, 자동차 경주, 사다리 타기, 체스처럼 **도메인은 단순하지만 설계 연습이 되는 문제**가 적합하다. 거창한 서비스가 아니라 작은 문제에서 반복 훈련이 목적.

### 2. TDD 전에 학습 테스트부터

학습 테스트 = 사용하는 API, 라이브러리의 동작을 확인하려고 작성하는 테스트 (문자열 분리, 날짜 처리, 컬렉션 동작 등). 코드보다 테스트를 먼저 생각하는 습관을 부드럽게 만들어 준다.

### 3. TDD 사이클은 쪼개서

Red → Green → Refactor 세 단계를 처음부터 완벽히 하려 하면 부담스럽다. 처음엔 테스트 작성과 통과만, 그다음 리팩토링에 집중. **많은 사람이 통과에서 멈추고 리팩토링을 생략하는데, 진짜 학습은 리팩토링 단계에서 일어난다.** 테스트는 안전망, 리팩토링이 설계 훈련이다. (사이클 자체는 [[TDD-BDD|TDD, BDD]] 참조)

### 4. 메서드 분리 → 5. 클래스 분리 (원시값 포장)

- **메서드 분리**: 조건문, 반복문, 변환이 한 메서드에 몰린 코드에 들여쓰기, 길이 제약을 걸면 자연스럽게 나뉜다. 역할이 드러나면 읽는 순서도 좋아진다 — 좋은 코드는 모든 세부사항을 한 번에 보여주지 않고, 필요할 때 안쪽으로 들어가게 한다
- **클래스 분리**: 문자열 숫자를 int로만 다루지 않고 `PositiveNumber`처럼 감싼다. 변환, 음수 검증, 값 보관의 책임이 한곳에 모인다. 처음엔 코드가 늘어 보여도 검증이 흩어지지 않고, 의미 있는 타입이 생기고, 변경에 강해진다

### 6. 의존성 있는 코드는 나중에

DB, UI, 외부 API가 얽힌 코드부터 TDD를 시도하면 테스트 작성에 지쳐 포기한다. 순수 로직으로 연습한 뒤, **테스트하기 쉬운 부분과 어려운 부분을 분리**하는 연습으로 넘어간다 (DB 접근은 바깥으로, 계산과 판단은 별도 객체로). 핵심 감각: 테스트하기 어려운 코드를 억지로 테스트하는 것이 아니라 **테스트하기 쉬운 구조로 바꾸는 것** (경계 설계는 [[Service-Layer-Testing|서비스 레이어와 테스트 경계]]).

### 7. 레거시는 마지막에

레거시는 의존성이 많고 설계 의도가 흐려졌고 안전망도 없는 **가장 어려운 대상**. 작은 프로젝트에서 자신감을 만든 뒤, 회사 코드에는 영향 범위가 작은 부분부터. 작은 성공이 쌓여야 업무 코드에 적용할 힘이 생긴다.

## 성능보다 가독성이 먼저인 경우가 많다

역할별로 메서드를 나누면 반복문을 여러 번 돌 수도 있다. 그래도 대부분의 병목은 애플리케이션 코드가 아니라 DB, 네트워크, 외부 API에서 발생한다. 먼저 읽기 쉽고 변경하기 쉬운 코드를 만들고, **측정된 병목에 대해서만 최적화** ([[Cognitive-Biases-Programming|Premature Optimization]]).

## 팀에 전파하기 — 강요보다 신뢰

- 사람은 변화를 부담스러워하고, 경험 많은 개발자는 과거 실패 탓에 새 시도를 경계한다. 처음부터 팀 전체를 바꾸려 하면 반발이 생긴다
- 먼저 자신의 영향 범위 안에서 좋은 코드를 만들고, 관심을 보이는 동료와 작게 공유 — **변화는 설득보다 증거에서 시작된다**
- 리더는 답을 지시하기보다 질문한다("어떻게 하면 좋을까요?"). 팀원 입에서 나온 개선안이 실행 가능성이 높다. 1대1 면담으로 신뢰를 쌓고, 작은 개선 주제 하나에 익숙해질 때까지 집중
- 팀 변화도 TDD 연습과 같은 구조 — 작게 시작해 피드백 받으며 키운다 ([[Growth-Principles|성장의 7가지 원칙]]의 반복, 소셜화)

## 면접 체크포인트

- 의식적인 연습의 4조건과 **정량적 제약이 피드백 장치가 되는** 원리
- 들여쓰기 제한, else 금지 같은 제약이 **설계 감각을 만드는** 메커니즘 (제약 → 기존 방식 불가 → 새 구조 탐색)
- TDD에서 리팩토링 단계가 **진짜 학습 지점**인 이유
- 원시값 포장의 효과 (검증 응집, 의미 있는 타입, 변경 내성)
- 테스트하기 어려운 코드에 대한 태도 — 억지 테스트가 아니라 구조 변경
- 레거시보다 장난감 프로젝트부터 시작해야 하는 이유
- 팀에 TDD 문화를 도입한다면? — 강요 대신 증거, 작은 성공, 질문하는 리더
