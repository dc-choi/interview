# 장애복구와재발방지

> 출처: [[Toss-Runners-High-Seminar|토스 러너스하이 세미나]] 기술 운영 평가

## 장애복구설계

### 스스로에게물어볼질문
1. 재시작이 필요한가, 수동 작업이 요구되는가?
2. 일부 기능만 멈추는가, 서버 전체가 중단되는가?
3. 내가 설계한 요소가 예기치 못한 상황을 만나도 충분히 회복할 수 있는가?

### 복구수준
| 수준 | 설명 | 목표 |
|------|------|------|
| 수동 복구 | 사람이 개입하여 수동으로 복구 | 최소화해야 함 |
| 자동 재시작 | 프로세스/컨테이너 자동 재시작 | 기본 수준 |
| 부분 장애 격리 | 일부 기능만 멈추고 나머지는 정상 운영 | 권장 수준 |
| 자동 복구 | 시스템이 스스로 상태를 감지하고 복구 | 이상적 수준 |

## 재발방지(RootCause분석)

### 스스로에게물어볼질문
1. 현재 대처가 문제를 회피하는 것인가? **Root Cause**는 무엇인가?
2. 동일한 장애가 다시 발생하지 않을 수 있는가?
3. **사람에 의존하는가, 시스템에 의존하는가?**

### 참고사례
- SLASH24 - eBPF를 활용한 Root Cause 분석
- Pay Cluster: 사람 의존 → 시스템 의존으로 전환

### 재발방지프로세스
```
장애 발생 → 임시 대응 → Root Cause 분석 → 재발 방지 대책 수립 → 시스템에 반영 → 검증
```

## 운영자동화와지속적개선

- 성능 최적화/운영 자동화를 위해 지속적으로 노력하는가?
- **사업의 확장은 최전선에서 일어남**
- 지나간 일에 파묻히면 앞으로 나아갈 동력을 잃게 됨
- 과거 장애 대응에만 매몰되지 않고, 미래를 위한 자동화에 투자

## 면접포인트
- "장애 경험?" → 감지 → 대응 → Root Cause → 재발 방지까지 전체 사이클로 답변
- "사람 의존 vs 시스템 의존?" → 수동 대응을 자동화로 전환한 경험
- "운영 자동화 경험?" → 반복 작업을 시스템화하여 본업에 집중할 수 있도록 개선
