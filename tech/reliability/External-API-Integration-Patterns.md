# 외부 API 연동 실전 패턴

여행 플랫폼·이커머스·결제 서비스는 **다수의 외부 API**(공급사·OTA·결제 PG·포인트·할인)에 의존한다. 각 API의 **장애·지연·응답 불일치**가 내 서비스로 전파되지 않게 해야 한다. [[External-Service-Resilience|타임아웃·벌크헤드·서킷]]이 기초라면, 이 문서는 **도메인별 실전 패턴**.

## 두 축의 연동 유형

연동 성격에 따라 접근이 전혀 다르다.

| 유형 | 성격 | 핵심 가치 | 예시 |
|---|---|---|---|
| **조회·카탈로그** | 읽기 위주, 부분 실패 허용 | **내결함성·Fallback** | 상품 상세, 가격 비교, 리뷰 |
| **거래·예약** | 쓰기, 금전·재고 관련 | **일관성·멱등성** | 결제, 예약 확정, 재고 차감 |

## 조회형 연동 패턴

### 1. API별 타임아웃 차등
중요도·평균 응답시간에 따라 **API마다 다른 타임아웃**.
- 핵심 API (상품 메인 정보): 2초
- 보조 API (할인 가격): 500ms
- 부가 API (리뷰 요약): 300ms

→ 느린 보조 API가 전체 페이지 렌더링을 막지 않음.

### 2. Fallback 체인
특정 API 실패 시 대체 값으로 응답 유지.
```
할인가 조회 실패 → 원가 표시
추천 상품 실패 → 인기 상품 리스트
사용자 프로필 조회 실패 → 익명 모드
```

**원칙**: "핵심 경로가 살아있으면 페이지는 뜨게 한다". Graceful Degradation.

### 3. 병렬 호출 + Race
여러 API를 동시에 호출해 **빠른 응답 먼저 사용**. Promise.race 활용. 둘 이상의 소스에서 같은 데이터 얻을 수 있을 때.

### 4. 응답 캐싱
- 자주 변하지 않는 데이터는 **CDN·Redis**에 캐시
- TTL 기반 + 이벤트 기반 무효화 병용
- **Stale-While-Revalidate**: 만료된 캐시로 즉시 응답 + 백그라운드 재조회

### 5. 회로 차단 (Circuit Breaker)
지속 실패 시 일정 시간 **호출 자체를 중단**하고 캐시·기본값으로 응답. [[External-Service-Resilience#Circuit Breaker]] 참고.

## 거래형 연동 패턴

### 1. 상태 머신으로 복잡도 단순화
예약은 `요청 → 결제 대기 → 결제 완료 → OTA 확정 → 취소 가능 → 취소 완료` 등 단계가 많음. **상태 전이를 명시적으로 모델링**하면:
- 각 상태에서 가능한 전이가 명확
- 비정상 전이를 코드·DB에서 차단
- 복구·조회·재처리 로직이 상태 기준으로 통일

### 2. 멱등성 키
클라이언트가 생성한 **idempotency key**를 모든 쓰기 요청에 필수로:
- 네트워크 오류 시 재시도 안전
- 중복 결제·이중 예약 방지
- 서버는 key 기준 캐시로 이전 응답 그대로 반환

상세는 [[Idempotency]] 참고.

### 3. 취소 불가능한 작업은 마지막에
연동 순서를 **"롤백 가능 → 롤백 불가"** 로 배치.

예시 (예약 플로우):
1. 내부 DB에 `PENDING` 생성 (롤백 가능)
2. 결제 PG 호출 (승인 취소 가능)
3. 포인트 차감 (원복 가능)
4. **OTA 실제 예약 확정** (롤백 불가 — 마지막)
5. 내부 `CONFIRMED` 상태 전이

앞단계 중 실패하면 이전까지 롤백. OTA 확정은 모든 선행 조건이 OK일 때만.

### 4. 보상 트랜잭션 (Compensation)
분산 환경에서 분산 트랜잭션(2PC)은 현실적으로 불가 → **Saga 패턴**.

각 단계가 실패하면 **이미 성공한 단계를 역순으로 보상**:
- OTA 예약 후 포인트 적립 실패 → OTA 예약 취소 API 호출
- 보상 로직은 **멱등**해야 함 (보상도 재시도될 수 있음)

### 5. 배치 대사 (Reconciliation)
내 시스템 상태와 외부 시스템 상태가 **진짜 일치하는지 주기적 확인**.
- 매 10분: 최근 1시간 예약 목록을 OTA와 cross-check
- 불일치 발견 시 알림 + 자동 보정(가능하면) 또는 수동 처리 큐로
- 실시간 동기화가 실패해도 **최종 일관성**을 배치로 보장

### 6. 재시도 정책
- **멱등 요청**만 자동 재시도
- **지수 백오프 + 지터** (1s, 2s+random, 4s+random, 8s+random...)
- 최대 재시도 횟수 명시 (무한 재시도 금지)
- 재시도 실패 시 **DLQ(Dead Letter Queue)** 로 보내 사람이 개입

## 도메인별 조합

### 여행 플랫폼 (OTA 연동)
- 상품 상세: 내결함성 중심 (타임아웃·Fallback·캐싱)
- 예약 확정: 일관성 중심 (상태 머신·멱등·보상·대사)

### 결제 시스템
- 결제 승인: **반드시 멱등 키**
- 승인 → 매입 분리 (승인 취소 가능 기간 활용)
- 실패 시 PG사별 에러 코드를 **공통 도메인 에러**로 변환

### 이커머스 재고
- 조회: 캐시 + 실시간 DB 병용
- 차감: **분산 락** + 낙관적 락 + 보상 ([[Distributed-Lock]])
- 동시성: 재고가 잔여 1개일 때가 가장 위험 → 락 필수

## 관찰 가능성 (Observability)

외부 API 연동은 **블랙박스**라 모니터링이 생명.

- **응답시간 분포**: API별 p50·p95·p99 대시보드
- **에러율**: 4xx vs 5xx 구분, 상대 비율 추이
- **재시도 통계**: 재시도 빈도가 올라가면 외부 서비스 상태 악화 신호
- **Correlation ID**: 내 요청 ↔ 외부 요청을 연결해 분산 추적

## 흔한 실수

- **모든 API에 동일 타임아웃** → 보조 API 지연이 전체 페이지 블록
- **Fallback 없음** → 보조 API 장애로 핵심 페이지 안 뜸
- **멱등 키 없이 결제 재시도** → 이중 결제
- **보상 로직 없음** → 부분 성공 상태 영구 잔존
- **대사 배치 없음** → 조용히 데이터 불일치 누적
- **외부 API에 동기 응답 직결** → 외부 지연이 내 스레드 풀 고갈

## 면접 체크포인트

- 조회형과 거래형 연동의 핵심 가치 차이 (내결함성 vs 일관성)
- 상태 머신이 예약·주문 도메인에서 중요한 이유
- "롤백 불가 작업은 마지막에" 원칙의 의미
- Saga 패턴과 보상 트랜잭션
- 배치 대사(Reconciliation)의 역할
