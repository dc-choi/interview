# 동시성과프로세스

## 운영체제 개요

시스템 리소스를 제어하고 프로그램에 대한 일반적인 서비스를 지원하는 시스템 소프트웨어. 컴퓨터는 운영체제 없이도 동작하지만 더 유연하고 많은 기능을 위해 운영체제의 도움이 필요.

### 운영체제가 관리하는 것
- 프로세스 관리 / 메모리 관리 / 하드웨어 관리 / 파일 시스템 관리

### 운영체제 역사

| 시기 | 변화 |
|------|------|
| 1940 | 에니악(1943). 배선+스위치로 프로그래밍. IO 중 계산 불가 |
| 1950 | IC 개발. 펀치카드 사용. 오퍼레이터 오버헤드 → 싱글스트림 배치시스템 |
| 1960 | 시분할 시스템, 멀티 프로그래밍, 파일시스템, UNIX 등장. 메모리 침범 이슈 → 베이스 레지스터 도입. OS가 학문으로 자리잡음 |
| 1970 | 하드웨어 저렴화, 개인용 PC 보급, 매킨토시 GUI |

### 운영체제 구조
- **커널**: 프로세스, 메모리, 저장장치 관리의 핵심. 직접 접근 불가
- **GUI/CLI**: 사용자가 커널에 접근하는 인터페이스
- **시스템 콜**: 애플리케이션이 커널에 접근하는 인터페이스 (직접 접근 차단)
- **드라이버**: 하드웨어가 커널에 접근하는 인터페이스

### 부팅 과정
1. ROM에 저장된 BIOS가 전기 신호를 받고 실행
2. 주요 하드웨어 이상 유무 확인 (이상 있으면 실행 중지)
3. 하드디스크의 MBR(마스터 부트 레코드)에 저장된 부트로더를 메모리로 로드
4. 운영체제를 메모리로 불러오면 부팅 완료

### 인터럽트
- 폴링 방식의 문제를 해결. CPU가 IO 관리자에게 명령 후 다른 작업 수행
- IO 완료 시 CPU에게 신호 → CPU가 ISR(인터럽트 서비스 루틴)을 실행하여 작업 완료
- **비동기적** 동작으로 성능 이점
- 하드웨어 인터럽트: 입출력
- 소프트웨어 인터럽트: 유효하지 않은 메모리 접근, 0으로 나누기 등

### 컴퓨터 하드웨어 (폰 노이만 구조)
- CPU와 메모리를 버스로 연결. 버스는 데이터를 전달하는 통로
- **메인보드**: 다른 하드웨어를 연결하는 장치
- **CPU**: 산술논리연산장치(ALU), 제어장치, 레지스터
- **메모리**: RAM, ROM

## 동시성vs병렬성

| 구분 | 정의 | 예시 |
|------|------|------|
| 동시성(Concurrency) | 여러 가지 일이 동시에 진행되는 것 | 1개 CPU가 시분할로 여러 작업 처리 |
| 병렬성(Parallelism) | 같은 일을 여러 주체가 함께 진행하는 것 | 멀티코어에서 동시에 연산 수행 |

## 원자성,동기화,교착상태

### 원자성(Atomicity)
- 동시에 일이 일어나도 자원을 선점하고 있다면 그것을 보장하는 것
- 화장실에 진입하면 Lock, 나오면 Unlock → Lock 구간이 원자성 보장 구간

### 동기화(Synchronization)
- 누군가 자원을 선점하는 동안 원자성을 보장할 수 있도록 하는 메커니즘
- Lock/Unlock으로 임계 영역(Critical Section) 보호

### 공유자원과 임계구역
- **공유자원**: 프로세스 간 통신 시 공동으로 이용하는 변수나 파일
- 접근 순서에 따라 결과가 달라질 수 있고 컨텍스트 스위칭으로 실행 순서 예측이 어려움 → **동기화 문제**
- **임계 구역**: 여러 프로세스가 동시에 사용하면 안 되는 영역
- **경쟁 조건(Race Condition)**: 공유 자원을 서로 사용하기 위해 경쟁하는 것

### 상호배제(Mutual Exclusion)의 요구사항
1. 임계 영역에는 동시에 하나의 프로세스만 접근
2. 여러 요청에도 하나의 프로세스 접근만 허용
3. 임계 영역에 들어간 프로세스는 최대한 빠르게 나와야 함

### 세마포어
- 임계 영역에 들어간 프로세스가 세마포어를 가지면 다른 프로세스는 접근 불가
- **wait()**: 세마포어를 획득 / **signal()**: 세마포어를 반환
- 정수형 변수이며 여러 개 가질 수 있음 (공유자원 수에 비례)
- 카운팅 세마포어: N개 스레드 접근 가능 (DB 커넥션 풀 등)
- 이진 세마포어: 뮤텍스처럼 동작하나 소유권 없음
- 단점: wait/signal 호출 순서를 잘못하면 문제 발생

### 뮤텍스 (Mutex)
- 하나의 스레드만 공유 자원에 접근할 수 있도록 제한하는 락
- **소유권**이 존재: 뮤텍스를 얻은 스레드만 해제 가능
- 커널 기반(OS 제공)과 사용자 공간 기반(경량 스핀락)으로 구현 가능

### 모니터
- 세마포어의 단점을 해결. 프로그래밍 언어 차원에서 지원
- Java의 **synchronized** 키워드가 대표적
- synchronized가 붙은 함수가 실행 중이면 다른 프로세스에서 synchronized 함수를 모두 실행 불가
- 모니터 구현이 완벽하면 프로그래머는 편하게 코딩 가능

### 뮤텍스 vs 세마포어 vs 모니터 비교

| 항목 | 뮤텍스 | 세마포어 | 모니터 |
|------|--------|---------|--------|
| 동시 접근 수 | 1개 | N개 | 조건에 따라 다름 |
| 소유권 | 있음 (획득한 스레드만 해제) | 없음 (누구나 해제 가능) | 언어 차원에서 관리 |
| 구현 방식 | 커널/유저 공간 | 커널 | 프로그래밍 언어 |

**언제 뭘 써야 할까?**
1. 단일 자원 보호 → **뮤텍스**
2. 제한된 자원 수 관리 (커넥션 풀 등) → **세마포어**
3. 복잡한 조건 동기화 → **모니터**

### 교착상태(Deadlock)
- 여러 프로세스가 서로 다른 프로세스의 작업 완료를 기다리며 아무도 작업하지 못하는 상태
- 공유자원이 없다면 교착상태는 발생하지 않음

**발생 조건 4가지 (모두 동시에 충족 시 발생):**

| 조건 | 설명 |
|------|------|
| 상호배제(Mutual Exclusion) | 자원은 한 번에 하나의 프로세스만 사용 |
| 점유대기(Hold and Wait) | 자원을 보유한 채 다른 자원을 요청하며 대기 |
| 비선점(No Preemption) | 다른 프로세스의 자원을 강제로 빼앗을 수 없음 |
| 순환대기(Circular Wait) | 프로세스들이 순환 형태로 서로의 자원을 대기 |

### 교착상태 해결

**교착상태 회피 — 은행원 알고리즘**
- 전체 자원의 수와 할당된 자원의 수를 기준으로 안정/불안정 상태를 판단
- 프로세스에게 자원 할당 전에 안전 상태인지 확인
- 불안정 상태라고 반드시 교착상태는 아니지만 확률이 올라감
- 비용이 비싸고 비효율적

**가벼운 교착상태 검출**
- 타이머 사용. 일정 시간 동안 작업이 진행되지 않으면 교착상태로 간주
- 체크포인트를 만들어 작업을 저장하고 타임아웃 시 마지막 체크포인트로 롤백
- 억울하게 종료되는 프로세스가 발생할 수 있음

**무거운 교착상태 검출**
- 자원 할당 그래프를 유지. 순환 구조가 생기면 교착상태 검출
- 교착상태를 일으킨 프로세스를 강제종료 후 체크포인트로 롤백
- 지속적인 그래프 유지/검사 오버헤드 있으나 억울한 종료는 발생하지 않음

## 프로세스간통신(IPC)

프로세스는 독립적으로 실행되지만 다른 프로세스와 데이터를 주고받으며 통신하는 경우도 있다.

| 방식 | 설명 |
|------|------|
| 파일 | 통신하려는 프로세스들이 하나의 파일에 읽고 쓰기 |
| 파이프 | 운영체제가 생성한 파이프를 통해 데이터를 읽고 씀 (같은 컴퓨터) |
| 쓰레드 통신 | 한 프로세스 안에서 데이터/힙 영역을 이용한 쓰레드간 통신 |
| 소켓 | 네트워크를 이용한 통신 |
| RPC | Remote Procedure Call, 다른 컴퓨터의 함수를 호출 |

## 멀티프로세싱vs멀티스레딩

| 구분 | 멀티프로세싱 | 멀티스레딩 |
|------|------------|-----------|
| 단위 | 프로세스 | 스레드 |
| 메모리 | 각각 독립된 메모리 공간 | 힙을 공유, 스택은 개별 |
| 통신 | IPC(파이프, 소켓 등) 필요 | 공유 메모리로 직접 통신 |
| 안정성 | 하나가 죽어도 다른 프로세스 영향 없음 | 하나가 죽으면 전체 프로세스 영향 |
| 오버헤드 | 컨텍스트 스위칭 비용 큼 | 컨텍스트 스위칭 비용 작음 |
| 동기화 | 상대적으로 단순 | 공유 자원 동기화 필요 (Race Condition) |

## 스레드종류

| 종류 | 설명 |
|------|------|
| 커널스레드 | OS가 직접 관리. 스케줄링 가능하지만 생성/전환 비용이 큼 |
| 유저스레드 | 사용자 공간에서 라이브러리로 관리. 빠르지만 OS가 인식 못함 |
| 하이브리드 | 유저 스레드를 커널 스레드에 매핑 (M:N 모델). Go의 goroutine이 대표적 |

## 프로그램→프로세스→스레드

| 개념 | 설명 |
|------|------|
| 프로그램 | 설치된 상태, 디스크에 저장된 코드 |
| 프로세스 | 프로그램을 실행하면 생성되는 인스턴스 |
| 스레드 | 프로세스 속의 실행 단위, 프로세스 자원을 공유 |

### 메모리영역
| 영역 | 소유자 | 특징 |
|------|--------|------|
| 스택 | 스레드별 개별 소유 | 함수 호출, 지역 변수 |
| 힙 | 프로세스 전체 공유 | 동적 할당, 동시성 문제 발생 가능 |

- 동시성 문제로 인해 기억 공간을 스택/힙으로 구분

### 프로세스메모리구조(상세)
| 영역 | 저장내용 | 방향 |
|------|---------|------|
| 코드(Text) | 실행할 프로그램의 기계어 코드 | 고정 |
| 데이터(Data) | 전역 변수, static 변수 | 고정 |
| 힙(Heap) | 동적 할당 (new, malloc) | 낮은 주소 → 높은 주소 |
| 스택(Stack) | 함수 호출, 지역 변수, 매개변수 | 높은 주소 → 낮은 주소 |

### Recursion이stack/heap에미치는영향
- 재귀 호출마다 스택 프레임이 쌓임 → 깊은 재귀는 **Stack Overflow** 발생
- 꼬리 재귀 최적화(TCO)가 없는 언어에서는 반복문으로 변환하는 것이 안전
- 재귀 내에서 동적 할당을 하면 힙도 함께 증가

### 단편화(Fragmentation)
| 종류 | 설명 |
|------|------|
| 외부단편화 | 할당/해제를 반복하면 메모리 사이에 사용할 수 없는 작은 빈 공간이 생김 |
| 내부단편화 | 할당된 블록이 요청보다 커서 블록 내부에 사용되지 않는 공간이 생김 |

- 외부 단편화 해결: 압축(Compaction), 페이징
- 내부 단편화 해결: 적절한 블록 크기 선택, 버디 시스템

### 메모리지역성(Locality)
| 종류 | 설명 |
|------|------|
| 시간적지역성(Temporal) | 최근 접근한 데이터에 다시 접근할 가능성이 높음 |
| 공간적지역성(Spatial) | 접근한 데이터 근처의 데이터에 접근할 가능성이 높음 |

- 캐시가 효율적으로 동작하는 근거
- 배열 순차 접근이 연결 리스트보다 빠른 이유 (공간적 지역성)

## 커널모드vs유저모드

### 컴퓨터를국가에비유
| 국가 | 컴퓨터 |
|------|--------|
| 영토/영공/영해 | 하드웨어 |
| 정부 조직 | OS (시스템 소프트웨어) |
| 국민 | 프로그램(프로세스) |
| 국민의 공간 | 프로세스 메모리 (다른 프로세스 침범 불가) |
| 경찰 | 디버거 (다른 프로세스 접근 가능) |

### 커널(Kernel)
- OS의 핵심, 컴퓨터의 법을 구현
- **역할**: 입출력 제어, 자원(CPU+RAM) 관리, 접근 제어
- OS는 하드웨어에 의존적 (64비트 OS → 64비트 CPU 필요)

### 두모드의관계

```
[유저 모드]  프로세스 → 디바이스 파일(인터페이스) → read/write/execute
                              ↓
[커널 모드]  엔진/코어 → 드라이버(펌웨어) → 하드웨어(디바이스)
```

- **디바이스 파일**: 커널 구성 요소에 대한 추상적 인터페이스를 유저 모드에 제공
- 프로세스는 하드웨어를 직접 통제하지 않음 → OS가 전부 통제
- 프로세스가 디바이스 파일에 접근하려면 OS로부터 권한 부여가 필요

### 디스크접근예시
```
[하드웨어] 디스크
    ↑
[커널모드] 드라이버(펌웨어) → 파일시스템
    ↑
[유저모드] 파일시스템 추상화 인터페이스 → 프로세스
```

## 가상메모리

- 메모리를 추상적/논리적으로 관리하는 체계
- 물리적 RAM을 넘어서 디스크까지 메모리처럼 사용

### 페이징
| 용어 | 설명 |
|------|------|
| 페이지 아웃(Swap Out) | RAM → 디스크로 이동 |
| 페이지 인(Swap In) | 디스크 → RAM으로 복귀 |
| 논페이징 | 페이징을 하지 않는 영역 (커널 등) |
| 페이징 파일 | RAM처럼 사용하는 디스크 영역 |

- 애플리케이션은 실제 데이터가 RAM에 있는지 디스크에 있는지 모름

### 가상메모리를사용하는이유
1. 각 프로세스 공간을 완벽하게 분리하고 통제
2. 프로세스 오류가 OS까지 영향을 주지 못하도록 차단
3. 메모리가 부족해도 여러 프로그램이 동작할 수 있음 (자원의 효율적 사용)

## 동기화문제가발생하는경우

| 문제 | 설명 |
|------|------|
| Race Condition | 여러 요청이 동시에 같은 데이터를 읽고 수정할 때 잘못된 순서로 실행되어 잘못된 데이터가 저장 |
| Dirty Read | 트랜잭션이 완료되지 않았는데 다른 트랜잭션이 해당 데이터를 읽는 문제 |
| Lost Update | 두 트랜잭션이 동시에 같은 데이터를 업데이트하면 한쪽이 덮어씌워지는 문제 |

## Node.js에서의동기화

Node.js는 싱글 스레드 + 이벤트 루프 기반이라 전통적인 뮤텍스/세마포어 이슈가 직접 발생하지 않는 경우가 많다. 하지만 Worker Thread, libuv 스레드 풀, Promise.all 병렬 DB 처리 시 발생 가능.

### DB 드라이버별 동작 차이

| 드라이버 | 방식 |
|----------|------|
| pg, mysql2 | 네이티브 C++ 바인딩 → libuv 스레드 풀 사용 |
| mongodb, ioredis, Prisma | JavaScript TCP 소켓 / Rust 엔진 → 이벤트 루프에서 직접 처리 |

### Prisma의 동작 방식
Node.js → Prisma Client (JS) → Query Engine (Rust) → DB
- Rust 엔진이 TCP 소켓으로 DB와 직접 통신
- libuv 스레드 풀을 사용하지 않음
- Promise.all 사용 시에도 Rust 엔진 내부에서 비동기 네트워크 처리

### 해결 방법

| 방법 | 적용 상황 |
|------|----------|
| 트랜잭션 + FOR UPDATE | DB 레벨에서 row 단위 Lock |
| SharedArrayBuffer + Atomics | Worker Thread 사용 시 멀티스레드 안전 공유 자원 접근 |
| async-mutex | Promise.all 병렬 처리 시 Mutex 개념 적용 (단일 프로세스 한정) |
| Redis 분산락 (SETNX, Redlock) | 멀티 프로세스/멀티 인스턴스 환경에서 분산락 구현 |
