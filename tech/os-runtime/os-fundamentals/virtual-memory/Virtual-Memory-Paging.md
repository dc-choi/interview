# 가상 메모리와 페이징

## 가상 메모리

프로세스 입장에서는 운영체제 영역 위치나 물리 메모리 크기를 알 필요가 없음. 프로그래머는 0번지에서 시작한다고 가정하고 개발.

- 프로세스는 메모리 관리자를 통해 메모리에 접근
- 메모리 관리자가 가상 주소를 물리 주소로 변환 = **동적 주소 전환**
- 가상 메모리 크기 = 물리 메모리 + swap 영역
- 이론적으로 무한대이나 실제로는 물리 메모리 크기와 CPU 비트수로 결정

## 세그멘테이션

가변 분할 방식을 사용. 프로그램을 함수/모듈 등 세그먼트로 구성.

### 동작 방식
1. CPU가 논리주소를 전달
2. 메모리 관리자가 세그먼트 번호를 알아냄
3. 세그먼트 테이블 베이스 레지스터로 물리 메모리의 세그멘테이션 테이블을 찾음
4. 세그먼트 번호를 인덱스로 **base address**와 **bound address**를 확인
5. 논리주소가 bound 주소보다 크면 → 메모리 침범 에러
6. 논리주소가 bound 주소보다 작으면 → 논리주소 + base 주소 = 물리 주소

컨텍스트 스위칭 시마다 세그먼트 테이블 베이스 레지스터를 해당 프로세스의 것으로 교체해야 하므로 컨텍스트 스위칭은 무거운 작업.

### 장단점
- 장점: 코드/데이터/힙/스택을 모듈로 나눠 관리 가능, 공유와 메모리 접근 보호가 편리
- 단점: **외부 단편화** 발생

## 페이징

고정 분할 방식을 사용. 외부 단편화 해결을 위해 고안.

### 핵심 개념
- **페이지**: 논리주소공간을 일정 크기로 균일하게 나눈 것
- **프레임**: 물리주소공간을 페이지 크기와 동일하게 나눈 것
- **페이지 테이블**: 페이지 번호 → 프레임 번호 매핑

### 동작 방식
1. CPU가 논리주소를 전달
2. 메모리 관리자가 페이지 번호와 오프셋을 알아냄
3. 페이지 테이블 베이스 레지스터로 페이지 테이블을 찾음
4. 페이지 번호를 인덱스로 프레임 번호를 확인
5. 프레임 번호 + 오프셋 = 물리 주소
6. Invalid 표시 → 스왑 영역에 있음

### 세그멘테이션과의 차이

| 비교 | 세그멘테이션 | 페이징 |
|------|-------------|--------|
| 크기 | 가변 (bound 주소 필요) | 고정 (bound 불필요) |
| 단편화 | 외부 단편화 | 내부 단편화 (심각하지 않음) |
| 논리 구분 | 코드/데이터/힙/스택 분리 가능 | 논리적 영역 분리 불가 |
| 공유/권한 | 특정 영역만 공유/권한 부여 쉬움 | 어려움 |

### 페이지 테이블 크기 문제
- 프로세스마다 페이지 테이블을 가지므로 프로세스가 많을수록 메모리 사용 증가
- 페이지 테이블도 물리 메모리의 운영체제 영역에 저장되므로 크기가 크면 사용자 영역 부족

## 페이지드 세그멘테이션

세그멘테이션 + 페이징의 장점을 결합.

### 동작 방식
1. 가상주소로 세그먼트 번호를 알아냄
2. 메모리 접근 권한(읽기/쓰기/실행) 검사 → 위반 시 프로세스 종료
3. 세그멘테이션 테이블에서 페이지 번호와 페이지 개수를 가져옴
4. 페이지 테이블에서 프레임 번호를 확인하여 물리 주소 계산

### 메모리 접근 권한

| 영역 | 읽기 | 쓰기 | 실행 |
|------|------|------|------|
| 코드 | O | X | O |
| 데이터 | O | O/X | X |
| 스택/힙 | O | O | X |

단점: 물리 메모리에 두 번 접근 (세그멘테이션 테이블 + 페이지 테이블). 현대 OS는 페이징과 페이지드 세그멘테이션을 적절히 혼합하여 사용.

## 디맨드 페이징

90:10 법칙 (도널드 커누스): 프로그램 실행 시 90%의 시간이 10%의 코드에서 소비됨.

### 지역성 이론
- **공간의 지역성**: 현재 위치에서 가까운 데이터에 접근할 확률이 높음
- **시간의 지역성**: 최근 접근한 데이터에 다시 접근할 확률이 높음
- goto문을 지양하는 이유: 코드 구조 파악 어려움 + 지역성 위반으로 성능 저하

조만간 쓸 데이터만 메모리에 올리고 당분간 불필요한 데이터는 swap 영역으로 이동시키는 정책.

### 페이지 테이블 엔트리 (PTE) 구성

| 비트 | 역할 |
|------|------|
| 접근 비트 | 메모리에 올라온 후 읽기/실행이 있었으면 1 |
| 변경 비트 | 메모리에 올라온 후 쓰기 작업이 있었으면 1 |
| 유효 비트 | 1이면 swap 영역, 0이면 물리 메모리에 있음 |
| 권한 비트 | 읽기/쓰기/실행 접근 권한 검사 |

### Page Fault
- 프로세스가 접근한 페이지가 물리 메모리에 없으면 발생하는 인터럽트
- 보조저장장치 swap 영역에서 데이터를 메모리로 불러옴
- 해당 프로세스는 대기 상태가 됨

### 세 가지 참조 시나리오
1. **swap 불필요**: 유효 비트 확인 → 프레임 번호로 물리 메모리 직접 접근
2. **swap 영역 참조**: 물리 메모리에 빈 공간 있음 → swap에서 임의 공간에 데이터 할당, 유효 비트 업데이트
3. **물리 메모리 꽉 참**: 물리 메모리에서 불필요한 영역을 swap으로 옮기고 그 공간에 swap 데이터 할당

## 페이지 교체 정책

### Random
- 무작위 선택. 지역성을 고려하지 않아 자주 사용되는 페이지가 선택될 수 있음. 거의 사용되지 않음

### FIFO
- 메모리에 들어온 지 가장 오래된 페이지를 교체
- 구현이 간단하고 성능도 괜찮아 변형해서 많이 사용
- 단점: 자주 쓰이는 페이지가 먼저 들어왔다는 이유로 교체될 수 있음
- **빌레이디의 역설**: 프레임을 늘렸는데 Page Fault가 오히려 증가 (FIFO에서만 발생)

### Optimum
- 앞으로 가장 오래 쓰이지 않을 페이지를 선택. 이론적 방법으로 구현 불가능
- 다른 알고리즘과 성능 비교 시 참조용

### LRU (Least Recently Used)
- 최근에 가장 사용이 적은 페이지를 선택. Optimum에 근접한 성능
- 단점: 시간 기록 비트가 많이 필요, 오버플로우 발생 가능
- 실제 구현 시 접근 비트를 사용하여 LRU에 근접하게 구현

### Clock (클락 알고리즘)
- LRU를 접근 비트 하나만으로 근사 구현
- 일정 시간 간격마다 모든 페이지의 접근 비트를 0으로 초기화
- 페이지를 원형으로 연결하고 **클락 핸드**(포인터)가 시계 방향으로 이동
- Page Fault 발생 시:
  - 접근 비트 1 → 0으로 바꾸고 다음 페이지로 이동
  - 접근 비트 0 → 해당 페이지를 swap으로 보냄

### Enhanced Clock (향상된 클락)
- 접근 비트 + 변경 비트를 함께 확인
- 교체 우선순위: (접근=0, 변경=0) > (접근=0, 변경=1) > (접근=1, 변경=0) > (접근=1, 변경=1)

### 2차 기회 페이지 교체 (FIFO 개선)
- FIFO 방식에서 자주 사용되는 페이지에 한 번 더 기회를 줌
- Page Fault 없이 페이지 접근에 성공하면 해당 페이지를 큐의 맨 뒤로 이동하여 수명 연장
- 성능: LRU > 2차 기회 > FIFO

하드웨어적으로 접근 비트를 지원하지 않는 시스템에서는 FIFO를 사용할 수밖에 없음.

## 스레싱과 워킹셋

### 스레싱 (Thrashing)
- 멀티프로그래밍의 정도가 높아질수록 물리 메모리에 올릴 프로세스가 많아짐
- Page Fault가 빈번하게 발생하고 CPU가 작업하는 시간보다 swap 시간이 길어짐
- CPU 사용률이 낮아지면 스케줄러가 더 많은 프로세스를 올리고 이것이 반복되면 CPU 사용률이 더 떨어지는 악순환
- 근본 원인: 물리 메모리 크기 부족

### 소프트웨어적 해결
- 프로세스 실행 시 일정량의 페이지를 할당
- Page Fault가 많이 발생하면 → 페이지를 더 할당
- Page Fault가 너무 적으면 → 페이지를 과하게 할당한 것으로 판단하여 회수

### 워킹셋 (Working Set)
- 적절한 페이지 수가 결정된 후 어떤 페이지들을 유지할 것인가는 지역성 이론을 따름
- 현재 메모리에 올라온 페이지는 다시 사용할 확률이 높으므로 하나의 세트로 묶어 메모리에 올림
- 컨텍스트 스위칭 시 워킹셋 단위로 관리
