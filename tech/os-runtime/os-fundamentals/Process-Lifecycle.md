# Process Lifecycle

## PCB (Process Control Block)

운영체제는 프로세스가 만들어지면 해당 프로세스의 정보를 가지고 있는 PCB를 만들어 관리한다. PCB는 연결리스트로 저장되고 프로세스가 종료되면 해당 PCB를 제거한다.

### PCB 구조

| 항목 | 설명 |
|------|------|
| 포인터 | 부모/자식 프로세스 포인터, 할당 자원 포인터, 상태 전환시 저장 포인터 |
| 프로세스 상태 | 생성, 준비, 실행, 대기, 완료 |
| 프로세스 ID (PID) | 프로세스 식별 숫자 |
| 프로그램 카운터 | 다음 실행될 명령어의 주소. 다른 프로세스 실행 후 돌아왔을 때 이어서 실행하기 위해 필수 |
| 레지스터 정보 | 실행 시 사용했던 레지스터 값들. 복귀 시 값 복구 용도 |
| 메모리 정보 | 프로세스의 메모리 위치, 경계 레지스터 값 (메모리 침범 방지) |
| CPU 스케줄링 정보 | 우선순위, 최종 실행시간, CPU 점유시간 |

## 프로세스 상태

| 상태 | 설명 |
|------|------|
| 생성(New) | PCB 생성, 메모리 적재 요청 상태. 승인되면 준비 상태로 전환 |
| 준비(Ready) | CPU 사용을 위해 기다리는 상태. CPU 스케줄러에 의해 CPU 할당됨 |
| 실행(Running) | CPU를 할당받아 실행 중. 실행 프로세스 수 = CPU 수. 부여된 시간만큼만 사용 후 준비로 복귀 |
| 대기(Waiting) | IO 요청 시 완료까지 기다리는 상태. IO 완료 후 준비 상태로 복귀 |
| 완료(Terminated) | 프로세스 종료. 메모리에서 데이터 제거, PCB 제거 |

## 프로세스 생성과 종료

### 생성 과정
1. 운영체제가 프로그램의 코드/데이터 영역을 메모리에 로드
2. 빈 스택과 빈 힙 공간을 확보
3. PCB를 만들고 값을 초기화

이 과정은 부팅 시 0번 프로세스 생성 때 딱 한 번만 실행됨.

### fork()와 exec()
- **fork()**: 0번 프로세스를 복사하여 자식 프로세스를 생성. 새로 생성하는 것보다 빠름
- 자식 프로세스는 부모의 코드, 데이터, 스택, PCB 내용을 전부 복사
- 그대로 복사하면 부모와 똑같이 동작하므로 **exec()** 함수로 코드/데이터 영역을 원하는 값으로 덮어씀
- exec() 이후 자식 프로세스는 부모와 완전히 다르게 동작

### wait()와 exit()
- **wait()**: 부모 프로세스가 자식의 정상 종료를 기다림
- **exit()**: 자식 프로세스가 부모에게 정상 종료를 알림

### 좀비 프로세스
- 부모가 자식보다 먼저 종료되거나, 자식이 비정상 종료하여 exit() 신호를 보내지 못한 경우
- exit status를 읽지 못해 메모리에 계속 남아있는 상태
- 컴퓨터를 오래 켜두면 느려지는 원인 중 하나

## 쓰레드

프로세스를 생성하면 PCB, 코드/데이터/스택/힙 영역을 모두 만들어야 하고 프로세스 간 통신(IPC)은 비용이 크다. 이를 해결하기 위해 등장.

### 쓰레드의 특징
- 프로세스 안에 존재하며 한 프로세스당 최소 1개 이상
- 같은 프로세스 내 쓰레드들은 PCB, 코드, 데이터, 힙 영역을 **공유**
- **스택만 쓰레드마다 개별** 소유
- 쓰레드 ID와 TCB(Thread Control Block)로 관리
- 운영체제가 작업을 처리하는 단위는 프로세스 내의 쓰레드

### 프로세스 vs 쓰레드

| 관점 | 프로세스 | 쓰레드 |
|------|---------|--------|
| 안정성 | 하나가 죽어도 다른 프로세스 영향 없음 | 프로세스에 문제 생기면 모든 쓰레드에 영향 |
| 자원/속도 | 고유 자원, IPC 오버헤드 큼 | 스택 외 영역 공유, 오버헤드 작음 |
| 통신 | IPC 필요 (비용 큼) | 공유 영역으로 쉽게 통신 가능하나 동기화 문제 발생 가능 |

## 컴파일과 프로세스

### 컴파일 과정 (C언어)
1. **전처리기**: #include, #define 등 매크로 처리 → `.i` 파일
2. **컴파일러**: C → 어셈블리어 변환 (기계어와 1:1 매칭) → `.s` 파일
3. **어셈블러**: 어셈블리어 → 기계어(오브젝트 파일, 코드/데이터 영역 분리) → `.o` 파일
4. **링커**: 여러 오브젝트 파일과 라이브러리를 연결, 실행 주소 매핑 → 실행파일(`.exe`)

### 실행파일 → 프로세스
1. 실행파일의 코드/데이터 영역을 프로세스의 코드/데이터 영역에 적재
2. 빈 힙과 스택 할당
3. PCB 생성, 프로그램 카운터를 코드 영역 첫 번째 주소로 설정
4. CPU 스케줄링에 따라 실행

### 컴파일 vs 인터프리터

| 방식 | 특징 | 언어 |
|------|------|------|
| 컴파일 | 미리 기계어로 변환, 문법 검사, 실행 속도 빠름 | C, C++, C# |
| 인터프리터 | 실행 시 한 줄씩 해석, 오류 가능성, 상대적으로 느림 | JS, Python, Ruby |
