# Process Lifecycle

## PCB (Process Control Block)

운영체제는 프로세스와 실행 가능한 스레드의 상태를 커널 자료구조로 관리한다. 교재에서는 이를 PCB와 TCB로 구분하지만 실제 자료구조와 큐 구성은 운영체제 구현에 따라 다르다. Linux는 스케줄링 가능한 task를 `task_struct`로 표현한다.

### PCB 구조

| 항목 | 설명 |
|------|------|
| 포인터 | 부모/자식 프로세스 포인터, 할당 자원 포인터, 상태 전환시 저장 포인터 |
| 프로세스 상태 | 생성, 준비, 실행, 대기, 완료 |
| 프로세스 ID (PID) | 프로세스 식별 숫자 |
| 프로그램 카운터 | 다음 실행될 명령어의 주소. 다른 프로세스 실행 후 돌아왔을 때 이어서 실행하기 위해 필수 |
| 레지스터 정보 | 실행 시 사용했던 레지스터 값들. 복귀 시 값 복구 용도 |
| 메모리 정보 | 프로세스의 메모리 위치, 경계 레지스터 값 (메모리 침범 방지) |
| CPU 스케줄링 정보 | 우선순위, 최종 실행시간, CPU 점유시간 |

## 프로세스 상태

| 상태 | 설명 |
|------|------|
| 생성(New) | task와 실행 문맥을 만드는 중인 개념적 상태. 실행 가능해지면 준비 상태로 전환 |
| 준비(Ready) | CPU 사용을 위해 기다리는 상태. CPU 스케줄러에 의해 CPU 할당됨 |
| 실행(Running) | 하나 이상의 스레드가 논리 CPU에서 실행 중. 동시에 실행 가능한 스레드 수의 상한은 사용 가능한 논리 CPU와 스케줄링 제약에 좌우되지만 runnable task가 적으면 CPU가 유휴일 수 있고, 한 멀티스레드 프로세스가 여러 CPU에서 동시에 실행될 수도 있음 |
| 대기(Waiting) | I/O, lock, timer, event 등 조건을 기다려 실행할 수 없는 상태. 조건이 충족되면 준비 상태로 복귀 |
| 완료(Terminated) | 실행이 끝난 상태. 부모가 종료 상태를 회수하기 전에는 최소한의 종료 정보가 남을 수 있음 |

## 프로세스 생성과 종료

### 프로세스 생성과 프로그램 이미지 실행은 별개

- Unix 계열의 `fork()`/`clone()` 경로는 새 task를 만들고 부모의 실행 문맥과 주소 공간을 복제하거나 지정 방식으로 공유한다. 자식은 새 executable을 적재하지 않고도 `fork()` 다음 지점에서 계속 실행할 수 있다
- `execve()`는 새 프로세스를 만들지 않고 호출한 PID의 기존 process image를 executable, loader가 요구하는 새 매핑과 초기 stack, entry point로 교체한다. 실제 물리 페이지는 보통 접근 시 demand fault로 적재된다
- `posix_spawn()` 같은 API는 생성과 프로그램 실행을 하나의 상위 작업처럼 제공할 수 있지만, 모든 운영체제와 생성 경로가 동일한 세 단계로 동작한다고 일반화하지 않는다

부팅 초기 task는 커널이 특별한 경로로 만들지만, 일반 프로세스도 시스템 호출과 런타임을 통해 반복해서 생성된다.

### fork()와 exec()
- **fork()**: 호출한 프로세스가 자식 프로세스를 생성한다. 부모에는 자식 PID를, 자식에는 0을 반환한다.
- 자식은 별도 PID와 커널 task를 가지며 부모의 논리적 주소 공간을 복제한다. 현대 Unix 계열은 일반적으로 Copy-on-Write로 물리 페이지를 공유하다 쓰기 시 복사한다.
- `execve()` 계열은 현재 PID 안에서 기존 주소 공간의 매핑, stack, heap을 포함한 process image를 새 프로그램 이미지로 교체하고 새 entry point에서 시작한다
- PID와 명세에 따라 보존되는 속성, `FD_CLOEXEC`이 없는 file descriptor 등은 남을 수 있다. 성공한 `execve()`는 기존 프로그램으로 반환하지 않는다

### wait()와 exit()
- **wait()/waitpid()**: 부모가 종료한 자식의 상태를 회수한다. 정상 종료뿐 아니라 시그널 종료 상태도 확인할 수 있다.
- **exit()/_exit()**: 프로세스를 종료하고 종료 상태를 남긴다. 커널은 부모에게 `SIGCHLD`를 알릴 수 있다.

### 좀비 프로세스
- 자식이 종료했지만 부모가 `wait()` 계열로 종료 상태를 아직 회수하지 않은 상태다. 정상 종료와 비정상 종료 모두 zombie가 될 수 있다.
- 부모가 먼저 종료하면 살아 있는 자식은 orphan이 되고 init 또는 subreaper에 재부모화된다. 그 자체가 zombie는 아니다.
- zombie는 일반 실행 메모리를 계속 점유하는 프로세스가 아니라 PID와 종료 상태 등 최소 커널 엔트리를 남긴다. 대량 누적되면 PID와 task table 자원을 고갈시킬 수 있다.

## 쓰레드

스레드는 한 주소 공간 안에서 여러 실행 흐름이 자원과 데이터를 공유하도록 한다. 프로세스 간 IPC와 별도 주소 공간 관리에 비해 효율적일 수 있지만 동기화와 장애 격리의 tradeoff가 있다.

### 쓰레드의 특징
- 프로세스 안에 존재하며 한 프로세스당 최소 1개 이상
- 같은 프로세스 내 쓰레드들은 주소 공간, 코드, 데이터, 힙과 여러 프로세스 자원을 공유한다.
- 각 스레드는 스택, 레지스터 상태, 스레드 ID와 스케줄링 상태를 개별 소유한다.
- 쓰레드 ID와 TCB(Thread Control Block)로 관리
- 운영체제가 작업을 처리하는 단위는 프로세스 내의 쓰레드

### 프로세스 vs 쓰레드

| 관점 | 프로세스 | 쓰레드 |
|------|---------|--------|
| 안정성 | 주소 공간이 분리되어 메모리 오류 전파를 줄이지만 shared resource, supervisor와 IPC 관계를 통한 영향은 가능 | 같은 주소 공간의 치명적 오류나 프로세스 종료가 모든 쓰레드에 영향 |
| 자원/속도 | 별도 주소 공간과 IPC 경계로 추가 비용 가능 | 주소 공간을 공유해 생성, 통신 비용을 줄일 수 있지만 동기화 비용 존재 |
| 통신 | IPC 필요 (비용 큼) | 공유 영역으로 쉽게 통신 가능하나 동기화 문제 발생 가능 |

## 컴파일과 프로세스

### 컴파일 과정 (C언어)
1. **전처리기**: #include, #define 등 매크로 처리 → `.i` 파일
2. **컴파일러**: 전형적인 네이티브 C toolchain에서는 C를 대상 아키텍처의 어셈블리로 변환 → `.s` 파일. 어셈블리 명령과 최종 기계어의 대응은 ISA와 인코딩에 따라 달라진다.
3. **어셈블러**: 어셈블리어 → 기계어(오브젝트 파일, 코드/데이터 영역 분리) → `.o` 파일
4. **링커**: 여러 오브젝트 파일과 라이브러리를 연결하고 relocation, symbol resolution을 수행해 대상 플랫폼의 실행 파일 또는 라이브러리 생성

### 실행파일 → 프로세스
1. 커널이 executable 형식을 확인하고 새 주소 공간과 필요한 파일 매핑을 구성
2. dynamic loader가 필요하면 공유 라이브러리를 매핑하고 relocation, 초기화를 수행
3. 인자와 환경을 포함한 초기 stack, 레지스터와 entry point를 구성
4. 스케줄링 뒤 실행하며 매핑된 페이지는 필요할 때 물리 메모리로 fault-in

### 컴파일 vs 인터프리터

| 방식 | 특징 | 언어 |
|------|------|------|
| AOT 네이티브 컴파일 | 실행 전에 대상 기계어 생성 | 전형적인 C/C++ toolchain |
| 바이트코드/VM/JIT | 중간 표현을 실행하고 필요하면 런타임에 기계어 생성 | JVM, .NET, 주요 JavaScript 엔진 등 |
| 인터프리터 실행 | 구현이 소스나 중간 표현을 해석해 실행 | 일부 Python/Ruby 구현과 개발용 런타임 |

언어 이름만으로 컴파일 언어와 인터프리터 언어를 고정 분류할 수 없다. 같은 언어도 구현체와 배포 방식에 따라 AOT, JIT, interpreter를 조합한다.
