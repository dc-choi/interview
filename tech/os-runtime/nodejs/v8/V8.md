# V8 엔진

Google이 개발한 오픈소스 C++ JavaScript 엔진. Chrome, Electron, Node.js의 핵심 컴포넌트.

## 왜 엔진을 이해해야 하는가

엔진은 **개발자가 정해진 룰을 따른다는 가정** 아래 코드를 최적화한다. 우리가 짠 코드가 엔진의 전략 조건을 만족하지 못하면 **최적화 이점을 전혀 받지 못한다** — GC, Inline Cache, TurboFan 모두 마찬가지. 엔진 관점의 시야를 가진 개발자와 그렇지 않은 개발자 사이에는 분명한 간극이 있다.

## 역할

JavaScript 소스코드를 받아 기계어로 컴파일하고 실행한다. 메모리 할당, 가비지 컬렉션, 최적화를 모두 V8이 담당한다.

## 주요 JS 엔진

| 엔진 | 개발사 | 사용처 |
|---|---|---|
| **V8** | Google | Chrome, Electron, Node.js |
| **JSC** (JavaScriptCore) | Apple | Safari, React Native |
| **SpiderMonkey** | Mozilla | Firefox, MongoDB |

1·2차 브라우저 전쟁을 거쳐 살아남은 세 엔진. 모든 엔진은 공통적으로 **JIT 컴파일**과 **다계층 컴파일러** 아키텍처를 채택한다.

## 고수준 언어 vs 저수준 언어

- **고수준 언어**: C, C++, JS, Java 등. 가독성·유지보수성 좋음, 성능은 상대적으로 낮음
- **저수준 언어**: 어셈블리, 기계어. 실행 속도 빠르고 성능 좋음, 가독성 낮고 유지보수 어려움

개발자가 작성한 고수준 JS를 CPU가 이해할 수 있는 저수준 언어로 변환하는 작업을 엔진이 한다.

## 인터프리터 vs 컴파일러 vs JIT

| 방식 | 특징 | 예시 |
|---|---|---|
| **인터프리터** | 소스코드를 한 줄씩 실행. 컴파일 단계 없어 시작 빠르지만 반복 실행 느림 | Python, JS, Ruby |
| **컴파일러** | 실행 전 전체 코드를 기계어로 변환. 시작 느리지만 실행 빠름. 단 한 번만 수행 | C, C++, C#, Java |
| **JIT** | 런타임에 코드를 프로파일링해 hot 부분만 컴파일. 인터프리터의 빠른 시작 + 컴파일러의 빠른 실행 혼합 | 모든 주요 JS 엔진 |

## 컴파일 파이프라인 (9.1 이후)

```
JS 소스
  ↓ Parser → AST
  ↓ Ignition → Bytecode 실행 + 프로파일링
  ↓
  ├─ SparkPlug → 비최적화 기계어
  └─ TurboFan  → 최적화 기계어 (Deopt 시 Bytecode로 복귀)
```

상세: [[V8-Ignition-TurboFan|V8 컴파일 파이프라인]]

## 역사 요약

- **~5.9 (2017 이전)**: Full-codegen(전체 선컴파일) + Crankshaft(최적화). 메모리 과소비, ES6 최적화 한계
- **5.9+ (2017 초)**: Ignition(인터프리터) + TurboFan(최적화)로 전면 재설계
- **9.1**: SparkPlug(비최적화 중간 계층) 도입 → Ignition↔TurboFan 간극 해소
- **114+**: Maglev(경량 최적화) 추가 → SparkPlug↔TurboFan 사이 추가 계층

## 바이트코드

Ignition이 생성하는 중간 표현(IR). 플랫폼 독립적인 V8 전용 명령어 셋.

```bash
node --print-bytecode app.js
```

```
LdaSmi [1]            // accumulator에 정수 1 로드
Star r0               // r0에 저장
LdaSmi [2]
Add r0, [0]           // r0과 accumulator 더하기
Return
```

## 메모리 할당 (TLAB)

V8은 Young Generation에 객체를 할당할 때 스레드마다 미리 확보된 **TLAB(Thread-Local Allocation Buffer)**에서 **포인터 범프(pointer bump)** 방식으로 즉시 할당한다.

- 중앙 할당자 락 경합 없음
- free-list 탐색 불필요 → **O(1) 할당**
- TLAB 꽉 차면 새 TLAB 배정 또는 GC 트리거

세대 가설(대부분 객체는 금방 죽는다)과 맞물려 극도로 빠른 할당 속도를 낸다.
