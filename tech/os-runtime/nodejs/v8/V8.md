### V8 엔진
Google이 개발한 오픈소스 C++ JavaScript 엔진. Chrome과 Node.js의 핵심 컴포넌트.

## 역할
```
JavaScript 소스코드를 받아 머신코드로 컴파일하고 실행한다.
메모리 할당, 가비지 컬렉션, 최적화를 모두 V8이 담당한다.
```

## 컴파일 파이프라인

### 전체 흐름
```
JS 소스코드
    │
    ▼
  Parser  ─────────→  AST (Abstract Syntax Tree)
    │
    ▼
  Ignition (인터프리터)  ─→  Bytecode 생성 및 실행
    │
    │  (핫 함수 감지: 자주 실행되는 코드)
    ▼
  TurboFan (최적화 컴파일러)  ─→  최적화된 Machine Code
    │
    │  (최적화 가정이 깨지면)
    ▼
  Deoptimization  ─→  다시 Bytecode로 복귀
```

### Ignition (인터프리터)
```
V8의 바이트코드 인터프리터. 모든 JS 코드는 먼저 Ignition을 통해 실행된다.

1. Parser가 생성한 AST를 바이트코드로 변환
2. 바이트코드는 레지스터 기반 (스택 기반이 아님)
3. 빠른 시작 시간: 전체 코드를 미리 컴파일하지 않아 메모리 효율적
4. 실행 중 프로파일링 데이터 수집 (어떤 함수가 자주 호출되는지, 어떤 타입이 쓰이는지)
```

### TurboFan (최적화 컴파일러)
```
Ignition이 수집한 프로파일링 데이터를 바탕으로 핫 코드를 최적화된 머신코드로 컴파일한다.

주요 최적화 기법:
- Inline Caching: 프로퍼티 접근 패턴을 캐싱하여 반복 접근 가속 (아래 섹션 참고)
- Inlining: 작은 함수를 호출부에 직접 삽입하여 호출 오버헤드 제거
- Dead Code Elimination: 실행되지 않는 코드 제거
- Constant Folding: 컴파일 시점에 계산 가능한 상수 표현식을 미리 계산
- Loop Unrolling: 반복문의 일부를 풀어 분기 비용 절감
```

### Inline Caching 상세
```
동일한 프로퍼티 접근이 반복될 때, 매번 Hidden Class 조회를 하는 대신
직전 접근 결과를 호출 지점(call site)에 "캐싱"해 즉시 재사용한다.

IC는 관찰된 Hidden Class의 개수에 따라 4가지 상태로 전이한다:

1. Uninitialized: 아직 한 번도 실행되지 않은 상태
2. Monomorphic: 한 종류의 Hidden Class만 관찰됨 → 가장 빠름 (캐시 히트 1회 비교)
3. Polymorphic: 2~4개의 Hidden Class 관찰 → 여러 캐시를 순차 비교
4. Megamorphic: 5개 이상의 Hidden Class 관찰 → 캐시를 포기하고 전역 해시 테이블 조회 (느림)

같은 함수에 다양한 모양의 객체를 넘기면 polymorphic → megamorphic으로 전락하여
TurboFan 최적화가 포기될 수 있다. "같은 함수에는 같은 모양의 객체만 넘기기"가
IC 최적화의 핵심 원칙이다.
```

### Deoptimization
```
TurboFan이 최적화 시 세운 가정(예: "이 변수는 항상 number")이 깨지면 발생.
최적화된 머신코드를 버리고 Ignition의 바이트코드로 되돌아간다.

흔한 deopt 원인:
- 타입 변경 (number → string)
- 새로운 프로퍼티 추가로 Hidden Class 변경
- try-catch 블록 내 코드 (일부 최적화 불가)
```

## Hidden Classes (Maps)
```
V8은 JS 객체의 프로퍼티 접근을 빠르게 하기 위해 내부적으로 Hidden Class를 사용한다.
(V8 내부에서는 "Map"이라 부름)

동적 언어인 JS는 객체 구조가 런타임에 변할 수 있지만,
V8은 같은 구조의 객체들이 같은 Hidden Class를 공유하도록 만들어 접근 속도를 높인다.
```

### 동작 원리
```javascript
// 두 객체가 같은 순서로 프로퍼티를 추가하면 같은 Hidden Class를 공유
const a = {};  // Hidden Class C0
a.x = 1;       // Hidden Class C0 → C1 (x 추가)
a.y = 2;       // Hidden Class C1 → C2 (y 추가)

const b = {};  // Hidden Class C0
b.x = 3;       // Hidden Class C0 → C1 (같은 전이!)
b.y = 4;       // Hidden Class C1 → C2 (같은 전이!)
// a와 b는 같은 Hidden Class C2를 공유 → Inline Cache 히트
```

### 성능 팁
```
1. 객체 프로퍼티는 항상 같은 순서로 초기화하라
   → Hidden Class 전이 경로가 동일해져 IC(Inline Cache)가 잘 동작

2. 생성자에서 모든 프로퍼티를 초기화하라
   → 나중에 프로퍼티를 추가하면 Hidden Class가 바뀌어 deopt 발생 가능

3. delete 연산자를 피하라
   → 프로퍼티 삭제는 Hidden Class를 변경시켜 최적화를 무효화
```

## 바이트코드
```
Ignition이 생성하는 중간 표현(IR). 플랫폼 독립적이며 V8 전용 명령어 셋을 가진다.

Node.js에서 확인하는 방법:
node --print-bytecode app.js

예시:
LdaSmi [1]            // 레지스터 a에 정수 1 로드
Star r0               // a를 레지스터 r0에 저장
LdaSmi [2]            // 레지스터 a에 정수 2 로드
Add r0, [0]           // r0과 a를 더하기
Return                // 결과 반환
```

## 메모리 할당 (TLAB)
```
V8은 Young Generation에 객체를 할당할 때 스레드마다 미리 확보해둔
TLAB(Thread-Local Allocation Buffer) 영역에서 "포인터 범프(pointer bump)" 방식으로
즉시 할당한다.

- 중앙 할당자에 락을 걸지 않아 스레드 경합이 없음
- free-list 탐색 없이 포인터 하나만 증가시키면 되므로 O(1) 할당
- TLAB가 꽉 차면 새 TLAB를 배정받거나 GC를 트리거

이 방식은 "모든 객체는 결국 TLAB에서 시작한다" 전제 하에,
대부분의 객체가 금방 죽는다는 세대 가설과 맞물려 극도로 빠른 할당 속도를 낸다.
```
