# OS 개요와 동시성

## 운영체제 개요

시스템 리소스를 제어하고 프로그램에 대한 일반적인 서비스를 지원하는 시스템 소프트웨어. 컴퓨터는 운영체제 없이도 동작하지만 더 유연하고 많은 기능을 위해 운영체제의 도움이 필요.

### 운영체제가 관리하는 것
- 프로세스 관리 / 메모리 관리 / 하드웨어 관리 / 파일 시스템 관리

### 운영체제 역사

| 시기 | 변화 |
|------|------|
| 1940 | 에니악(1943). 배선+스위치로 프로그래밍. IO 중 계산 불가 |
| 1950 | IC 개발. 펀치카드 사용. 오퍼레이터 오버헤드 → 싱글스트림 배치시스템 |
| 1960 | 시분할 시스템, 멀티 프로그래밍, 파일시스템, UNIX 등장. 메모리 침범 이슈 → 베이스 레지스터 도입. OS가 학문으로 자리잡음 |
| 1970 | 하드웨어 저렴화, 개인용 PC 보급, 매킨토시 GUI |

### 운영체제 구조
- **커널**: 프로세스, 메모리, 저장장치 관리의 핵심. 직접 접근 불가
- **GUI/CLI**: 사용자가 커널에 접근하는 인터페이스
- **시스템 콜**: 애플리케이션이 커널에 접근하는 인터페이스 (직접 접근 차단)
- **드라이버**: 하드웨어가 커널에 접근하는 인터페이스

### 부팅 과정
1. ROM에 저장된 BIOS가 전기 신호를 받고 실행
2. 주요 하드웨어 이상 유무 확인 (이상 있으면 실행 중지)
3. 하드디스크의 MBR(마스터 부트 레코드)에 저장된 부트로더를 메모리로 로드
4. 운영체제를 메모리로 불러오면 부팅 완료

### 인터럽트
- 폴링 방식의 문제를 해결. CPU가 IO 관리자에게 명령 후 다른 작업 수행
- IO 완료 시 CPU에게 신호 → CPU가 ISR(인터럽트 서비스 루틴)을 실행하여 작업 완료
- **비동기적** 동작으로 성능 이점
- 하드웨어 인터럽트: 입출력
- 소프트웨어 인터럽트: 유효하지 않은 메모리 접근, 0으로 나누기 등

### 컴퓨터 하드웨어 (폰 노이만 구조)
- CPU와 메모리를 버스로 연결. 버스는 데이터를 전달하는 통로
- **메인보드**: 다른 하드웨어를 연결하는 장치
- **CPU**: 산술논리연산장치(ALU), 제어장치, 레지스터
- **메모리**: RAM, ROM

## 동시성vs병렬성

| 구분 | 정의 | 예시 |
|------|------|------|
| 동시성(Concurrency) | 여러 가지 일이 동시에 진행되는 것 | 1개 CPU가 시분할로 여러 작업 처리 |
| 병렬성(Parallelism) | 같은 일을 여러 주체가 함께 진행하는 것 | 멀티코어에서 동시에 연산 수행 |

## 커널모드vs유저모드

### 컴퓨터를국가에비유
| 국가 | 컴퓨터 |
|------|--------|
| 영토/영공/영해 | 하드웨어 |
| 정부 조직 | OS (시스템 소프트웨어) |
| 국민 | 프로그램(프로세스) |
| 국민의 공간 | 프로세스 메모리 (다른 프로세스 침범 불가) |
| 경찰 | 디버거 (다른 프로세스 접근 가능) |

### 커널(Kernel)
- OS의 핵심, 컴퓨터의 법을 구현
- **역할**: 입출력 제어, 자원(CPU+RAM) 관리, 접근 제어
- OS는 하드웨어에 의존적 (64비트 OS → 64비트 CPU 필요)

### 두모드의관계

```
[유저 모드]  프로세스 → 디바이스 파일(인터페이스) → read/write/execute
                              ↓
[커널 모드]  엔진/코어 → 드라이버(펌웨어) → 하드웨어(디바이스)
```

- **디바이스 파일**: 커널 구성 요소에 대한 추상적 인터페이스를 유저 모드에 제공
- 프로세스는 하드웨어를 직접 통제하지 않음 → OS가 전부 통제
- 프로세스가 디바이스 파일에 접근하려면 OS로부터 권한 부여가 필요

### 디스크접근예시
```
[하드웨어] 디스크
    ↑
[커널모드] 드라이버(펌웨어) → 파일시스템
    ↑
[유저모드] 파일시스템 추상화 인터페이스 → 프로세스
```

## 프로그램→프로세스→스레드

| 개념 | 설명 |
|------|------|
| 프로그램 | 설치된 상태, 디스크에 저장된 코드 |
| 프로세스 | 프로그램을 실행하면 생성되는 인스턴스 |
| 스레드 | 프로세스 속의 실행 단위, 프로세스 자원을 공유 |

### 메모리영역
| 영역 | 소유자 | 특징 |
|------|--------|------|
| 스택 | 스레드별 개별 소유 | 함수 호출, 지역 변수 |
| 힙 | 프로세스 전체 공유 | 동적 할당, 동시성 문제 발생 가능 |

- 동시성 문제로 인해 기억 공간을 스택/힙으로 구분

### 프로세스메모리구조(상세)
| 영역 | 저장내용 | 방향 |
|------|---------|------|
| 코드(Text) | 실행할 프로그램의 기계어 코드 | 고정 |
| 데이터(Data) | 전역 변수, static 변수 | 고정 |
| 힙(Heap) | 동적 할당 (new, malloc) | 낮은 주소 → 높은 주소 |
| 스택(Stack) | 함수 호출, 지역 변수, 매개변수 | 높은 주소 → 낮은 주소 |

### Recursion이stack/heap에미치는영향
- 재귀 호출마다 스택 프레임이 쌓임 → 깊은 재귀는 **Stack Overflow** 발생
- 꼬리 재귀 최적화(TCO)가 없는 언어에서는 반복문으로 변환하는 것이 안전
- 재귀 내에서 동적 할당을 하면 힙도 함께 증가

### 단편화(Fragmentation)
| 종류 | 설명 |
|------|------|
| 외부단편화 | 할당/해제를 반복하면 메모리 사이에 사용할 수 없는 작은 빈 공간이 생김 |
| 내부단편화 | 할당된 블록이 요청보다 커서 블록 내부에 사용되지 않는 공간이 생김 |

- 외부 단편화 해결: 압축(Compaction), 페이징
- 내부 단편화 해결: 적절한 블록 크기 선택, 버디 시스템

### 메모리지역성(Locality)
| 종류 | 설명 |
|------|------|
| 시간적지역성(Temporal) | 최근 접근한 데이터에 다시 접근할 가능성이 높음 |
| 공간적지역성(Spatial) | 접근한 데이터 근처의 데이터에 접근할 가능성이 높음 |

- 캐시가 효율적으로 동작하는 근거
- 배열 순차 접근이 연결 리스트보다 빠른 이유 (공간적 지역성)
