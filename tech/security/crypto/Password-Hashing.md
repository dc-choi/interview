# Password Hashing

## 해시 vs 암호화 해시

암호화는 기본적으로 NP 문제를 활용한다.

암호화 해시가 만족해야 하는 조건:
- **역상 저항성**: 해시를 보고 입력값을 찾기 어려워야 함
- **제2역상 저항성**: 입력의 해시를 유지하면서 입력을 변경하기 어려워야 함
- **충돌 저항성**: 같은 해시를 가진 두 입력을 찾기 어려워야 함
- 같은 입력이라도 prefix가 같으면 안 됨

## 해시 DoS

해시 충돌을 의도적으로 발생시키는 DDoS 공격이다.
알고리즘 특성상 해시 충돌이 일어날 수밖에 없고, 입력이 같으면 출력이 같으므로 무수히 많은 충돌을 유발할 수 있다.

## 사용하면 안 되는 알고리즘

| 알고리즘 | 이유 |
|---|---|
| MD5, SHA-1 | 절대 사용 금지 (충돌 발견됨) |
| SHA-256 | 패스워드 해시로 부적합. 비트코인 채굴기가 이 알고리즘을 가장 빠르게 연산함 |
| bcrypt | 현재 기준으로 안전하지 않음 |

## Salt의 올바른 이해

- Salt는 **레인보우 테이블 방어용**이지, 해시를 암호화하는 역할이 아님
- "salt + 복잡한 password = 안전"이라는 생각은 위험
- DB가 침해당해도 암호화되어 있으니 괜찮다는 생각도 위험

## 권장 패스워드 해시 알고리즘

하드웨어 발전으로 연산 속도는 빨라졌지만 **메모리는 저렴해지지 않았다**.
따라서 메모리를 많이 쓰도록 강제하는 것이 핵심 전략이다.

| 알고리즘 | 설명 |
|---|---|
| **argon2** | blake2를 반복한 해시, 메모리 사용 강제 |
| **balloon** | SHA-3를 여러 번 반복 (KISA 권고 SHA-256 필요 시 대안) |
| **yescrypt** | 메모리 하드 해시 |

## 암호화 원칙

- 모든 암호화의 안전성은 **키에 의해서만** 보장되어야 함
- 현대 보안에서는 감청당해도 안전하게 통신 가능 -> **HTTPS 필수**
- TLS 1.3 이상 권고, 최소 1.2 사용 필요

## JWT 보관

- JWT 자체는 토큰 포맷일 뿐
- 실제 고민 포인트: 토큰을 어떻게 보관할 것인가
- OTP에 입력 횟수 제한이 없으면 위험
