# RSA 암호화

RSA(Rivest–Shamir–Adleman)는 **큰 정수의 소인수분해가 어렵다**는 수학적 난제에 기반한 공개키 암호 시스템. 1977년 MIT에서 발표된 이래 TLS·SSH·PGP·JWT 서명 등 현대 보안의 주요 자리를 차지해왔다. 암호화와 디지털 서명을 동시에 지원한다.

## 핵심 명제

- **한 방향 함수**: 두 소수 p·q를 곱해 n을 만드는 건 쉽지만, n에서 p·q를 분해하는 건 현실적 시간 안에 불가능
- 공개키·개인키 쌍은 이 한 방향성을 이용해 **"잠그기 전용"과 "풀기 전용"** 의 분리를 만들어낸다
- 안전성은 키 길이에 달려 있으며, **2048비트 미만은 현재 비권장**

## 키 생성 절차

1. **큰 소수 p, q 선택** (보통 1024비트 이상씩 → n은 2048+비트)
2. **n = p × q** 계산 → 공개·개인키 모두에서 사용되는 모듈러스
3. **오일러 totient Φ(n) = (p − 1)(q − 1)** 계산
4. **공개 지수 e** 선택: 1 < e < Φ(n), gcd(e, Φ(n)) = 1. 관례적으로 e = 65537(2^16 + 1)
5. **개인 지수 d** 계산: e × d ≡ 1 (mod Φ(n)) — 확장 유클리드 알고리즘
6. **공개키 = (n, e)**, **개인키 = (n, d)**
7. p, q, Φ(n)은 **즉시 파기** — 유출되면 d를 쉽게 재구성 가능

## 암호화 · 복호화

메시지 M을 정수로 인코딩(0 ≤ M < n).

- **암호화**: `C = M^e mod n` — 공개키로 누구나 수행
- **복호화**: `M = C^d mod n` — 개인키 보유자만 수행

수학적 성립 근거는 **오일러 정리** `M^(φ(n)) ≡ 1 (mod n)` → `M^(e·d) ≡ M (mod n)`.

## 디지털 서명

암호화와 키 사용 방향이 반대.

- **서명 생성**: `S = H(M)^d mod n` — 개인키로 메시지 해시를 서명
- **검증**: `H(M) == S^e mod n` — 공개키로 검증

실전에서는 서명 전에 해시(SHA-256 등) + 패딩(PSS)을 적용하여 선택 평문·재사용 공격을 막는다.

## 패딩의 중요성

순수 RSA(textbook RSA)는 결정적 함수라 **같은 평문 → 같은 암호문**이 되어 패턴 분석에 취약. 따라서 반드시 패딩을 적용한다.

| 패딩 | 용도 | 특징 |
|---|---|---|
| **PKCS#1 v1.5** | 레거시 | Bleichenbacher 공격에 취약 → 비권장 |
| **OAEP** (Optimal Asymmetric Encryption Padding) | 암호화 | 현대 표준 |
| **PSS** (Probabilistic Signature Scheme) | 서명 | RSA-PSS가 현대 서명 표준 |

## 성능 특성

- **암호화 지수 e가 작음**(65537) → 공개키 연산은 상대적으로 빠름
- **복호화 지수 d는 n만큼 큼** → 개인키 연산이 느림. CRT(중국인의 나머지 정리) 최적화로 4배 가속
- **대칭키 대비 약 1000배 느림** → 본 데이터를 직접 암호화하지 않고, **세션 키(AES)를 RSA로 감싸서 전달**

## 보안 강도 · 키 길이

| RSA 키 | 동등 대칭키 | 비고 |
|---|---|---|
| 1024비트 | ~80비트 | **취약, 사용 금지** |
| 2048비트 | 112비트 | 현재 최소 권장 |
| 3072비트 | 128비트 | 장기 보관 문서 |
| 4096비트 | ~140비트 | 오버스펙, 성능 저하 |

ECC 256비트 = RSA 3072비트 수준 보안 → 모바일·IoT는 ECC가 유리.

## 공격 벡터

- **소인수분해 발전** — NFS(General Number Field Sieve) 등 알고리즘 개선 + 하드웨어 → 1024비트는 이미 위험
- **사이드 채널** — 타이밍 공격, 전력 분석 → 일정 시간 구현 필요
- **약한 난수** — p, q 선택 시 PRNG이 약하면 충돌. Debian OpenSSL 2008년 사고 사례
- **e·d 재사용 오류** — 같은 메시지를 여러 다른 n에 같은 e로 암호화(Håstad broadcast attack)
- **양자 컴퓨터(Shor 알고리즘)** — 충분히 큰 양자컴이 등장하면 RSA·ECC 모두 다항시간에 깨짐 → Post-Quantum Crypto(CRYSTALS-Kyber 등) 이행 필요

## 실무 사용처

- **TLS 인증서** — 서버 공개키 인증서로 사용(단, 세션 키 교환은 ECDHE로 이행 중)
- **JWT RS256 / PS256** — 발급자만 서명, 모든 검증자가 공개키로 확인
- **SSH 키** — `ssh-keygen -t rsa -b 4096`. 최근은 Ed25519 권장
- **코드 서명·패키지 서명** — APT·RPM·npm 등
- **PGP/GPG** — 이메일·파일 암호화 및 서명

## 흔한 실수

- 평문을 그대로 RSA로 암호화해 큰 파일을 감싸려 함 → 느리고 위험. **세션 키 교환에만 사용**
- 키 길이를 1024비트로 남겨두기 → 2048비트 이상으로 rotate
- 패딩 없이 `textbook RSA` 사용 → 결정적 암호문으로 공격당함
- 개인키를 서버 평문 저장 → HSM·KMS·Vault로 보호
- 같은 키 쌍을 암호화와 서명 모두에 재사용 → 용도별 분리 권장

## 면접 체크포인트

- RSA 안전성의 수학적 근거(소인수분해 난제) 한 문장
- 키 생성 단계에서 e = 65537이 관례인 이유(속도·안전성 균형)
- 암호화 키 방향 vs 서명 키 방향의 차이
- RSA vs ECC 키 길이 비교와 모바일 선호도
- Post-Quantum 시대에 RSA가 위협받는 이유(Shor 알고리즘)
- 왜 대용량 데이터를 RSA로 직접 암호화하지 않는가
