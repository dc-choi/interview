# 실시간 위치 기반 매칭 (공간 분할)

주문과 배달원처럼 **움직이는 두 집합을 실시간으로 짝짓는** 문제. 지역을 나눠 계산하면 병렬화가 쉬워 보이지만, 공간은 단순 파티셔닝만으로 나뉘지 않는다. 경계 문제와 동시 계산 충돌을 함께 설계해야 한다.

## 공간은 단순 파티션보다 어렵다

지역을 행정구역이나 격자로 딱 잘라 각 구역을 독립 계산하면 **경계 문제**가 생긴다.

- 주문은 A구역, 가장 가까운 배달원은 바로 옆 B구역에 있으면, 구역을 잘라 계산했을 때 서로 보이지 않는다.
- 매칭은 "내 구역 안"이 아니라 "일정 반경 안"의 후보를 함께 봐야 한다. 즉 계산 단위(구역)와 탐색 범위(반경)가 어긋난다.

그래서 공간 분할은 데이터를 나누는 것(where)뿐 아니라 **언제 계산할지(when)**까지 설계해야 안정적이다.

## H3 — 육각형 격자 인덱싱

지구 표면을 육각형 격자로 나누는 공간 인덱싱 라이브러리(Uber). 위치(위, 경도)를 육각형 셀 ID(정수)로 변환해 공간 계산의 단위로 쓴다.

- **해상도 계층**: 큰 육각형을 더 작은 육각형으로 재귀 분할. 해상도를 올리면 셀이 작아진다.
- **육각형의 장점**: 인접 셀과의 중심 거리가 균일. 사각형 격자는 직각 인접과 대각선 인접의 거리가 달라 왜곡이 생긴다.
- **k-ring**: 특정 셀의 반경 k 이웃 셀을 O(1)에 열거 → "반경 안 후보"를 격자 ID 집합으로 치환. 경계 문제를 이웃 셀 포함으로 완화.
- 격자 ID는 정수라, R-Tree 같은 공간 인덱스 없이 **정수 인덱스, 해시 키**로 조회를 단순화할 수 있다.

H3만 쓴다고 경계 문제가 사라지지는 않는다. 이웃 셀과의 겹침, 계산 순서, 동시 처리 충돌을 함께 봐야 한다. (DB 저장, 배송 영역 판별 관점의 H3는 [[Spatial-Index-MySQL#사례: 쿠팡 로켓배송 — H3 공간 색인|공간 색인]] 참고.)

## 시간 분할 — 경계 충돌을 시간으로 푼다

모든 지역을 **동시에** 계산하면, 인접한 두 구역이 같은 배달원을 동시에 후보로 잡아 이중 배차하는 충돌이 생긴다.

해법은 공간을 여러 그룹으로 나눠 **시간 순서로 처리**하는 것.

- 예: 전체 격자를 7개 그룹으로 나누고, 짧은 주기 안에서 1번 → 2번 → 3번 그룹 순으로 계산.
- 서로 인접해 후보가 겹칠 가능성이 큰 영역이 **같은 시각에 계산되지 않도록** 그룹을 배치(체스판처럼 인접끼리 다른 그룹).
- 결과적으로 동시성 충돌을 락 없이 스케줄링으로 회피. 분산 락, 낙관적 락의 재시도 비용을 줄인다.

공간 분할 + 시간 분할을 합치면, "어느 격자를(공간) 언제(시간) 누가(파티션, 인스턴스) 계산하는가"가 모두 정해진다.

## Kafka Streams와의 결합

이 매칭 계산은 보통 [[MQ-Kafka-Streams|Kafka Streams]] 위에서 돈다.

- 주문, 배달원 위치 이벤트를 **H3 셀 ID를 키로** 발행 → 같은 셀 이벤트가 같은 파티션에 모여 지역 지역성 확보.
- 셀별 현재 주문 수, 후보 배달원 목록을 **상태 저장소**에 누적 → 매번 DB 조회 없이 최신 상황판 유지.
- 시간 분할 그룹은 스케줄, 윈도우와 맞물려 처리 순서를 제어.

단, H3 셀을 키로 삼을 때 특정 도심 셀에 트래픽이 몰리는 핫 파티션 (작성 예정: `Hot-Partition`) 위험이 있다. 셀 해상도, 키 조합으로 분산을 조정해야 한다.

## 면접 체크포인트

- 지역을 격자로 잘라 독립 계산하면 왜 문제인가 → 경계의 후보를 놓친다
- H3가 사각형 격자, geohash보다 나은 점 → 인접 셀 거리 균일, k-ring 탐색
- 인접 구역의 동시 배차 충돌을 락 없이 어떻게 막나 → 공간 그룹의 시간 분할
- 위치를 파티션 키로 쓸 때의 리스크 → 도심 셀 핫 파티션
