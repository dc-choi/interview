# 모바일 서비스 개발 방식 4유형

클라이언트를 어떻게 만드느냐가 배포 주기, 비용, 기기 기능 활용 범위를 결정한다. 정답은 없고, 기능의 정교함과 업데이트 속도와 리소스 사이의 트레이드오프다.

## 4유형 비교

| 구분 | 네이티브 앱 | 모바일 웹 | 웹 앱 (SPA) | 하이브리드 앱 |
|---|---|---|---|---|
| 기술 | OS별 SDK (Kotlin, Swift) | HTML, CSS, JS | HTML, CSS, JS | 네이티브 셸 + 웹뷰 |
| 진입 | 앱스토어 다운로드 | URL 접속 | URL 접속 | 앱스토어 다운로드 |
| 업데이트 | 스토어 심사 필요 | 즉시 반영 | 즉시 반영 | 웹 영역은 즉시, 셸은 심사 |
| 비용 | 매우 높음 (OS별 별도 개발) | 낮음 | 낮음 | 중간 |
| 성능, 기기 기능 | 최고 — 카메라, GPS, 센서, 푸시 전부 | 브라우저 공통 기능만 | 모바일 웹과 동일 제약 | 중요 기능만 네이티브로 |

- **모바일 웹 vs 웹 앱** — 둘 다 브라우저 기반이지만, 모바일 웹은 풀 브라우저 방식(전체 새로고침)이고 웹 앱은 SPA로 변경 부분만 Ajax로 갱신해 앱과 유사한 화면 전환을 낸다.
- **하이브리드** — 네이티브 앱 안에 웹뷰를 띄운다. 자주 바뀌는 화면은 웹으로(즉시 배포), 정교한 성능이 필요한 기능(카메라, 결제, 모션)은 네이티브로. 한 코드로 양 OS를 커버해 리소스를 아낀다.

## 선택 기준

1. **기능** — 고성능 카메라 필터, 정교한 모션이 핵심이면 네이티브. 콘텐츠와 폼 중심이면 웹 계열로 충분.
2. **배포 주기** — 빠른 실험과 핫픽스가 중요하면 웹, 웹뷰 비중을 높인다. 스토어 심사는 며칠 단위(플랫폼별 상이)라 긴급 대응이 막힌다.
3. **리소스** — OS별 네이티브 2벌 개발은 인력과 비용이 2배. 초기 팀은 하이브리드나 웹 앱으로 시작하는 경우가 많다.

## 백엔드 관점: 클라이언트 유형이 서버에 미치는 영향

- **여러 버전의 클라이언트가 공존한다** — 네이티브, 하이브리드 앱은 스토어 심사와 사용자의 업데이트 지연 때문에 구버전이 오래 살아남는다. 서버는 항상 다중 버전을 동시에 서빙하므로 **API 하위 호환성 유지와 버저닝**([[API-Conventions-Response|API 컨벤션]])이 강제되고, 강제 업데이트 정책(최소 지원 버전)이 함께 설계돼야 한다. 방치하면 다중 API 버전 동시 운영의 유지보수 부담으로 돌아온다.
- **웹뷰, SPA는 전부 API 의존** — 화면이 서버 렌더링이 아니라 API 바인딩이므로 CORS, 인증 토큰 저장 위치, 응답 스키마 안정성이 클라이언트 유형과 직결된다.
- **푸시 알림은 네이티브 영역** — 웹 계열은 푸시가 제한되므로, 알림이 핵심인 서비스는 최소한 하이브리드 셸이 필요하다 (메시징 채널 설계의 전제).

## 면접 체크포인트

- 앱 배포 주기 질문에 스토어 심사 + 구버전 공존 → API 하위 호환과 강제 업데이트 정책으로 답하는 것이 백엔드다운 답.
- 하이브리드 선택 이유는 변경 빈도 기준으로 — 자주 바뀌는 화면은 웹뷰, 성능 민감 기능은 네이티브라는 분할 원칙.
- 기획, 클라이언트와의 협업에서 배포 비용(심사 며칠)을 아는 것이 일정 합의의 기본기.
