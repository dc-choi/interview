# HTTP 상태 코드

HTTP 상태 코드는 클라이언트 요청의 **수행 결과를 알리는 약속**이다. 강제 사항은 아니지만, 브라우저, HTTP 통신 라이브러리, 검색 엔진, 프레임워크가 모두 이 표준을 전제로 동작하므로 지키지 않으면 예측 못 한 부작용이 생긴다. 코드는 첫 자리로 5개 부류로 나뉜다.

| 부류 | 의미 |
|---|---|
| 1xx | 정보 (프로토콜 전환, 계속 진행) |
| 2xx | 성공 |
| 3xx | 리다이렉션 |
| 4xx | 클라이언트 오류 (요청이 잘못됨) |
| 5xx | 서버 오류 (서버가 처리 실패) |

## 왜 표준을 지켜야 하나 — 200으로 실패를 숨기지 말 것

흔한 안티패턴은 **요청이 실패해도 200 OK를 내려주고 본문에 `{ "success": false }`로 실패를 표현**하는 것이다. 클라이언트의 HTTP 라이브러리(fetch, axios 등)는 상태 코드로 성공, 실패를 판단해 4xx, 5xx에서만 에러를 던지도록 설계되어 있다. 그래서 올바른 상태 코드를 쓰면 `try/catch` 한 겹으로 에러가 잡히지만, 200으로 실패를 숨기면 **본문을 다시 파싱해 `if (!success) throw` 같은 중복 분기**를 강제하게 된다 — 게다가 서버가 죽으면 500, 502가 내려오므로 `try/catch`도 여전히 필요해, 핸들링이 이중으로 지저분해진다.

브라우저도 상태 코드를 엄격히 본다. 200이면 성공으로 처리하고, 3xx면 자동 리다이렉트하며, 4xx, 5xx는 다르게 인식한다. 검색 엔진 봇은 301을 받으면 색인을 갱신한다(SEO). 그러니 상황에 맞는 코드를 내리는 것이 클라이언트와의 인터페이스를 명확히 한다.

## 2xx — 성공

- **200 OK**: 단순 성공. 대부분의 성공 응답을 이 하나로 처리한다.
- **201 Created**: 요청 성공 + **새 리소스 생성**(회원가입, 글 작성 등). 보통 `Location` 헤더에 생성된 리소스 URI를 함께 준다.
- **204 No Content**: 요청 성공 + **응답 본문 없음**(삭제 등). 삭제가 Soft인지 Hard인지는 클라이언트의 관심사가 아니다 — 리소스가 사라졌다는 사실만 전달.

## 3xx — 리다이렉션

- **301 Moved Permanently**: 리소스가 영구 이동. 브라우저는 `Location` 헤더의 URL로 자동 이동한다. HTTP로 들어온 요청을 HTTPS로 강제하거나(80→443), URL 정규화에 쓴다. 검색 엔진이 색인을 갱신하므로 SEO에 중요. 일시 이동은 302, 메소드 보존이 필요하면 307, 308.
- **304 Not Modified**: 조건부 요청(If-None-Match, If-Modified-Since) 결과 **리소스가 안 바뀜**. 본문을 재전송하지 않고 클라이언트가 캐시를 쓰게 해 페이로드를 아낀다. 캐시가 없는데 304가 오면 빈 화면이 날 수 있다. 캐싱 헤더는 [[HTTP-Content-Type|Content-Type, 표현 헤더]] 인접.

## 4xx — 클라이언트 오류

- **400 Bad Request**: 요청이 문법적, 의미적으로 잘못됨. 무엇이 잘못됐는지는 본문에 담아 주는 것이 친절하다.
- **401 Unauthorized**: **인증 안 됨** — 서버가 요청자가 누구인지 모른다("신원을 밝혀라"). 로그인 필요 API를 비로그인으로 호출한 경우. 클라이언트는 로그인 페이지로 보내기도 한다. (이름과 달리 인증(authentication) 문제다.)
- **403 Forbidden**: **인가 거부** — 누구든 이 리소스 접근은 금지("신원과 무관하게 안 됨"). 401과의 핵심 차이: 401은 신원을 모르는 것, 403은 신원과 무관하게 권한이 없는 것.
- **404 Not Found**: 요청한 리소스가 없음.
- **405 Method Not Allowed**: 리소스에 허용되지 않는 메소드 사용. 프레임워크가 해당 핸들러가 없을 때 자동으로 내리기도 한다.
- **406 Not Acceptable**: **서버 주도 컨텐츠 협상** 결과 클라이언트가 `Accept` 헤더로 요청한 타입의 리소스가 없음. 서버는 Accept의 우선순위(예: text/html → application/xml → */*)를 앞에서부터 맞춰 응답 `Content-Type`을 정한다 ([[HTTP-Content-Type|Content-Type, MIME]]).
- **408 Request Timeout**: 연결은 됐으나 **요청 본문이 도착하지 않아** 서버가 기다리다 끊음. (클라이언트 측 타임아웃)
- **429 Too Many Requests**: 짧은 시간에 과도한 요청, 또는 쿼터 초과. `Retry-After` 헤더로 재시도 시점을 안내할 수 있다 ([[Rate-Limiting|Rate Limit]]).

## 5xx — 서버 오류

- **500 Internal Server Error**: 서버 내부에서 **핸들링되지 않은 에러**. 원인을 그대로 노출하면 보안 사고가 되므로 발생 사실만 알리는 경우가 많다 — 원인은 서버 로그나 에러 모니터링(Sentry 류)으로 추적.
- **502 Bad Gateway**: 앞단 프록시(Nginx, 로드밸런서 등)가 뒤의 백엔드로부터 **유효한 응답을 못 받음** — 흔히 백엔드 앱이 죽은 경우. 백엔드 아키텍처는 보통 프록시 + 앱 2개 이상이고, 그 사이 추상적 통로가 게이트웨이다. 프록시를 앞에 두는 이유는 보안(문지기)과 처리 효율(정적 파일은 프록시가 직접 처리) ([[Reverse-Proxy|리버스 프록시]], [[Load-Balancer|로드밸런서]]).
- **503 Service Unavailable**: 서버가 **일시적으로** 처리 불가(과부하, 점검). 502와 달리 곧 회복되는 상황이라 `Retry-After`로 재시도를 안내한다. 서버리스에서 동시 실행 한도 초과, 실행 시간 초과 시에도 나타날 수 있다.
- **504 Gateway Timeout**: 408과 같은 타임아웃이지만 **백엔드 내부**에서 발생 — 프록시가 백엔드로 요청을 전달했는데 백엔드가 제때 응답하지 않음. 408은 클라이언트→서버, 504는 프록시→백엔드의 타임아웃.

## 헷갈리는 쌍 정리

- **401 vs 403**: 신원을 모름(인증) vs 신원과 무관하게 금지(인가).
- **408 vs 504**: 클라이언트→서버 본문 미도착 vs 프록시→백엔드 응답 지연.
- **502 vs 503 vs 504**: 백엔드 무응답(죽음) vs 일시적 처리 불가(과부하) vs 백엔드 응답 지연(타임아웃).
- **429 vs 503**: 둘 다 `Retry-After`를 쓰지만 429는 클라이언트의 과도한 요청, 503은 서버 측 일시 불능.

## 면접 체크포인트

- 200으로 실패를 내릴 때 클라이언트가 겪는 문제(중복 분기, 라이브러리의 상태 코드 가정)
- 401과 403의 차이(인증 vs 인가)
- 201, 204를 언제 쓰는가(생성, 본문 없는 성공)
- 301과 304의 동작(Location 리다이렉트, 조건부 캐시)
- 502/503/504의 구분과 게이트웨이의 의미
- 429/503의 `Retry-After`
