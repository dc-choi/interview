# 세션, 프레젠테이션, 애플리케이션 계층 (L5/6/7)

OSI 상위 3계층은 현대 네트워크에서 칼같이 나뉘기보다 **하나의 애플리케이션 영역**으로 묶여 다뤄진다. 전체 지도는 [[OSI-7-Layer]], 아래 L4는 [[Transport-Layer]]. 실용적 이해: **L5는 연결 유지, L6은 데이터 해석, L7은 서비스 로직 처리.**

## 왜 셋을 묶어 보나 — OSI는 오래된 모델

OSI 모델은 1970년대 후반, 강력한 메인프레임에 단순한 터미널 여러 대가 붙던 환경을 배경으로 만들어졌다. 그래서 L5와 L6은 오늘날 실제 구현과 딱 맞지 않는다. 현대에 널리 쓰는 **TCP/IP 모델은 L5/6/7을 하나의 애플리케이션 계층으로** 묶는다. 세 계층을 외우기보다 각자 푸는 문제로 이해하는 편이 실용적이다.

## L5 세션 — 논리적 대화 상태 유지

세션 계층은 누가 누구와 통신 중인지를 구분하고 그 상태를 이어가는 일을 맡는다. 물리적 연결이나 IP/포트가 아니라 **그 위의 논리적 대화 상태**를 관리한다. 메인프레임 시절 여러 터미널이 한 서버에 붙으면 IP/포트만으로 사용자를 가리기 어려웠고, 그 구분자가 세션의 출발점이었다.

### 현대의 세션 — IP가 바뀌어도 로그인이 유지된다

지하철에서 이동하면 기지국이 바뀌며 IP가 달라질 수 있는데도 영상은 이어지고 로그인은 풀리지 않는다. **애플리케이션 차원에서 세션을 유지**하기 때문이다. 대표가 HTTP 쿠키다. 서버가 로그인 사용자에게 쿠키나 토큰을 발급하고, 클라이언트는 요청마다 이를 실어 보내, 서버는 IP가 달라져도 같은 사용자로 인식한다. [[Cookie]], 세션 저장은 [[Session-Store]].

### HTTP vs FTP — 연결 의존 vs 논리적 세션

- **HTTP**: 쿠키로 세션을 유지한다. IP가 바뀌어도 같은 쿠키면 인증 상태가 이어진다.
- **FTP**: 전통적으로 특정 연결(IP)을 기준으로 인증을 판단해, IP가 바뀌면 다시 로그인해야 할 수 있다.

차이의 본질은 연결 자체에 기대느냐, 연결 위의 논리적 세션을 따로 유지하느냐다.

## L6 프레젠테이션 — 데이터를 해석 가능한 형태로

네트워크로 오는 데이터는 결국 0과 1의 바이너리다. 같은 바이너리도 숫자로도 문자열로도 읽힌다. HTTP의 `POST`도 전송 중엔 바이너리이고, 프로토콜에 맞게 문자열로 해석해야 의미가 생긴다. 프레젠테이션 계층은 **이 데이터를 어떤 형식으로 읽을 것인가**를 다룬다.

여기엔 압축 해제, 문자 인코딩, 직렬화 형식(JSON 등) 해석이 포함된다. 예를 들어 응답 헤더가 `Content-Encoding: gzip`, `Content-Type: application/json; charset=utf-8`이면, 본문은 gzip으로 압축돼 있고 풀면 UTF-8로 인코딩된 JSON이라는 뜻이다. 이 문맥으로 데이터를 애플리케이션이 이해할 형태로 변환한다. [[HTTP-Content-Type]]. TLS 암호화/복호화도 OSI 매핑상 이 계층에 둔다. [[HTTPS-TLS]]

## L7 애플리케이션 — 받은 데이터로 실제 일을 한다

최종적으로 데이터로 무엇을 할지 결정하는 계층이다.

- 브라우저: HTML, CSS, JavaScript를 해석해 화면을 그린다.
- 게임: 공격, 아이템 사용 같은 요청을 게임 규칙으로 처리한다.
- HTTP 서버: 메서드, 경로, 헤더, 인증, 본문을 읽고 응답을 만든다.

네트워크로 도착한 데이터를 실제 서비스의 의미로 바꾸는 마지막 단계다.

## AWS 로드 밸런서 — 어느 계층을 이해하는가

로드 밸런서가 동작하는 계층이 곧 라우팅에 쓸 수 있는 정보를 정한다. [[ELB]]

| | NLB (L4) | ALB (L7) |
|---|---|---|
| 보는 정보 | IP, 포트, TCP/UDP | HTTP 도메인, 경로, 헤더 |
| 라우팅 | 주소와 포트 기준 빠른 전달 | 내용 기반 라우팅 |
| 성격 | 초고성능, 단순 | 똑똑하지만 처리 단계 많음 |

ALB는 L7 정보를 해석하므로 `web.example.com`은 웹 서버로, `dashboard.example.com`은 대시보드로 보내는 식의 **호스트/경로 기반 라우팅**이 된다. 더 많은 정보를 보는 만큼 NLB보다 처리 단계가 많다. NLB가 주소와 문 번호만 보고 빠르게 배달하는 택배기사라면, ALB는 상자 안 주문서까지 읽고 부서로 보내는 접수 담당자다. L4 관점은 [[Transport-Layer]].

## 면접 체크포인트

- OSI 상위 3계층이 TCP/IP에서 애플리케이션 계층으로 묶이는 이유(오래된 모델)
- 세션 = 논리적 대화 상태, IP가 바뀌어도 쿠키/토큰으로 로그인이 유지되는 원리
- HTTP(논리적 세션)와 FTP(연결 의존) 세션 유지 차이
- 프레젠테이션 = 바이너리 해석, 인코딩/압축/직렬화, `Content-Encoding`/`Content-Type`의 의미
- 애플리케이션 = 데이터를 서비스 의미로 처리하는 마지막 단계
- NLB(L4) vs ALB(L7): 동작 계층이 라우팅 정보와 성능 특성을 가르는 점
