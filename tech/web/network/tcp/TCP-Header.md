# TCP 헤더 구조

TCP 헤더는 신뢰성, 흐름 제어, 혼잡 제어를 굴리는 데 필요한 메타데이터를 담는다. 옵션이 없으면 **20바이트(160비트)**, 옵션을 붙이면 최대 **60바이트**까지 늘어난다. 헤더 한 장만 훑어도 TCP가 어떤 기능을 갖췄는지 윤곽이 보인다. 연결 수립/종료 절차는 [[TCP-Handshake]], L4 전체 맥락은 [[Transport-Layer]].

## 왜 이런 필드들이 필요한가

회선 교환은 통신할 상대와 물리 회선을 하나 잡아두므로 중계 노드 하나만 끊겨도 통신이 죽고 회선 효율도 낮다. 이를 대체한 **패킷 교환**은 데이터를 잘게 쪼개 여러 경로로 흘려보내 노드 하나가 죽어도 전체가 유실되지 않고 회선 효율도 높다. 대신 새 문제가 생겼고, TCP 헤더 필드는 그 해법이다.

- 패킷이 사라지거나 훼손되면? → 그 조각만 재전송 (ARQ, 시퀀스, ACK 번호)
- 쪼갠 순서를 수신 측이 어떻게 아나? → 순서 번호 동봉 (시퀀스 번호)
- 수신 측 처리 속도보다 빠르게 보내면? → 받을 수 있는 양을 알려 그만큼만 (윈도우, 슬라이딩 윈도우)

## 필드별 의미

| 필드 | 크기 | 역할 |
|---|---|---|
| Source / Destination Port | 각 16비트 | 출발지, 목적지 애플리케이션 식별. IP 주소는 한 계층 아래 [[Network-Layer\|IP 헤더]]에 있어 TCP엔 포트만 |
| Sequence Number | 32비트 | 보내는 데이터의 순서 번호 |
| Acknowledgment Number | 32비트 | 수신 측이 기대하는 다음 시퀀스 번호 |
| Data Offset | 4비트 | 헤더 길이(데이터 시작 위치) |
| Reserved | 3비트 | 예약. 모두 0 |
| Flags | 9비트 | 세그먼트의 속성 (아래) |
| Window Size | 16비트 | 한 번에 받을 수 있는 양 (흐름 제어) |
| Checksum | 16비트 | 오류 검출 |
| Urgent Pointer | 16비트 | 긴급 데이터 위치 (URG=1일 때) |
| Options | 가변 (~40바이트) | 기능 확장 |

### Sequence / Acknowledgment Number

시퀀스 번호로 수신자는 쪼개진 세그먼트를 올바른 순서로 재조립한다. 최초 전송 시 **랜덤 값(ISN)으로 초기화**하고, 보내는 데이터의 **1바이트당 1씩 증가**하며, 32비트 최댓값(약 43억)을 넘으면 0으로 순환한다.

승인 번호는 **다음에 받을 데이터의 시작점**을 가리킨다. 계산 방식이 두 갈래다.

- 핸드셰이크(연결 수립/종료): 상대 시퀀스 번호 **+ 1**
- 데이터 전송: 상대 시퀀스 번호 **+ 받은 바이트 수**

예를 들어 100바이트를 시퀀스 0부터 받으면(0~99) 다음 기대 번호는 100이고, 수신 측은 ACK 100을 돌려준다. 시퀀스 범위는 `시작:끝` 형태로 끝 값은 범위에 포함되지 않으므로(반열린 구간), 그 끝 값이 곧 다음 시작점이자 승인 번호가 된다.

### Data Offset — 헤더가 어디서 끝나나

옵션 필드 길이가 가변이라 데이터가 어디서 시작되는지 표시가 필요하다. 단위는 **32비트 워드(1워드 = 4바이트)**라 필드 값에 4를 곱하면 헤더 길이가 나온다. 4비트로 0~15워드를 표현하지만, 옵션 없는 필수 헤더가 20바이트이므로 **최솟값은 5워드(20바이트), 최댓값은 15워드(60바이트)**다.

### Flags — 세그먼트의 속성

원래 6개의 제어 플래그를 쓰다가 혼잡 제어를 위해 ECN 관련 3개가 더해졌다.

| 플래그 | 의미 |
|---|---|
| URG | Urgent Pointer 필드가 유효함. 가리키는 데이터를 우선 처리 (요즘 거의 안 씀) |
| ACK | 승인 번호 필드가 유효함. 0이면 승인 번호 무시 |
| PSH | 버퍼를 기다리지 말고 데이터를 즉시 애플리케이션에 올리라는 요청 |
| RST | ESTABLISHED 상대에게 연결을 강제로 리셋 |
| SYN | 연결 수립 시 시퀀스 번호 동기화 |
| FIN | 연결 종료 요청 |

ECN(명시적 혼잡 통보)용 추가 플래그는 혼잡을 타임아웃으로 감지하던 방식 대신, 혼잡을 **명시적으로 통보**한다. `ECE`는 SYN과 함께면 ECN 사용 협상, SYN 없이 1이면 혼잡하니 윈도우를 줄이라는 신호. `CWR`은 그 신호를 받아 윈도우를 줄였다는 응답. `NS`는 ECE/CWR 은폐를 막기 위한 보호 비트. (NS 플래그는 이후 RFC 9293에서 폐기되어, 최신 표현은 제어 플래그 8개 + 예약 4비트로 보기도 한다.)

### Window Size — 흐름 제어

한 번에 받을 수 있는 데이터 양(바이트). 16비트라 기본 최대 64KB지만, 고속 대용량 환경엔 부족해 옵션의 **WSCALE**로 비트를 좌측 시프트해 상한을 키운다. 이 값으로 송신량을 조절하는 흐름 제어와 슬라이딩 윈도우 동작은 [[TCP-Flow-Error-Control|흐름 제어와 오류 제어]].

### Checksum — 오류 검출

데이터를 16비트 단위로 잘라 차례로 더하고, 자리 올림(캐리)이 넘치면 그 부분을 결과에 다시 더하는 **wrap-around** 방식으로 합을 만든 뒤 **1의 보수(비트 반전)**를 취한 값이다. 수신 측은 같은 합에 받은 체크섬을 더해 **모든 비트가 1이면 정상**, 0이 하나라도 있으면 변조로 판단한다.

### Options — 기능 확장

가변 길이(최대 40바이트)로 TCP 기능을 확장한다. 대표적으로 윈도우 상한을 키우는 **WSCALE**, 선택적 재전송([[TCP-Flow-Error-Control|Selective Repeat]])을 위한 **SACK** 등 약 30종이 있다. 옵션을 안 써도 Data Offset이 20바이트를 넘게 잡혔다면 남는 자리를 0으로 패딩해 헤더 경계를 맞춘다.

## 면접 체크포인트

- 기본 헤더 20바이트, 옵션 포함 최대 60바이트 — Data Offset이 경계를 표시
- 시퀀스 번호는 바이트 단위 증가(ISN 랜덤), 승인 번호 = 다음 기대 시작점
- 핸드셰이크의 ACK(+1) vs 데이터의 ACK(+받은 바이트) 계산 차이
- 6개 제어 플래그(URG/ACK/PSH/RST/SYN/FIN)의 의미
- 윈도우 사이즈와 흐름 제어, WSCALE로 64KB 한계를 넘는 법
- 체크섬의 1의 보수 검증 원리
