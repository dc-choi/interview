# 문제 대응의 세 가지 패턴

문제에 조치했다는 사실과 문제를 해결했다는 사실은 다르다. 국소 지표만 개선하거나 기존 이해관계를 지키고, 하나를 없애면서 더 큰 문제를 만들 수도 있다. 문제 이동, 문제 보존과 새 문제 부상이라는 세 가지 휴리스틱은 조치의 실제 결과를 시스템 경계에서 다시 보게 한다.

이 분류는 세 대응을 모두 나쁘다고 판정하는 보편 이론이 아니다. 비용과 책임을 명시한 이동, 우선순위에 따른 한시적 보존, 감당 가능한 다음 병목은 합리적인 선택일 수 있다. 선택한 비용을 숨기거나 소유자와 검토 시점을 두지 않을 때 안티패턴이 된다.

## 1. 문제 이동

한 구간을 개선하면서 비용과 실패를 다른 팀, 시스템이나 시간대로 넘기는 국소 최적화다.

- API 응답을 빠르게 하려고 작업을 큐로 넘겼지만 backlog와 중복 처리는 운영팀 몫이 됨
- DB 부하를 줄이려고 캐시를 추가했지만 무효화 실패와 오래된 응답이 사용자에게 이동
- 한 팀의 배포 승인을 단순화했지만 공통 플랫폼 팀의 수동 검증이 늘어남

사람을 탓하기보다 **측정 경계와 인센티브**를 본다. 팀 KPI만 최적화하면 시스템 전체 비용을 옮기는 행동이 합리적인 선택이 될 수 있다. 상위 책임자가 전체 리드 타임, 오류율과 고객 결과를 함께 보도록 목표를 재설계해야 한다.

큐로 부하를 흡수하거나 전문 팀에 책임을 위임하는 것처럼 의도적인 이동도 있다. 이동한 비용, 소유자, SLO와 복구 경로를 함께 정하면 설계 선택이고, 이를 숨겨 국소 지표만 좋아 보이게 하면 안티패턴이다.

## 2. 문제 보존

역할, 예산, 권한이나 정체성이 특정 문제를 전제로 만들어져 해결 자체가 손실로 느껴지는 상태다.

- 반복 장애를 처리하는 절차와 조직은 커졌지만 근본 자동화 투자는 계속 밀림
- 복잡한 승인 과정을 관리하는 역할이 생긴 뒤 승인 단계 축소에 저항
- 수동 보고서를 만드는 체계가 지표 자동화를 위협으로 받아들임

문제로 이익을 얻는 사람을 악의로 단정하면 전환이 더 어려워진다. 해결 뒤 바뀌는 역할, 잃는 권한과 필요한 역량을 함께 설계하고, 영향받는 사람을 전환 계획에 포함한다.

위험을 받아들이고 해결을 미루는 결정 자체가 항상 문제는 아니다. 보존 비용, 만료일과 재검토 조건 없이 임시 조치가 영구 구조가 되거나, 이해관계 때문에 근본 해결을 반복해서 막을 때 안티패턴이다.

## 3. 새 문제 부상

하나를 해결하면 다음 병목이 드러나거나 해결책이 새로운 부작용을 만든다. 이는 실패일 수도 있지만 시스템 개선의 자연스러운 결과일 수도 있다.

- 처리량을 높이자 DB가 아니라 외부 API rate limit이 새 병목이 됨
- 경보 수를 줄이자 알림 피로는 개선됐지만 중요한 이상도 늦게 발견됨
- 자동화를 도입하자 수동 오류는 줄었지만 잘못된 자동 결정의 영향 범위가 커짐

문제가 완전히 사라질 것이라는 기대보다 **어떤 다음 문제를 받아들일지** 선택한다. 해결 전 2차 효과와 새 병목을 가정하고, 배포 뒤 실제로 부상한 문제가 감당 가능한지 검증한다. 더 큰 위험을 만들고도 관측, 소유와 대응 계획이 없다면 안티패턴이고, 더 나은 제약으로 이동했다면 개선의 다음 단계다.

## 시스템 경계에서 해결 여부 판단하기

| 질문 | 확인할 것 |
|---|---|
| 어디가 좋아졌나 | 국소 지표와 사용자 결과를 분리 |
| 비용이 어디로 갔나 | 다른 팀, 저장소, 미래 운영과 예외 처리 |
| 누가 현 상태에 의존하나 | 역할, 예산, 권한과 전환 손실 |
| 무엇이 다음 병목이 되나 | 용량, 품질, 보안과 사람의 검토 부담 |
| 되돌릴 수 있나 | 롤백 기준, 관측 지표와 검토 시점 |

공유 다이어그램에 흐름, 대기, 책임과 피드백을 함께 표시하면 서로 다른 팀이 같은 문제 경계를 볼 수 있다. 모든 문제를 동시에 없애려 하지 말고 사용자와 시스템 결과에 가장 큰 영향을 주는 몇 개를 고른다.

## 실행 체크리스트

1. 현재 상태와 목표 결과를 시스템 전체 지표로 정의한다.
2. 조치가 비용을 옮기는 대상과 시점을 적는다.
3. 현 문제에 의존하는 역할과 전환 계획을 확인한다.
4. 예상되는 다음 병목과 허용 한도를 정한다.
5. 제한된 범위에서 실행하고 baseline과 비교한다.
6. 검토 날짜에 해결, 이동, 보존과 부상 중 실제 결과를 다시 분류한다.

## 면접 체크포인트

- 국소 최적화가 전체 시스템을 악화시킨 사례와 측정 경계를 넓힌 방법
- 기술 문제가 인센티브와 책임 구조 때문에 반복될 수 있는 이유
- 해결책의 2차 효과와 다음 병목을 사전에 검토하는 방법
- 사람을 비난하지 않고 시스템과 전환 계획을 바꾸는 접근
