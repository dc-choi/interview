# 프로그래밍에서의 인지 편향

인간의 뇌는 **빠른 판단을 위해 지름길(휴리스틱)**을 쓴다. 대부분은 유용하지만, 복잡한 엔지니어링 결정에서는 체계적 오류를 만든다. 편향을 없앨 수는 없지만 **이름을 알고 있으면** 토론에서 호명해 의사결정 품질을 높일 수 있다.

## 1. Hyperbolic Discounting — 즉시 이익 선호

**"나중에 받을 큰 이익보다 지금의 작은 이익을 선호하는 경향."**

- 테스트 작성 미루기 → 당장 구현 속도는 빠르지만, 몇 주 뒤 회귀 버그 추적에 수 배의 시간 소비
- 익숙한 키보드 단축키만 쓰고 새 단축키 학습을 미룸 → 평생 느린 타이핑
- 리팩토링을 "지금 바쁨"으로 계속 미룸 → 몇 달 뒤 손댈 수 없는 코드

**대응**
- "미래의 비용"을 **정량화**해 현재의 불편과 비교 — 테스트 15분 vs 디버깅 3시간
- 스프린트에 **기술 부채 할당 비율** 강제 (예: 20%)
- "Invest vs Improve"를 명시적 주기로 평가

## 2. IKEA Effect — 내가 만든 것 과대평가

**"직접 만든 것에 과도한 가치를 부여하는 편향."**

- 사내 도구가 비슷한 오픈소스보다 열등해도 "우리가 만들었으니까 유지"
- 몇 달 들여 만든 아키텍처에 애착이 생겨 **더 나은 대체안이 와도 인정 못함**
- NIH(Not Invented Here) 증후군의 인지적 뿌리

**대응**
- **Build vs Buy** 결정을 설계 초기에 기록 (시간이 지나면 sunk cost와 얽힘)
- 정기적인 **도구·라이브러리 재평가** 세션
- 외부 리뷰어·컨설턴트에게 "이거 왜 자체 제작해?" 질문받기

## 3. Premature Optimization — 시기상조 최적화

**"실제 병목을 측정하기 전에 코드를 최적화하는 경향."**

- 쓰이지도 않을 캐시 레이어를 미리 넣기
- 전혀 병목이 아닌 함수를 인라인화·메모이제이션
- 엔진이 고장난 차에 공기역학 스포일러 다는 꼴

Donald Knuth의 명언: *"Premature optimization is the root of all evil."*

**대응**
- **측정 먼저** — 프로파일러·APM으로 실제 병목 확인 후에만 최적화
- "Make it work → Make it right → Make it fast" 순서 준수
- "이 최적화가 사용자 지연을 몇 ms 줄이는가?"를 질문

## 4. Planning Fallacy — 일정 과소추정

**"작업에 걸리는 시간을 일관되게 짧게 추정하는 경향."**

- "하루면 끝날 줄" → 실제 3일
- **90-90 법칙**: 코드의 처음 90%는 개발 시간의 90%를, **나머지 10%가 또 90%를** 차지 → 체감 90%지점에서 실제는 반만 끝남
- 낙관 편향과 결합하면 **프로젝트 전체**가 수 배 지연

**대응**
- **과거 유사 작업의 실제 소요 시간**을 기준으로 (reference class forecasting)
- 추정에 **버퍼**(예: 50% 추가) 자동 포함
- 큰 일정은 **여러 명이 각자 추정 후 평균** (Wideband Delphi)
- 일단 작은 범위로 프로토타입 → 실측 후 재추정

## 5. Recency Bias — 최근 경험 과대평가

**"최근에 겪은 문제·패턴을 다른 상황에도 쉽게 투영하는 경향."**

- 방금 쓴 디자인 패턴을 모든 문제에 적용하려 함
- 최근 장애 원인을 다음 설계 논의에서 과도하게 방어
- 새로 배운 프레임워크를 맞지 않는 도메인에 도입

**대응**
- **다양한 맥락**에서 같은 문제를 다뤄본 동료의 의견 수렴
- 아키텍처 결정을 **내리기 전 며칠 숙고** — 뜨거운 직후 결정 금지
- "이게 진짜로 지금 문제에 맞는가?"를 문서로 근거 작성

## 추가로 자주 등장하는 편향

### Confirmation Bias — 확증 편향
원하는 결론에 맞는 증거만 수집. 장애 분석에서 초기 가설에 맞는 로그만 읽음 → 진짜 원인 놓침.

**대응**: "이 가설을 **반증**할 증거는?"을 의식적으로 질문. PR 리뷰에서 "문제없는 것 같다"가 아닌 "무엇이 문제일 수 있을까"로 접근.

### Dunning-Kruger Effect
지식이 얕을수록 자신감이 높고, 깊을수록 조심스러워지는 경향. 주니어가 큰 결정을 과감히 내리고 시니어가 더 많이 묻는 현상의 인지적 뿌리.

**대응**: "내가 모르는 것은 무엇일까?"를 체크리스트화. 고위 엔지니어일수록 **의문의 표명**을 미덕으로 모델링.

### Sunk Cost Fallacy — 매몰 비용 오류
이미 투입한 시간·비용이 아까워 실패하는 방향을 계속 추진. "이만큼 했는데 지금 접을 순 없잖아."

**대응**: "만약 지금 **처음부터** 시작한다면 이 방식을 선택할까?"를 매 의사결정에서 질문.

### Availability Heuristic — 가용성 휴리스틱
쉽게 떠오르는 예시에 가중치 부여. 최근 뉴스의 장애 유형·유명 회사 사례가 과대평가됨.

**대응**: 실제 확률·빈도 데이터 수집. 일화 대신 통계로 근거.

## 편향을 조직 차원에서 줄이는 장치

- **의사결정 기록**(ADR·테크스펙) — 당시의 근거를 남겨 사후 확증 편향 방지
- **사전 부검(Pre-mortem)** — "이 프로젝트가 실패했다고 가정하고 원인을 나열" → 낙관 편향 상쇄
- **디바이스 어드보킷(Devil's Advocate)** 지정 — 반대 입장을 의도적으로 맡기기
- **체크리스트** — 편향이 개입할 틈을 없앰 (항공·의료에서 검증된 기법)
- **다양한 경력의 팀 구성** — 공통 편향을 줄이기 가장 효과적

## 면접 체크포인트

- 본인이 **과거에 겪었던 편향 경험**과 극복 방법
- "측정 없이 최적화" 같은 **Premature Optimization의 구체 사례**
- **90-90 법칙**을 아는가·일정 추정에서 어떻게 반영하는가
- **사전 부검(Pre-mortem)** 기법의 의미
- 주니어의 과신 vs 시니어의 조심 — **Dunning-Kruger**와 연결해 설명
- 편향을 **조직 차원 장치**(ADR·테크스펙·체크리스트)로 줄이는 방법
