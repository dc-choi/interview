# 좋은 테크스펙 구조

## 1. 배경, 문제 정의
- 현재 상황과 한계
- 왜 지금 이 문제를 푸는가 (시급성)
- 해결하지 않으면 무엇이 안 됨

## 2. 목표, 비목표
- **목표 (Goals)**: 이 스펙이 달성할 것
- **비목표 (Non-goals)**: 의도적으로 제외 — 스펙 범위 경계 명시

비목표 명시가 특히 중요. "이건 이번 스펙에서 안 함"을 분명히 하면 리뷰 집중도 올라감.

## 3. 제안 (Proposal)
구체적 설계:
- 데이터 모델
- API, 인터페이스
- 컴포넌트 구조
- 주요 플로우 (시퀀스 다이어그램)

## 4. 대안 검토 (Considered Alternatives)
**선택하지 않은 안**을 명시.
- A안, B안, C안 비교
- 각각의 장단점
- 왜 현재 안을 택했는지

이 섹션이 없으면 리뷰어가 "왜 X 안 썼어?" 매번 같은 질문. 명시하면 상위 대안은 리뷰 대상에서 빠짐.

## 5. 트레이드오프
- 이 설계가 포기하는 것
- 잠재적 리스크, 완화 방법
- 장기적 영향 (기술 부채 여부)

## 6. 마이그레이션, 롤아웃
- 기존 시스템에서 어떻게 전환
- 점진적 배포 전략
- 롤백 계획

## 7. 측정, 성공 지표 (Measuring Impact)
- 구현 후 어떻게 성공을 판단할지
- 모니터링 지표
- SLO, SLA 영향
- **Goals와 1:1 매핑** — 각 목표별 측정 가능한 수치 (예: "첫 2개월 내 DAU 1만", "첫 응답 p95 200ms 이하")
- 데이터 사이언스, PM과 협업해 정의 — 엔지니어 단독으로 정하지 않기
- 우선순위 논쟁 해결에도 활용 — "한정된 자원, 어떤 기능이 더 중요한가?"

## 8. 보안, 프라이버시, 리스크 (Security, Privacy, Risks)
외부 노출 프로젝트는 **악용 시나리오**를 명시. 리스크를 적는 게 불안해 보여도, 리뷰어가 **질문, 완화책**을 제시할 기회를 줌 → 결과적으로 더 견고한 기능.

## 9. 마일스톤 (Milestones)
작업을 주요 달성 지점으로 나누고 **예상 날짜** 부여.
- 예: "Dog API 연동 완료: 10/14", "QA 완료: 10/21"
- **지연되면 즉시 업데이트** — 이 섹션이 프로젝트 상태의 Source of Truth
- 큰 팀, 조직에서 "언제 끝나?" 질문에 대한 공식 답

## 10. 열린 질문 (Open Questions)
**미결 사항을 명시**. "이 부분은 아직 모르겠다"를 숨기지 말고 드러내면 리뷰어가 답을 줄 수 있음.
- **특정 전문가 호출** — "@데이터분석팀, 이 트윗 주기가 최적인지 확인해줄 수 있나요?"
- 미해결 질문을 눈에 띄게 두어 리뷰어가 답할 기회를 명시적으로 제공
