# 테크스펙 리뷰 프로세스와 운영

## 리뷰 프로세스

### 초안 단계
- "최소 써지는 만큼" 쓰고 **즉시 공유**
- 완벽 추구 금지 — 리뷰가 채워줌
- `status: draft` 명시

### 분배 순서 — 점진 확장

피드백 품질은 **스펙의 완성도**에 비례. 완성도 낮은 상태로 넓은 청중에 뿌리면 같은 기초 지적을 수십 번 받는다. 순차 확장이 효율적.

1. **팀원 1명 사전 리뷰** — 명백한 오류, 누락 포착
2. **팀 전체** — 기술 정합성, 다른 컴포넌트 영향
3. **직접 이해관계자(Stakeholder)** — PM, 디자이너, 의존 팀 리드
4. **전체 관심자** — 공용 메일링리스트, Slack 채널

각 라운드마다 스펙이 다듬어져 다음 라운드 리뷰가 **더 유용한 피드백**에 집중할 수 있다.

### 리뷰 세션
- 주요 스테이크홀더와 **30분~1시간** 회의 or 비동기 댓글
- "의견 있으면 댓글 주세요"보다 **대면 리뷰가 밀도 높음**
- 팀이 크면 RFC처럼 **2주 오픈 → 반대 없으면 진행**

### 피드백 처리 원칙

- **모든 댓글에 응답** — 본인 또는 다른 리뷰어가. 답 없는 댓글은 "묻혔다"는 인상
- **해결된 이슈는 명시적으로 표기** — 댓글 스레드에 resolution, 또는 본문에 반영 후 감사
- **피드백을 닫을 시점 판단** — 더 이상 유용한 코멘트가 없으면 "close to feedback" 선언. 비핵심 의견은 **v2로 이관**
- **주요 변경은 날짜 기록** — 살아있는 문서지만 독자가 "지금 이 스펙이 최신인지" 판단할 수 있어야 함

### 반복, 승인
- 피드백 반영 후 `status: ready for review`
- 주요 결정권자 명시적 승인 (`LGTM` 댓글 등)
- 승인되면 `status: approved` → 구현 착수

### 종료
- 구현 완료 후 `status: implemented` 또는 `shipped`
- 스펙은 **구현과 함께 수명 종료** — 계속 최신 상태 유지할 필요 X
- 지속 유지 필요한 내용은 **README, IDL, 테스트 코드**로 옮김

테크스펙이 "계속 업데이트되는 살아있는 문서"가 되면 의도 왜곡. **프로젝트 단위**로 완결되는 게 맞음.

## ADR(Architecture Decision Record)과의 차이

- **테크스펙**: 구현 전 "무엇을 어떻게 할지" 토론
- **ADR**: 이미 내린 결정의 "왜"를 간단히 기록

둘은 보완 관계. 큰 설계 결정은 테크스펙으로 논의 → 핵심 결정만 ADR로 요약해 영구 보관.

## 흔한 실수

- **스펙 없이 큰 기능 착수** → 중간에 방향 틀어서 1주일 날림
- **스펙을 문서화로만 취급** → 리뷰 요청 없이 "올려놨어요"
- **모든 작업에 스펙 강제** → 의식화되고 업무 속도 저하
- **스펙 쓰고 피드백 안 반영** → 다음부터 리뷰 참여 안 함
- **완벽 추구로 초안 못 냄** → 타이밍 놓쳐 이미 구현됨

## 면접 체크포인트

- 테크스펙과 "문서화"의 차이
- 테크스펙을 쓰는 상황과 쓰지 않는 상황
- 비목표(Non-goals) 명시가 중요한 이유
- 대안 검토 섹션의 가치
- 테크스펙이 구현 완료와 함께 수명 종료되는 이유
- ADR과의 차이
