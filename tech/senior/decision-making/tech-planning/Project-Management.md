# 프로젝트 관리

프로젝트를 리딩하는 역할은 기획서, 설계 문서를 쓰는 데서 끝나지 않는다 — 디자인, 개발, 유관 부서를 조율해 시작부터 종료까지 끌고 가는 관리가 따라온다. 시작 전(킥오프), 진행 중(상태 관리), 종료 후(회고)의 3구간으로 나뉜다. 모든 방법론을 모든 과제에 적용할 필요는 없다 — 알고 생략하는 것과 모르고 일하는 것은 다르며, 프로젝트 성격에 맞게 선택한다.

## 시작 전: 킥오프 미팅

목적 세 가지 — ① 프로젝트의 필요성과 목표 합의 ② 영향 범위(scope) 규정 ③ 참여자 간 라포 형성. 초기에 범위와 목표를 못박지 않으면 프로젝트는 끝없이 확장된다.

아젠다:

| 항목 | 내용 |
|---|---|
| 개요와 배경 | 왜 지금 이 프로젝트인가 |
| 목표와 성과지표 | SMART 원칙(구체적, 측정 가능, 달성 가능, 관련성, 기한)으로 구체화 |
| 주요 과업 (Scope) | 포함 범위와 제외 범위를 함께 명시 |
| 타임라인 | WBS에서 도출 |
| 운영 방식 | 스크럼 주기, 협업 툴, 의사결정자 |
| 프로젝트 팀 | 과업별 담당자와 역할 |

## WBS와 마일스톤

**WBS (Work Breakdown Structure)** — 목표를 산출물 단위까지 계층 분해하는 구조:

목표(Goal) → 결과물(Outcome) → 주요 업무(Task) → 세부 업무(Activity) → 산출물(Output)

타임라인은 WBS에서 도출한다 — 작업을 다 쪼갠 뒤에야 기간 추정이 의미를 가진다.

**마일스톤**은 모든 작업이 아니라 중요한 지점만 찍는다: 단계가 끝나는 시점, 주요 이벤트와 결과물, 목표와 직결된 주요 결과.

## 추정과 약속을 분리한다

**추정(estimate)**은 현재 정보로 계산한 예상 범위이고, **약속(commitment)**은 조직이 책임지고 맞추기로 합의한 목표다. 둘을 같은 숫자로 취급하면 불확실성이 숨겨지고, 일정이 흔들릴 때 엔지니어의 추정 정확도만 문제 삼게 된다.

- 추정에는 전제, 범위, 불확실성과 신뢰 수준을 함께 기록한다.
- 출시일이 고정돼 있다면 범위, 인력, 품질, 위험 가운데 무엇을 조정할지 명시적으로 협상한다.
- 새 정보가 생기면 조기에 다시 추정하고, 기존 수치와 달라진 이유를 공유한다.
- 목표 날짜는 의사결정의 제약 조건으로 사용하되, 근거 없는 낙관을 강요하는 수단으로 쓰지 않는다.

## 진행 중: Project Status Report

주기적 상태 보고는 리스크가 심각한 지연이 되기 전에 감지하는 장치다. 구성:

| 항목 | 내용 |
|---|---|
| 프로젝트 이름과 상태 | 타임라인 대비 정상, 지연 여부를 한눈에 |
| 전체 진행 상황 | 이전 리포트 대비 진척 |
| 주요 성과 | 완료한 구체적 과업 |
| 주요 이슈 | 문제와 해결책을 함께 공유 |

지연 상태만 보고하지 않고 원인과 다음 액션을 붙이는 것이 핵심이다.

### 진행 중의 두 가지 함정

- **Executor 전환 함정** — 일정 압박 속에 스펙이 깎여 나갈 때마다 어떻게든 완성하는 것 자체가 목표가 되면(실행가 관점), 이 기능이 사용자에게 정말 가치 있는가라는 본질 질문(기획자 관점)을 놓친다. 스펙이 변경될 때마다 가치 질문을 다시 던지는 것이 리딩하는 사람의 역할이다 — 결과물이 축소되는 것보다 그 과정에서 관점을 잃는 것이 더 큰 손실.
- **이상 신호 재확인** — 회의에서 상대의 반응이 미심쩍으면(필수 기능 얘기에 갸웃하는 표정 등) 상대의 경력이나 직급을 신뢰하는 것으로 넘기지 말고 그 자리에서 한 번 더 확인한다. 필수 스펙이 전달되지 않은 채 진행되면 배포 직전의 위기로 돌아온다.

## 리딩에 필요한 소프트 스킬

- **문제 해결 (전략적 사고)** — 문제 정의 → 원인 파악 → 아이디어 생성 → 최선책 선택 → 실행의 5단계.
- **커뮤니케이션** — 결론만 던지지 않고 배경과 맥락을 먼저 제시해 공감을 만든 뒤 요청한다.
- **협업** — 타임라인 합의, 히스토리 기록, 상대 질문의 의도 파악.

## 종료 후: 회고

포스트모템(Postmortem)이라고도 부른다 — 전 과정을 되돌아보며 성공과 실패 요인을 분석하는 자리.

1. **진행 내용 요약** — 초기 목표 대비 실제 진행 비교.
2. **KPT 프레임워크** — Keep(계속할 것), Problem(문제였던 것), Try(다음에 시도할 해결책).
3. **목표 달성 확인** — 킥오프에서 합의한 성과지표 대비 실측을 확인하고 유관 부서와 공유한다. 의외의 유의미한 지표가 발견되면 그것도 함께 공유.

회고는 개인 학습이 아니라 팀 역량과 조직 문화에 누적되는 자산이다.

### 회고 운영 3단계

1. **아이스브레이킹** — 형식적 질문 대신 프로젝트에서 느낀 점 중심의 자유 대화. 주최자가 먼저 솔직한 감정(부담감, 감사)을 꺼내야 분위기가 열린다.
2. **이슈별 토론** — 사전에 기록해 온 항목(좋았던 점, 아쉬웠던 점, 시도할 점, 어려웠던 점)을 그룹핑해 칭찬과 개선점을 함께 도출한다.
3. **로드맵 공유** — 서비스의 다음 방향을 공유해 팀원의 시각을 단순 실행에서 전략적 이해로 끌어올린다.

### 회고가 드러내는 전형적 문제

- **리더 경유 커뮤니케이션의 누락** — 스펙을 가장 잘 아는 사람은 리더가 아니라 실무자인데, 소통이 리더를 경유하면 정보가 샌다. 개선은 실무자 간 직접 소통 전환 + 위키에 모듈별 담당자 정리.
- **상대가 모르는 구조적 부담** — 한쪽의 부담(다중 API 버전 동시 운영 등)을 다른 쪽이 인지조차 못 하는 경우가 흔하다. 회고에서 드러나면 구버전 페이드아웃 같은 부채 상환 계획으로 이어질 수 있다.
- **비대면 조직의 단절** — 재택은 집중에 좋지만 팀 간 단절을 만든다. 의도적인 연결의 시간(대면 회고)이 신뢰와 생산성을 함께 올린다.

## 면접 체크포인트

- 프로젝트 리딩 경험 질문에는 킥오프(범위 합의) → 상태 관리(리스크 조기 감지) → 회고(KPT, 지표 확인)의 전주기 구조로 답한다.
- 일정 추정 질문에는 WBS를 근거로 답하고 추정과 약속을 구분한다. 작업 분해 없이 나온 기간은 추정이 아니라 희망이다.
- 지연 보고는 상태 + 원인 + 다음 액션 세트로 말한다.
- 회고 경험 질문에는 발견한 문제와 그것이 만든 실제 개선 액션(담당자 문서화, 기술 부채 상환 계획)까지 한 세트로 답한다.
