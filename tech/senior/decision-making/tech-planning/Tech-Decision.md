# 기술 의사결정

기술적 선택이 비즈니스에 미치는 영향을 고려하여 근거 기반으로 결정하는 역량이다. "왜 이 기술을 선택했는가?"에 트레이드오프를 포함한 답변을 할 수 있어야 한다.

## 의사결정 프레임워크

1. **문제 정의** — 현재 상태와 목표 상태의 차이를 명확히 한다
2. **선택지 나열** — 가능한 대안을 최소 2개 이상 도출한다
3. **트레이드오프 분석** — 각 선택지의 장단점, 리스크, 비용을 비교한다
4. **결정 + 근거 기록** — 선택한 이유와 배제한 이유를 문서화한다
5. **검증 기준 정의** — 결정이 올바른지 판단할 지표를 사전에 정한다

## Design It Twice

중요한 선택은 첫 번째로 떠오른 설계를 곧바로 구현하지 않고, 같은 제약을 만족하는 **서로 다른 설계 두 개 이상**을 짧게 만든 뒤 비교한다.

- 같은 문제와 성공 기준을 두고 경계, 데이터 흐름이나 운영 방식이 다른 대안을 만든다.
- 인터페이스 복잡도, 실패 모드, 마이그레이션, 관측과 되돌리기 비용을 같은 기준으로 비교한다.
- 다이어그램, 얇은 prototype이나 AI 브레인스토밍으로 탐색 비용을 줄일 수 있지만, 생성된 선택지를 실제 근거로 검증한다.
- 선택 뒤에는 채택 이유뿐 아니라 버린 대안과 다시 검토할 조건을 남긴다.

이는 시스템을 두 번 완성하라는 뜻이 아니다. 가역적인 결정은 탐색을 짧게 하고, 되돌리기 어렵고 영향 범위가 큰 결정일수록 비교에 더 투자한다.

## 결정의 전달과 진화

아키텍처 결정은 결정권자의 머릿속에서 끝나지 않는다. 기술 팀에는 구현 제약과 검증 방법을, 비기술 이해관계자에게는 비용, 위험과 전달 속도에 미치는 영향을 설명해야 실제로 같은 방향으로 움직일 수 있다.

- ADR에는 맥락, 검토한 선택지, 결정, 결과와 재검토 조건을 기록한다.
- 결정과 함께 품질 속성을 측정할 [[Architecture-Fitness-Functions|fitness function]]을 정해 의도가 코드와 운영에서 유지되는지 확인한다.
- 설계자는 코드와 팀의 피드백에서 떨어지지 않고, 새 정보가 생기면 기록과 결정을 함께 갱신한다.

## 사례 1: 계정 모델 전환

**문제:**
- 공유 계정 구조가 구조적 이탈을 유발 (여러 교리교사가 하나의 계정 공유 → 혼란 → 사용 중단)
- 개인 식별 불가로 권한 분리, 활동 추적, IDOR 방지가 구조적으로 불가능
- 2단계 기능(행사 카드, 사제 보고, 구독)의 공통 선행 조건

**선택지:**
1. 공유 계정 유지 + 닉네임 구분 — 최소 변경, 근본 해결 안 됨
2. 개인 계정 + 조직 계층 (Parish→Church→Organization) — 대규모 변경, 근본 해결

**결정: 선택지 2**
- 근거: IDOR 취약점의 근본 원인 해결, 향후 기능의 기반, 이탈 원인(A유형) 제거
- 리스크: 마이그레이션 실패 시 데이터 손실 → 완화: 백업 + 1곳 우선 실행 + 수동 검증
- 검증: WAU 12명 달성, A유형 본당 재활성화 1곳 이상, 합류 플로우 완료율 70%+

## 사례 2: 포지셔닝 전환 (Pivot)

**문제:**
- 주일학교 중고등부만 타겟 → 결제 주체 검증 어려움
- 시장이 좁아 성장에 한계

**당시 결정:** "주일학교 중고등부" → "모든 가톨릭 모임"으로 타겟 확대를 검토

**현재 재평가 (2026-07-15):** PMF 타깃은 주일학교 중고등부로 유지한다. 전 그룹 확장은 회의록 출시와 90일 사업 검증을 거친 뒤 2026-11-30 최종 판정 이후 별도로 결정한다.

**영향 범위:** 확대를 확정할 경우에만 용어(학생→멤버), 브랜딩, 마케팅 채널, UI 라이팅을 변경한다.
**근거:** 타깃 확대의 장기 가설보다 현재 사용자군의 문제와 유료 운영 지원 검증을 우선한다.

## 사례 3: 보안 취약점 대응 (IDOR)

**문제:** Account-리소스 소유권이 1:N 직결 구조라 수평 권한 상승 가능

**단기 대응:** BulkCreateStudents 핫픽스 (즉시)
**구조적 해결:** 공통 소유권 검증 유틸(`assertGroupIdsOwnership`, `assertStudentIdsOwnership`) + 회귀 테스트

**결정 기준:** 보안 취약점은 단기 패치와 구조적 해결을 병행. 패치만으로 끝내지 않고 같은 유형의 취약점이 재발하지 않도록 공통 유틸을 만들었다.

## 결정을 기록하는 이유

- 미래의 자신과 팀원에게 "왜 이렇게 했는지" 설명
- 상황이 변했을 때 재검토 근거 제공
- 같은 실수를 반복하지 않기 위한 학습 자산

## 면접 포인트

Q. 중요한 기술적 의사결정 경험은?
- 계정 모델 전환: 공유→개인 계정 + 조직 계층. IDOR 구조적 해결 + 2단계 기반
- 포지셔닝 피벗: 범용 확장 가설을 세웠지만, 현재는 중고등부 PMF 검증을 우선하고 판정일 이후 재평가
- 보안 대응: 핫픽스(단기) + 공통 유틸(구조적) 병행

Q. 의사결정을 어떻게 문서화하는가?
- PRD 형식: 배경/문제 → 목표/성공 기준 → 범위 → 요구사항 → 리스크 → 롤아웃/검증
- 결정 근거, 배제 이유, 검증 기준을 명시
