# 테크스펙 (Tech Spec) 작성

테크스펙은 **문서가 아니라 "토론의 장"**. 애자일 선언을 오해해 "문서는 비효율"로 치부하는 조직이 많지만, 현대적 테크스펙은 구현 전에 **값싸게 피드백을 받는 협업 도구**. 애자일의 "작동하는 소프트웨어가 포괄적 문서보다 중요"는 **일방적·고정된 문서**를 비판한 것이지, 대화를 배제한 게 아님.

## 테크스펙이 막아주는 3대 시나리오

"문서 쓸 시간에 코드 한 줄 더 짜"라는 반론이 흔하지만, 실제 장애·낭비 사례는 스펙 부재에서 시작되는 경우가 많다.

1. **배포 직전에 발견하는 외부 의존성 변경** — 몇 주 만든 기능이 다른 팀이 이미 deprecated한 서비스에 의존. 초기 스펙 공유만 있어도 **리뷰 단계에서 포착**됐을 문제
2. **오래된 스펙으로 잘못된 구현** — 클라이언트 엔지니어가 구버전 스펙 기준으로 몇 주 일한 뒤에 서버 통합 시 전부 무효. **버전·날짜 표기**만 있어도 방지
3. **Scope Creep** — 기능 발표 후 요구사항이 끝없이 붙어 출시 불가. **Non-Goals 명시**로 경계 방어

이 3가지 실패는 모두 **구현 전 한두 시간**의 스펙 리뷰로 대체할 수 있는 수 주 분량의 재작업을 동반한다. 테크스펙의 ROI가 가장 잘 드러나는 지점.

## 테크스펙의 3대 이점

### 1. (거의) 버그 없는 릴리스

엔드포인트 이름·에러 코드 같은 **저수준 세부**까지 폭넓은 청중에 노출되면, 버그·회귀가 **사전에** 발견될 확률이 급격히 높아진다. 모든 버그를 없앨 수는 없지만 "여러 눈"이 대부분을 걸러준다.

### 2. 구현 중·후 문서화

- 구현 중: "정확히 어떤 일을 해야 하는가"를 명확히
- 구현 후: 신입·전입자가 **기능의 내부 동작·트레이드오프**를 빠르게 파악
- 문서는 **접근 가능할 때만** 유용 — 공용 저장소·메일링리스트·위키로 **누구나 열람·기여**할 수 있어야 함. 주니어가 자신감 있게 공헌할 기반

### 3. Fast Iteration

설계·구현 합의가 스펙 단계에서 이뤄지면 이후 논쟁이 줄어든다. 마감 직전 크런치에서 **"왜 이렇게 했지?"** 를 다시 싸울 시간을 안 쓴다. 출시 후에도 스펙은 **신뢰할 수 있는 참조 자료**로 남아 의사결정 근거·범위·타 시스템 통합 지점을 제공.

## Purpose-First — 쓰기 전에 목적을 정의

Pareto 원칙(20% 투입이 80% 결과)은 스펙 작성에도 적용된다. **"이 스펙으로 무엇을 달성하고 싶은가?"** 를 먼저 정하지 않으면 산만한 문서가 되어 읽히지도, 쓸모 있지도 않다.

대표 목적 예:
- **팀 내 구현 방법 공유** → 저수준 세부(API·에러 코드)까지
- **이해관계자 설득** → 배경·목표·영향 중심, 저수준 최소
- **아키텍처 대안 검토** → Plan 섹션에 여러 안 비교
- **타 팀 협업 인터페이스 확정** → Contract·스키마 집중

목적이 **기술적 상세도·청중·분량**을 자동으로 결정한다.

## 라이프사이클 — High-level → Low-level 전환

프로젝트 단계에 따라 스펙의 성격이 바뀐다. **하나의 스펙에 평생을 걸지 않는다.**

| 단계 | 스펙 성격 | 목적 |
|---|---|---|
| 기획 초기 | **High-level** | 이해관계자 buy-in, 방향성 합의 |
| Buy-in 이후 | **Low-level** | 엔지니어 실행용 — API·에러·분석 이벤트까지 |
| 구현 중 | 살아있는 문서 | 변경 사항 날짜 기록 |
| 구현 후 | 동결 | 참조용, README·ADR로 장기 보존 요약 이관 |

## 문서 vs 토론 공간

| 전통적 "문서" | 현대적 테크스펙 |
|---|---|
| 일방향 전달 | 양방향 피드백 |
| 완성 후 배포 | 초안 단계부터 공유 |
| 수정 비용 큼 (워드 파일) | 낮음 (Notion·Google Docs) |
| 읽히지 않음 | 댓글·인라인 피드백 활발 |
| 정적 | 구현 전까지 살아 있음 |

Notion·Google Docs·Slack 같은 현대 도구가 **실시간 양방향 협업**을 가능하게 함. 애자일 선언이 2001년일 때와 맥락이 다름.

## 테크스펙의 목적

### 구현 전 피드백 수집
영화 스토리보드·만화 콘티와 같다. **그림을 완성하기 전에** 방향·구성·캐스팅에 대한 의견을 받아 잘못된 방향으로 가지 않게.

코드도 마찬가지. 4주 구현한 뒤 "이 방향 아닌데요" 듣는 것보다, 스펙 단계에서 1시간 리뷰로 돌아가는 게 압도적으로 쌈.

### 여러 관점의 합산
혼자 생각한 설계는 자기 경험의 한계를 넘지 못한다. 다른 엔지니어·PM·디자이너 관점이 합산되면:
- 놓친 엣지 케이스 발견
- 더 나은 대안 제시
- 팀 전체가 설계 맥락 이해

### 의사결정 히스토리
"왜 이렇게 설계했지?"를 6개월 뒤에 다시 답할 수 있는 근거. 특히 **하지 않기로 한 선택**에 대한 기록이 값짐 (돌아오는 질문 방지).

## 언제 쓰는가 / 쓰지 않는가

### 쓰는 경우
- **여러 명이 협업**하는 프로젝트
- **창의적 판단**이 필요한 문제 (단순 구현 아닌 설계)
- 다양한 관점으로부터 이득 볼 수 있는 문제
- 팀 경계를 넘는 영향 (인프라·보안·데이터 모델 변경)
- **되돌리기 어려운** 선택 ([[One-Way-vs-Two-Way-Door]])

### 쓰지 않는 경우
- 자명한 버그 수정
- 단순·반복적인 작업
- 혼자 판단 가능한 소규모 변경
- 이미 패턴이 정해진 유사 작업

**작게·자주·필요할 때만** 써야 스펙 작성 자체가 의식(儀式)으로 변질되지 않음.

## 좋은 테크스펙 구조

### 1. 배경·문제 정의
- 현재 상황과 한계
- 왜 지금 이 문제를 푸는가 (시급성)
- 해결하지 않으면 무엇이 안 됨

### 2. 목표·비목표
- **목표 (Goals)**: 이 스펙이 달성할 것
- **비목표 (Non-goals)**: 의도적으로 제외 — 스펙 범위 경계 명시

비목표 명시가 특히 중요. "이건 이번 스펙에서 안 함"을 분명히 하면 리뷰 집중도 올라감.

### 3. 제안 (Proposal)
구체적 설계:
- 데이터 모델
- API·인터페이스
- 컴포넌트 구조
- 주요 플로우 (시퀀스 다이어그램)

### 4. 대안 검토 (Considered Alternatives)
**선택하지 않은 안**을 명시.
- A안·B안·C안 비교
- 각각의 장단점
- 왜 현재 안을 택했는지

이 섹션이 없으면 리뷰어가 "왜 X 안 썼어?" 매번 같은 질문. 명시하면 상위 대안은 리뷰 대상에서 빠짐.

### 5. 트레이드오프
- 이 설계가 포기하는 것
- 잠재적 리스크·완화 방법
- 장기적 영향 (기술 부채 여부)

### 6. 마이그레이션·롤아웃
- 기존 시스템에서 어떻게 전환
- 점진적 배포 전략
- 롤백 계획

### 7. 측정·성공 지표 (Measuring Impact)
- 구현 후 어떻게 성공을 판단할지
- 모니터링 지표
- SLO·SLA 영향
- **Goals와 1:1 매핑** — 각 목표별 측정 가능한 수치 (예: "첫 2개월 내 DAU 1만", "첫 응답 p95 200ms 이하")
- 데이터 사이언스·PM과 협업해 정의 — 엔지니어 단독으로 정하지 않기
- 우선순위 논쟁 해결에도 활용 — "한정된 자원, 어떤 기능이 더 중요한가?"

### 8. 보안·프라이버시·리스크 (Security, Privacy, Risks)
외부 노출 프로젝트는 **악용 시나리오**를 명시. 리스크를 적는 게 불안해 보여도, 리뷰어가 **질문·완화책**을 제시할 기회를 줌 → 결과적으로 더 견고한 기능.

### 9. 마일스톤 (Milestones)
작업을 주요 달성 지점으로 나누고 **예상 날짜** 부여.
- 예: "Dog API 연동 완료: 10/14", "QA 완료: 10/21"
- **지연되면 즉시 업데이트** — 이 섹션이 프로젝트 상태의 Source of Truth
- 큰 팀·조직에서 "언제 끝나?" 질문에 대한 공식 답

### 10. 열린 질문 (Open Questions)
**미결 사항을 명시**. "이 부분은 아직 모르겠다"를 숨기지 말고 드러내면 리뷰어가 답을 줄 수 있음.
- **특정 전문가 호출** — "@데이터분석팀, 이 트윗 주기가 최적인지 확인해줄 수 있나요?"
- 미해결 질문을 눈에 띄게 두어 리뷰어가 답할 기회를 명시적으로 제공

## 리뷰 프로세스

### 초안 단계
- "최소 써지는 만큼" 쓰고 **즉시 공유**
- 완벽 추구 금지 — 리뷰가 채워줌
- `status: draft` 명시

### 분배 순서 — 점진 확장

피드백 품질은 **스펙의 완성도**에 비례. 완성도 낮은 상태로 넓은 청중에 뿌리면 같은 기초 지적을 수십 번 받는다. 순차 확장이 효율적.

1. **팀원 1명 사전 리뷰** — 명백한 오류·누락 포착
2. **팀 전체** — 기술 정합성·다른 컴포넌트 영향
3. **직접 이해관계자(Stakeholder)** — PM·디자이너·의존 팀 리드
4. **전체 관심자** — 공용 메일링리스트·Slack 채널

각 라운드마다 스펙이 다듬어져 다음 라운드 리뷰가 **더 유용한 피드백**에 집중할 수 있다.

### 리뷰 세션
- 주요 스테이크홀더와 **30분~1시간** 회의 or 비동기 댓글
- "의견 있으면 댓글 주세요"보다 **대면 리뷰가 밀도 높음**
- 팀이 크면 RFC처럼 **2주 오픈 → 반대 없으면 진행**

### 피드백 처리 원칙

- **모든 댓글에 응답** — 본인 또는 다른 리뷰어가. 답 없는 댓글은 "묻혔다"는 인상
- **해결된 이슈는 명시적으로 표기** — 댓글 스레드에 resolution, 또는 본문에 반영 후 감사
- **피드백을 닫을 시점 판단** — 더 이상 유용한 코멘트가 없으면 "close to feedback" 선언. 비핵심 의견은 **v2로 이관**
- **주요 변경은 날짜 기록** — 살아있는 문서지만 독자가 "지금 이 스펙이 최신인지" 판단할 수 있어야 함

### 반복·승인
- 피드백 반영 후 `status: ready for review`
- 주요 결정권자 명시적 승인 (`LGTM` 댓글 등)
- 승인되면 `status: approved` → 구현 착수

### 종료
- 구현 완료 후 `status: implemented` 또는 `shipped`
- 스펙은 **구현과 함께 수명 종료** — 계속 최신 상태 유지할 필요 X
- 지속 유지 필요한 내용은 **README·IDL·테스트 코드**로 옮김

테크스펙이 "계속 업데이트되는 살아있는 문서"가 되면 의도 왜곡. **프로젝트 단위**로 완결되는 게 맞음.

## ADR(Architecture Decision Record)과의 차이

- **테크스펙**: 구현 전 "무엇을 어떻게 할지" 토론
- **ADR**: 이미 내린 결정의 "왜"를 간단히 기록

둘은 보완 관계. 큰 설계 결정은 테크스펙으로 논의 → 핵심 결정만 ADR로 요약해 영구 보관.

## 흔한 실수

- **스펙 없이 큰 기능 착수** → 중간에 방향 틀어서 1주일 날림
- **스펙을 문서화로만 취급** → 리뷰 요청 없이 "올려놨어요"
- **모든 작업에 스펙 강제** → 의식화되고 업무 속도 저하
- **스펙 쓰고 피드백 안 반영** → 다음부터 리뷰 참여 안 함
- **완벽 추구로 초안 못 냄** → 타이밍 놓쳐 이미 구현됨

## 면접 체크포인트

- 테크스펙과 "문서화"의 차이
- 테크스펙을 쓰는 상황과 쓰지 않는 상황
- 비목표(Non-goals) 명시가 중요한 이유
- 대안 검토 섹션의 가치
- 테크스펙이 구현 완료와 함께 수명 종료되는 이유
- ADR과의 차이
