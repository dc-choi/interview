# 기술 로드맵 수립

제품과 기술의 중장기 방향을 단계별로 설계하고 관리하는 역량이다. 로드맵은 "지금 뭘 해야 하는가"와 "왜 이 순서인가"에 대한 답이다.

## 로드맵 설계 원칙

### 1. 단계별 가치 전달
각 단계가 독립적으로 사용자에게 가치를 전달해야 한다. 4단계를 모두 완료해야 의미 있는 것이 아니라, 1단계만으로도 핵심 문제를 해결한다.

### 2. 의존성 기반 순서
기능 간 의존 관계를 분석하여 순서를 결정한다. "인증/권한 설계"가 완료되어야 "조직 기반 기능"이 가능하다.

### 3. 리스크 후순위
비용/계약 리스크가 큰 외부 연동은 PMF 확인 후로 미룬다. 확실하지 않은 것에 일찍 투자하지 않는다.

## school-manage 현재 로드맵 (2026-07-16 정본)

| 단계 | 전략 | 목표 |
|---|---|---|
| 1단계 | Free Core 안정화 | 기존과 신규 조직에 현재 핵심 기능을 무과금으로 제공 |
| 2단계 | 계측 정비와 중고등부 운영 | 같은 코호트의 단체 생성, 학생, 출석과 복귀 행동을 연결 |
| 3단계 | 90일 Paid Parish Operations 실험 | 2026-09-01부터 진단과 학기 운영지원의 실제 수납, 납품 시간과 공헌이익 검증 |
| 4단계 | 게이트 후 장기 옵션 판단 | 2026-11-30에 Basic/Pro, 광고, 외부 연동과 전 그룹 확장 여부 재검토 |

**핵심 전략: "Free Core 유지 → 계측 신뢰성 확보 → 운영지원 실수납 → 장기 옵션 판단"**
- 소프트웨어 유료 기능을 먼저 만들지 않음
- 90일 실험은 구독 전환율이 아니라 고정 범위 운영지원의 현금 수납과 노동 조정 공헌이익을 검증

## 기술 부채와 로드맵

로드맵에 기술 부채 해결을 명시적으로 포함한다.

**심각도 기반 분류:**
| 심각도 | 예시 | 대응 |
|---|---|---|
| Critical (빨강) | 보안 취약점(IDOR, RTR 미구현) | 즉시 대응, 기능 개발 중단 |
| Warning (노랑) | 성능(ErrorBoundary, 번들 최적화) | 다음 스프린트 포함 |
| Low (초록) | DX 개선(컴포넌트 분리) | 여유 있을 때 |

## 로드맵 관리

### 진행 추적
- STATUS.md에 현재 상태와 핵심 지표의 기준일 스냅숏을 갱신
- 주간 단위 데이터 스냅샷으로 트렌드 파악
- 기준일과 단위를 함께 기록 (예: "2026-07-15 MAO 34곳, 목표 50곳")

### 유연한 조정
로드맵은 고정이 아니다. 데이터에 기반하여 조정한다.
- 포지셔닝 전환 가설: "주일학교 중고등부" → "모든 가톨릭 모임"은 보류. 2026-07-15 현재 중고등부 PMF 타깃을 유지하고 2026-11-30 최종 판정 이후 재평가
- 우선순위 변경: 보안 취약점 발견 시 기능 개발보다 보안 패치 우선

### 완료 기준 명확화
각 단계의 완료 조건을 사전에 정의한다.
- 계측 단계: 가입 사용자와 단체의 `first_*` 이벤트를 같은 코호트로 연결
- 운영 단계: MAO 50곳은 운영 목표로 추적하되 사업 검증의 진입 조건으로 사용하지 않음
- 유료 실험: 30만원 선입금 1건, 4시간 이내 납품과 노동 조정 공헌이익 양수를 확인한 뒤에만 60만원 연간 지원 제안

## 면접 포인트

Q. 기술 로드맵을 어떻게 수립하는가?
- 의존성 기반 순서 설계: 계측 신뢰성 → 운영지원 실수납 → 손익 게이트 → 장기 제품 옵션
- 각 단계가 독립적 가치를 전달하도록 설계
- 리스크가 큰 외부 연동은 PMF 확인 후로 미룸

Q. 로드맵은 어떻게 관리하는가?
- STATUS.md로 핵심 지표 실시간 추적
- 데이터 기반으로 유연하게 조정 (포지셔닝 전환, 우선순위 변경)
- 기술 부채를 심각도별로 분류하여 로드맵에 명시적 포함
