# 아키텍처의사결정과경제적관점

> 출처: [[Why-Architecture-Matters|소프트웨어 아키텍처의 중요성]]

## 아키텍처의두가지핵심

### 1.지식의공유
- 아키텍처 = 전문 개발자들이 공유하는 시스템 디자인에 대한 이해 (Martin Fowler)
- 다이어그램과 문서는 지식을 공유하는 불완전한 표현
- 프로젝트가 잘 성장할 때 = 팀원 간 이해도가 잘 공유될 때

### 2.결정의비가역성
- 아키텍처 디자인은 프로젝트 초기에 올바르게 결정되어야 함
- **한번 결정하면 변경하기 매우 어려움** — 이것이 아키텍처가 중요한 이유

## 경제적관점에서의아키텍처

### 장인정신vs경제
- 장인 정신과 경제의 싸움에서 경제적 관점이 항상 이김
- 아키텍처의 중요성을 **경제적 논리**로 설명할 수 있어야 함

### 외부품질vs내부품질

| 구분 | 설명 | 가시성 |
|------|------|--------|
| 외부품질 | 기능, UX, 성능 등 사용자가 경험하는 것 | 보임 |
| 내부품질 | 코드 구조, 아키텍처, 테스트 가능성 | **보이지 않음** |

- 아키텍처 = 내부 품질
- 외부 이해관계자는 내부 품질을 볼 수 없고 평가할 수도 없음
- 하지만 **장기적으로 가장 중요**

### DesignStaminaHypothesis

Martin Fowler의 설계 체력 가설:

- **아키텍처 무시**: 초기 빠름 → 시간이 지날수록 기능 추가가 급격히 느려짐
- **아키텍처 투자**: 초기 느림 → 기존 코드가 플랫폼처럼 동작하여 점점 빨라짐
- 이것이 CI/CD가 보편화되는 이유 — 기능을 자주, 잘 추가할 수 있어야 하므로

## 시니어엔지니어로서활용

### 기술부채설명
- "왜 리팩토링이 필요한가?" → Design Stamina Hypothesis로 설명
- 내부 품질 저하 = 개발 속도 감소 = 비즈니스 손해

### 아키텍처의사결정
- "좋은 아키텍처란?" → 팀의 공유된 이해 + 변경하기 어려운 결정을 올바르게 내리는 것
- "아키텍처가 왜 중요한가?" → 경제적 관점에서 기능 추가 속도에 직접 영향

### 리팩토링정당화
- 비개발자에게 리팩토링의 필요성을 설득할 때 경제적 논리 사용
- "지금 투자하면 6개월 후 기능 추가가 2배 빨라진다"
