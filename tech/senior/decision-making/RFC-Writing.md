# RFC / PRD 작성

기술적 의사결정이나 제품 변경을 구조화된 문서로 정리하여 팀과 공유하는 역량이다. RFC(Request for Comments)와 PRD(Product Requirements Document)는 형식은 다르지만 목적은 동일하다: **결정의 배경, 범위, 리스크를 투명하게 기록한다.**

## RFC vs PRD

| 구분 | RFC | PRD |
|---|---|---|
| 초점 | 기술적 설계와 구현 방향 | 제품 요구사항과 사용자 시나리오 |
| 독자 | 엔지니어 | 엔지니어 + PM + 디자이너 |
| 핵심 질문 | "어떻게 만들 것인가?" | "무엇을 왜 만들 것인가?" |

실무에서는 둘을 혼용하거나 하나로 합치는 경우가 많다.

## PRD 구조 (실전 템플릿)

### 1. 배경/문제 요약
- 현재 상태와 문제점을 구체적으로 기술
- 관련 문서 참조 링크
- **"무엇이 문제인가"를 데이터와 사례로 증명**

예: "공유 계정 구조가 구조적 이탈을 유발한다. 신당동/금호동에서 실증: 2명 이상 교리교사가 개별 가입 시 혼란 발생 → 사용 중단"

### 2. 목표/성공 기준
- 정량적 성공 지표를 사전에 정의
- 측정 기간 명시

예: "WAU 12명 이상 달성, 합류 플로우 완료율 70%+, 데이터 손실 0건, 측정 기간 배포 후 4주"

### 3. 사용자/대상
- 주요 사용자와 사용 맥락
- 사용자 시나리오 (최소 3개: 신규, 기존, 예외)

### 4. 범위
- **포함(In Scope):** 이번에 구현할 것
- **제외(Out of Scope):** 의도적으로 빼는 것과 이유

범위를 명시적으로 정의하면 스코프 크립(scope creep)을 방지한다.

### 5. 요구사항
- **필수(Must):** 없으면 출시 불가
- **선택(Should):** 있으면 좋지만 없어도 출시 가능
- **제외(Out):** 명시적으로 하지 않는 것

MoSCoW 우선순위 기법과 동일한 원리다.

### 6. 제약/가정/리스크
- **제약:** 변경 불가능한 조건 (인력, 기술 스택, 일정)
- **가정:** 검증이 필요한 전제 (사용자가 거부감이 없을 것)
- **리스크:** 실패 가능성과 완화 방안

리스크에는 반드시 **완화(mitigation) 방안**을 함께 기술한다.

### 7. 의존성
- 내부 의존: 다른 기능, 스키마 변경
- 외부 의존: 사용자 동의, 외부 서비스

### 8. 롤아웃/검증
- 출시 단계 (점진적 배포 계획)
- 추적할 이벤트/로그
- 검증 방법과 성공 판단 기준

## 좋은 RFC/PRD의 특징

1. **의사결정을 내릴 수 있다** — 읽은 후 "하자/말자/이렇게 바꾸자"를 판단할 수 있음
2. **이유가 있다** — "무엇"뿐 아니라 "왜"와 "왜 다른 것은 아닌가"가 포함
3. **범위가 명확하다** — 포함/제외가 명시적이어서 스코프 논쟁을 줄임
4. **검증 가능하다** — 성공/실패를 판단할 수 있는 정량 지표 존재
5. **오픈 이슈가 있다** — 아직 결정하지 못한 것도 투명하게 기록

## Spec-Driven Development (SDD)

명세(Spec)가 개발 전 과정을 주도하는 접근 방식이다. 코드를 먼저 쓰는 것이 아니라, **"왜 만드는가 → 어떻게 동작하는가 → 어떻게 구현하는가"를 문서로 먼저 정의**한 뒤 구현에 들어간다.

### 실제 적용: 7단계 워크플로우

school-manage 프로젝트에서 SDD를 다음과 같이 구체화했다.

| 단계 | 설명 | 핵심 질문 |
|---|---|---|
| 0 | 작업 선택 + 사업 문서 확인 + README 등록 | 무엇을 할 것인가? |
| 1 | PRD 작성 | 왜 만드는가? |
| 2 | 기능 설계 작성 | 어떻게 동작하는가? |
| 3 | Task 작성 (역할별 업무 분할) | 누가 무엇을 하는가? |
| 4 | Development 작성 (역할별 구현 명세) | 어떻게 구현하는가? |
| 5 | 구현 + 테스트 | 코드 작성 |
| 6 | 자동 검증 + 문서 정리 + PR 생성 | 품질 보장 |

### 문서 계층 구조

PRD(왜) → 기능 설계(어떻게 동작) → Task(누가 무엇을) → Development(어떻게 구현) → 코드

각 계층이 상위 문서를 기반으로 하며, 하위로 갈수록 구체화된다.

### SSoT (Single Source of Truth)

구현 완료된 기능의 진실 원천은 **기능 설계 문서 + 코드베이스**이다. Task/Development 문서는 구현 완료 후 **삭제**한다. 코드와 문서의 불일치를 원천 방지하는 핵심 원칙이다.

### 역할 분리

- **사업 에이전트:** 문제 정의, 가치, 우선순위, 로드맵 (별도 워크스페이스)
- **SDD 에이전트:** 문서 작성 + 구현 + 자동 검증 + PR 생성
- **사용자:** PR 리뷰 → 머지 결정

### 자동 검증 + 서브에이전트 리뷰

구현 완료 후 lint, typecheck, build, test를 자동 실행하고, PR 생성 전 **서브에이전트**(security-reviewer, design-reviewer, performance-analyzer)가 보안/디자인/성능을 자동 리뷰한다.

### 비기능 요구사항은 간소화

성능, DX 개선 등 비기능적 변경은 Task/Development를 생략하고 기능 설계 → 바로 구현 → 자동 검증으로 진행한다. 완료 후 코드 자체가 SSoT가 되므로 별도 문서를 유지하지 않는다.

## 면접 포인트

Q. 기술 문서 작성 경험은?
- SDD(스펙 주도 개발)를 직접 설계하여 프로젝트에 적용
- 21개 PRD + 16개 기능 설계 문서 작성/관리
- 문서 계층: PRD(왜) → 기능 설계(동작) → Task(역할 분할) → Development(구현 명세) → 코드
- SSoT 원칙: 구현 완료 후 Task/Development 삭제, 기능 설계 + 코드만 유지

Q. 문서와 코드의 관계는?
- 기능 설계는 "동작 명세" 수준만 유지, 구현 상세는 코드에 위임
- PRD의 성공 지표가 테스트 케이스와 롤아웃 검증의 기반
- 서브에이전트(보안/디자인/성능)가 PR 전에 자동 리뷰하여 품질 게이트 역할
