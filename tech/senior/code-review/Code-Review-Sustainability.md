# 코드 리뷰의 정당화와 지속 (Sustainability)

리뷰 원칙(→ [[Code-Review-Culture]]), 리뷰어 댓글 기법(→ [[Code-Review-Reviewer-Guide]]), 우선순위 태그(→ [[Code-Review-Pn-Priority]])와 별개로, 리뷰를 **조직에 정착시키고 비즈니스로 정당화하며 갈등을 관리하는** 층이 따로 있다. 좋은 리뷰 문화는 좋은 도구, 작은 PR, 빠른 피드백, 정중한 언어에 더해 **리더의 지원과 경제적 정당화**가 있을 때 오래간다.

## 전제: 품질은 비용을 줄인다

리뷰에 시간을 쓰는 것을 정당화하려면 먼저 품질의 경제학을 이해해야 한다.

- **소프트웨어에서 설계는 곧 코드다.** 건축은 설계도와 시공이 분리되지만, 소프트웨어에서 동일한 결과물을 재현할 수 있는 가장 정확한 설계물은 코드 그 자체다. 좋은 코드를 쓴다는 것은 동작하는 프로그램이 아니라 **좋은 설계를 남기는 일**이고, 개발자는 코드를 쓰면서 동시에 설계를 수행한다.
- **"빨리 만들려고 품질을 낮춘다"는 장기적으로 더 느리다.** 설계를 무시한 코드는 초반엔 빨라 보이지만, 기능이 쌓이고 변경이 반복되면 속도가 급격히 떨어진다. 품질은 개발자의 취향이 아니라 **버그 수정, 장애 대응, 변경 요청 처리 시간을 줄이는 비즈니스 전략**이다 → [[Code-Quality-Criteria|코드 품질의 기준]].

## 사람보다 도구가 잘하는 일은 자동화한다

리뷰어가 들여쓰기, 포맷팅, import 정리, 정적 분석 같은 일을 반복해서 지적하게 만들면 안 된다. 이런 일은 **포매터, 린터, 테스트, 정적 분석 도구**가 처리한다.

- 사람은 로직, 설계, 의도, 유지보수성처럼 **판단이 필요한 부분**에 집중한다.
- 스타일 가이드는 팀에서 합의하고, 한번 정하면 **불필요한 논쟁을 줄이는 기준**으로 쓴다.
- 이는 결정적 규칙은 도구로 강제하고 사람은 판단에 쓰는 원칙과 같다(스타일=Hook/CI, 사람=설계). 리뷰어의 `nit` 폭격을 구조적으로 없애는 방법이다.

## 비즈니스 언어로 품질을 설명한다

"좋은 코드가 필요합니다"만으로는 경영진, 다른 직군을 설득하기 어렵다. 비즈니스가 이해하는 지표로 번역한다. 네 가지 배포 성과 지표(DORA)가 공통 언어다.

| 지표 | 의미 |
|---|---|
| **변경 처리 시간 (Lead Time)** | 코드 변경이 배포되기까지 걸리는 시간 |
| **배포 빈도 (Deployment Frequency)** | 얼마나 자주 배포하는가 |
| **장애 복구 시간 (MTTR)** | 장애에서 복구하는 데 걸리는 시간 |
| **배포 실패율 (Change Failure Rate)** | 배포가 장애로 이어지는 비율 |

코드 리뷰의 가치는 "개발자가 좋아해서"가 아니라 **"더 빠르고 안정적으로 변경하기 위해서"**다. 리뷰는 이 지표들을 개선하는 수단으로 설명해야 예산과 시간을 확보한다 → [[Software-Productivity-Measurement|소프트웨어 생산성 측정의 함정]].

## 갈등이 커지면 온라인에서 멈춘다

텍스트 리뷰는 톤이 안 보여 갈등이 증폭되기 쉽다. 위험 신호와 대응을 미리 정해 둔다.

- **위험 신호**: 댓글이 점점 공격적으로 변함, 같은 주제로 반박이 반복됨.
- **대응**: 댓글로 계속 싸우지 말고 **직접 대화하거나 화상으로** 옮긴다. 그래도 안 풀리면 리더나 다른 리뷰어에게 도움을 요청한다.
- **관계 > 완벽한 코드 하나**: 때로는 완벽하지 않아도 받아들이는 선택이 필요하다. 코드 하나를 완벽하게 만들려다 팀의 협업 관계를 망가뜨리면 더 큰 손해다.

## 완벽을 한 번에 요구하지 않는다 (점진 상향)

리뷰의 목표는 모든 코드를 최고 수준으로 만드는 게 아니다. 현재 D 수준이면 C나 B로 올리는 것만으로 의미가 있다. 작성자의 역량과 상황을 무시하고 A+를 요구하면 **학습보다 방어가 먼저** 일어난다. 좋은 리뷰는 상대를 압도하는 게 아니라 다음 단계로 올라가게 돕는 일이다 → [[Code-Review-Reviewer-Guide|숙제 검사가 아닌 학습 과정]].

## 리더의 역할과 리뷰 보상

리뷰 문화는 개인의 선의만으로 오래가기 어렵다.

- 리더가 리뷰의 가치를 인정하고 **리뷰 시간을 업무의 일부**로 본다.
- 리뷰를 한 사람도 팀 생산성에 기여한 것이다. 성과와 보상에서 **코드 작성만 인정하고 리뷰 기여를 무시하면** 리뷰 문화는 유지되지 않는다.

## 점진적 도입

처음부터 모든 규칙을 한 번에 적용하면 실패하기 쉽다. **작은 규칙 하나**부터 시작하고 익숙해지면 다음을 추가한다. 새 습관은 초기엔 생산성이 떨어지는 것처럼 느껴지지만(J-curve), 그 구간을 지나면 팀의 학습 속도와 코드 품질이 함께 올라간다.

## 면접 포인트

Q. 코드 리뷰의 가치를 비개발 직군에 어떻게 설득하나?
- DORA 4지표(변경 처리 시간, 배포 빈도, MTTR, 배포 실패율)로 번역한다. 리뷰는 "더 빠르고 안정적으로 변경하기 위한 수단"이지 개발자 취향이 아니다.

Q. 리뷰에서 갈등이 격해지면?
- 위험 신호(공격적/반복 반박)를 감지하면 온라인 댓글을 멈추고 직접 대화로 옮긴다. 안 풀리면 리더 에스컬레이션. 완벽한 코드 하나보다 협업 관계가 우선이다.

Q. 리뷰어가 스타일만 지적하는 문제는?
- 포매터, 린터, 정적 분석으로 자동화하고 사람은 로직, 설계, 의도에 집중하게 만든다. 스타일 가이드는 합의로 고정해 논쟁을 없앤다.

Q. 리뷰 문화를 어떻게 정착시키나?
- 작은 규칙 하나부터 점진 도입(J-curve 인지), 리더가 리뷰 시간을 업무로 인정하고 보상에 반영해야 지속된다.
