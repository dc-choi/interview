# Pn 룰 · 코드 리뷰 우선순위 체계

텍스트 기반 코드 리뷰에서 **리뷰어의 의도와 블로킹 수준**이 명확하지 않으면, 작성자는 사소한 지적도 중대한 거부로 해석해 감정 소모·지연이 발생한다. **Pn 룰**은 모든 리뷰 코멘트에 P1~P5의 태그를 붙여 **수용·토론·무시 가능 여부**를 첫 문자에서 드러내는 관행. 원래 뱅크샐러드에서 제안됐고 11번가 등 여러 조직이 도입했다.

## 핵심 명제

- **커뮤니케이션 비용이 성장하는 회사의 가장 큰 비용**
- 같은 코멘트도 **톤에 따라** Request / Comment / Suggestion으로 달리 해석됨
- Pn 태그로 **기계적 분류** → 해석 비용 제거
- 코드 리뷰뿐 아니라 **Slack·문서·회의록** 같은 텍스트 전반에 적용 가능

## 5단계 체계

### P1 — 꼭 반영해주세요

- **GitHub 상태**: Request Changes
- **의미**: 서비스·보안에 중대한 오류를 야기할 수 있음
- **작성자 대응**: 반영 또는 **합리적 근거**로 리뷰어 설득
- **예시**: 보안 취약점, 프로덕션 장애 유발, 데이터 무결성 파괴

### P2 — 적극적으로 고려해주세요

- **상태**: Request Changes
- **의미**: 반드시 해결되어야 하나 대안 토론 여지 있음
- **작성자 대응**: 수용 또는 **논의 후 결론**
- **예시**: 구조 결함, 성능 이슈, 아키텍처 일관성

### P3 — 웬만하면 반영해주세요

- **상태**: Comment
- **의미**: 이번 PR에서 고치면 좋지만 필수 아님
- **작성자 대응**: 수용 or **사유 명시 후 후속 티켓**
- **예시**: 가독성 개선, 네이밍 제안, 소규모 리팩터링

### P4 — 참고해주세요

- **상태**: Approve + Comment
- **의미**: 개인 의견·경험 공유
- **작성자 대응**: 자유롭게 판단
- **예시**: "저라면 이렇게", "다음에 이 방향도 고려"

### P5 — 무시해도 됩니다

- **상태**: Approve
- **의미**: 사소한 취향·아이디어
- **작성자 대응**: 무시 OK
- **예시**: "재밌네요", "오타 같아요"

## 도입 효과

- **해석 비용 감소** — 태그만 봐도 대응 방식 결정 가능
- **감정 소모 감소** — 사소한 지적을 P5로 명시하면 방어 반응 차단
- **블로커 명확화** — P1·P2는 머지 전 해결, P3 이하는 협의
- **작성자 주도권** — P3 이하는 수용 여부를 작성자가 결정 가능
- **전사 확산** — 코드 리뷰 외 모든 텍스트 커뮤니케이션에 적용 가능

## 사용 예시

```
P1: N+1 쿼리 발생. 현재 구조로는 사용자 수 증가 시 DB 타임아웃 위험.
→ fetch join 또는 DataLoader로 해결 필요

P3: 함수명 getData가 조금 모호해 보입니다.
fetchActiveUserOrders 같이 구체적이면 읽기 편할 것 같아요.

P5: 이 리팩터링 정말 좋네요 👍
```

작성자가 한눈에 **무엇을 반드시 대응하고 무엇은 넘길지** 파악.

## 유사 체계와의 비교

### `nit:` · `MUST:` 태그

- Google·Meta에서 사용하는 2단계 축약
- `nit:` = 사소, `MUST:` = 필수 반영
- Pn은 **5단계로 더 세밀**

### 이모지 기반

- 일부 팀은 🚨(긴급), ⚠️(주의), 💡(제안), 💬(잡담)으로 구분
- 직관적이지만 5단계 구조보다는 **해석 편차** 큼

### 블로킹 토글 기반

- GitHub Request Changes / Approve로만 구분
- 단순하지만 **사소한 의견과 중대한 지적**이 섞임

**Pn의 우위**: 정교함 + 작성자 주도권 + 교차 텍스트 적용성.

## 도입 체크리스트

팀에 Pn 룰을 도입할 때:

- [ ] **팀 워크숍**에서 5단계 정의 합의 (조직마다 범위 조정 가능)
- [ ] 리뷰 PR 템플릿·가이드에 **태그 사용법** 명시
- [ ] 초기 2주는 **리뷰어 모두가 강제 사용**으로 습관화
- [ ] 첫 달 **회고** — P1 남발 여부, 명확성 점검
- [ ] Slack·문서에도 **확산** — 태그의 교차 활용
- [ ] 신규 입사자 **온보딩**에 포함

## 흔한 실수

- **P1 남발** — 모든 의견을 블로커로 두면 효과 소실. P1은 "서비스 영향"으로 엄격히
- **태그 없이 작성** — 일관성 안 지켜지면 체계 무너짐
- **작성자가 P3 이하를 다 반영해야 한다고 착각** — P3 이하는 **작성자 재량**
- **P4·P5의 가치 폄하** — 칭찬·공유 코멘트는 팀 시너지 증폭제
- **Pn 태그가 톤을 대체한다고 생각** — 톤·근거 설명은 여전히 중요, 태그는 **보조**

## 효과 측정

도입 후 **정성·정량 지표**로 효과 확인.

- **리뷰 라운드 횟수** — P1 명확화로 1~2라운드에 수렴
- **PR 머지까지 소요 시간** — 체감 감소 확인
- **리뷰 스레드 길이** — 핵심만 논의하므로 짧아짐
- **팀 설문** — "리뷰로 인한 스트레스" 감소 여부

## 면접 체크포인트

- 코드 리뷰에서 **커뮤니케이션 비용**이 발생하는 이유
- Pn 룰 5단계의 **블로킹 여부**와 작성자 대응
- P1·P2 vs P3·P4·P5의 **작성자 재량 경계**
- `nit:`/`MUST:` 같은 **2단계 체계**와의 비교
- 팀에 Pn을 도입할 때의 **전환 전략**
