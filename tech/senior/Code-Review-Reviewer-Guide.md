# 코드 리뷰어의 자세 · 피드백 작성 가이드

좋은 코드 리뷰 **문화**를 만드는 것과 별개로, **리뷰어 개인의 댓글 작성 방식**이 리뷰의 체감 품질을 결정한다. 같은 문제 지적도 **왜를 설명하는가·답을 주는가 방향을 주는가·개발자 성향을 공격하는가 코드에 집중하는가**에 따라 리뷰이가 성장하거나 위축된다. 리뷰어의 실전 댓글 5원칙 + 추가 팁. 문화·프로세스 관점은 [[Code-Review-Culture|생산적 코드 리뷰 문화]].

## 핵심 명제

- **좋은 리뷰는 길고 나쁜 리뷰는 짧다** — 변경 이유·근거를 담으면 자연스럽게 길어짐
- 답보다 **방향·키워드** — 리뷰이가 스스로 학습할 기회
- **개발자 성향이 아닌 코드**에 집중
- 지적할 게 없으면 **칭찬** — 리뷰를 위한 리뷰는 감성 소비

## 5가지 원칙

### 1. "왜 개선이 필요한가"를 충분히 설명

리뷰가 주관적·추상적이면 리뷰이는 혼란. 구체적 **근거**를 필수로.

**나쁜 예**: "data 변수 말고 다른 이름으로 하세요"
- 이유 없음 → 리뷰이는 왜 바꿔야 하는지 모름

**좋은 예**: "data라는 이름은 자료구조의 의도를 드러내지 못합니다. 학점 정보를 담고 있으니 이를 반영한 이름으로 바꾸면 읽는 사람이 자료구조를 즉시 이해할 수 있습니다"
- 현재 문제 + 바꿨을 때 이득 + 방향 제시(답은 아님)

### 2. 답 대신 방향·키워드

업무와 리뷰 병행에서 "`A.filter()` 쓰세요"처럼 **정답**을 던지기 쉽다. 하지만 리뷰이의 학습 기회를 빼앗는다.

**나쁜 예**: "`arr.filter(item => item % 2 === 0)`으로 사용하세요"
- filter 동작 이해 없이 코드만 바꿈
- 비슷한 코드가 여러 곳 있어도 이 부분만 고침

**좋은 예**: "자바스크립트 배열에는 다양한 내장 메서드가 있습니다. 그중 `filter` 메서드를 활용해 보세요. 코드량을 줄이고 의도를 명확히 할 수 있습니다"
- 키워드(`filter`)만 제시 → 리뷰이가 동작 방식 학습
- 유사 패턴을 **본인이 찾아** 함께 리팩터링 가능

**언제 답을 줘도 되는가**: 단순 오타·명확한 컨벤션 위반·보안 취약점. 학습 포인트가 없는 지점.

### 3. 일관성: 전체 코드 맥락에서 보기

독립적 조각만 보면 놓친다. 같은 PR·같은 프로젝트 안의 **일관성 불균형**을 짚어야 한다.

**나쁜 예**: "함수 표현식을 사용하셨군요. reduce도 잘 썼네요"
- 조각만 보고 칭찬. 전체 관점에서 일관성 문제를 놓침

**좋은 예**: "두 함수가 함수 표현식과 선언식을 섞어 쓰고 있습니다. 특별한 이유가 아니라면 스스로 컨벤션을 정하고 따르세요. 또 같은 반복 처리인데 하나는 `reduce`, 다른 하나는 `for`입니다. 비슷한 고차 함수로 통일하면 더 일관된 코드가 됩니다"
- 두 함수를 **교차 비교**해 불일치 지적
- 이유(일관성의 가치) + 방향(고차 함수 통일) 제시

### 4. 숙제 검사가 아닌 학습 과정으로

리뷰이가 "**감사**"나 "**평가**"로 느끼면 방어적이 된다. 의도를 **먼저 묻는** 접근이 신뢰를 만든다.

**나쁜 예**: "클래스 상속은 필요 없습니다"
- 단정 + 의도 무시. 방어 유발

**좋은 예**: "`Component`를 상속받는 건 좋은 시도네요. 다만 자식 클래스에서 부모 호출만 하고 있어 중복 코드처럼 보이는데, 이렇게 설계한 이유가 있을까요? (예: 향후 확장을 고려)"
- **칭찬 먼저** → 의도 질문 → 필요하면 후속 논의
- 리뷰이는 스스로 생각하고 답을 만들 기회

### 5. 리뷰를 위한 리뷰 지양 · 칭찬 활용

흔적 남기려는 의무감의 리뷰는 관계만 상한다. 깔끔한 코드에 억지로 "다른 디자인 패턴 권유"하지 말기.

**활용 가능한 칭찬 예시**:
- "코드량이 적당해서 읽기 편하네요"
- "고민이 많이 엿보입니다"
- "성능에 유리한 설계네요"
- "전보다 훨씬 좋아졌네요"
- "예외 처리가 꼼꼼합니다"
- "함수·변수명이 직관적이네요"

칭찬은 **시너지 증폭제**. 긴 스레드에 칭찬이 섞여야 지속 가능.

## 추가 팁

### 커밋이 아닌 전체 PR 맥락으로 리뷰

- 커밋 단위만 보면 **부분 최적화 함정**에 빠짐
- 작은 조각은 깔끔해 보여도 **전체에서 어색**할 수 있음
- 신규 프로젝트·투입 초기 개발자는 커밋 단위 리뷰로 흐름 파악 어려움
- **PR 전체를 한 번 훑고** 상세 리뷰에 들어가기

### Pull Request 템플릿 활용

PR 실수와 고민 시간 축소.

**템플릿에 포함할 요소**:
- **체크리스트**: target 브랜치 확인, 리뷰 라벨 부착, Approve 후 머지 규칙
- **작업 내용**: 무엇을 왜 했는지 요약
- **참고 링크**: 설계 문서, 이슈, 스펙
- **테스트 방법**: 재현·검증 절차
- **스크린샷·마크다운**: 말로 설명하기 어려운 부분

리뷰이가 맥락을 미리 써두면 **리뷰어 인지 시간이 단축**되어 리뷰 품질이 올라간다.

### 개발자 성향 리뷰는 피하기

**나쁜 예**: if/else를 삼항 연산자로 바꾸라고 강제
- 가독성이 **개인 취향** 영역인 경우 강제 지양
- 추가 연산이 없는 단순 분기의 가독성은 주관적

**좋은 예**: 추가 연산이 있거나 코드 베이스 컨벤션이 있을 때만 제안
- 컨벤션이 없으면 **스레드로 토론** → 팀 합의 후 적용

**원칙**: 개발자의 **스타일·성격**이 아니라 **코드·팀 컨벤션**에만 집중.

## 좋은 리뷰 vs 나쁜 리뷰의 길이 비대칭

- **나쁜 리뷰는 짧다**: 이유 없이 변경 요청만
- **좋은 리뷰는 길다**: 이유·근거·방향·칭찬이 섞여 있음
- 온라인 리뷰일수록 **상대방의 표정·상황**이 안 보이므로 더 신중히
- 한 줄의 무심한 피드백이 리뷰이의 **기분·동기**를 해칠 수 있음

길이를 두려워 말 것 — 설명하지 않을 거면 차라리 쓰지 않는 편이 낫다.

## 톤 컨벤션

팀 전체가 합의하면 유용한 태그·톤.

- **`nit:`** — 사소한 취향·스타일, 머지 차단 안 함
- **`MUST:` 또는 `blocking:`** — 머지 전 반드시 해결
- **`question:`** — 답을 요구하기보다 의도 파악
- **`suggestion:`** — 강제 아닌 제안
- **`praise:`** — 잘한 부분 짧은 칭찬
- 이모지 한두 개(👍·🤔·💡)로 감정 톤 완화

## 공감 만들기

- **"저라면 이렇게 할 것 같은데요"** — 주관 표시
- **"혹시 ... 이유가 있을까요?"** — 의도 존중
- **"이 부분은 잘 이해가 안 돼서..."** — 책임을 리뷰어가 짐
- **"... 부분은 좋습니다"** — 좋은 점 명시

명령형("이렇게 하세요")은 숙제검사 톤을 만든다.

## 사례: 좋은 리뷰 vs 나쁜 리뷰 길이 비교

### 나쁜 리뷰

> "if/else 말고 삼항으로 하세요. 깔끔해집니다."

### 좋은 리뷰

> "현재 if/else와 삼항 연산자 둘 다 가독성 차이는 크지 않아 보입니다. 만약 팀 컨벤션이 있다면 따르는 게 좋고, 없다면 어느 쪽이 읽기 편한지 먼저 논의해봐도 좋겠습니다. 추가 연산이 없는 단순 분기라면 개인 취향 영역일 수 있어요."

길이의 차이는 **사고의 깊이 차이**. 나쁜 리뷰는 빨리 끝나지만 리뷰이를 위축시키고, 좋은 리뷰는 시간을 들이지만 **팀 학습·신뢰**를 만든다.

## 흔한 실수

- 이유 없이 변경만 요구
- 답을 즉시 던져서 학습 기회 삭제
- 조각만 보고 **일관성 불균형 놓침**
- 의도 질문 없이 단정
- 리뷰를 위한 리뷰 (흔적 남기기)
- 삼항·if/else 같은 **주관적 스타일 강제**
- PR 전체 맥락 없이 **커밋 단위 리뷰**
- PR 템플릿 없이 매번 새로 쓰기
- 온라인 리뷰에서 **톤 조절 없이** 직설

## 면접 체크포인트

- 좋은 리뷰의 **5원칙**을 자기 언어로 말할 수 있는가
- 최근 본인이 작성한 리뷰 댓글을 **좋은/나쁜**으로 평가 가능한가
- "답 대신 방향"의 구체 예시
- 일관성 리뷰가 왜 **전체 맥락**을 필요로 하는가
- PR 템플릿의 구성 요소와 효과
- 개발자 성향 vs 코드 리뷰의 경계
