# MOA백엔드스터디커리큘럼

PBL(Project Based Learning) 방식으로 설계한 백엔드 스터디 커리큘럼.

## 교육철학

### PBL방식
- 학습자가 요구사항을 구현하기 위해 주도적으로 학습
- 요구사항에 적히지 않은 문제는 스스로 정의하고 분석

### 기본기중심
- 기본기 = 주어진 요구사항을 분석하고 구현할 수 있는 능력
- 문제를 정의하고 요구사항을 분석하는 것이 개발자의 핵심 역량
- AWS 등 인프라는 직접 경험하는 것이 더 효과적이므로 커리큘럼에서 제외

### 동료학습
- 페어를 구성하여 함께 요구사항 분석 → 정의 → 설계
- "혼자 42시간 고민할 것을 같이 하면 42분"
- 매주 월/목 정기 모임 + 페어 자율 모임
- 스프린트 종료 시 랜덤 발표 (2명, 사다리타기)

### 학습방향확인
- 요구사항 분석/정의 후 멘토에게 공유 → 방향성 검증
- Commit/Push 생활화 → 업무 내역 추적 + 방향성 확인

## 커리큘럼구조

```
1차: 배달 플랫폼          2차: 커머스 플랫폼
├─ Sprint 1: 설계         ├─ Sprint 3: 설계
│  DB설계, OOP 설계       │  DB설계, OOP 설계
│                         │  (재고 관리 추가)
└─ Sprint 2: 성능개선      └─ Sprint 4: 동시성
   트랜잭션, 실행계획,       DB Lock,
   인덱스                   동시성 제어
```

### 기술스택제약
- Java 17+, Spring Boot 3+, MySQL 8+

## Sprint1:배달플랫폼-주문설계

### 구현API
- 주문 완료
- 가게의 주문 확인 (페이징 없음)

### 주문플로우
`가게 선택 → 메뉴 선택 → 장바구니 담기 → 주문 완료`
(주문 완료 API 구현 시 앞 프로세스는 완료된 전제)

### 상세요구사항
- 메뉴 안에 옵션, 옵션 안에 옵션 상세 존재
- 결제는 고려하지 않음
- 객체 지향적 코드 구조 설계

### 제약사항
- 장바구니 데이터는 서버에 저장하지 않음 (클라이언트 보관)
- 도메인별 구조를 확실히 분리
- **도메인 사이 양방향 참조 금지**
- DB 설계는 본인이 직접 수행

### 제출물
- GitHub Repository (delivery-server)
- ERD + 클래스 다이어그램 발표자료

### 학습포인트
- DB 설계 능력
- OOP 설계 (도메인 분리, 단방향 의존성)
- 장바구니 미저장 시 메뉴 검증 문제 (변경 불일치 대응)

## Sprint2:배달플랫폼-성능개선

### 상황
주문 100만건 발생. 가게에서 주문 확인 기능 수정 요청.

### 필수요구사항
- 페이징 구현
- 주문을 월간/주간/일자별로 확인

### 선택요구사항
- 트랜잭션 개념 학습 후 결제 부분 구현 (외부 PG 연동 없이)

### 제출물
- GitHub Repository
- 성능 개선 경험 발표자료
- 트랜잭션 발표자료 (선택)

### 학습포인트
- 실행 계획(EXPLAIN) 분석
- 인덱스 설계와 튜닝
- 페이징 전략
- 트랜잭션 이해 (선택)

## Sprint3:커머스플랫폼-주문설계

### 구현API
- 주문 완료
- 상품 내역 조회
- 상품 상세 조회

### 주문플로우
`상품 선택 → 장바구니 담기 → 주문 완료`

### 상세요구사항
- 가상 물류 창고에서 제품 보관, 소비자에게 판매
- 하나의 상품은 여러 제품(옵션별 상품)을 포함
- **제품 재고 관리 필수**
- 결제는 고려하지 않음
- 객체 지향적 설계

### 제약사항
- Sprint 1과 동일 (장바구니 미저장, 도메인 분리, 양방향 참조 금지)
- GitHub: commerce-server Fork

### 제출물
- GitHub Repository
- ERD + 클래스 다이어그램 발표자료

### 학습포인트
- 상품-제품(SKU) 관계 설계
- 재고 관리 도메인 모델링
- 1차와의 차이를 통한 설계 성장 확인

## Sprint4:커머스플랫폼-동시성

### 상황
재고가 한정적인데 여러 사용자가 몰리면 재고 소진 후에도 주문이 되는 현상 발생.

### 요구사항
- 동시성 문제 원인 분석
- 해결 방안 구현

### 제출물
- GitHub Repository
- 문제 해결 경험 발표자료

### 학습포인트
- Race Condition 이해
- DB Lock (비관적 락, 낙관적 락)
- 동시성 제어 전략 (SELECT FOR UPDATE, 버전 관리 등)

## 커리큘럼설계의도

| 스프린트 | 핵심역량 | 난이도진행 |
|---------|---------|-----------|
| 1 | DB설계 + OOP | 기본 도메인 모델링 |
| 2 | 성능 분석 + 개선 | 대용량 데이터 대응 |
| 3 | DB설계 + OOP (심화) | 재고라는 상태 관리 추가 |
| 4 | 동시성 제어 | 분산 환경의 핵심 문제 |

- 1차(배달)에서 기본기를 다지고, 2차(커머스)에서 심화
- 각 차수의 첫 스프린트는 기능적 요구사항(설계), 두번째 스프린트는 비기능적 요구사항(성능/동시성)
- 점진적으로 복잡도를 높여가며 실무에서 마주치는 문제를 경험
