# 토스내부평가기준

> 출처: [[Toss-Runners-High-Seminar|토스 러너스하이 세미나]]

## 기술코드/설계

### 1.라이브러리내부분석으로문제해결
- 라이브러리 내부 구현을 분석/파악하여 문제 해결 가능한가?
- 예: DB Pool의 다양한 설정이 존재하는 이유? Jackson ObjectMapper Exception 발생 시 원인 파악?

### 2.모범사례비교로더나은설계제시
- 현재 구현이 최선이라고 어떻게 확신하는가?
- 더 좋은 사례를 참고해서 부족한 점을 개선
- 피드백, 레퍼런스, 컨퍼런스 발표를 통해 학습

### 3.시스템성능최적화
- 리소스를 30% 덜 사용할 방법이 있는가? (Hyper-Thread)
- 적정 사용량은? 어느 정도의 여유 자원이 낭비인가? (IPO Projection)
- HA? 이중화? 속도? API 호출 수?

## 제품

### 비즈니스성장기여
- 우리의 월급은 어디서 오는가?
- 중요한 건 만들어가는 것. 10개 중 1개 성공
- 외부에서는 잘하는 것처럼 보이지만, 내부에서는 매일 치열하고 안 되는 것도 많음
- **결과로 말한다**는 생각으로 일함

### 담당제품규모와임팩트
- 한 영역의 깊이뿐만 아니라 다양한 영역에 관심을 갖고 커버
- **내가 있을 때와 없을 때의 차이**를 팀이 느끼고 있는가?

## 커뮤니케이션

### 데이터기반의견조율
- 큰 일은 팀으로 해결할 수밖에 없기에 커뮤니케이션이 중요
- 미래의 일을 가정으로만 조율하면 합의가 어려움
- 때로는 일을 두 배로 처리하는 것이 더 효율적 (Vector DB: pgvector, ES, Redis, Milvus)

### 기록을통한문맥전달
- 여러 사람에게 피드백을 받기 위한 기록
- **피드백/설명 시간 = ROI**로 생각

## 문화

### 정확한피드백
- 아는 만큼 보인다
- 돕는 사람과 스스로 모두 부족한 부분을 파악할 수 있는 좋은 경험

### 지속적결과를위한방법제시
- 단순히 코드를 살펴보다가 개선점 찾기보다:
  - 응답 속도 목표를 설정
  - 높은 우선순위 작업부터 순차적 처리
  - 이상값은 알림 설정
- 도구 발전: async profiler → pipeline async profiler → summary profiler
- 프로세스 발전: 피드백 → 피드백 문서 → 내부 평가지표 → 회고

## 면접포인트
- "토스가 원하는 개발자?" → 결과를 만들어내는 사람, 데이터로 말하는 사람
- "기술 역량을 어떻게 증명하나?" → 라이브러리 내부 분석, 모범사례 비교, 성능 측정 결과
- "비즈니스 이해가 왜 중요한가?" → 기술은 비즈니스 성장의 수단, 임팩트 측정이 핵심
