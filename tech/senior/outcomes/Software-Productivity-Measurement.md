# 소프트웨어 생산성 측정의 함정

소프트웨어 생산성은 **시간·라인 수·티켓 수로 잴 수 없다**. 그럼에도 관리자는 측정하려 하고, 잘못된 측정은 역선택·품질 저하·시니어 이탈로 이어진다. 타임 트래킹이 대표적 안티패턴. 시니어는 **결과·영향·품질 기반 측정**으로 대안을 설계해야 한다.

## 왜 소프트웨어 생산성 측정이 본질적으로 어려운가

### 1. 작업 단위의 비균질성

- **버그 두 개의 난이도가 100배 차이** 나는 경우가 흔함
- "같은 제목의 티켓"이 코드 깊이·의존성·테스트 영향에 따라 10분일 수도 3일일 수도 있음
- 제조업의 "마요네즈 병 라벨 붙이기"처럼 **반복·균질**이 아님

### 2. 생산적 활동이 관찰 불가능

- 3시간 고민한 뒤 15분 코딩 → 외부에서는 **15분만 일한 것처럼 보임**
- 회의·코드 리뷰·문서화·멘토링은 개인 지표에 잡히지 않지만 팀 생산성엔 결정적
- 리팩토링으로 1,300줄 → 200줄로 줄인 성과가 "삭제"로만 잡힘 → 부정적 지표

### 3. 최고의 성과가 관찰 직전에 생김

- 어려운 아키텍처 결정은 **며칠 정체처럼 보이다가 한 번에 큰 진전**
- 버그 추적은 **막판 몇 분이 며칠을 해결**
- 시간 평균으로 평가하면 이런 spike 본질을 놓침

## 타임 트래킹이 해로운 5가지 이유

### 1. 난이도 비대칭을 무시
어려운 일을 잘 푼 시니어가 시간만 보면 "느린 개발자"로 보임.

### 2. 어려운 일 회피 유도
"시간 많이 쓰면 무능해 보인다"는 공포 → 쉬운 티켓만 선택하는 역선택. 조직 전체 난제가 미해결 상태로 누적.

### 3. 불안감·투명성 착각
시간을 보고한다고 진실한 보고가 되는 게 아님. 사람은 숫자를 맞추기 위해 **시간을 조작**한다 (Goodhart's Law).

### 4. 고민 시간의 평가절하
좋은 설계는 **코딩 전에 결정**된다. 코딩만 보면 이 가치가 사라짐.

### 5. 신뢰 기반 문화 훼손
"시간을 의심받는 대상"이라는 메시지가 자율성·몰입을 꺾음. 시니어일수록 빠르게 이탈.

## 제조업식 관리가 소프트웨어에 이식되는 이유

산업혁명이 제조업에 준 교훈 — **측정 가능한 생산성이 곧 경쟁력**. 이 성공 경험이 소프트웨어 시대에도 적용된다.

| 제조업 | 소프트웨어 |
|---|---|
| 균질 단위 (제품 1개) | 비균질 (같은 "기능"이 전혀 다름) |
| 반복 작업 | 반복이면 자동화됨. 남는 건 새 것 |
| 물리적 산출 관찰 가능 | 가치는 내부 구조·신뢰성·유지보수성에 숨어있음 |
| 시간 = 산출 | 고민·실험·폐기도 가치 창출 단계 |
| 제품은 "완성됨" | "소프트웨어는 결코 완성되지 않음" |

관리자의 **측정 욕구는 인간 본성**이지만, 잘못된 측정은 측정 안 하는 것보다 나쁘다.

## 흔한 잘못된 측정 지표 (Anti-Metrics)

- **라인 수 (LOC)** — 많을수록 좋다? 좋은 리팩토링은 LOC를 줄임
- **커밋 수** — 잘게 커밋하면 올라가지만, 진척과 무관
- **PR 개수** — 작게 자른 리팩토링 PR만 많이 내면 높게 나옴
- **작업 시간 / 티켓 소요 시간** — 난이도 비균질성 무시
- **버그 수정 수** — 버그를 애초에 안 만들면 0
- **출근 시간 / 사무실 체류 시간** — 원격근무 생산성 동등 증명 이후 무효
- **주간 업무 보고 분량** — 글 쓰기 잘하는 사람이 이김

공통점: **쉽게 측정되지만 측정 대상이 되면 왜곡**된다 (Goodhart's Law — "지표가 목표가 되면 더 이상 좋은 지표가 아니다").

## 올바른 방향 — 결과·영향·품질 기반 측정

측정을 완전히 포기하라는 건 아님. **측정 대상을 바꾸면** 된다.

### 결과 지표 (Outcome)
- 서비스 SLO 달성률, 사용자 영향 있는 장애 빈도
- 비즈니스 KPI 기여 (전환율·매출·리텐션)
- DORA 4 메트릭: 배포 빈도·리드 타임·MTTR·변경 실패율

### 영향 지표 (Impact)
- 팀 생산성의 복리 기여 (공통 모듈·도구·문서)
- 멘토링으로 팀원 역량 상승
- 기술 부채 감소로 후속 작업 가속

### 품질 지표 (Quality)
- 코드 리뷰 피드백의 깊이·유용성
- 테스트 커버리지 (단, 이것도 목표가 되면 왜곡)
- 포스트모템 품질·재발 방지 실효성

### 자기 보고 기반 (Self-Assessment)
- 1on1에서 본인이 말하는 **가장 힘든 문제와 그 해결**
- 동료·매니저 피드백의 질적 수렴
- 회고에서 본인이 제시하는 개선 액션의 실행률

## 시니어의 대응 원칙

조직이 타임 트래킹을 강제할 때 / 설계할 때 시니어가 취할 방침:

### 조직이 강제할 때
- **이유를 물어보기** — 규제·고객 정산 목적이면 수용 가능(외부 정당성). 그 외엔 대안 제시
- **집단 저항보다 개별 설명** — "이 지표가 왜 왜곡을 만드는가"를 데이터로 설명
- 정 안 되면 **이직 신호**로 해석 (장기적으로 건강한 조직이 아님)

### 시니어가 측정 체계를 설계할 때
- **한 지표를 목표로 삼지 않음** — 항상 3~4개 상보 지표 조합
- **측정 대상자와 합의** — 지표를 위에서 내리꽂지 말 것
- **분기마다 지표 점검** — 왜곡되지 않았는지 재검토
- 개인 지표보다 **팀·시스템 지표** 우선
- **정성 평가 병행** — 숫자가 놓치는 영역은 정기 리뷰로

## 자주 헷갈리는 포인트

- **"측정 없이는 개선도 없다" 격언의 남용** — 맞지만, 잘못된 측정은 개선이 아니라 왜곡을 부름
- **DORA 메트릭도 맹신 금지** — 좋은 출발점이지만 팀·도메인 특성으로 조정 필요
- **원격근무 = 저생산** 편향 — 팬데믹 데이터가 반박. 생산성은 근무 방식이 아닌 **명확성·자율성·도구**가 결정
- **개인 성과 평가 = 측정 강화**가 아님 — 좋은 1on1과 정성 평가가 더 정확한 경우 많음
- **측정 거부 = 성과 회피** 오해 — 측정을 **올바르게** 하자는 주장
- **타임 트래킹이 일부 맥락에서는 유용** — 컨설팅 빌링·고객 정산 등 외부 정당성 있는 경우

## 면접 체크포인트

- **타임 트래킹이 해로운 5가지 이유** (난이도 비대칭·역선택·Goodhart·고민 평가절하·신뢰 훼손)
- **제조업식 관리 vs 소프트웨어**의 본질 차이 (균질성·반복성·가시성)
- **Goodhart's Law** — "지표가 목표가 되면 더 이상 좋은 지표가 아니다"
- **Anti-Metrics 5가지 이상** (LOC·커밋 수·작업 시간 등)
- **DORA 4 메트릭** (배포 빈도·리드 타임·MTTR·변경 실패율)
- 결과·영향·품질·자기 보고의 **4축 대체 지표**
- 시니어가 잘못된 측정에 **개별 설명으로 대응**하는 원칙
