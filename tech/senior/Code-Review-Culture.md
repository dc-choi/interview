# 생산적 코드 리뷰 문화

코드 리뷰는 **"품질 게이트"**가 아니라 **"팀의 학습·정렬·예측 가능성 확보 장치"**. 규칙을 많이 만들수록 오히려 병목이 되고, 리뷰가 버그를 막아준다는 기대를 하면 테스트 문화가 약해진다. 조직 성숙도에 맞는 원칙과 프로세스를 설계하는 게 핵심.

## 핵심 명제

- **리뷰는 실수를 막는 도구가 아니다** — 버그 방지는 **테스트**의 몫. 리뷰는 가독성·설계 타당성·컨텍스트 공유
- **예측 가능성 > 엄격함** — "언제까지 리뷰될지 모름"이 가장 큰 비용
- **규칙은 최소로** — 경미한 변경은 자율, 중요한 변경은 명확한 기준
- **부담 분산** — 팀장·시니어에게 몰리면 조직 전체 속도가 한 명에 의해 정해짐

## 코드 리뷰의 실제 목적

흔히 "버그 방지"로만 이해되지만 실제 역할은 더 다층적.

| 목적 | 설명 |
|---|---|
| **변경사항 공유** | 팀원이 비즈니스 로직·아키텍처 변경을 인지. bus factor 감소 |
| **일관성 유지** | 코드 스타일·패턴을 팀 전체로 정렬 |
| **가독성 점검** | 작성자가 아닌 읽는 사람 관점에서 이해 가능한지 |
| **설계 타당성** | 구현 접근이 문제에 맞는지, 더 단순한 방법이 있는지 |
| **학습 기회** | 주니어가 시니어의 리뷰를 보며 성장, 시니어도 새로운 아이디어 접함 |
| **멘탈 모델 정렬** | "왜 이렇게 짰는가"의 맥락이 PR 설명·리뷰 대화로 팀에 전파 |

**버그 방지는 여기에 없다** — 사람은 실수하므로 리뷰로 막으려 하면 누락이 반드시 생김. 버그는 **테스트와 CI**로 방어.

## 리뷰 문화가 망가지는 전형적 패턴

팀이 커지면 나타나는 증상:

- **시니어에게 리뷰 집중** — 팀장이 하루에 20개 PR을 봐야 하는 상태
- **순차 지연의 누적** — 기술 스펙 리뷰 → 코드 작성 → 코드 리뷰 → 머지 경로가 병목
- **대기 시간 예측 불가** — "언제 리뷰될지 모름"으로 일정 산정 불가
- **2명 이상 approve 강제** 같은 규칙 → 사소한 변경도 며칠 대기
- **리뷰가 버그 방어선으로 착각됨** → 테스트 소홀
- **PR이 지나치게 커짐** → 리뷰어가 대충 보거나 미루게 됨

## 개선 원칙 5가지

### 1. 예측 가능한 리뷰 시간

- 구체 기준 예: **"요청 후 4시간 내 또는 다음날 오전까지 최초 응답"**
- 기한을 지킬 수 없으면 즉시 알림 → 요청자가 다른 일로 전환 가능
- 평균 대기 시간보다 **분산** 을 줄이는 게 체감에 중요

### 2. 예상 소요 시간 공지

PR 요청 시 리뷰어 부담을 표시:
- "5분", "15분", "1시간 이상" 같은 T-shirt 사이즈
- 리뷰어가 **우선순위**를 쉽게 판단 — 대형 PR을 바쁠 때 뒤로 미룰 수 있음
- PR 크기에 대한 자연스러운 압력 (1시간짜리 PR을 내면 반성하게 됨)

### 3. 자율적 생략 허용

- 오타·사소한 refactor·문서 오타 수정 등은 **self-merge** 가능
- 책임은 작성자가 지되, 팀은 이를 신뢰
- 경직된 "모든 PR 2명 approve" 규칙은 **생산성의 주적**

### 4. 부담 분산

- 경험 많은 멤버가 의도적으로 리뷰를 **내려놓고** 다른 팀원이 리뷰하도록
- 주니어를 리뷰어로 초대 — 학습 효과 + 부담 분산
- 리뷰어 풀이 N명이면 1명이 모든 PR을 보지 않음

### 5. PR 크기 관리

- PR은 **작고 자주**. 한 주제·한 맥락·300줄 이내가 이상적
- 큰 작업은 stacked PR 또는 feature branch + 단계적 머지
- "리뷰하기 쉬운 PR이 좋은 PR"

## 리뷰어 체크리스트 (추천)

리뷰할 때 보는 순서:

1. **PR 설명** — 무엇을·왜 하는지, 테스트 방법
2. **테스트 코드** — 테스트가 의도를 드러내는가, 커버리지가 합리적인가
3. **인터페이스 변경** — public API·DB 스키마·이벤트 payload 변경 여부
4. **핵심 로직** — 복잡도·예외 처리·경계 조건
5. **가독성** — 이름·구조·주석
6. **부가 사항** — 로그·메트릭·문서 업데이트

"오타·스타일 지적만 쏟아내는 리뷰" 금지 — 상위 지적부터.

## 리뷰 피드백의 톤

- 명령보다 **질문** — "이 경우에 어떻게 동작하나요?" vs "여기 고치세요"
- 의견과 사실 분리 — "저라면 이렇게 할 것 같습니다" vs "이건 틀렸습니다"
- 블로킹 여부 명시 — 사소한 의견은 `nit:`, 머지 전 해결 요청은 `MUST:` 또는 `blocking:`
- 좋은 부분 언급 — 문제만 지적하지 말고 잘한 부분도 짧게

## 스타트업 vs 대기업 관점

| 축 | 스타트업 | 대기업 |
|---|---|---|
| 우선순위 | 예측 가능성·속도 | 일관성·리스크 관리 |
| 규칙 엄격도 | 최소 (자율 허용) | 명시적 규칙 다수 |
| 리뷰어 수 | 1명 + 자율 생략 | 2명 이상 필수 |
| 대기 시간 | 4~8시간 목표 | 1~2영업일 허용 |

**단 하나의 정답 없음** — 조직 규모·리스크 프로파일·팀 성숙도에 맞춘 설계.

## 흔한 실수

- **엄격한 규칙 = 좋은 리뷰** 오해 — 규칙이 늘수록 병목 증가
- **시니어가 모든 PR 리뷰** — bus factor 1이 되고 팀 성장 저해
- **버그 방지를 리뷰에 의존** — 테스트·CI 강화가 먼저
- **리뷰를 교육 도구로만 봄** — 맥락 공유·설계 점검의 가치가 더 큼
- **PR을 크게 만들기** — 큰 PR은 꼼꼼히 리뷰 불가
- **`nit` 폭격** — 세부 스타일을 길게 지적하면 작성자는 리뷰 자체를 회피

## 면접 체크포인트

- **코드 리뷰의 실제 목적 6가지** (버그 방지가 아님)
- **예측 가능성 > 엄격함** 원칙
- **버그 방지는 테스트의 몫**이라는 관점
- 리뷰가 병목이 될 때의 **5가지 개선 원칙** (시간·크기·분산·자율·예측)
- 리뷰어 체크리스트 순서 (설명 → 테스트 → 인터페이스 → 로직 → 가독성)
- 리뷰 피드백의 **톤** (질문·분리·블로킹 명시)
- 스타트업·대기업 맥락 차이
