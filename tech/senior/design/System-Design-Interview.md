# 시스템 설계 인터뷰 대응

## 설계 접근 프레임워크

시스템 설계 질문은 계층별로 나누어 답변하면 체계적이다.

1. **엣지 & CDN** — 사용자 가까이에서 처리할 수 있는 것
2. **애플리케이션 계층** — 서비스 배포와 런타임
3. **데이터 전략** — 읽기/쓰기 분리, 소유권, 복제
4. **비동기 처리** — 알림, 이벤트, 큐
5. **검색 & 집계** — 읽기 최적화, 오프로딩
6. **라우팅 & 일관성** — UX 관점의 일관성 설계
7. **보안 & 개인정보** — 최소 수집, 암호화, 마스킹
8. **관측성** — SLO, 모니터링, 알림
9. **배포 전략** — 캔러리, 롤백, 스키마 변경
10. **비용** — 핵심 레버, 환경별 구성

## 사례: Mony-Calendar (글로벌 캘린더 서비스)

> 하루 수십만 명이 사용하는 글로벌 캘린더 서비스의 인프라 설계

### 도메인 특성 파악
- 읽기(월/주/일 조회)가 압도적, 쓰기(일정/공유/알림 CRUD)는 중간
- 알림의 정확한 시각 트리거가 핵심
- 서버 내부는 UTC, 클라이언트에서만 사용자 타임존 적용

### 1. 엣지 & CDN
- Route53 지연 기반 라우팅으로 가장 가까운 리전의 CloudFront로 유도
- 정적 자산/이미지/캐시 가능한 GET 응답을 CDN에서 처리하여 오리진 부하 감소
- **캐시 키 설계가 핵심**: 인증된 GET은 사용자 ID + 질의 범위를 캐시 키에 반영
- 공개 공유(ICS 링크)는 긴 TTL, 개인화 응답은 짧은 TTL + stale-while-revalidate
- 엣지에서 WAF/봇 차단, Authorization/쿠키는 캐시 키에 불필요하게 섞이지 않도록 분리

### 2. 애플리케이션 계층
- 리전마다 ECS로 API/워커/스케줄러를 독립 배포
- NestJS 모듈형 모놀리식 구조를 동일 태그로 리전 단위 배포
- 릴리즈는 한 리전에서 캔러리 시험 후 점차 확산
- 레이트리밋, 토큰 버킷, 세션 캐시는 리전별 Redis로 분리 (전역 상태 공유 없음)

### 3. 데이터 전략 — "사용자 홈 리전 소유권"
- 가입 시 지연/국가 기준으로 User.homeRegion을 결정, JWT 클레임에 포함하여 스티키 유지
- **쓰기**: 사용자 홈 리전의 DB로만 진입 → autoincrement 키 충돌 방지, 트랜잭션 무결성 보장
- **읽기**: Aurora Global Database 또는 DMS/CDC 기반 복제로 읽기 전용 엔드포인트 제공
- 복제 지연은 조회(허용 가능한 도메인)에 한정, 강한 일관성이 필요하면 홈 리전으로 프록시

### 인덱스 전략
- startDate, endDate, (startDate, endDate) 복합 인덱스로 월/주 범위 교차 쿼리 커버
- ownerId 결합 인덱스로 추가 최적화
- 공유: (calendarId, userId) 유니크 인덱스로 중복 방지
- 알림: alarmTime, (isTriggered, alarmTime) 인덱스로 대량 스케줄/취소/재예약 효율 보장
- 확장 시 ownerId/homeRegion 해시 기반 파티셔닝

### 4. 알림 처리 플로우
- CalendarAlarmService가 알림 생성 → NotificationService가 delay 계산 → BullMQ에 예약
- 알림 소유 사용자 기준으로 워커 리전을 결정하여 지연 최소화, 외부 게이트웨이 비용 최적화
- 홈 리전이 아닌 경우 해당 리전의 Redis/Bull 큐로 위임하거나 EventBridge로 "축약 알림 이벤트" 전달
- Job ID를 notification-<alarmId>로 고정하여 재예약/취소 **멱등성** 확보

### 5. 검색 & 집계
- 리전마다 OpenSearch를 독립 운영
- 캘린더 CRUD/공유 변경 시 홈 리전에서 CDC(DMS, Debezium)로 이벤트 추출 → 각 리전 인덱스로 송신
- 검색에 필요한 축약 이벤트만 전파하여 결합도를 낮춤
- 월/주/키워드 검색이 거의 실시간 반영, API의 DB 부하 감소

### 6. 라우팅 & 일관성 UX
- 쓰기 페이지(일정 편집/공유 관리) → 홈 리전 고정
- 읽기 페이지(달력 보기/공개 링크) → 가장 가까운 리전에서 처리
- "최근 변경이 몇 초 지연될 수 있음"을 UX로 명시
- 공유/권한 변경은 빈도가 낮으므로 홈 리전 처리 후 캐시/검색 무효화를 이벤트로 전파

### 7. 보안 & 개인정보
- 최소 수집 원칙, ICS 링크는 서명된 토큰 + 짧은 TTL
- 로그에 제목/메모 등 민감 필드를 남기지 않도록 공통 마스킹 적용
- 알림 전송 시 이메일/전화번호는 전송 직전까지 복호화하지 않는 설계
- 리전별 Secrets Manager/SSM에 동일 키로 비밀 저장, 수명/롤테이션은 중앙 관리

### 8. 관측성
- "리전 x 릴리즈 x 엔드포인트"로 슬라이스 가능한 대시보드
- **SLO (p95 기준)**:
  - GET /v1/calendars 범위 조회 지연/오류율
  - POST /v1/calendars/:id/shares 권한 변경
  - POST /v1/calendars/:id/alarms 알림 예약
- 알림 엔드투엔드 지연(예약→전송)과 실패 재시도율을 별도 지표로 관리
- BullMQ 큐 길이/지연, DB 복제 지연, OpenSearch 색인 지연 모니터링

### 9. 배포 전략
- 리전 독립 원칙, 확장-수축(expand-contract) 패턴으로 스키마 변경
- 기능 플래그로 점진 릴리즈, 한 리전 캔러리 → 확산, 문제 시 해당 리전만 롤백
- DB는 홈 리전 단일 라이터 원칙이므로 마이그레이션 리스크 낮음
- 페일오버: CloudFront 오리진 그룹, Aurora 리더 승격, Route53 헬스체크
- 쓰기 경로는 홈 리전 재지정이 수반되므로 **게임데이**로 전환 시간 정기 점검

### 10. 비용
- 핵심 레버: 달력 범위 조회의 캐시 적중률, 검색 오프로딩
- OpenSearch/Redis는 리전 독립으로 스케일/상한 관리
- 야간 저부하 시 워커/스케줄러 최소 태스크 수 하향 조정
- 스테이징: 대표 리전만 풀스택(검색/큐/복제), 나머지는 축소 구성

## 과제 후속 — 추가로 작업하고 싶은 것

실제 프로젝트라고 가정했을 때 개선하고 싶은 영역:

| 영역 | 내용 |
|------|------|
| 인증/인가 | JWT/OIDC 기반 Guard, OWNER/READ_WRITE/READ_ONLY RBAC 정책 중앙화 |
| 타임존 일관성 | 서버 전역 UTC 통일, DTO 검증 강화, DST/날짜 경계 테스트 케이스 추가 |
| 알림 안정성 | BullMQ 재시도/백오프 정책, 멱등성, 지연/만료 윈도 도입 |
| 에러 처리 | 에러 코드/메시지 규격 통일, 예외 필터에 런북 링크 포함 |
