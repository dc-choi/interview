# 엔지니어링 팀 기여 문화

엔지니어가 **자기 도메인을 넘어 팀 공통 영역에 기여하는 문화**. 코드 한 줄을 누가 쓰느냐의 문제가 아니라, **공통 플랫폼·도구·문서·설계를 누구나 개선할 수 있는가**의 문제. 시니어 영향력의 핵심 척도 중 하나.

## 기여의 범위 — 코드 너머

"기여(contribution)"라고 하면 기능 코드만 떠올리기 쉽지만, 실제로는 훨씬 넓다.

| 유형 | 예시 |
|---|---|
| **공통 모듈/라이브러리** | 사내 utility, SDK, 공통 인증 패키지 개선 |
| **개발 도구** | CLI, 코드 생성기, 린터 규칙, IDE 플러그인 |
| **문서화** | 온보딩 가이드, 런북, 아키텍처 ADR, API 문서 |
| **테스트 인프라** | 테스트 헬퍼, 픽스처, E2E 하네스, mock 서버 |
| **CI/CD 파이프라인** | 빌드 속도 개선, 캐싱, 배포 템플릿 |
| **관측성** | 로깅 표준, 메트릭 대시보드, 알림 정의 |
| **팀 프로세스** | 코드 리뷰 체크리스트, RFC 양식, 회고 템플릿 |
| **멘토링** | 1on1, 온보딩 페어링, 스터디 운영 |
| **문화 형성** | 회고·포스트모템·직설 피드백 문화 유지 |

기여는 **"내 업무 도메인보다 더 낮은 공통 엔지니어링 영역"** 으로 향하는 흐름. 이 흐름이 살아있으면 팀 전체 생산성이 복리로 올라간다.

## 기여 문화가 작동하기 위한 3가지 조건

기여하고 싶어도 할 수 없는 환경이 대부분이다. 문화가 실제로 작동하려면 다음이 필요하다.

### 1. 엔지니어링 플랫폼 — "안전하게 기여할 수 있는 기반"

기여자가 자기 업무 도메인 밖 코드를 건드릴 때 리스크가 크면 손대지 못한다. 이를 낮추는 3요소:

- **문서화** — 이 패키지가 무엇을 하고, 어떻게 쓰며, 어떤 호출자가 있는지
- **테스트 커버리지** — 바꾼 뒤 깨졌는지 즉시 확인 가능
- **내보내는 인터페이스 추적** — export된 API가 어디서 쓰이는지, 안 쓰이는 건 폐기 가능

**핵심 규칙**: 공통 패키지의 public API는 **사용처가 없으면 삭제 가능**해야 한다. 그러려면 usage tracking이 필수.

### 2. 낮은 기여 난이도 — "격리된 코드 + 점진적 적용"

결합도 높은 모놀리스에서는 작은 변경도 파급 효과가 예측 불가. 리스크 대비 동기가 꺾인다. 해결:

- **모듈 경계 명확화** — 공통 모듈을 독립 패키지로 분리, 의존 방향 일관성
- **점진적 적용 가능한 변경 방식** — 기존 API를 두고 새 API 추가 → 마이그레이션 경로 → 구 API 제거
- **실험 가능한 feature flag** — 위험한 개선을 일부 트래픽에 먼저 적용
- **롤백 용이성** — 기여자가 실수해도 복구 가능하다는 안전감

"소소한 기여가 거의 불가능한 코드베이스"는 기여 문화 이전에 **설계 문제**. 구조 개선이 선행.

### 3. 친화적 조직 문화 — "문제 제기자를 비판하지 않음"

기여의 가장 원초적 동기는 "도움이 되고 싶은 마음". 조직이 이걸 꺾으면 다음부터 아무도 입을 열지 않는다.

**기여를 죽이는 패턴**:
- 문제를 제기하면 **"그럼 네가 해결해봐"** 역공격 (제기 = 숙제 배정으로 해석됨)
- 개선 PR에 사소한 흠집 찾기로 반려
- 공통 모듈에 기여한 사람이 이후 **모든 버그의 담당자**로 낙인
- 시니어/창립자의 코드에 대한 비판이 금기
- "우리 팀 방식"이라는 이유로 변경 거부

**기여를 키우는 패턴**:
- 문제 제기 자체를 기여로 인정 (해결까지 강요 안 함)
- 작은 PR도 공식 채널에서 감사 표현
- 코드 리뷰에서 **"목적 > 문체"** 우선순위
- 공통 코드는 여러 메인테이너가 공동 소유
- 기여 내역을 성과 평가·승진에 반영

## 기여가 시니어 역량으로 평가되는 이유

주니어는 자기 티켓을 끝내면 된다. 시니어는 **자기 티켓을 끝내면서 팀의 미래 티켓을 쉽게 만든다**.

| 관점 | 주니어의 기여 | 시니어의 기여 |
|---|---|---|
| 범위 | 내 할당 작업 | 팀 전체 코드·도구·문서 |
| 영향 범위 | 개인 생산성 | 팀 생산성의 복리 |
| 시간 지평 | 현재 스프린트 | 6개월~2년 후 유지보수 |
| 측정 | 완료 티켓 수 | "내 부재가 느껴지지 않을 정도로 문서·자동화" |

시니어가 "팀 기여가 많다"는 건 단순 친절함이 아니라 **팀 bus factor를 낮추고 온보딩 속도를 높이는 레버리지 활동**. 성과 평가에서 중요하게 다뤄지는 이유.

## 기여 vs 자기 업무 분산 — 경계

모든 기여가 선은 아니다. 자기 본업을 소홀히 하면서 남의 코드를 고치는 행동은 오히려 평가 저하.

**건강한 기여의 기준**:
- 본업 임팩트 대비 **부담이 명확히 작은가** (본업의 20% 내외가 일반적)
- 기여가 **반복적 불편**을 해결하는가 (일회성 이슈는 제자리)
- **다른 메인테이너와 합의**되었는가 (기습 리팩토링 금지)
- 기여 PR이 **리뷰 부담**을 남들에게 떠넘기지 않는가 (작고 맥락 있는 단위)

**경계를 넘는 신호**:
- 본업 데드라인이 밀리는데 공통 모듈 리팩토링을 하고 있다
- 누가 요청하지도 않은 대규모 변경을 혼자 진행 후 머지 요청
- 기여 내역을 본업 실적처럼 과대 포장
- 기여 후 해당 모듈의 후속 유지보수를 외면

## 면접 답변 탄약

### "팀에 어떤 기여를 하셨나요?"

> "제 담당 기능 외에, 팀 전체가 반복적으로 겪는 불편을 줄이는 기여를 해왔습니다. 구체적으로는 (A) 온보딩 문서·런북 정비로 신규 입사자 적응 시간 단축, (B) 공통 (모듈/도구/테스트 헬퍼) 개선으로 개발 속도 향상, (C) (포스트모템 템플릿/코드 리뷰 체크리스트/회고 형식) 같은 프로세스 도입 — 이런 기여가 제 본업 20% 이내 시간으로 가능하도록 도구·모듈 경계를 먼저 정돈했습니다."

### "시니어의 조건이 뭐라고 생각하세요?"

> "본인 티켓을 끝내면서 동시에 **팀의 미래 티켓을 쉽게 만드는 사람**이라고 생각합니다. 구체적으로는 공통 모듈·도구·문서·온보딩 같은 영역에 반복적으로 기여해 팀 생산성의 복리를 만드는 역량입니다. 주니어는 자기 할당 작업을 잘하면 되지만, 시니어는 자기 부재가 느껴지지 않을 만큼 팀이 돌아가도록 만드는 것이 목표라고 봅니다."

### "기여 문화를 어떻게 만드시겠어요?"

> "세 가지를 봅니다. 첫째로 **안전하게 기여할 수 있는 플랫폼**(문서·테스트·인터페이스 추적)이 있는가. 둘째로 **기여 난이도가 낮은가**(모듈 경계·점진적 변경 가능성·롤백 용이성). 셋째로 **문제 제기자를 비판하지 않는 문화**가 있는가. 이 셋 중 하나라도 부족하면 기여 시도가 점차 줄어든다는 걸 여러 팀에서 관찰했습니다."

## 주의 포인트 (면접 시)

- **추상적 "팀워크 좋아요" 피하기** — 구체 기여 내역과 그 효과를 숫자로 (온보딩 시간 N일 단축, 빌드 N% 빨라짐)
- **본업 성과와의 균형** 함께 말하기 — 기여만 많고 본업 없으면 경계 없는 사람으로 보임
- **팀 기여가 없는 회사에서는 역효과** — 지원하는 회사 컬처를 보고 톤 조절
