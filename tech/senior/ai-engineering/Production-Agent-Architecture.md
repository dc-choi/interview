# 프로덕션 에이전트 아키텍처 (Production Agent Architecture)

LLM 단발 호출을 도구로 쓰는 단계를 넘어, **에이전트에게 일을 위임해 끝내게 만드는** 시스템 설계. 두 축이 필요하다.

- **하니스(Harness)** — 행동을 제어하는 장치 (분업·권한 분리·안전망·고가용성)
- **지식 체계(Knowledge)** — 결정론적 답변을 위한 정의 (지표 레지스트리·메타데이터·정책)

도구는 사람이 들고 쓰고, 동료는 위임받아 끝낸다. 프로덕션 에이전트의 목표는 **위임 가능한 수준의 신뢰**다.

## 컴포넌트 분업

단일 LLM이 모든 단계를 처리하면 컨텍스트가 폭증하고 비결정성이 쌓인다. 역할을 쪼개 토큰·권한·실패 범위를 분리한다.

| 컴포넌트 | 역할 | 비고 |
|---|---|---|
| **Listener** | 외부 채널(슬랙 등) 이벤트 수신 | 상시 실행, LLM 미사용 |
| **Dispatcher** | 잡담/유효 요청 분류, 중복·세션 관리 | 가벼운 모델(예: Sonnet) |
| **Parent Worker** | 요청 의도 정리·라우팅·재확인 | 6개 컴포넌트 컨텍스트(요청자·범위·개념·취소분 포함/제외·합의된 사실) |
| **Sub-agents** | 단순 분석·복잡 분석·검색·보고서·시각화·시트 출력 | 도메인별 워커, 권한 분리 |

Parent의 핵심 책무는 **"정보 부족 시 재확인"**이다. 모호한 요청을 추측으로 채우면 다층 안전망이 무용지물이 된다. 예: "거래 뽑아줘" → "숙박만요, 아니면 전체요?", "취소분 포함요?"로 분기.

## Lazy Load 컨텍스트

Parent가 모든 규칙을 메모리에 들고 있으면 세션당 수십만 토큰이 낭비된다. 헤드리스 모드(매 요청마다 새 세션 생성)에서는 더 치명적이다.

전략은 Parent가 규칙 본문이 아닌 **주소만** 보유하는 것.

```
Parent: "SQL 작업이면 #sql-rules 참조"
        "보고서면 #report-rules 참조"
              ↓
Sub-agent가 자기 규칙만 런타임에 로드
```

부작용 효과:
- **유지보수**: 한 워커 규칙만 수정 → 다른 워커 무영향
- **권한 분리**: 보고서 워커가 BigQuery 직접 접근 못 함 (규칙 자체가 분리됨)
- **토큰 절감**: 세션당 200K+ 절약 가능

## Defense in Depth (4겹 안전망)

LLM의 비결정성은 단일 가드로 막을 수 없다. 단계별 차단을 쌓는다.

| 레이어 | 위치 | 차단 대상 |
|---|---|---|
| **Role** | 워커 선언 단계 | 권한 외 도구 호출 (분석 워커가 Confluence 쓰기 불가) |
| **Gate** | 요청 처리 진입 | 컨텍스트 부족 → 재확인으로 분기 |
| **Rule** | 워커 내부 정책(자연어) | 읽기 전용 쿼리, 스캔 한도, 취소분 처리 규칙 |
| **Hook** | 실행 직전(코드) | 쓰기 명령·스캔 비용 초과·PII 컬럼·파티션 누락 차단 |

**Hook는 마지막 방어선**이다. 자연어 Rule이 무시되더라도 코드 레벨에서 SQL을 가로채 거절한다. 자연어 정책에만 의존하면 LLM이 "잠깐, 이번엔 예외야"로 빠져나갈 수 있다.

## 지식 체계: 정의의 명확성

같은 질문에 매번 다른 답이 나오는 원인은 보통 **정의 부재**다. 검증 가능한 지식을 명시적으로 인코딩한다.

### Metric Registry

핵심 지표를 YAML로 정의해 자연어 표현을 단일 산식으로 수렴시킨다.

```yaml
- id: gmv_total
  description: 총거래액 (Gross Merchandise Value)
  keywords: ["GMV", "거래", "총거래"]
  kind: sum
  expression: SUM(<payment_amount>)
  notes: 기본은 결제 거래액 기준. 취소분 반영 여부는 요청 맥락에 따라 분기.
```

- "거래 얼마야" = "GMV 뽑아줘" = "총거래액" → 모두 `gmv_total` 산식으로 수렴
- 정의 변경은 Registry 한 줄 수정 = 전사 일관성

### 조회 우선순위

1. **Metric Registry** (사람이 관리하는 핵심 지표 — 가장 강한 정의)
2. **DBT 메타데이터** (컬럼 단위 정의)
3. **직접 쿼리 탐색** (새 데이터 — 가장 약한 정의, 재확인 권장)

### Auto Memory 비활성화

LLM의 자동 메모리는 **무엇을 기억하고 있는지 불투명** → 회사 기준과 어긋난 답변이 누적된다. 명시적·검증 가능한 지식만 정책으로 인정한다.

## Eval 시스템과 운영 루프

배포 전 자동 검증(예: 100+ shell test/eval)을 4축으로 본다.

| 축 | 측정 |
|---|---|
| **정확성** | 수치가 맞는가 |
| **행동 경로** | 어떤 워커를 거쳤는가 |
| **안정성** | 같은 요청에 같은 답인가 |
| **형식** | 요청한 출력 형식인가 (엑셀/시트/이미지) |

운영 루프는 피드백을 지식·테스트로 환원한다.

```
사용자 피드백 ("이거 이상해요")
    ↓
근본 원인 분석 (Knowledge 부재? Rule 누락? Skill 결함?)
    ↓
Knowledge 수정 (Metric Registry / Rule / Skill)
    ↓
재발 방지 eval 추가 (테스트 케이스화)
```

지표·규칙·테스트가 동시에 성장하며 엣지 케이스 커버리지가 누적된다.

## 고가용성: 워커 풀

활성 워커와 대기 워커를 풀로 두고, Dispatcher가 리밋(주간/시간당)·API 에러를 감지하면 즉시 교체한다. 사용자는 "워커 죽었으니 새 스레드 파세요" 안내를 받지 않는다 → **동료로서의 신뢰** 유지.

비용 관점에서는 토큰당 API 호출 대신 Claude Code Max 다중 계정 + 헤드리스 모드 + Docker 병렬 실행으로 단가를 떨어뜨릴 수 있다.

## Closure-loop: 위임 경계

반복적 워크플로우는 에이전트에 **완전 위임**하고, 사람은 정의·정책·신규 에이전트 개발에 집중한다.

| 사람 담당 | 에이전트 담당 |
|---|---|
| 신규 지표 설계 | 기존 지표 추출·조합 |
| 정책 의사결정 | 정책 적용 |
| 새 에이전트·워크플로우 개발 | 기존 워크플로우 실행 |

Human-in-the-loop이 "사람이 매번 확인"이라면, Closure-loop은 "사람은 정의에만 개입, 실행은 닫힌 루프로 위임"이다.

## 사례

마이리얼트립 데이터 에이전트(2026): 슬랙 → Dispatcher(Sonnet) → Parent Worker → 분업 워커(Claude Code Max + Docker)로 구성. 라이브 7일 누적 2,000건+, 일 500~600건 처리. 분석가 인당 일 4~7건이던 반복 분석을 위임하고 신규 지표 설계·정책 결정에 인력 재배치.

## 면접 포인트

Q. 단발 LLM 호출과 프로덕션 에이전트의 차이는?
- 단발은 "도구", 에이전트는 "동료". 위임의 신뢰 수준이 다름.
- 신뢰는 **분업 · Lazy Load · Defense in Depth · 명시적 지식 · eval · 고가용성** 6축으로 만든다.

Q. LLM의 비결정성을 어떻게 통제하는가?
- 정의의 명확성(Metric Registry/DBT)으로 **입력 분산**을 줄이고
- 4겹 안전망(Role/Gate/Rule/Hook)으로 **실행 분산**을 차단하며
- Eval 회귀 테스트로 **변경 분산**을 검출한다.

Q. 멀티 에이전트는 왜 필요한가? 단일 거대 에이전트로 안 되는가?
- 컨텍스트 폭증·권한 비대화·실패 범위 확산.
- 분업하면 워커별 권한 최소화, 규칙 Lazy Load, 장애 격리가 가능하다.

Q. 자동 메모리를 끄는 이유는?
- 무엇을 기억하는지 불투명 → 회사 기준과 어긋난 답변이 누적될 위험. 명시적·버전 관리되는 정의만 정책으로 인정한다.
