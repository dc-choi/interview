# AI 시민 개발 — 도메인 전문가가 직접 만드는 내부 도구

시민 개발(Citizen Development)은 현업 담당자(비개발자)가 자기 업무의 도구를 직접 만드는 패턴이다. 로우코드 시절에는 플랫폼이 허용하는 범위에 갇혔지만, AI 코딩 에이전트가 구현을 대신하면서 실제 서비스 수준(웹 앱, 외부 API 연동, 인증)까지 범위가 확장됐다. 핵심 명제: **AI가 구현 비용을 무너뜨리면 희소 자원은 코딩이 아니라 도메인 지식이 되고, 문제를 가장 잘 아는 사람이 도구를 쥐는 것이 최적이 된다.**

## 내부 도구의 전통적 딜레마

- 사내 불편(흩어진 시트, 반복 질문, 수동 예약)은 **보이지 않는 소통 비용**으로 쌓이지만, 개별 건의 ROI가 작아 보여 개발팀 백로그에 진입하지 못한다
- 결과적으로 현업 담당자가 게이트키퍼가 된다 — 정보 요청이 모두 한 사람을 거치고, 그 사람의 시간이 반복 응대로 소진된다
- 개발자가 만들자니 도메인 파악 비용이 크고, 현업이 만들자니 구현 능력이 없었다 — AI가 이 두 번째 벽을 제거

## 도메인 전문가의 우위

구현이 싸지면 남는 병목은 문제 정의다. 현업 담당자는 개발자가 갖기 어려운 세 가지를 이미 안다.

| 우위 | 내용 |
|---|---|
| 불편의 우선순위 | 어떤 반복 질문이 가장 잦은지, 무엇부터 풀어야 체감이 큰지 |
| 데이터의 실체 | 어느 시트에 어떤 필드가 있고 무엇이 최신인지 (파편화의 지도) |
| 조직 수용성 | 어느 팀의 협조가 필요하고 누가 이해관계자인지 |

## AI의 역할 — 초안 기계가 아니라 동료

비개발자 주도 구축에서 AI가 실제로 담당하는 것은 코드 생성만이 아니다.

- **기술 초안** — 코드, 문서, 구조 설계
- **판단 근거 제공** — 보안 트레이드오프(아래 사례)처럼 선택지의 위험을 설명해, 비개발자가 보안팀과 같은 언어로 대화할 수 있게 함
- **이해관계자 식별** — 어떤 팀에 무엇을 요청해야 하는지 함께 정리
- **계획 우선 진행** — 구현 전에 Plan Mode로 전략을 합의하고 시작 ([[Claude-Code-Fundamentals|Claude Code 기초]])

## 기술 판단은 생략되지 않는다 — 두 사례

비개발자가 만들어도 아키텍처와 보안 판단은 그대로 등장한다. AI가 근거를 제공하고 사람이 선택한다.

### 사내망 봇의 인바운드 노출 문제

- 문제: 슬랙 슬래시 커맨드는 공개 Request URL을 요구하는데, 시스템은 사내망 전용 — 외부 노출은 보안 범위를 키움
- 해결: **Slack Socket Mode** — 봇이 슬랙으로 아웃바운드 WebSocket을 먼저 열어, 공개 엔드포인트 없이 커맨드를 수신 ([[WebSocket|WebSocket]]의 아웃바운드 연결 패턴)

### 대표 계정 하나로 전 직원 예약 생성 문제

- 문제: 초기 구현이 관리자 계정 하나로 모든 회의실 예약을 생성 → 모든 예약의 주최자가 담당자로 기록, 전사 알림이 한 계정에 집중, 도메인 전체 위임은 키 하나로 전 직원 캘린더에 접근하는 마스터키 리스크
- 해결: **사용자별 OAuth + 최소 스코프**(calendar.events만), 토큰은 세션별 암호화 보관, 퇴사 시 권한 자동 정리 ([[OAuth2|OAuth 2.0]]의 위임 범위 설계)

## 조직 수용 전략 — 강제 대신 취지

내부 도구는 채택이 절반이다. 데이터 통일(예: 프로필 사진)이 필요할 때 변경을 강요하면 반발이 생기고, 일관된 정보 제공이라는 취지와 가이드라인(AI 생성 프로필 제안 등)을 전달하면 자발적 참여가 따라온다. 구성원 각자가 참아온 불편을 공동의 문제로 재정의하는 것이 확산의 동력.

## 효과의 구조

- 조회 작업이 담당자 경유에서 **셀프서비스**로 전환 — 게이트키퍼 병목 제거
- 담당자의 반복 응대 시간이 판단이 필요한 업무로 재배치
- 파편화된 데이터(여러 시트)가 단일 진입점으로 통합

## 한계와 전제

- **운영 책임** — 만든 뒤의 유지보수, 장애 대응 주체가 필요하다. 개발자의 코드 리뷰와 피드백을 받는 협업 구조가 안전판
- **보안 거버넌스** — 사내 데이터에 접근하는 도구이므로 보안팀을 초기부터 참여시켜야 한다. 비개발자일수록 최소 권한, 노출 면적 판단을 AI 근거와 함께 검증받을 것
- 개발자 대체 서사가 아니라 [[AX-Transformation|AX 조직 전환]]의 산출자에서 판단자로의 역할 이동이 비개발 직군까지 확장된 형태 — 문제 정의자가 주도하고 AI가 산출한다

## 면접 체크포인트

- AI가 구현 비용을 낮추면 병목이 어디로 이동하는가 (문제 정의, 도메인 지식)
- 내부 도구가 개발 백로그에 못 들어가는 구조적 이유와 시민 개발이 그 틈을 메우는 방식
- 비개발자 주도 구축에서도 등장하는 기술 판단 사례 (인바운드 노출 회피, 최소 권한 위임)
- 내부 도구 확산에서 강제와 취지 공유의 차이
- 시민 개발의 운영 책임, 보안 거버넌스 전제

## 사례

아임웹 GA(총무) 담당자가 Claude와 함께 좌석 배치도, 회의실 현황과 예약, 슬랙 커맨드 조회, 인사 데이터 통합 대시보드를 구축. 첫 GitHub PR을 올린 비개발자가 Socket Mode 채택과 개별 OAuth 전환 같은 기술 판단을 AI의 근거 설명과 함께 통과시켰고, 개발자들의 리뷰를 받으며 운영 중.
