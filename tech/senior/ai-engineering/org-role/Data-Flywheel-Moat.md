# 데이터 플라이휠과 제품 해자

AI 제품의 기반 모델과 기능은 빠르게 복제될 수 있다. 사용 과정에서 나온 **검증 가능한 피드백과 고객별 운영 맥락**을 제품 개선으로 되돌리는 루프는 더 오래 축적되는 경쟁력이 될 수 있다. 다만 데이터 자체가 곧 해자는 아니다. 합법적인 접근 권한, 신호 품질, 개선 속도와 사용자가 체감하는 가치까지 연결되어야 한다.

## 데이터 플라이휠

```text
도입
  → 실제 작업과 상호작용
  → 결과 검증 신호
  → 평가, 라우팅과 제품 개선
  → 품질과 고객 적합도 향상
  → 재사용과 추가 도입
```

핵심은 사용량보다 **결과를 판정할 수 있는 신호**다.

- 코드 제안 수락, 거절과 후속 수정
- 상담 답변 뒤 재문의, 에스컬레이션과 해결 여부
- 운영 조치 뒤 경보 해소, 롤백과 재발 여부
- 초안이 최종 산출물에 남은 비율과 사람이 고친 부분

클릭과 체류 시간처럼 해석이 여러 개인 지표만 모으면 제품이 왜 좋아지거나 나빠졌는지 알기 어렵다. 업무 결과와 연결된 신호일수록 개선 루프가 짧고 명확해진다.

## 문제 복잡도와 도입 난이도

AI 제품의 데이터 축적 조건은 문제 난이도와 도입 난이도를 함께 봐야 한다.

| | 도입이 쉬움 | 도입이 어려움 |
|---|---|---|
| 문제가 비교적 단순함 | 사용량은 빨리 늘지만 대체도 쉬움 | 기업 통합과 고객별 규칙이 전환 비용을 만들 수 있음 |
| 문제가 복잡함 | 반복 사용과 세밀한 피드백이 빠른 개선을 도울 수 있음 | 가치가 크지만 신호가 적고 검증, 구축 주기가 길어짐 |

도입이 어렵다는 사실만으로 해자가 생기지는 않는다. 불필요한 구축 비용은 단순한 마찰이다. 통합 과정에서 고객의 승인 규칙, 예외 처리, 시스템 관계와 성공 기준을 학습하고 이를 안정적으로 재사용할 때 비로소 고객별 맥락이 된다.

## 범용 데이터와 고객별 맥락

- **범용 신호**는 여러 사용자에게 공통으로 적용할 수 있어 제품 전체의 품질을 높인다.
- **고객별 맥락**은 조직의 용어, 워크플로우, 권한과 예외를 반영해 교체 비용을 높일 수 있다.

둘을 섞으면 개인정보, 계약과 과적합 문제가 생긴다. 테넌트별 데이터 경계, 학습 사용 동의, 삭제와 내보내기 정책을 먼저 정하고 범용화할 수 있는 신호만 분리해야 한다.

## 해자가 되기 위한 조건

1. **신호의 정체성** — 누가 어떤 입력과 버전에서 어떤 결과를 판정했는지 추적한다.
2. **검증 가능성** — 수락, 거절만 보지 않고 실제 업무 결과와 연결한다.
3. **짧은 개선 지연** — 수집한 신호가 평가와 제품 변경으로 돌아오는 시간을 줄인다.
4. **권리와 거버넌스** — 수집, 학습, 보존과 삭제 권한을 계약과 정책으로 보장한다.
5. **체감 가능한 개선** — 모델 점수뿐 아니라 완료 시간, 오류와 재작업이 줄어야 한다.
6. **도달 경로** — 개선된 제품이 다시 사용자에게 닿아야 다음 신호가 생긴다.

마지막 조건 때문에 [[Distribution-As-Moat|도달력]]과 데이터 플라이휠은 대체 관계보다 연결 관계에 가깝다.

## 실패 패턴

- 데이터 양을 해자로 착각하고 판정 기준 없이 로그만 쌓음
- 쉬운 사용자와 성공 사례만 남아 실제 실패 분포를 가림
- 자동 수락을 품질 신호로 오해해 편향을 증폭
- 고객 데이터를 학습에 쓸 권리가 없는데 제품 전략은 이를 전제로 함
- 고객별 규칙에 과적합해 다른 고객에게 확장하지 못함
- 긴 평가 주기 때문에 개선 전에 모델, 시장과 요구가 바뀜
- 일반 기능이 기반 모델에 흡수되는데 고객 문제와 워크플로우 학습은 남지 않음

## 측정 체크포인트

- 결과를 자동 또는 사람 검토로 판정할 수 있는 작업의 비율
- 수락 뒤 재수정, 롤백과 에스컬레이션 비율
- 신호 수집부터 평가, 배포까지 걸리는 시간
- 신규 고객과 기존 고객의 품질 차이, 초기 적응 시간
- 고객별 규칙이 늘 때 운영 비용과 오류율이 어떻게 변하는지
- 데이터 삭제, 내보내기와 학습 제외 요청을 실제로 이행할 수 있는지
