# Software 3.0

소프트웨어 개발의 세 번째 패러다임. Andrej Karpathy가 제시한 분류에서 **LLM이 프로그램의 실행 주체가 되는 시대**를 가리킨다. 전통 엔지니어링이 없어지는 게 아니라, **어떤 레이어에서 일하느냐**가 바뀌는 것.

## 세 패러다임의 구분

| 버전 | 프로그램의 정체 | 작성 주체 | 표현 매체 |
|---|---|---|---|
| **Software 1.0** | 명시적 로직 | 인간 개발자 | 코드 (Python·Java·C++) |
| **Software 2.0** | 신경망 가중치 | 학습 알고리즘 | 데이터셋 + 모델 |
| **Software 3.0** | 프롬프트 | 인간 + LLM 협업 | **자연어 명세** |

- Software 1.0: "어떻게(How)"를 코드로 정의. 결정론적
- Software 2.0: 데이터로 모델을 학습시켜 특정 함수를 근사. 2010년대 딥러닝 붐
- Software 3.0: LLM에 **"무엇을(What)" 원하는지 자연어로 전달** → LLM이 실행 · 조합 · 도구 호출로 구현

세 버전은 공존한다. Software 3.0 시스템 내부에도 1.0(도구·런타임)과 2.0(LLM 자체)이 살아 있다.

## Harness — 3.0을 실제 작동시키는 것

LLM만으로는 파일 읽기·API 호출·DB 접근이 불가능. **LLM을 실제 행동 가능한 에이전트로 만들어주는 주변 인프라**가 필요하고 이것을 Harness라 부른다. ([[Harness-Engineering|하네스 엔지니어링]] 참고)

Claude Code 같은 도구가 Harness의 구체 예시. LLM + 파일 시스템 접근 + 터미널 실행 + MCP 연결 + Sub-agent 구조로 "명령을 입력하면 작업하는 에이전트"가 된다.

## 전통 아키텍처와의 매핑

Software 3.0 에이전트 시스템은 **고전 레이어드 아키텍처와 놀라울 정도로 닮아 있다**.

| 에이전트 구성 요소 | 레이어드 아키텍처 대응 |
|---|---|
| Slash Command | Controller (요청 진입점) |
| Sub-agent | Service Layer (여러 컴포넌트 조율) |
| Skills | 단일책임원칙 기반 컴포넌트 |
| MCP (Model Context Protocol) | Infrastructure / Adapter (외부 시스템 연결) |
| CLAUDE.md | 프로젝트 맥락 (설정·규칙·제약) |

**의미**: 좋은 에이전트 시스템 설계는 **기존 소프트웨어 설계 원칙(응집도·결합도·추상화·SRP)**을 그대로 요구한다. 아키텍처 지식이 Software 3.0 시대의 무기가 된다.

## Software 1.0과의 결정적 차이: 질문할 수 있다

전통 코드는 **모든 분기를 사전 정의**해야 한다. `if (A) ... else if (B) ...`. 예외 케이스를 빠뜨리면 에러.

Software 3.0 에이전트는 **Human-in-the-Loop**로 실행 중에 "A와 B 중 뭘 원하세요?" 질문 가능. 사전 정의할 분기 수가 크게 줄어든다. 이건 레퍼런스 없는 새로운 능력.

동시에 이 능력이 **신뢰의 경계**를 만든다 — 에이전트가 질문 없이 실행하면 안 되는 지점(금전 거래, 데이터 삭제, 외부 공개)을 명시해야 함.

## 실무 함의

### 1. 스펙 작성이 핵심 스킬
3.0에서는 "잘 쓴 프롬프트 = 잘 설계된 프로그램". 구현을 LLM이 하니 **스펙 작성 능력**이 곧 생산성. ([[Agent-Spec-Writing|에이전트 스펙 작성법]] 참고)

### 2. 아키텍처가 도구 선택이 아니라 "대화 경로 설계"
"어떤 슬래시 커맨드 → 어떤 sub-agent → 어떤 skill → 어떤 MCP"를 연결할지가 곧 아키텍처.

### 3. 결정론과 비결정론의 혼합
Software 1.0 부분은 여전히 결정론적이어야 함(런타임·테스트). LLM이 끼어드는 경계에서 **검증 루프(Verify)** 로 비결정론을 감싸야 함.

### 4. 개발자의 가치 이동
- 약해지는 것: 보일러플레이트 코드 작성, CRUD 반복 구현
- 강해지는 것: **시스템 경계 정의, 스펙 작성, 검증 설계, 예외 판단**
- [[Developer-Role-AI-Era|AI 시대 개발자 역할]] 참고

## 흔한 오해

- "LLM이 모든 걸 대체한다" — 에이전트는 Harness·도구 없이는 아무것도 못 함. 전통 인프라가 더 중요해짐
- "3.0이 오면 1.0은 사라진다" — 반대. 3.0의 토대로 1.0이 더 많이 쓰임
- "자연어만 알면 된다" — 자연어로 스펙 쓰는 것도 아키텍처 지식이 있어야 잘 됨
- "AI가 시니어 개발자를 만든다" — AI는 시니어의 판단력을 **증폭**할 뿐, 대체 못 함

## 면접 체크포인트

- Software 1.0/2.0/3.0의 차이를 한 문장씩으로 설명
- Harness가 왜 LLM만으로 부족한지
- 에이전트 시스템이 레이어드 아키텍처와 닮은 이유
- Software 3.0에서 시니어 개발자의 가치가 오히려 커지는 이유
- Human-in-the-Loop가 없으면 3.0 에이전트가 무엇을 못 하는가
