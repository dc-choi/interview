# AI-Native 조직 (AI-Native Org)

[[AI-Native-System|AI Native 레포]]가 개인의 코딩 생산성을 풀었다면, 다음 경계는 조직 전체가 AI를 호출하고 검증하고 기억하는 **공용 실행 계층**이다. 개인 IDE 안의 AI는 세션 내 단발이고, 협업 흐름(요구→승인→구현→검증→기억)과 세션 밖 기억을 갖지 못한다. 이를 푸는 것이 팀의 공용 인터페이스 위에 올린 전사 AI 실행 하네스다.

## 왜 조직 단위인가 — 개인 IDE의 한계

- 세션 내 단발이라 협업 맥락(누가, 왜, 어떤 제약으로 요청했는지)이 없다
- 세션이 끝나면 결정과 발견이 사라져 매번 재설명해야 한다
- 비개발자는 IDE 안의 AI에 접근할 수 없다

조직 실행 계층이 주는 것: 요구사항과 승인, 피드백, 결과가 한 스레드에 시간순으로 기록되어 감사 가능성이 자동 확보되고, 세션 밖 지식이 누적되며, 누구나 같은 인터페이스로 접근한다.

## 설계 원칙 4가지

1. **기존 흐름을 따른다** — 새 UI로 사람을 옮기는 대신, 조직이 이미 일하는 인터페이스(공용 팀챗) 안으로 AI를 넣는다. 채택 비용이 낮고, 협업이 이미 스레드 단위라 맥락이 자연스럽게 쌓인다.
2. **세션 밖 메모리는 필수** — pod 재시작이나 세션 종료로 기억이 날아가면 매번 재설명이 발생하고 조직 학습이 불가능하다. 기억을 세션 바깥 계층에 둔다.
3. **이미 좋은 것은 재구현하지 않는다** — 성숙한 코딩 루프(계획→수정→테스트→Git)는 재활용하고, 조직 실행 계층(인터페이스, 상태 관리, 메모리)에만 집중한다. 하네스 위에 하네스를 얹는 전략이다.
4. **복구가 기능만큼 중요하다** — 중복 처리, stale lock 회수, pod 중단 중 상태 복구가 시스템 신뢰도를 결정한다. 신규 기능보다 실패 복구에 더 많은 시간이 든다.

## 아키텍처 — 팀챗 위 멀티 에이전트 플랫폼

실행 흐름은 단방향 파이프라인이다.

팀챗 webhook 수신 → API 서버(stateless: 스레드 문맥 복원, 상태머신 관리, 태스크를 PostgreSQL에 저장) → Orchestrator(DB 스캔, 워커 pod 생성과 회수) → Kubernetes Job 워커 pod(코딩 에이전트 코어 실행: 탐색, 수정, 테스트, PR) → 결과를 팀챗 스레드에 보고.

| 컴포넌트 | 역할 |
|---|---|
| API 서버 (stateless) | 스레드 문맥 복원, 상태머신 관리, 태스크 저장 |
| Orchestrator | DB를 스캔해 워커 pod 생성과 회수 |
| Conversation Agent | 팀챗 메시지 해석, 스레드 root와 reply history로 문맥 회수 |
| Specialized Agent | 도메인 작업 (브라우저 자동화, 로그 분석 등) |
| Worker (K8s Job) | 코딩 에이전트 코어 루프 실행 |

저장소: PostgreSQL(태스크와 상태), Redis(중복 처리와 lock), 벡터 DB(의미 검색), 그래프 DB(심볼 의존성), 지식 베이스(결정과 규칙).

### 상태머신 제어 + Human-in-the-loop

암묵적 협업 규칙을 명시적 상태 전이로 강제한다.

PLAN_REVIEW → (사용자 승인) → IMPLEMENTING → CODE_REVIEW → COMPLETED

각 전이에 사람의 개입점이 고정된다: 계획 승인(구현 전 검증), 코드 리뷰(PR 생성 후 피드백), 머지 판단(최종은 사람). 개입 모드는 유연하다 — 질문만 하고 끝낼 수도, 같은 스레드에서 구현으로 전환할 수도, 진행 중 언제든 제약을 추가할 수도 있다.

핵심은 하이브리드 결정론이다. **결정론적 상태 전이 + 비결정론적 대화 해석**을 결합한다. 실제 사용자 대화는 비선형(질문→실행→제약 추가→재실행)이라 순수 결정론으로는 담을 수 없다. 결정론을 포기하는 게 아니라 그 경계를 어디에 둘지 찾는 일이다.

### Kubernetes Job 워커 격리

태스크당 일회성(ephemeral) pod로 워커를 실행한다. init container가 대상 레포를 clone하고, main container가 코딩 에이전트를 실행하며, 작업이 끝나면 자동 정리된다. 이점은 실행 환경 격리, 실패 격리, 역할별 이미지 차별화다. 같은 webhook이 여러 pod에 중복 유입되는 것은 Redis 기반 dedup으로 막고, stale lock 회수로 재시도 가능성을 보장한다.

### MCP 프록시 — 자격증명 중앙화

워커 pod에 자격증명을 직접 두지 않는다. pod 안의 MCP 서버가 내부 API를 거치고, 자격증명과 조직별 권한, 설정은 서버에서 중앙 관리한 뒤 외부 API를 호출한다. 권한이 중앙화되어 감사가 쉽고 pod 유출 위험이 줄어든다. → [[MCP]]

## 메모리 계층 — 조직 학습의 누적

세션 밖 기억을 4계층으로 둔다.

| 계층 | 범위 | 저장 내용 |
|---|---|---|
| L1 | 현재 세션 | 진행 중 컨텍스트 |
| L2 | 태스크 | 결정, 발견 |
| L3 | 레포/서비스 | 코드 패턴, 아키텍처 |
| L4 | 조직 | 규칙, 제약, 경험 |

검색은 세 방식을 동시에 운영한다: 벡터 검색(코드와 문서의 의미 기반), 그래프 검색(심볼 의존성 관계 추적), 메모리 검색(축적된 결정과 규칙 조회). 저장 대상은 어떤 태스크에서 어떤 결정이 내려졌는지, 반복되는 레포 구조와 패턴, 조직 차원의 제약과 규칙, 관찰된 실패와 사이드이펙트다.

주의: 잘못된 규칙이 쌓이면 오히려 독이 된다. 메모리 정확성은 명시적 검증과 짝을 이뤄야 한다. 불투명한 자동 메모리를 끄고 검증된 정의만 정책으로 인정하는 [[Production-Agent-Architecture|프로덕션 에이전트 아키텍처]]의 Auto Memory 논의와 같은 긴장이다.

## 활용 양상 — 코딩을 넘어

같은 스레드 인터페이스 위에서 개발 외 작업도 처리된다.

- **보안 전수조사**: 취약점 영향 프로젝트를 자동 트리아주하고 역할별 액션 아이템으로 분배
- **배포 직후 관측**: 관측 도구의 에러율과 latency 이상징후를 자동 확인해 후속 검토 항목만 필터
- **조직 지식 조회**: 스펙 문답을 비개발자도 같은 인터페이스로 질의
- **자기 수정**: 플랫폼이 자기 레포를 한 스레드에서 계획→승인→구현→PR→머지까지 완성

운영 알람 채널의 자동 트리아주가 도입 확산의 주된 동력이 된다.

## 트레이드오프, 한계

- 결정론과 대화형의 균형 — 경계 설정 자체가 설계의 본질
- 복구 로직이 기능 개발보다 시간이 더 든다 (dedup, stale lock, pod crash 복구)
- 확장성 — 수천 태스크 동시 처리 시 DB와 Orchestrator가 병목
- 메모리 정확성 — 잘못된 규칙 누적 위험
- 보안과 권한 — 역할별 접근 제어와 감사 로깅 필요

## AI-Native 조직의 6요소

조직이 AI를 실제로 굴리려면 여섯 가지가 함께 있어야 한다.

1. 좋은 레포 ([[AI-Native-System|AI Native 코드베이스]])
2. 강제되는 규칙 (lint, 테스트, 아키텍처 검증)
3. 조직 지식 (문서, 아키텍처 결정)
4. 세션 밖 메모리 (L2~L4 계층)
5. 공용 인터페이스 (팀챗 스레드)
6. 조직 문화 (AI를 실제로 쓰려는 의지)

## 면접 체크포인트

- 개인 IDE의 AI와 조직 AI 플랫폼의 차이 (협업 맥락 + 세션 밖 메모리 + 접근성)
- 왜 새 UI 대신 기존 팀챗 위에 올리나 (채택 비용, 감사 가능성, 맥락 누적)
- 상태머신과 대화 해석을 결합하는 이유 (비선형 대화에서 결정론의 경계 찾기)
- K8s Job ephemeral 워커 격리의 이점 (실행, 실패 격리 + 역할별 이미지)
- 복구가 기능만큼 중요한 이유 (dedup, stale lock, crash 복구가 신뢰도 결정)
- AI를 전사 워크플로우로 쓰는 조직에서 백엔드가 만드는 것 = 이 실행 인프라(상태 관리, 워커 격리, 메모리 계층)
