# Claude Code 내부 구조 — 에이전트 루프, 도구 파이프라인, 설계 패턴

프로덕션 코딩 에이전트가 실제로 어떻게 구현되는지의 레퍼런스 아키텍처 (약 1,884개 TypeScript 파일 분석 기준. TUI는 Ink 기반 React, 상태는 Zustand, 번들은 bun). 설계 3원칙은 안전성(위험 작업 차단), 성능(스트리밍, 병렬, 캐싱), 확장성(도구, 스킬, 플러그인, MCP).

## 쿼리 루프 — "버그 고쳐줘" 한 번의 실제

전체 구조는 시작 → 쿼리 루프 → 도구 실행 → 렌더링 4단계이고 심장은 쿼리 루프다. 사용자 요청 하나에 Read → Edit → Bash(test) 같은 여러 API 턴이 자동 반복된다. 턴마다 5단계:

1. **전처리 — 컴팩션 3종**: Snip(오래된 메시지 제거), Microcompact(tool_use 블록 축소), Auto-Compact(임계값 = 컨텍스트 윈도우 - 13,000토큰에서 서브에이전트가 요약. 요약 후 참조 상위 5개 파일을 50K 토큰 예산으로 복원하고 스킬을 25K 예산으로 재주입. 3연속 실패 시 서킷 브레이커)
2. **스트리밍 API 호출** (과부하 시 폴백 모델로 전환)
3. **에러 보류와 복구**: 복구를 먼저 시도하고 실패할 때만 사용자에게 표면화. 출력 토큰 초과는 상한을 단계 확대하며 최대 3회 이어쓰기
4. **도구 실행**
5. Stop 훅과 예산, 턴 한도 확인 후 tool_use가 남았으면 반복

## 도구 시스템

- 45개 이상의 내장 도구가 공통 인터페이스를 구현: Zod inputSchema, checkPermissions, isConcurrencySafe, isReadOnly, render
- **도구 등록 순서는 고정** — 순서가 바뀌면 시스템 프롬프트가 달라져 프롬프트 캐시가 무효화된다. 캐시 친화성이 코드 구조의 제약이 되는 사례
- 실행 파이프라인 10단계: 이름 조회 → 중단 확인 → Zod 검증 → PreToolUse 훅 → 권한 → 실행 → API 형식 매핑 → **대용량 출력은 디스크에 영속하고 참조만 전달** → PostToolUse 훅 → 텔레메트리 ([[Tool-Output-Filtering|도구 출력 관리]]의 구현측)
- 동시성 파티셔닝: 연속된 읽기 전용 도구(Read, Grep, Glob)는 최대 10개 병렬, 쓰기 도구(Edit, Bash, Write)는 단독 순차 배치. API 스트리밍 중 이미 도착한 tool_use를 먼저 실행하는 스트리밍 실행기로 지연을 숨긴다
- BashTool: tree-sitter로 명령을 AST 분석해 **허용 목록 노드만 통과시키는 fail-closed** 설계 (Bash 보안 모듈만 888KB). 15초 초과 명령은 백그라운드 태스크로 자동 전환

## 신뢰성 계층

- API 클라이언트: 프롬프트 캐싱 1시간, 에러별 차등 재시도 — 레이트리밋은 대기 시간이 짧으면 캐시를 보존한 채 대기, 용량 초과 3연속이면 폴백 모델, 인증 만료는 토큰 강제 갱신, 무인 세션은 지수 백오프로 최대 6시간 재시도
- 중단 내성: API 호출 전에 트랜스크립트를 디스크에 저장 — 크래시해도 대화가 유실되지 않는다

## 시작과 UI 성능

- 진입 파일을 의도적으로 800KB 단일 파일로 유지(디스크 읽기 1회로 시작 단축), 무거운 임포트가 도는 동안 인증 프리페치를 병렬 실행, 피처 게이트로 비활성 기능은 빌드에서 데드 코드 제거
- 자체 TUI: 커스텀 react-reconciler + yoga(터미널 Flexbox), 이중 버퍼링(변경 셀만 출력), 문자열 인터닝, 프레임 스로틀링 — 터미널에서 React 렌더링 성능을 확보하는 기법들

## 멀티 에이전트 — 코디네이터

리더는 코드를 직접 수정하지 않고 위임만 한다. 4페이즈: 조사(병렬) → 종합(**리더가 결과를 반드시 직접 이해, 이 단계만은 위임 금지**) → 구현(파일 영역별 워커 1명씩 배정해 충돌 방지) → 검증(병렬 테스트). [[Harness-Engineering|멀티 에이전트 오케스트레이션]] 원칙의 실제 구현.

## 8대 설계 패턴

제너레이터 스트리밍, 피처 게이트 데드코드 제거, 메모이즈드 컨텍스트(Git 상태와 CLAUDE.md는 세션 수명 동안 캐시), 보류 후 복구(withhold & recover), 지연 임포트(순환 의존 회피), 불변 상태(단일 DeepImmutable 상태 트리), 중단 내성(쿼리 전 저장), 의존성 주입(모드별 교체와 테스트 목킹).

## 체크포인트

- 사용자 요청 한 번이 내부에서 도는 턴당 5단계 루프
- 도구 등록 순서가 고정인 이유 (프롬프트 캐시 무효화)
- 컴팩션 3종의 역할 분담과 요약 후 복원 예산, 서킷 브레이커
- 읽기와 쓰기 도구의 동시성 파티셔닝 기준 (isConcurrencySafe)
- BashTool이 차단 목록이 아니라 허용 목록 AST인 이유 (fail-closed)
- 코디네이터에서 종합 단계만 위임을 금지하는 이유
