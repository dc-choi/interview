# AI-Native 시스템 (AI-Native System)

AI가 코드를 안정적으로 생산하게 만드는 결정 변수는 모델 성능이 아니라 **레포와 조직이 AI에게 일관된 패턴을 강제하는가**이다. AI-Native 시스템은 에이전트를 감싸는 [[Harness-Engineering|하네스]]의 Verify, Constrain 축을 자연어 부탁이 아니라 코드베이스 자체의 결정론적 장치로 구현한 형태다.

## 정의 — AI-Native 레포의 속성

AI가 어느 파일을 참조해도 동일한 패턴을 발견하고, 그 패턴을 벗어나는 코드는 자동으로 머지 전에 차단되는 코드베이스.

- 일관된 컨벤션이 전체 코드에 균일하게 적용된다
- 규칙이 문서가 아니라 테스트와 린터로 강제된다
- CI, 포매터, 프리커밋 같은 자동 검증이 갖춰져 있다

**핵심 등가**: AI-Native 레포 = 좋은 온보딩 환경. 새 팀원이 빠르게 기여할 수 있는 레포는 AI도 빠르게 기여한다. 사람의 온보딩 문서와 AI의 컨텍스트가 지향점이 같다. 컨벤션이 흩어져 있고 규칙이 사람 머릿속에만 있는 레포는 신입에게도 AI에게도 똑같이 지뢰밭이다.

## 핵심 명제 — 부탁이 아니라 강제

자연어 지침은 부탁이고, 시스템은 강제다. 같은 규칙이라도 어디에 두느냐가 신뢰도를 가른다.

자연어 지침(컨텍스트 문서)의 구조적 한계:

- **모호성**: "지양해주세요"가 절대 금지인지 권장인지 모델이 확정할 수 없다
- **긴 컨텍스트 무시**: 지침이 길어지면 후반부 규칙의 준수율이 떨어진다 (Context Rot)
- **규칙 충돌**: 문서가 커질수록 상충하는 지침이 생기고 우선순위가 불명확해진다
- **추가의 역효과**: 컨텍스트 파일을 늘릴수록 작업 성공률이 오히려 소폭 하락하고 토큰 비용은 증가한다

결론: 지침을 늘려서 통제하려는 시도는 비선형으로 효과가 떨어진다. 반드시 지켜야 하는 규칙은 **결정론적 게이트**로 내려야 한다. 자연어로 "하지 마세요"를 100번 쓰는 것보다 린터 규칙 1개가 확실하다. 토큰 경제학 관점의 같은 논의는 [[Context-Engineering|컨텍스트 엔지니어링]]의 Hook vs Advisory.

## 결정론적 제어 4계층

| 계층 | 역할 | 강제 시점 |
|---|---|---|
| 프리커밋 훅 | 커밋 전 자동 검사 (포맷, 린트, 금지 패턴) | 커밋 |
| 포매터 | 스타일을 기계적으로 통일 (해석 여지 제거) | 저장/커밋 |
| 린터 | 정적 분석으로 안티패턴 검출 | 커밋/CI |
| CI | 아키텍처, 유닛, 통합 테스트로 합격선 판정 | PR 머지 |

공통 성질: 통과하지 못하면 머지가 불가능하다. AI 출력에 대한 객관적 합격선이 코드로 존재하므로, 사람도 AI도 해석으로 우회할 수 없다. 합격선이 코드에 있다는 점이 자연어 지침과의 결정적 차이다.

## 강제 메커니즘 구체화

### 아키텍처 테스트 (AST 정적 분석)

가장 강력한 신규 장치. 의존성 방향과 레이어 경계를 코드로 검증한다.

- **문제**: God Object(모든 것에 의존하는 단일 패키지)와 무분별한 cross-import가 있으면, AI가 그 잘못된 의존 구조를 학습해 재생산한다. 자연어로 "이 패키지를 의존하지 마"라고 적어도 강제력이 없다.
- **해결**: 언어의 AST(추상 구문 트리)를 파싱해 import 방향, 레이어 위반, 도메인 경계 침범을 검출하는 테스트를 CI에 둔다. JVM 진영의 ArchUnit이 같은 계열이고, Go 등은 커스텀 AST 프레임워크로 구현한다.
- **효과**: 아키텍처 위반을 CI에서 0으로 유지한다. [[DDD&Hexagonal|DDD]]의 도메인 분리와 Saga 경계가 문서가 아니라 빌드 실패로 강제된다.

### 워크플로우 자동화 (Skill)

AI가 반복적으로 틀리는 절차를 절차 자체의 자동화로 박제한다. 예를 들어 잘못된 base branch 선택이나 빈약한 PR 제목, 설명은 PR 생성 자체를 자동화하는 Skill로 만들면 실수가 구조적으로 0이 된다. 모델을 더 설득하는 게 아니라 실수할 경로를 없앤다.

### MCP 지식 그래프 (서비스 간 의존성)

MSA에서 한 서비스의 수정이 다른 서비스에 미치는 영향을 AI는 자기 컨텍스트만으로 알 수 없다. 아키텍처 의존 관계를 지식 그래프로 만들고 [[MCP]] 도구로 실시간 조회하게 하면, 크로스 도메인 영향 분석이 자동화된다. AI에게 코드만이 아니라 시스템 지도를 쥐여주는 것이다.

## 마인드셋 — 실수를 시스템으로 흡수하는 루프

핵심 전환은 "LLM이 이것도 못하네"에서 "이 실수를 막는 시스템을 만들자"로 가는 것이다. AI의 실수는 모델의 한계가 아니라 **환경의 빈틈을 드러내는 신호**로 읽는다.

| 관찰된 실수 | 흡수한 시스템 | 결과 |
|---|---|---|
| 임의 커밋 행동 | 커밋 규칙 + 프리커밋 훅 | 커밋 실수 차단 |
| 잘못된 base branch | PR 자동화 Skill | 브랜치 실수 0 |
| 아키텍처 위반 | AST 정적 분석 CI | 위반 0 |
| 스타일 불일치 | 포매터 + 프리커밋 | 일관된 코드 |
| 서비스 간 영향 미파악 | MCP 지식 그래프 | 의존성 가시화 |

이 루프는 복리로 작동한다. 환경이 좋아질수록 AI가 더 잘 기여하고, 그 여력으로 환경을 더 개선한다. 시스템 개선 자체도 AI에게 맡기는 또 하나의 병렬 작업이 된다.

## 작업 방식 — 작은 워터폴이 애자일을 이긴다

LLM은 명확한 전체 스펙을 먼저 받고 한 번에 구현하는 방식이, "일단 이것만 해봐" 식 점진적 피드백보다 효과적이다. 사람은 전체 그림을 미리 그려 주고, AI가 자율 계획을 세운 뒤 **의사결정 지점(A 방식 vs B 방식)에서만** 개입한다. 단계별 커밋으로 세이브 포인트를 남기고, self-review와 테스트를 AI가 스스로 돌린다. 스펙을 앞단에 명확히 두는 설계는 [[Agent-Spec-Writing|에이전트 스펙 작성]]과 같은 결이다.

## 세 축 — 시스템, 사람, 문화

결정론적 시스템만으로는 부족하고 세 축이 동시에 있어야 작동한다.

- **시스템**: 위의 결정론적 제어와 강제 메커니즘
- **사람**: 직접 타이핑한 코드에 대한 소유권을 내려놓고 판단자로 역할을 재정의한다. 엔지니어의 가치는 타이핑 속도가 아니라 판단의 질이다. 상세는 [[Developer-Role-AI-Era|AI 시대 개발자 역할]]
- **문화**: 실험을 허용하고(결과 중심 평가), AI가 생성한 코드도 사람 코드와 동등하게 공정히 리뷰한다

시스템만 깔고 문화가 따라오지 않으면, 코드 소유권 집착과 AI 결과물에 대한 편향이 도입을 막는다. 조직 전환의 단계론은 [[AX-Transformation|AX 조직 전환]].

## 트레이드오프, 주의점

- 결정론적 게이트(AST 프레임워크, CI 튜닝)는 선행 투자가 크다. 작은 레포에서는 과할 수 있다
- 게이트가 과도하면 정당한 변경까지 막는다. 규칙 자체도 유지보수 대상이다
- 병렬 작업의 의존성과 우선순위 판단은 아직 사람 몫이라 오케스트레이션이 병목이 된다
- 순서가 중요하다. AI-Native 시스템이 선행되지 않은 채 멀티 에이전트 자율 작업부터 도입하면 결과를 신뢰할 수 없다. 시스템을 먼저 깔고 자율성을 나중에 올린다 → [[Production-Agent-Architecture|프로덕션 에이전트 아키텍처]]

## 사례

- 6개월간 직접 작성 코드 0줄로 560개 이상 PR 머지, 약 97만 줄 수정, 동시 최대 10개 작업 병렬 진행
- 커스텀 AST 아키텍처 테스트 약 105K 줄 구축 → 아키텍처 위반 0 유지
- CI 최적화를 AI가 27회 이상 반복 실험으로 자율 수행: Go 10분 → 2분(-78%), Java 36분 → 21분(-43%), 1년 묵은 기술 부채를 2일에 해소
- 새 언어 컨벤션으로 기존 코드 3만 줄 이상을 외부 연동 무사고로 전면 전환

## 면접 체크포인트

- AI 생산성의 비결이 모델이 아니라 시스템인 이유 (부탁 vs 강제)
- 자연어 지침(컨텍스트 문서)의 구조적 한계 3가지 (모호성, 긴 컨텍스트 무시, 추가의 역효과)
- 아키텍처 위반을 0으로 만드는 법 (AST 정적 분석을 CI 게이트로)
- AI의 실수를 어떻게 다루나 (모델 탓이 아니라 시스템의 빈틈 신호로 보고 흡수)
- AI를 팀 전체 워크플로우로 쓰는 조직에서 백엔드 엔지니어의 기여 = 결정론적 게이트와 강제 메커니즘 설계 (코드 생성이 아니라 환경 설계)
