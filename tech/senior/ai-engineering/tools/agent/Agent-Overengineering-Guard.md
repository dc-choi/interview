# 에이전트 과잉설계 방지 — YAGNI 사다리를 스킬로 주입

코딩 에이전트는 요청받은 것보다 더 많이 만드는 경향이 있다. "날짜 선택기"를 요청하면 라이브러리를 설치하고 래퍼 컴포넌트와 스타일시트를 만들고 타임존을 논의한다. 이 과잉을 억제하는 방법은 **코드를 쓰기 전에 통과해야 하는 결정 사다리를 에이전트 스킬로 상시 주입**하는 것이다. 핵심 원칙은 토큰 최소화가 아니라 **"필요한 것만 쓰되, 검증/에러처리/보안/접근성은 절대 자르지 않는다"** — 코드가 작아지는 것은 골프가 아니라 필요가 최소여서다.

## YAGNI 사다리 — 첫 번째로 성립하는 칸에서 멈춤

문제를 이해한 다음(대체가 아니라), 코드를 쓰기 전에 위에서부터 성립하는 첫 칸에서 멈춘다.

1. **존재할 필요가 있나?** → 아니면 건너뛴다 (YAGNI)
2. **이미 코드베이스에 있나?** → 재사용, 재작성 금지
3. **표준 라이브러리가 하나?** → 그걸 쓴다
4. **네이티브 플랫폼 기능이 있나?** → 그걸 쓴다 (예: 날짜 선택기는 `<input type="date">`)
5. **설치된 의존성이 하나?** → 그걸 쓴다
6. **한 줄이면 되나?** → 한 줄
7. **그제서야: 동작하는 최소 구현**

사다리는 문제를 이해한 **뒤에** 돈다 — 변경이 닿는 코드를 읽고 실제 흐름을 추적한 다음 칸을 고른다. **해법에는 게으르되 코드 읽기에는 부지런하다.**

## 게으름과 태만의 구분

이 접근의 안전성은 "무엇을 자르지 않는가"에 있다. 신뢰 경계 검증(trust-boundary validation), 데이터 손실 처리, 보안, 접근성은 절대 삭제 대상이 아니다.

- 벤치마크에서 "YAGNI + 한 줄로 써" 같은 단순 프롬프트는 코드는 줄이지만 **안전 가드를 하나 떨어뜨렸다**(safe 95%). 사다리 방식은 안전 100%를 유지하며 줄였다
- 교훈: "짧게 써"는 위험하고, "필요한 것만 쓰되 안전은 유지"가 안전하다. 감축과 안전의 양립이 설계의 핵심

## 측정 — 감축은 과잉 함정에서 크다

실제 에이전트가 실제 저장소를 편집한 결과(git diff 기준)로 측정한다. 단발 생성 벤치마크는 대화형 baseline이 답변에 산문과 옵션을 덧붙여 감축 폭이 부풀려지므로, **에이전트형 baseline 대비**가 정직한 수치다.

- LOC 약 54% 감소(평균), 토큰 약 22%, 비용 약 20%, 시간 약 27% 감소, 안전 100% 유지
- **감축은 과잉 함정에서 최대**(날짜 선택기 404줄 → 23줄: 컴포넌트 대신 네이티브 input), 이미 최소인 코드에서는 거의 0
- 역효과 주의: 사다리 칸을 일일이 숙고하는 추론 모델은 오히려 사고 토큰이 늘어 비용이 오를 수 있다 — 지시를 그대로 따르는 모델에서 이득이 난다

## 에이전트 스킬로서의 배포

- **상시 주입 룰셋 + 커맨드**: 강도 레벨(lite/full/ultra/off)로 조절, 리뷰(`review`, diff의 과잉설계를 삭제 목록으로), 감사(`audit`, 저장소 전체), 부채 수확(`debt`, 미룬 단축을 원장으로 모아 "나중"이 "영원히 안 함"이 되지 않게)
- **다중 에이전트 이식성**: 플러그인/스킬을 지원하는 호스트(Claude Code, Codex 등)엔 커맨드까지, 지원 안 하는 호스트엔 AGENTS.md/규칙 파일로 상시 룰셋만. 같은 원칙을 여러 도구에 이식하는 패턴 ([[claude-code|Skills]], [[Agent-Spec-Writing]])
- CLAUDE.md/AGENTS.md에 룰을 심는 것과 같은 계열 — 매 턴 주입되는 [[Context-Engineering|영속 컨텍스트]]로 에이전트 행동을 교정

## 체크포인트

- 에이전트가 요청보다 많이 만드는 경향을 무엇으로 억제하는가 (결정 사다리 상시 주입)
- YAGNI 사다리 7칸을 순서대로 설명할 수 있는가 (필요성 → 재사용 → 표준 → 네이티브 → 의존성 → 한 줄 → 최소)
- 게으름과 태만의 경계 — 무엇은 줄이고 무엇은(검증, 보안, 접근성) 절대 안 줄이는가
- "짧게 써" 프롬프트가 안전 가드를 떨어뜨리는 이유와 사다리 방식이 안전을 유지하는 이유
- 벤치마크에서 대화형 baseline이 감축 폭을 부풀리는 이유 (산문, 옵션 패딩)
- 이 스킬이 역효과인 모델 (사다리를 숙고하며 사고 토큰을 태우는 추론 모델)
