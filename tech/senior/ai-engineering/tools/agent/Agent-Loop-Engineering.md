# 루프 엔지니어링 (Loop Engineering)

코딩 에이전트를 매번 직접 프롬프팅하는 대신 **에이전트를 프롬프팅하는 루프(제어 구조)를 설계**하는 실천. 에이전트가 대화형 어시스턴트에서 장기 실행 시스템으로 진화하면서 — 스케줄이나 이벤트로 일감을 스스로 발견하고, 별도 검증자가 완료를 판정하고, 상태 파일로 다음 실행이 이어받는 — 빌딩 블록이 커스텀 스크립트가 아니라 제품 프리미티브로 내장되며 2026년 중반에 이름이 붙었다. 루프의 본질은 트리거 종류가 아니라 **정지 조건이 충족될 때까지의 반복**이다.

## 계보 — ralph loop에서 loop engineering으로

- **ralph loop**: 완료 기준이 주어지면 목표에 가까워질 때까지 반복 시도하는 패턴. 성공 여부는 에이전트 자신의 판단이 아니라 **외부 검증**으로 결정된다. 가장 순수한 형태는 Bash 루프.
- **loop engineering**: 약 1년 뒤에 생긴 신조어. 구호로 요약하면 — 코딩 에이전트를 프롬프팅하지 말고, 에이전트를 프롬프팅하는 루프를 설계하라. ralph와의 결정적 차이는 **뭘 할지 루프가 스스로 찾는다**는 것 (이슈, PR, 피드백 채널을 상시 스캔해 트리아지하고 다음 작업을 표면화).
- 두 해석은 경쟁 정의가 아니라 **같은 스택의 다른 층**이다. ralph는 부정된 게 아니라 정지 조건, 검증 분리, 예산을 입고 안쪽 실행 프리미티브로 흡수됐다.
- 초기 논쟁 구도: 차세대 추상화 계층이다 vs 모자 씌운 cron job이다 — 층위를 구분하면 해소된다.

## 아키텍처 축 — 4중 루프 (LangChain 프레임)

| 층 | 루프 | 역할 |
|---|---|---|
| 안쪽 | agent loop | 모델이 상태를 읽고 작업이 끝날 때까지 도구를 호출 |
| 안쪽 | verification loop | 루브릭으로 채점하고, 미달이면 피드백과 함께 되돌림 |
| 바깥 | event-driven loop | 외부 신호(스케줄, 웹훅, 새 문서)가 에이전트를 발화 |
| 바깥 | hill-climbing loop | 누적 실행 기록을 분석해 프롬프트, 도구, 채점기를 개선 |

- 안쪽 2층이 ralph형 수렴이고, 바깥 2층이 자율 운영이다.
- 바깥층의 본질은 시간 간격이 아니라 **사람 대신 트리거가 실행을 개시하는 제어 시스템** — cron은 여러 트리거 중 가장 흔한 심장박동일 뿐이다.
- hill-climbing은 앞의 세 루프가 일 자체를 자동화한다면 **개선을 자동화**한다 — 실행이 남긴 트레이스를 분석해 다음 실행 자체를 좋게 만드는 층.

## 위임 축 — 공식 4분류 (Claude Code, 2026-07 공식 아티클 기준)

공식 정의: 루프란 **정지 조건이 충족될 때까지 작업 사이클을 반복하는 에이전트**다. 네 유형 모두 정지 기준이 1급 시민으로 명시된다.

| 유형 | 트리거 | 정지 조건 | 사람이 넘기는 것 |
|---|---|---|---|
| Turn-based | 사용자 프롬프트 | 스스로 완료 판단 | 검증 |
| Goal-based (/goal) | 사용자가 검증 가능한 완료 기준을 부여 | 별도 evaluator 모델의 판정, 최대 턴 상한 | 정지 조건 |
| Time-based (/loop, /schedule) | 시간 간격 | 사용자 취소, 일감 소진 | 트리거 |
| Proactive | 이벤트/스케줄, 실시간 개입 없음 | 태스크는 목표 달성 시, 루틴은 끌 때까지 | 프롬프트 자체 |

- Goal-based의 정지 판정은 작업한 모델이 아니라 **별도 evaluator**가 한다 — maker와 checker의 분리를 정지 조건 자체에 적용한 것.
- Proactive는 버그 리포트/이슈 트리아지, 마이그레이션, 의존성 업그레이드처럼 **잘 정의된 반복 작업 스트림**에 쓴다.
- 위임 사다리를 오른쪽으로 갈수록 사람의 역할이 검증자(checker)에서 **루틴을 설계하고 언제 끌지 판단하는 director**로 이동한다.
- 두 축의 대응: agent loop는 turn-based에, verification loop는 goal-based의 evaluator에, event-driven loop는 time-based/proactive의 트리거에 대응한다. hill-climbing은 공식 분류에서 별도 루프가 아니라 운영 원칙으로 들어간다 — 기준 미달 건을 그것만 고치고 멈추지 말고 **이후 모든 반복이 좋아지도록 시스템에 인코딩**하라.

## 성숙형 합성 — 바깥은 발견, 안쪽은 수렴

표준 레시피: 스케줄이 주기적으로 일감을 발견하고(time-based), 발견된 항목마다 goal 루프가 완료 기준까지 수렴하고(goal-based), 복수 해법을 병렬 worktree에서 만들어 judge가 적대적으로 리뷰하고, 상태 파일이 뭘 시도했고 뭐가 통과했는지 기억해 다음 실행이 멈춘 지점에서 이어받는다. 리뷰와 판정은 코드를 만든 에이전트가 아니라 **컨텍스트가 신선한 두 번째 에이전트**에게 맡긴다.

## 가드 — bare loop는 열린 청구서

상한 없는 bare ralph 루프는 그대로 승인되지 않는다.

- 3종 가드: **반복 횟수 상한**, **진전 없음 체크**(최근 실행에서 diff가 더 이상 변하지 않으면 종료), **토큰/비용 지출 상한**
- 좋은 루프의 설계 요소: 명확한 목표, 유용한 컨텍스트, 작은 행동, 신뢰할 만한 관찰, 명시적 정지 규칙
- 절제: 모든 작업에 복잡한 루프가 필요한 건 아니다 — 가장 단순한 해법부터 시작해 선택적으로 쓴다

## 하네스와의 관계

- loop engineering은 [[Harness-Engineering|하네스 엔지니어링]]의 한 층 위다 — 타이머로 돌고, 작은 헬퍼들을 낳고, 스스로를 먹여 살리는 하네스. 한 줄 정리: **루프 = 시간 축과 일감 발견 축으로 확장된 하네스, ralph는 그 원형**.
- 진화 경로: chat, tools, goals, automations 순. 문제가 생기면 아래(안쪽) 루프로 내려가 신뢰성을 회복하고, 모델이 좋아지면 위(바깥) 루프로 올라가 레버리지를 키운다.

## 면접 체크포인트

- 루프의 본질을 트리거 종류가 아니라 정지 조건으로 정의할 수 있는가
- ralph loop와 loop engineering의 관계 — 경쟁 정의가 아니라 안쪽 프리미티브로의 흡수
- 아키텍처 축(4중 루프)과 위임 축(공식 4분류)의 대응 관계
- 위임 사다리 — 검증, 정지 조건, 트리거, 프롬프트를 차례로 넘기며 사람이 director로 이동
- bare loop에 필요한 3종 가드와 각각의 이유
- evaluator 분리 — maker/checker 분리가 정지 조건 판정에 적용되는 방식
- hill-climbing 층이 따로 있는 이유 (일의 자동화와 개선의 자동화 구분)
