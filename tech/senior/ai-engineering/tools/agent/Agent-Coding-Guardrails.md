# LLM 코딩 가드레일 — 실패 패턴과 교정 4원칙

코딩 에이전트의 사고는 능력 부족보다 행동 습관에서 온다. 확인 없이 가정하고 질주하는 것, 100줄이면 될 코드를 1000줄로 부풀리는 것, 태스크와 무관한 코드를 건드리는 것, 검증 기준 없이 완료를 선언하는 것. 이 반복 패턴은 모델 교체가 아니라 **짧은 행동 규칙을 영속 컨텍스트로 상시 주입**해 교정한다. 4원칙(Think Before Coding, Simplicity First, Surgical Changes, Goal-Driven Execution)이 대표 룰셋이고, CLAUDE.md 병합, 스킬/플러그인, Cursor rule 등 여러 경로로 배포된다.

## 반복 관찰되는 실패 패턴

1. **가정 질주** — 사용자를 대신해 가정을 세우고 확인 없이 진행한다. 혼란을 관리하지 않고, 명확화를 구하지 않고, 불일치를 표면화하지 않고, 트레이드오프를 제시하지 않고, 반박해야 할 때 반박하지 않는다.
2. **과잉 복잡화** — 코드와 API를 부풀리고 추상화를 비대하게 만들고 dead code를 정리하지 않는다. 100줄이면 될 구현을 1000줄로 짓는다.
3. **부수효과 편집** — 충분히 이해하지 못한 주석과 코드를, 태스크와 직교인데도 변경하거나 삭제한다.
4. **낭비되는 루프 지렛대** — LLM은 구체적 목표를 만족할 때까지 루핑하는 데 탁월한데, "되게 해줘" 같은 모호한 지시는 이 강점을 쓰지 못하게 하고 상시 개입만 남긴다.

## 교정 4원칙

각 원칙은 위 실패 패턴 순서와 대응한다.

### 1. Think Before Coding — 가정을 표면화

구현 전에:

- 가정을 명시적으로 선언하고, 불확실하면 추측 대신 질문한다
- 해석이 여러 개면 모두 제시한다. 조용히 하나를 고르지 않는다
- 더 단순한 접근이 있으면 말한다. 필요하면 반박한다
- 혼란스러우면 멈추고, 무엇이 혼란스러운지 명명하고 묻는다

### 2. Simplicity First — 요청 범위의 최소 코드

- 요청 밖 기능 금지, 단일 사용 코드에 추상화 금지
- 요청받지 않은 유연성과 설정 가능성 금지
- 불가능한 시나리오의 에러 처리 금지
- 200줄을 썼는데 50줄이면 충분하다면 다시 쓴다

판정 질문: 시니어 엔지니어가 보면 과잉이라고 하겠는가. 그렇다면 단순화한다.

단, 줄이는 대상은 요청 밖 범위지 안전이 아니다 — 신뢰 경계 검증, 보안, 접근성, 데이터 손실 처리는 감축 대상에서 제외한다 ([[Agent-Overengineering-Guard|게으름 vs 태만]] 구분).

### 3. Surgical Changes — 닿아야 할 것만, 자기 뒷정리만

기존 코드를 편집할 때:

- 인접 코드, 주석, 포맷을 개선하지 않는다
- 안 깨진 것을 리팩토링하지 않는다
- 내 취향과 달라도 기존 스타일을 따른다
- 무관한 dead code를 발견하면 언급만 하고 삭제하지 않는다

내 변경이 고아를 만들면:

- 내 변경 때문에 안 쓰이게 된 임포트, 변수, 함수는 제거한다
- 원래 있던 dead code는 요청 없이는 제거하지 않는다

판정 질문: 변경된 모든 라인이 사용자 요청으로 소급되는가.

### 4. Goal-Driven Execution — 성공 기준과 검증 루프

명령형 지시를 검증 가능한 목표로 변환한다.

| 명령형 | 검증 가능한 목표 |
|---|---|
| 검증 추가해줘 | 잘못된 입력 테스트를 먼저 쓰고 통과시킨다 |
| 버그 고쳐줘 | 재현 테스트를 먼저 쓰고 통과시킨다 |
| X 리팩토링해줘 | 전후로 테스트 통과를 보장한다 |

다단계 작업은 단계마다 verify를 붙인 계획을 먼저 선언한다 (단계, verify: 확인 방법). 강한 성공 기준은 에이전트의 독립 루프를 허용하고, 약한 기준은 상시 개입을 요구한다. 핵심 발상은 무엇을 할지 지시하는 대신 성공 기준을 주고 지켜보는 것.

## 왜 작동하나 — 복잡성은 옳고 그름이 아니라 타이밍 문제

과잉 설계된 코드는 명백히 틀린 코드가 아니다. 디자인 패턴과 모범 사례를 따른 코드다. 문제는 필요해지기 전에 복잡성을 넣는 타이밍이고, 이것이 이해를 어렵게 하고 버그를 늘리고 구현과 테스트를 느리게 한다. 단순 버전은 이해와 테스트가 쉽고, 복잡성이 실제로 필요해질 때 리팩토링하면 된다. 좋은 코드는 오늘의 문제를 단순하게 푸는 코드지, 내일의 문제를 미리 푸는 코드가 아니다.

## 배포 — 같은 룰셋, 여러 주입 경로

- **CLAUDE.md 병합**: 프로젝트 영속 컨텍스트에 append. 프로젝트 특화 규칙(테스트 규칙, 기존 에러 처리 패턴 준수 등)과 병합해 쓰는 전제
- **스킬/플러그인**: SKILL.md의 description 트리거로 코드 작성, 리뷰, 리팩토링 시 로드. 마켓플레이스 플러그인으로 설치하면 전 프로젝트 공용 ([[Agent-Skills]])
- **다른 호스트**: 같은 내용을 Cursor project rule(.mdc)로. 원칙을 도구별 주입 포맷으로 이식하는 패턴은 [[Agent-Overengineering-Guard]]의 다중 에이전트 이식성과 같은 축
- **트레이드오프**: 신중 편향 룰셋이라 사소한 작업(오타 수정, 자명한 한 줄)엔 과잉이다. 목적은 단순 작업의 감속이 아니라 비단순 작업의 비싼 실수 감축

## 작동 확인 신호

- diff에 요청 밖 변경이 줄었다
- 과잉 복잡화로 인한 재작성이 줄었다
- 명확화 질문이 실수 뒤가 아니라 구현 앞에 온다
- 드라이브바이 리팩토링 없는 최소 PR이 나온다

## 기존 도구와의 관계

- [[Agent-Overengineering-Guard|YAGNI 사다리]]는 Simplicity First 하나를 결정 절차(7칸 사다리)로 정밀화한 도구다. 4원칙은 가정, 범위, 편집 반경, 목표 정의까지 커버하는 상위 행동 프레임
- Goal-Driven Execution의 성공 기준 정의는 [[Agent-Spec-Writing]]의 경계 시스템, 적합성 테스트와 같은 축
- 규칙을 매 턴 주입하는 메커니즘 자체는 [[Context-Engineering|영속 컨텍스트]] — 권장(CLAUDE.md)과 강제(Hook)의 구분도 거기서 다룬다

## 면접 체크포인트

- LLM 코딩의 반복 실패 패턴을 구체적으로 말할 수 있는가 (가정 질주, 과잉 복잡화, 부수효과 편집, 모호한 목표)
- 4원칙 각각의 판정 테스트 — 시니어가 과잉이라 보겠는가, 모든 변경 라인이 요청으로 소급되는가, 성공 기준이 독립 루프를 허용하는가
- 명령형 지시를 선언적 성공 기준으로 변환하는 예 (버그 수정이면 재현 테스트부터)
- 과잉 설계를 옳고 그름이 아니라 타이밍 문제로 설명할 수 있는가
- 행동 가드레일의 트레이드오프(신중 vs 속도)와 적용 예외
- 같은 룰셋의 주입 경로별 차이 — 상시 주입(CLAUDE.md)과 온디맨드 로드(스킬)의 구분
