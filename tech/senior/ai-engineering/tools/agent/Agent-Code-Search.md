# 에이전트 코드 검색 — grep+read를 시맨틱 검색으로 대체하기

코딩 에이전트가 낯선 저장소를 탐색하는 기본 동작은 grep으로 후보를 찾고 매칭된 파일을 통째로 읽는 것이다. 이 방식은 **매칭 파일 전체가 컨텍스트로 흘러들어** 토큰을 태우고, 표현이 다르면(동의어, 다른 명명) 아예 못 찾는다. 대안은 코드베이스를 미리 인덱싱해 자연어 질의에 **관련 스니펫만 반환**하는 시맨틱 검색이다. grep+read 대비 토큰을 약 98% 줄이면서(2k 토큰에 recall 94% vs grep+read는 100k 컨텍스트로 85%) 검색 품질은 코드 특화 트랜스포머급을 유지한 사례가 있다.

이는 [[Tool-Output-Filtering|도구 출력 필터링]](도구가 컨텍스트를 채운다)과 [[Agent-Context-Budget|컨텍스트 예산]]의 코드 탐색 버전이다 — 읽는 양 자체를 줄이는 접근.

## 왜 grep+read가 비싼가

- **토큰**: 매칭된 파일을 전부 읽는 것이 baseline. 스니펫만 필요한데 파일 단위로 읽으면 대부분이 노이즈
- **재현율**: 정확 토큰 매칭이라 "인증은 어떻게 처리되나?" 같은 자연어 의도나 다른 표현의 구현을 놓친다
- 절감 추정: `(파일 문자수 - 스니펫 문자수) / 4`가 호출당 아낀 토큰. 익숙지 않은 코드를 탐색할수록 파일 전체 읽기 대비 이득이 크다

## 파이프라인 — 청킹, 이중 검색, 융합

1. **코드 인식 청킹**: tree-sitter로 파일을 구문 단위(함수, 클래스)로 분할 — 고정 길이 청킹이 함수 중간을 자르는 문제를 피함 ([[RAG-Retrieval-Engineering|청킹 전략]])
2. **이중 검색**: 정적 임베딩(의미 유사도) + BM25(식별자, API명 정확 매칭)를 병행. 의미는 임베딩이, 정확 토큰은 BM25가 맡는 분업
3. **RRF 융합**: 두 점수 리스트를 Reciprocal Rank Fusion으로 단일 랭킹으로 합침 (하이브리드 검색 일반론은 [[RAG-Retrieval-Engineering#하이브리드 검색: 벡터 + BM25]])

## 정적 임베딩 — CPU에서 밀리초

핵심 성능 트릭은 **쿼리 시점에 트랜스포머 forward pass가 없는** 정적 임베딩(Model2Vec 계열)을 쓰는 것이다.

- 코드 특화 트랜스포머 대비 인덱싱 약 200x, 쿼리 약 10x 빠르면서 검색 품질(NDCG@10)은 그 99% 수준
- **CPU만으로, API 키/GPU/외부 서비스 없이** 동작 — 평균 저장소 인덱싱 약 250ms, 쿼리 약 1.5ms
- 트레이드오프: 정적 임베딩은 문맥 의존 의미를 트랜스포머만큼 못 잡지만, 코드 인식 리랭킹으로 그 손실을 메운다

## 코드 인식 리랭킹 시그널

융합 후 코드 도메인 지식으로 재정렬하는 것이 일반 텍스트 검색과의 차별점이다.

- **적응형 가중치**: 심볼형 질의(`getUserById`, `Foo::bar`)는 BM25 비중을 높이고, 자연어 질의는 의미/키워드 균형 유지
- **정의 부스트**: 심볼을 정의하는 청크(class, def, func)를 단순 참조하는 청크보다 위로
- **식별자 스템 매칭**: 질의 토큰을 어간 분해해 `parse config` → `parseConfig`, `ConfigParser`, `config_parser`를 함께 매칭
- **파일 응집**: 한 파일에서 여러 청크가 매칭되면 파일 레벨 관련성으로 부스트 (문맥 이탈한 단일 청크 방지)
- **노이즈 페널티**: 테스트 파일, legacy/compat 심, 예제 코드, 타입 선언 스텁을 낮춰 정규 구현이 먼저 뜨게

## 제공 형태와 인덱스 관리

- **세 가지 통합**: MCP 서버(에이전트가 도구로 직접 호출), CLI(AGENTS.md/CLAUDE.md에 사용법 안내), 전용 서브에이전트. 어느 코딩 에이전트든 붙일 수 있음 ([[MCP]])
- **인덱스 캐싱과 무효화**: 첫 검색 시 디스크에 캐싱, 이후 파일 수정 시간 비교로 변경 감지 시 재빌드. MCP 모드는 파일 워처로 세션 내 자동 갱신
- **인덱싱 범위 제어**: `.gitignore`/`.sembleignore`로 제외, `!` 접두사로 비기본 확장자 강제 포함. node_modules 같은 비소스 디렉토리는 항상 제외

## 체크포인트

- 에이전트가 낯선 코드를 탐색할 때 토큰이 어디서 새는가 (grep 매칭 파일 전체 읽기)
- 시맨틱 코드 검색이 grep 대비 나은 두 축 (토큰 효율, 자연어/동의어 재현율)
- 정적 임베딩이 CPU 실시간 검색을 가능케 하는 원리 (쿼리 시 트랜스포머 forward pass 없음)과 트레이드오프
- 코드 검색이 일반 텍스트 검색과 다른 리랭킹 시그널 (정의 부스트, 식별자 스템, 노이즈 페널티)
- 인덱스 무효화 전략 (파일 mtime 비교, 워처 기반 자동 재빌드)
