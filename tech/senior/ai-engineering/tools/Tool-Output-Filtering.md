# 도구 출력 필터링 (Tool Output Filtering)

## 정의

에이전트의 컨텍스트를 실제로 채우는 것은 사용자 프롬프트가 아니라 도구 출력이다. MCP 호출, 파일 읽기, 코드 검색, 터미널 실행 결과가 모델과 외부 세계를 잇는 순간 응답 전체가 컨텍스트로 흘러든다. 도구 출력 필터링은 이 흐름의 중간에서 모델이 판단에 쓸 필드만 남기고 나머지를 버려, 토큰 소비와 Context Rot를 동시에 줄이는 기법이다. 프롬프트를 짧게 쓰는 것과는 다른 축의 최적화다.

## 토큰은 어디서 새는가

- 사용자가 던지는 프롬프트는 대체로 짧다. 컨텍스트 예산을 실제로 태우는 쪽은 도구가 돌려주는 대용량 구조체다.
- 대표 누수원: MCP 서버 응답(전체 노드 트리, 메타데이터), 파일 전체 읽기, 넓은 코드 검색 결과, 장황한 터미널 로그.
- 정량 감각: 외부 MCP를 필터 없이 물리면 수 시간 안에 컨텍스트 여유가 급락한다. 디자인 MCP 연동에서 5시간 사용 여유가 91%에서 38%로 떨어진 사례가 있다. 한 번의 무거운 도구 호출이 세션 전체의 남은 예산을 좌우한다.

## 원칙 1 - 사전 필터링 (요약보다 먼저)

필터링을 언제 하느냐가 핵심이다. 요약은 이미 토큰을 다 먹은 뒤에 일어나므로 늦다.

- 나쁜 흐름: 외부 도구 → 응답 전체가 컨텍스트로 → 모델이 요약
- 좋은 흐름: 외부 도구 → 중간 계층에서 필터링 → 필요한 필드만 컨텍스트로

세부:

- 필터는 모델이 아니라 도구와 모델 사이의 코드 계층(래퍼, MCP 프록시)에서 돈다. 모델에게 요약을 시키면 그 요약 입력분만큼 이미 과금된 상태다.
- 구현은 단순한 필드 추출로 충분한 경우가 많다. 응답 객체에서 구현 판단에 필요한 키만 골라 새 객체로 만들어 넘긴다.
- 판단 기준: 이 필드가 없으면 모델이 다음 행동을 못 정하는가. No면 버린다. CLAUDE.md 줄 단위 자문("없으면 실수하나?")을 도구 출력에 그대로 적용하는 논리다.

## 원칙 2 - 상태 기반 컨텍스트 관리

컨텍스트는 저장소가 아니라 작업 공간이다. 지나온 모든 것을 쌓는 로그가 아니라, 지금 판단에 필요한 상태만 올려둔 화이트보드로 다룬다.

- 남길 것: 현재 목표, 확정된 결정, 실패한 접근(반복 방지), 남은 위험.
- 버릴 것: 이미 해결된 에러 로그, 폐기된 계획, 중간 시행착오의 원본 덤프.

실패한 접근을 남기는 이유는 같은 벽에 다시 부딪히지 않기 위해서다. 성공으로 대체된 중간 산출물은 지운다. 이 관리를 사람이 매번 하긴 어려우므로, 상태를 컨텍스트 밖 파일(PROGRESS, decisions)로 빼는 Write 전략과 함께 쓴다.

## 원칙 3 - 에이전트 탐색 범위 제한

필터링하지 않으면 에이전트는 읽기 → 오류 → 수정 → 재읽기 루프를 돌며 매 회전마다 도구 출력으로 컨텍스트를 채운다. 루프 횟수 자체를 줄이는 게 근본 처방이다.

작업을 넘길 때 네 가지를 명시한다:

- 목표: 무엇을 끝내야 완료인가
- 허용 범위: 어떤 파일, 디렉토리까지 건드려도 되는가
- 금지 사항: 손대면 안 되는 곳
- 검증 방법: 완료를 어떻게 확인하는가

범위가 열려 있으면 에이전트가 주변 코드까지 탐색하며 불필요한 읽기를 늘린다. 경계를 좁힐수록 도구 호출 수가 줄고 컨텍스트가 깨끗해진다.

## 사례 - 디자인 도구를 코드로 동기화

디자인 컴포넌트를 스토리북 컴포넌트로 옮기는 작업에서 실제로 필요한 정보는 레이아웃, 색상, 타이포그래피, 간격, variant 상태뿐이다. 그런데 디자인 MCP의 노드 응답에는 내부 변수, 편집 메타데이터, 렌더링 속성이 대량으로 딸려 온다.

- 전체 node 객체를 그대로 넘기면 대부분이 구현과 무관한 노이즈다.
- 응답에서 구현 판단 필드(layout, color, typography, spacing, variant)만 추출해 넘기면 같은 결과를 훨씬 적은 토큰으로 낸다.
- 무엇을 버릴지는 도메인 지식이 정한다. 스토리북 변환에 편집 히스토리는 필요 없다는 판단이 곧 필터 규칙이 된다.

## 트레이드오프

- 필터링 로직 유지비: 도구 응답 스키마가 바뀌면 필드 추출 코드도 따라가야 한다. 대신 매 호출의 토큰을 항구적으로 아낀다.
- 과필터 위험: 나중에 필요한 필드를 미리 버리면 모델이 재조회하며 오히려 왕복이 는다. 버리는 기준은 이 작업에 불필요인지이지, 영원히 불필요인지가 아니다.
- 사전 필터 vs 사후 요약: 요약은 범용이지만 이미 과금된 뒤다. 필터는 도구별 코드가 들지만 과금 전에 막는다. 무겁고 정형화된 도구부터 필터를 붙이는 게 이득이 크다.

## 핵심 원칙

좋은 토큰 최적화는 모델에게 정보를 부족하게 주는 일이 아니다. 모델이 판단해야 할 정보만 정확하게 주는 일이다. 목표는 프롬프트 단축이 아니라 컨텍스트 품질이다.

## 면접 포인트

Q. 에이전트가 토큰을 빨리 소진하는데 프롬프트는 짧다. 어디서 새나?
- 도구 출력이다. MCP 응답, 파일 전체 읽기, 넓은 검색 결과가 컨텍스트를 채운다. 프롬프트가 아니라 도구와 모델 사이에 필터를 넣어야 한다.

Q. 요약으로 줄이면 되지 않나?
- 요약은 이미 토큰을 먹은 뒤라 늦다. 도구와 모델 사이 코드 계층에서 필요한 필드만 남기는 사전 필터가 과금 전에 막는다.

Q. 무엇을 버릴지 어떻게 정하나?
- 이 필드가 없으면 다음 행동을 못 정하는가로 판단한다. 도메인 지식이 필터 규칙을 정하고, 이 작업에 불필요를 기준으로 삼되 영원히 불필요와 혼동하지 않는다. 과필터하면 재조회 왕복이 늘어 오히려 손해다.
