# LLM 워크플로우 패턴 (체인 vs 에이전트, Function Calling)

LLM 서비스의 중심축은 "질문하면 답하는 챗봇"에서 **업무 흐름 안에서 실제 행동을 수행하는 에이전트**로 이동한다. "이번 분기 성과 정리해줘" 한마디에 AI가 Jira, 문서, OKR, 평가 기준을 찾아 초안을 만드는 식이다. 다만 모든 문제를 에이전트로 풀 필요는 없고, 문제 성격에 맞는 워크플로우 구조를 고르는 것이 설계의 핵심이다. 에이전트 위임의 신뢰 설계 자체는 [[Production-Agent-Architecture|프로덕션 에이전트 아키텍처]]를 참고.

## 체인 워크플로우 vs 에이전트 워크플로우

| 구분 | 체인 워크플로우 | 에이전트 워크플로우 |
|------|----------------|---------------------|
| 흐름 | 정해진 순서대로 처리 | 상황에 따라 다음 행동을 동적 결정 |
| 적합 | 목적이 명확하고 흐름이 단순 (문서 요약, 고정 포맷 변환) | 질문 유형이 다양하고 단계별 정확도 점검이 필요 (서포트 채널 답변) |
| 비용/리스크 | 낮음, 예측 가능 | 높음, 비결정성 증가 |

핵심 원칙은 **모든 문제를 에이전트로 풀지 않는 것**이다. 단순하고 결정론적인 작업에 에이전트를 쓰면 토큰과 비결정성만 늘어난다(오버엔지니어링).

### 그래프 워크플로우

에이전트 워크플로우를 그래프로 구성하면, 각 노드가 **프롬프트와 LLM을 가진 작은 처리 단위**처럼 동작한다. 일반 처리 노드 사이에 **검색 결과나 생성 답변을 평가하는 노드**를 끼워, 품질이 미달이면 재검색이나 재생성으로 분기시킨다. 질문 분류 → 유형별 체인 선택(SQL 생성인지, 문서 검색인지, 채널 답변인지) → 동적 프롬프트 조합으로 이어진다.

## Function Calling: 모델이 도구를 다루는 능력

Function Calling은 LLM이 외부 도구(함수)를 호출하는 능력이다. 모델은 **사용자 발화와 함수 스키마**를 보고, 어떤 함수를 호출할지와 어떤 파라미터를 넣을지를 생성한다. API 결과(보통 JSON)는 사람이 읽기 어려우므로, 모델이 다시 사람 친화적인 문장이나 표로 정리한다.

연결을 표준화한 프로토콜 관점은 [[MCP|MCP]](모델-도구 연결 표준)이고, Function Calling은 그 위에서 모델이 실제로 함수를 고르고 호출하는 **모델 능력** 레이어다.

### 스킬 시스템: Function Calling의 서비스화

특정 API 묶음을 LLM에 연결해 에이전트를 만드는 구조다.

- **스킬** = API 하나, **스킬셋** = 한 서비스를 위한 API 묶음
- 여행 API 묶음을 붙이면 여행 에이전트, 쇼핑 API 묶음을 붙이면 쇼핑 에이전트 — **묶음을 바꾸면 에이전트의 성격이 바뀐다**

처리 3단계:

| 단계 | 역할 |
|------|------|
| **Detector** | 사용자 질문이 API로 풀 문제인지 판별 |
| **CoT** | 어떤 API를 호출하고 어떤 파라미터를 넣을지 추론 (Chain of Thought, 중간 추론으로 행동 결정) |
| **Answer** | API 결과를 사람이 이해할 문장이나 표로 정리 |

- **멀티턴**: 이전 맥락을 활용("ABC 동물병원 전화번호" → 다음에 "BB도" 하면 BB 동물병원으로 해석)
- **멀티쿼리**: 여러 API 호출이 필요한 복잡 요청을 여러 단계로 분할 실행
- **강건성**: 함수 이름이나 속성을 난독화해도 동작하도록 학습시켜 모델이 표면 명칭에 과의존하지 않게 함
- **평가**: Berkeley Function Calling Leaderboard로 형식 준수, 실행 가능성, 관련성, 멀티턴 처리를 측정. 형식, 실행 성능은 성숙했으나 **멀티턴 성능이 공통 개선 과제**

학습 데이터(질문, 생각, API 호출, 답변 형식)는 처음부터 손으로 만들 필요 없이, 기본 CoT 모델이 초안을 생성하고 잘못된 부분만 사람이 고치는 방식으로 누적한다. 답변 포맷(표, 추천 URL 위치 등)도 학습 대상이며, 수백 건이 쌓이면 거의 일정한 형식으로 생성된다.

## Text-to-SQL과 데이터 디스커버리

데이터 리터러시(데이터를 이해, 생성, 인사이트 도출, 소통하는 능력)를 비전문가에게 확장하는 패턴이다.

- **Text-to-SQL**: 자연어 질문에서 필요한 테이블, 조건, SQL 함수를 고려해 쿼리를 생성
- **데이터 디스커버리**: 테이블의 의미를 설명하고 어떤 데이터를 써야 하는지 안내

품질의 관건은 모델이 아니라 **구조화된 메타데이터**다. 테이블 의미, 업무 용어, 지표 정의를 벡터 스토어와 사전에 적재해야 한다. 모호한 의미 검색을 결정론적 조회로 바꾸는 방향은 [[RAG-Retrieval-Engineering|엔티티 추출 후 구조화 조회]], 지표 정의 수렴은 [[Production-Agent-Architecture|Metric Registry]]와 같은 결이다.

## 데이터와 모델 중 무엇이 더 중요한가

- **도메인 특화 서비스 → 데이터가 더 중요**: 사내 용어, 테이블 의미, 업무 규칙은 범용 LLM이 알 수 없다
- **일반 지식, 범용 작업 → 모델 성능 영향이 큼**

실무 개발자가 당장 개선하기 쉬운 쪽은 보통 **데이터**다. 모델 자체 고도화는 어렵지만, 좋은 문서, 정확한 메타데이터, 정제된 로그, 명확한 평가셋은 직접 만들 수 있다.

## 환각 통제와 운영

환각은 확률적 생성 모델의 속성이라 제거가 아니라 **통제**의 문제다. 두 갈래다.

- **근거 기반 답변**: RAG와 Function Calling으로 자기 기억이 아니라 외부 문서, API 결과를 근거로 답하게 함 (모른다고 말하는 능력과 시스템 보완은 [[LLM-Abstention|LLM Abstention]])
- **기획 단계의 영역 선택**: 환각에 덜 민감한 작업(초안, 요약)은 사람이 수정 가능하지만, 법률 판단, 인사 평가, 금융 거래처럼 오류 비용이 큰 영역은 더 강한 검증 장치가 필요하다

운영은 한 번 만들고 끝이 아니라 **빠른 피드백 루프**다. 모델, 프롬프트, 검색 데이터, 사용자 요구가 계속 바뀌므로 빠르게 배포하고 피드백을 받아 개선한다(피드백을 지식, 테스트로 환원하는 운영 루프는 [[Production-Agent-Architecture]], 평가 철학은 [[LLM-Eval-Strategy]]).

**시작 전략**: 거창한 에이전트부터 만들기보다 반복적이고 검증 가능한 업무(문서 요약, FAQ 자동 응답, SQL 초안 생성, 업무 기록 정리)부터. 사람이 최종 확인할 수 있는 작업이 좋은 출발점이다. LLM은 사람을 대체하는 도구라기보다 **사람이 더 빠르게 판단하고 더 나은 결과를 만들게 돕는 업무 증폭기**다.

## 면접 체크포인트

- 체인 워크플로우와 에이전트 워크플로우의 선택 기준 (단순/결정론은 체인, 다양/단계점검은 에이전트, 다 에이전트로 풀지 않기)
- Function Calling 3단계(Detector → CoT → Answer)와 멀티턴/멀티쿼리, 멀티턴이 공통 과제인 이유
- 스킬 vs 스킬셋, 묶음 교체로 에이전트 성격이 바뀌는 구조
- 데이터 vs 모델 — 도메인 특화면 데이터, 실무자가 당장 손댈 수 있는 쪽도 데이터
- 환각은 제거가 아니라 통제(근거 기반 + 기획 단계 영역 선택)
- 시작은 검증 가능한 반복 업무부터, 사람이 최종 확인

## 사례
- 데이터 분석 보조: Text-to-SQL + 데이터 디스커버리 + 지식 공유(위키, Slack 서포트 채널 자동 답변 + 근거 링크), 질문 분류 후 멀티 체인, 일부는 평가 노드를 둔 그래프 에이전트로 구성. Slack을 UI로, LLMOps로 배포/모니터링.
- 성과 평가 보조: Jira 이슈/에픽 → OKR 연결 → 회사 평가 기준 반영으로 성과 노트 초안 작성. 동료 평가는 AI가 질문을 던져 구체화하고 AAT(행동-감사-기대), SBI(상황-행동-영향) 피드백 구조를 적용. 데이터 전처리(누락값, 형식 오류)와 평가 윤리(편향, 개인정보, 최종 판단은 사람)가 난제.
- API 에이전트 플랫폼: 스킬셋 연결로 도메인 에이전트 생성, Detector-CoT-Answer 처리, 리트리벌 스킬(CoT+랭킹으로 RAG 확장)과 근거를 XML로 표시하는 랭킹 모델로 출처 추적.
