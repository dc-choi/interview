# RAG 검색 엔지니어링 (RAG Retrieval Engineering)

RAG(Retrieval-Augmented Generation)에서 답변 품질을 결정하는 건 생성이 아니라 검색이다. 모델이 아무리 좋아도 검색이 엉뚱한 문서를 물어오면 환각이 시작된다. 따라서 엔지니어링의 무게중심은 프롬프트가 아니라 무엇을, 어떻게 꺼내오느냐에 있다.

## 청킹 전략: 고정 길이의 한계

가장 단순한 청킹은 문서를 N 토큰 단위로 자르는 고정 길이 방식이다. 구현은 쉽지만 표, 이미지, 그래프처럼 한 청크를 넘어가는 구조를 끊어버린다. 표 상단 헤더와 하단 데이터가 다른 청크로 갈리면 둘 다 의미를 잃는다.

대안은 문서 구조 기반 청킹이다. 제목, 섹션, 본문의 위치를 인식해 의미 단위로 자른다.

| 방식 | 기준 | 약점 |
|---|---|---|
| 고정 길이 | N 토큰마다 절단 | 표, 그래프, 논리 단위가 청크 경계에서 깨짐 |
| 구조 기반 | 제목, 섹션, 문단 경계 | 파서 구현 비용, 비정형 문서엔 전처리 필요 |

PDF, 스캔본 같은 비정형 문서는 청킹 전에 OCR 전처리로 텍스트, 레이아웃을 복원해야 구조 기반 청킹이 가능하다.

## 하이브리드 검색: 벡터 + BM25

벡터 검색은 의미 유사도를 잡지만 리터럴(정확 문자열) 매칭에 약하다. 주문번호 `A1234`, 모델명 `SM-G998N` 같은 토큰은 임베딩 공간에서 의미적으로 인접한 다른 번호와 뒤섞인다.

BM25(키워드 기반 랭킹)를 병행해 정확 매칭을 보완한다. 의미는 벡터가, 정확한 식별자는 BM25가 맡는 분업이다.

| 검색 | 강점 | 약점 |
|---|---|---|
| 벡터(임베딩) | 의미, 패러프레이즈, 동의어 | 주문번호, 코드, 고유명사 리터럴 |
| BM25(키워드) | 정확 토큰 매칭 | 표현이 다르면 못 잡음 |

두 결과를 점수 융합(예: RRF)해 단일 랭킹으로 합친다. 검색 엔진 레벨의 BM25, kNN 메커니즘은 [[OpenSearch]] 참고.

## 쿼리 재구성과 재탐색

1차 검색이 부족하면 끝내지 않고 에이전트가 쿼리를 재작성해 다시 탐색한다. 사용자의 모호한 질문을 검색 친화적 표현으로 분해, 확장하고, 결과가 빈약하면 키워드를 바꿔 재시도하는 ReAct 루프다. 한 번의 검색을 잘 짜는 것보다, 부족하면 다시 찾는 루프를 설계하는 게 견고하다.

## Progressive Disclosure 탐색

문서 전체를 한 번에 컨텍스트에 밀어넣지 않는다. 폴더 구조와 메타데이터만 먼저 컨텍스트로 주고, 에이전트가 tool-calling으로 매 턴 탐색 방향을 스스로 정한다. 코딩 에이전트가 디렉터리를 훑고 필요한 파일만 열어보는 패턴과 같다.

- 필요한 만큼씩만 로드 → 토큰 절약, 노이즈 감소
- 검색된 문서의 인접 문서를 자동 탐색해 끊긴 맥락을 보완
- 단발 검색이 아니라 다단계 탐색 세션으로 동작

[[Context-Engineering]]의 Select, Isolate 원칙이 검색 단계에 적용된 형태다.

## 계층적(Hierarchical) RAG

평면 청크 풀에서 top-k를 뽑는 방식은 문서 간 위계, 문서 내 구조를 무시한다. 계층 구조(문서 → 섹션 → 청크, 또는 요약 → 상세)를 두면 상위 레벨에서 후보를 좁히고 하위 레벨에서 정밀 검색해 기존 RAG 대비 정확도를 끌어올릴 수 있다. 상담, 실시간 서비스처럼 파일시스템 기반 탐색이 어려운 환경에서 특히 효과가 크다.

## 엔티티 추출 후 구조화 조회

RAG 고도화의 방향은 비정형 검색에서 구조화 조회로의 전환이다. 문서에서 엔티티(주문, 고객, 상품, 날짜)를 추출해 구조화 데이터로 적재하면, 모호한 의미 검색 대신 결정론적 조회(쿼리)로 답한다. 검색(retrieve)이 아니라 조회(lookup)가 되는 순간 정의가 명확해지고 비결정성이 줄어든다. [[Production-Agent-Architecture]]의 Metric Registry, 조회 우선순위와 같은 결.

## 도메인 사전: 구축 vs 미택

용어 사전(동의어, 약어, 도메인 명칭)을 구축하면 검색 재현율이 오른다. 하지만 수천 개 고객사가 제각각의 도메인 용어를 쓰는 멀티테넌트 환경에서는 사전을 전부 관리하는 운영 비용이 효과를 넘어선다. 이럴 땐 사전 구축을 의도적으로 포기(미택)하고 하이브리드 검색, 쿼리 재구성으로 커버하는 게 합리적이다. 모든 정확도 기법이 항상 옳은 게 아니라, 운영 복잡성 대비 효과로 취사선택한다.

## 면접 포인트

Q. RAG 품질이 낮을 때 어디부터 보나?
- 생성이 아니라 검색부터. 청킹이 논리 단위를 깨는지, 리터럴 매칭이 빠지는지, top-k가 엉뚱한지 순으로 확인한다.

Q. 벡터 검색만으로 부족한 이유는?
- 주문번호, 모델명 같은 정확 식별자에 약하다. BM25를 병행해 의미는 벡터가, 식별자는 키워드가 맡게 하고 점수를 융합한다.

Q. RAG를 더 고도화한다면?
- 엔티티를 추출해 구조화 데이터로 조회로 전환한다. 검색의 비결정성을 줄이고 정의를 명확히 한다. 계층적 RAG로 후보를 단계적으로 좁히는 것도 한 방향이다.

Q. 도메인 사전은 무조건 만드는 게 좋은가?
- 아니다. 멀티테넌트에서 용어 다양성이 크면 관리 비용이 효과를 넘는다. 운영 복잡성 대비 효과로 판단하고, 안 만드는 선택도 설계다.
