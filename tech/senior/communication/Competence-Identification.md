# 능력있는 개발자 판별 · Competent 가려내기

"개발자의 진짜 실력"을 경력·학력·말솜씨로 판단하는 것은 거의 불가능하다. 특히 **자기보다 윗레벨**의 실력은 면접으로 드러나지 않는다. 해결책은 단순하지만 강력하다 — **1시간 안에 코드를 교과서적으로 리팩토링하게** 한다. 레벨 간 격차가 결과물에 즉시 나타난다. 드레퓌스 모델 관점에서 Competent 이상을 가려내는 실전 기법 정리.

## 왜 면접으로는 안 되는가

### 선문답처럼 들리는 상급자의 답변

- 표면을 보는 사람에게 심층을 보는 사람의 대답은 **모호·사기꾼**처럼 들림
- 예: "객체가 뭐냐"는 질문에 "객체"라고 답함 — 객체를 가장 잘 표현하는 단어가 객체
- 반대로 고급 주제로 어필하면(Prototype·Duck Typing·온톨로지 Class) **뜬구름 잡는 설명**으로 인식됨
- 결국 말 잘하는 사람·화려한 용어 구사자가 오히려 유리해지는 역전

### 자기보다 아래는 쉬움

- 15분 면접이면 Advanced Beginner 이하는 파악 가능
- 하지만 이걸로는 Competent를 못 뽑는다

### 지식 양 vs 지혜

- Advanced Beginner: 책을 많이 읽고 키워드 나열 가능 → 면접에서 유창해 보임
- Competent: 키워드보다 **맥락 연결·트레이드오프 판단**이 강함 → 질문 하나에 짧은 답, 그 뒤의 긴 뉘앙스가 포착 안 됨

**결론**: 말로 실력을 가리면 Advanced Beginner가 Competent로 포장되기 쉽다.

## 실전 테스트: 1시간 코드 리뷰 과제

방법:

1. **현장에서** 기존 코드(적당한 난이도의 오픈소스 또는 회사 샘플) 제시
2. **1시간 동안** 리팩토링 과제
3. 작성자에게 **질문 가능** — 코드 의도 파악 허용
4. 결과물을 **교과서적 구조**와 비교

### 레벨별 예상 결과

| 레벨 | 1시간 내 결과 | 판별 |
|---|---|---|
| Novice | 큰 변화 없음, 기능도 깨뜨림 | 실력 부족 |
| Advanced Beginner | 흉내는 내지만 **책 보면서 고친다** | 아직 체화 안 됨 |
| **Competent** | **교과서적 구조로 정리** (디자인 패턴·분리·중복 제거) | 합격선 |
| Proficient | 5~15분 내 교과서적 구조 | 상위 |
| Expert | 구조 + 도메인·성능까지 | 최상위 |

Competent는 **자기 몸이 교과서**라 책 없이 1시간 안에 정리 가능.

### 체크 포인트

- 데이터 **묶음 구조** 식별 (Class·Struct) — 관련 필드의 응집
- **추상화** 적용 — Switch 지옥 → 다형성·전략 패턴
- **함수 분리** — 100줄 함수를 책임 단위로 쪼개기
- **확장 시 변경 범위** — 새 구현체 추가가 파급 최소화되는가
- **이름** — 변수·함수가 의도를 드러내는가

## 실제 관찰 포인트

### 나쁜 코드의 시그널

- 멤버 변수 **prefix 중복** + 클래스·구조체로 묶이지 않음 (예: `markerNames`, `markerIds` 따로)
- **Switch 폭증** — 한 조건(map type 등)이 여러 함수에 반복
- **거대 함수** — 100줄 이상
- 패턴 학습의 흔적은 있지만 실제 **적용 실패**

### 좋은 리팩토링의 시그널

- 관련 필드를 **Class로 캡슐화** (`Marker` class)
- **Adapter·Strategy**로 Switch 제거
- 공통 인터페이스 + 구체 구현 분리
- `type → Constructor` 매핑 테이블로 Switch를 **데이터화**
- 함수가 **단일 책임** 내 50줄 이하

## 팀 버프: Competent 이상을 중용해야 하는 이유

Competent 이상의 역할은 **본인 생산성 3배**가 전부가 아니다. "히어로의 팀 버프"처럼 조직 전체 능력을 증폭.

### 조직에 미치는 효과

- **지뢰 회피**: Advanced Beginner가 밟아야 할 실수의 절반 회피 → 팀 총 소실 시간 ↓
- **결과물 정리**: 품이 많이 드는 작업을 Advanced Beginner가 해도, Competent가 **정리·체계화**하면 사용성·안정성이 Competent 단독 작업 수준
- **학습 가속**: 아래 레벨이 **코드 변화 과정**을 관찰하며 빠르게 성장
- **조언 품질**: 업무 지시·피드백의 밀도가 높아 낭비가 준다
- **팀 생산성 1.2~1.5배** 추가 상승 (개인 3배 × 팀 버프 1.5배)

### 중용의 의미

- 단순 연봉 상승이 아니라 **기술 리딩 권한** 부여
- **팀 방식을 Competent 스타일로 강권** — 가볍게 권장하면 변화 없음
- 신입·Advanced Beginner를 **Competent 아래에 붙여** 가까이 관찰·모방 가능하게

## 조직이 Competent를 놓치는 이유

- **경력 기반 등급제** — 시간 채웠다고 올라가는 구조는 Competent를 판별 안 함
- **브랜드 채용** — 학력·전 회사 이름으로 뽑으면 말 잘하는 Advanced Beginner가 섞여 들어옴
- **화이트보드 알고리즘 일변도** — 구현 능력은 보지만 **설계 품질**은 못 봄
- **온라인 과제** — 시간 제한 없고 검색 가능해 실제 역량 구분 어려움

해결: **현장 1시간 코드 리팩토링** + **질문 허용** + **왜 이렇게 고쳤는가 인터뷰**

## 실전 채용 프로세스 제안

1. **온라인 스크리닝** — 기본기 필터 (Advanced Beginner 수준 이상)
2. **현장 코드 리팩토링 과제** — 1시간, 질문 허용, 결과물 채점
3. **리팩토링 설명 인터뷰** — 왜 이렇게 고쳤는가. 선택·버린 대안을 설명
4. **시스템 설계 대화** — 실제 업무 규모의 문제
5. **컬처 체크** — 매니저·동료 적합성

## 흔한 함정

- **리팩토링 과제 대신 알고리즘 퀴즈만** → 코드 품질 평가 누락
- **온라인으로 기간 길게** → 지식 없어도 검색으로 통과 가능
- **질문 금지** → 현실과 괴리. Competent도 모호한 코드는 작성자에게 물어본다
- **정답 비교만** → 구조의 이유를 묻지 않으면 진짜 이해인지 모름
- **코드 품질 기준 없음** → 면접관마다 편향. 교과서적 구조·디자인 패턴 기준으로 합의 필요

## 면접 체크포인트 (채용 담당자 시점)

- 1시간 리팩토링 과제에서 **데이터 묶기·추상화·함수 분리** 3가지가 모두 개선되었는가
- 작성자가 **왜 그렇게 고쳤는지**를 원리로 설명하는가
- Switch → Strategy/Adapter 같은 **디자인 패턴**을 이름 아는 수준이 아니라 실제 적용하는가
- 새 요구사항(예: 다음 지도 추가)에 대한 **변경 범위**를 정확히 예측하는가
- **짧은 답변의 밀도** — 말보다 행동(코드)에서 Competent 신호가 드러남
