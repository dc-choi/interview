# 개발자 커뮤니케이션 — 사람은 컴파일러가 아니다

개발자의 직업적 사고방식(엄밀함, 정확성 우선, 이분법적 판정)을 인간 소통에 그대로 적용하면 무의식적으로 갈등이 생긴다. 핵심 전제: **커뮤니케이션은 타고나는 재능이 아니라 코딩처럼 배우고 연습 가능한 기술**이다.

## 정확성 화법의 간극

컴파일러에게는 엄밀한 답이 미덕이지만 사람에게는 다르게 전달된다.

- 개발자의 "확인해봐야 알아요" (정직한 불확실성 표현) → 비개발자에게는 "모른다", "회피"로 해석
- 같은 단어도 맥락과 직군에 따라 다른 의미로 도착한다 — 말한 내용이 아니라 **도착한 내용**이 커뮤니케이션의 결과
- 보완: 불확실성을 말할 때 다음 행동과 시점을 붙인다 — "지금은 확답이 어렵고, 오늘 중 확인해서 알려드릴게요"

## 코드 리뷰의 온도

리뷰는 코드가 아니라 **동료와의 대화**다. "이 쿼리 성능 완전 망했네요"(판단)와 "N+1이 될 수 있을 것 같아요, Lazy loading 검토해볼까요?"(관찰 + 제안)의 차이가 심리적 안전감을 좌우한다. 리뷰 문화 전반(Pn 룰, 리뷰어 체크리스트, 톤 원칙)은 [[Code-Review-Culture|생산적 코드 리뷰 문화]].

## 메신저 — 응답 기대 시간을 명시적으로

- 플로우 상태가 끊기면 재집중에 평균 23분이 걸린다. 즉답 요구가 기본값이 되면 팀 전체의 몰입이 무너진다
- 해법: **채널별 응답 기대 시간을 팀 규범으로 명시** — 예: 프로덕션 장애 15분, 일반 논의 당일, 코드 리뷰 다음날
- 묻는 쪽 실천: 맥락을 갖춘 구체적 질문으로 왕복 횟수를 줄인다

## 비개발자와 일할 때 — 첫 질문은 왜

"이 기능 추가해주세요"에 "네, 알겠습니다"도 "그건 안 됩니다"도 부적절하다.

1. **"왜 이 기능이 필요한가요?"** 부터 — 요구사항 뒤의 문제를 파악하면 더 싼 해법이 나올 수 있다
2. **옵션과 비용으로 제시** — "X 방식은 2주, Y 방식은 1일" 식으로 의사결정을 상대에게 돌려준다 ("안 된다" 대신 대안 제시 패턴은 [[Cross-Functional-Product-Collaboration|직군 간 프로덕트 협업]])

## 기술 부채의 번역

"레거시 코드 리팩토링이 필요합니다"는 비개발자에게 의미가 도착하지 않는다. **비즈니스 임팩트로 재구성**한다 — "지금 구조로는 기능 추가마다 시간이 2배로 들고 있어서, 이번에 정리하면 다음 분기 개발 속도가 올라갑니다". 개발 언어를 성과 언어로 바꾸는 축은 [[Engineering-Business-Alignment|개발팀의 비즈니스 align]]과 같다.

## 갈등 — 피하는 것이 아니라 잘 다루는 것

- 갈등이 없는 조용한 팀이 건강한 팀은 아니다 — 이견이 표면화되지 않을 뿐
- "그건 아닌 것 같은데요"(차단) 대신 "제가 우려하는 건 X인데, 잠깐 들어주실 수 있나요?"(우려 + 공간) — 상대가 표현할 공간을 남기는 화법
- 솔직함과 배려의 균형 프레임은 [[Radical-Candor-Feedback|Radical Candor, I-message]]

## 실천 체크리스트

- 코드 리뷰에 **왜**를 함께 쓴다
- PR에 맥락 설명을 붙인다
- 회의 요청 전에 안건을 먼저 공유한다
- 메신저 답변은 구체적으로, 질문은 맥락을 갖춰서
- 불확실성엔 다음 행동과 시점을 붙인다

## 체크포인트

- 정확성 화법이 비개발자에게 어떻게 오해되는지 예시로 설명할 수 있는가
- 판단형 리뷰와 관찰 + 제안형 리뷰의 차이
- 채널별 응답 기대 시간을 규범화하는 이유 (플로우 보호)
- 기능 요청에 대한 첫 반응이 수락/거절이 아니라 **왜**인 이유
- 기술 부채를 비즈니스 언어로 번역하는 구조 (현상 → 비용 → 개선 후 이득)
- 갈등 회피와 갈등 관리의 차이
