# UI 용어 정리

기획서와 스펙 문서에서 쓰이는 UI 컴포넌트 용어. 회사마다 조금씩 다르게 쓰지만, 문서에서 용어를 명확히 구분해야 디자이너와 개발자가 의도(특히 동작 차이)를 정확히 구현한다.

## 기본 컴포넌트

| 용어 | 정의 | 비고 |
|---|---|---|
| 스플래시 화면 | 앱 실행 후 메인 로드 전 1~5초 노출되는 화면 | 초기화 시간 확보 + 브랜드 인식 |
| 당겨서 새로고침 (Pull to Refresh) | 화면 상단을 아래로 끌어 갱신 | 모바일 표준 갱신 UX |
| GNB (Global Navigation Bar) | 상단 고정 메뉴 — 어느 화면에서든 주요 메뉴로 이동 | 하단 내비게이션과 역할 분담 |
| 하단 내비게이션 | 어디서든 접근해야 하는 필수 기능 배치 | 모바일에서 엄지 접근성 최상 |
| 플로팅 버튼 (FAB) | 콘텐츠 위에 떠서 고정 위치를 지키는 버튼 | 핵심 액션 1개 강조 |
| 툴팁 | 요소 옆에 부연 설명을 띄우는 작은 레이어 | 신기능 안내, 짧은 도움말 |

## 선택 컴포넌트: 체크박스 vs 라디오 버튼

- **체크박스** — 단일 또는 다중 선택. 각 항목이 독립적 on/off.
- **라디오 버튼** — 2개 이상 중 **반드시 1개만** 선택하는 배타적 선택.

스펙에 어느 쪽인지 명시해야 한다 — 선택 모델(다중 허용 여부)이 데이터 구조와 검증 로직까지 결정한다.

## 팝업 계열의 구분 (가장 헷갈리는 영역)

화면 위에 새로 뜨는 것을 뭉뚱그려 팝업이라 부르지만, 동작이 다르다:

| 용어 | 동작 | 핵심 차이 |
|---|---|---|
| 팝업(창) | 독립된 새 창을 연다 | **기존 화면 조작 가능** |
| 모달 | 기존 화면 위에 레이어를 얹는다 | **닫기 전까지 기존 화면 조작 불가 (블로킹)** |
| 레이어 팝업 | 별도 레이어로 만든 팝업 (논블로킹도 가능) | 모달 여부를 따로 명시해야 함 |
| 시스템 팝업 | OS 기본 다이얼로그 사용 | 권한 요청 등 — 디자인 커스텀 불가 |

### 컨펌 vs 알럿

- **컨펌(Confirm)** — 취소/확인처럼 **2개 이상의 선택지**를 주고 사용자의 결정을 받는다.
- **알럿(Alert)** — 확인/닫기 **단일 버튼**으로 정보만 전달한다.

파괴적, 비가역적 액션(삭제, 결제, 환불 승인)에는 알럿이 아니라 컨펌을 쓴다 — 사용자에게 결정의 기회를 줘야 한다.

## 터치 제스처

| 제스처 | 동작 | 주 용도 |
|---|---|---|
| 탭 (Tap) | 짧게 한 번 두드림 | 선택, 버튼 — PC의 클릭. 가장 기본이고 안전 |
| 더블 탭 | 두 번 연속 두드림 | 확대/축소 토글, 좋아요 — 실수 유발 위험 있음 |
| 프레스 (Press, 길게 누르기) | 꾹 누르고 유지 | 단축, 보조 기능 — 발견과 학습이 어려움 |
| 스와이프 (Swipe) | 한 직선 방향으로 이동 | 스크롤, 콘텐츠 넘김 — 가장 광범위하게 적용 |
| 패닝 (Panning) | 떼지 않고 여러 방향으로 이동 | 지도 등 큰 화면의 세밀한 탐색 |
| 플리킹 (Flick) | 빠르게 긋고 손을 뗌 | 모멘텀(관성) 스크롤 — 뗀 뒤에도 움직임 지속 |
| 핀치 (Pinch) | 두 손가락을 모음 | 축소 |
| 스프레드 (Spread) | 두 손가락을 벌림 | 확대 |
| 로테이트 (Rotate) | 두 손가락 회전 | 화면, 지도 회전 |

### 제스처 설계 유의점

- **프레스는 보조 기능에만** — 눈에 보이지 않아 IT 리터러시가 낮은 사용자는 학습하기 어렵다. 핵심 기능을 프레스 뒤에 숨기면 안 된다.
- **더블 탭은 신중하게** — 의도치 않은 두 번 탭으로 발동될 수 있어 파괴적 액션에는 쓰지 않는다.
- **플리킹과 스와이프의 경계는 구현자가 정의한다** — 추상적 범주일 뿐 명확한 경계가 없어서, 시작점, 이동 거리, 속도 임계값으로 자의적으로 규정된다. 대부분의 앱은 둘을 구분하지 않는다 — 사용자는 제스처 구분을 신경 쓰지 않으므로 구분 자체가 피로를 만든다. 플리킹이 명확히 분리되는 드문 예는 공 던지기 게임(손을 뗀 뒤에도 관성으로 이동)뿐.
- **대체 접근 경로를 함께 제공한다** — 제스처는 눈에 보이지 않는 인터페이스라서, 스와이프로 여는 기능은 메뉴나 탭 같은 가시적 경로로도 접근 가능해야 한다.
- **메타포로 제스처 가능성을 암시한다** — 그래버/드래그 핸들(≡ 아이콘, 순서 변경 가능 암시), 닷 네비게이션(점 개수로 추가 콘텐츠 존재 표시), 다음 콘텐츠의 끄트머리 일부 노출(스와이프 유인). 신호 없는 제스처는 발견되지 않는다.
- **온보딩에서 안내한다** — 코치 마크(레이어 형식의 조작법 안내)나 첫 항목에 거는 애니메이션(숨겨진 버튼을 살짝 노출)으로 학습 시점을 만든다. 안내 배치 원칙은 [[User-Guidance-Patterns|사용자 가이드 제공 패턴]].
- **플랫폼 디자인 가이드를 준수한다** — OS 표준과 충돌하는 제스처(하단 탭 영역의 좌우 스와이프 등)는 다른 앱에서 학습된 기대([[UX-Laws|제이콥의 법칙]])를 깨서 혼동을 만든다.
- **의도적 마찰의 예외** — 실수 방지가 목적이면 일부러 개입이 더 필요한 제스처를 쓴다 (밀어서 결제). 금전적, 파괴적 액션에서 진짜 의사가 있는 사용자만 통과시키는 마찰 설계.

## 면접 체크포인트

- 스펙 리뷰에서 모달인지 팝업인지의 구분은 단순 용어 문제가 아니라 동작 차이다 — 모달은 블로킹이므로 그 뒤 화면의 상태 갱신, 포커스 처리까지 영향을 준다.
- 파괴적 액션의 컨펌 강제는 어드민, 백오피스 설계의 기본기 — [[Visual-Hierarchy|시각적 위계]]의 위험 액션 대비 설계와 한 세트.
- 용어 통일은 협업 비용 절감 장치다 — 기획, 디자인, 개발이 같은 단어로 같은 동작을 떠올리게 하는 것이 스펙 문서의 역할.
