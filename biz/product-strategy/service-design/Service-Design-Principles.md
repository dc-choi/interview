# 서비스 설계 10원칙 (GOV.UK)

영국 정부 디지털 서비스 조직(GDS)이 정립한 설계 원칙. 부처별로 난립하던 정부 웹사이트를 단일 서비스(gov.uk)로 통합하며 검증됐고, 공공 서비스를 넘어 어드민, B2B 도구, 일반 프로덕트 설계에 그대로 적용된다.

## 10원칙

| 원칙 | 핵심 |
|---|---|
| 1. 사용자 니즈 우선 | 추측 금지 — 데이터, 조사, 사용자와의 대화로 설계 |
| 2. 적게 하라 | 핵심만 직접 만들고, 나머지는 재사용 플랫폼과 API로 제공 |
| 3. 데이터 기반 설계 | 프로토타입 테스트와 분석이 의사결정에 선행 |
| 4. 단순성 추구 | 항상 그래왔다는 이유를 거부, 복잡성은 시스템 뒤로 숨김 |
| 5. 반복과 개선 | MVP → 알파 → 베타 단계 반복, 작은 실패 권장 |
| 6. 포용적 설계 | 접근성이 최우선, 미적 요소는 후순위 |
| 7. 맥락 이해 | 사용자의 환경과 능력까지 고려 |
| 8. 디지털 서비스 | 웹사이트가 아니라 끝까지 완결되는 서비스를 만든다 |
| 9. 일관성 | 디자인 시스템으로 컴포넌트와 톤을 통일 |
| 10. 투명성 | 코드, 아이디어, 실패까지 공개 공유 |

## 주요 원칙의 실무 해석

### 검색엔진이 우리의 홈이다

사용자는 서비스 사이트를 직접 방문하지 않고 검색으로 진입한다. 따라서 첫 화면 디자인보다 **SEO에 전력**하는 것이 진짜 입구 설계다 ([[Content-Marketing|콘텐츠 마케팅]]의 SEO). 검색 결과 상단에 공식 페이지가 노출되는가가 서비스 도달률을 결정한다.

### 포용적 설계는 대체 수단 제공

휴대폰을 잃어버렸거나 청각 장애가 있을 수 있다는 전제로 설계한다 — 문자 인증만 제공하지 않고 음성 인증을 병행하는 식. 접근성은 마지막에 덧붙이는 체크리스트가 아니라 기능 명세의 일부다.

### 명사보다 동사

도메인 용어를 사용자 언어로 번역한다. 수수료 감면(제도 명칭)보다 수수료 비용을 지원하다(사용자가 얻는 것)가 이해도 낮은 사용자도 움직이게 한다. [[Product-Roadmap|프로덕트 로드맵]]의 Theme 작성 원칙(기능이 아니라 고객 가치 문장)과 같은 축.

### 디자인 시스템

서비스가 커지면 담당자와 페이지가 늘며 일관성이 무너진다. 컴포넌트와 톤을 묶은 디자인 시스템이 해법이고, GOV.UK 디자인 시스템은 공개돼 있어 어드민, 백오피스 기획의 참고 자료로 바로 쓸 수 있다 — 검증된 공공 표준이라 개발자 설득도 쉽다.

### 적게 하라

모든 것을 직접 만들지 않는다. 공통 기능은 재사용 가능한 플랫폼과 API로 제공하고 각 팀은 고유 문제에 집중한다 — 플랫폼 팀, 공통 컴포넌트 사고의 원형.

기능 단위에서도 같은 원칙이 적용된다 — 혹시 모르니까 만들어 둔 기능은 주머니 열 개 달린 건빵바지처럼 실제로는 기본 주머니 두 개만 쓰인다. 기능의 기본값은 추가가 아니라 **필요성 증명**이고, 비즈니스 목표와 고객 UX 양쪽에서 효과를 검증하지 못한 기능은 제거도 설계 결정이다. 미사용 기능도 공짜가 아니다 — 화면 복잡도, 유지보수, 테스트, 마이그레이션 비용을 계속 낸다.

## 면접 체크포인트

- 어드민, 백오피스 설계 질문에 GOV.UK 디자인 시스템 같은 공개 표준을 참고했다는 답은 근거 있는 설계 결정의 좋은 예가 된다.
- 접근성 질문에 포용적 설계 원칙으로 — 대체 수단(인증 채널 다중화)은 장애 대응 설계와도 연결된다.
- 적게 하라 원칙은 공통 플랫폼화, API 우선 설계와 연결해 말할 수 있다.
- 단순성 원칙의 한 줄 — 사용자에게 단순해 보이려면 복잡성을 없애는 게 아니라 시스템 뒤로 옮겨야 한다. 백엔드가 무게를 지는 이유.
