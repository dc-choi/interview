# PRD (Product Requirements Document) 작성법

제품을 만들거나 업데이트할 때 **무엇을 왜 만들지**를 기획 단계에서 정리하는 문서. 개발·디자인·QA·비즈니스 조직이 같은 맥락을 공유하고 의사결정 시 판단 기준으로 쓰는 **협업 계약서** 성격을 가진다.

## 단계별 역할

| 단계 | PRD가 하는 일 |
|---|---|
| 기획 | 제품 비전·전략 정의, 문제·기회 정량화 |
| 디자인 | 사용자 요구·흐름 검증, 화면 단위 요구 정합 확인 |
| 개발 | 기능 목적·구현 범위 확인, 스코프 기준선 |
| QA·출시 | 기술·시장 요구 충족 검증, 회귀 테스트 기준 |

## 구성 요소 (7축)

1. **제품 개요** — 목적, 해결하려는 문제, 근거 데이터(설문·인터뷰·시장 지표)
2. **비즈니스 목표** — 기대 성과·KPI·목표 수치 (정성 목표만 쓰면 의사결정 기준이 안 됨)
3. **핵심 사용자** — 페르소나·세분화. "모두를 위한 제품"은 PRD에서 금지 신호
4. **사용자 여정** — CUJ(Critical User Journey), 사용자 스토리 문법 `[사용자]는 [목적]을 위해 [행동]을 원한다`
5. **기능적 요구사항** — MoSCoW 우선순위(Must / Should / Could / Won't)
6. **일정 및 배포 계획** — 개발·출시 타임라인, 마케팅·CS 부서 협의 포인트
7. **참고 문서** — 와이어프레임, 스토리보드, UX 리서치, 경쟁사 레퍼런스

## 작성 원칙

### 기능 나열 ≠ 요구사항 문서

모든 기능에 `이것이 해결하는 Pain은 무엇인가`를 물어 답을 남긴다. 답이 없으면 기능에서 빼거나 우선순위를 내린다.

### 근거 기반 우선순위

- **사용자 근거**: 인터뷰 수, 설문 응답, 정량 지표
- **비즈니스 근거**: 매출 영향, 리텐션·전환 영향, 전략 적합도
- **기술 근거**: 개발 공수, 의존성, 리스크

근거 없이 "중요하다"로 우선순위를 매긴 PRD는 개발 단계에서 우선순위가 뒤집힌다.

### 범위(Scope)와 비범위(Non-Scope) 명시

무엇을 한다보다 **무엇을 하지 않는다**를 적는 게 분쟁 예방에 더 효과적. "이번에는 A까지만, B는 차기"가 명확해야 스코프 크리프를 막는다.

### Living Document

배포 후 피드백을 반영해 계속 업데이트한다. 첫 작성 시 완벽을 노리기보다 `작게 만들고 → 검증하고 → 개선하는 반복`이 실질적으로 유효하다. 처음부터 방대한 PRD를 써내려가면 개발 중 현실과 괴리돼 아무도 안 본다.

## MoSCoW 우선순위 예시 (음식 주문 앱)

| 레벨 | 예시 |
|---|---|
| **Must Have** | 메뉴 선택·결제 — 없으면 제품이 성립 안 함 |
| **Should Have** | 메뉴 검색 — 매우 필요하지만 없어도 주문은 가능 |
| **Could Have** | 할인 배너·이벤트 노출 — 있으면 좋지만 MVP에선 제외 가능 |
| **Won't Have** | 리뷰 작성 — 이번 릴리즈에서는 의도적으로 제외 |

## 협업에서의 역할

PRD는 **의견 대립 시 판단 기준점**이다. "디자이너와 개발자가 기능 해석이 다름" 상황에서 PRD가 최종 근거가 되므로:

- 애매한 문장·형용사(빠르게, 직관적으로, 쉽게) 대신 **측정 가능한 기준**을 쓴다
- 이해관계자(개발·디자인·QA·영업·CX)와 초안부터 함께 리뷰한다 — 최종본 일방 공유는 PRD의 역할을 스스로 포기하는 것
- 변경 이력을 남긴다 (언제·왜 바뀌었는지)

## 좋은 PRD의 기준

- 신규 인원이 PRD만 읽고 **제품의 목적·사용자·핵심 흐름**을 파악할 수 있다
- 구현 중 "이게 스코프 안인지" 물어봤을 때 PRD로 **답을 낼 수 있다**
- 릴리즈 후 목표 달성 여부를 **PRD의 KPI로 검증**할 수 있다
- Non-Scope가 명시되어 있어 "이것도 해야 하는 거 아니냐"는 사후 분쟁이 없다

## 흔한 실수

- **기능 나열서**로 쓴다 — 왜 필요한지 없음
- **형용사만 가득** — "빠르게", "쉽게", "직관적으로"로만 기준을 서술
- **페르소나 없이 시작** — 사용자 없는 기능 정의
- **일정만 있고 근거 없음** — 왜 그 시점에 내야 하는지, 왜 이만한 공수가 필요한지 없음
- **배포 후 업데이트 안 함** — Dead Document로 전락

## 면접 체크포인트

- 실제로 작성·리딩한 PRD가 있다면 **구성 7축 중 어떤 부분에서 팀 내 의사결정이 바뀌었는지** 에피소드 하나를 준비
- MoSCoW 우선순위를 실제로 조정한 사례 (Must → Should로 내린 근거)
- Non-Scope를 명시해서 스코프 크리프를 막은 경험
- 배포 후 KPI가 기대치에 못 미쳐 PRD를 업데이트·재기획한 과정
