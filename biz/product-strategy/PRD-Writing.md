# PRD (Product Requirements Document) 작성법

제품을 만들거나 업데이트할 때 **무엇을 왜 만들지**를 기획 단계에서 정리하는 문서. 개발, 디자인, QA, 비즈니스 조직이 같은 맥락을 공유하고 의사결정 시 판단 기준으로 쓰는 **협업 계약서** 성격을 가진다.

## 단계별 역할

| 단계 | PRD가 하는 일 |
|---|---|
| 기획 | 제품 비전, 전략 정의, 문제, 기회 정량화 |
| 디자인 | 사용자 요구, 흐름 검증, 화면 단위 요구 정합 확인 |
| 개발 | 기능 목적, 구현 범위 확인, 스코프 기준선 |
| QA, 출시 | 기술, 시장 요구 충족 검증, 회귀 테스트 기준 |

## 구성 요소 (7축)

1. **제품 개요** — 목적, 해결하려는 문제, 근거 데이터(설문, 인터뷰, 시장 지표)
2. **비즈니스 목표** — 기대 성과, KPI, 목표 수치 (정성 목표만 쓰면 의사결정 기준이 안 됨)
3. **핵심 사용자** — 페르소나, 세분화. "모두를 위한 제품"은 PRD에서 금지 신호
4. **사용자 여정** — CUJ(Critical User Journey), 사용자 스토리 문법 `[사용자]는 [목적]을 위해 [행동]을 원한다`
5. **기능적 요구사항** — MoSCoW 우선순위(Must / Should / Could / Won't)
6. **일정 및 배포 계획** — 개발, 출시 타임라인, 마케팅, CS 부서 협의 포인트
7. **참고 문서** — 와이어프레임, 스토리보드, UX 리서치, 경쟁사 레퍼런스

## 유저 시나리오 vs 유저 스토리

구성 요소의 사용자 여정 축을 채우는 두 도구. 본질은 같다 — 기획 과정에서 스며드는 공급자 마인드를 깨고 **사용자 관점을 강제하는 장치**다.

| 구분 | 유저 시나리오 | 유저 스토리 |
|---|---|---|
| 형태 | 상세한 서사 텍스트 | 한 문장 |
| 목적 | 사용자 맥락의 이해 | 요구사항의 스펙화 |
| 시점 | 초기 기획, 리서치 | 개발 단계 (애자일 백로그) |
| 상세도 | 높음 | 낮음 |

**유저 시나리오** — 타겟 페르소나가 특정 상황에서 목표를 이루기 위해 어떻게 행동할지를 서술한다. 5요소: Who(페르소나 디테일), What(이루려는 목표), When과 Where(시점과 장소 — 장애물 포함), Why(왜 필요한지). 기술 배경이 없는 사람도 이해하게 쓰고, 문화적 특성이나 정보 습득 차이처럼 사용에 영향을 주는 요인까지 담는다.

**유저 스토리** — As a [역할] / I want [행동] / So that [목적]의 한 문장. 단순해서 기획, 디자인, 개발의 공용어가 되고, 애자일에서 작은 마일스톤 단위로 업무를 쪼개는 기준이 된다. JTBD의 Job Statement([[User-Feedback|사용자 피드백 관리]])와 형제 포맷 — JTBD는 상황(When)에서, 유저 스토리는 역할(Who)에서 출발한다.

유저 스토리의 한계: 정확한 스펙 정의가 아니라서 협의가 따라야 하고, **비기능 요구(성능, 보안, 데이터 정합성)를 놓치기 쉽다** — 백엔드가 백로그 리뷰에서 짚어야 할 단골 누락 지점이다. 수백 케이스의 대규모 프로젝트에는 부적합.

## 유저플로우 (User Flow)

사용자가 목표를 달성하기 위해 거치는 행동의 연속을 시각화한 다이어그램. 시나리오(맥락), 스토리(요구) 다음 단계의 도구로, 화면이 아니라 동선을 설계하라는 원칙의 실행 방법이다. 기획서 초반에 배치해 전체 사용 동선을 먼저 이해시키는 커뮤니케이션 장치로 쓴다.

| 단계 | 산출물 | 내용 |
|---|---|---|
| 1. User Goal | 한 문장 | 유저가 이루려는 목표 (예: 상품을 스크랩한다) |
| 2. Task Flow | 단계 목록 | 목표 달성의 단계 구분 (리스트 진입 → 상세 이동 → 스크랩 클릭) |
| 3. Wire Flow | 와이어프레임 + 플로우차트 | 화면 뼈대 위에 흐름을 얹어 화면 간 이동을 시각화 |
| 4. User Flow | 조건 분기 포함 다이어그램 | 만약에 시나리오(비로그인이면 로그인 유도, 리스트에서 바로 스크랩 등)까지 정리 |

핵심은 4단계의 **조건 분기**다 — 정상 경로만 그린 플로우는 절반이고, 비로그인, 오류, 빈 상태 분기가 위 화면 명세의 부정 케이스 누락을 막는다. 백엔드 관점에서 플로우의 분기점은 곧 API의 상태 분기와 권한 체크 지점이다 — 플로우에 없는 분기는 엣지 케이스 버그로 돌아온다.

## 작성 원칙

### 기능 나열 ≠ 요구사항 문서

모든 기능에 `이것이 해결하는 Pain은 무엇인가`를 물어 답을 남긴다. 답이 없으면 기능에서 빼거나 우선순위를 내린다.

### 근거 기반 우선순위

- **사용자 근거**: 인터뷰 수, 설문 응답, 정량 지표
- **비즈니스 근거**: 매출 영향, 리텐션, 전환 영향, 전략 적합도
- **기술 근거**: 개발 공수, 의존성, 리스크

근거 없이 "중요하다"로 우선순위를 매긴 PRD는 개발 단계에서 우선순위가 뒤집힌다.

### 범위(Scope)와 비범위(Non-Scope) 명시

무엇을 한다보다 **무엇을 하지 않는다**를 적는 게 분쟁 예방에 더 효과적. "이번에는 A까지만, B는 차기"가 명확해야 스코프 크리프를 막는다.

### Living Document

배포 후 피드백을 반영해 계속 업데이트한다. 첫 작성 시 완벽을 노리기보다 `작게 만들고 → 검증하고 → 개선하는 반복`이 실질적으로 유효하다. 처음부터 방대한 PRD를 써내려가면 개발 중 현실과 괴리돼 아무도 안 본다.

## MoSCoW 우선순위 예시 (음식 주문 앱)

| 레벨 | 예시 |
|---|---|
| **Must Have** | 메뉴 선택, 결제 — 없으면 제품이 성립 안 함 |
| **Should Have** | 메뉴 검색 — 매우 필요하지만 없어도 주문은 가능 |
| **Could Have** | 할인 배너, 이벤트 노출 — 있으면 좋지만 MVP에선 제외 가능 |
| **Won't Have** | 리뷰 작성 — 이번 릴리즈에서는 의도적으로 제외 |

## 화면 명세(스토리보드)의 실무 디테일

문서 전체 구조가 잡혀도 화면 단위 명세에서 반복되는 실수들이 있다.

1. **여백을 남긴다** — 페이지를 처음부터 빽빽하게 채우면 추가 케이스(빈 상태 화면 등)가 생겼을 때 페이지 재배치, 번호 재정렬, 상호 참조(16p 참고 식) 수정이 연쇄적으로 터진다. 미래의 추가를 전제로 널널하게.
2. **화면이 아니라 동선을 설계한다** — 정적 문서를 보다 보면 개별 화면 사고에 갇힌다. 닫기 버튼을 그렸으면 닫은 다음에 어디로 가는지까지가 스펙이다. 사용자는 평면이 아니라 흐름으로 움직인다.
3. **용어 일관성** — 같은 기능을 페이지마다 다른 이름으로 부르면 개발자는 다른 기능으로 오인한다. 문서 서두나 Appendix에 용어를 정의하고, 공용 용어는 별도 용어집([[UI-Terminology|UI 용어 정리]])에 링크.
4. **타이틀에 기능과 범위를** — 플러스 버튼 클릭 시 화면(모호)이 아니라 플러스 버튼 레이어 - 스티커, 구분선 첨부 (1/3)처럼 내용, 순서, 범위가 읽히게.
5. **부정 케이스를 초반에** — 정상 흐름만 설계하면 이탈, 오류, 빈 상태 케이스가 디자인과 개발 단계에서 역류해 전원의 재작업이 된다. 초반에 네 번 고민하는 쪽이 전체 공수가 싸다.

## 협업에서의 역할

PRD는 **의견 대립 시 판단 기준점**이다. "디자이너와 개발자가 기능 해석이 다름" 상황에서 PRD가 최종 근거가 되므로:

- 애매한 문장, 형용사(빠르게, 직관적으로, 쉽게) 대신 **측정 가능한 기준**을 쓴다
- 이해관계자(개발, 디자인, QA, 영업, CX)와 초안부터 함께 리뷰한다 — 최종본 일방 공유는 PRD의 역할을 스스로 포기하는 것
- 변경 이력을 남긴다 (언제, 왜 바뀌었는지)

## 좋은 PRD의 기준

- 신규 인원이 PRD만 읽고 **제품의 목적, 사용자, 핵심 흐름**을 파악할 수 있다
- 구현 중 "이게 스코프 안인지" 물어봤을 때 PRD로 **답을 낼 수 있다**
- 릴리즈 후 목표 달성 여부를 **PRD의 KPI로 검증**할 수 있다
- Non-Scope가 명시되어 있어 "이것도 해야 하는 거 아니냐"는 사후 분쟁이 없다

## 흔한 실수

- **기능 나열서**로 쓴다 — 왜 필요한지 없음
- **형용사만 가득** — "빠르게", "쉽게", "직관적으로"로만 기준을 서술
- **페르소나 없이 시작** — 사용자 없는 기능 정의
- **일정만 있고 근거 없음** — 왜 그 시점에 내야 하는지, 왜 이만한 공수가 필요한지 없음
- **배포 후 업데이트 안 함** — Dead Document로 전락

## 면접 체크포인트

- 실제로 작성, 리딩한 PRD가 있다면 **구성 7축 중 어떤 부분에서 팀 내 의사결정이 바뀌었는지** 에피소드 하나를 준비
- MoSCoW 우선순위를 실제로 조정한 사례 (Must → Should로 내린 근거)
- Non-Scope를 명시해서 스코프 크리프를 막은 경험
- 배포 후 KPI가 기대치에 못 미쳐 PRD를 업데이트, 재기획한 과정
