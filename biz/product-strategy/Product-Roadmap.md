# 프로덕트 로드맵

제품의 비전, 방향, 우선순위를 시간 축 위에 요약한 전략 문서. 용도는 세 가지 — 일의 우선순위 결정, 제한된 리소스의 분배, 이해관계자에게 제품 방향을 설명하고 설득하는 것. 엔지니어 관점의 단계 설계와 운영(의존성 순서, 리스크 후순위, 완료 기준)은 [[Tech-Roadmap|기술 로드맵 수립]]에서 다룬다.

## 수립 4단계

| 단계 | 내용 |
|---|---|
| 1. Vision 확립 | 서비스가 고객과 사회에 제공할 궁극적 가치, 미션 |
| 2. Product Goal | Vision을 측정 가능한 목표로 전환 |
| 3. Theme으로 묶기 | 과제들을 고객 가치 중심의 주제로 통합 |
| 4. 우선순위 | 제한된 리소스 안에서 중요도 판단 |

위에서 아래로 도출된다 — Vision이 명확해야 담을 과제가 뚜렷해지고, Goal이 있어야 과제가 산발적이지 않다.

### 1. Vision

서비스의 궁극적 목표. 추상적이어도 되지만 방향은 분명해야 한다 (예: 전 세계 정보를 체계화해 누구나 편리하게 접근하게 한다).

### 2. Product Goal

추상적 Vision을 이행 가능한 목표로 구체화한다. 자주 쓰이는 유형: 경쟁 차별화, 고객 만족도 향상, 기술 개선, LTV 증대, 이탈률 감소, 신규 서비스 업셀, 지역 확장, 모바일 전환. 목표가 정해지면 세부 과제가 따라 나온다 — 이탈 고객 줄이기 → 가입 단계 축소, CRM 푸시 알림 같은 식.

### 3. Theme

과제를 묶을 때 기능 단위가 아니라 **고객이 느낄 가치 문장**으로 표현한다.

- 나쁜 예: 푸시 알림 개선
- 좋은 예: 사용자는 푸시 알림으로 필요한 정보를 적시에 받아본다

로드맵에 스토리가 생기고, 이해관계자에게 단순 개선 목록이 아니라 고객 가치 창출로 읽힌다. Outcome을 기능이 아닌 지표, 가치로 정의하는 [[User-Feedback|Continuous Discovery]]의 원칙과 같은 축.

### 4. 우선순위

- **Kano 모델** — 과제가 당위적(없으면 불만), 일차원적(충족 비례 만족), 매력적(있으면 감동) 품질 중 어디 속하는지로 시급도를 가른다. 5유형 상세 표는 [[User-Feedback|사용자 피드백 관리]].
- **Story Mapping** — 고객이 제품을 사용하는 여정 단계에 기능을 매핑하고, 여정에서 먼저 쓰이는 기능부터 만든다. 기능 A → B → C → D 순서로 사용된다면 제한된 일정에서 A부터. MVP 범위와 배포 순서를 정하는 도구.

정량 점수(RICE, ICE)와 범위 합의(MoSCoW)는 [[User-Feedback|사용자 피드백 관리]]의 우선순위 프레임워크 참조.

## 시각화와 운영

- 시각화는 타임라인 기반(기간 축에 작업 배치) 또는 파트별(기획, 디자인, 서버, 앱 별 진행 현황) 형태가 일반적이다.
- **Living Document 원칙** — 로드맵은 한 번 만들고 방치하는 산출물이 아니라 부서 의견과 데이터가 주기적으로 반영되는 살아있는 문서다. [[PRD-Writing|PRD 작성법]]의 Living Document, [[Tech-Roadmap|기술 로드맵]]의 데이터 기반 조정과 같은 원칙.

## 면접 체크포인트

- 로드맵 수립 질문에는 Vision → Goal → Theme → 우선순위의 도출 순서로 답한다 — 과제 나열이 아니라 비전에서 내려오는 구조를 보여주는 것이 핵심.
- Theme을 고객 가치 문장으로 말하는 습관 — 기술 과제도 그 결과 고객이 얻는 것으로 번역.
- 엔지니어로서는 의존성 기반 순서, 리스크 후순위, 단계별 독립 가치([[Tech-Roadmap|기술 로드맵]])를 보완 축으로 함께 답할 수 있다.
