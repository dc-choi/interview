# 개인화와 추천

## 개인화 vs 커스터마이징

| 구분 | 개인화 (Personalization) | 커스터마이징 (Customizing) |
|---|---|---|
| 주체 | 시스템 — 고객이 요청하지 않아도 자동으로 맞춤 | 고객 — 직접 UI나 서비스를 설정 |
| 입력 | 행동 데이터 분석 | 명시적 설정값 |

상반된 개념이 아니라 같은 화면에서 동시에 쓰일 수 있다 (추천 피드 + 관심 카테고리 직접 설정).

## 추천의 3요소

- **추천 대상** — 상품, 기획전, 카테고리. 무엇을 추천하는가
- **추천 범위** — 전체 풀 vs 수동 등록된 후보군
- **추천 로직** — 고정 추천(사람이 지정) vs 자동 선별(알고리즘)

## 데이터 수준에 따른 개인화 5단계

보유한 데이터의 수준이 가능한 개인화의 수준을 결정한다. 수준이 올라갈수록 인프라와 운영 비용도 커진다.

| 수준 | 데이터 | 활용 | 한계 |
|---|---|---|---|
| 1 | 고객이 저장한 데이터 (위시리스트, 장바구니) | 명시적 관심사 기반 노출 | 주문내역 재노출은 박탈감 유발 — 이미 산 상품은 제외 |
| 2 | 성별, 연령 | 기본 타겟팅 | 개인정보보호 이슈로 사용 감소 추세 |
| 3 | 쿠키 기반 행동 (최근 본 상품) | 세션 맥락 추천 | 멀티 디바이스에서 휘발 — 서버 저장으로 보완 |
| 4 | 연관 분석 (함께 구매한, 함께 본) | 협업 필터링형 추천 | 입력 데이터 오염, 관심사 변화 미반영 |
| 5 | 실시간 로그 분석 (EBM, Event Based Marketing) | 행동에 즉각 대응 | 효과적인 시나리오 설계가 어려움 |

### 끝단: 구매 여정의 축약 (선제 커머스)

개인화의 종착점은 구매 여정 4단계(니즈 인식 → 탐색 → 결제 → 배송)를 줄이는 것이다 — IoT가 소모품 수명을 감지해 니즈 인식을 대신하고, 대화형 커머스(챗봇, 음성)가 탐색을 대행하고, 생체인증과 자동 결제가 결제를 지운다. 신뢰받는 비서 모델: 멤버십으로 고정 고객층을 묶고 음성 주문과 구매 대행까지 가져가면, 추천은 제안을 넘어 구매 의사결정 자체를 위임받는다. 신뢰가 쌓일수록 할인 경쟁 없이 전환되므로 수익 구조도 함께 좋아진다.

## 운영 함정

- **데이터 오염** — 오픈 전 테스트 주문 데이터가 연관 분석에 섞이면 무관한 상품이 함께 구매한 상품으로 고정 노출된다. 입력 데이터 검증과 재학습 주기가 운영 요건이다.
- **유사 상품 도배** — 한 번 보거나 산 상품과 유사한 것만 반복 노출하면 추천 효용이 떨어진다. 정확도와 탐색 다양성의 트레이드오프.
- **이미 구매한 상품의 재추천** — 명시적 제외 규칙이 필요하다 (소모품 재구매는 예외).
- 알고리즘이 자동화돼도 무시할 데이터와 집중할 고객층을 가르는 판단은 사람의 몫으로 남는다.

## 추천 AI의 윤리: FAT

딥러닝 기반 추천은 인간이 과정을 평가하기 어려운 알고리즘을 단시간에 만들어낸다 — 블랙박스 문제. 그래서 공정성(Fairness), 책임감(Accountability), 투명성(Transparency)의 FAT 원칙이 추천 시스템의 운영 요건으로 들어온다.

| 원칙 | 추천에서의 문제 | 대응 |
|---|---|---|
| 공정성 | 광고비 많은 상품, 마진 높은 상품으로 편향된 추천 | 고객 유용성 기준의 중립 평가, 컴플라이언스 절차 |
| 책임감 | 민감 콘텐츠가 알고리즘을 타고 노출 | 카테고리별 포커스 그룹, 노출 조정 |
| 투명성 | 추천 근거 불명 (왜 이게 나에게?) | 추천 이유를 사용자에게 명시 — 상호 신뢰 장치 |

실무 장치로는 항의 버튼(이상 결과를 사용자가 신고 — 피드백 루프), 핵심 가치를 정의하는 포커스 그룹, 윤리 시나리오 학습이 제안된다. 주의할 역설 하나 — **추천이 정확할수록 위험할 수 있다.** 본인이 밝히지 않은 사실(임신 등)을 추천 결과가 주변에 노출하는 사례처럼, 정밀한 프로파일링은 그 자체가 프라이버시 침해가 된다.

## 백엔드 관점

- 최근 본 상품을 쿠키(디바이스 종속)에 둘지 서버(계정 종속)에 둘지는 멀티 디바이스 일관성의 저장 위치 결정 문제다.
- 데이터 수준 4와 5의 차이는 파이프라인의 차이다 — 연관 추천은 배치 집계, EBM은 실시간 스트림 처리.
- 추천 모델의 출력과 실제 노출 사이에는 서빙 시점 필터(품절, 이미 구매, 연령 제한)라는 비즈니스 규칙 계층이 있다.
- 투명성 요건은 API 설계로 내려온다 — 추천 응답에 추천 이유 메타데이터를 함께 내리는 구조, 신고(항의 버튼) 이벤트를 모델 개선으로 되돌리는 피드백 파이프라인.

## 면접 체크포인트

- 추천 기능 제안은 데이터 수준부터 — 지금 보유한 데이터로 가능한 단계와, 다음 단계에 필요한 수집을 구분해 말해야 현실적인 답이 된다.
- 알고리즘 고도화 이전에 제외 규칙(이미 구매, 품절, 테스트 데이터)이 추천 품질의 체감을 좌우한다.
