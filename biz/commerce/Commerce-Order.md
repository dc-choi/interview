# 커머스 주문(Order) 도메인

## 주문의 위치

주문은 거래의 중심축이다. 주문에 의해 결제와 배송이 발생하고, 클레임에 따라 추가 배송이나 결제금액 변경이 발생한다. 따라서 주문 영역을 고민할 때는 **결제, 배송, 클레임, 정산을 함께** 고려해야 한다 — 주문만 떼어 설계할 수 없다.

## 주문 상태와 전환 트리거

상태는 명사(현재 어디에 있는가), 전환 트리거는 이벤트(무엇이 일어났는가)로 구분된다.

| 전환 트리거 (이벤트) | 전환 후 상태 |
|---|---|
| 상품 주문완료 (결제완료) | 주문 접수 |
| 주문확인 | 상품준비중 (출고처리중) |
| 출고 완료처리 (송장등록, 집하지시) | 배송준비중 (집하대기중) |
| 택배사 인계 (집하완료) | 배송중 |
| 배송완료 (도착처리) | 배송완료 |
| 구매확정 | 구매확정 |

### 출고와 배송의 세부 상태

- **상품준비중 (출고 과정)**: 피킹완료 → 포장완료 → 상차대기
- **배송중 (배송 과정)**: 집하접수 → 상품이동중 → 배송지도착 → 배송지출발

### 처리 상태의 일반화

다양한 서비스의 처리 상태는 **지시 → 처리중 → 종결**로 일반화할 수 있다. 출고든 배송이든 클레임이든 이 패턴의 변주다.

## 장바구니와 주문서

| 구분 | 장바구니 | 주문서 |
|---|---|---|
| 목적 | 주저하는 고객의 주문 유도, 추가 구매 유도 | 주문 체결 |
| 성격 | 주문 전 임시저장, 관심 상품 보관 | 거래 계약 체결과 결제 (동의 절차 = 계약 서명) |
| 주요 기능 | 상품 추가와 삭제, 추천, 옵션과 혜택 비교, 수량과 가격과 배송비 확인 | 최종 금액 확인, 배송정보 입력, 결제 연결 |
| 핵심 지표 | 더 많이 담고 주문서로 전환되는 비율 | 빠르게 동의하고 주문완료로 넘어가는 속도 |

주문서는 오래 머물수록 유실률이 높아진다 — 장바구니는 오래 머물게, 주문서는 빨리 통과하게 설계한다. 목적은 다르지만 사용자가 장바구니를 생략하고 주문서로 직행(바로 구매)하는 경우가 많아 두 화면의 정보 구성은 비슷해진다.

### 주문서를 빨리 통과시키는 3가지 설계

| 기법 | 내용 |
|---|---|
| 경로 차단 | 배너, 액션바 등 이탈 경로를 제거하고 결제에만 집중하는 화면 |
| 디폴트 처리 | 배송지, 결제수단, 할인 혜택을 자동 설정 — 입력 마찰 최소화 |
| 확인 단계 생략 | 멘탈 모델에 맞춰 순서를 재배치하고 단계를 줄임. 극단이 원클릭 주문(페이지 없이 결제 완료) |

순서 자체도 사용 환경의 멘탈 모델을 따라 바뀐다 — PC에서는 상품 → 배송지 → 금액 → 결제수단이었지만, 모바일에서는 배송지 → 상품과 가격 → 결제로 단순화됐다. 백엔드 관점에서 디폴트 처리는 최근 배송지와 결제수단의 저장, 자동 적용 문제이고, 원클릭 주문은 결제수단 토큰(빌링키) 저장 위에서만 성립한다 — UX 단순화의 비용을 보안(토큰 관리, 본인 인증)이 치르는 구조다.

### 주문서 이원화 (간단 주문서)

주문서까지 도달한 고객도 다수가 이탈한다. 데이터로 보면 모바일 주문의 대부분은 단순 케이스다 — 단건 주문이 60% 이상, 1개 상품에 일반택배가 전체의 절반, 신용카드 비중 약 82%에 같은 카드 재사용률 약 90%, 주문서 체류 5초 미만이 절반. 고객이 반드시 확인하는 것은 **최종결제금액, 결제수단, 배송지** 셋뿐이고 나머지 부가 기능은 유사시에만 쓰인다.

그래서 가장 흔한 케이스(1개 상품 + 일반택배)에는 부수 옵션 입력창을 숨기고 디폴트(최대 할인 자동 적용, 이전 배송지와 결제수단)를 강화한 **간단 주문서**를 자동 노출하고, 복수 상품과 특수 배송은 기존 주문서로 보내는 이원화가 성립한다 — 힉스의 법칙(선택지가 많을수록 결정이 느려짐, [[UX-Laws|Laws of UX]])의 적용이다. 단, 디폴트가 강할수록 잘못된 배송지로 발송되는 류의 사고가 따라오므로 배송지 변경 가능 시점 정책을 같이 설계해야 한다.

진행 방법도 전형적이다: 정성 분석(자가 관찰 — 화면 녹화, 주변 피드백)으로 가설을 세우고 → 정량(주문 로그)으로 검증하고 → 오픈 후 KPI로 효율을 추적한다 ([[Commerce-Member|커머스 회원 도메인]]의 데이터 기반 개선 프레임과 같은 축). 간단 주문서 분기 조건(상품 수, 배송 유형) 판정은 클라이언트가 아니라 서버의 몫이다.

### 주문서에 임시저장이 없는 이유

문서 도구라면 당연한 임시저장이 주문서에 없는 것은 누락이 아니라 의도된 설계다.

1. **정합성** — 재고, 가격, 할인은 실시간 정보라 주문 생성 시점에 재계산해야 한다. 임시저장된 주문서는 저장 시점의 낡은 스냅샷이어서 불러오는 순간 전부 재검증해야 하므로 저장의 의미가 없다. 모놀리식은 주문 시점 재계산으로 끝나지만, 프론트 주문 DB와 주문 생성 시스템이 분리된 구조에서는 정합성 비용이 더 커진다.
2. **쓰레기 데이터** — 주문서의 구매 전환율은 1~3% 내외다. 임시저장을 만들면 결제로 이어지지 않는 97%의 데이터가 축적되고, 그대로 스토리지 비용이 된다.

대안은 저장이 아니라 재구성이다 — 자주 바뀌지 않는 값은 디폴트로 기억하고(위 간단 주문서), 미완료 주문은 장바구니가 받친다. 장바구니가 곧 주문서의 임시저장 역할인 셈이다. 기능 요구를 받았을 때 데이터의 수명과 전환율을 근거로 거절하거나 대안을 제시하는 전형적 사례이기도 하다.

### 위시리스트 vs 장바구니: 기능 아이덴티티 문제

| 구분 | 위시리스트 (찜, 관심상품, 스크랩) | 장바구니 |
|---|---|---|
| 목적 | 갖고 싶은 상품의 저장 | 한 번에 구매할 상품의 저장 |
| 옵션 선택 | 불필요 — 상품 목록에서 바로 담기 가능 | 필수 (수량, 옵션 확정) |
| 구매 전환율 | 낮음 | 높음 |

둘 다 담는 리스트라 고객은 잘 구분하지 않고, 전환율은 장바구니가 압도적이라 위시리스트는 존재 이유가 계속 흔들려 온 기능이다. 단골 중심 시대에는 재방문 도구(품절 상품을 담아두고 입고를 기다림)였지만, 가격 비교와 단타 방문이 일반화되며 위상이 약해졌다. 재정의 방향은 세 갈래다:

1. **마찰 없는 저장** — 옵션 선택 없이 목록에서 바로 담기. 장바구니와의 차별점을 진입 마찰 최소화에 둔다.
2. **정기 구매 상품 관리** — 단골 상품 전용 리스트. 단, 장바구니에 삭제 방지 옵션을 주는 식으로 대체 가능해 차별성이 약하다.
3. **저장 대상의 확장** — 상품 단위를 넘어 매장, 브랜드, 기획전 찜(단골 매장). 장바구니가 못 하는 영역이라 가장 확실한 차별화다.

설계 교훈: 비슷한 기능 둘이 공존하려면 각자의 아이덴티티(이름, 진입점, 핵심 지표)가 분명해야 한다 — 구분이 안 되는 기능은 활성화 시책을 아무리 부어도 살아나지 않는다. 백엔드 관점에서는 장바구니가 SKU(옵션, 수량) 단위 모델인 반면, 위시리스트는 찜 대상이 상품, 매장, 브랜드로 확장될 수 있어 폴리모픽 참조 모델이 된다.

## 결제 프로세스

고객이 선택한 상품의 가격을 지불하고 구매를 완료하는 과정.

결제 수단 선택 → 결제 정보 입력 → 결제 정보 검증 → 결제 승인 또는 거절

이 뒤에서 도는 인프라(PG와 VAN, 인증 → 승인 → 매입 4단계, 간편결제 토큰)는 [[Payment-Service|결제 서비스]] 참조.

## 주문 클레임

주문의 상태에 영향을 주는 고객 요청. 판단 트리로 분류한다.

1. **주문과 관련 있는가** — 아니면 개선 요청, 기타 불만 (클레임 아님)
2. **주문 상태에 영향을 주는가** — 예: 취소, 교환, 반품 / 아니오: 배송지 변경, 수량 변경, 옵션 변경
3. **후속처리가 필요한가** — 예: 교환, 반품 (회수, 재출고) / 아니오: 취소

### 클레임 유형별 핵심 구분: 상품 이동 시작 여부

| 유형 | 가능 시점 | 처리 |
|---|---|---|
| 주문 취소 | 상품 이동 시작 전 | 거래 자체를 철회 — 없던 계약으로 만들고 환불 |
| 반품 | 상품 이동 시작 후 | 거래 철회 불가 — 상품 반납 절차 후 환불 |
| 교환 | 상품 이동 시작 후 | 기존 상품 회수와 교환 상품 출고를 동시 진행 |
| 배송지, 수량, 옵션 변경 | 주로 상품 이동 전 (취소 가능 상태) | 출고와 배송에 영향을 주는 기타 클레임 |
| 결제수단 변경 | 이동 상태와 무관 | 출고와 배송에 영향 없는 기타 클레임 |

단, 중개(오픈마켓) 구조에서는 구매확정 후 플랫폼 시스템상 반품, 교환, 부분 취소가 제한되는 경우가 많다. 다만 전자상거래 청약철회권, 하자 담보, 판매자와의 별도 처리 가능성은 남으므로 법적 불가와 플랫폼 정책 제한을 구분해야 한다 ([[Commerce-Overview|커머스 도메인 개요]]의 판매자 구분 참조).

### 취소의 회색 지대: 고객 상태와 물류 상태의 불일치

상품 이동 시작 전이라는 기준은 화면에 보이는 것보다 이르다 — 주문이 들어가는 순간 물류센터에서는 픽업과 포장이 이미 진행되고, 이 단계의 취소는 상품을 원위치로 되돌리는 인건비를 실제로 발생시킨다. 고객에게는 발송 전(취소 가능해 보임)인데 셀러에게는 작업 진행 중인 회색 지대가 생기는 것. 배송 상태 추적이 정교해질수록 발송 전 취소 요청은 늘어나는데(정보 공개의 역설) 작업 비용은 사라지지 않으므로, 셀러와 구매자의 마찰이 구조화된다.

중개 플랫폼은 이 마찰을 강제로 정리하지 못한다 — 셀러는 통신판매업자로서 고유 권한을 가진 거래 당사자라 플랫폼이 취소 수용을 강제할 수 없다. 재고 뻥튀기로 인한 주문 후 품절 취소([[Commerce-Korea-Market|한국 이커머스 시장의 특수성]])와 함께, 중개 구조에서 플랫폼이 거래 품질을 직접 보장하지 못하는 두 단면이다. 시스템 관점의 절충은 취소 가능 판정을 화면 상태가 아니라 **물류 작업 상태(피킹 시작 여부)와 동기화**하는 것 — 상태 머신에 출고 작업 단계를 반영해야 회색 지대가 줄어든다.

### 반품 프로세스

상품 회수 → 입고 → 검수 → 귀책 여부 체크 → 환불 → 재판매 또는 폐기

귀책 여부(판매자 귀책 vs 구매자 단순변심)에 따라 반품 배송비 부담 주체와 환불 금액이 달라진다.

### 유형 상품 vs 무형 상품

| 구분 | 주문의 결과 | 클레임 종류 |
|---|---|---|
| 유형 상품 | 상품의 배송 처리 | 취소, 반품, 교환, 재배송 |
| 무형 상품 | 티켓의 발급 처리 | 취소, 재발급 |

### 클레임 정책이 다뤄야 하는 것

- 요청 주체, 요청 내용, 사유에 따른 클레임 구분
- 클레임 처리에 따른 주문 상태와 조건 변경
- 추가 비용 결제, 환불, 재결제 처리
- 추가 출고와 배송 처리
- 주문 정산의 처리 (이미 정산된 건의 차감)
- 접수와 처리 주체에 따른 승인 업무 프로세스

## 면접 체크포인트

- 주문 상태를 단일 ENUM 컬럼으로 관리하면 생기는 문제 — 주문, 결제, 배송, 클레임은 각자의 상태 머신을 가진다. [[Ecommerce-Shopping-Mall-ERD|이커머스 쇼핑몰 ERD]] 참조.
- 주문, 결제, 배송이 1:1이 아닌 이유 — 부분 취소, 분할 배송, 복수 결제 수단.
- 취소와 반품의 경계가 왜 상품 이동 시작 시점인가 — 물류 비용 발생 여부가 거래 철회 가능성을 가른다.
- 결제 정보 검증 단계의 역할 — 클라이언트가 보낸 금액을 신뢰하지 않고 서버에서 재계산, 결제 승인의 멱등성 처리.
- 클레임 처리가 정산에 미치는 영향 — 환불은 결제만의 문제가 아니라 판매자 정산 차감까지 이어지는 흐름.
