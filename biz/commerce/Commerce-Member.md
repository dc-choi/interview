# 커머스 회원(Member) 도메인

## 회원의 정의: Consumer > Customer > Member

잠재 이용자 전체에서 서비스 가입자로 좁혀지는 계층 구조.

| 계층 | 정의 | 예시 (패션 커머스 기준) |
|---|---|---|
| Consumer (소비자) | 그 상품군을 소비하는 모든 사람 | 옷을 입는 사람 전부 |
| Customer (고객) | 서비스에 접점이 생긴 사람 | 쇼핑몰을 둘러보는 사람 |
| Member (회원) | 특정 서비스에 가입한 이용자 | 계정을 만든 사람 |

비회원 주문을 허용하는 커머스라면 Customer와 Member의 구분이 시스템에도 그대로 반영된다 (비회원 주문 모델, 주문 조회용 별도 인증).

### 계층 전환의 사업 가치

계층 구분이 중요한 이유는 객단가 격차다 — 팬덤 비즈니스 추정에서 코어 팬의 연 매출은 일반 소비자의 약 7배(14만 원 vs 2만 원). 사업의 핵심은 하위 계층(일반 소비자, 라이트 유저)을 상위 계층(코어)으로 끌어올리는 전환 장치의 설계다.

전환 장치의 고전이 **문간에 발 들여놓기(foot-in-the-door)** — 거절하기 힘든 가벼운 개입부터 시작해 점진적으로 깊은 개입으로 끌어올린다. 로그인 없는 웹 기반 가벼운 콘텐츠(꾸미기 놀이) → 앱 설치 → 일정 확인 같은 심화 활동 → 소비로 이어지는 단계 설계. 통합 플랫폼의 진입장벽이 부담스러운 일반 소비자를 타겟 전용 경량 앱으로 받는 구조가 대표 사례다.

## 회원 가입

계정을 생성해 서비스 이용 자격과 식별 수단(열쇠)을 부여하는 절차. 동의는 네 범주로 나뉜다:

| 동의 항목 | 성격 |
|---|---|
| 서비스 기본 약관 | 필수 — 이용 계약 조건 |
| 개인정보 수집과 활용 | 필수 + 선택 — 행동정보 수집 범위 |
| 마케팅 정보 수신 | 선택 — 이메일, SMS 광고 |
| 제3자 정보 제공 | 선택 — 타 법인과의 공유, 데이터 통합의 법적 관문 |

개인정보는 직간접적으로 개인을 식별할 수 있는 정보 전부를 말한다 — 선호를 드러내는 수준의 행동정보도 포함된다. 확실한 개인 식별에는 본인확인 기반 식별값(CI)이 쓰인다.

## 로그인과 사용자 식별 수단

| 수단 | 특징 |
|---|---|
| ID/PW | 기본 수단. 탈취, 재사용 공격에 취약해 단독 사용은 지양 추세 |
| 휴대폰 인증 | 본인확인 겸용, 한국 커머스에서 사실상 표준 |
| 이메일 인증 | 가입 확인, 비밀번호 재설정에 주로 사용 |
| 공동인증서 | 고액 결제, 금융 연동 시 |
| OTP | 2차 인증(2FA) |
| 생체인증 | 모바일 앱 간편 로그인 (FIDO) |
| 카드인증 | 결제 수단 기반 본인확인 |

식별 수단이 강할수록 단일 계정 정책을 강제하기 쉬워진다.

## 계정 정책: 단일 vs 복수

| 정책 | 내용 | 트레이드오프 |
|---|---|---|
| 단일 계정 | 사용자 1명당 계정 1개. 강한 인증 수단(휴대폰 본인확인 등)으로 강제 | 어뷰징 방지, 데이터 일관성 확보 / 가입 마찰 증가 |
| 복수 계정 | 사용자 1명당 여러 계정 허용 | 사용성과 가입 전환율 우수 / 쿠폰 어뷰징 등 악의적 사용 여지 |

신규 가입 쿠폰, 첫 구매 혜택이 있는 커머스에서 복수 계정은 곧 어뷰징 비용이다. 혜택 설계와 계정 정책은 함께 결정해야 한다.

## 통합회원제(SSO)와 데이터 통합의 한계

여러 계열 서비스를 한 ID로 쓰는 통합회원제를 도입해도 통합 빅데이터는 자동으로 생기지 않는다.

| 제약 | 내용 |
|---|---|
| 이중약관 | SSO든 SNS 로그인 매핑이든, 통합 멤버십 약관 + 개별 서비스 약관의 2단계 동의 구조가 된다 |
| 소유 vs 보유 | 행동정보는 발생한 서비스(법인)의 소유다. 통합 멤버십사는 보유만 할 뿐 활용, 처분을 결정할 수 없고, 타사 공유에는 개인별 제3자 제공 동의가 필수 |
| 발생 위치 분산 | SSO는 기본정보만 내려받는다. 로그와 주문 같은 행동정보는 각 서비스 서버에 적재되므로 통합회원 레벨에서는 모아볼 수 없다 |

CI가 있어도 본인 동의 없이 여러 회원제의 정보를 엮을 수 없다. 겉으로 합쳐진 회원제도 속에서는 찢어져 있는 셈이고, 데이터 통합이 목적이라면 인증 통합(SSO)이 아니라 **처음부터 회원과 행동정보를 하나의 데이터 통으로 설계**해야 한다. 결제라는 단일 접점에서 합법적으로 데이터를 모으는 우회 구조는 [[Payment-Service|결제 서비스]]의 데이터 시야 참조.

오프라인 임대몰은 이 단절의 물리적 형태다 — 한 쇼핑몰에서 여러 브랜드 상품을 사면 영수증이 두 장 나온다 (브랜드: 상품 상세 / 몰 운영사: 총액, 결제수단, 방문시간). 각 주체는 단편만 갖고, 같은 그룹이라도 법인이 다르면 지점 간 고객정보 공유가 불가능하며, 포인트 적립 동의가 없으면 거래는 익명으로 남는다. 추천에 필요한 두 조건(구매자 식별 + 상품 단위 상세)이 기술이 아니라 **법인 구조와 동의 체계**에 막히는 것 — 쓸만한 데이터는 무턱대고 쌓이는 게 아니라 목적에 맞는 형태로 설계해야 쌓인다는 원칙의 실증이다.

백엔드 관점: SSO는 인증의 통합이지 데이터의 통합이 아니다 — 이 구분이 시스템 경계(어느 DB에 무엇이 쌓이는가)와 법적 컴플라이언스(어떤 동의로 무엇을 조인할 수 있는가)를 동시에 가른다.

## 멤버십과 구독

- **멤버십(Membership) 회원**: 특정 혜택을 받을 자격을 가진 회원. 무료 멤버십(등급제 등)과 유료 멤버십으로 나뉜다.
- **구독 서비스**: 월정액 또는 정기 결제를 통해 제공되는 서비스 (쿠팡 와우, 네이버플러스 멤버십).
- 유료 멤버십은 락인 수단으로 작동한다 — 멤버십 비용을 회수하려는 심리가 플랫폼 내 구매 빈도를 높인다.

### 번들형 멤버십의 선순환

멤버십에 비커머스 혜택(OTT 콘텐츠 등)을 묶으면 락인이 한 단계 강해진다 — 주기적 시청이 플랫폼 재방문을 만들고, 멤버십 회원의 객단가는 비회원의 2배 이상이라는 선순환(콘텐츠 → 재방문 → 구매 증가)이 프라임 모델의 골격이다. 멤버십 가입비를 낮게 두는 것은 그 자체가 수익원이 아니라 **진입장벽을 낮춰 선순환에 태우려는 설계**다.

번들 혜택은 외부 제휴로 채우는 방식과 직접 운영(자체 OTT — PB 상품처럼 주도권 확보)으로 갈린다. 직접 운영은 통제력을 얻지만 콘텐츠 수급이 과제로 남는다.

### 자체 간편결제(페이)의 다중 효과

플랫폼이 자체 페이를 내재화하면 세 가지를 동시에 얻는다 — ① 결제 수수료라는 수익원 ② 충전 잔액 기반의 락인 ③ 개인 간 거래에서 계좌번호 노출 없이 결제하는 **익명성 = 신뢰 장치**. C2C 플랫폼이 수익모델 부재 상태에서 페이를 첫 수익화 카드로 꺼내는 이유가 이 조합이다.

## 마이페이지

회원이 자기 상태를 확인하고 스스로 처리하는 공간. 기능은 여섯 범주로 정리된다:

| 범주 | 항목 |
|---|---|
| 개인 자산 | 포인트, 쿠폰 |
| 대시보드 | 주문, 배송 상태 확인 |
| 클레임 처리 | 변경, 취소, 교환, 반품 |
| 알림 | 푸시 등 알림 리스트 |
| 개인 활동 | 상품평, Q&A |
| 기타 | 이벤트, CS 서비스, 회원정보 |

이용률은 주문배송조회 > 쿠폰과 포인트 > 위시리스트와 최근 본 상품 > 상담 > 상품평과 이벤트 > 회원정보 순이 전형적이다. 마이페이지의 핵심 비즈니스 지표는 **CS 자가처리율** — 클레임 메뉴를 찾지 못한 고객은 그대로 상담 비용이 되므로, 주문 대시보드와 클레임 동선이 1순위 직무다.

### 데이터 기반 페이지 개선 프레임

마이페이지처럼 구성이 단순한 페이지일수록 감에 의한 개선안은 설득력이 없다 — 클릭 로그가 출발점이고, 메뉴 축소 같은 이해관계자 설득에도 데이터만 한 근거가 없다.

1. **유입 경로 분석** — 어디서 들어오는가 (헤더, 햄버거 메뉴, 퀵메뉴)
2. **기능 중복 검토** — 다른 페이지와 겹치는 기능 확인
3. **버튼별 이용률 측정** — 클릭 수 순위
4. **저사용 요소 정리** — 유사 메뉴 통폐합, 하단 축소 배치
5. **전략적 강조** — 사업이 키우려는 기능을 상단 배치 (작성 가능 상품평 건수 노출 등)

효과 측정은 1, 3, 6개월 단위로 나눠 본다 — 동선 변화에 고객이 익숙해지는 데 3~6개월이 걸리므로, 오픈 직후의 급등(숨어 있던 메뉴 클릭이 10배 이상 뛰는 등)과 3개월 후의 안착 수치를 구분해 성공 여부를 판단한다. 데이터 기반 기획은 강제 장치 없이는 관행이 되기 어렵다 — 측정 시점을 프로세스로 박아두는 이유다.

## 판매자 회원

오픈마켓과 입점형 플랫폼에서 판매자는 구매자와 별도의 회원 유형이다. 판매자 역시 플랫폼의 직접, 간접 매출에 기여하는 고객으로 본다.

판매자와의 주요 거래 관계:
- 쇼핑몰 입점 고객 (입점, 판매 수수료)
- 물류 위탁 화주 (풀필먼트 수수료)
- 마케팅 광고주 (광고 집행)

### 입점의 데이터 비대칭

브랜드사(판매자) 입장에서 온라인 입점은 오프라인 입점과 본질이 다르다 — 오프라인은 장소 임대(부동산)라 고객을 직접 만나고 VOC도 직접 받지만, 온라인은 데이터 플랫폼이라 **고객 정보(성별, 연령, 구매 경로)를 플랫폼이 독점**하고 브랜드는 자기 고객을 볼 수 없다. 상품 정보는 브랜드 동의 없이 계열 채널과 제휴 채널로 재판매되기도 한다. 입점 채널은 판매 채널인 동시에 **고객 접점 주도권의 경쟁자**다 — 셀러 어드민의 리포트 제공 범위가 플랫폼의 협상 카드가 되는 이유이고, 브랜드사가 자사몰(D2C)을 병행하는 동기이기도 하다. 브랜드 측의 대응 전략 전체는 [[D2C-Brand-Strategy|D2C와 브랜드 데이터 전략]] 참조.

## 구매자와 판매자의 플랫폼 Flow

- **구매자 Flow**: 회원가입 → 상품 탐색 → 상품 주문 → 개인화 서비스
- **판매자 Flow**: 입점 계약 → 상품 전시 → 상품 판매 → 리포트 분석 서비스

같은 플랫폼이라도 구매자용 서비스와 판매자용 셀러 어드민은 사실상 별개의 제품이며, 회원 모델도 분리 설계되는 경우가 많다.

## 면접 체크포인트

- 비회원 주문을 지원해야 한다면 회원 모델을 어떻게 설계할 것인가 — Customer와 Member의 분리.
- 구매자와 판매자 계정을 한 모델로 둘 것인가 분리할 것인가 — 인증은 공유하되 역할(Role)과 프로필을 분리하는 패턴이 일반적.
- 신규 가입 혜택 어뷰징 방지(본인확인 기반 단일 계정)와 가입 전환율 사이의 트레이드오프를 설명할 수 있는가.
