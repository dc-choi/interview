# 이력서 15초 승부 · 5원칙

채용담당자가 이력서 한 장을 판단하는 시간은 **수 초, 길어야 15초**. 첫 페이지에서 "면접으로 이어갈 만한가"가 결정된다. **30초 광고 영상을 설계하듯** 임팩트·성취·정량화·맞춤·가독성 5원칙으로 써야 한다. 기본 구성·포맷·자소서 작성법은 [[Resume-Portfolio-Guide]]에 있고, 이 문서는 **"읽히기 위한 원칙"** 에 집중.

## 원칙 1 — 임팩트에 집중

첫 페이지의 밀도가 전체 운명을 결정한다.

- **첫 페이지가 가장 중요** — 본편(면접)으로 관심을 끌어오는 트레일러
- **짧은 문장** — 한 줄, 최대 두 줄. 장문은 반드시 글머리 기호로 분해
- **기술 스택 나열은 아래로** — 단순 "Python·Java·AWS"는 정보 가치 낮음. 대신:
  - **난이도 있는 자격증**(AWS Solutions Architect Associate 등) 학습 노력 증빙
  - **실체 기여**로 증명 (프로젝트·블로그·OSS 링크)
- 추가 정보는 **블로그·GitHub·포트폴리오 링크**로 확장

### 흔한 실수

- 첫 페이지에 학력·자격증·긴 자기소개를 몰아넣어 **본인 성과가 2페이지부터** 시작
- 기술 스택을 **분홍·파랑 배지**로 화려하게 나열 → 정보보다 시선 분산
- 첫 줄을 "안녕하세요 ...입니다"로 시작 → 생산성 저하

## 원칙 2 — 성취 중심

"했다"가 아니라 **"이만큼 했다"**.

### 구체적 서술

- "담당 개발"(X) → **"DAU 10만 서비스에서 결제 모듈 신규 개발"**(O)
- 개선 작업은 **Before/After 수치** 필수:
  - "API 성능 개선"(X) → "API 응답시간 220ms → 80ms (64% 감소)"(O)
  - "비용 절감"(X) → "월 운영비 300만원 → 50만원"(O)

### 큰 성취 우선 배치

- 여러 프로젝트 중 **기여도가 높은 프로젝트**를 최상단
- 대표 프로젝트는 "무엇을 물어봐도 술술 나올 수준" — **10분 분량 서사** 준비
- 면접관이 **먼저 배치된 것부터** 질문함을 전제로 순서 배치

### 대표 프로젝트 필수 구성

1. **프로젝트 소개** — 무엇을 만들었는가
2. **시스템 구성** — 아키텍처·기술 스택
3. **기여 사항** — 본인이 담당한 범위
4. **성과** — 수치화된 결과
5. **트러블슈팅** — 겪은 문제·해결 과정
6. **아쉬운 점** — 돌이켜봤을 때의 개선점

## 원칙 3 — 정량화

숫자 없는 성과는 **기억에 남지 않는다**.

### 숫자로 영향력 강화

- "많은 사용자"(X) → **"1,000명 중 65.7%"**(O)
- 감소보다 **개선율로 표현** — "에러 70% 감소" vs **"에러율 3.3배 축소"** — 후자가 임팩트
- "빨라졌다"(X) → **"P99 응답시간 800ms → 180ms"**(O)

### 다양한 측정 항목

프로그래머의 성과는 코드 속도에 국한되지 않는다.

- **수익·비용** — 매출 기여, 운영비 절감, FinOps
- **프로세스** — 배포 시간, 리드타임, MTTR
- **사용자** — DAU·MAU·리텐션·전환율
- **리더십** — 멘토링 대상자 수·신규 채용 기여
- **지식 공유** — 사내 세미나·블로그·기술 문서

코딩 외 **기획·회의·장애 대응** 성과도 함께 기록할 가치가 있다.

## 원칙 4 — 각 역할에 맞게 수정

### 은탄환은 없다

- **모든 회사에 통용되는 이력서는 불가능**
- 지원 포지션 JD의 **요구사항 키워드**를 이력서에 반영
- 관련 경험이 있다면 반드시 **전진 배치**
- JD에 "대용량 트래픽 경험"이 있으면 해당 경험을 최상단으로

### 필요 없는 내용은 삭제

- 해당 역할과 무관한 프로젝트·경험은 **제거**
- 단, **특정 산업 관련 경력**은 상황 따라 강점 — 핀테크·커머스·게임·의료
- 고민 포인트: "이 문장이 이 포지션 채용에 0.1%라도 기여하는가?"

### 버전 관리

- 회사별로 **fork** 해서 관리
- 공통 마스터 이력서 + 회사별 커스터마이즈 버전
- 지원 시점마다 최신 정보 반영

## 원칙 5 — 보기 좋은 떡

가독성이 떨어지면 **아무리 내용이 좋아도 안 읽힌다**.

### 템플릿

- **글꼴 3개 이하** (제목·본문·코드)
- **색상 최소** — 강조 1~2색
- **적절한 여백** — 섹션 구분이 명확
- **인쇄했을 때 잘 읽히는가** 테스트 — 화면에서만 예쁜 건 신호

### 피드백 받기

- **직무를 모르는 사람**에게도 이해되는가 체크 — 전문용어 과다 방지
- **종이로 인쇄**해서 빨간 펜으로 피드백 받기 — 화면보다 집중도 높음
- 지인에게 **JD와 함께 전달**하면 모의 면접관 역할 가능
- 전문 리쿠르터·채용 담당 경험자 리뷰는 **유료라도 가치** 있음

### 포맷

- **PDF**로만 제출 — Word·HWP 호환성 문제
- 파일명에 이름·포지션 포함 — `홍길동_백엔드개발자_2026.pdf`
- 2~3페이지 이내 — 경력자도 대부분 3페이지를 넘기면 안 읽힘

## 반복이 원칙

이력서는 한 번에 완성되지 않는다.

- **분기 단위 업데이트** — 최근 성과 즉시 반영
- 구직 시작 후 쓰려 하면 **기억이 흐려져** 성과를 숫자화 못 함
- 평소 **성과 로그**를 별도로 관리 → 이력서 갱신 시 참조
- **리쿠르터 질의**를 이력서 개선 신호로 활용

## 흔한 실수

- 수치 없는 "많이·크게·빠르게"
- 팀 성과를 개인 것처럼 — 꼬리 질문에서 드러남
- JD와 무관한 내용이 페이지 상단 차지
- 기술 스택을 **배지 나열**로만 — 증빙 없음
- 모든 회사에 같은 파일 제출
- 화려한 디자인으로 **가독성 훼손** — 인포그래픽 이력서의 함정
- 3페이지 넘는 "모든 걸 다 썼다" 이력서

## 체크리스트

- [ ] 첫 페이지만 보고도 "누구이고 어디에 강한 사람인지" 15초 내 파악 가능한가
- [ ] 모든 성과에 수치·Before/After가 있는가
- [ ] 대표 프로젝트가 **10분 분량 STAR**로 답변 가능한가
- [ ] 지원 회사별로 **JD 키워드 매칭**을 했는가
- [ ] 모르는 사람에게 **종이 피드백**을 받았는가
- [ ] 분기 단위 업데이트 습관이 있는가
