# PM · PO · PMM · 프로젝트 매니저 · 프로그램 매니저

"PM"이라는 같은 약어 뒤에 **전혀 다른 직무**가 숨어 있다. 프로덕트 매니저·프로젝트 매니저·프로그램 매니저·프로덕트 오너·프로덕트 마케팅 매니저. 조직 규모에 따라 통합되기도 분리되기도 하지만, **책임 범위와 판단 기준**이 다르다. 개발자가 이 구분을 모르면 의사결정 주체를 찾지 못해 일이 밀린다.

## 한 문장 요약

| 역할 | 한 문장 정의 | 비유 |
|---|---|---|
| **Product Manager (PM)** | 제품의 **방향**을 결정하고 로드맵 관리 | 감독 |
| **Product Owner (PO)** | 제품 전략을 **백로그·스프린트**로 실행 | 코치 |
| **Project Manager** | 일정·예산·리스크 관리 | 팀 매니저 |
| **Product Marketing Manager (PMM)** | 제품의 **가치를 시장에 전달** | 스포크스퍼슨 |
| **Program Manager** | 여러 프로젝트·팀을 **통합 조율** | 감독관 |

## 1. Product Manager (PM)

- **책임**: 무엇을 왜 만들지 결정, 시장·경쟁 분석, 비전 정렬, 분기·연간 로드맵
- **판단 기준**: 비즈니스 임팩트 > 기술 취향, 사용자 문제, ROI·시장 점유
- **파트너**: CEO·VP Product·디자인·엔지니어링 리드·마케팅·세일즈
- **오해**: "엔지니어링 관리자" 아님 — 방향 결정이 본질. 코드 이해는 필요하지만 개발자는 아님

## 2. Product Owner (PO)

- **책임**: PM 전략을 실행으로, 백로그·우선순위, 스프린트 플래닝, 스토리·인수 조건
- **판단 기준**: 스프린트 내 가치 최대화, 블로커 제거, 요구사항 명확성
- **파트너**: 개발팀·QA·디자이너·PM
- **PM vs PO**: PM이 "무엇을 왜", PO는 "언제 어떻게". 소규모는 겸직, 스크럼은 명시적 분리

## 3. Project Manager

- **책임**: 일정·예산·리스크, 마일스톤 추적, 리소스 배분, 의존성 관리
- **판단 기준**: 일정 준수, 예산 내 실행, 이해관계자 만족
- **차별점**: 제품 방향 결정권 없음. SI·대형 프로젝트 필수. "예정대로 완성·전달"이 본질

## 4. Product Marketing Manager (PMM)

- **책임**: 가치 제안·포지셔닝·메시징·GTM, 보도자료·세일즈 인에이블먼트
- **판단 기준**: 시장 인식·리드·전환
- **파트너**: PM·마케팅·세일즈·CS·PR
- **차별점**: PM이 "만들 것 결정", PMM은 "만든 것 팔 수 있게 포장". B2B SaaS에서 특히 중요

## 5. Program Manager

- **책임**: 여러 프로젝트·팀 통합 조율, 크로스 조직 의존성, 전사 이니셔티브
- **판단 기준**: 전사 목표, 조직 간 정렬, 장기 이니셔티브 진행
- **차별점**: Project = 하나의 프로젝트, Program = 포트폴리오. 대형 조직 전용, CPO·CTO의 운영 파트너

## 역할 매트릭스

| 축 | PM | PO | Project Mgr | PMM | Program Mgr |
|---|---|---|---|---|---|
| 범위 | 제품 | 스프린트 | 프로젝트 | 제품 시장 | 다수 프로젝트 |
| 시간축 | 분기·연간 | 2주 | 프로젝트 기간 | 출시·캠페인 | 반기·연간 |
| 결정 | 제품 방향 | 백로그 우선순위 | 일정·예산 | 포지셔닝 | 포트폴리오 |
| 성공 지표 | 제품 KPI | 스프린트 속도 | On-time·On-budget | 리드·매출 | 전사 목표 |

## 조직 규모별 통합·분업

### 초기 스타트업 (10명 이하)

- 창업자가 PM + PO + Project Mgr 겸직
- PMM은 없거나 마케터가 겸업
- Program Mgr 부재

### 시리즈 B~C (30~100명)

- PM·PO 분리 시작
- 마케팅 내 PMM 1~2명
- Project Mgr은 대형 이니셔티브 전담

### 중대형 (수백 명 이상)

- PM·PO·PMM·Project Mgr·Program Mgr 모두 분리
- 각 도메인·제품 라인별 다수

## 개발자가 이들과 협업할 때

### PM에게는

- **왜**를 물음. 기능의 배경·사용자 시나리오·KPI
- 기술적 트레이드오프를 **비즈니스 언어**로 전달
- 과도한 추상 요구는 범위 조정 요청

### PO에게는

- 스토리의 **인수 조건**을 함께 다듬음
- 견적(Effort)을 정직하게 제시
- 블로커 즉시 에스컬레이션

### Project Manager에게는

- 일정 리스크를 조기에 공유
- 의존성·외부 블로커를 명확히
- 범위 변경은 **영향 평가 포함**해 제안

### PMM에게는

- 기술 용어를 **사용자 언어**로 번역해 제공
- 데모·스크린샷·성능 수치 지원
- 출시 전 충분한 리드타임 확보

### Program Manager에게는

- 다른 팀과의 **접점**을 명시
- 주간 상태 업데이트를 구조화해 제공

## 개발자의 전환 경로

- 엔지니어 → **PM**: 가장 흔한 경로. 기술·사용자 이해가 강점
- 엔지니어 → **PO**: 스크럼 경험 + 제품 감각 있는 시니어
- 엔지니어 → **Program Manager**: 대형 시스템 운영 경험 + 조율 능력
- 엔지니어 → **PMM**: 드물지만 DevRel·개발자 대상 제품에서 존재

## 흔한 오해

- "모든 PM은 같다" — 프로덕트/프로젝트/프로그램 매니저가 완전히 다른 직무
- "PO는 PM보다 아랫단계" — 역할이 다름. 조직에 따라 같은 레벨
- "PMM은 세일즈" — 마케팅 측 제품 전문가. 세일즈와 구분
- "스크럼 없으면 PO 불필요" — 스프린트 기반 팀은 대부분 PO 존재

## 면접 체크포인트

- 같은 PM 약어의 **3가지 이상 해석**을 구분할 수 있는가
- PM과 PO의 **시간축·범위** 차이
- Project Manager와 Program Manager의 차이(단일 vs 포트폴리오)
- PMM이 개발 제품과 어떻게 연결되는가
- 본인이 전환을 고려한다면 **어느 역할**인가, 근거는?
