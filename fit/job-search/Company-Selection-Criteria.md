# 개발자 회사 선택 기준

연봉·사무실·네임밸류만으로 회사를 고르면 2년 후 "왜 왔지?"라는 질문을 마주한다. 중요한 건 **성장 환경의 질**이며, 그건 **조직마다 다르고 심지어 같은 회사 안에서도 팀마다 다르다**("진리의 팀바팀"). 회사 전체 브랜드보다 **지원 팀의 문화**를 검증하는 것이 필수.

## 평가 축 4가지

### 1. 보상 · 복지

기본이자 협상 가능한 정량 요소.
- **연봉 구성**: 베이스 / 성과급 / 스톡·RSU / 사이닝 보너스
- **복리후생**: 교육비·도서비·컨퍼런스·장비 지원
- **근무 형태**: 원격·하이브리드·출근·유연 근무
- 주의: 총보상(TC) 기준 비교. 베이스만 비교하면 왜곡

### 2. 동료의 질

"훌륭한 동료가 곧 복지"라는 격언이 자주 인용되는 이유. **학습의 70%는 동료로부터**.

- **시니어 비율** — 팀 안에 배울 만한 사람이 있는가
- **채용 기준** — "이 팀이 어떤 사람을 데려오는가"가 미래의 팀 수준
- **이직률** — 6개월·1년 차 이탈률은 조직 상태의 선행 지표
- **면접관 인상** — 같이 일하고 싶은가를 본인이 역으로 평가

### 3. 제품 · 트래픽

기술적 성장의 밀도는 **문제의 난이도**에서 온다.

- **자체 서비스 vs 대행·SI** — 자체 제품은 장기적 도메인 축적
- **트래픽 규모** — 대용량 경험은 시니어 커리어의 레버리지
- **기술 스택의 업계 포지션** — 업계 표준인가, 갈라파고스인가
- **확장성·안정성 요구** — 단순 CRUD인지 분산·고가용인지
- 주의: "대용량 트래픽 경험"이라는 환상의 함정 → [[Large-Scale-Traffic-Experience]]

### 4. 리더십 · 문화

"극단에 치우친 리더보다는 **균형을 잘 잡는 리더**"가 건강한 조직을 만든다.

- **개발-사업 밸런스** — CTO가 사업만 보는지, 기술만 보는지, 둘 다 보는지
- **결정 구조** — 톱다운 vs 합의형. 본인 성향과 매치
- **실패 허용** — 실험·실패가 학습으로 처리되는가, 개인 책임으로 돌아가는가
- **CEO·CTO 배경** — 개발자 출신 여부가 의사결정 품질에 영향

## 면접에서 확인할 구체 질문

### 개발팀 팀장에게

- 팀의 시니어/주니어 비율은?
- 팀 규모는? (6명 전후가 관리·협업에 유리하다는 경험칙)
- 코드 리뷰는 어떻게 진행되나요? 필수인가요, 권장인가요?
- 자동화(CI/CD·테스트·배포)는 어디까지?
- 온보딩 기간에 기대하는 바는?

### 리더(CTO/실장)에게

- 개발 조직의 향후 1~2년 로드맵은?
- 개발-사업의 의사결정 밸런스는?
- 기술 부채 관리 방식은?
- CTO 본인의 개발 베이스는?

### 팀 문화 검증

- 토론 문화가 개방적인가, 연공서열인가?
- 교육·컨퍼런스 지원이 실제로 쓰이는가?
- 일정 관리는 합리적인가(번다운·추정·버퍼)?
- **실패 사례**를 들려달라 — 어떻게 대처했는지

## 성장 환경의 구조적 조건

개발자가 성장하기 좋은 회사는 다음이 **구조적으로** 구비되어 있다.

- **도전적 과제** — 70% 시간이 "해본 적 있는 일"에만 쓰이면 정체
- **빠른 피드백 루프** — 코드 리뷰·모니터링·장애 리뷰로 학습이 축적
- **자율성 + 책임** — 자기 영역을 소유하되 결과로 평가
- **명확한 경력 경로** — 시니어·스태프·프린시펄 같은 트랙이 실존
- **실패를 학습으로** — Blameless Postmortem 문화

## 시그널: 이 회사는 피하라

- JD가 모호하거나 여러 역할을 섞어놓음 (책임 불명확)
- 면접관이 질문에 방어적·감정적
- 평가 기준·승진 기준 설명 회피
- 이직률 질문에 "괜찮다"만 반복 (구체 숫자 없음)
- 야근 빈도가 상시화되어 있다는 뉘앙스
- 주니어 비율이 과반 (시니어 밑천이 없음)
- 연봉만 강조하고 **성장·문화** 질문에 답을 못 함
- 자세한 구조적 판별은 [[Toxic-Org-Detection|독성 조직 판별 프레임]]

## 단계별 접근

| 단계 | 확인 |
|---|---|
| **공고 분석** | JD의 R&R·기술 스택·업무 범위. 너무 광범위하면 경계 |
| **회사 정보** | 제품·트래픽·재무(스타트업이면 런웨이) |
| **1차 인터뷰** | 팀 분위기·면접관 성향·기본 커뮤니케이션 |
| **2차·대면** | 리더 철학·기술 결정 구조·온보딩 |
| **Offer 협상** | TC·성장 약속·근무 조건 |
| **Reference Check** | 가능하면 재직자·퇴직자 1:1 대화 |

## 수용 전 최종 검토

Offer를 받은 후 **24~48시간 냉정 검토**.

- 현 직장 불만이 **새 회사의 장점**과 연결되는가, 아니면 탈출만인가
- 2년 후 이 회사에서 **어떤 스킬**이 생길지 구체적으로 그릴 수 있는가
- 팀장·동료와 **주 1회 이상** 대화하며 성장을 논할 수 있는가
- 실패·이직이 **블랙마크**가 아닌 **학습**으로 간주되는가

## 면접 체크포인트

- 회사 브랜드가 아닌 **지원 팀** 단위로 평가했는가
- 4가지 평가 축(보상·동료·제품·리더십)에 각각 점수를 매겨보았는가
- 면접에서 **역질문 10개 이상**을 준비했는가
- 성장 환경의 구조적 조건 5가지 중 몇 개가 충족되는가
- 피해야 할 시그널이 2개 이상 감지되면 재고려
