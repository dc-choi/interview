# 경력기술서 · 직무 전환

경력기술서(Career Description of Work)는 **경력직 전형에서 가장 중요한 서류** 중 하나. 이력서가 '이 사람은 누구인가'를 요약한다면, 경력기술서는 **'이 사람이 그 직무에서 어떻게 일했는가'** 를 증명한다. 채용 담당자는 수많은 서류 중 **직무 적합성·성과의 구체성**을 기준으로 필터링하며, 글쓰기 방식이 곧 합격률을 결정한다. 이력서 일반론은 [[Resume-Portfolio-Guide]], 15초 5원칙은 [[Resume-15-Second-Rules]] 참조.

## 경력기술서 vs 이력서

| 축 | 이력서 | 경력기술서 |
|---|---|---|
| 목적 | 본인 소개·총체적 요약 | **특정 직무에서의 성과 증명** |
| 길이 | 1~2페이지 | 2~4페이지 (프로젝트·성과 상세) |
| 구조 | 학력·경력·스킬·프로젝트 요약 | **주요 업무 + 프로젝트별 상세** |
| 대상 | 1차 스크리닝(인사 담당) | 실무 담당·팀장 심사 |
| 업데이트 | 분기 | 지원마다 회사별 커스터마이즈 |

경력직 지원은 두 문서가 **한 쌍**이다. 이력서가 첫인상, 경력기술서가 본편.

## 작성 전 준비: JD 역분석

경력기술서를 쓰기 전에 **채용 공고의 JD를 먼저 자세히 분석**.

- "이 회사는 **어떤 직무**를 원하는가?"
- "**어떤 역량**을 가진 지원자를 찾는가?"
- 본인의 경험에서 **JD와 교집합**이 큰 항목이 **셀링 포인트**
- 내가 **보여주고 싶은 것**이 아니라, 회사가 **확인하고 싶은 경험** 중심으로 서술

### JD 역분석 절차

1. JD 전문 복사 → 핵심 키워드·요구 역량 하이라이트
2. 본인 경험 중 해당 키워드에 매칭되는 **프로젝트·성과** 추출
3. 해당 경험의 **수치·역할·기여도**를 회사 관점에서 재구성
4. 셀링 포인트 3~5개를 경력기술서 상단에 배치

## 핵심 원칙

### 1. 객관적 사실 + 수치 중심

모호한 서술은 효과 없음. **Before/After·%·규모**로 표현.

나쁜 예 → 좋은 예:
- "서비스 성장 기여" → "월 활성 사용자 1만 → 3만명 (3배 증대)"
- "마케팅 성과 개선" → "고객군별 맞춤 광고 세분화로 클릭률 30% 개선"
- "시스템 안정화" → "장애율 월 2회 → 분기 1회, MTTR 60분 → 15분"
- "팀 생산성 향상" → "주간 배포 횟수 1회 → 3회, 코드 리뷰 TAT 3일 → 1일"

### 2. 모든 경험 나열 금지

- 수행한 프로젝트 전부 적으면 **핵심이 흐려짐**
- 업무·기술이 반복되는 프로젝트는 **요약해서 묶기**
- 지원 직무와 무관한 경력은 **제외** — 경력기술서가 허전해도 상관없음
- 회사 관심사가 아닌 내용은 "채우기"용으로 안 넣는다

### 3. 간단명료 + 구조화

- 빼곡한 서술은 **서류 통과 자체가 어려움**
- **한 줄로 핵심 키워드 + 성과** 표현
- 업무·프로젝트를 bullet과 헤더로 **구조화**
- 1차 서류 검토는 **인사 담당**이 하는 경우 많음 → 전문 용어 남발 금지
- **일반적 언어**로 전달력 확보

## 경력기술서 기본 양식

### 헤더: 주요 업무

- **보유 기술** — Java/Spring, MySQL, AWS, Docker 등
- **반복되는 핵심 업무 스킬** — 예: "대용량 트랜잭션 처리, API 설계, CI/CD 운영"
- 프로젝트마다 반복되는 업무를 상단에 요약 → 각 프로젝트에서 같은 내용 반복 생략

### 프로젝트별 블록

```
[프로젝트명]
- 기간: 2023.03 ~ 2024.05 / 투입 인원 5명, 기여도 40%
- 배경: 기존 주문 처리 시스템 장애율 월 2회, 피크 시 응답시간 2초 초과
- 역할: 백엔드 리드 / 아키텍처 설계·DB 최적화 주도
- 성과:
    - API 응답시간 P99 2.0s → 350ms (82% 감소)
    - 장애율 월 2회 → 분기 1회
    - 처리 TPS 500 → 3,000 (6배)
- 주요 사용 기술: Java 17, Spring Boot 3, Kafka, MySQL 샤딩, Redis
```

필수 5요소: **기간 / 투입·기여도 / 배경 / 역할 / 성과(수치)**.

### 역량 중심 vs 프로젝트 중심

- **프로젝트 중심**: 프로젝트 단위로 시간 순 나열. 이력과 매칭 쉬움
- **역량 중심**: "대용량 트랜잭션", "팀 리딩" 같은 역량별 묶음. 다양한 프로젝트를 관통하는 강점 강조
- 보통 **프로젝트 중심이 기본**, 시니어급에서 역량 중심 병행

## 직무 전환(Career Pivot) 전략

현 직무가 마음에 들지 않아 다른 직무로 이직하려는 경우. 관련 경험 부족이 약점.

### 공통 원칙: 교집합 찾기

- 과거 업무와 새 직무의 **교집합**을 최대한 드러냄
- 완전한 0에서 시작하는 게 아니라 **기존 강점을 전이 가능한 스킬**로 재해석
- 예: 개발자 → PM 전환 시 "데이터 기반 의사결정·이해관계자 조율" 경험

### 사무직 전환 (인사·재무·회계 등)

- **업계 간 이동 자유로운 편** — 어느 회사에나 필요
- 동종 **규모의 기업**으로 이직이 많음 — 업무 방식 적응이 빠름
- 중소 → 대기업: 중소에서 **여러 업무 겸업**했더라도, 대기업은 **세분화**됨. 본인이 **주력으로 보여줄 역량 하나**에 집중
- 유사 규모·업계 경험을 상단에

### 전문 직무 전환 (엔지니어·개발)

- 업종 전환이 **상대적으로 어려움**
- 과거 경력과 교집합이 없으면 **연봉 유지 어려움**
- 잠재성·학력·성실성으로 보완 가능하지만, **처음부터 경력을 쌓는다는 자세** 필요
- 교집합이 있는 인접 분야부터 점진 이동 (백엔드 → 데이터 엔지니어 → ML 엔지니어)
- 개인 프로젝트·OSS 기여·포트폴리오로 **전이 가능성 증명**

### 전환 시 경력기술서 작성 포인트

- 과거 업무를 **직접 나열하지 말고**, 새 직무 관점에서 **재해석**
- 예: "운영 업무를 통해 배운 시스템적 사고 / 장애 대응 커뮤니케이션 역량"
- 새 직무와 연결되는 **사이드 프로젝트·학습 경로**를 명시
- 커리어 공백을 설명하고, 전환의 **동기·계획**을 자연스럽게

## 버전 관리

- **마스터 경력기술서** 1개 유지 — 모든 프로젝트·성과 포함한 원천 자료
- 지원마다 **fork해서 맞춤 편집** — JD 키워드에 맞춰 순서·강조 조정
- 분기 단위로 마스터 갱신 — 최근 성과를 **기억이 흐려지기 전에** 수치화
- 파일명: `홍길동_백엔드_네이버_2026.pdf` (회사·직무·연도 포함)

## 흔한 실수

- **JD 분석 건너뛰고 일반 이력 나열** — 맞춤화 부재
- 수치 없이 **"많이·크게·빠르게"** — 설득력 0
- **팀 성과를 개인 성과처럼** — 꼬리 질문에서 드러남
- **모든 프로젝트 나열** — 핵심이 흐려지고 분량만 늘어남
- 전문 용어 남발 → 1차 스크리닝(인사팀)에서 이해 불가
- **빼곡한 문단** — 구조화·bullet 없이 서술만
- 직무 전환 시 과거 업무만 강조 → 새 직무 연결 부재
- 경력기술서 한 번 쓰고 방치 — 분기 업데이트 누락으로 최신 성과 누락

## 체크리스트

- [ ] 지원 JD를 열어두고 **역매칭**하며 썼는가
- [ ] 모든 프로젝트에 **기간·기여도·수치 성과**가 있는가
- [ ] 헤더의 **주요 업무 요약**이 JD 키워드와 매칭되는가
- [ ] 각 프로젝트가 **5줄 이내** 핵심으로 구조화되었는가
- [ ] 전문 용어가 **인사 담당자도 이해 가능**한 수준인가
- [ ] 지원 회사별로 **fork해서 맞춤** 편집했는가
- [ ] 팀 성과와 개인 기여도가 **분리 기술**되었는가

## 면접 체크포인트

- 경력기술서와 이력서의 **역할 차이**를 설명할 수 있는가
- JD 역분석으로 셀링 포인트를 도출하는 **구체적 절차**
- 성과를 **수치화한 Before/After** 예시 3가지 이상
- 직무 전환 시 **교집합 전략**의 구체적 적용
- 경력기술서의 **회사별 커스터마이즈** 습관이 있는가
