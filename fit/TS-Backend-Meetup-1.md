## Cryptographic Hash 101

> "그 패스워드 해싱은 틀렸다"

### 해시vs암호화해시
- P-NP 문제: 암호화는 기본적으로 NP 문제를 활용
- 해시를 보고 입력값을 찾기 어려워야 함 (역상 저항성)
- 입력의 해시를 유지하면서 입력을 변경하기 어려워야 함 (제2역상 저항성)
- 같은 해시를 가진 두 입력을 찾기 어려워야 함 (충돌 저항성)
- 같은 입력이라도 prefix가 같으면 안 됨

### 해시DoS
- 해시 충돌을 의도적으로 발생시키는 DDoS 공격
- 알고리즘 특성상 해시 충돌이 일어날 수밖에 없음
- 입력이 같으면 출력이 같으므로, 무수히 많은 충돌을 유발 가능

### 절대쓰면안되는알고리즘
- **MD5, SHA-1**: 절대 사용 금지
- **SHA-256**: 패스워드 해시로 부적합 (비트코인 채굴기가 이 알고리즘을 가장 빨리 품)
- **bcrypt**: 사실 이것도 안전하지 않음

### Salt의올바른이해
- Salt는 레인보우 테이블 방어용이지, 해시를 암호화하는 역할이 아님
- "salt + 복잡한 password = 안전"이라는 생각은 위험
- DB가 침해당해도 암호화되어 있으니 괜찮다는 생각도 위험

### 권장패스워드해시알고리즘
- **argon2**: blake2를 반복한 해시, 메모리를 많이 사용하도록 강제
- **balloon**: SHA-3를 여러 번 반복 (KISA 권고 SHA-256 필요 시 대안)
- **yescrypt**
- 핵심: 하드웨어 발전으로 연산 속도는 빨라졌지만 메모리는 저렴해지지 않음 -> 메모리 사용을 강제

### 모든암호화의원칙
- 안전성은 **키에 의해서만** 보장되어야 함
- 현대 보안에서는 감청당해도 안전하게 통신 가능 -> **HTTPS 필수**
- TLS 1.3 이상 권고, 최소 1.2 사용 필요

### JWT활용팁
- JWT 자체는 토큰 포맷일 뿐
- 실제 고민 포인트: 토큰을 어떻게 보관할 것인가
- OTP에 입력 횟수 제한이 없으면 위험

### 금융권보안현실
- 망분리를 기반으로 한 20세기적(물리적) 보안이 오히려 튼튼
- 암호화를 여러 차례 돌리면 웬만하면 통과

## 로그 적재 비용 개선기

> 300달러에서 2달러로

### 기존문제
- 에러 발생 시 로그 파일 확인 불가 (EC2 CPU/메모리 과다 사용)
- 로그가 일정 기간 지나면 삭제됨
- 빠르게 로그 데이터를 확인할 수 있는 환경 부재

### 중앙집중형로그적재
- 사람이 아닌 **기계가 읽을 수 있는 로그 형식** (Structured Logging)
- 글로벌 트레이스 ID: MSA 환경에서 전체 흐름 추적용
- 트레이스 ID: 개별 요청/응답 식별용

### 아키텍처변천
1. **CloudWatch Logs**: 버튼 한번으로 적재 가능하나 **비용이 너무 비쌈**
2. **FluentBit + Firehose + S3**: 최종 선택
   - FluentBit: 경량성 (저사양 인스턴스에 적합), Firehose와 쉬운 통합
   - 사이드카 패턴: 애플리케이션과 로그 관리 로직 디커플링
   - FluentBit에서 S3 직접 적재 가능하나, S3 호출비용 때문에 중간에 Firehose 유지

### 운영기준
- **로깅 레벨**: 경고(warn)는 적재하지 않음, 인터셉터와 FluentBit에서 필터링
- **마스킹**: 전역적으로 로그 처리, 개인정보 누출 방지, 필요 시 전역 마스킹 적용

### 개선포인트
- 백프레셔 문제: 1초에 1000건까지 처리 가능, 초과 시 Kinesis Data Streams 고려
- FluentBit/Fluentd fail-over 문제도 추가 검토 필요
