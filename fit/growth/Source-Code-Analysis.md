# 소스 코드 분석 · 남의 코드 읽는 법

개발자가 쓰는 시간의 **절반 이상은 남의 코드를 읽는 것**. 레거시 유지보수, 오픈소스 적용, 라이브러리 내부 이해, 인수인계 코드 분석 — 모두 "처음 보는 코드를 빠르게 머리에 넣는 능력"에 달려 있다. 이 스킬은 단순 반복이 아니라 **위에서 아래로·중심에서 바깥으로·기록하며 읽기** 같은 전략적 기법으로 가속된다.

## 왜 필요한가

- **새 프로젝트는 대부분 기존 코드 위에서 진행** — 0에서 만드는 일은 드물다
- 오픈소스 라이브러리의 **문서만으로는 해결 안 되는** 버그·튜닝·확장 상황이 흔함
- 이직·팀 이동 시 빠른 온보딩의 핵심
- 내가 쓰는 도구(Spring·React·DB 드라이버 등)의 **내부 이해**가 곧 실력의 천장

## 준비: 시간 확보 · 침착함

- 바쁘다고 **훑어보기만** 하면 잘못된 멘탈 모델이 굳어져 오히려 독
- "3일 만에 파악"은 거의 없다 — 규모에 따라 1주~수개월
- **녹음·메모 습관** — 구두 인수인계는 기억에 남지 않음
- 이메일·메신저로 **자료를 글로** 남겨달라 요청

## 위에서 아래로: 큰 숲 먼저

파일·함수 단위부터 들어가면 **맥락 없이 지엽말단**만 얻는다. 반드시 상위 구조 먼저.

### 파악 순서

1. **제작 의도** — 왜 만들었는가, 어떤 문제를 해결하는가
2. **프레임워크·구조** — Spring·Django·FastAPI 등 기반 스택
3. **디렉토리 트리** — controllers/services/repositories 같은 상위 분류
4. **주요 엔트리 포인트** — `main()`·`app.py`·`index.ts`·`Application.java`
5. **핵심 파일·함수** — 비즈니스 로직의 중심
6. 그 다음에야 세부 함수·클래스

### 뷰 도구

- IDE 트리 뷰 (IntelliJ·VS Code)
- `tree` 명령 또는 **온라인 트리 시각화** 도구
- **Class Diagram 자동 생성** (PlantUML·Mermaid·IDE 기능)
- 코드베이스 스캐너(Sourcegraph, Sourcetrail)

## 메타 자료 최대 활용

### 문서

- **README·ARCHITECTURE.md** — 있다면 먼저 정독
- **공식 docs** — 프레임워크·라이브러리의 공식 문서
- **CHANGELOG** — 최근 변경 맥락
- **위키·Confluence·Notion** — 사내 정리물

### 코드 외 자료

- **Git 로그** — `git log --oneline` + 특정 파일의 `git log -p`
- **커밋 메시지** — 왜 이렇게 짰는지 단서
- **이슈·PR** — 문제와 해결의 맥락. 토론이 보석
- **주석** — 이상해 보여도 **흘려 쓰지 말 것**. 다 이유가 있다

### 묻기

- 알려진 작성자에게 **이메일·GitHub 이슈**로 질문
- 오픈소스는 Slack·Discord·메일링 그룹에 질문 허용
- 질문 전 **본인 가설** 한 줄 붙이기 → 답변자 부담↓, 답변 품질↑

## 코드 작성자의 눈으로 읽기

같은 문제도 사람마다 **작성 스타일**이 다르다. 원 작성자 관점을 재구성해야 빠르다.

- **스타일** — 함수형 vs OOP, 짧은 함수 vs 긴 함수, 명시적 vs 암묵적
- **목적** — 이 모듈이 만드는 가치 한 줄
- **자료구조** — **변수·타입 정의부터** 철저히. 데이터가 구조를 정한다
- **배경지식** — 사용된 알고리즘·패턴·도메인 용어를 사전 학습

### 자료구조가 먼저다

"프로그램 = 자료구조 + 알고리즘" 중 **자료구조를 먼저** 이해하지 않으면 로직이 떠다닌다.

- `struct`·`interface`·`class` 정의 먼저
- 데이터베이스 스키마·ER 다이어그램
- 주요 타입·DTO·Entity
- 이것만 정리해도 전체 흐름의 50%가 보임

## 실행 환경을 갖춰라

- **빌드·실행 먼저** — 동작하는 걸 보면 이해가 배가 된다
- 로컬 개발 환경·Docker·테스트 DB 세팅
- **디버거 걸고 한 스텝씩** — 코드 읽는 가장 빠른 방법
- 로그 레벨 DEBUG로 내리기
- 테스트 스위트 실행 → 통과하는 걸 확인

## 초기화 코드에서 시작

상위 구조 파악 후 디테일로 들어갈 때는 **초기화·부트스트랩 코드**부터.

- `main()`·`App.init()`·`Application.run()`
- DI 컨테이너 등록·설정 로딩
- 의존성 와이어링이 한 곳에 집중 → **어떤 모듈들이 어떻게 연결되는지** 한눈에
- 여기서부터 호출 체인을 따라 내려가면 자연스러운 순서로 이해

## 디자인 패턴 지식이 가속기

- Factory·Strategy·Adapter·Observer 같은 **표준 패턴**을 알면 이름만 봐도 구조 파악
- Spring·Django·React의 **관용 패턴**도 마찬가지
- 패턴을 모르면 **왜 이런 구조**인지 의문만 쌓임
- GoF 23개 + 프레임워크별 관용구 학습 권장

## 기록하며 읽기 (Analysis-as-Documentation)

코드만 보면 금방 잊는다. **동시에 기록**.

### 기록 방식

- **모듈별 한 줄 요약** — 파일마다 "이 모듈이 뭐 하는지"
- **주요 호출 흐름 다이어그램** — PlantUML·Mermaid
- **의문점 로그** — "왜 이렇게 짰지?"를 별도로 쌓아두기
- **개인 위키** — Obsidian·Notion
- **PR 본문·블로그 글** — 제일 강력. 남이 볼 글을 쓰면 **가장 정확하게** 정리됨

### 효과

- 분석 완료 시점에 **이미 인수인계 문서가 됨**
- 나중에 같은 코드를 다시 볼 때 재학습 비용 급감
- 팀 전체의 지식 자산화

## 좋은 코드 많이 읽기

코드 분석 능력은 **좋은 예시 누적**으로 자란다.

- **사용 중인 라이브러리·플러그인의 소스** — Express·Axios·Lodash 같은 기본
- **인상 깊은 소프트웨어의 코드** — Redis·Kafka·Linux 커널
- **존경하는 사람이 만든 코드** — 특정 커미터의 변경 로그 추적
- **실제 다룰 수 있는 규모**부터 — 되도록 **초기 버전**
- 대형 오픈소스는 **초기 몇 커밋**부터 따라가면 진화 과정이 보임

## 실전 시나리오별 접근

- **오픈소스 라이브러리 문제 추적**: 이슈·PR 검색 → API 진입점 → 호출 체인 → 최신 변경 로그
- **레거시 유지보수**: 문서·주석·테스트 확인 → 핵심 경로만 식별 → 수정 범위 최소화 → 회귀 테스트
- **인수인계 받은 코드**: 녹음·메일로 구두 정보 기록 → 전임자 Q&A 2~3회 → 90일 내 전체 맵 공유 → ARCHITECTURE.md 작성

## 흔한 함정

- **함수 하나부터** 들여다보기 — 맥락 없이는 시간 낭비
- **모든 파일 이해 시도** — 핵심 경로 20%만 제대로 알면 80% 커버
- 주석을 **못 믿겠다고 삭제** — 무지로 인한 파괴
- **빌드·실행 없이** 읽기만 — 이해도 절반 이하
- **기록 없이 머리로만** — 1주일 뒤면 상당수 증발
- 디자인 패턴·관용구 **사전 학습 없이** — "왜 이렇게?"의 벽에 갇힘

## "엉덩이의 힘"

- 소스 코드 분석의 **8할은 끈기**
- 한 줄씩·한 단어씩·API 단위로 내려가며 포기하지 않기
- **3일 걸리던 것도 3년 후엔 30분** — 누적이 답
- 좌절 시 **규모를 작게 쪼개기**

## 체크리스트

- [ ] 제작 의도·프레임워크·트리 구조를 **3문단 이내**로 설명 가능한가
- [ ] 핵심 자료구조 5~10개를 **정의부터** 파악했는가
- [ ] 빌드·실행·테스트가 로컬에서 돌아가는가
- [ ] 초기화 코드를 디버거로 **한 번** 따라가 봤는가
- [ ] 분석 내용을 **개인 위키·문서**에 기록하고 있는가
- [ ] 의문점·미해결 사항을 **별도 로그**로 쌓고 있는가
- [ ] 알려진 버그·이슈·PR을 훑었는가

## 면접·자기 평가 체크포인트

- 최근에 분석한 **오픈소스·레거시**의 구조를 2~3분으로 설명 가능한가
- 코드 분석 시 본인만의 **루틴·도구**가 있는가
- 디자인 패턴·프레임워크 관용구를 **10개 이상** 자연스럽게 인식하는가
- 빌드·디버거 환경을 **빠르게 구축**하는 습관이 있는가
- 분석 결과를 **문서화**해서 팀과 공유한 경험
