# Learning Domain Rules

이 파일은 `fit/growth/learning/`의 학습 계획, 로드맵, 우선순위와 진행 판단에 적용한다.

ORM 관련 학습 우선순위를 다룰 때는 `tech/CLAUDE.md`도 읽는다.

## 학습 원칙

- **아웃풋 우선**: 블로그, 면접 답변과 주일학교 플랫폼 개발에 시간을 쓴다. 강의와 블로그가 충돌하면 아웃풋을 우선한다. 다만 수학 기초 단계에서는 개념 입력과 손계산 반복 자체를 필수 과정으로 인정하고 블로그나 코드 산출물을 억지로 요구하지 않는다.
- **기존 장기 기술 축**: JS, TS와 Node.js 코어 및 블로그, CS와 OOP 및 디자인 패턴과 헥사고날 및 클린 아키텍처, 시스템 디자인, MySQL 순서다. 2026-07-22 이후 개인 학습 일정은 아래 수학 우선순위가 덮어쓴다.
- **단일 주축 원칙**: 체계적인 개인 학습은 한 번에 한 트랙만 진행한다. 공고, 면접 준비와 주일학교 플랫폼 개발은 필요에 따라 수행하되 학습 로드맵 진도로 계산하지 않는다.
- **현업 예외**: GraphQL, gRPC, Kafka, Redis, Terraform과 Kubernetes처럼 현업에 즉시 필요한 지식은 먼저 일이 되게 만드는 범위만 학습한다. 부족한 기반은 현재 장기 트랙을 해치지 않는 시점에 보강한다. AI는 자료 탐색, 설명과 검산에 활용하되 중요한 결론은 소스, 실행 결과나 계산으로 확인한다. 현재 주축과 별개의 커리큘럼, 진도표나 자격증 준비로 확장하지 않고 로드맵 단계 통과로 계산하지 않는다.
- **전체 그림 우선 이해형**: 부분 개념보다 전체 아키텍처가 먼저 보여야 이해가 붙는다. 2026-07-14 OpenSearch 학습에서 확인한 방식대로 전체 구조를 먼저 보고 세부 내용을 학습한 다음 문서를 보지 않고 구조를 재현한다.
- **수학 출발점**: 2026-07-23 기준, 《개발자를 위한 필수 수학》 1장부터 6장과 《컴퓨팅 사고력을 키우는 이산수학 3판》 1장부터 12장을 장기 기반 범위로 끝까지 학습한다. 필수 수학 7장 신경망은 제외하고 8장 경력 조언은 선택 독서로 두며 수학 gate에 포함하지 않는다. 외부 기초 자료를 먼저 완주하지 않고 교재 예제에서 막힌 선행 개념만 보충한다. 6주 완주 가정이나 교재 진도율로 압박하지 않고 손계산, 풀이 설명과 작은 코드 검산으로 확인한다.
- **현재 실행 순서**: `2026-H2-Math-Roadmap.md`의 수학, `2027-Search-Recommendation-Roadmap.md`의 검색 baseline과 품질 평가, 같은 문서의 추천 baseline과 taxonomy 실습, `2027-DevOps-Practical-Roadmap.md`의 DevOps 실전 순서다. 추천 디스커버리의 남은 고급 단계는 DevOps 핵심 4 통과 뒤에 재개하거나 현업에서 필요한 범위만 다룬다. 코딩 테스트는 제외한다.
- **gate 우선 전환**: 현재 트랙의 명시된 gate를 닫기 전에는 다음 개인 학습 로드맵을 열지 않는다. 문서의 연도는 가장 이른 시작 목표이지 마감일이 아니다. 선행 gate가 늦어지면 압축하거나 병행하지 않고 뒤 로드맵을 순서대로 미룬다.

## 시간대별 학습 모드

- 퇴근 후에는 기술 아웃풋이나 수학 개념 이해와 손계산처럼 집중이 필요한 일을 한다.
- 근무 중 자투리 시간에는 짧은 강의, 책 복습, 이미 푼 문제와 용어 복습을 한다. 새로운 수학 개념은 시작하지 않는다.
- 기술 강의는 짧은 챕터, 훑기와 복습 같은 자투리 시간 적합도로 판단한다. 깊은 집중이 필요하면 퇴근 후 아웃풋 트랙으로 옮긴다. 수학에서는 손으로 풀이 과정을 남기는 것을 아웃풋으로 인정한다.

## 학습 상태 판단

- 저장소의 `status: done` 수량을 숙련도 프록시로 사용하지 않는다. CS 64/65나 Node.js 45/53 같은 수치는 문서 상태이지 숙련도가 아니다.
- 사용자가 CS는 밀린다고 자가 진단했다. 정리된 문서가 있어도 면접이나 글에서 스스로 설명하지 못하면 숙련으로 보지 않는다. 숫자만 보고 CS 강의가 필요 없다고 결론 내리지 않는다.
- 학습 우선순위를 제안하기 전에 자가 진단을 묻거나 해당 주제가 말이나 글로 바로 나오는지 확인한다. 문서 수는 맥락 참고로만 사용한다.

## CS 학습 이유

- CS를 배우는 이유는 웹 인프라의 논리적 흐름과 구조 이해, 비용과 성능 효율, 장애와 디버깅, 설계 의사결정의 언어를 확보하기 위해서다. ORM 이상 쿼리, 커넥션 풀 고갈과 GC처럼 추상화가 새는 문제에서는 밑단 지식이 원인 파악의 근거가 된다.
- 코딩은 AI에 위임할 수 있어도 무엇을 왜 만들지에 대한 의사결정과 설계는 개발자의 책임이라는 멘토링 합의를 따른다.
- 학습 깊이는 해당 CS 주제가 실제 설계 의사결정으로 말이나 글에 나오는지를 기준으로 정한다. 인덱스 내부 구조를 외우는 데서 끝내지 않고 특정 쿼리에 그 인덱스를 선택한 이유를 설명할 수 있어야 한다. EventBridge와 SQS 비용 비교, 트래픽 정량 비교 및 Prisma `relationLoadStrategy` 실행계획 cost 검증처럼 실제 판단과 연결한다.
