# CS 지식 학습법 · 키워드 기반 책 탐색

CS 교과서 한 권을 처음부터 끝까지 읽는 방식은 **시간 대비 효과가 낮다**. 현장에서 필요한 건 "이 개념을 지금 이해하고 넘어갈 수 있는 수준의 지식". 답은 **문제→키워드→목차→필요한 장만** 순서로 접근하고, 막히면 **순환 탐색**으로 선행 개념을 찾아 들어가는 방식. 책 폭증을 경계하고 동료·구글링을 함께 쓴다.

## 핵심 명제

- **완독 금지** — 한 권 끝까지 읽으려 하면 타이밍 놓침
- **키워드 기반 접근** — 문제에서 출발해 목차를 검색
- **순환 탐색** — 모르는 단어 등장 시 그 단어로 다시 들어감
- **책 폭증 경계** — 참고 책이 너무 많아지면 **지금은 그만두기**
- **동료·검색 병행** — 책만이 답은 아님

## 학습 루프

### 1. 문제·호기심에서 출발

- 업무에서 부딪힌 개념, 면접 꼬리 질문, 코드 리뷰 피드백
- "왜 이 함수가 이렇게 동작하나", "이 에러 메시지의 근본 원인은?"
- **학습 목적이 명확할수록** 몰입·기억 유지가 강함

### 2. 키워드로 목차 검색

- 책의 **목차 인덱스**에서 해당 키워드 찾기
- 관련 섹션만 **핀포인트 읽기**
- 책 전체 읽기 금지 — 시간 대비 효율 급락

### 3. 모르는 단어 발견 → 순환 탐색

- 현재 섹션에 **앞에서 언급된 듯한 단어**가 있으면 그 섹션으로 이동
- 그 섹션에서 또 모르는 단어가 있으면 또 이동
- 필요한 **선행 개념**까지 거슬러 올라간 뒤, 다시 원래 섹션으로 복귀

### 4. 책 바꾸기 판단

- **현재 책에 해당 키워드가 없거나 너무 부실**하면 다른 책 참고
- 단, **참고 책이 쌓이기 시작**하면 멈춰라 — 주의가 분산되어 아무 것도 못 배움
- "1~2권으로 충분"이 원칙. 3권 이상 펼쳐지면 접근법 재점검

### 5. 동료·구글링 활용

- 동료에게 **핵심 질문**으로 묻기 (본인 가설을 먼저 덧붙여서)
- Google·Stack Overflow로 실전 사례 탐색
- **공식 문서·표준 명세**도 일종의 "책" — 같은 원리로 접근

### 6. 반복

- 이 사이클을 여러 번 돌리면 책이 자연스럽게 **체화**
- 1회 완독보다 **N회 순환**이 장기 기억에 유리

## 왜 완독이 비효율인가

- **낯선 개념이 연속으로** 등장 → 집중력 고갈
- 지금 당장 쓰지 않는 개념은 **일주일 내 잊힘**
- 책 후반으로 갈수록 **앞 개념 누적 전제** → 중간에 포기하면 투자 시간 낭비
- 시간 투자가 커질수록 **경험·실전**을 쌓을 시간이 줄어듦

대조적으로 키워드 접근은:
- 즉시 쓸 개념에 집중 → **즉각적 효용**
- 실전 문제 해결로 **동기 유지**
- 순환 탐색으로 **트리 구조**의 지식 형성

## 책 폭증 경계

### 신호

- 책상에 참고 도서 3권 이상이 펼쳐져 있다
- "이 책 다 읽으면 저 책으로 넘어가야지" 계획이 누적
- 실제로는 진도가 거의 없음

### 대응

- **1~2권으로 제한** — 필요 시 한 권만 기본, 한 권만 보조
- 책 탭보다 **코드·실전** 비중을 늘림
- 답이 안 나오면 책을 덮고 **동료·검색**으로 전환

## Novice 단계의 예외

순환 탐색은 **어느 정도의 기초 체력**을 전제로 한다. 완전 입문자(Novice)라면:

- **학부 교재의 목차를 섹션별로 키워드 나열 가능**할 정도까지 교과서 체화가 선행되어야 함
- 권장 10권: 언어 2~3권 / 자료구조 / 알고리즘 / DB / OS / 네트워크 / OOP / 소프트웨어 공학 / 병렬 컴퓨팅
- 기초 없이 키워드 접근만 시도하면 **계속 거슬러 올라가야** 해서 끝이 없음
- 상세: [[Dreyfus-Skill-Model#Novice · 입문자]]

## 실전 팁

- **핵심 챕터 목차를 종이로** 뽑아 책상에 둠 — 순환 탐색의 지도
- 읽은 섹션은 **1문단 요약**을 메모 — 재탐색 시 빠르게 복귀
- **실제 코드로 재현**해봐야 이해가 완성됨 — 읽기만 하면 휘발
- 면접 대비 목적이면 **질문·답변 형식**으로 변환해 기록
- 일주일 단위로 **본인 지식 트리** 갱신 — 공백·누락 지점 파악

## 조직 관점

- 팀 내 **학습 지도자(Competent)** 가 있으면 키워드 추천·순환 탐색 지름길 제공
- [[Code-Review-Culture|코드 리뷰]]가 자연스러운 키워드 학습 계기
- **내부 위키·스터디**는 순환 탐색의 앵커 역할
- 주니어에게는 책 한 권을 통째로 주기보다 **문제 상황 + 2~3개 키워드**를 제시하는 것이 효과적

## 흔한 함정

- 책을 다 읽어야 **준비가 끝난다**는 강박
- 모든 참고서를 동시에 펼쳐두고 **진도 제로**
- 키워드 접근이라면서 **정답만 긁기** — 순환 탐색으로 이해 깊이를 만들지 않으면 지식이 얕음
- 기초 없이 **최신 기술 키워드**로 점프 → 용어 암기만 쌓이고 본질 이해 누락
- 동료에게 **완성된 답**을 기대 → 답보다 사고 과정을 물어야 남음

## 면접 체크포인트

- CS 교과서를 읽는 **본인의 루틴**을 설명할 수 있는가
- 모르는 개념을 만났을 때 **순환 탐색**이 몸에 있는가
- 완독 vs 키워드 접근의 **시간 대비 효용** 판단
- Novice와 이상의 **접근법 차이**를 아는가
- 참고 도서가 폭증할 때의 **멈춤 신호**를 자각하는가
