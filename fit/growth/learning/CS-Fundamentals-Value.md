# CS 기초의 가치 (코딩이 아니라 원리)

컴퓨터 공학 전공의 진짜 가치는 프로그래밍 실력이 아니라 **컴퓨터 시스템의 원리와 구조적 사고**다. 그리고 이 기초 지식은 새 기술이 쏟아질 때 학습곡선을 완만하게 만드는 레버다. 전공 여부와 무관하게, 기초를 갖춘 개발자가 변화 빠른 시장에서 오래 경쟁력을 유지한다.

## 컴퓨터 공학과는 코딩을 가르치는 곳이 아니다

흔한 오해는 컴퓨터 공학 전공자가 당연히 프로그래밍을 더 잘하리라는 것이다. 실제로 전공 교육이 가르치는 것은 Git, 프레임워크, 상태 관리 패턴(MVC, Flux 등), 개발 방법론이 아니다. 그것은 **시스템의 원리와 구조적 사고**다. 예컨대 OS 과목의 프로세스 스케줄링(FCFS, SJF, RR)은 일반 애플리케이션에서 직접 쓸 일은 드물지만, 런타임의 동시성 모델(예: 이벤트 루프)을 이해하는 밑바탕이 된다.

- 그래서 신입 단계에서 **프로그래밍 실력 자체는 전공과 비전공 간 큰 차이가 없다**(애플리케이션 레이어 기준). 전공자도 졸업 후 코딩, 협업 도구를 따로 배우는 경우가 흔하다.
- 전공자의 진짜 강점은 코딩 속도가 아니라 **CS 기초**다 — 자료구조, 알고리즘, 네트워크, 수학, 암호학 같은 근본 지식의 보유 비중이 높다는 것.

## 신기술은 기반기술 위에 쌓인다

새 기술은 혁신처럼 보여도 대부분 **기존 기반기술 위에서 발전한 것**이다. 밑바닥 기반은 잘 변하지 않는다.

- HTTP/1.1에서 HTTP/2로의 변화는 새로운 무언가가 아니라, 이미 쓰던 TCP 커넥션을 더 효율적으로 쓰게 바꾼 것이다. 기반 키워드는 TCP.
- 컨테이너(Docker)는 무거운 VM 가상화를 대신해, 게스트 OS를 통째로 올리는 대신 공통 레이어를 공유하고 프로세스만 격리한 것이다. 기반 키워드는 가상화, 레이어, 리눅스 namespace, cgroup.

밑바닥의 기반기술을 아는 사람은 새 기술을 **기존 개념의 재조합**으로 인식해 학습곡선을 완만하게 그린다. 기초가 없으면 새 기술이 나올 때마다 사용법만 외우거나 남이 정리한 정보를 암기하게 되고, 비슷한 기술이 또 나오면 **같은 암기 루프를 반복**한다(이 암기 함정은 [[CS-Learning-Method|CS 학습법]], [[Dreyfus-Skill-Model|드레퓌스 모형]]의 Novice 단계에서도 다룬다).

이는 기술 변화를 **무엇을 골라 배울지**의 관점에서 본 [[Tech-Trend-Learning-Strategy|기술 변화와 학습 전략]]과 짝을 이룬다. 그쪽이 변화의 패러다임 전복과 선택 판단(도구 vs 근본)을 다룬다면, 여기서는 그 변화조차 **기반기술 위에 적층**되며 기초가 흡수를 돕는다는 면을 본다.

## 실무에서 기초가 갈리는 순간

기초는 당장의 취업에는 덜 보여도 실무 판단에서 조용히 갈린다. 개념 자체는 각 레퍼런스에 있고, 여기서는 왜 중요한지만 짚는다.

- **자료구조**: 서버 응답 목록을 그대로 배열에 두면 특정 항목 조회가 O(n)이지만, 키 기반 맵(해시)으로 바꾸면 O(1)이다. 배열이 메모리에 연속 할당되는지, 자바스크립트에서 배열이 내부적으로 리스트처럼 동작하는 경우는 언제인지를 모르면 잘못된 선택을 한다 ([[자료구조(DataStructure)|자료구조]], [[Hash-Collision|해시 충돌]]).
- **알고리즘**: 문제 푸는 법을 외우는 게 아니라 **내 코드의 시간복잡도를 계산해 병목을 줄이는 것**이 핵심이다. 삼중 for를 하나라도 줄일 수 있는지 판단하려면 복잡도를 읽을 줄 알아야 한다 ([[Algorithm-Complexity|시간복잡도]]).
- **네트워크**: 웹은 네트워크 위에서 동작한다. CORS 같은 이슈에서 통신 과정과 보안을 모르면, 운영 환경에 와일드카드 Access-Control-Allow-Origin을 열어 보안을 망가뜨리는 식의 사고로 이어진다 ([[CORS|CORS]]).

## 무엇부터 — 자료구조, 알고리즘, 네트워크

기초는 이론 비중이 높아 독학이 지루하지만, 최소한 **자료구조, 알고리즘, 네트워크** 세 가지는 실무에서 알게 모르게 가장 자주 도움이 된다.

- 알고리즘은 **정렬, 탐색**부터 시작한다(버블, 병합, 퀵, 이진 탐색은 직관적). 풀이는 자연스럽게 자료구조와 엮이므로 함께 본다.
- 코딩 테스트 사이트의 극단적 라인 줄이기 풀이는 실무 코드가 아니므로 학습 모델로 삼지 않는다.
- 키워드 기반으로 필요한 부분만 파고드는 학습법은 [[CS-Learning-Method|CS 학습법]].

## 체크포인트

- 새 기술을 만나면 그 **기반기술 키워드**(예: TCP, 가상화)로 환원해 이해하는가
- 내가 짠 코드의 **시간복잡도**를 계산할 수 있는가
- 자료구조 선택(배열 vs 맵)을 **접근 패턴과 메모리**로 설명할 수 있는가
- 기초 없이 사용법만 외우는 **암기 루프**에 빠져 있지 않은가
- 자료구조, 알고리즘, 네트워크 중 약한 축을 알고 보완하는가
