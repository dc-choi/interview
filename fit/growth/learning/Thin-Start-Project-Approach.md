# 얇게 시작하기 · 프로젝트 MVP 접근

개인·토이·사이드 프로젝트에서 가장 흔한 실패 공식은 **"시작 전에 모든 걸 완벽히 설계하려는" 지체**. 해결은 거꾸로 — **돌아가는 최소 기능을 먼저 만들고, 쓰면서 보완**한다. Agile 스프린트의 개인 버전이며, 토이 프로젝트·부트캠프·면접 과제 모두에 적용된다. 단, "계획 없이 아무렇게나"가 아니라 **"계획은 간단히, 시작은 빨리"** 가 핵심.

## 핵심 명제

- **작동하는 최소본 먼저** — 기능 1개만 완결되는 버전
- **main 문부터 쓴다** — 추상 설계 전에 구체부터
- **쉬운 기능부터** — 어려운 기능으로 시작하면 도파민 고갈
- **사용하면서 보완** — 직접 써봐야 진짜 요구사항이 드러남
- **계획이 없다는 뜻은 아니다** — README에 가볍게 윤곽, 그리고 바로 코딩

## 왜 얇게 시작해야 하는가

### 완벽 설계의 함정

- 설계 단계에서 **현실에 없는 요구사항**을 많이 가정
- 실제 코딩 시작 전에 피로 누적 → 실행력 저하
- 설계가 너무 크면 **첫 코드까지의 거리**가 멀어져 포기율 상승
- 초기 설계는 대부분 **틀린다** — 직접 써봐야 제대로 보임

### 얇게 시작의 효과

- **빠른 실패·빠른 학습** — 설계 단계의 가정을 즉시 검증
- **도파민 유지** — 작동하는 코드가 주는 성취감이 동력
- **구체적 대화 가능** — 피드백 받을 수 있는 물체가 있음
- **숨은 요구사항 발견** — 써보면서 진짜 필요가 드러남

## 5단계 얇은 시작

### 1. README 가볍게 작성

- 무엇을 만들지 **1~2문단**
- 핵심 기능 **3~5개 bullet**
- 기술 스택 · 실행 방법
- **40줄을 넘기지 말 것** — 나중에 계속 갱신

### 2. main 진입점 즉시 작성

```
int main() { return 0; }
```

- 일단 **빌드·실행되는 빈 프로그램**부터
- 아키텍처·추상화·폴더 구조 고민은 **나중에**
- "불필요한 고민을 길게 하지 않기"가 핵심

### 3. 돌아가는 최소 기능

- 하드코딩이라도 좋으니 **1개 경로**를 완결
- 입력 → 처리 → 출력이 한 번이라도 완전히 동작하는 버전
- 이 버전은 **나쁜 코드**여도 된다. 일단 돈다는 게 중요

### 4. 쉬운 기능부터 추가

- 어려운 기능은 나중에
- 예: `printf`를 직접 만든다면, 먼저 **매개변수 그대로 출력**하는 버전부터 → 이후 포맷 지정자·플래그 하나씩
- 각 기능이 **작동함을 확인**하고 다음으로

### 5. 직접 써보면서 보완

- 본인이 사용자가 되어 **실제 사용 시나리오** 실행
- 불편·버그·누락이 드러남 → 다음 개선 대상
- 이 루프가 **가장 빠른 학습**

## "바로 main 쓰라"의 정확한 의미

**오해 방지**: 이 말은 "계획 없이 코드부터"가 아니다.

- 기능 단위로 나누려면 **계획이 선행**되어야 함
- 그 계획은 **README에 가볍게**
- "main부터"의 진짜 의미: **생각을 너무 오래 하지 말고, 빨리 구체로**
- 5분 생각 → 5분 코딩 → 다시 5분 생각의 짧은 사이클

## Agile 스프린트와의 연결

개인 프로젝트의 얇은 시작은 **Agile 스프린트의 개인 버전**.

| Agile 원칙 | 개인 프로젝트 적용 |
|---|---|
| 작동하는 소프트웨어 우선 | 돌아가는 최소 기능 먼저 |
| 변화에 대응 | 쓰면서 드러난 요구에 반응 |
| 짧은 주기 | 하루·반나절 단위의 진전 |
| 사용자 피드백 | 본인이 써보면서 발견 |

대조적 접근인 Waterfall(전체 설계 후 구현)은 **요구사항이 고정된 환경**에서만 유효. 개인 프로젝트·학습용 프로젝트에서는 거의 Agile이 맞다.

## 이슈 트래킹: 얇은 시작의 반대급부

얇게 시작해도 **무엇을 남기고 무엇을 할지** 관리는 필요.

- **GitHub Issues / Projects / Jira** 같은 도구로 기록
- 생각난 개선점·버그를 **즉시 이슈로** — 잊히지 않게
- 우선순위 태그로 "지금 할 것" vs "나중에"
- 이슈 수가 폭증해도 괜찮음 — 실행 여부는 우선순위로 필터링
- Todo 문서보다 **이슈 트래커**가 공유·검색·상태 관리 모두 유리

## 실전 패턴

### 토이 프로젝트

- 오늘 안에 **`hello world` → 핵심 기능 1개** 완결
- 한 주 안에 **최소한의 사용 시나리오** 돌아감
- 한 달 안에 **공개 가능한 버전**

### 부트캠프·학교 과제

- 과제 시작일에 바로 **껍데기 코드** 커밋
- 매일 1기능씩 추가, 커밋 기록으로 진전 증명
- 평가 당일까지 **개선 루프** 지속

### 면접용 포트폴리오

- 아키텍처 고민은 **최소 기능 작동 후**
- 실제 사용자 입장에서 **한 바퀴** 돌려봄
- README에 **돌아가는 데모 링크** — 이것만으로도 지원자 80% 상회

### 업무용 MVP

- 기능 범위를 **반으로 줄이고** 출시 후 확장
- Feature Flag로 **On/Off 가능한 구조**
- 사용자 피드백이 **진짜 요구사항**을 알려줌

## 흔한 함정

- **설계 완벽주의** — "완벽한 아키텍처를 정한 다음에"로 6개월 지체
- **기술 스택 선택에만 2주** — 어떤 스택이든 **일단 시작**이 더 중요
- **어려운 기능부터** — 초반 에너지 소진, 도파민 없음
- **직접 써보지 않음** — 기능만 쌓고 사용성 체감 없음
- **README 없이 무작정** — 계획 없음이 아니라, README에 가볍게 윤곽
- **이슈 없이 머릿속에만** — 아이디어 유실, 우선순위 혼란

## 예외적 상황

- **보안·결제·의료**처럼 **One-Way Door**([[One-Way-vs-Two-Way-Door]]) 성격 강한 영역 — 설계 비중 높여야 함
- **팀 규모가 커서 인터페이스 합의 선행** 필요 — API·계약 우선 설계
- 그 외 대부분은 **얇게 시작**이 안전

## 면접 체크포인트

- 토이·사이드 프로젝트 시작 시 **본인의 루틴**이 있는가
- "돌아가는 최소 기능 먼저"의 구체 경험
- 초기 설계 후 **직접 써보고 바꾼** 사례
- 이슈 트래킹(GitHub Issues/Jira) 활용 경험
- Agile 스프린트와의 연결을 본인 언어로 설명 가능한가
