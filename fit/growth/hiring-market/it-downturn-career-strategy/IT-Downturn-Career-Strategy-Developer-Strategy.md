# 개발자 대응 전략

상위 문서: [[IT-Downturn-Career-Strategy|긴축기 IT 커리어 전략]]

> 검토 기준일: 2026-07-15. 시장 평균을 개인의 정답으로 사용하지 않는다. 역할, 경력 구간, 회사 재무, 생활 조건을 함께 비교한다.

## 대응 원칙

### 1. 총보상과 위험을 함께 비교한다

현재 보상을 유지하거나 소폭 올리는 것이 모든 긴축기의 기본값은 아니다. 공개된 보상 자료, 실제 면접 결과, 역할 범위를 근거로 기대 범위를 갱신한다.

- 베이스, 상여, 사이닝, 지분, 복지, 근무 형태를 같은 표에서 비교한다.
- 지분 보상은 유동성, 희석, vesting, 퇴사 조건을 반영한다.
- 회사의 재무 위험이 크면 기대수익뿐 아니라 실직과 재구직 비용도 포함한다.
- 개인 처우 금액은 vault에 저장하지 않고 원본 오퍼에서 확인한다.

안정성은 무조건적인 최우선 가치가 아니라 개인 런웨이와 목표에 따라 가중치가 달라지는 판단 축이다.

### 2. AI 활용 역량을 역할에 맞게 쌓는다

모든 개발자가 ML 이론이나 특정 도구를 같은 깊이로 익혀야 하는 것은 아니다. 제품 개발자는 다음 역량을 우선한다.

- 모델이 잘하는 작업과 실패하는 작업을 구분한다.
- 실제 업무 평가셋과 회귀 테스트를 만든다.
- 프롬프트뿐 아니라 검색, 데이터 품질, 관측 가능성, 비용을 다룬다.
- 개인정보, 저작권, 보안 경계를 설계한다.
- 모델 교체와 장애를 전제로 시스템을 분리한다.

도구 이름보다 평가와 운영 원리를 익히면 제품과 모델이 바뀌어도 재사용하기 쉽다.

### 3. 데이터 회사라는 이름보다 데이터 사용 방식을 본다

데이터를 다루는 회사가 그렇지 않은 회사보다 항상 좋은 선택인 것은 아니다. 보상 차이를 무시하고 데이터 회사를 고르는 고정 규칙도 두지 않는다.

판별 질문:

- 어떤 의사결정을 어떤 지표로 바꾸는가
- 개발자가 데이터에 접근하고 품질을 개선할 권한이 있는가
- 수집한 데이터가 제품 개선과 고객 가치로 돌아오는가
- 데이터 사용 권리와 개인정보 관리가 명확한가
- 내 역할이 파이프라인 유지에 그치는지 제품 판단까지 연결되는가

### 4. CS 기초와 도메인 이해를 결합한다

언어와 프레임워크의 변화 속도가 빨라도 네트워크, 운영체제, 데이터베이스, 동시성 같은 기초는 설계 판단에 재사용된다. 다만 기초를 암기 목록으로 만들지 않고 현재 업무의 장애, 성능, 데이터 문제와 연결한다.

- [[CS-Learning-Method|키워드 기반 CS 학습법]]
- [[Dreyfus-Skill-Model|드레퓌스 모델]]

### 5. 영어를 접근 비용을 낮추는 도구로 본다

영어 능력이 모든 개발자 채용의 필수 조건이라고 단정하지 않는다. 다만 공식 문서, 연구, 글로벌 협업을 직접 다룰수록 번역과 요약의 지연이나 손실을 줄일 수 있다.

- 읽기: 공식 문서와 변경 이력을 원문으로 확인한다.
- 쓰기: 이슈, 설계 결정, 장애 설명을 짧게 남긴다.
- 듣기와 말하기: 실제 협업이나 지원 목표가 있을 때 우선순위를 높인다.

## 상황별 전략

| 상황 | 확인할 증거 | 우선순위 |
|---|---|---|
| 채용 수요 증가 | 공고 수, 면접 전환율, 제안 범위가 함께 상승 | 선택지 비교, 역할 확장, 협상 |
| 수요 둔화 | 공고 감소, 채용 기간 증가, 요구 경력 상향 | 현재 성과 증명, 탐색 기간 확보, 지원 범위 조정 |
| 회사 위험 증가 | 현금흐름 악화, 반복된 투자 지연, 지급 이상 | 권리 확인, 개인 런웨이 점검, Plan B 실행 |
| 특정 분야 회복 | 투자뿐 아니라 고객과 채용이 함께 증가 | 도메인 적합성 검증 후 선별 이동 |

## 준비 항목

### 지금 유지할 것

- 이력서와 포트폴리오를 최근 성과 기준으로 갱신한다.
- 장애 대응, 비용 절감, 매출과 운영 개선을 검증 가능한 사례로 정리한다.
- 핵심 동료와 외부 네트워크를 끊기지 않게 유지한다.
- 개인 생활비와 구직 기간을 기준으로 필요한 런웨이를 계산한다.

### 다음 역할을 위해 늘릴 것

- 인접 기술 하나를 실제 결과물로 검증한다.
- AI 기능은 평가, 비용, 보안까지 포함해 운영해 본다.
- 한 도메인의 고객, 규제, 수익 구조를 설명할 수준까지 파고든다.
- 시니어 IC와 리더 트랙 중 다음 역할에서 검증할 책임을 정한다.

## 피해야 할 단정

- 긴축 직후 신입사원이 항상 먼저 감원된다고 가정하지 않는다.
- 특정 업종이나 회사 규모만으로 안전성을 판단하지 않는다.
- 정해진 기간 안에 무조건 이직해야 한다는 규칙을 두지 않는다.
- 모델 경쟁과 데이터 경쟁을 양자택일로 보지 않는다.
- 번역 도구가 있으니 영어 학습이 불필요하다고 결론 내리지 않는다.

## 자기 점검

- 현재 팀의 재무와 채용 계획을 직접 확인했는가
- 이직 후보의 고객, 현금흐름, 역할 범위를 확인했는가
- 최근 지원 결과로 시장 가정을 갱신했는가
- AI 기능을 평가하고 운영할 수 있는가
- CS 지식을 현재 설계 결정과 연결해 설명할 수 있는가
- 개인 런웨이와 Plan B의 시작 조건을 정했는가
