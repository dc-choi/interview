# '대규모 트래픽 경험'을 어떻게 봐야 하는가

채용공고의 "대규모 트래픽 경험 우대" 문구와 면접 질문 "대규모 트래픽 경험 있나요?"를 어떻게 해석하고 답변할 것인가에 대한 정리. "체계 없는 환경에서의 성장"이라는 인접 주제도 함께 다룬다.

## 핵심 명제

"대규모 트래픽 경험"은 **DAU/TPS 숫자를 가진 적이 있는가**가 아니라, **트래픽이 만들어내는 문제 유형을 다뤄본 적이 있는가**다. 이력서 한 줄로 적힐 만한 화려한 경력이 아니라, **장애 대응·근본 원인 분석·재발 방지·비용/성능 트레이드오프 판단**이 누적된 경험을 가리킨다.

## 채용 측이 실제로 평가하는 것

| 표면 키워드 | 실제 평가 영역 |
|---|---|
| 대규모 트래픽 경험 | 메트릭→로그→분산추적으로 원인을 좁히는 사고 흐름 |
| 장애 대응 경험 | 응급 조치 + 포스트모템 + 재발 방지의 3단 사이클 |
| 인프라 운영 경험 | 문서 없는 레거시를 리버스 엔지니어링하는 끈기 |
| 비용 최적화 경험 | 기술 선택의 트레이드오프 판단 (예: Redshift→ES 같은 결정) |
| 자율적 성장 경험 | 체계가 없는 환경에서 본인이 만든 체계 (런북·포스트모템 템플릿·온콜 로테이션) |

**숫자(DAU·TPS·QPS)는 진입권일 뿐**, 그 위의 사고 깊이가 변별 요소다. 대기업 출신과 같은 트랙으로 숫자 경쟁을 하면 거의 진다. 같은 숫자를 다뤘어도 **사고의 질**로 차별화한다.

## 흔한 오해 3가지

1. **"대규모 트래픽을 안 받는 회사에 있으면 경험을 못 쌓는다"**
   - 트래픽 자체를 만들지 못해도, 같은 자원에서 부하 테스트(k6·Locust)·SLO 설계·메트릭 두 레이어 교차 분석 같은 **사고 훈련**은 가능하다.

2. **"체계가 없어서 못 배웠다"는 이직 사유가 통한다"**
   - 감점 사유다. 체계가 없었다면 **본인이 만든 체계**(런북·ADR·온콜 로테이션·모니터링 대시보드)를 이야기해야 한다. 그게 시니어 시그널.

3. **"개고생 = 성장"의 서사로 포장한다"**
   - "밤새서 해결했다"는 주니어 톤. **"같은 유형이 재발하지 않게 만들었다"**가 시니어 톤이다.

## 자주 다루게 되는 문제 유형

큰 트래픽이 만들어내는 사건은 도메인이 달라도 패턴이 비슷하다. 다음을 한 가지라도 직접 다뤄봤다면 "대규모 트래픽 경험"으로 말할 수 있다.

- 인메모리 캐시 포화 (Redis/Memcached) — BGSAVE 실패, 메모리 단편화, 키 만료 폭주
- DB 연결 풀 고갈 또는 락 경합
- 메시지 큐 백프레셔 부재 → consumer lag → OOM
- 외부 API 응답 지연 → 호출 스레드 누적 → 시스템 행
- 로그 폭증 → 디스크 포화 → 디스크 I/O 병목 (CPU는 멀쩡)
- 페이지 분할/단편화로 인한 DB 성능 저하
- 트래픽 급증에 오토스케일링 반응이 늦어 SLO 위반
- 비용(스토리지·로그·트랜잭션) 폭증

## 면접 답변 탄약

### "대규모 트래픽 경험이 있으신가요?"

> "제가 다룬 서비스 규모는 (DAU/TPS 숫자)였습니다. 다만 저는 트래픽 숫자보다 **트래픽이 터졌을 때 어떻게 대응했는가**가 더 중요하다고 생각합니다. 예를 들어 (구체 장애)에서 (메트릭→로그→분산추적)으로 원인을 좁혀 (해결책)을 적용했고, 재발 방지를 위해 (런북/알림/아키텍처 변경)을 도입했습니다."

### "이전 회사를 떠나려는 이유가 무엇인가요?"

> "이전 회사는 문서·프로세스가 부족한 환경이었고, 그 안에서 (런북·포스트모템·모니터링 대시보드 등)을 제가 직접 만들면서 배운 것이 많습니다. 다만 이제는 **이미 체계가 자리 잡힌 팀에서 더 높은 수준의 문제**(예: 글로벌 트래픽, 특정 도메인의 신뢰성 요구)를 다뤄보고 싶어서 이직을 준비하고 있습니다."

### "가장 고생했던 장애 대응 경험을 말해주세요."

> "(구체 장애)가 (원인)으로 발생했습니다. 처음에는 (잘못된 가설) 때문에 시간을 허비했습니다. (전환 포인트)를 계기로 (진짜 원인)을 찾았고, (응급 조치)로 서비스를 복구한 뒤 (포스트모템)으로 (재발 방지책)을 도입했습니다. 그 뒤로 같은 유형의 장애는 재발하지 않았습니다."

**구조 공식**: `현상 → 잘못된 가설 → 전환점 → 진짜 원인 → 응급 조치 → 재발 방지`. 5단계 모두 있어야 시니어로 들린다.

## 주의 포인트

- **숫자만으로 경쟁하지 말 것** — 사고의 질로 승부
- **"개고생 서사" 금지** — "재발 방지"가 핵심 시그널
- **"체계 없음"을 핑계로 쓰지 말 것** — 그 안에서 만든 체계를 말한다
- **"원인 → 조치 → 재발 방지"** 3단을 모두 말한다 (하나라도 빠지면 주니어)
- 빅테크 사례를 빌리지 말고, **본인 컨텍스트 안의 문제**를 다룬다

## 사례 (참고)

작은 회사·레거시 환경에서 문서 없이 인프라를 떠맡아 다음을 동시에 처리한 사례가 종종 회자된다 — Fleet 같은 컨테이너 오케스트레이션 붕괴, 8년 운영 MongoDB 디스크 포화 + 버전업 경로 차단, Redis Codis 메모리 포화로 인한 BGSAVE 실패와 데이터 유실, Redshift 로그 검색을 ElasticSearch로 이전해 연 3억 절감. 같은 패턴(레거시·문서 부재·동시 다발 장애)이 다른 회사에서도 반복적으로 나타난다.
