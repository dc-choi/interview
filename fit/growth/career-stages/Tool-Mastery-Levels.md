# 개발 도구 숙련도 5단계 (Level 0~4)

Jenkins·AWS·Kubernetes 같은 개발 도구를 "배웠다"는 사실은 단기적 실력이다. 진짜 실력은 **새 도구가 나와도 스스로 익히고, 여러 도구 중 적절한 것을 선택하고, 필요하면 내부를 수정할 수 있는가**에서 드러난다. 이 5단계 모델은 함께 일할 개발자에게 **무엇을 기대할지** 조직이 가늠하는 기준으로 쓰인다.

## 핵심 명제

- **도구 사용법 = 단기 실력**. 학습력·적응력·판단력이 누적되어 **지속 가능한 실력**
- 신입사원이라도 **Level 2**는 기본 기대치
- Level 3 이상은 팀 기술 결정에 참여할 수 있는 수준
- Level 4는 드문 인원 — 한 팀에 1명 있어도 조직 전체의 기술 밀도를 올린다

## 레벨 정의

### Level 0 — 문서 줘도 못 씀

- 이미 쓰고 있는 개발 도구의 **사용법을 알려주거나 가이드 문서를 줘도** 잘 활용하지 못함
- 기초 체력(언어·OS·네트워크) 부족이 원인인 경우가 많음
- 실무 협업자로서는 부적합 수준

### Level 1 — 가이드 안에서만

- 팀이 만든 가이드 문서·튜토리얼이 **정확히 커버하는 범위 안**에서만 사용
- 문서에 없는 상황이 오면 팀원에게 물음
- 독립적 문제 해결은 아직 어려움
- 신입 초반 1~3개월 수준

### Level 2 — 공식 레퍼런스로 자습 + 문서화

- **공식 레퍼런스**를 보고 스스로 사용법 익힘
- 자신이 경험한 사용법을 **팀 내에 문서로 공유**
- 가이드에 없는 문제도 검색·공식 문서로 해결
- **신입사원 기대 수준** — 이 수준이 되지 않으면 팀 업무가 계속 블로킹

### Level 3 — 도구 비교·선택 + 공식 문서 기여

- **여러 도구를 비교 분석**해서 상황에 맞는 것 선택 (예: CI는 Jenkins vs GitHub Actions, 배포는 ArgoCD vs Spinnaker)
- 공식 레퍼런스 문서의 **부족한 부분을 수정해 PR** 가능
- 팀의 도구 스택 결정에 참여
- 시니어·테크 리드의 기본 자질

### Level 4 — 소스 수정·Fork·패치

- 도구의 **소스 코드를 직접 수정**해서 Fork·패치 적용
- 버그·성능 이슈를 직접 고쳐 운영에 반영
- 오픈소스 커뮤니티에 **큰 기여** 가능
- 팀에 1명 있어도 **조직 전체 기술 수준을 한 단계 끌어올린다**
- Expert 레벨 근처

## 왜 이 분류가 유용한가

### 조직 관점

- 채용·평가에서 **기대 수준 명확화** — "Level 2 이상 기대"라는 기준
- 팀 빌드 시 **레벨 분포** 확인. Level 4가 1명도 없으면 심화 문제 해결 역량 부족
- 온보딩 목표 설정 — 3개월 내 Level 1 → Level 2

### 개인 관점

- 본인의 **현재 수준** 자각
- 다음 단계로 가는 **구체적 행동** 파악
- 평가 면담·이직 협상에서 **레벨 기반 커뮤니케이션**

## 레벨업 전략

### Level 1 → 2

- **공식 문서를 읽는 습관** — 블로그·튜토리얼만 의존하지 말기
- 새로운 도구를 만나면 **팀 위키에 튜토리얼 작성**
- 모호한 동작은 **공식 문서에서 정답을 찾는** 루틴
- 목표: **6개월 내**에 Level 2 진입

### Level 2 → 3

- 유사 도구 **2~3개 병행 학습** (예: Jenkins + GitHub Actions + CircleCI)
- 각 도구의 **트레이드오프 문서화** — 블로그·사내 공유 발표
- 공식 문서 PR 1~2건 시도
- 팀 기술 선택 회의에서 **근거 있는 의견** 제시
- 시니어 직전의 자연스러운 발달

### Level 3 → 4

- 사용 중인 도구의 **소스 읽기·빌드**
- 버그를 **직접 고쳐 PR**
- 오픈소스 커뮤니티 활동(이슈 응답·문서·버그 수정)
- 1~2년 꾸준한 누적이 필요
- 모두가 도달해야 하는 수준은 아님 — **한 분야의 깊이**가 필요한 사람에게

## 각 단계의 함정

### Level 1에서 고착

- 가이드가 없는 상황에서 **질문으로만** 해결 → 스스로 찾는 근육이 안 생김
- 대응: 질문 전 **30분 스스로 시도** 규칙

### Level 2에서 고착

- 단일 도구만 깊이 파서 대체재·경쟁 도구를 모름
- 대응: 유사 카테고리 **주기적 비교 학습**

### Level 3에서 고착

- 비교·선택은 하지만 **내부 동작**은 블랙박스
- 대응: 1~2개 도구의 **소스 코드·내부 아키텍처** 파고들기

### Level 4가 모든 정답 아님

- 모든 도구에 대해 Level 4를 목표로 삼으면 시간이 부족
- **한두 개 핵심 도구**에서만 Level 4, 나머지는 Level 2~3이 현실적 목표

## 적용 예시

### 신입 채용 질문

- "최근에 새로 배운 도구가 있다면 **어떤 자료를 통해** 익혔는가?"
- "팀 내에 해당 도구에 대한 **문서를 남긴 적** 있는가?"
- Level 2 이상 신호 확인

### 시니어 채용 질문

- "현재 팀의 도구 선택에 어떤 **비교·분석** 과정을 거쳤는가?"
- "공식 문서 또는 오픈소스에 **기여한 경험**이 있는가?"
- Level 3~4 신호 확인

### 자기 평가

- 최근 6개월간 새 도구를 **몇 개 학습**했고 **어느 레벨까지** 도달했는가?
- 본인 주력 도구에 대해 **소스 수정 가능 수준**까지 가봤는가?

## 드레퓌스 모델과의 관계

[[Dreyfus-Skill-Model|드레퓌스 5단계]]와 부분적으로 매핑된다.

| Tool Mastery | Dreyfus |
|---|---|
| Level 0~1 | Novice |
| Level 2 | Advanced Beginner |
| Level 3 | Competent |
| Level 4 | Proficient / Expert |

도구 숙련은 전체 기술 숙련의 한 부분. 드레퓌스는 전반적 판단·직관까지 포괄하는 더 큰 틀.

## 면접 체크포인트

- 5단계 레벨 정의를 자기 언어로 말할 수 있는가
- 본인의 **현재 레벨**(도구별)을 객관적으로 평가할 수 있는가
- Level 2에서 3으로 가는 **구체적 전략**
- Level 4가 조직에 주는 가치를 설명할 수 있는가
- 공식 문서·오픈소스 기여 경험의 구체적 사례
