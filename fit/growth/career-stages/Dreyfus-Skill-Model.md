# 드레퓌스 기술 습득 모형

Hubert·Stuart Dreyfus 형제가 제안한 **기술 습득의 5단계 모델**. 규칙 기반의 초보자가 직관 기반의 전문가로 진화하는 과정을 서술하며, "경력 연수 = 실력"이 아니라는 근거의 이론적 배경이 된다. 특히 Competent 경계를 기준으로 **자기 주도·타인 코칭** 능력이 갈리며, 실력 판별·성장 전략의 핵심 프레임.

## 5단계 요약

| 단계 | 판단 방식 | 주변 인식 | 가르치기 |
|---|---|---|---|
| **Novice (초심자)** | 규칙을 그대로 따름 | 전체 맥락 파악 불가 | 불가 |
| **Advanced Beginner (고급 입문자)** | 상황에 맞춰 규칙 조합 | 일부 패턴 인식 | 불가 |
| **Competent (중급자)** | 계획을 세우고 문제 해결 | 전체 맥락 일부 파악 | **가능 — 코칭 시작점** |
| **Proficient (숙련자)** | 직관적 + 분석 혼합 | 맥락 전반 파악 | 가능 |
| **Expert (전문가)** | 직관 중심, 무의식적 실행 | 완전한 맥락 파악 | 최고 수준 |

- **Competent부터 "능력있는 개발자"** — 조직·타인에게 실질 기여
- 업계 통계상 **Expert는 약 1%**. Competent 이상도 전체의 10~20% 수준

## 국내 초중고급 등급제의 부조화

- **옛 등급제**: 대졸 + 경력 13년 = 특급. 평생 근무 시 약 40%가 특급이 된다는 계산
- **드레퓌스**: Expert ≤ 1%. 산술적으로 성립 불가
- 이유: 국내 등급제는 **시간 기반**, 드레퓌스는 **실제 역량 기반**. "소처럼 햇수만 채운다고 1등급 안 된다"
- 직급·직위·경력 연수와 독립적으로 **어떻게 개발하는가**를 봐야 한다

## Competent가 "경계"인 이유

Competent부터 다음이 가능해진다.

- **계획 수립** — 문제를 체계적으로 쪼개고 접근 순서 설계
- **교과서의 체화** — 지식의 양이 아니라 **몸에 밴 지혜**
- **코칭** — 타인에게 기술을 전달 가능 (아래 단계는 불가)
- **지뢰 절반 회피** — Advanced Beginner가 모두 밟아야 하는 실수의 절반을 예측으로 피함

Competent 이하는 **교과서를 흉내만** 낸다. 고치라고 해도 책 보면서 고친다. Competent는 **자기 몸이 교과서**라 1시간이면 교과서적 구조로 정리 가능.

## 국내 개발자 실력의 실제 분포 (경험치)

- **Novice 포함 Advanced Beginner 이하**: 80~90%
- **Competent 이상**: 10~20%
- **Proficient 이상**: 소수
- **Expert**: 1% 수준 (드레퓌스 이론과 일치)

"경력 많은데 왜 Competent가 아닌가"는 학습·환경·코칭 부재의 결과. 햇수만 쌓이고 역량은 정체된 경우가 흔함.

## 레벨별 성장 전략

### Novice · 입문자

- **교과서 10권**을 외울 정도로 학습. 학부 교재 목차만 보고 섹션별 키워드 전부 나열 가능한 수준
- 권장 범위: 언어 2~3권 / 자료구조 / 알고리즘 / DB / OS / 네트워크 / OOP / 소프트웨어 공학 / 병렬 컴퓨팅
- 기초 지식 없이는 Competent를 만나도 **조언을 이해할 수 없다** — 기회가 와도 놓친다
- 개발 코칭도 보통 **Novice는 수용 불가** — "차범근 코치를 만나도 기본 드리블·체력이 없으면 배울 수 없다"

### Advanced Beginner · 신입/3~4학년

- **Competent를 찾아 무조건 모방** — 수단·방법 가리지 말 것
- 이해가 안 가도 기술을 따라 해봄. 모방 → 혼남 → 이해의 실마리
- **자기 코드를 공개할 용기** — 코드 리뷰는 발가벗는 기분이지만, 실패를 보여주지 않으면 배움이 없음
- 회사를 고를 때 **기술 역량 있는 사람**이 있는지 판단. 소스 보유량이나 브랜드가 아님
- 같이 일할 기회가 오면 잡기 — "능력있는 개발자를 따라 아래 직원들이 함께 이직하는 이유"

### Competent

- **Proficient·Expert를 찾아 모방**
- 팀에서 기술 리딩 포지션을 받아 팀 생산성 1.5배 이상 끌어올리기
- 혼자 잘하기보다 **팀 버프** 극대화

### Proficient / Expert

- 후진 양성 + 조직의 기술 방향 설정
- 문제의 본질을 드러내 **전체 시스템의 악순환을 끊는 고리**

## 팀 버프: Competent 이상이 팀에 주는 효과

"10배 개발자"의 가치는 본인 생산성만이 아니다.

- **지뢰 회피** — 팀이 삽질에 쓰는 시간을 절반으로
- **학습 가속** — 주변의 Advanced Beginner가 더 빨리 성장
- **코드 품질 상향** — 쓰는 사람의 오류를 사전에 막는 구조
- **결과 정리** — 품을 들여 만든 코드를 정리·체계화해 팀 전체 생산성 상승
- **조직 생산성 1.5~3배** — 개인 + 팀 버프 합산

"슈퍼 개발자 = 빨리 만드는 사람"은 오해. 슈퍼 개발자는 **타인의 실수를 사전에 막는 사람**(영국·실리콘밸리 개발 문화의 공통 인식).

## 모델을 실전에 쓰는 체크리스트

### 자가 진단

- 책 없이 교과서적 구조로 코드를 **1시간 안에** 리팩토링할 수 있는가?
- 팀 후배에게 "왜 이렇게 해야 하는가"를 **패턴·원리**로 설명할 수 있는가?
- 새 기술을 만나면 **기존 개념과 연결**해서 빠르게 흡수하는가?

세 가지 모두 "예"면 Competent 이상.

### 조직 진단

- 팀 안에 Competent 이상이 **적어도 1명**인가?
- 신입이 Advanced Beginner로 성장하는 데 **2년 이내**로 수렴하는가?
- 시니어가 **다른 사람의 실수를 사전에 막는** 활동에 시간 투자하는가?

## 흔한 오해

- **"경력 = 실력"** — 틀림. 실전 훈련·피드백 없이는 햇수만 쌓임
- **"Novice에게 어려운 책을 쥐여주면 빨리 성장"** — 기초 없이는 교본 이해 불가. 교과서 체화가 선행
- **"Advanced Beginner도 다른 사람 코칭 가능"** — 불가. 지식이 있어도 지혜가 없어 오용 가능성
- **"Expert가 가장 빠르다"** — 직관 판단이라 Competent보다 빨리 결정하지만, 모든 상황에서 빠른 건 아님
- **"모든 개발자가 Expert를 목표로"** — 조직에는 Competent 다수 + Proficient 소수 + Expert 극소수가 건강한 구성

## 면접 체크포인트

- 드레퓌스 5단계를 **한 문장씩** 정의 가능한가
- Competent가 경계선인 **이유**(계획·체화·코칭)
- 국내 등급제와 드레퓌스가 어긋나는 **구조적 근거**
- 본인의 **현재 단계**와 근거를 말할 수 있는가
- 팀 구성에서 단계별 **적정 비율**을 감각으로 알고 있는가
- "슈퍼 개발자 = 실수를 사전에 막는 사람" 명제의 의미
