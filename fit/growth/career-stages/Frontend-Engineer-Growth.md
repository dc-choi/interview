# 프론트엔드 엔지니어 성장

HTML, CSS를 다루는 사람에서 **프론트엔드 엔지니어**로 넘어가는 길. 북극성은 "좋은 프론트엔드 개발자"가 아니라 **문제를 해결하는 좋은 개발자**다. 화면을 그리는 사람이 아니라 사용성을 개선하고 팀의 생산성을 높이는 사람을 목표로 한다. 성장의 공통 방법론(작게 만들고, 깊게 개선하고, 함께 피드백)은 [[Thin-Start-Project-Approach|얇게 시작하기]], [[Growth-Principles|성장 원칙]], [[Developer-Growth-Stages|성장 단계]]와 공유하고, 이 문서는 프론트엔드 고유의 학습 축을 다룬다.

## 북극성: 사용자와 가장 가까운 개발자

프론트엔드는 사용자와 물리적으로 가장 가까운 계층이다. 버튼이 늦게 반응하거나, 입력이 버벅이거나, 애니메이션이 끊기면 사용자는 곧바로 불편을 느낀다. 그래서 프론트엔드 엔지니어는 화면 구현자가 아니라 **사용성(UX)을 개선하는 사람**으로 자신을 정의해야 한다. 이 관점이 배울 기술의 우선순위를 정한다.

## JavaScript는 문법이 아니라 프로그래밍 사고

JavaScript를 잘한다는 것은 문법 암기가 아니라 아래를 다룰 줄 안다는 뜻이다.

- **상태 관리** — 화면이 가진 데이터를 어디에 두고 어떻게 바꿀지
- **모듈화** — 코드를 어떤 경계로 나눌지
- **비동기 흐름** — 네트워크, 타이머, 이벤트를 순서와 실패까지 포함해 다룰지
- **렌더링과 이벤트 제어** — 언제 다시 그리고, 이벤트를 어떻게 흘려보낼지

프로토타입, 클로저, 스코프, 실행 컨텍스트는 면접 단골이지만 글로만 외우면 금방 잊힌다. 프로젝트에서 코드를 쓴 뒤 "내가 쓴 이 코드가 이 개념과 어떻게 연결되나"를 되짚어야 오래 남는다. 개념을 코드 경험에 앵커링하는 것이 핵심.

## UX를 목표로 한 기술 선택

프론트엔드 기술은 결국 더 나은 UX를 위한 도구다. 도구를 UX 목적과 묶어서 배운다.

| 기술 | UX 목적 |
|---|---|
| CSS transition, transform | 상태 변화를 부드럽게 인지시키기 |
| DOM 이벤트, 이벤트 버블링 | 사용자 입력을 정확한 지점에서 처리 |
| 디바운스(debounce) | 입력 폭주 억제 (검색창 마지막 입력만 처리) |
| 스로틀링(throttle) | 고빈도 이벤트 처리량 제한 (스크롤, 리사이즈) |
| 애니메이션 제어 | 끊김 없는 피드백, 로딩 체감 단축 |

라이브러리로 빠르게 구현하는 것도 좋지만, 디바운스나 간단한 애니메이션은 **한 번쯤 직접 만들어봐야** 내부 동작과 트레이드오프가 보인다.

## HTML/CSS에서 프론트엔드 엔지니어로

HTML, CSS를 잘하는 것은 큰 강점이지만, 엔지니어로 가려면 JavaScript로 **프로그래밍 사고**를 익혀야 한다. 결정적 이유는 요즘 프론트엔드가 **컴포넌트 기반**이라는 점이다.

- 하나의 컴포넌트 안에 **구조(마크업), 스타일, 상태, 동작**이 함께 응집된다.
- HTML/CSS와 JavaScript를 분리된 업무로만 생각하면 이 방식에 적응하기 어렵다.
- 작은 UI(토글, 탭, 모달)를 직접 만들며 **상태 변화 → 리렌더 → 이벤트**의 흐름을 손으로 익히는 것이 전환의 지름길.

## 프레임워크보다 바닐라, 원리가 먼저

React, Vue, Angular는 강력하지만 내부 원리를 모르면 **단순 사용자**에 머물기 쉽다.

- 프레임워크 전에 **바닐라 JavaScript**로 상태 관리, 렌더링, 모듈화, 이벤트 처리를 직접 경험한다.
- 여러 개를 얕게 아는 것보다 **하나를 깊게** 파며 "왜 이런 구조가 필요한가"를 이해하는 편이 성장에 유리하다.
- 가능하면 **작은 프레임워크를 흉내 내보는** 훈련(가상 DOM 흉내, 상태-구독 렌더링)이 원리 이해를 크게 끌어올린다.

이는 [[Reverse-Engineering-Learning|역설계 학습]], [[Source-Code-Analysis|소스 코드 분석]]과 같은 축이다. 도구를 쓰기 전에 도구가 푸는 문제를 먼저 겪는다.

## 웹 기초와 백엔드 한 스푼

프론트엔드라도 아래 웹 기초가 곧 문제 해결의 밑천이다.

- **HTTP** 요청과 응답, 상태 코드, 쿠키, [[CORS|CORS]], [[XSS|XSS]] — 브라우저 네트워크 탭으로 실제 트래픽을 관찰하며 익힌다.
- **자료구조와 알고리즘** — 취업 시험용이 아니라 문제를 분해하고 효율적으로 푸는 기초 체력. 운동선수의 기초 체력과 같다.
- **백엔드 한 스푼** — Node.js와 Express로 간단한 API 서버를 만들고 프론트에서 요청을 보내보면, 웹이 어떻게 연결되는지 훨씬 선명해진다. 클라이언트와 서버의 경계를 몸으로 안다.

## 성장 방식은 직군 공통

프론트엔드 고유 축 위에 얹히는 성장 방법은 직군을 가리지 않는다. 상세는 각 노트로.

- **작게 만들고 깊게 개선** — 투두, 검색 UI, 게시판처럼 작게 시작해 성능, UX, 구조, 테스트, 배포까지 한 프로젝트를 조금씩 개선 → [[Thin-Start-Project-Approach|얇게 시작하기]]
- **프로젝트는 이력서 장식이 아니라 학습 기록** — 왜 막혔고 어떻게 풀었는지 회고 → [[Growth-Principles|성장 원칙(회고, 실험)]], [[Retrospective-Methods|회고 기법]]
- **함께 자라는 경험** — 여러 명이 GitHub Issues, PR, Project Board로 협업하며 이슈 관리, 역할 분담, 코드 리뷰, 의견 충돌을 겪는다 → [[Interview-Soft-Skills|소프트 스킬]]
- **좋은 개발자는 팀 생산성을 높인다** — 코드 리뷰에서 "틀렸다"가 아니라 "왜 그렇게 생각했는지, 다른 방식의 장단점"을 존중하며 말한다 → [[Great-Developer-Habits|뛰어난 개발자 습관]]
- **취업은 시장 피드백** — 이력서로 결핍을 확인하고, 탈락 시 피드백을 구하고, 포트폴리오는 화면보다 문제 해결 과정을 담는다 → [[구직(JobSearch)|구직]]
- **학습은 생활 습관** — 매일 같은 강도가 아니라 힘들면 줄이고 회복되면 늘려 다시 돌아올 리듬을 만든다(10문제가 실패하면 1문제로) → [[Burnout-Sustainable-Pace|지속 가능한 페이스]]

## 면접 체크포인트

- "JavaScript를 잘한다"를 상태, 모듈, 비동기, 렌더링 제어로 풀어 말할 수 있는가
- 디바운스, 스로틀을 직접 구현해봤고 언제 무엇을 쓰는지 아는가
- 프레임워크의 특정 동작을 바닐라 관점(원리)으로 설명할 수 있는가
- 컴포넌트에 구조, 스타일, 상태, 동작이 응집되는 이유를 이해하는가
- 협업 프로젝트에서 겪은 코드 리뷰, 의견 충돌을 구체적으로 말할 수 있는가
