# 개발자채용인사이트

## 기업이원하는개발자

### 채용시장의현실
- 신입에게도 경력직 수준의 스펙과 우대사항을 요구하는 추세
- 채용 혹한기로 공급은 증가하고 수요는 감소하지만, 채용 눈높이는 낮아지지 않음
- 결과적으로 소수의 경력직 채용 위주로 시장이 움직임

### 신입채용의우대사항으로올라온것들
- CS지식(운영체제, 네트워크, 자료구조)
- 협업도구(Slack, Jira, Git 브랜치 전략)
- 운영경험(배포, 모니터링, 장애 대응)
- 이런 역량은 현장 경험이 필요하지만, 학교에서는 얻기 어려움

### 차별화포인트
- 똑같은 포트폴리오로는 안 됨
- 기획 → 설계 → 구현 → 배포 → 운영까지 전과정을 경험한 프로젝트가 필요
- 많은 트래픽, 보안, 안정성에 대한 고민이 담긴 경험이 중요

## CS기초의중요성

- 기초 지식이 없으면 금방 바닥이 드러남
- 장기적인 연봉에 영향을 줌
- 소통을 위한 최소한의 학습이 반드시 필요
- 모든 CS를 다 알 수는 없지만:
  - **몇몇 핵심 주제**는 심도 있게 이해하고, 코드와 이론을 매핑할 수 있어야 함
  - **나머지 주제**는 적어도 개론 수준의 이해가 필요

## 문제해결역량

- 성장 가능성 = 문제를 잘 정의하고 해결할 수 있는 능력
- 문제 해결의 과정:
  1. 팀 안에서 문제의 관점을 논의
  2. 해결을 위한 기술과 협업 방법을 논의
  3. 필요한 지식을 나열하고, 부족한 부분은 학습
  4. 점진적으로 디버깅, 코드 리뷰, 테스트, 릴리즈, 배포
  5. 회고를 통해 지식 습득과 협업을 점검
- 해결 방안이 현장에서 잘 동작하는지 확인하는 **운영 경험**이 핵심

## 협업에대한인식

### 기존방식의문제
- 철저한 분업으로 다른 팀원이 문제를 어떻게 해결했는지 모름
- 다른 영역에 대한 기술 이해가 없음
- 그걸 알 수 있는 협업 도구나 활동이 부재

### 좋은협업이란
- 지식을 공유하고 다른 사람에게 질문하는 것
- 개선할 점을 받아들이고 실제로 개선하는 것
- 모든 구성원이 개발 전 과정을 함께 경험하는 것
- 기술적/방법적/태도적 측면에서 회고를 하는 것

## 동기유지와성장

- 개발의 재미는 결과를 확인할 때, 더 높은 기술적 목표에 대한 열정이 생길 때 느낌
- 동기를 지속적으로 유지하려면 **성취의 단계를 더 잘게 나누어야** 함
- 깊은 고민을 먼저 해본 뒤 질문해야 의미 있는 답을 얻을 수 있음

## 개발자역량

결과적으로 일이 되게 하는 능력. 일이 되게 만드는 것을 고민하는 것이 전문가적 역량.
예측 가능한 퍼포먼스를 보여주는 것이 프로의 역량. 설득하고 이해시키는 커뮤니케이션 역량.
이것이 모여 성과로 연결된다.

> 경험과 의지가 일정 수준의 숙달을 만들고, 그 숙달이 전문적인 기술을 만들어냄.

### 기술적역량

**1. 프로그래밍 언어와 인프라의 공고한 이해**
- CS, 알고리즘과 코딩 테스트
- 개발 스택 (모바일, 프론트, 백)
- 데이터베이스와 SQL
- 기본적인 서버 인프라
- 개발 스택에 필수적인 프레임워크의 이해

**2. 소프트웨어 엔지니어링**
- 시스템 디자인
- 코드 품질과 테스트
- 디자인 패턴, OOP
- 데이터베이스 설계와 NoSQL, Redis
- ER 다이어그램, 엔티티 디자인
- Lock, 캐싱 전략
- 프레임워크와 에코시스템

### 업무역량
- 업무 분석 능력, 데이터의 흐름 파악, 호출 구조 파악
- 비즈니스 요구사항 분석 (정책 관련 코드 분석, 요구사항 정의)
- 도메인의 이해
- 큰 그림을 그리는 능력과 그 안에서 디테일을 챙기는 꼼꼼함
- 요구사항을 프로그래밍으로 구현하는 능력
- 주어진 미션을 되게 만드는 노력

## 문제해결능력(심화)

종합적인 논리, 창의적 사고, 경험치가 필요한 영역.
해결까지 필요한 전문적 기술 지식 필요. 문서화, 전파, 공유, 재발 방지.

### 취업준비자가 갖춰야 할 것
- CS 기본 지식과 자료구조, 알고리즘을 바탕으로
- 언어 스펙 공부
- RDBMS, ANSI SQL
- 라이브러리 뜯어보기, 프레임워크 뜯어보기
- 프레임워크를 활용한 프로젝트
- 무엇인가 만드는 것을 좋아하는 사람이라면 반드시 결과물로 어필 (덕업일치)

### 이직준비자가 갖춰야 할 것
- 연차에 비례하는 경험과 실력
- 시스템 디자인, 엔티티 설계
- OOP와 대표적인 디자인 패턴
- RDBMS와 NoSQL, Redis
- 서비스를 운영하기 위한 네트워크와 인프라
- 클라우드 서비스의 이해
- 애플리케이션의 장애 트래킹과 문제 해결 방법
- 성능 테스트, 성능 개선 (애플리케이션 튜닝, SQL 튜닝)
- 비동기 아키텍처에 대한 이해
- RabbitMQ, Kafka
- 컨테이너 플랫폼 환경의 이해 (Docker, K8s)
- MSA

## 인터뷰준비점검

### 제대로 질문하고 제대로 설명하기
- 생각보다 많은 초보자들이 스스로 뭘 모르는지 모름
- 잘 모르다 보니 질문 자체도 굉장히 막연함
- 질문이 모호하다 보니 답변이 불가능
- 알고는 있는데 설명을 잘 못하겠다 → 완벽하게 이해하지 못한 경우
- 따라서 **정리하는 습관이 중요**
- 참고: [How to Ask](https://stackoverflow.com/help/how-to-ask)

### 면접 준비 전략
1. 지원처별 이력서 정리
2. 이력서 첨삭
3. 멘토링
4. 모의 면접
5. 이력서를 기준으로 예상 질문 대비
6. 예상 질문 이후 꼬리질문 대비
7. 면접 경험
8. 복기와 회고

## 스터디와프로젝트를통한성장

팀 혹은 개인으로 프로젝트 혹은 꾸준한 스터디가 가장 확실한 성장의 발판.
지속적인 시간을 투자해야 하므로 체력적, 시간적 부담이 크다는 단점이 존재.

### 내적 성장
- 성실함: 끝까지 완주하는 인내심, 꾸준한 노력
- 지식 공유: 공부한 것을 정리하는 노하우

### 외적 성장
- 사람: 오랜 기간 도움을 주고받을 수 있는 동료, 멘토
- 네트워크: 다양한 업계의 다양한 시각과 정보, 커뮤니티

### 스터디 운영 팁
- 미리 회비를 걷어서 진행, 그라운드 룰 작성
- 발표자 결석 시 즉시 배제 (무임승차 방지)
- 온라인 스터디라도 완주 시 회비로 식사 (매몰비용 효과 — 무료 스터디는 결석이 많음)
- 코드를 짜보고 리뷰하는 것이 가장 적절
- 발표 양식 등의 포맷을 공유
- 실무에서 사용하기 어려운 주제는 지양

### 스터디 예시
- 몽고DB 파헤치기 (도서 완독 및 발표 자료 정리)
- 백엔드 개발자를 위한 Go 스터디 (기초 문법과 앱 개발)
- AWS + NestJS 활용 (AWS에 배포해보기)
- Docker와 k8s 파헤치기 (컨테이너 인프라 환경 구축과 배포, 운영)
- Node.js, NestJS 레퍼런스 번역과 코드 튜토리얼 예제 완성하기

### 개인 과제 구현 예시
다중 로그인 제어 / 공공 데이터 API 연동 / 회원 가입 플로우 / 게시물 정렬과 페이징 / 파일 업로드·다운로드 / 미디어 최적화 / 협업 도구 / 음성 명령 제어 / 화상 채팅 동작원리 / CSV·Excel 파싱

### 팀 과제 구현 예시
- 소셜 로그인 (OAuth 2.0, JWT, 세션, 쿠키)
- 미니 WAS, HTTP (Request, Response, Socket)
- 간단 채팅 (WebSocket, STOMP, PUB/SUB)
- 알림시스템 (이벤트처리, 비동기처리, 성능최적화)
- Shorten URL (Hashing, HTTP 리다이렉션, 악성URL, CSRF)
- REST API 호출기 (REST 설계 원칙, HTTP 메서드/헤더, TEST)
- 공공정보 수집/가공 (스크래핑, 데이터 처리/정제, 데이터 인사이트 확장)
- Key/Value Store (데이터 구조, 동시성 제어, 캐싱)

## 주니어에서시니어로

> "주니어를 넘어 시니어로 가는 길"에서 체감해야 하는 역량 축.

### 기술적·업무적 역량
- **조직 리딩** — 업무 관리, 일정 관리, 주니어 성장 지원
- **코드 품질** — 장애의 개선과 방지, 지속 가능한 코드베이스 유지
- **아키텍처 이해와 스케일링** — 시스템이 성장할 때 깨지지 않는 구조
- **비즈니스 도메인 설계** — 요구사항을 데이터/상태 플로우로 전환하는 능력
- **비기능 요구사항(NFR)** — 가용성, 성능, 보안, 관측성, 운영성
- **대용량 트래픽/분산 처리** — 캐시, 메시징, 샤딩, 파티셔닝에 대한 학습
- **기술 부채 관리** — 우선순위 매기기, 점진적 개선, 로드맵에 포함
- **소규모 프로젝트 리드 역량** — 공식 리더가 아니어도 주도할 수 있는 수준

### 마인드셋
- **성장**에 대한 지속적 투자
- **영향력**을 만들어내는 방식 (코드 + 문서 + 지식 공유 + 멘토링)
- **업무와 팀의 개선**에 대한 고민이 습관화되어 있는가
- **리딩 능력** — 합의 형성, 의견 조율, 결정과 책임
- **마일스톤과 실행 계획** — 큰 목표를 실행 가능한 단위로 쪼개는 능력
- **단기 목표 vs 장기 목표**의 균형
- **메타인지** — 내가 뭘 알고 뭘 모르는지를 정직하게 인식 ([[Interview-Soft-Skills#메타인지 — 가장 먼저 훈련해야 할 것|메타인지 훈련]] 참조)

### 방법론
- **채용공고를 통한 트렌드 분석** — 시니어 포지션 JD가 요구하는 역량을 내 로드맵과 비교
- [[Retrospective-Methods|회고 기법]]을 꾸준히 적용해서 성장 속도를 체감 가능한 지표로 만들기
- [[Growth-Principles|성장의 7가지 원칙]] 중 **소셜화** — 혼자 아는 것으로는 시니어가 될 수 없음

> 더 상세한 시니어 역량은 [[시니어역량(SeniorEngineer)|시니어역량]] 문서 참조.

## 개발자학습로드맵

> 채용공고를 통해서 현재 트렌드를 분석하면서 로드맵을 짜는 방법이 가장 실전적이다. 공고에 계속 등장하는 키워드가 곧 시장이 요구하는 로드맵.

### 공식 로드맵 사이트
- **[roadmap.sh](https://roadmap.sh)** — 역할별 로드맵의 표준
  - [JavaScript](https://roadmap.sh/javascript)
  - [TypeScript](https://roadmap.sh/typescript)
  - [Node.js](https://roadmap.sh/nodejs)
  - 백엔드 / 프론트엔드 / 데브옵스 / 시스템 디자인 등 분야별 로드맵 제공

### 커뮤니티 로드맵 (villainscode/tech-interview-for-junior)
- [신입사원 로드맵 2024](https://github.com/villainscode/tech-interview-for-junior/blob/main/image/new_roadmap_2024.png)
- [백엔드 로드맵](https://github.com/villainscode/tech-interview-for-junior/blob/main/image/roadmap.png)

### 로드맵을 활용하는 방법
1. **체크리스트로 쓰지 말 것** — 모든 칸을 채우는 게 목표가 아님
2. **우선순위 선정** — 내 분야/연차/다음 목표에 맞춰 2~3개 핵심 영역만 깊게
3. **채용공고와 매칭** — 지원하려는 회사의 JD와 공고를 놓고 공통 키워드 추출
4. **실전 프로젝트로 검증** — 로드맵의 각 항목을 **내가 만든 프로젝트로 증명**할 수 있어야 의미
5. **분기별 재점검** — 트렌드는 빠르게 변하므로 로드맵 자체를 업데이트

### 주의사항
- 로드맵은 **지도**이지 **정답**이 아님
- 로드맵에 있는 기술을 "써본 경험"보다 **"왜 썼고, 어떤 문제를 해결했는지"**가 훨씬 중요
- 남이 만든 로드맵을 그대로 따라가면 **나만의 차별점이 사라짐** — 반드시 내 경험과 맥락으로 커스터마이즈할 것
