# 뛰어난 개발자의 행동 패턴

경력 많은 엔지니어들을 관찰하며 반복적으로 드러나는 7가지 행동 패턴. 주니어가 흔히 놓치는 지점과 개선 방향 포함. 성장 전반은 [[Developer-Hiring-Insights-Growth]] 참조.

## 1. 기본기에 충실

- 공식 문서 레퍼런스를 철저히 따름 (Stack Overflow 답변 복붙 금지)
- 가능한 **예외 케이스를 사전에 다룸** → 프로덕션 배포 시 장애가 0에 가까움
- 라이브러리 기본값·동작 원리를 확인 후 사용

**주니어의 함정**: "기술적 자부심으로 기본을 경시" → 화려한 패턴·최신 프레임워크만 추구하고 기본 에러 처리·테스트·문서 확인 소홀

**개선 방향**: 거만함을 버리고 공식 문서 정독 + 예외 처리 꼼꼼히 수행

## 2. 주도적 역량 개발

- 업무 공백 시간을 **팀 발전 주제**에 투자 (공용 모듈·도구·문서)
- 학습 결과를 **팀에 전파** — 지식이 개인 자산에 머무르지 않음
- 미래 대비(기술 부채·확장성)에 에너지 집중

**주니어의 함정**: 개인 공부나 기능 테스트로만 시간 소비. 팀 단위 레버리지 효과 부재

**개선 방향**: 남는 시간을 **팀 공동 주제**로 전환. 문서화·테스트 헬퍼·린터 룰 같은 공공재 기여

## 3. 빠른 코드 분석

- 장애 발생 시 **1시간 내 해결책 방향** 도출 — 평소 축적된 패턴 덕
- 오픈소스 코드를 **의도적으로 읽고** 분석
- **상향식**(세부 → 전체)과 **하향식**(전체 → 세부) 분석을 동시 진행
- 실패 가설을 **빠르게 버리고** 다른 가설로 전환

**주니어의 함정**: 자신이 써본 프레임워크·코드 패턴에만 의존. 새 코드베이스 앞에서 무력

**개선 방향**: 주기적으로 다양한 오픈소스 코드 읽기 훈련. 특정 부분을 가져와서 내 프로젝트에 적용해보기

## 4. 논리적 증명 (엔지니어링 사고)

- 가설에 **논리적 빈틈**이 없는지 자가 점검
- 로그·재현·다이어그램으로 주장을 뒷받침
- 시퀀스 다이어그램·상태 다이어그램으로 시각화해 동료와 공유
- 제안 방법의 **다른 영역 영향** 사전 점검

**주니어의 함정**: "이것 같은데"·"아마"라는 직관만으로 여러 가설 제시. 근거 없이 리더를 설득하려 함

**개선 방향**: 근거(로그·재현 스크린샷·코드 스니펫·다이어그램)로 주장 뒷받침. "저는 X라고 생각하는데, 이유는 Y 로그에서 Z를 관측했기 때문" 형태의 주장

> 엔지니어는 **증명**을 해야 한다. 가정·가설만으로는 부족하며 로그·재현·시각화로 입증.

## 5. 책을 실전과 연결

- 기술서를 정독 후 **실제 경험 중인 문제와 연결**
- 예: False Sharing·Cache Line·TCP 혼잡 제어 같은 교과서 개념을 **며칠 뒤 프로젝트에서 즉시 인지·해결**
- 책 속 개념을 "언젠가 쓸 지식"이 아니라 "지금 내 문제의 렌즈"로

**주니어의 함정**: 책을 흘려 읽고 실무와 연결 못함. "이 책 봤는데 도움 안 돼"로 결론

**개선 방향**: 책 읽을 때 **내 현재 문제·코드와 대조**하며 읽기. 읽는 중 떠오른 적용처를 메모해 뒀다가 즉시 실험

## 6. 프로덕트 중심 마인드

- 기술 집착 ≠ 성과. **효율적 솔루션으로 가치 전달** 이 우선
- 여가 시간의 **Dev Toy**(개인 프로젝트)로 새 기술·원리 이해·포트폴리오를 동시 구축
- **문제 해결 범위를 넓게** — 기획·설계·운영까지 포함
- 장애 해결을 "개발을 방해하는 요소"가 아닌 **성장 기회**로

**주니어의 함정**: 기술 완벽주의로 일정 지연. 프로덕트와 코드를 분리된 영역으로 봄

**개선 방향**: "이 기술이 제품에 어떤 가치를 주는가"를 먼저 질문. Dev Toy는 실제 사용 가능한 형태까지 완성

## 7. 깊이 있는 학습

- "**가져다 쓰면서도, 이해하고 쓰기**"
- 프레임워크/패키지를 쓰더라도 원리를 이해해야 이슈 발생 시 **독립 해결** 가능
- 인접 분야 기초 이해 (백엔드가 프런트 일부 이해, 역도 동일) → **더 나은 기술 결정**
- 블랙박스 도구를 줄이고 **내부 모델**을 가질 것

**주니어의 함정**: "쓸 줄만 알면 되지"로 끝남. 해당 도구의 이슈 발생 시 문제 해결 능력 부재

**개선 방향**: 자주 쓰는 라이브러리 1~2개는 **소스 레벨**까지 들여다보기. 장애 시 내부 이해가 결정적

## 종합 — 주니어가 시니어로 가는 공통 지표

위 7가지 패턴의 공통점은 **"현재의 나"에 머무르지 않는 자세**.

- 기본기 충실 → **현재의 완성도**
- 주도적 역량 개발 → **팀 단위 임팩트**
- 빠른 코드 분석 → **학습 속도**
- 논리적 증명 → **의사결정 품질**
- 책을 실전과 연결 → **지식의 활용**
- 프로덕트 중심 → **가치 전달**
- 깊이 있는 학습 → **자립성**

이 7가지를 **매 스프린트 하나씩** 점검해보는 것이 실질적인 성장 루틴.

## 면접 활용

### "어떻게 성장하고 있으신가요?"

> "세 가지 습관을 계속 다듬고 있습니다. 첫째, **기본기와 예외 처리** 를 공식 문서에서 재확인하는 것. 둘째, **오픈소스 코드를 주기적으로 읽고 분석** 하는 것. 셋째, **제 가설을 반드시 로그·재현·다이어그램** 으로 증명한 뒤 제안하는 것. 이 세 가지가 문제 해결 속도와 의사결정 품질에 가장 큰 영향을 줬습니다."

### "뛰어난 동료에게서 무엇을 배우셨나요?"

> "특히 **논리적 증명 습관**이 인상 깊었습니다. 같은 이슈를 봤을 때 주니어였던 저는 '이것 같은데요'로 말했는데, 그 동료는 로그 발췌·재현 시나리오·시퀀스 다이어그램까지 준비해서 근거로 제안했습니다. 그 이후로 저도 주장할 때는 근거를 먼저 준비하는 방식으로 바꿨습니다."
