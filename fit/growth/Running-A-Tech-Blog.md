# 기술 블로그 운영 · 지속 가능한 글쓰기 문화

기술 블로그는 **개인·팀·회사 모두의 성장 레버**. 외부에 보여주는 브랜드이자 내부 지식 축적 도구이자, 개발자 본인의 이력서 강화 수단. 시작은 쉽지만 **지속 운영**이 어렵다. 맞춤법·동기부여·글쓰기 장벽이 가장 흔한 실패 원인이고, 해결은 **프로세스·자동화·템플릿** 3축으로.

## 왜 기술 블로그인가

### 개인 관점

- **이력서보다 강한 증거** — 링크 하나로 실력 노출
- 글을 쓰려면 **이해가 완성**되어야 함 → 학습 품질 상승
- 커리어 네트워크·기회의 **수동적 유입**
- 자기 커리어의 **포트폴리오 역사**

### 팀·회사 관점

- 기술 인재 **채용 브랜딩**
- 내부 지식의 **외부 공유 통로**
- 조직의 **학습 문화** 가시화
- 오픈소스·컨퍼런스 연결

### 독자 관점

- 기술 레퍼런스
- **간접 경험** 습득
- 회사 내부 동료에게도 학습 자료

## 시작하기: 플랫폼 선택

| 플랫폼 | 장점 | 단점 |
|---|---|---|
| **Jekyll + GitHub Pages** | 무료·버전 관리·개발자 친화 | 빌드·커스터마이즈 학습곡선 |
| **Hugo + Netlify/Vercel** | 빠른 빌드·테마 풍부 | 비슷한 장점 |
| **Medium·Velog** | 즉시 쓰기 가능·검색 노출 쉬움 | 플랫폼 종속 |
| **Notion + 도메인** | 작성 편의 | SEO 약함 |
| **워드프레스·Ghost** | 개인 운영 자유도 | 유지보수 |

**팀 블로그는 Jekyll/Hugo + Git 기반**이 권장 — PR로 리뷰·버전 관리·자동화 가능.

## 운영 프로세스 (5단계)

지속성을 위한 표준 프로세스.

### 1. 글감 등록

- 아이디어 단계 → Issue/Board에 등록
- **누구에게 도움이 되는지**·**기대 효과** 한 줄 기재
- 제목 후보 1~2개

### 2. 글감 리뷰

- **예상 독자** 구체화 (주니어·시니어·비개발자)
- **핵심 메시지** 3~5줄
- 쓸 가치·범위 확정
- 여러 주제가 겹치면 **시리즈로** 쪼개기

### 3. 글 작성

- 초안은 **구조화 위주** — 제목·목차·핵심 예시
- 완벽하지 않아도 **일단 끝내기**
- 템플릿 활용 (도입-본문-정리-참고)

### 4. 초안 피드백

- **Git PR로 리뷰** — 문체·저작권·정확성
- 리뷰어 2~3명
- 맞춤법·읽기 흐름·보충 자료

### 5. 보완 + 배포

- 피드백 반영
- SEO·이미지·링크 최종 점검
- 배포 후 공유

## 배포 타이밍

- **화~목 퇴근 1~2시간 전** (오후 4~5시)이 경험적으로 트래픽 좋음
- 월·금은 상대적으로 낮음
- 컨퍼런스·핫이슈와 겹치는 시기는 회피 or 활용

## 성과 측정

### 정량

- **WAU** (Weekly Active Users) 지표
- Google Analytics·Search Console로 유입 경로·키워드 분석
- 월 1회 글 배포 목표(팀 기준)

### 정성

- 개발 커뮤니티 언급 수
- LinkedIn·X·커뮤니티 반응
- 블로그 글이 면접·외부 강연으로 연결되는 빈도
- 사내 구성원의 **자발적 기여 증가**

## 흔한 운영 어려움

### 1. 맞춤법·문체 일관성

**문제**: 리뷰어마다 판단 기준 다름, 수작업 피곤함.

**해결**:
- **GitHub Action으로 맞춤법 검사 ChatOps** — PR에 자동 코멘트
- 팀 내 문체 가이드 1~2 페이지
- 기계적 오류는 자동화, 문체는 사람 리뷰

### 2. 동기부여 부족

**문제**: "일이 바빠서 못 썼다"가 상시.

**해결**:
- **사내 기존 콘텐츠 재가공** — 회고·내부 위키·세미나 자료를 블로그화
- **글감 발굴 전략** — 문제 해결 사례·도입기·장애 회고
- **주간 작가 로테이션** 시도
- 성공 사례를 **팀 전체에 공유**해 인정 경험 만들기

### 3. 글쓰기 진입 장벽

**문제**: 마크다운·이미지·코드 블록 다루기 부담.

**해결**:
- **템플릿 제공** — 자주 쓰는 구조를 fork용으로
- 마크다운 에디터 추천·단축키 정리
- 시니어가 **초안 구조만** 잡아주고 디테일은 저자가

## 글 유형별 접근

| 유형 | 특징 | 적합 대상 |
|---|---|---|
| **문제 해결기** | 실전 장애·성능·디버깅 | 모든 레벨 |
| **도입기** | 새 기술·도구 적용 과정 | 중급 이상 |
| **아키텍처 설명** | 시스템 구조와 결정 | 시니어 |
| **회고** | 프로젝트·이직·성장 | 모든 레벨 |
| **튜토리얼** | 단계별 가이드 | 주니어 |
| **의견·생각** | 업계 트렌드·원칙 | 시니어 |

**문제 해결기**가 가장 안정적으로 독자를 유입시키는 유형.

## 개인 블로그의 경우

팀 블로그 대비 개인 블로그 팁.

- **주 1회**보다 **월 2~3회**가 지속 가능
- **시리즈**로 묶으면 꾸준함 동력
- 포트폴리오 링크와 **연결된 주제** 중심
- SEO 키워드 의식 (커리어·이직 검색 트래픽 큼)
- 너무 완벽한 글 추구 → **완결**이 우선
- **GitHub·LinkedIn**과 교차 링크

## 흔한 실수

- **완벽주의** — 초안 3개월 방치
- 주제가 본인 **레벨과 맞지 않음** — 너무 얕거나 너무 전문적
- **논란 예민 주제**를 감정적으로 — 경력에 마이너스
- 회사 기밀 포함 — 배포 전 반드시 확인
- 플랫폼 선택에 **시간 낭비** — 선택보다 **시작**이 10배 중요
- 댓글·피드백에 과민 반응 — 긍정적 대응이 장기 자산
- **SEO만 의식** — 독자 가치가 우선

## 운영 체크리스트

Git 기반 버전 관리 / 글감→배포 프로세스 정의 / 템플릿·문체 가이드 / 맞춤법·링크 자동 검증 / 월 1회 이상 정기 배포 / 성과 지표(WAU·유입) / 동기부여 루틴

## 개인 활용 관점

전문 영역 1~2개 집중, 3~5년 장기 자산 관점, 글마다 구체 사례·수치, 옛 글 반복 갱신, 포스팅을 이력서·면접·커리어 협상의 증거 자료로.

## 면접 체크포인트

- 본인의 **블로그 글 3편**을 한 문장씩 요약 가능한가
- 글감 → 배포까지의 **개인 루틴**을 설명할 수 있는가
- 가장 인기 있던 글과 **배운 점**
- 블로그가 본인 **커리어·학습**에 기여한 구체 사례
- 팀 블로그 운영 경험이 있다면 **정량·정성 성과**
