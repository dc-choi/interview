# -10배 엔지니어 안티패턴 (20가지)

"10배 엔지니어"의 거울 이미지 — **한 명이 팀의 10명분 생산성을 삭제**할 수 있는 20가지 행동 패턴. 풍자적으로 쓰였지만 실제 조직에서 매일 일어나는 현상들. 본인에게 이 패턴이 있는지 자가 진단용으로, 팀에 있다면 **조직 건강 경고**로 읽는다.

## 본질

- **개인 생산성은 평균의 3배가 천장**
- 반면 팀 생산성을 깎는 데는 **한 명이 10명분을 삭제 가능**
- 좋은 개발자는 **자기 효율**, 나쁜 패턴은 **남의 시간·주의·동기**를 삭제
- 의도적으로 나쁜 사람은 드물다 — 대부분 **자각 없이** 한다

## 요구사항·스코프 파괴 (1~4)

### 1. 빈번한 요구사항 변경으로 팀 작업 무효화

- 며칠 걸린 결과물을 "방향이 바뀌었어"로 폐기
- **엔지니어 10명의 작업물 = 쓰레기**
- 개선: Two-way Door는 빠르게 바꿔도 OK, 하지만 **방향 결정 자체의 품질**을 높여야

### 2. 무의미한 반복 작업 강제

- 프레젠테이션·다이어그램·티켓 관리에 과도한 시간 요구
- 산출물보다 **산출물 문서**가 커짐
- 개선: 진짜 필요한 문서화·의식적 회의만 남기기

### 3. 번아웃 조장

- 감사 없이 책임 전가, 과로 유도
- 팀의 **심리적 안전**·**동기**가 파괴됨
- 개선: 개인 책임과 팀 책임을 구분, 성취 인정

### 4. 기술 논쟁으로 시간 낭비

- 아무도 **결정권을 갖지 못하는** 구조로 토론 지속
- 평행선 토론은 **시간 + 신뢰 + 관계**를 모두 소모
- 개선: [[One-Way-vs-Two-Way-Door|의사결정 권한]] 명확화, 시한 설정

## 커뮤니케이션·도구 오염 (5~8)

### 5. 비효율적 커뮤니케이션

- 긴 메시지·장황한 문서로 **남의 시간을 눈에 띄지 않게** 삭제
- 10명이 5분씩 허비하면 본인의 5분 절약보다 50배 손실
- 개선: **TL;DR 상단**, 한 주제당 메시지 분리

### 6. 클라우드 비용 낭비

- 느린 코드·미사용 인덱스·과도한 메모리로 리소스 낭비
- 클라우드 월 청구서가 말 없이 증가
- 개선: FinOps·리소스 리뷰 관행

### 7. 불필요한 맞춤 도구 개발

- 단일 담당자만 이해 가능한 스크립트 다수
- 문서 없이 사람이 떠나면 **블랙박스** 남음
- 개선: 기존 도구 우선, 커스텀은 문서·테스트 동반

### 8. 빌드·컴파일 시간 증대

- **최소 20초** 이상 재컴파일이면 집중력 분산
- 수천 번의 개발 루프에서 기하급수적 손실
- 개선: 빌드 캐시·증분 빌드·모듈 분할

## 테스트·아키텍처 파괴 (9~11)

### 9. 무의미한 테스트 작성

- 실제 기능 대신 **모의 함수**·무작위성 포함된 테스트
- 테스트 통과가 **신뢰를 주지 못함**
- 개선: 의도 있는 테스트, [[TDD-BDD]] 프레임 기반

### 10. 잘못된 아키텍처 설계

- 시스템 진화를 고려하지 않은 구조
- 매 변경이 **전체 재작성**으로 이어짐
- 개선: [[Layered-Clean-Hexagonal|Clean/Hexagonal]] 기반 경계 설계

### 11. 복잡한 배포 프로세스

- 환경 간 불일치·취약한 빌드 시스템
- 배포 때마다 사람이 며칠 소요
- 개선: IaC·컨테이너화·[[Blue-Green]]·자동화

## 품질·안정성 방치 (12~13)

### 12. 버그·보안 문제 방치

- **심각한 버그를 감지하지만 해결하지 않음**
- 사용자가 떠나는 진짜 이유
- 개선: 정기 품질 리뷰, [[Incident-Detection-Logging|장애 감지]]

### 13. 불완전한 문서화

- **개인 메시지**·미사용 위키로 지식 고립
- 사람이 떠나면 지식도 같이 사라짐
- 개선: [[Running-A-Tech-Blog|기술 블로그 운영]]·사내 위키 체계

## 인재 관리 실패 (14~19)

### 14. 무가치한 프로젝트에 인력 묶기

- 우수 엔지니어의 잠재력 낭비
- 기한까지 **거짓 진행 상황** 보고
- 개선: OKR 정렬, Kill Criteria 명시

### 15. 복잡한 의존성 추가

- 라이브러리마다 개별 학습 강요
- 유지보수 시간 기하급수
- 개선: 필요 최소 의존성, LTS 스택 기준

### 16. 실패 인정 거부

- **매몰비용**에 빠뜨리고 개선 기회 무시
- "여기서 포기하면 지금까지 한 게 아까워" 증후군
- 개선: [[One-Way-vs-Two-Way-Door|Two-Way Door]] 사고, Kill Switch

### 17. 기여도 없는 엔지니어 충원 (0x 수준)

- 유능한 인재의 자리를 평범한 엔지니어가 차지
- **팀 평균이 실제 하한**
- 개선: 채용 기준 엄격, [[Competence-Identification|Competent 판별법]]

### 18. 역할 모델로 -1x 엔지니어 고용

- 재앙을 일으키고 학습에 저항
- **문화 전염** — 후배가 따라함
- 개선: 시니어 채용에 특히 엄격, 문화 적합성 검증

### 19. 문제 엔지니어 보호

- 실수 지적 없이 부실 엔지니어링 방조
- **성과 우수자가 떠나는 직접 원인**
- 개선: Blameless Postmortem ≠ 면책. 반복 실수는 개입

## 디버깅 불가 설계 (20)

### 20. 디버깅 불가능한 코드 작성

- 스파게티 구조·과다한 추상화·순수 함수 회피
- 문제 원인 **규명 자체가 불가능**
- 개선: 단순성 우선, [[Code-Quality-Criteria|코드 품질 기준]], 관측가능성 설계

## 자가 진단 체크리스트

카테고리별로 본인·팀을 점검.

- **스코프**: 요구사항 변경 빈도 / 문서·회의 비중 / 번아웃 신호 / 기술 논쟁 종결
- **커뮤니케이션**: 읽는 사람 기준 / 클라우드 청구서 리뷰 / 커스텀 스크립트 문서화 / 빌드 시간
- **엔지니어링**: 테스트가 버그 잡음 / 아키텍처 진화성 / 배포 무인화 / 알려진 심각 버그 기록
- **문화**: 팀 지식 위키화 / 프로젝트 목표 정렬 / 의존성 리뷰 / 실패 수용
- **인재**: 채용 실력 기준 / 시니어 문화 적합성 / 문제 행동 정면 대응 / 디버깅 가능 코드

5개 이상 "아니오"면 팀 생산성 위험. 10개 이상이면 즉시 개입 필요.

## 팀 레벨 대응

- **1:1에서 객관적 피드백** — 인격이 아닌 행동
- **명시적 기준** — 코드 리뷰·설계 리뷰·운영 룰
- **자동화로 인간 실수 제거** — 린트·CI·정책
- **심각 케이스는 해결 or 격리** — 방치가 최악
- **좋은 패턴을 가시화** — 회고·칭찬·공개 인정

## 개인 레벨 대응

- 20가지 중 본인의 **주요 함정 2~3개** 식별
- 매 분기 **1개씩** 개선 목표 설정
- 동료에게 **피드백 요청** — 본인이 모르는 패턴은 주변이 안다
- 읽기 전용 안티패턴이 아니라 **자신부터 점검**하는 용도로

## 면접 체크포인트

- 본인이 경험한 -10x 행동 패턴의 구체 사례
- 팀에서 안티패턴을 **어떻게 개입**했는가
- 본인의 약점으로 인지한 패턴 1~2가지와 개선 노력
- 채용·조직 문화에서 **방치 vs 개입**의 기준
