## 0. 왜 백엔드 (정체성)

> 윤회·스페이스맵·비단팝팝·씨이랩 후속 면접에서 "이 사람 진짜 백엔드 깊이 추구하는가" 판단 기준.

> "저는 백엔드를 **물리적 사물·데이터·시간을 디지털로 연결하는 일**이라고 봅니다. 트라이포드랩에서 IoT 디바이스가 보내는 재고 데이터를 정합성 있게 받아 발주·알림·리포트로 연결해오면서, 이 연결이 깨지지 않게 만드는 게 사업 자체를 떠받친다는 걸 체감했습니다. 사용자에게 가장 가까운 결과를 만드는 게 프론트라면, **그 결과가 어떤 상황에서도 깨지지 않게 떠받치는 게 백엔드**라 생각하고, 이 책임을 다음 5~10년 깊게 가져가고 싶습니다."

**꼬리**: "프론트 안 해봤어요?" → 기본 학습은 있지만 본인이 가장 깊게 책임지고 싶은 영역이 백엔드. 풀스택 흩어지기보단 깊이 우선. / "데이터·ML로 안 갔나요?" → AI를 안정적으로 서빙하는 인프라가 본인 강점.

## 1. 장단점 (한 흐름으로 묶기)

**장점: 문제를 끝까지 파고들어 구조로 해결한다**.
- Prisma JOIN 성능 저하를 직접 발견 → 공식 문서 파고들어 `relationLoadStrategy` 발견 → 실행계획 비교 검증 → API 90% 개선. **AI·추측 의존 X, 끝까지 검증**
- 슬로우 쿼리를 인덱스 하나로 끝내지 않고 카디널리티 분석 → "데이터 쌓여도 인덱스 1탐색으로 끝나는 구조" 설계 → 99.3% 개선

**단점: 규모 있는 시스템·표준 인프라를 직접 운영해본 경험이 얇다**.
- 트라이포드랩 발주 자동화 시 Kafka 검토했으나 규모·비용 고려해 EventBridge+SQS 결정 — 판단은 했으나 직접 운영 기회는 없었음
- **보완**: 학습한 내용을 '언제·왜 쓰는지' 기준으로 문서화. 메시지 브로커 4종(RabbitMQ·BullMQ·SQS·Kafka) 성능·운영 부담·유스케이스·선택 기준 비교 정리. **이벤트 아키텍처 직접 설계로 전달 보장·멱등성 본질 체화 → 도구 전이는 빠를 것**

**한 흐름**: 구조로 파고드는 성향은 있으나 무대가 중소 규모라 대규모 인프라는 못 다뤄봤음 → 그 파고드는 성향을 지금 인프라 학습에 쓰는 중.

**꼬리**: "Kafka 안 써봤는데 가능?" → EventBridge·SQS로 DLQ·멱등성·status 머신까지 다룸. 메시징의 본질은 같고 도구 문법만 익히면 됨 / "작은 회사만 선호?" → 0→1·풀스택 범위는 충분히 경험, 다음 단계로 규모 있는 환경 (지원 동기와 연결)

## 2. 갈등·소통 (일정·우선순위 조율 패턴)

> [[My-Motivation-Reasons#기획자와 의견 충돌 사례|FIT-Q-Motivation 원본]] + 비누커머스 회고 시드.
> **핵심 패턴**: 갈등의 본질 = 양쪽 우선순위가 다른 곳에 있음. **데이터로 우선순위를 가시화해서 합의** — 누가 맞고 틀리고가 아니라 무엇을 먼저 풀지 정렬.

### 2-A. 환불 정책 — 일정 vs 리스크 (메인 사례, 5/15 검증)

> "기획자가 환불 정책을 다음 주까지 반영해달라고 요청했는데, 시스템상 새 정책이 기존 정산과 충돌해서 환불 금액 오차 위험이 있었습니다. **그 위험을 데이터로 정리해서 공유**하고, '오차 시 고객 보상·역추적 비용'을 함께 보고 **일주일 미루기로 합의**했습니다. 일정 1주 추가됐지만 오픈 후 민원·회계 문제 없이 안정 운영."

### 2-B. 일정 조율 일반 — 우선순위 분해 (2-A 외 갈등 요청 시)

> 환불 사례 외 "다른 갈등" 요청 시. **구체 회사·금액 디테일 없이 패턴만**.

> "일정 조율 갈등은 자주 있었습니다. 흔한 패턴은 **요청한 쪽은 일정이 우선, 받는 쪽(개발)은 품질·리스크가 우선**이라 평행선이 되는 경우. 이 때 본인이 한 건 **요구사항을 필수 vs 보조로 분해해서 시간 추정을 정량으로 공유** — '전체 일정 안에서 핵심부터 단계적으로 릴리스하면 양쪽 우선순위가 만난다'는 합의를 만들었습니다. 일정 vs 품질 trade-off를 말로 다투지 않고 **분해된 작업 단위로 옮겨서** 결정 가능한 형태로 만드는 게 핵심이었습니다."

### 공통 꼬리

- **"상대가 끝까지 우기면?"** → 리스크/옵션 문서화 → 팀 리드/PM·결정권자 공유 → 최종 판단 위임. 개발자는 **리스크 알리는 역할**이지 최종 결정자 아님.
- **"감정적 갈등이었나?"** → 본질은 우선순위 차이지 사람의 문제가 아님. 일정·리스크 데이터를 같은 화면에 올려놓으면 감정 빠지고 결정만 남음.

## 3. 기획자·비IT 소통

> "기획자와는 **기획 의도를 먼저 묻고 이해한 게 맞는지 재확인**합니다. 시스템 아는 입장에서 '이 의도면 이런 방식도 가능' 제안. 기술용어는 상대방 언어로 풀어서. 프론트엔드와는 **API 문서를 최대한 상세하게** — 협업 포인트가 가장 빈번한 지점이라."

**꼬리**: "비개발자에게 기술 설명한 사례?" → 비누커머스 면접에서 인사담당자·경영실장께 스택 설명 (개념 → 예시 → 비유 → 빡집중) / "API 문서 어떻게?" → OpenAPI 스펙 + 예시 요청/응답 + 에러 코드 + 변경 이력

## 4. 5년 후 / 커리어 목표

**5년 후**: 8년차 백엔드 리드 / **Staff Engineer 트랙**. **권한이 아니라 영향력** — 직접 코드 짜면서 팀 기술 방향 제시·멘토링.

**최종 목표**: CTO. 단 직책 자체가 목표라기보다 **기술과 사업의 연결을 한 사람이 책임지는 자리**. 10~15년 후 방향. 기술적 깊이(3~5년) → 리드(5~8년) → 큰 조직(8~10년) 단계적.

**꼬리**:
- "왜 CTO?" → 트라이포드랩에서 백엔드 단독 운영하면서 기술 결정이 사업 단계와 분리될 수 없다는 걸 체감
- "관리자 트랙(EM)은?" → 사람·일정 관리는 필요하면 하지만 본인 강점은 **기술 결정 깊이**. Staff/CTO 트랙 정렬
- "5년 후에도 코드?" → 코드 짜는 사람이 결정해야 결정이 흔들리지 않음. 멈춘 시점이 시야 멈추는 시점

## 9. 번아웃 관리

> "단독 책임 자리가 길어지면서 **암묵지를 명시지로 옮기는** 습관. 결정 이력을 RFC·ADR로 남기고, 외부 활동(하코 발표·카카오테크 캠퍼스 멘토링)으로 본인만 알지 않게 채널 만든 게 페이스 유지 방법."

**신호 캐치**: 코드 짜는 게 재미없음, 작은 일에 짜증, 집중력 저하 → 육체적 피로(휴가·운동) vs 정신적 피로(프로세스 개선·새 도전) 구분.

## 10. AI 도구 활용 깊이 ★ (자주 받음)

> [[Sensitive-Question-Answers]] 6번 시드. 본인 깊이 어필 카드라 거의 확실히 받음.

> "단순 코드 생성을 넘어 **개발 파이프라인 자체로 재설계**해서 씁니다. Claude Code 위에 **계층형 CLAUDE.md로 컨텍스트 분할**, MCP(context7)로 라이브러리 최신 문서, **Subagent 3종**(api-documenter·security-reviewer·performance-analyzer), **PostToolUse Hook**으로 편집 파일만 lint·prettier, **Stop Hook 자가 리뷰**로 작업 종료 시 버그·품질 점검 자동화."

**원칙**: **설계·의사결정은 직접, 반복 작업은 AI와 협업**.

**꼬리**:
- "AI 추천이 틀린 적은?" → EventBridge vs Kafka 선택 시 AI는 일반 비교만, 도메인 특성(발주 월 10만 건) 정량 비교는 직접 — 99.7% 비용 절감
- "Cursor는?" → Claude Code 기반 (Cursor는 IDE 통합 강점, 본인은 터미널 + 하네스 우선)

## 11. 워라밸·강도 ★ (역질문 패턴)

> [[Sensitive-Question-Answers]] 7번 시드. JD에 "강한 몰입·업무 강도" 명시되면 100% 들어옴.

> "솔직히 의미 있는 일이라면 시간 투자 괜찮습니다. 트라이포드 PoC 밤샘·50% 오류율 주말 작업 사례. 다만 궁금한 건 **지속 가능성** — {N}년째 이 문화를 유지한다는 건 단순 강도 아니라 미션 공감·팀워크 같은 요소가 있다는 뜻 같은데, 팀원들은 어떻게 느끼는지 궁금합니다."

**핵심 기교**: "괜찮습니다" = 절박 / "힘들겠다" = 부정 → **둘 다 회피, 역질문으로 마이크 넘김**.

## 12. 프로젝트 실패 / 품질 사고 (시솔지주 사례)

> "시솔지주 환율 Batch 운영 중 **오류율 50%**를 모니터링으로 발견. PMF 단계라 빠른 실험 우선이라 초기엔 수동 테스트로 커버했는데 기능 늘면서 지속 불가. **400건 오류를 치명도·빈도·원인으로 분류해 우선순위 잡고**, 테스트 0→70%로 오류율 1% 미만 개선. **문제를 발견하고 시스템으로 해결하는 게 개발자의 역할**임을 그 경험으로 배웠습니다."

**꼬리**: "지금이라면?" → 발견 후 대응이 아니라 **출시 전 PR 게이트** 처음부터. 트라이포드는 jest+supertest 게이트 정착으로 같은 사고 0건.

## 13. 개발 환경·도구 ★

> "터미널 중심. **Claude Code를 IDE 수준으로 커스터마이징** — 계층형 CLAUDE.md, MCP, Subagent 3종, PostToolUse Hook, Stop Hook 자가 리뷰. dotfile은 zsh + tmux + neovim 통일. 회사·개인 프로젝트 모두 같은 환경에서 즉시 작업 가능."

**자연스러운 흐름**: 13번 → 10번 (AI 도구 깊이 본문) → CARE ID 6개월 도입 4단계 같은 매핑.

## 14. 디버깅·문제 해결 (6단계)

> "**재현 → 원인 가설 → 분리·검증 → 해결 → 영향 범위 점검 → 회고**. 트라이포드 Prisma API 1000ms 사례 — APM 로그로 재현 조건 식별 → EXPLAIN으로 N개 쿼리 가설 검증 → 공식 문서 `relationLoadStrategy` 발견 → 90% 개선 → Grafana로 다른 endpoint 점검 → 회고 기록."

**도구**: APM·DB slow log·EXPLAIN·`git bisect`·Chrome DevTools·로컬 프로파일러

**꼬리**: "바로 안 풀리는 문제?" → 가설 글로 적고 24h 두고 다시 봄, 새벽 결정·해결 시도 X

## 16. 마지막 한마디 ★ (모든 면접 클로징)
