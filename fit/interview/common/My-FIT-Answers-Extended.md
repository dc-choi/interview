# 내 FIT/컬처핏 답변 마스터 — Extended (보조 11개)

> 핵심 12개는 [[My-FIT-Answers|메인]]. 여기는 나오면 답하지만 매 면접 외워가는 우선순위는 아닌 답변.
> **본문은 여기에만**. 원천 가이드는 [[My-FIT-Answers]], [[My-Motivation-Reasons]].

## 5. 별명 / 자아 인식

**별명**: 어릴 때 똥철, 동물철권 → 친근하고 유들유들한 이미지.

**본인 인식**: "T라서 개발자같다" 모먼트 자주 듣음. 처음엔 친근해 보이는데 일 시작하면 어려워하는 사람도 있음.

**대응 (비누커머스 회고 보강 — 행동 변화 사례 1개)**:
> "프론트엔드 동료 중 처음엔 제 코드 리뷰가 직설적이라 어려워하시던 분이 있었음. 그분께는 리뷰 시 **결론 먼저 + 의도 질문 + 대안 제시** 순서로 톤을 바꿨고, 두 달 뒤엔 그분이 먼저 리뷰 자주 부탁하시는 관계가 됨."

## 6. 이상적인 회사

> "회사는 **공동의 목적을 가진 단체**. 개인이 그 목적 달성에 필요한 것들이 있고, 그건 **소통으로 해결**되어야 함. 결정 권한이 명확하고, 결정 근거가 데이터로 공유되는 환경이 이상적."

**꼬리**: "수평적 vs 위계적?" → 수평적 의사결정이 잘 맞음. 단 결정 권한과 책임은 명확해야. 무한 합의는 비효율.

## 7. 회사 비전 매칭 (객관적 판단)

> "결정이 데이터로 정리되어야 소통이 원활하다고 봅니다. 트라이포드랩에서 클라우드 비용이 규모 대비 높게 나오는 걸 발견하고, **지표로 정리해서 인프라 개선안을 제안해 30% 비용 절감**한 게 그 사례."

**다른 비전 (실행력 등)이 매칭이면 카드 1 장단점에서 사례 가져옴.**

## 8. 빠르게 변하는 기술 / 학습 방식

> "기술 자체를 쫓기보다 **'이 단계의 서비스에 필요한가?'**로 판단. 새 도구는 1~2일 프로토타입으로 비용, 레이턴시, 정확도 정량 비교 후 도입. AI 도구는 그 자체가 빠르게 변하는 영역이라, **CLAUDE.md, MCP, Subagent, Hook으로 팀 협업 파이프라인까지 만들어 본 게** 가장 큰 학습."

**꼬리**: "최근 관심?" → Claude MCP와 Agent SDK. 백엔드에서 MCP 서버 구축 시 AI 에이전트가 DB, API, 문서 직접 조회 가능.

## 15. 사이드 프로젝트, 취미

> "**주일학교 출석부**를 Express 5+tRPC 모노레포로 만들어 운영 중입니다. 2026-07-15 기준 **보고서상 모임 레코드 108개, 랜딩 누적 본당 85곳, 누적 학생 3,669명, 출석 약 22,051건, MAO 34곳**입니다. 누적 본당과 모임 레코드는 활성 수치가 아닙니다. Parish→Church→Organization 3단계 위계 테넌트, 명시적 권한 가드, Snapshot 패턴(이력 보존)까지 직접 설계하고 운영했습니다. 작은 트래픽이어도 **멀티테넌트 운영 문제(권한, noisy neighbor, tenant 자동화)를 직접 다뤄온 영역**입니다. 그 외 **하코 3000명 커뮤니티 Prisma 발표와 카카오테크 캠퍼스 멘토링**으로 가르치며 이해를 점검했습니다."

**꼬리**:
- "본업 영향?" → **"퇴근 후 1시간 정도 + 주말에 시간 내서 합니다. 본업 시간은 침해하지 않게 가드 두고 있습니다."**
- "가장 큰 학습?" → 내가 만든 게 다른 사람 실서비스로 쓰일 때 책임감
- "정말 직접 운영?" → 사업자 등록과 유저 인터뷰, 운영 지표 점검까지 직접 수행. 현재는 MAU와 MAO를 단위별로 분리하고, Activation 전환율과 이탈 기준은 계측을 정비한 뒤 다시 산출
- "본격 사업?" → "거의 본격 사업처럼 운영하지만 본업은 별개. 사이드로 0→1 운영 사이클 학습하는 채널"

## 17. 긴급 이슈 동시 발생 — 판단 기준

> [[My-Motivation-Reasons#같이 일하기 힘든 동료 / 긴급 이슈 판단|FIT-Q-Motivation 원본]]

1. **비즈니스 임팩트**: 사용자 직접 영향(서비스 중단 > 사용성 저하) → 빈도(전체 > 특정) → 복구 시간
2. 혼자 해결 vs 팀 도움 필요 빠르게 판단. **혼자 5시간 걸릴 것 같으면 바로 도움 요청**
3. 판단 어려우면 팀 리드에게 바로 물어보기 — "모르면 빠르게 물어보는 것"도 능력

**꼬리**: "3순위가 더 커지면?" → 왜 그렇게 판단했는지 설명 + 개선책 (모니터링 강화 등). **실수 인정 + 개선이 핵심**

## 18. 함께 성장하는 개발 문화 만들기

> "**먼저 공유하는 사람**이 되기: 하코 3000명 발표, 카카오테크 멘토, 사내 트러블슈팅 노트 정리. 코드 리뷰는 **'이게 틀렸다'가 아니라 '왜 이렇게 했는지' 먼저 묻고 '이런 방법도 있다' 제안**. 동기부여는 말이 아니라 행동으로 — 칭찬 먼저, 개선 제안은 부드럽게. 문화는 강요가 아니라 자연스럽게 만들어지는 것."

## 19. 몰입 경험

> "**시스템 전반 효율 개선에 몰입**: API 느림 → Prisma 분석 → 90% 개선 / 비용 증가 → 분석 도구 도입 → 30% 절감. '코드만 잘 짜는 게 아니라 서비스 전반을 보는 게 진짜 개발자'라는 깨달음. 시간 관리: **오전 Deep Work (미팅 차단, 몰입 2~3시간) + 오후 Shallow Work (회의, 리뷰)**."

**인생 최고로 노력한 순간 (라피신)**: 42서울 입과 시험인 라피신 한 달이 떠오릅니다. 주 7일을 거의 그것만 했습니다. 눈 뜨면 클러스터에 가고 돌아오면 자는 생활이었는데, 아예 처음 보는 개념이 많아서 그 시기엔 다른 걸 할 여유가 없었습니다. 강제된 게 아니라 모르는 걸 알아가는 게 재미있어서 스스로 몰아붙인 시간이었고, 이때 기본기를 처음 제대로 깔았습니다. 또 하나는 직전 회사에서 백엔드를 혼자 맡으며 인사이트가 부족하다고 느껴, 외부 발표와 멘토링으로 피드백을 채우며 회사 밖 성장까지 끌고 온 시기입니다.

> 메모: 강도, 헌신 질문이면 이 버전, 효율 개선 질문이면 기존 버전.

## 20. 스스로를 어떻게 평가

> "**시행착오를 통해 방향을 찾아온 사람**. 문제를 발견하면 그냥 넘어가지 못하고 해결하려고 파고드는 스타일."

**꼬리**:
- "후회되는 선택?" → 첫 회사 6개월 퇴사. 단 그 경험이 **'내가 원하는 환경'을 명확히** 해줌
- "해결 못 한 문제?" → 레거시 리팩토링. 비즈니스 우선순위 때문에 완전 해결 못 했지만 **PR마다 조금씩 정리 + 새 코드는 베스트 프랙티스**. "아직 진행 중"

## 21. 동료에게 받은 기억에 남는 피드백

> "**'조금 더 너의 주장을 강하게 해도 좋을 것 같아'** — 프론트엔드 동료. 깨달음: **겸손과 자신감 없음은 다름**. 근거 있으면 당당하게. 이후 변화: '~인 것 같아요' → '입니다. 왜냐하면'. 데이터 기반 + 명확한 주장 연습."

**두 번째 자주 받는 피드백 (실행력 vs 성격 급함)**: 가장 많이 받은 칭찬은 실행력과 리서치 능력입니다. 일을 빠르게 굴리고 필요한 자료를 잘 찾아온다는 평을 받았습니다. 반대로 단점은 성격이 급한 편이라는 것이고, 몸이 먼저 앞선다는 피드백을 받았습니다. 더 나은 해결책이 있을 수 있는데 바로 손이 나가는 경향이라, 지금은 의도적으로 여유를 갖고 먼저 경청한 뒤 스스로 더 고민하는 단계를 한 번 넣는 습관을 들이고 있습니다.

## 22. 같이 일하기 힘든 동료

> "**기술만 잘하는데 협업 안 되는 사람 / 커뮤니케이션은 좋은데 실력이 따라오지 않는 사람 / 책임을 회피하는 사람**. 기술, 커뮤니케이션, 책임감의 균형이 중요."

## 추가: 성과를 다른 사람이 가져간다면

> 상황 파악 먼저 (고의 vs 실수). **일회성이라도 한 번은 직접 확인** — "이 부분 제가 작업한 건데 혹시 오해가?" 가볍게. 한 번 지켜본 뒤 반복되면 명확히 — 감정 X, 사실 기반으로 선 긋기.
