## 2. FIT 답변 (전화용)

### 1분 자기소개 (전화 30초 압축 + AX 매핑)

> 안녕하세요, 4년차 백엔드 개발자 [이름]입니다. 트라이포드랩에서 NestJS 기반 IoT VMI 서비스를 사실상 단독으로 설계하고 운영하면서, 수기 발주를 EventBridge와 SQS 기반 이벤트 아키텍처로 자동화하고, 슬로우 쿼리를 99.3% 줄였고, Claude Code를 단순 코드 생성이 아니라 개발 파이프라인 자체로 재설계해서 씁니다. 에고이즘이 미뇽맨션, 바쏠을 운영하며 마케팅, CS, 물류의 반복 업무를 LLM 기반으로 자동화하는 단계라는 점에서, 제가 풀어온 업무 자동화와 AI 도구 활용 경험이 정확히 맞다고 봤습니다.

### 지원동기 (제출문 위에 쌓기)

**이미 제출한 지원동기의 뼈대**(§7 원문 참조): ① 최근 1년 Claude Code를 개발 파이프라인 자체로 재설계, ② 거기서 **AI를 어디까지 도구로 쓰고 어디서 사람이 판단하는가를 직접 설계하는 일이 가장 임팩트**라는 깨달음, ③ AX Engineer = 사내 파이프라인에 해온 일을 **운영 중인 브랜드들의 업무 전반(마케팅, CS, 물류)으로 확장**하는 자리. **전화에서 이 뼈대를 벗어난 새 카드를 갑자기 꺼내지 말 것.**

전화용 압축(원문 톤 유지):
> 최근 1년 동안 Claude Code를 단순 코드 생성이 아니라 개발 파이프라인 자체로 재설계하면서, AI를 어디까지 도구로 맡기고 어디서 사람이 판단해야 하는가를 직접 설계하는 일이 가장 임팩트가 크다는 걸 체감했습니다. 에고이즘 AX는 그 사고를 사내 파이프라인이 아니라 실제 브랜드들의 마케팅, CS, 물류 업무 전반으로 확장하는 자리라 봐서 지원했습니다.

### 이직 사유 (자기 귀책, 외부 귀책 금지)

> 트라이포드랩에서 백엔드를 단독으로 책임지며 0→1 구축과 운영 자동화를 폭넓게 잡았습니다. 주도성은 충분히 쌓였고, 다음 구간은 그 자동화 역량을 **더 많은 사용자와 더 빠른 비즈니스 사이클에 직접 연결하는 것**이라고 봤습니다. 에고이즘은 자체 브랜드를 흑자로 운영하면서 AI 자동화로 다음 성장을 만드는 단계라 그 조건이 맞습니다.

**금지**: 회사 망함, SI라 비전 없음, 회사 사정 같은 외부 귀책 톤.

## 3. 핵심 어필 카드 (꼬리 대비)

### A. 가장 임팩트 있던 결과 — 발주 자동화 (제출함, 꼬리 깊게 들어옴)

> 제출 원문은 수치 없이 정성적으로 썼다(§7). 전화에선 **원문 톤대로 먼저 말하고**, "더 설명해달라" 들어오면 아래 수치를 꼬리로 얹는다. 원문에 없던 수치를 먼저 쏟지 말 것.

수천 대 IoT 디바이스가 보내는 재고 데이터를 기준으로, 사람이 일일이 확인하고 발주하던 흐름을 **AWS EventBridge, SQS 기반 이벤트 아키텍처로 재설계**. 조직 규모상 Kafka 도입이 부담이라는 제약을 EventBridge, SQS로 풀었고, **재시도, DLQ, 알림까지 실패를 전제로 설계**해 운영 중 누락 없이 동작. 결과적으로 사람이 붙어 있던 반복 운영 업무가 사라지고 작은 팀이 수천 대 규모 트래픽을 감당. 핵심은 코드가 아니라 **어떤 업무를 어떤 단위로 자동화할지 직접 정의하고 문제 정의부터 배포, 운영까지 끌고 간 것**.

- **수치 꼬리**: 수기 발주 1시간 자동화, 수기 재고관리 4시간을 10분으로(95.8% 절감).
- **왜 EventBridge+SQS (vs Kafka/MSK) 꼬리**: MSK 약 574달러/월 vs EventBridge+SQS 0~18달러/월(월 10만 발주 × 5액션 = 50만 SQS, Free Tier 범위). 운영 인력과 트래픽 규모에서 관리 부담과 비용이 유리 — 사업 단계와 기술 결정을 함께 본 사례.

### A'. 같은 자동화 사고의 확장 — Claude Code 파이프라인 + 실DB 사업지표 분석 (제출함, AX 정조준)

제출 원문이 발주 자동화를 "최근엔 같은 자동화 사고를 한 단계 확장"했다고 마무리한 그 부분. **AX와 가장 직결되는 카드**라 또렷하게:

- **Claude Code를 개발 파이프라인으로 재설계**: 도메인별 CLAUDE.md로 컨텍스트 분할, MCP로 최신 문서 실시간 연결, Subagent와 PostToolUse Hook, Stop Hook 자가 리뷰로 **코드 품질 회귀를 자동 통제**.
- **실DB 기반 사업지표 분석까지 AI 워크플로우로 자동화**: 운영 DB의 지표를 사람이 매번 들여다보지 않게 AI 워크플로우로 묶음 = 반복 업무를 구조로 푸는 같은 사고를 데이터 분석으로 확장.
- **관통하는 원칙**: AI를 어디까지 도구로 맡기고 어디서 사람이 판단할지 직접 설계 — 이게 발주 자동화부터 지금까지 일관된 축이고, 에고이즘 AX가 마케팅, CS, 물류에 하려는 일과 같은 사고.

### B. AI 도구 활용 깊이 (JD 정조준, 거의 확실히 받음)

> 단순 코드 생성을 넘어 개발 파이프라인 자체로 재설계해서 씁니다. Claude Code 위에 계층형 CLAUDE.md로 컨텍스트를 분할하고, MCP로 라이브러리 최신 문서를 주입하고, Subagent 3종(api-documenter, security-reviewer, performance-analyzer), PostToolUse Hook으로 편집 파일만 lint와 prettier, Stop Hook 자가 리뷰로 작업 종료 시 버그와 품질 점검까지 자동화합니다. 원칙은 설계와 의사결정은 직접, 반복 작업은 AI와 협업입니다.

- 꼬리 "AI가 틀린 적은?" → EventBridge vs Kafka에서 AI는 일반 비교만, 도메인 정량 비교(발주 월 10만 건)는 직접.
- 꼬리 "Cursor는?" → Claude Code 기반. Cursor는 IDE 통합이 강점, 본인은 터미널과 하네스 우선.

### C. E2E 주도 + 비개발 직군 소통

- E2E: 트라이포드랩에서 문제 정의부터 설계, 구현, 배포(ECS Fargate 전환), 관측(Grafana, Prometheus, Loki)까지 단독 사이클.
- 소통: 기획 의도를 먼저 묻고 이해가 맞는지 재확인한 뒤 시스템을 아는 입장에서 대안을 제안. 기술 용어는 상대 언어로 풀어서. AX는 마케팅, CS, 물류 등 비개발 직군의 업무를 자동화 구조로 번역하는 일이라 이 패턴이 그대로 쓰인다.

## 7. 제출 원문 (톤 앵커 — 통화 직전 읽기)

> ninehire로 실제 제출한 원문. 전화 답변은 이 두 글의 뼈대 위에서만 쌓는다.

**지원동기**
> 저는 트라이포드랩에서 백엔드를 단독으로 책임지며, 최근 1년은 Claude Code를 단순 코드 생성 도구가 아니라 개발 파이프라인 자체로 재설계하는 일에 가장 많은 시간을 썼습니다. 도메인별 CLAUDE.md로 컨텍스트를 분할하고, MCP로 최신 문서를 실시간 연결하고, Subagent, PostToolUse Hook, Stop Hook 자가 리뷰로 코드 품질 회귀를 자동 통제했으며, 실DB 기반 사업지표 분석까지 AI 워크플로우로 묶었습니다. 이 과정에서 AI를 어디까지 도구로 쓰고 어디서 사람이 판단해야 하는가를 직접 설계하는 일이 가장 임팩트가 크다는 걸 체감했습니다. 에고이즘 그로스팀의 AX Engineer는 LLM과 코딩 에이전트를 마케팅, CS, 물류의 반복 업무를 재설계하는 전략적 자산으로 다루고, 그 효율을 비즈니스 성과로 직접 증명하는 자리라고 이해했습니다. 제가 사내 개발 파이프라인을 대상으로 해온 일을, 실제 운영 중인 브랜드들의 업무 전반으로 확장할 수 있는 자리라고 봐서 지원했습니다.

**가장 임팩트 있던 결과**
> 트라이포드랩에서 수기로 처리되던 발주 업무를 이벤트 기반으로 완전 자동화한 것이 가장 임팩트가 컸습니다. 수천 대 IoT 디바이스가 보내는 재고 데이터를 기준으로, 사람이 일일이 확인하고 발주하던 흐름을 AWS EventBridge, SQS 기반 이벤트 아키텍처로 재설계했습니다. 조직 규모상 Kafka 도입이 부담이라는 제약을 EventBridge, SQS로 풀었고, 재시도, DLQ, 알림까지 실패를 전제로 설계해 운영 중에도 누락 없이 돌아가게 만들었습니다. 결과적으로 사람이 붙어 있던 반복 운영 업무가 사라지고, 작은 팀이 수천 대 규모의 트래픽을 감당할 수 있게 됐습니다. 단순히 코드를 짠 게 아니라 어떤 업무를 어떤 단위로 자동화할 것인가를 직접 정의하고 문제 정의부터 배포, 운영까지 끌고 간 경험이라 가장 기억에 남습니다. 최근에는 같은 자동화 사고를 한 단계 확장해, Claude Code를 개발 파이프라인으로 재설계하고 실DB 사업지표 분석까지 AI 워크플로우로 자동화하고 있습니다. 반복 업무를 구조로 풀어내는 이 흐름이 에고이즘 그로스팀이 하려는 일과 정확히 맞닿아 있다고 봤습니다.
