## 2. 서비스 모델 — 양방향 + AI URL 구매

- **sazo.shop (2024.05, 일본)**: 일본 거주자가 한국 상품 직구 (역직구). 수수료 0엔.
- **사줘 sazo.kr (2025.8~9, 한국)**: 한국 소비자가 일본 상품 직구. 제휴몰 입점 상품 검색 구매 + **임의 쇼핑몰 URL 입력 → AI가 페이지 분석 → 환율, 관세, 배송비 포함 최종 결제 금액 사전 확정 → 1회 결제 → 통관, 배송 자동 처리**. 금액 확정 후 추가 청구 없음, 대행수수료 0원.
- 일본 현지 물류센터에서 검수 후 국제 배송. 피규어, 만화, 게임, 굿즈 등 일본 한정 상품이 주력.
- 채용공고 표현: 세계 최초 AI 기반 크로스보더 구매 경험 상용화. AI-First 조직 (팀 전체가 AI를 핵심 워크플로우로).

**BM의 기술적 의미**: 배송대행(배대지)이 아니라 **구매대행** — 결제부터 일본 내 구매, 입고, 검수, 국제 출고까지 풀 파이프라인을 직접 소유. 즉 OMS + WMS + 멀티 캐리어 추적이 전부 자사 백엔드 문제. 일본우정(물류), 스즈요(물류), 폴라오르비스(유통)의 전략 투자 = 물류 인프라 모델을 대기업이 검증한 구조.

## 3. 백엔드 도메인 문제 7개

### 3.1 스크래핑, 파서 파이프라인 (가장 차별적)
- N개 이질 소스: 라쿠텐(API 있으나 샵별 페이지 제각각), 아마존JP(봇 차단 강함), 메루카리(퍼블릭 API 없음), 야후 플리마켓, 임의 URL. 소스별 파서가 독립적으로 깨짐 → 파서 모듈 격리 + 소스별 파싱 성공률 메트릭이 운영의 본질.
- 깨짐 감지: 파싱 실패율 급증 알림, 필드 단위 결측률 추적, 골든 URL 회귀 테스트.
- Puppeteer 리소스: 브라우저 풀, 메모리 누수, 동시성 제한, 큐 기반 백프레셔(Kafka가 들어갈 자리).
- 메루카리 특수성: 중고 1점물 — 스크래핑 시점과 구매 시점 사이 품절 race 상시 존재. 상품 데이터 TTL 극단적으로 짧음.

### 3.2 환율 (3종 시점 분리)
- 표시용 실시간 환율 vs 결제 시점 고정 스냅샷 vs 통관 신고용 관세청 고시 환율(주 단위) — 이중 삼중 체계.
- 견적 유효시간, 주문 레코드에 환율 박제, 변동 리스크는 스프레드 버퍼. 배치 실패 시 이전 값 폴백 + 알림.

### 3.3 관세, 관부가세
- 자가사용 미화 **150달러 이하 목록통관 면세** (미국발 특송만 200달러). 초과 시 공제 없이 **전액 과세**.
- 과세가격 = 물품가 + 국제운송비 + 보험료. 관세 = 과세가격 × HS코드 관세율, 부가세 = (과세가격 + 관세) × 10%.
- **합산과세**: 같은 날 같은 통관 건이면 건별 150달러 이하라도 합산 초과 시 과세 → 합포장 vs 분리출고 결정이 고객 세금을 바꿈 = WMS 출고 로직과 세금 엔진이 결합.
- 목록통관 배제 품목(식품, 건기식, 의약품, 기능성 화장품)은 금액 무관 일반 수입신고 → 상품 카테고리 분류가 통관 경로를 결정.

### 3.4 통관, 국제배송 추적
- 개인통관고유부호 수집 검증, 목록통관 vs 일반통관 분기, 통관 보류와 검사 이벤트 처리.
- 멀티 캐리어 합성 추적: 일본 내(야마토, 사가와) → 국제(EMS, 특송) → 한국 내 택배. API 있는 캐리어 + 없는 캐리어 스크래핑 혼합, 캐리어별 상이한 상태 코드를 단일 상태 머신으로 정규화.

### 3.5 주문 파이프라인 (주문 → 구매대행 → 입고 → 검수 → 출고)
- 일반 커머스와 달리 **결제 시점에 재고를 소유하지 않음** — 주문은 구매 시도의 약속. 구매 실패(품절, 가격 변동), 부분 실패(N개 중 1개 품절)의 보상 트랜잭션이 일상 → 긴 라이프사이클 상태 머신 + Saga + 이벤트 기반 전이가 자연스러운 구조.
- 단계별 SLA 추적(구매까지 X시간, 입고까지 Y일)과 지연 알림이 CS 비용을 좌우. 앱 평점 2.7의 상당 부분이 이 구간일 가능성.

### 3.6 WMS (일본 현지 창고)
- 입고 매칭(주문 없는 입고, 합배송 대기), 검수(중고품 상태 확인 — 메루카리 물량에서 핵심), 합포장과 분할포장(합산과세 회피 로직과 결합), 출고와 통관 서류 생성 연동.

### 3.7 상품 검색 (Elasticsearch)
- 일본어 상품명 → 한국어 검색: 일본어 형태소(kuromoji) + 한국어(nori) 멀티 필드, 애니메이션과 피규어 도메인 고유명사 이표기 동의어 사전, 스크래핑 → 색인 갱신 신선도(품절 상품이 검색에 남는 문제).

## 4. 내 경험 매핑 (직격도 순)

| 사줘 도메인 문제 | 내 경험 | 직격도 |
|---|---|---|
| 멀티 캐리어 추적 (API + 스크래핑 혼합) | 시솔지주 15개 택배사 연동, API 없는 곳 스크래핑 + Sentry 변경 감지 | **직격** — 깨지는 외부 의존성을 모니터링으로 운영하는 동일 문제 구조 |
| 환율 처리 | 시솔지주 하나은행 API 30분 배치, 재시도 4회 | **직격** — 외부 금융 API 실패 내성 그 자체 |
| 통관 | 시솔지주 통관 환율 배치 + 통관 서류 번역(캐싱 3초→0.9초) | **직격** — 통관을 실무로 만져본 후보는 드묾 |
| OMS, WMS | 시솔지주 CBT OMS, WMS 구축 | **직격** — 국제물류 주문 창고 도메인 모델링 |
| 이벤트 기반 주문 상태 전이 (Kafka) | 트라이포드 EventBridge+SQS, DLQ, 채널별 재시도 차등 | **강한 전이** — 브로커만 다름, 멱등성과 실패 격리는 동일 |
| 1점물 동시 주문 race | 트라이포드 IoT 수천 대 동시 재고, SELECT FOR UPDATE NO WAIT | **강한 전이** — 동시성 정합성 경험 그대로 적용 |
| 조회 성능, 대량 데이터 | 트라이포드 복합 인덱스 99.3% 개선 | 전이 — ES 도입 판단의 근거 어휘 |
| 파서 성공률 메트릭 운영 | 트라이포드 Grafana, Prometheus, Loki SLO 알림 | 전이 — 스크래핑 관측 제안과 결합 시 직격 격상 |

**요약: 시솔지주 = 도메인 직격 (CBT, 통관, 환율, 추적, OMS, WMS), 트라이포드랩 = 아키텍처 직격 (이벤트, 동시성, 관측). 두 경험의 합집합이 사줘 JD를 거의 전부 덮는다.**

## 5. 도메인 이해 한 줄 멘트 (핏 인터뷰용)

1. "국제 배송 추적에서 API 없는 해외 택배사는 스크래핑으로 풀었는데, 그때 배운 건 스크래핑은 만드는 것보다 깨짐을 빨리 아는 게 본질이라는 점입니다. 사줘의 파서 파이프라인도 결국 소스별 성공률 모니터링 싸움이라고 생각합니다."
2. "환율 배치를 30분 간격 재시도 4회로 운영해봐서, 표시 환율과 결제 시점 환율, 관세청 고시 환율이 다 다른 시점 문제라는 걸 압니다. 직구 가격 엔진은 환율 스냅샷을 주문에 어떻게 고정하느냐가 핵심이라고 봅니다."
3. "메루카리 상품은 재고가 1개라 스크래핑 시점과 구매 시점 사이 품절 race가 상시 존재할 텐데, IoT 수천 대 동시 재고 정합성을 잠금 전략으로 풀어본 경험이 이 문제에 그대로 이어진다고 생각합니다."
4. "구매대행 주문은 결제 시점에 재고를 소유하지 않는 긴 상태 머신이라, 단계 전이를 이벤트로 풀고 실패를 DLQ와 보상 플로우로 다루는 설계가 자연스럽습니다. EventBridge와 SQS로 같은 구조를 운영해봤습니다."
5. "150달러 합산과세 때문에 합포장 한 번이 고객 세금을 바꾸는 도메인이라, WMS 출고 로직과 세금 계산이 결합되는 게 직구 백엔드의 재미라고 생각합니다."

> 양방향 포인트: 사줘는 직구 + 역직구 **양방향**. 한국 상품을 일본에 파는 역방향(일본 수입 통관, 엔화 결제)까지 언급하면 조사 깊이 시그널.

## 6. 도메인 특화 예상 질문 5

| 질문 | 답변 방향 |
|---|---|
| 일본 쇼핑몰 개편으로 파서가 깨지면 어떻게 감지, 대응? | 필드 단위 결측률 메트릭 + 임계 알림, 골든 URL 회귀 테스트, 소스별 파서 모듈 격리, 실패 건 DLQ 적재 후 파서 수정 뒤 재처리. 시솔 Sentry 경험 앵커 + Grafana 관측 보강 |
| 메루카리 1점물에 두 유저가 동시 주문하면? | 내부 선점은 DB 잠금(NO WAIT 경험 인용) 또는 Redis 분산락 + TTL. 외부 품절은 못 막으므로 주문 = 구매 시도로 모델링, 실패 시 자동 환불 보상 플로우를 1급 시나리오로. 내부 정합성과 외부 불확실성 분리 |
| 환율은 어떻게 다루나? 표시가와 결제가가 달라지면? | 환율 3종 분리(표시 실시간, 결제 스냅샷, 관세청 고시). 견적 유효시간 + 주문에 환율 박제 + 스프레드 버퍼. 배치 실패 시 이전 값 폴백 + 알림 — 하나은행 배치 경험 그대로 |
| 관부가세 150달러와 합산과세를 시스템에 어떻게 반영? | 과세가격 = 물품가 + 운송비 + 보험료, 초과 시 전액 과세, 관세 후 부가세 10% 계산 순서를 정확히 말하는 것 자체가 차별화. 합포장 결정과 세금 엔진 결합 설계, 배제 품목은 카테고리 분류 단계 플래그 |
| 주문 → 구매 → 입고 → 검수 → 출고를 MSA + Kafka로 설계한다면? | 단계별 서비스가 도메인 이벤트로 전이하는 코레오그래피 Saga. 멱등 컨슈머 + 단계별 DLQ와 재시도 차등(채널별 차등 경험 인용) + 보상 이벤트(구매 실패 → 환불). 입고 매칭, 부분 실패를 엣지로 먼저 언급 = OMS, WMS 실무 시그널 |
