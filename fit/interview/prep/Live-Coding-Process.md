# 라이브 코딩·시스템 설계 면접 프로세스

라이브 코딩은 **코딩 테스트가 아니라 실무 시뮬레이션**이다. 알고리즘 정답 풀이가 아니라 모호한 요구사항을 정의·분해·구현·개선하는 과정을 보여주는 자리.

## 코딩 테스트 vs 라이브 코딩

| 축 | 코딩 테스트 | 라이브 코딩 |
|---|---|---|
| 문제 | 명확함 | 모호함 (실무 문제) |
| 정답 | 1개 | 여러 접근 가능 |
| 평가 | 결과(통과/실패) | 사고·소통·우선순위 |
| 시간 분배 | 풀이 시간이 전부 | 정의/분석에 ⅓ 이상 |

**탈락 1순위 시그널**: 문제를 받자마자 코드부터 짜기 시작. "요구사항 파악 능력 없음 / 실무에서도 똑같이 덤빌 사람"으로 읽힌다.

## 60분 표준 분배

| 단계 | 시간 | 산출물 |
|---|---|---|
| 1. 문제 명확화 | 5분 | 면접관에게 질문 → 모호한 부분 확정 |
| 2. 제약·가정 정리 | 5분 | 칠판/노트에 제약, 규모, DB 스키마 가정 |
| 3. 접근 방식 비교 | 5~10분 | 2~3가지 접근 + 장단점 + 선택 이유 |
| 4. 구현 | 35~40분 | 핵심 로직 + 주석 + 말하면서 |
| 5. 개선안 | 5분 | 확장성·성능·Trade-off |

시간이 짧은 과제(30~40분)는 1·2단계를 합쳐 5~7분으로 압축하되 **생략하지 않는다**.

## 1단계: 문제 명확화

모호한 단어를 골라내 면접관에게 되묻는다. "면접관의 시간을 뺏는 게 아니라, 평가받는 행위 자체"다.

예: "초기 유저에게 비슷한 주제의 추천 피드"라는 과제를 받았다면:
- "초기 유저"는 가입 후 며칠 이내인가
- "비슷한 주제"는 무엇으로 판단하나 (가입 시 태그? 게시물 키워드 분석?)
- "친구가 적다"는 몇 명 이하인가
- "유입을 늘린다"의 지표는 무엇인가 (MAU/DAU/체류시간)
- 피드는 몇 개를, 어떤 순서로 보여주나

면접관이 정해주지 않으면 **본인이 가정하고 명시**한다 ("X로 가정하고 진행하겠습니다"). 침묵으로 추측하는 게 가장 나쁘다.

## 2단계: 제약·가정 정리

칠판/노트에 적어 면접관과 합의한다.

```
[제약 조건]
- 초기 유저: 가입 후 7일 이내
- 친구 수: 3명 이하
- 추천 개수: 10
- 정렬: 인기도 순

[가정 - 데이터 모델]
users(id, created_at, tags[])
posts(id, user_id, content, created_at, likes, views)
post_tags(post_id, tag_id)
```

규모도 같이 가정한다 ("DAU 10만, 게시물 1000만건"). 캐싱·인덱스 결정의 근거가 된다.

## 3단계: 접근 방식 비교

**한 가지만 던지면 위험**하다. "선택지 → 장단점 → 선택" 흐름을 보여주면 의사결정 능력이 평가된다.

예 (추천 피드):
1. 태그 기반 인기 게시물 — 구현 단순, 개인화 약함
2. 유사 유저 협업 필터링 — 개인화 강함, Cold Start
3. 하이브리드 (1 + 최신 게시물 섞기) — 다양성, 가중치 튜닝 필요

"시간 제약상 1로 시작하고, 남으면 3로 확장하겠습니다" — 우선순위 표명.

## 4단계: 구현 — 말하면서 짜기

침묵하면 사고 과정이 평가 불가능해진다. 한 줄 짤 때마다 짧게 설명한다.

- "먼저 엔드포인트부터 뚫겠습니다"
- "여기서 초기 유저 판별 → 가입 후 7일 이내"
- "태그 IN 절로 후보 게시물 뽑고, 인기도(likes+views)로 정렬"
- "최근 7일 게시물로 한정해서 신선도 확보"

막혔을 때도 침묵 금지. "이 부분이 두 가지 방식이 떠오르는데요" 같은 사고 출력으로 대응. 면접관이 힌트를 줄 여지가 생긴다.

## 5단계: 개선안

시간이 남거나, 끝나갈 때 "지금은 단순하게 짰지만 실서비스라면…"으로 확장. 캐싱, 다양성, 개인화, A/B 테스트, 비동기 처리 등.

시간이 부족하면 코드를 미완성으로 두고 마지막 1분에 "남은 부분은 X로 마무리하면 됩니다"라고 말하는 게, 끝까지 코드만 붙잡고 있는 것보다 낫다.

## 흔한 실패 패턴

| 패턴 | 시그널 | 대응 |
|---|---|---|
| 정의 없이 코딩 시작 | 받자마자 IDE 켬 | 1·2단계를 의식적으로 끼워넣기 |
| 침묵 코딩 | 면접관이 보고만 있음 | 모든 결정에 한 줄 설명 |
| 너무 복잡한 접근 | ML/협업필터링부터 꺼냄 | 단순 → 점진적 확장 |
| 한 부분 매몰 | DB 스키마에 30분 | 타이머 의식, MVP 우선 |
| 막혔을 때 정지 | "음…" 30초 | "두 가지가 떠오르는데" |
| 끝까지 코드만 | 개선안 없이 종료 | 마지막 3~5분은 무조건 회고 |

## 연습법

- **타이머 60분**으로 매일 1개 주제. 5단계 시간 분배 지킬 것.
- **소리 내서 설명하며 코딩** — 혼자 있을 때도. 익숙해질 때까지.
- **녹화 후 복기** — 침묵 구간, 같은 말 반복, 우선순위 실수 확인.
- 완성 후: 뭘 놓쳤나 / 더 나은 방법 / 시간 분배는 적절했나 — 3개 체크.

## 면접 포인트

Q. 라이브 코딩에서 가장 중요한 게 뭐라고 생각하는가?
- 코드의 완성도가 아니라 **문제 정의와 의사결정 과정**. 실무 문제는 정답이 없고, 면접관은 모호함을 어떻게 다루는지를 본다.

Q. 시간이 부족할 것 같으면?
- 끝까지 코드만 잡고 있지 않는다. 마지막 5분은 회고·개선안에 무조건 배정. "MVP는 여기까지, 남은 부분은 X 방식으로"가 침묵 종료보다 훨씬 강한 마무리.

Q. 막혔을 때 어떻게 하는가?
- 침묵 대신 사고를 출력한다. "두 가지가 떠오르는데 A는… B는…"으로 면접관과의 대화로 전환. 그게 협업 시그널이다.
