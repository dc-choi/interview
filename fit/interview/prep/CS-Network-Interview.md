# CS 네트워크 면접: 본질 질문 + 응용 시나리오

"CS 공부는 어디에 써먹나?"를 **네트워크 관점**으로 되묻는 형식의 면접 가이드. 단순 TCP/IP 정의 질문이 아니라, **왜 네트워크가 성능의 본질적 병목이며, 자원(CPU·메모리·디스크·네트워크)을 어떻게 교차 최적화하는가**를 꼬리 질문으로 이어간다. 꼬꼬무(꼬리에 꼬리를 무는) 질문 대비용 정리.

## Part 1. 본질 질문

### Q1. 네트워크 통신은 왜 성능을 저하시키는가?

- **물리적 거리 + 빛의 속도 제약** — 신호는 빛보다 빠를 수 없고, 실제로는 광섬유·라우팅 오버헤드로 훨씬 느림
- 내부 처리는 CPU 나노초·메모리 접근 수십 나노초인데, **네트워크는 밀리초**. 4~5자릿수 차이
- 결국 분산 시스템의 병목은 거의 항상 **네트워크와 디스크**

### Q2. 지구 반대편 서버와 통신하면 얼마나 걸리나?

- 빛의 속도로 지구 반 바퀴만 **~0.066초(66ms)** 편도 이론값
- 실제는 TCP/TLS/HTTP 프로토콜 오버헤드로 더 큼
- **CDN·엣지 캐싱**이 등장한 근본 이유

### Q3. TCP 통신 과정을 한 요청 기준으로 풀어보라

한 번의 HTTP 요청-응답에 필요한 통신 수:
- **3-way handshake** (SYN / SYN-ACK / ACK) — 3번
- **HTTP 요청 + 응답** — 2번(간단화)
- **4-way handshake** (FIN / ACK / FIN / ACK) — 4번
- **총 9번** 패킷이 왕복

지구 반대편 기준 편도 66ms면 **9 × 2 × 66ms ≈ 왕복 거리로 4바퀴 반**. 단일 요청에 0.5초 이상 소요 가능.

### Q4. 이걸 줄이려면?

- **Keep-Alive** — 매 요청마다 handshake 반복 안 함
- **HTTP/2 멀티플렉싱** — 한 TCP 연결로 다중 스트림
- **HTTP/3 (QUIC)** — UDP 기반, 0-RTT 재개
- **TLS 1.3 0-RTT** — 재방문 시 handshake 단축
- **CDN** — 물리적 거리 단축
- **연결 풀링** — 앱/DB 간 영구 연결 재사용

### Q5. 개발 중 네트워크로 주고받는 데이터는 뭐가 있나?

시각적 무거움 순:
- **미디어**: 이미지·폰트·동영상 (정적 자원)
- **API 페이로드**: JSON·XML·바이너리
- **DB 통신**: 쿼리·결과셋
- **서비스 간 호출**: MSA gRPC·Kafka·HTTP
- **외부 API**: 결제·SMS·인증

### Q6. 대역폭보다 큰 데이터를 주고받으면 어떻게 되나?

- **지연 누적** — 패킷 큐잉, 혼잡 제어(CWND 감소)로 TCP 전송률 하락
- **패킷 손실·재전송** 증가 → 악순환
- 특정 연결 하나가 대역폭을 독점하면 **이웃 요청도 지연** (혼잡)
- 해결: 압축·샘플링·버킷팅·페이지네이션

## Part 2. 응용 시나리오

### 시나리오 1. 크롤링 속도 튜닝

**상황**: 여러 쇼핑몰을 모아보는 서비스(지그재그·에이블리 류). 수백 사이트를 주기적으로 크롤링.

**동원되는 자원**:
- **CPU** — HTML/JSON 파싱
- **네트워크** — 대상 사이트 호출
- **디스크·DB** — 결과 저장
- **메모리** — 진행 중 데이터 버퍼링

**네트워크 튜닝**:
- **Blocking + 스레드 풀 확대** — 동시 요청 수↑, 단 스레드당 메모리 비용
- **Non-Blocking I/O** — 소수 스레드로 수천 요청 다중화 ([[Sync-Async-Blocking|동기·비동기]])
- **Connection Pool** — 재연결 비용 제거 ([[Connection-Pool]])
- **Bulk Insert** — 결과 저장 시 1건씩이 아니라 N건 묶어 한 번의 쿼리로. 통신 횟수 감소

**CPU 튜닝**:
- 멀티 스레드·멀티 프로세스로 코어 활용 ([[Single-vs-Multi-Thread]])
- I/O 대기를 비동기화해 CPU 유휴 최소화
- 스레드를 **과도하게 만들면** 컨텍스트 스위칭 비용이 역효과

**메모리 튜닝**:
- 페이지 단위로 스트리밍 파싱 → 전체 로드 금지
- OOM 방지 위해 진행 중 버퍼 크기 제한
- 백프레셔(downstream이 못 따라오면 상류 속도 조절)

**답변 포인트**: "네트워크가 가장 느린 자원이므로 동시성으로 가리고, 저장은 배치로 통신 횟수를 줄인다. CPU·메모리는 과다 병렬화의 역효과를 견제하는 상한 역할."

### 시나리오 2. 웹페이지 로딩 속도 개선

**상황**: 사용자 체감 "페이지가 느림" 민원.

**진단**:
- 브라우저 개발자 도구 **Network 탭** — 큰 요청, 긴 TTFB, 블로킹 리소스 식별
- **Core Web Vitals** — LCP·CLS·INP 지표

**네트워크 관점 처리**:
- **정적 자원 최적화** — 이미지 압축(WebP/AVIF), 사이즈 축소, lazy loading
- **폰트 최소화** — 서브셋·variable font
- **JSON 최소화** — 불필요한 공백·필드 제거
- **HTTP 압축** — gzip/brotli
- **CDN + 캐싱 헤더** — `Cache-Control`, `ETag`
- **HTTP/2·3** — 멀티플렉싱·0-RTT

**렌더링 관점 처리**:
- Critical CSS 인라인
- `async`·`defer`로 JS 파서 블로킹 제거
- Preload·Preconnect 힌트

**답변 포인트**: "측정→가장 큰 병목부터. 네트워크가 병목이면 자원 크기·압축·캐싱, 렌더링이 병목이면 Critical Path 축소."

### 시나리오 3. 대용량 트래픽 처리

**상황**: DAU 급증, 서버 CPU는 여유 있는데 응답 지연 증가.

**네트워크가 병목일 때의 신호**:
- 응답 시간은 길지만 CPU·DB는 한산
- `netstat`·`ss`에서 소켓 대기 다수
- 네트워크 인터페이스 사용률 한계 근접

**처리 전략**:
- **JSON 크기 축소** — 개행·공백 제거, 불필요 필드 제외, 압축(gzip/brotli)
- **응답 페이로드 정리** — over-fetching 제거, Projection DTO
- **페이지네이션·커서** — 대형 목록은 분할
- **필드 마스킹·선택 반환** — 프런트가 필요한 필드만
- **GraphQL·Partial Response** — 요청 측에서 필드 선택
- **HTTP/2 멀티플렉싱** 활용
- **CDN/엣지 캐싱** — 반복 요청 오프로드
- **Rate Limit** — 악성·버그 요청 차단

**답변 포인트**: "CPU가 놀고 있는데 느리다면 네트워크·디스크를 의심. 페이로드 축소와 캐싱이 첫 타자."

## Part 3. 자주 꼬리 붙는 질문

- "네트워크와 디스크 중 어느 게 더 느린가?" → 둘 다 느리지만 **디스크 랜덤 I/O**와 **원거리 네트워크**가 최악. SSD + 근거리는 네트워크가 더 예측 가능
- "Connection Pool 크기를 늘리면 성능이 좋아지나?" → **아니다**. DB 하드웨어·락 경합 한계 때문에 정점 후 하락. [[Connection-Pool|Little's Law]] 참조
- "Blocking vs Non-Blocking 언제 뭘 쓰나?" → 요청당 스레드 모델이 단순하면 Blocking. C10K 이상은 Non-Blocking·이벤트 루프. [[Sync-Async-Blocking]]
- "Bulk Insert의 단점은?" → 한 트랜잭션의 크기가 커지면 **롤백 비용·락 보유 시간** 증가. 배치 단위 타협
- "JSON 압축으로 얻는 이득을 수치로?" → gzip으로 60~80% 축소 가능. 모바일·해외 사용자에 특히 체감
- "HTTP/2가 모든 상황에서 빠른가?" → Head-of-Line blocking은 TCP 레벨에 남음 → HTTP/3(QUIC)이 그것까지 해결

## 면접 답변 템플릿

시나리오형 질문의 공통 구조:

1. **자원 교차 관점 선언** — "CPU·네트워크·디스크·메모리 중 어느 게 병목인가 측정부터"
2. **네트워크 쪽 조치**(동시성·통신 횟수·페이로드 축소·캐싱)
3. **그 다음 자원 리밋**(CPU 과병렬화·메모리 OOM 경계)
4. **측정 지표 명시** — P99·TTFB·네트워크 사용률·CPU 사용률
5. **트레이드오프 언급** — "추가 복잡도·비용과 성능 이득의 균형"

## 면접 체크포인트

- TCP 한 요청에 실제로 **몇 번 통신이 오가는지** 계산할 수 있는가
- 네트워크·CPU·메모리·디스크를 **동시에 얹어 사고**하는가
- 크롤링·로딩·트래픽 시나리오를 각각 2~3분으로 설명할 수 있는가
- Connection Pool·Bulk Insert·Non-Blocking·캐싱을 **조합**해서 답할 수 있는가
- "측정 없이 최적화 금지" 원칙을 발화할 수 있는가
