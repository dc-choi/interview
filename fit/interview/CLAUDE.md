# Interview Domain Rules

이 파일은 `fit/interview/` 아래의 면접 답변, 준비 가이드, 회사별 문서와 회고에 적용한다. 신규 회사 면접 준비의 실행 절차는 `interview-prep` 스킬에서 관리한다.

ORM 관련 답변이나 기술 vault 내용을 다룰 때는 `tech/CLAUDE.md`도 읽는다.

## common과 prep 역할 분리

- `common/`은 개인 경험, FIT, 기술 경험 답변 본문을 관리하는 마스터다. `My-*.md`, `Common-Interview-Questions.md`, `Interview-Retro-Template.md`가 여기에 속한다.
- `prep/`은 가이드, 질문 풀, 메타, 치환과 자기분석을 관리한다.
- 실제 파일 목록은 작업 시점에 확인하고 각 파일의 frontmatter `aliases`와 첫 헤더로 역할을 검증한다.

## 답변 본문 배치

- 개인 경험, FIT, 개인 기술 경험의 재사용 답변 본문은 `common/`에만 둔다.
- `prep/`의 기술 질문 풀에는 특정 개인 경험에 의존하지 않는 간결한 사실형 답변을 포함한다. 상세 기술 설명은 `tech/` 정본을 반복하지 말고 링크한다.
- 회사별 문서는 선택한 마스터 답변을 fork하고 회사 매핑 끝줄을 붙인다. 이 문서는 사용자가 연습할 때 vault나 `prep/`으로 이동하지 않아도 되도록 자기완결적으로 작성한다.

## 예상 질문 필수 구성

- 질문만 저장하지 않는다. 사용자가 바로 연습하고 오답을 교정할 수 있는 준비 내용을 붙인다.
- 사실형 기술 질문에는 질문 의도, 핵심 답변, 흔한 오답, 꼬리질문, 학습 정본이나 출처를 포함한다.
- 설계형 기술 질문에는 단일 정답 대신 답변 골격, 판단 기준, 대안과 트레이드오프, 꼬리질문을 포함한다.
- `prep/`의 일반 질문 풀에서 개인 경험과 FIT 질문을 다룰 때는 준비 포인트와 해당 `common/` 마스터 링크만 두고 개인 답변 본문을 복제하지 않는다. 회사별 문서에서는 위 답변 본문 배치 규칙에 따라 선택한 마스터 답변을 fork하고 회사 매핑 끝줄을 붙인다.
- 새 답변은 처음부터 `Self-Analysis`의 WHY, `Developer-Interview-13-Questions`의 질문 의도, `Developer-Interview-Signals`의 답변 톤과 정렬한다. 면접관이 함께 일하고 싶은 사람이라는 시그널을 받을 수 있어야 하며, 작성 후 별도 다듬기 단계에 의존하지 않는다.

## 답변 원칙

- **외부 귀책 금지**: `My-Motivation-Reasons`의 자기 귀책, 학습과 현재 적용 구조를 따른다. 비누커머스와 윤회 회고에서 반복된 패턴을 유지하고 시솔지주 이직 사유는 마스터 옵션 B인 시니어 사고 격상을 사용한다.
- **현재 이직 사이클 확인**: 현재 회사의 퇴사 이유를 작성하기 전에 `fit/CLAUDE.md`의 현재 커리어 상태와 `My-Motivation-Reasons`의 현재 사이클 변수를 대조한다. 다음 사이클 슬롯이 비어 있거나 사용자가 현재 이직 사유를 확정하지 않았다면 먼저 사용자에게 확인하고, 종료된 과거 사이클을 현재 답변으로 재사용하지 않는다.
- **기술 답변 단순화 조건 표기**: 말하기 쉽게 단순화하면 적용 전제, 제품 버전, 예외 중 필요한 조건을 답변 카드 끝줄에 남긴다. 정확성을 바꾸는 생략을 조건 표기 없이 단정하지 않는다.
- 마스터에 없는 회사 특이 질문은 `prep/`의 Sensitive, Behavioral, Tech-Basics와 Tech-Scale 일반 질문 풀에서 fork할 수 있다. 반복되는 새 답변은 마스터에 역반영한다.
- 사전 통화나 커피챗에서 본인이 실제로 말한 카드와 말하지 않은 카드를 회고에서 구분한다. 본 면접에서는 앞서 말한 카드를 우선하고 면접관이 직접 물을 때만 새로운 큰 카드를 답한다. 앞선 대화에서 꺼내지 않은 큰 카드를 이유 없이 추가해 진정성 시그널을 깨지 않는다.

## vault 링크 위치

| 위치 | 허용 | 이유 |
|---|---|---|
| 답변 카드 본문 위의 vault 매핑 줄 | 아니오 | 답변 흐름을 방해한다. |
| `common/` 마스터 끝의 `### vault 카테고리 인덱스` | 예 | 한 문서 안에서 보강 위치를 찾을 수 있다. |
| 회사별 문서의 vault 매핑 | 아니오 | 필요한 콘텐츠를 본문에 흡수해 자기완결적으로 만든다. |
| 별도 통합 vault 인덱스 | 아니오 | 탐색 단계를 늘린다. |
| 같은 답변 시스템 안의 cross-reference | 예 | 답변 시스템 내부 연결이다. |
| `prep/` 가이드의 톤과 시그널 매핑 | 예 | 답변 정렬을 점검하는 메타 정보다. |

## 면접 후 마스터 역반영

- 회고는 회사별 차수 폴더에 `Interview-Retro-{회사명}-{차수}.md`로 저장하고 `common/Interview-Retro-Template.md`를 fork한다.
- 회고에서 마스터에 없는 새 답변이나 받은 질문이 발견되면 마스터를 업데이트한다.

## 회사별 폴더

- 경로는 `fit/interview/{회사명}/`이며 회사명은 영문 소문자를 사용한다.
- 신규 회사 문서는 마스터 fork와 회사 고유 내용만 두는 슬림 패턴을 사용한다.
- 차수별 서브폴더 `{회사명}/{차수}/`를 반드시 사용한다. 차수는 `1st/`, `2nd/`, `coffeechat/`처럼 표현한다.
- 회사 폴더 루트에는 모든 차수에 공통인 자료만 둔다. 예를 들어 회사 도메인 브리프는 루트에 둘 수 있다. JD, Tech, FIT, Lead, 체크리스트, Cheatsheet와 회고는 해당 차수 폴더에 둔다.
- 회사별 폴더는 leaf 폴더 파일 수 분할 규칙보다 회사와 차수 단위 묶음을 우선한다.
- 액션파워, 윤회와 스페이스맵 문서는 마스터 도입 전에 작성되어 답변 본문 중복이 남아 있다. 기존 문서를 관련 작업으로 수정할 때 마스터 fork 형태로 점진적으로 정리한다.
