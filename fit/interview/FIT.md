# FIT
정답이 없는 질문. 다만, 미리 생각해두는게 좋음.

## 면접 준비 프레임워크

면접 전에 회사를 4가지 관점으로 분석해두면 답변에 구체성이 생긴다.

### 1. 내가 이 회사에 가지는 강점
- 내 경험 중 이 회사 도메인/문제와 직접 연결되는 것
- 기술 역량뿐 아니라 도메인 적응 속도, 구조화 능력, 기술 의사결정 능력까지

### 2. 회사가 나에게 기대할 것
- 회사의 현재 단계에서 필요한 역할 (안정성, 속도, 설계 등)
- 기획서가 불완전할 때 구조를 제안하는 능력
- 비즈니스 구조를 시스템으로 전환하는 능력

### 3. BM 구조 추정
- 돈 버는 방식, 핵심 고객, 데이터 흐름
- 운영 안정성이 중요한지 vs 트래픽 대응이 중요한지 판단

### 4. 어떤 역할을 맡아야 하는가
- 아키텍처 설계 / 기술 의사결정 / 운영 관점 설계 / 팀 리드 등
- 나중에 합류할 개발자들이 이해할 수 있는 구조를 남기는 역할

### 전달해야 하는 핵심 메시지 (예시)
- 빠르게 만드는 것보다 **운영에서 깨지지 않는 구조**를 중시
- 기술 선택은 스펙이 아니라 **단계의 문제** (Kafka vs SQS, Redis Lock vs DB Lock)
- 초기에 구조를 잘 잡으면 이후 인력/운영/장애 비용이 줄어든다
- 혼자 잘하는 구조보다 **다음 사람이 와도 이해할 수 있는 구조**

### 연봉 협상 전략
- 제시 → 합의 목표 → 마지노선의 3단계로 준비
- 숫자보다 **역할과 책임** 기준으로 논리를 세울 것
- "이 포지션은 단순한 개발자 채용이 아니라 플랫폼 책임 역할이라고 이해하고 있습니다. 그 기준으로 의논하고 싶습니다."

## 면접 현장 주의사항

> Self-Analysis 기반 모순/불일치 점검 결과. 텍스트 자체의 모순은 아니지만 면접에서 답변이 연결될 때 리스크가 있는 항목.

**"성급함" 단점 말할 때 이썸테크 6개월 퇴사를 떠올리지 않기**
- "성급함" → "후회되는 선택: 6개월 퇴사" → "결국 성급하게 나온 거잖아요?"로 면접관이 꿰뚫을 수 있음
- 성급함의 예시는 **기술적 실행 판단**(ex: 충분히 검토 전에 구현 시작)으로 들 것
- 이썸테크 퇴사는 "방향 판단"이지 "성급한 실행"이 아니라는 구분을 유지

**CTO 목표 말할 때 "권한"이라는 단어 쓰지 않기**
- 무의식적으로 최종 권한이 없으면 힘이 빠지는 경향이 있음 (Self-Analysis)
- "CTO를 원하는 진짜 이유가 기술이 아니라 권한 아닌가?"로 해석될 위험
- **"영향력으로 문제를 해결하는 것"**, **"기여의 크기"**로 표현할 것
- "역할 자체보다 기여가 중요하다"는 방어 라인 유지

## 면접질문

### 1분 자기소개

답변 구조: **이름/역할 → 도메인 경험 요약 → 핵심 성과 → 핵심 철학 → 왜 이 회사 → 마무리**

- 보안/물류/VMI 분야에서 수작업 의존 운영 프로세스를 시스템으로 전환하는 역할
- 수천대 IOT 기기 동시 재고 데이터 환경에서 데이터 정합성 보장 + 운영 가능한 구조 설계
- DB Lock 기반 동시성 제어로 재고 오류 제거, 발주/수주/알림 이벤트 기반 재설계로 수기 운영 자동화
- 핵심 철학: 실시간 처리보다 정합성과 복구 가능성 우선, 실패 전제로 재시도/DLQ/알림 포함 구조 설계
- 비즈니스 흐름을 데이터와 상태로 정리하고 운영 효율을 높이는 시스템을 만드는 데 기여하고 싶다

### 저희 회사에 지원한 이유가 무엇인가요?
> 참고: [[Zerobase-Career-Seminar#지원동기|지원동기 작성법]] — 기업의 고민(퇴사 우려)에 답하는 구조

답변 구조: **회사의 미션 이해 → 내 관련 경험 → 연결점 → 기여하고 싶은 것**

- 회사가 풀고 있는 문제가 단순한 서비스가 아니라 복잡한 운영을 시스템으로 전환하는 시도라는 점에 공감
- 내 경험에서도 비슷하게 수작업 의존 프로세스를 시스템으로 대체해왔음
- 단순히 기능을 구현하는 것을 넘어, 비즈니스 구조를 이해하고 시스템 상태와 데이터 플로우로 바꾸는 역할을 해왔음
- 이 환경에서는 실시간 처리보다 정합성과 복구 가능성, 운영자가 이해할 수 있는 데이터 흐름이 더 중요

### 이전 직장에서 퇴사한 이유가 무엇인가요?
> 참고: [[11st-Engineer-Seminar|11번가 엔지니어 세미나]]

**이썸테크 (6개월)**: 프로젝트 단위 개발이라 한 서비스를 장기적으로 개선하는 경험을 쌓기 어려워 제품 중심 조직으로 이직 결심
- 꼬리: "6개월이면 너무 빠른 결정 아닌가?" → 3개월간 열심히 배웠지만 "내가 원하는 방향이 아니다"는 확신이 선 이상 더 머물러도 같은 고민 반복
- 꼬리: "적응력 부족 아닌가?" → 적응은 문제없었고 빠르게 기여. 문제는 적응이 아니라 방향. 이후 시솔지주 1년, 트라이포드랩 1년 8개월 안정 근무

**시솔지주 (1년, 권고사직)**: FIDO 인증 성공시키고 백엔드 팀 리드했지만, 시장 반응 부진으로 회사가 사업 종료 결정 → 팀 전체 정리
- 꼬리: "개인 역량 부족 때문 아닌가?" → 저만 나간 게 아니라 팀 전체 정리. 오히려 FIDO 인증 성공이 가장 큰 성과
- 8개월 공백기: 멋쟁이사자처럼 부트캠프에서 Java/Spring 체계적 재학습, 수강생 100명 중 최우수 코드 선정. 공백이 아니라 준비 기간

**트라이포드랩 (현재)**: 이직 사유 항목 참고

### 이직 사유 (현재 회사에서 왜 옮기려 하는가)

답변 구조: **현 회사에서 충분히 성장한 점 → 한계를 느낀 지점 → 다음 도전 목표 → 지원 회사와의 연결**

**B2C 지원 시**: PoC 단계의 넓은 경험 → 이제는 더 많은 사용자가 직접 사용하는 B2C에서 빠르게 피드백 받으며 개선하는 경험을 쌓고 싶음. "피드백 사이클"이 빠른 환경에서 성장하고 싶다
- 꼬리: "B2C도 느린데?" → 피드백 사이클 자체가 빠름 (PoC 수개월 vs B2C 주 단위)
- 꼬리: "B2B는 더 이상 싫은 건가?" → B2B 경험은 값졌지만 지금 커리어 단계에서는 빠른 개선 경험이 필요
- 꼬리: "피드백 느려지면 또 이직?" → 이직 기준은 속도가 아니라 **성장 기회**

**B2B 지원 시**: PoC 단계에서 넓게 경험 → 이제는 실제 운영 중인 서비스에서 대규모 트래픽과 확장성 문제를 다루고 싶음
- 꼬리: "운영은 재미없지 않나?" → 오히려 반대. "실제 사용자 문제를 해결하는 것"이 목표
- 꼬리: "우리 규모가 기대만큼인지 확인했나?" → 규모 자체보다 기술적으로 도전적인 문제가 있는가가 중요
- 꼬리: "더 큰 규모 원하면 나가는 거 아닌가?" → 규모가 아니라 성장 기회. 함께 성장하는 과정이 더 값짐

### 기획자와 의견 충돌 사례

답변 구조: **상황 → 상대 입장 이해 → 구체적 리스크 설명(비개발자 언어) → 대안 제시 → 결과**

- 결제 기능 개발 시 "환불 정책은 나중에" 제안에 대해 리스크를 비개발 언어로 설명 (사용자 민원, 회계 처리, 기술 부채)
- 대안 제시: 환불 정책 수립을 병렬 진행 + 결제 구조를 환불 고려해 미리 설계
- 일정 1주일 추가됐지만 오픈 후 민원/회계 문제 없이 안정 운영
- 꼬리: "기획자가 끝까지 우기면?" → 리스크 문서화 → 팀 리드/PM에 공유 → 의사결정권자 판단 존중. 개발자는 리스크를 알리는 역할이지 최종 결정자는 아님
- 이후 기획자가 먼저 "기술적으로 가능해요?" 물어보는 문화로 변화

### 백엔드 기술적 의견 충돌 사례

- FIDO 서버 개발 시 팀원이 라이브러리(SimpleWebAuthn)의 규격 준수 여부 의문 제기
- 리더로서 일정(3개월) vs 품질 딜레마 → "검증 → 이슈 제기 → 해결" 프로세스로 해결
- WebAuthn Spec 문서와 대조하여 실제 규격 상이 부분 발견 → GitHub 이슈 생성 → 메인테이너 패치 → 인증 통과
- 배운 점: 의심은 품질을 높이는 과정, 리더는 양자택일이 아니라 해결 방법을 찾아야 함, "누가 맞았냐"보다 "함께 해결했다"가 중요

### 긴급 이슈 동시 발생 시 판단 기준

1. **비즈니스 임팩트**: 사용자 직접 영향(서비스 중단 > 사용성 저하) → 빈도(전체 > 특정) → 복구 시간
2. 혼자 해결 vs 팀 도움 필요를 빠르게 판단. 혼자 5시간 걸릴 것 같으면 바로 도움 요청
3. 판단 어려우면 팀 리드에게 바로 물어보기. "모르면 빠르게 물어보는 것"도 능력
- 꼬리: "3순위 문제가 더 커지면?" → 왜 그렇게 판단했는지 설명 + 개선책 제시 (모니터링 강화 등). 실수 인정 + 개선이 핵심

### 같이 일하기 힘든 동료가 있을까요?

- 기술만 잘하는데 협업 안 되는 사람
- 커뮤니케이션은 좋은데 실력이 따라오지 않는 사람
- 책임을 회피하는 사람
- 기술, 커뮤니케이션, 책임감의 균형이 중요

### 장기적으로 어떤 개발자가 되고 싶은가요?
> 참고: [[Growth-Principles|성장의 7가지 원칙]]

**5년 후**: 개발 리드. 코드를 잘 짜는 것을 넘어 팀의 기술 방향을 제시하고 주니어 성장을 돕는 역할
- 꼬리: "리더십 경험은?" → 시솔지주에서 백엔드 3명 리드 FIDO 인증 성공 (기술 의사결정, 갈등 조율, 일정 관리)
- 꼬리: "매니저 vs 코드?" → Staff Engineer 지향. 코드 직접 짜면서 팀 기술 방향 제시 + 멘토링

**최종 목표**: CTO. 단 10~15년 후 방향. 기술적 깊이(3~5년) → 리드(5~8년) → 큰 조직(8~10년) 단계적 성장
- 꼬리: "왜 스타트업이 아니라 중견?" → 지금 부족한 건 기술적 깊이. 대규모 환경에서 먼저 깊이를 쌓겠다
- 꼬리: "회사 이용만 하고 나가겠네?" → CTO가 될 만한 사람으로 성장하고 싶은 것. 성장 가능하면 오래 함께할 것
- "비즈니스 임팩트를 만드는 개발자": Prisma 90% 성능 개선, 클라우드 30% 비용 절감, 발주 자동화(4시간→10분)

### 어떤 조직문화에서 일해오셨어요?

- 수평적인 조직에서 일했다. 직급이 없었다
- 동료 간에는 의견을 적극적으로 개진하는 편이지만, 과거에는 윗사람 앞에서 주저한 적이 있었음
- 동료로부터 "주장을 더 강하게 해도 좋겠다"는 피드백을 받고, 근거가 있으면 당당하게 말하는 연습을 해왔음
- 지금은 데이터와 논리를 준비한 뒤 명확하게 의견을 전달하려고 노력 중

### 이상적인 회사의 모습은?

- 회사는 공동의 목적을 가진 단체
- 공동의 목적을 향해 달려가되, 개인이 목적 달성을 위해 필요한 것들을 소통으로 해결해야 한다

### 소통을 어떻게 하시나요?

- **기획자**: 기획 의도를 먼저 물어보고, 이해한 내용이 맞는지 재확인. 시스템을 아는 입장에서 "이런 게 가능하다"를 알려줌. 기술 용어는 풀어서 설명
- **프론트엔드**: API 문서를 최대한 자세하게 작성 (Request/Response 예시, 에러 케이스, 예상 응답 시간). "문서가 곧 소통". 에러 처리 방식 통일 (Status Code 명확 구분, Error Response 형식 통일). API 변경 시 Breaking Change 사전 공지
- **백엔드 동료**: 의견 충돌은 감정이 아닌 논리로. 근거 기반 검증 후 함께 해결. "누가 맞았냐"보다 "함께 해결했다"가 중요

### 남들이 나를 어떻게 보는지, 그리고 나는 나를 어떻게 생각하는지

- 남들이 볼 때: 친근하고 유들유들한 이미지
- 본인 생각: T라서 "개발자같다" 모먼트가 자주 있고, 지내다 보면 어려워하는 사람도 있음
- 대응: 그런 부분은 바꿀 수 없지만, 소통을 위해 말을 부드럽게 하려고 노력한다

### 회사 비전 중 하나를 고르고 사례를 들어주세요

- **객관적인 판단**을 선택
- 지표나 데이터가 있어야 소통이 원활하다고 생각
- 사례: 클라우드 비용 지표를 확인하고 규모 대비 과다 비용 판단 → 인프라 개선안 제안 → 30% 비용 절감

### 비개발자에게 기술 개념을 설명해보세요

- 개념 먼저 설명 → 예시 → 비유적으로 설명하는 순서
- 실제 면접에서 스택(Stack)을 인사담당자/경영실장에게 설명하여 집중시킨 경험

### 마지막 한마디

- 인터뷰를 통해 느낀 점을 짧게 언급 (회사가 단순한 X가 아니라 Y를 풀어가는 팀이라는 걸 느꼈다)
- 내 경험을 바탕으로 기여하고 싶은 점
- "주어진 역할에만 머무르기보다, 팀이 더 안전하게 성장할 수 있는 구조를 함께 고민하는 동료가 되고 싶다"

### 이전 직장에서의 주요 성과가 무엇인가요?
> 참고: [[tech/senior/시니어역량(SeniorEngineer)|시니어역량]]
> 참고: [[Zerobase-Career-Seminar#직무관련성공경험|성공경험 정량화]] — 돈, 개선률, 상대적 위치로 수치화

답변 프레임: **상황 → 문제 인식 → 기술적 접근 → 결과(수치)**

- Metric: "P99 응답 속도가 300ms 개선되었다"
- Impact: "유지보수 비용이 줄어 신규 기능 개발 속도가 1.5배 빨라졌다"
- Scalability: "사용자가 10배 늘어도 인프라 비용 증가폭은 2배 이내로 억제했다"

**비용 절감 사례**: 메가존 클라우드 분석 도구 도입 → RDS/EC2 다운그레이드, 미사용 리소스 정리 → 월 $250→$80 (30% 절감) → 절감분으로 모니터링/SonarQube 재투자
**성능 개선 사례**: Prisma JOIN 미지원으로 N개 개별 쿼리 → relationLoadStrategy 적용 → 1개 SubQuery로 통합 → API 성능 90% 향상
**오류율 개선 사례**: 50% 오류율 → 400건 분류(치명도/빈도/원인) → 테스트 0에서 도입 → 1% 미만으로 개선
- 꼬리: "0%는 못 만드나?" → 현실적으로 불가능. "빠른 감지 & 복구"가 더 중요. 구글도 99.9% 가용성이 목표

### 본인의 장단점

- **장점: 실행력**. "이거 해야겠다" → 바로 움직임. Prisma 문제 발견 즉시 공식 문서 찾아 해결, 테스트 없는 환경에서 주저 없이 0→1 도입
- **단점: 실행 판단이 빠른 만큼 성급할 때가 있음**. 감정적으로는 신중한 편이지만, 실행으로 옮기는 속도가 빨라서 충분히 검토하기 전에 움직이는 경향이 있었음
- 개선: 중요한 결정 전 "이게 정말 최선인가?" 한 번 더 생각. 급할수록 우선순위 먼저 정리. 감정은 억제하되 실행 전 검증 단계를 의식적으로 추가

### 동료에게 받았던 기억에 남는 피드백

- "조금 더 너의 주장을 강하게 해도 좋을 것 같아" — 프론트엔드 동료
- 깨달음: "겸손"과 "자신감 없음"은 다름. 근거 있으면 당당하게 말해도 됨
- 이후 변화: "~인 것 같아요" → "입니다. 왜냐하면". 데이터 기반 + 명확한 주장 연습

### 함께 성장하는 개발 문화를 어떻게 만드는가

- "먼저 공유하는 사람"이 되기: 커뮤니티 3,500명 발표, 카카오테크 멘토, 사내 트러블슈팅 노트 정리
- 코드 리뷰: "이게 틀렸다"가 아니라 "왜 이렇게 했는지" 먼저 물어보고 "이런 방법도 있다" 제안
- 동기부여: 말이 아니라 행동으로. 칭찬 먼저 + 개선 제안은 부드럽게
- 문화는 강요가 아니라 자연스럽게 만들어지는 것

### 번아웃 관리

- 신호 캐치: 코드 짜는 게 재미없어짐, 작은 일에 짜증, 집중력 저하
- 원인 구분: 육체적 피로(→ 휴가, 운동) vs 정신적 피로(→ 업무 프로세스 개선, 새로운 도전)
- "통제 불가능한 것(면접 결과) 집착 → 번아웃" vs "통제 가능한 것(실력 쌓기) 집중 → 회복"

### 몰입 경험

- 시스템 전반 효율 개선에 몰입: API 느림 → Prisma 분석 → 90% 개선 / 비용 증가 → 분석 도구 도입 → 30% 절감
- "코드만 잘 짜는 게 아니라 서비스 전반을 보는 게 진짜 개발자"라는 깨달음
- 시간 관리: 오전 Deep Work (미팅 차단, 몰입 2~3시간) + 오후 Shallow Work (회의, 리뷰)

### 스스로를 어떻게 평가하나요

- "시행착오를 통해 방향을 찾아온 사람". 문제를 발견하면 그냥 넘어가지 못하고 해결하려고 파고드는 스타일
- 꼬리: "후회되는 선택?" → 첫 회사 6개월 퇴사. 하지만 그 경험이 "내가 원하는 환경"을 명확히 해줌
- 꼬리: "해결 못 한 문제?" → 레거시 리팩토링. 비즈니스 우선순위 때문에 완전히 해결 못 했지만, PR마다 조금씩 정리 + 새 코드는 베스트 프랙티스. "아직 진행 중"

### 성과를 다른 사람이 가져간다면

- 상황 파악 먼저 (고의 vs 실수)
- 일회성이라도 한 번은 직접 확인: "이 부분은 제가 작업한 건데 혹시 오해가 있으셨나요?" 정도로 가볍게 짚음
- 한 번 지켜본 뒤 반복되면 명확히 대응: 감정적이 아니라 사실 기반으로 선을 긋기
- 참고 넘어가는 것이 아니라, 확인하되 관계를 해치지 않는 방식으로 능동적으로 대응
