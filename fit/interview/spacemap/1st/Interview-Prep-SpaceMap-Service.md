## 1. 서비스 맥락 질문 (면접관 → 나)

> "우리 서비스에서는 이런 상황인데, 어떻게 하겠는가?" 패턴. 우주 데이터의 실제 형태를 모르는 상태이므로 — **단정하지 말고, 가정을 명시한 뒤 설계하고, 역으로 실제 구조를 되묻는** 흐름으로.

### Q1. 우주 데이터(위성 궤도, 추적 로그)를 실시간 수집, 처리하는 파이프라인을 새로 설계한다면?

- **가정 명시**: "데이터가 다수 관측 소스에서 시계열로 들어오고, 일부는 무거운 연산(궤도 전파 등)이 필요하다고 가정하겠습니다"
- **설계**: 수집 어댑터 → 큐 버퍼 → 검증, 멱등성 → 연산 워커(CPU 바운드는 분리) → append-only 원천 저장 + 조회 최적화 테이블 → API. (상세는 [[Interview-Prep-SpaceMap-Tech-JD|Tech-JD #5]])
- **실무 연결**: 트라이포드랩 IoT 파이프라인이 같은 골격
- **되묻기**: "실제로 데이터 소스 수와 유입 빈도, 연산 무게가 어느 정도인가요?" — 정답을 아는 척하지 않고 구조를 함께 그리는 태도

### Q2. 교육용 소프트웨어는 어떻게 접근하겠는가?

- 우주 데이터를 *교육 맥락*으로 풀어내는 제품 — 사용자가 학생, 교육자라 **정확성만큼 이해 가능성, 반응성**이 중요
- 실무 연결: 출석부 사이드 프로젝트 — **주일학교 멀티테넌트 SaaS** (2026-07-15 보고서상 모임 레코드 108개, 랜딩 누적 본당 85곳, 학생 3,669명, 출석 약 22,051건, MAO 34곳)를 **1인 설계, 운영**. 사용자 인터뷰, MAU/MAO 메트릭 직접 추적, Parish→Church→Organization 3단계 위계 테넌트 + 명시적 권한 가드 + Snapshot 패턴(이력 보존). 누적 본당과 모임 레코드는 활성 수치로 쓰지 않는다. "교육 도메인 사용자는 기술을 모른다"는 전제에서 출발하는 감각.
- 카카오테크 캠퍼스 BackEnd 멘토 — 교육 컨텐츠를 전달해본 경험

### Q3. 데이터 엔진의 응답이 갑자기 느려졌다. 어떻게 진단하겠는가?

- **계층별 切り分け**: 메트릭부터 — APM/Grafana로 어느 구간(수집, 연산, DB, API)이 느린지 / 슬로우 쿼리 로그, 실행계획 / 큐 깊이, 워커 처리율 / DB CPU, ReplicaLag / Event Loop Lag
- 실무 연결: Grafana, Prometheus, Loki를 직접 구축하고 SLO 알림을 설계한 경험 — "조용히 느려지는 것"을 탐지하는 체계를 만들어봄
- 단발성 spike vs 지속 패턴을 구분(`for: 5m`), 지속 패턴이면 인덱스, 쿼리 구조 의심

### Q4. 데이터 소스, 연동 대상이 자주 추가된다면, 확장성 있는 구조는?

- 소스별 **어댑터 패턴** — 수집 인터페이스를 표준화하고 소스별 구현만 추가. 파이프라인 본체는 불변
- 실무 연결: 시솔지주에서 약 15개 택배사 API를 통합 연동 — API 미제공 택배사는 스크래핑으로, Sentry로 변경 감지. "이질적인 외부 소스를 표준 인터페이스 뒤로 숨기는" 경험
- 트라이포드랩의 도메인별 모듈 분리 설계도 같은 원리

### Q5. 미 우주군, 우주청 과제 — B2G/방산 환경의 보안, 신뢰성 요구는 어떻게 다루겠는가?

- B2G, 방산은 **정합성, 감사 추적, 접근 통제**가 계약 조건. "조용히 틀리면 안 되는" 도메인
- 실무 연결: ① 정보관리보안학과 전공 ② 시솔지주에서 FIDO(WebAuthn) 인증 서버를 Spec부터 설계해 공식 인증 통과 ③ 관측 인프라로 감사 가능한 로그 체계 구축
- 접근: 최소 권한, 모든 데이터 변경의 추적 가능성(누가, 언제, 무엇을), 정합성 우선(가용성과 트레이드오프 시)

### Q6. 5년차 포지션인데 4년차다. 어떻게 그 갭을 메우고 기여할 것인가?

- **정면 인정 + 재정의**: "연차는 1년 모자랍니다. 다만 5년이라는 숫자가 보장하려는 건 *문제의 깊이, 범위를 감당한 경험*이라고 생각합니다"
- **밀도 근거**: 4개 회사 + 실서비스 사이드에서 — 동시성 정합성(DB Lock), 1억 건 데이터 인덱스 설계, 단일 서버→ECS 아키텍처 전환, 관측 인프라 0→1 구축, 레거시 마이그레이션(2회), FIDO 인증 서버 팀 리딩, AI 파이프라인 재설계. 5년차에게 기대하는 범위를 이미 통과
- **빠른 적응 증명**: Java→Node, JSP→Spring, MongoDB→MySQL — 새 스택을 반복 흡수. Python, Airflow, 우주 도메인도 같은 패턴
- **솔직한 한 줄**: "갭이 없다고 우기지 않겠습니다. 다만 그 갭이 6개월이면 닫힐 종류라는 걸, 제 이력이 보여준다고 생각합니다"

## 2. 컬처핏 / 소프트스킬 질문

> [[Common-Interview-Questions|FIT]], [[Self-Analysis|나찾기]] 기반. 스페이스맵에 특히 중요한 항목 위주.

- [ ] **장단점** — 장점: 데이터, 성능 문제를 끝까지 파고들어 구조로 해결. 단점: 완성도를 높이려다 속도가 느려질 때가 있음 → 우선순위, 타임박스로 관리 중
- [ ] **"강한 몰입, 업무 강도 환경"을 어떻게 받아들이나** (JD 명시 — 반드시 대비)
  - 솔직하게: 몰입 자체는 마다하지 않음 — 1인으로 실서비스를 무중단 운영하며 강도 높게 일해봄. 다만 **지속 가능한 몰입**을 위해 파이프라인, 자동화로 처리량을 구조적으로 끌어올리는 편(AI 협업 파이프라인 사례)
  - 되물을 것: "업무 강도가 높다"는 게 상시인지, 과제 마감 등 특정 구간인지 — 역질문으로 실제를 확인 (아래 역질문 참고)
  - ⚠️ 주의: 무조건 "야근 좋아합니다"도, 방어적으로 "워라밸 중요합니다"도 아닌 — *구조로 강도를 감당한다*는 프레임으로
- [ ] **동료와 기술적 의견 충돌** — 근거(실행계획, 벤치마크, 문서)로 대화. 시솔지주 Prisma 이슈를 공식 문서, 실행계획 비교로 풀었듯, 감정이 아니라 데이터로
- [ ] **기획자/PM과 충돌** — 트레이드오프를 숫자로 설명(이 일정엔 이 정합성을 포기해야 함 등). 결정은 존중하되 리스크는 기록
- [ ] **긴급 장애 판단** — 영향 범위, 데이터 손실 가능성으로 우선순위. 관측 인프라로 빠르게 원인 좁히고, 롤백 가능 여부를 먼저 확인
- [ ] **번아웃, 지속가능성 관리** — 강도 높은 환경일수록 중요. 자동화로 반복 노동 제거, 우선순위로 일을 줄임
- [ ] **성장 목표** — 데이터 난이도가 높은 도메인에서 파이프라인, 연산 백엔드를 깊게. 장기적으로 데이터 엔진 설계를 주도하는 엔지니어
- [ ] **새 언어, 도메인 학습 자세** — 이력 자체가 증거(반복된 스택 전환). 우주, Python을 학습 대상으로 명확히 인지하고 입사 전부터 시작할 의지

## 3. 역질문 (나 → 면접관)

> 핵심 2~3개는 외워서, 심화는 노트. "강한 업무 강도"는 반드시 실체를 확인할 것.

### 기술 / 아키텍처
- 우주 데이터는 실제로 어떤 형태인가요 — 데이터 소스 수, 유입 빈도, 한 건의 크기, 시계열인지 / 가장 무거운 연산은 무엇인가요?
- "데이터 엔진"은 구체적으로 어떤 역할인가요 — 수집, 연산, 서빙 중 어디에 무게가 있나요?
- 데이터 파이프라인은 현재 무엇으로 돌아가나요 — Airflow를 실제로 쓰고 계신가요, 아니면 도입 예정인가요?
- 코드 베이스에서 Python과 TypeScript의 비중, 역할 분담은 어떻게 되나요?

### 팀 / 조직
- 백엔드 팀 규모와 구성이 궁금합니다. 이 포지션은 기존 팀 합류인가요, 신규 라인인가요?
- 5년차 이상을 찾으시는데, 이 자리에 특히 기대하는 역할은 무엇인가요? (갭 우려를 역으로 확인 — 리딩 기대인지, 즉시 전력인지)
- 코드 리뷰, 기술 의사결정은 어떻게 이뤄지나요?

### 성장 / 기대치 / 근무
- 입사 후 6개월 동안 기대하는 모습은 무엇인가요?
- 공고의 "강한 몰입과 업무 강도"는 구체적으로 어떤 모습인가요 — 상시인가요, 과제 마감 등 특정 구간인가요? (실체 확인)
- 성과 기반 보상이라고 하셨는데, 성과는 어떤 기준으로 평가되나요?
- 채용 절차의 다음 단계는 어떻게 되나요?
