# 토스 계열 면접 조언 (멘토, 2026-03-11)

> 멘토 카카오톡 조언. 당시 맥락은 토스뱅크, 토스증권 Node.js Developer 집중채용 JD 분석 요청이었고, 내용은 토스 계열 공통 면접 성향이라 회사 폴더 루트(전 차수 공통)에 둔다. 이후 26.06 토스플레이스 1차 면접으로 상당 부분 실측 검증됨 — 아래 대조표 참조.

## 조언 요지

1. **토스 Node.js JD는 늘 올라온다.** 채용이 안 되는 건지, 간을 보는 건지, 벽이 정말 높은 건지 불명. 어느 쪽이든 쉽지 않은 포지션이라는 신호.
2. **면접에서 서비스에 대한 진심을 많이 요구한다.** 예: 데이터 정합성이 깨지면 사용자에게 어떻게 표현해줄 것인가, 어떻게 대처할 것인가. 기술 자체보다 서비스 관점의 사고를 굉장히 중요하게 본다.
3. **지원자의 커리어와 포트폴리오에 집중해서** 그 서비스 관점 질문이 많이 들어온다. 내 경험에서 출발하는 꼬리 질문을 가정할 것.
4. **JD 요구 기술에 대한 트러블 슈팅 경험**이 있는지 묻는다.
5. **기술 내부 프로세스**도 꽤 묻는다. 예: 내부적으로 통신이 안 되면 어떻게 동작하는지.
6. 단서: 모든 JD가 이 형태는 아닐 수 있고 멘토가 아는 범위 기준.

## 1차 면접 대조 (26.06.09 토스플레이스) — 적중 검증

| 조언 | 실제 1차에서 | 판정 |
|---|---|---|
| 커리어, 포트폴리오 집중 질문 | 과제 코드 라이브 디펜스가 면접 전체. 내 코드에서 출발한 꼬리 질문 연쇄 | 적중 |
| 데이터 정합성 관점 | 동시 예약 충돌, 확정끼리 안 겹침 불변식, SQLite 직렬화 vs MySQL 락 등 정합성이 핵심 축. 다만 사용자에게 어떻게 표현하고 대처할지의 서비스 관점 버전은 미출현 | 부분 적중 |
| 기술 내부 프로세스 | 단일 인스턴스 EventEmitter vs 분산 MQ 가정 구분은 출현. 통신 단절 시 동작 같은 장애 시나리오는 미출현 | 부분 적중 |
| 트러블 슈팅 경험 | 1차는 설계 깊이 중심이라 직접 미출현 | 미출현 (다음 단계 대비 유지) |
| 벽이 높다 | 막힐 때까지 미는 설계 깊이 probe 2개(엔티티 위치, 정책 폭발), 결과 탈락 | 적중 |

요약: 멘토 조언은 1차에서 과반 적중. 미출현 항목(서비스 관점 표현과 대처, 트러블 슈팅, 장애 시나리오)은 빗나간 게 아니라 1차가 설계 디펜스에 집중됐기 때문으로 보는 게 안전하다. 토스 계열 재지원 시 다음 단계(컬처핏, 2차)에서 나온다고 가정하고 준비를 유지한다.

## 다음 토스 계열 지원 시 적용

- **서비스 관점 답변 한 벌 상비**: 정합성이 깨졌을 때 (1) 어떻게 탐지하고 (2) 사용자에게 무엇을 보여주고 (3) 어떻게 대처(보상 트랜잭션, 재대사, 공지)하는지를 내 프로젝트 기준으로 시나리오화. [[Interview-Prep-TossPlace-Domain|도메인 브리프]]의 재고, 단말 상태 정합성 매핑과 같은 축.
- **트러블 슈팅 카드 매핑**: JD 요구 기술별 트러블 슈팅 경험을 [[My-Tech-Cards]]에서 미리 선별해 두기.
- **통신 단절 시나리오**: 외부 연동이 죽으면 어떻게 동작하는가 — 타임아웃, 재시도, 서킷 브레이커, 멱등 재처리. 토스플레이스 도메인(외부 물류 시스템 연동)과 직결.
- **지원 이력 진행 곡선**: 25.08 토스(지인 추천, Node) 과제 탈락 → 26.05 토스플레이스 과제 통과, 1차 면접 탈락. 과제 벽은 넘었고 현재 갭은 설계 깊이의 라이브 표현. 재지원 시 출발점은 [[Interview-Retro-TossPlace-1st]]의 모범답변 2개(Strategy 정책 추출, 도메인 모델 vs 퍼시스턴스 엔티티 분리).
