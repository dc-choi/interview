# 토스플레이스 도메인 브리프 (전 차수 공통)

> 지원 직무의 실제 소속과 업무, 회사 내 위치, 그리고 회의실 예약 과제가 이 직무와 어떻게 연결되는지. 차수 공통이라 회사 폴더 루트에 둔다.
> JD는 채용 페이지를 렌더링해 확보(job_id 7686375003). 이 직무는 결제 정합성 백엔드가 아니라 Operations Team의 사내 운영 플랫폼 직무다 — 이 점이 FIT과 역질문의 축이다.

## 지원 직무: Operations Team Node.js Developer

JD 원문 기준 정확한 정의:

- **소속**: 토스플레이스 Operations Team.
- **만드는 것**: 결제단말기 다변화와 물류 체인 고도화에 맞춘 **유연하고 확장 가능한 단말기 생애주기 관리 및 물류 관리 플랫폼**.
- **방향**: 구성원이 더 효과적으로 일하도록 **업무 자동화와 시스템**을 개발해 조직 전반에 영향과 인사이트를 제공.
- **범위**: 개발만이 아니라 **문제 정의부터 성과 측정까지** 전 과정 주도.

합류 후 업무(JD):

- 토스플레이스 여러 팀을 관찰하고, 반복적이거나 비효율적인 수작업을 기술로 자동화하고 시스템화한다.
- 다양한 외부 물류 시스템과 데이터를 연동하고, TypeScript, Node.js, Nest.js 기반의 **서버, 배치, Internal Product**를 개발한다.
- 반복되는 문제를 빠르게 패턴화하고, **재사용 가능한 모듈과 시스템**을 만들어 플랫폼을 점점 효율화한다.

핵심: 이 직무는 결제 승인 트래픽을 처리하는 대고객 백엔드가 아니라, **단말기와 물류라는 물리 자산의 생애주기를 데이터로 관리하고, 사내 반복 업무를 자동화하는 내부 플랫폼**이다.

## 토스플레이스는 무엇을 하는가

- 비바리퍼블리카(토스) 계열사로, 오프라인 200만 자영업자의 매장 운영을 다루는 대면결제 사업이다.
- 결제 단말을 가맹점에 공급하고 운영한다. 그래서 **물리 단말의 재고, 출고, 설치, 고장, 회수**라는 물류와 자산관리 문제가 사업의 중심에 있다.
- 이 직무(Operations)는 그 물류와 단말 운영을 떠받치는 내부 시스템과 자동화를 만든다.

## 토스 생태계 내 위치

- 토스(슈퍼앱), 토스뱅크, 토스증권, 토스페이먼츠(온라인 결제 PG), 토스플레이스(오프라인 결제와 단말).
- 참고로 이 공고는 3개 계열사(토스, 토스증권, 토스플레이스) 집중채용이고, 셋 다 Node.js Developer지만 팀이 다르다. 토스와 토스증권은 AI 에이전트와 MCP 기반 자동화 플랫폼, **토스플레이스는 단말 생애주기와 물류 운영 플랫폼**이다. 면접에서 직무 이해를 물으면 이 차이를 짚어주면 강하다.

## 과제와 이 직무의 매핑 (재프레이밍 — 결제가 아니라 물류/자산/자동화)

| 과제에서 보여준 것 | 이 직무(Operations)와의 연결 |
|---|---|
| ReservationSlot 상태 전이(PENDING → CONFIRMED 승격) | 단말기와 물류 자산의 **상태 머신**(재고 → 출고 → 설치 → 고장 → 회수) |
| 트랜잭션 경계, 확정끼리 안 겹침 불변식 | **재고 수량과 단말 상태 정합성**(한 단말이 동시에 두 상태일 수 없음, 재고 음수 방지) |
| 리마인더 @Cron + reminderSentAt 멱등 | **배치 자동화 + 멱등 재처리**(중복 처리 방지, at-least-once 설계) |
| 알림 Sender 전략 패턴, 레지스트리, OCP | **외부 물류 시스템 연동의 어댑터/플러그인 구조**(택배사, ERP 추가가 무수정 확장) |
| use-case per file, 1급 도메인 객체 분리 | **재사용 가능한 모듈과 시스템화**(JD 직접 키워드) |
| 권한 분리, 출력 whitelist, 운영 체크리스트 | **Internal Product의 안정성과 관측** |
| 도메인 이벤트 발행 + 커밋 후 처리 | 자산 상태 변화를 이벤트로 흘려 자동화 트리거로 잇는 seam |

정합성 카드는 버리는 게 아니라 **대상만 결제에서 재고/단말 상태로 바꾼다.** 동시성 설계 서사는 그대로 유효하다.

## 왜 이 직무인가 (FIT 톤)

- JD가 "문제 정의부터 성과 측정까지 주도"를 명시한다. **명확한 기술 문제와 측정 가능한 성과**에서 강한 내 성향과 정확히 정렬된다.
- 반복 수작업을 시스템으로 제거하는 일은 내가 사이드로 만드는 주일학교 출석 플랫폼(사내 도구 성격의 Internal Product)과 같은 결이다 — 관찰, 패턴화, 자동화, 지표.
- 물리 자산(단말)을 데이터 모델과 상태 머신으로 다루는 문제는 과제에서 보여준 도메인 모델링 접근과 직결된다.

## 회사 맞춤 역질문 (직무 정렬)

- 단말기 생애주기를 상태 머신으로 모델링하나요, 이벤트 소싱에 가깝나요? 상태 전이의 정합성은 어디서 강제하나요?
- 외부 물류 시스템 연동에서 재고 카운트 어긋남 같은 정합성 문제는 어떻게 맞추나요(보상 트랜잭션, 재대사)?
- 업무 자동화의 성과를 어떤 지표로 측정하나요(절감 시간, 수작업 에러율, 처리량)?
- Internal Product라 사용자가 사내 구성원인데, 여러 팀의 요구 우선순위는 어떻게 정하나요?
- 배치와 실시간 처리의 경계는 어떻게 두나요? 배치 실패의 재처리와 멱등성은 어떻게 보장하나요?
