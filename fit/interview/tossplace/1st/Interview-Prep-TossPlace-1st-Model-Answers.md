# 토스플레이스 직무 면접 — 과제 모범답변 스크립트

> [[Interview-Prep-TossPlace-1st-Assignment-Defense|디펜스 시트(3뎁스 드릴)]]의 짝. 코드에서 뽑은 **구어체 답변**. 통암기 X, 본인 말로 재구성용.
> ⚠️ 토스 저작권. 공개 저장소 푸시 금지.

## 1. 예약 대기 로직 (ReservationSlot)

> 예약 생성 시 그 시간에 확정 예약이 있으면 거부하지 않고 PENDING으로 만듭니다. PENDING끼리는 겹쳐도 됩니다. 기존 확정 예약이 취소될 때 그 슬롯의 대기들을 승격하는데, 승격 로직은 ReservationSlot이라는 순수 도메인 객체에 모았습니다.
>
> 알고리즘은 확정 집합과 대기 집합으로 나눈 뒤, 대기를 createdAt FIFO로 정렬하고 동률이면 id로 결정적 정렬합니다. 단일 패스로 돌면서 각 후보가 확정 집합 어디와도 안 겹치면 확정으로 올리고, 방금 승격한 예약도 즉시 확정 집합에 넣습니다. 막힌 후보는 건너뛰고 뒤를 봅니다. 선두가 막혔다고 전체를 막지 않는 가동률 우선 방식입니다.
>
> 부분 충돌 예로, C가 09:30~10:30 대기인데 A(09~10)와 B(10~11)가 점유 중이면 A만 취소돼도 B와 겹쳐 막히고 둘 다 취소돼야 확정됩니다. 반면 D가 09:00~10:00 대기면 A 취소 순간 확정 집합과 안 겹치니 같은 패스에서 바로 확정됩니다. 확정 집합 전체와 겹침을 보기 때문에 자연스럽게 처리됩니다.
>
> 단일 패스로 충분한 이유는, 승격이 확정 집합을 늘리기만 하지 다른 대기를 풀어주지 않기 때문입니다.

## 2. 동시성 / 트랜잭션

> 확정 예약을 읽고 상태를 결정하고 쓰는 구간 전체를 하나의 트랜잭션으로 묶었습니다. 직렬화 단위는 회의실의 특정 날짜이고 다른 방이나 날짜는 병렬로 처리돼도 무방합니다. 지금은 SQLite better-sqlite3라 동기 단일 writer 모델이어서 쓰기가 DB 레벨에서 직렬화됩니다. row lock이 없으니 트랜잭션 경계만으로 충분합니다.
>
> MySQL이나 PostgreSQL이었다면 두 가지를 더 했을 겁니다. 첫째, 트랜잭션 직후 Room 행을 PESSIMISTIC_WRITE, 즉 SELECT FOR UPDATE로 잡아 같은 회의실 트랜잭션을 직렬화합니다. 더 높이려면 room, date 단위 advisory lock입니다. 둘째, PG라면 btree_gist EXCLUDE 제약으로 확정끼리 안 겹친다는 불변식을 DB가 강제하게 합니다. 동시 확정 시 한쪽이 제약 위반으로 실패해 재시도하죠. 데드락은 락 순서를 room, date 사전순으로 통일해 피합니다.
>
> 어느 DB든 승격 알고리즘은 안 바뀌고 임계구역을 무엇으로 직렬화하느냐만 바뀝니다. 취소의 경우 nativeDelete가 DB에 반영돼야 비워진 자리가 보이므로 승격 재조회 전에 flush로 먼저 반영합니다.

## 3. API Key 인증

> 키는 prefix와 secret을 붙인 형식이고 DB에는 표시용 keyPrefix와 SHA-256 해시만 저장합니다. 평문은 발급 응답에 한 번만 노출됩니다. 검증은 prefix로 단건 인덱스 조회 후 상수 시간 비교로 해시를 확인하고, 활성과 유효기간을 본 뒤 lastUsedAt을 갱신합니다.
>
> 비밀번호는 scrypt인데 키는 SHA-256인 이유는 성격이 다르기 때문입니다. 비밀번호는 사람이 만든 저엔트로피라 brute-force를 막으려면 느린 해시가 필요하고, API Key는 192비트 고엔트로피 랜덤이라 brute-force가 사실상 불가능하며 인증 경로는 저지연이 중요해 빠른 해시가 맞습니다.
>
> revoke는 삭제가 아니라 revokedAt 소프트 방식이라 감사 추적이 남고, 남의 키 존재를 숨기려고 비소유자는 404로 통일합니다. 키로 또 키를 발급하는 권한 상승을 막으려 발급은 쿠키 세션만 허용합니다. JWT 쿠키와 API Key는 가드에서 통합해, Bearer가 있으면 API Key, 없으면 쿠키로 인증하고 어느 쪽이든 req.user를 채워 기존 코드를 안 건드립니다.

## 4. 알림 기능

> 세 층으로 나눴습니다. 알림 타입, 채널 무관한 내용을 만드는 MessageBuilder, 실제 발송 전략인 Sender입니다. 디스패처가 채널에서 sender로 라우팅하고요. 카톡 추가는 채널에 KAKAO를 넣고 KakaoSender를 구현해 등록만 하면 됩니다. 디스패처나 예약 유즈케이스는 안 바뀝니다.
>
> 예약 모듈이 알림 모듈을 직접 호출하지 않고 도메인 이벤트만 발행하면 리스너가 구독합니다. 결합을 끊은 것이자 나중에 큐로 빼는 지점입니다. 알림 페이로드는 트랜잭션 안에서 스냅샷으로 만들고 발송은 커밋 후에 합니다. 취소처럼 곧 삭제될 예약도 안전하고, 한 수신자 실패는 격리해 다른 발송과 예약 트랜잭션을 막지 않습니다.
>
> 10분 전 리마인더는 매분 크론이 타이밍만 트리거하고 도메인 룰은 별도 유즈케이스입니다. 중복 발송은 reminderSentAt 가드로 막는데, 후보를 확정이고 reminderSentAt이 null이고 날짜가 오늘이나 내일인 것으로 좁히고 수집과 마킹을 한 트랜잭션에서 원자적으로 합니다. 시간은 주입식이라 테스트에서 고정합니다.

## 5. 검색 개선

> 기존 like 쿼리가 레코드가 늘면 풀스캔이 되는 문제였습니다. username을 좌측 앵커 prefix로 검색하되 NOCASE 콜레이션 인덱스를 추가했습니다. SQLite 기본 LIKE는 case-insensitive인데 BINARY 인덱스로는 LIKE 최적화가 안 걸려서요. collate nocase 인덱스를 선언하니 실행계획이 풀스캔에서 인덱스 탐색으로 바뀝니다. 이걸 테스트에서 EXPLAIN QUERY PLAN으로 고정해 회귀를 막았습니다. 한계는 좌측 앵커만 인덱스를 타고 중간 검색은 못 한다는 의식적 트레이드오프입니다.

## 6. 권한 / 검증

> 인증과 인가를 분리했습니다. 인증 실패는 가드에서 401, 인가 실패는 도메인 유즈케이스에서 isOrganizedBy로 403입니다. 목록 조회 IDOR는 메모리 필터가 아니라 WHERE절에서 본인이 주최했거나 초대받은 것만 DB가 걸러 부하와 정보 누출을 동시에 해결합니다. 잘못된 날짜는 두 층입니다. DTO 검증에서 정규식이 못 거르는 2099-02-30이나 13월을 그레고리력에 실제 존재하는지 확인하고, KST 기준으로 과거 날짜를 차단합니다.

## 7. 아키텍처 / 테스트

> 컨트롤러에 다 있던 걸 책임별로 분리했습니다. 컨트롤러는 한 엔드포인트가 한 유즈케이스를 호출하는 HTTP 어댑터, 비즈니스는 유즈케이스, 슬롯 룰은 1급 도메인 객체입니다. 도메인 로직은 엔티티 메서드로 풍부하게 둬 anemic model을 피했습니다.
>
> 테스트는 두 갈래입니다. 단위는 DI 없이 순수 로직, 슬롯 승격이나 디스패처 격리, 토큰을 봅니다. 통합은 실제 SQLite 메모리 DB로 supertest를 돌리는데 모킹을 최소화하고 Sender만 no-op으로 바꿔 랜덤 실패의 플래키를 제거합니다. 알림은 EventEmitter를 spy해 올바른 이벤트와 수신자가 발행되는지 보고, 매 테스트마다 스키마를 드롭하고 시드해 격리합니다.

## 8. 약점 정면돌파 (드릴 들어오면)

- **스케줄러 다중 인스턴스 중복**: 단일 인스턴스 전제라 SQLite로 우연히 안전. 다중이면 리더 선출이나 분산 락 필요. create엔 동시성 주석 길게 달면서 스케줄러엔 안 단 비대칭 인정.
- **scope enforcement 부재**: 인증 확장이 이번 범위, 인가는 다음 단계. 지금은 전권이라 가드에 scope 체크 한 줄이 맞다. 구조만 선제한 미완성 인정.
- **과거 시각 미검증**: 과거 검증을 날짜 단위로 해석. 오늘의 지난 시각까지 막으려면 time을 봐야 함. 누락 인정.
- **리마인더 유실**: 발송 전 sent 마킹이라 실패 시 유실, at-most-once. 예약 흐름 보호 우선의 트레이드오프지만 중요 알림이면 발송 후 멱등키로 at-least-once가 맞다.
- **원칙**: 의도적 미구현은 트레이드오프로 선긋고, 진짜 누락은 변명 없이 인정 + 고칠 방법. [[Developer-Interview-Signals]] 솔직함 신호.
