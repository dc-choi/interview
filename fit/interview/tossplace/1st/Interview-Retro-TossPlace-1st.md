## 진행 흐름 (실제 받은 질문, 답한 핵심)

### A. 발표 본론 — 구조 설명은 일관됨

- 컨트롤러(표현 계층) → 유스케이스(애플리케이션) → 리치 도메인 모델. MikroORM이라 엔티티에 메서드를 둬 anemic model 회피. 도메인별 엔티티 + 예약을 슬롯 개념으로 1급 도메인 객체화. 대기 예약 FIFO.
- 동시성: 회의실 자원 동시 예약 충돌이 핵심. MySQL로 개발하다 과제가 SQLite라 자체 직렬화에 의존, MySQL이면 락 고려. 취소도 레이스 컨디션 인지. (직관 정확)
- 알림: NotificationSender 인터페이스 + 이메일, SMS 샌더. 예약 이벤트 → 리스너 → EventEmitter, 분산이면 MQ. (가정 구분 정확)
- API 키: 쿠키 토큰 → API 키 기반 요구. SHA-256(SHA-1보다 보안), 스코프, 테넌트, 기본 권한.

### B. 라이브 코딩 — 슬랙 샌더 추가 (참석자 2)

- **Q: 이메일, SMS만 있는데 슬랙 알림 요구가 들어오면 어떻게 추가할 건가. 실제로 구현해봐 달라.** (AI 없이 손으로, 수도코드 수준 OK) → SlackSender 파일 생성, 인터페이스 상속, 채널 타입에 SLACK 추가, 어댑터 패턴으로 레거시 안 건드림, 웹훅 URL 가정해 sendSlack 작성.
- **꼬리: 웹훅 정보는 보내는 쪽이 알 필요 없다, Config로 받아 처리한다면?** → 함수형이라 구조를 바꿔 클래스로 ConfigService 주입, 결국 트랜스포트 추상화를 없애는 쪽으로.
- **꼬리: 생성자 빨간 줄?** → ConfigService 주입 구현이 없어서.
- **꼬리: 기존에 트랜스포트로 분리한 의도?** → 레거시로 보고 안 건드리려 어댑터 패턴.

### C. 계층 분리 의도 (참석자 2)

- **Q: common, infra, domain 셋의 책임?** → 도메인=프로젝트 도메인 규칙(API키, 인증, 초대, 예약, 룸, 유저), common=공통(설정, 데코레이터, 가드, 필터, DTO, 유틸), infra=외부 연동(이메일 알림, 퍼시스턴스 DB도 외부로 봄).
- **Q: 프레젠테이션 vs 유스케이스 구분? 프레젠테이션에서 에러를 throw하는 경우가 있더라.** → 프레젠테이션은 요청 응답과 예외 처리, 유스케이스는 DB 조회. **유스케이스에서도 예외 처리가 일어나야 하는데 이 부분은 놓쳤다 인정.** (솔직함 좋음)
- **★ Q: 엔티티가 코어 도메인 안에 들어가 있는데, 같은 엔티티(User.person)가 auth 같은 다른 유스케이스에서도 쓰임. 엔티티를 특정 도메인 디렉터리에 종속시키는 게 맞나, 계층을 분리하면?** → 인증과 유저는 맞닿아 있다, 유스케이스는 auth에 두는 게 맞다, 메서드는 유저 속성 확인이니 유저 로직. 면접관이 리뷰 형태로 한 번 더 누름 → **되묻기 반복("또 도메인 객체 만든다는 말씀?")**, 복잡해지면 도메인 객체와 퍼시스턴스 엔티티를 분리하겠다, 퍼시스턴스 엔티티는 인프라.
- **꼬리: 지금 도메인 안에 같이 둔 게 의도냐 미고려냐?** → 내부 툴, 작은 규모라 의도적으로 겸함(YAGNI), 커지면 분리. (트레이드오프 프레이밍은 유효)

### D. 슬롯 분리와 정책 폭발 (참석자 4) — ★ 가장 크게 흔들린 구간

- **Q: 예약의 일부로 슬롯을 볼 수도 있는데 굳이 분리한 이유?** → 수정 삭제에서 예외가 많이 발생, 리치 도메인 모델로 분리. 슬롯을 DB 상태로 안 둠(분산 가정 안 함). 한 파일 200줄 넘으면 AI나 사람 인지가 어렵다 → 책임 분리.
- **Q: 시간 겹침 로직은 안 변하지만 승격 로직은 정책이 다양(회장님이 예약하면 기존을 밀어내고 우선). 유저 의존 정책이 들어오면 어떻게 구현?** → 시나리오 고민(비서 로그인 등). 면접관이 단순화 유도(인증된 유저에 조건 들어간 것까지만). checkPermission 메서드를 어디 둘지(auth vs user) 고민, 승격은 정렬로 회장님을 앞으로 보냄.
- **★★ Q: 그런 조건은 계속 추가된다. 승격 로직(ReservationSlot)이 비대해지고 200줄 원칙과 리치 도메인 모델이 깨진다. 구조적으로 개선할 방법?** → **메서드를 쪼갠다**로 답함 → 면접관 **"메서드를 나눠도 슬롯 안을 못 벗어난다. 메서드 나누는 걸로는 해결 안 된다"**고 직접 반려. 면접관 핵심: 가독성보다 **객체의 관심사 수가 인지 부하의 핵심이고, 슬롯이 수많은 정책을 다 안다는 것 자체가 문제**. → 본인 "도메인 객체를 더 만들 것 같다, checkPermission을 더 둔다"로 맴돌다 **구현 보여주려는 순간 시간 종료로 끊김.**

## ★ 핵심 두 질문 모범답변 (라이브에서 못 찍은 것)

### 1. 엔티티 위치 = 도메인 모델 vs 퍼시스턴스 엔티티 분리 (C의 ★)

지금 엔티티 하나가 **MikroORM 퍼시스턴스 매핑과 도메인 행위를 겸하고 있다.** 둘은 변경 이유가 다르다(퍼시스턴스는 DB 스키마가 바뀔 때, 도메인은 비즈니스 규칙이 바뀔 때 = SRP). 그래서 분리하면 퍼시스턴스 엔티티는 인프라(또는 persistence) 계층, 도메인 모델은 도메인 계층에 1급으로 두고 리포지터리 경계에서 매핑한다. **여러 유스케이스가 공유하는 엔티티(User, Reservation)는 특정 feature 디렉터리에 종속시키지 않고 공유 도메인(shared kernel)에 둔다.** 그러면 auth가 user를 참조해도 디렉터리 종속 문제가 사라진다. 작은 과제라 의도적으로 겸했다는 답(YAGNI)은 유효하지만, 면접관이 원한 한 줄은 **"분리할 줄 안다 + 분리 기준은 변경 이유"**를 또렷이 말하는 것이었다. 되묻기 대신 이 프레임으로 바로 들어갔어야.

### 2. 정책 폭발 = Strategy로 정책을 객체 밖으로 추출 (D의 ★★)

핵심은 **슬롯이 모든 정책을 알지 않게** 만드는 것. 정책을 객체 밖으로 빼는 패턴이 Strategy(또는 정책 객체, 명세 패턴)다. `PromotionPolicy` 인터페이스를 두고 `FifoPromotionPolicy`, `VipPriorityPromotionPolicy`를 구현체로 분리, 슬롯은 정책 목록을 주입받아 **정렬과 필터를 정책에 위임**만 한다. 정책 추가 = 새 클래스(OCP, 기존 슬롯 수정 없음), 슬롯의 관심사 수는 1로 고정된다. checkPermission도 유저 엔티티에 박는 게 아니라 인가 정책이나 도메인 서비스로 뺀다. 이러면 면접관이 정확히 짚은 **관심사 수 = 인지 부하**가 안 늘어난다. 200줄 논쟁의 진짜 답도 **줄 수가 아니라 관심사 수**다. 메서드 분리는 슬롯의 책임을 그대로 둔 채 줄만 옮기니 면접관이 반려한 것이고, 정답은 책임 자체를 빼내는 것. "도메인 객체를 더 만든다"는 방향은 맞았으나 **Strategy로 정책을 떼낸다**는 어휘로 못 찍었다.

## ✅ 잘 작동한 카드 (다음에도 재현)

1. **과제 디펜스 본론** — 승격 FIFO 알고리즘, SQLite 단일 writer 직렬화 vs MySQL SELECT FOR UPDATE, API 키 SHA-256 대 비밀번호 느린 해시 구분. 준비한 깊이가 그대로 나감.
2. **동시성과 분산 가정 구분** — SQLite 직렬화, MySQL 락, 단일 인스턴스 EventEmitter vs 분산 MQ. 가정 조건을 정확히 읽음.
3. **솔직한 약점 인정** — 프레젠테이션 예외 처리 누락, 더 고민해봐야 할 영역을 변명 없이 인정. [[Developer-Interview-Signals]] 솔직함 신호와 정렬.
4. **AI 없이 라이브 코딩 수행** — 더듬었어도 슬랙 샌더를 손으로 추가하고 ConfigService 주입까지 끌고 감.

## ⚠️ 다듬을 점 (감점 가설)

1. **정책 폭발에 Strategy 못 찍음(D)** — 가장 큰 감점 가설. 면접관이 메서드 분리를 직접 반려했는데도 정책 추출(Strategy)로 못 넘어가고 200줄 원칙에 집착. **관심사 수가 진짜 축**임을 놓침.
2. **엔티티 위치에 도메인/퍼시스턴스 분리 못 묶음(C)** — 되묻기만 반복. 변경 이유 기준 분리를 한 문장으로 말했어야.
3. **라이브 코딩이 헤맴(B)** — [[Interview-Prep-TossPlace-1st-Model-Answers|모범답변]] 4번에 MessageBuilder + Sender 전략 + 디스패처라는 깔끔한 버전이 이미 있었는데, 라이브에선 채널 타입 추가와 함수형 클래스 왕복으로 명료함이 안 나옴. 준비한 구조를 라이브 손코딩으로 재현하는 연습 부족.
4. **200줄 원칙 과집착** — 인지 부하의 축을 줄 수로 잡아 오히려 관심사 분리라는 정답을 가림. 다음엔 줄 수가 아니라 책임 수로 프레이밍.

- [x] **리치 도메인 모델의 god object 함정** — 메서드 분리는 책임을 안 줄인다 → [[Strategy패턴이란]] 위 섹션에 흡수(권한 판정도 엔티티에 박지 말고 인가 정책으로).
- [ ] **라이브 손코딩 드릴(미완, 실행 과제)** — 준비한 모범 구조(전략, 어댑터)를 화이트보드 수준으로 재현하는 연습. vault가 아니라 본인 연습. 다음 코딩 동반 면접 전 필수.

## 결정 (확정, 26.06.10)

- **회사 fit**: 강. 본인 1지망. 명확한 기술 문제와 트레이드오프 중심 평가 문화는 [[Self-Analysis|ISTP]]가 잘 맞는 논리와 근거 중심 환경 신호. 막힐 때까지 미는 강도는 거부감이 아니라 매치 쪽으로 읽힌다.
- **결과 체감**: 본인은 끝나고 조졌다는 감각이 컸으나, 토스 스타일은 막히는 지점을 보려는 설계라 깊은 두 probe까지 갔다는 것 자체가 정상 범위. 반반으로 봤고 26.06.10 탈락 통보.
- **피드백 경로 종료**: 피드백 요청 메일에 채용팀이 같은 날 회신 — 내부 가이드상 비공유. 위 감점 가설(D 정책 폭발에 Strategy 못 찍음, C 엔티티 위치)은 외부 확인 없이 자가 진단으로 확정한다. 역반영은 이미 vault에 반영돼 다음 설계 면접 탄약이 됐으니 이 회고는 여기서 닫는다. 트래커 갱신 완료.
- **재지원 여지**: 회신의 추후 더 좋은 기회 문구 + 토스 Node JD는 늘 올라옴([[Interview-Advice-Toss-Mentor]]). 이력 곡선이 우상향(25.08 과제 탈락 → 26.05 과제 통과, 1차 도달)이라 다음 사이클 재도전 가치 있음. 단 재도전 전에 체크리스트 미완 항목인 라이브 손코딩 드릴을 반드시 닫을 것.
