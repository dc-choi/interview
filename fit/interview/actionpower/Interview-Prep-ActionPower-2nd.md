## 0. 면접 전략
> 참고: [[FIT#면접 전략 (대화의 기술)|면접 전략]], [[Interview-Soft-Skills|인성면접 가이드]], [[Common-Interview-Questions|자주하는 면접 질문]]

### 2차 면접의 본질 — 채용은 "같이 일할 사람을 뽑는 활동"
> [[Interview-Soft-Skills#채용의 본질|채용의 본질]]

면접관(대표자 + 인사담당자)이 머릿속으로 던지는 질문:
- 잘 협업할 수 있는 동료인가?
- 그라운드 룰을 지킬 만한 사람인가? (고집세지 않은가, 너무 강한 화법은 아닌가)
- 보편 타당한 사고 방식을 지향하는가?
- 혼자 일하는 것만 좋아하는 사람은 아닌가?
- 개인 플레이를 넘어 이기적인 동료는 아닌가?

→ **모든 답변을 "팀과 함께", "근거 기반으로", "상대 의견 존중"의 톤**으로 통과시키는 것이 핵심.

### 면접관이 보는 5가지 평가축
> [[Interview-Soft-Skills#면접관이 보는 5가지|면접관이 보는 5가지]]

1. **성장 가능성** — 지금보다 나아질 수 있는 사람인가
2. **문제해결능력** — 정의→분석→실행→정량화의 과정이 있는가
3. **현재 실력** — 연차에 부합하는가
4. **적극적인 자기개발 의지** — 방향성에 맞는 학습이 있는가
5. **꾸준한 노력 여부** — 단기 스파이크가 아니라 지속성이 있는가

각 답변에 위 5가지 중 적어도 1~2개에 해당하는 신호를 의식적으로 심을 것.

### 초반 10분이 당락을 결정한다
- 자기소개에서 버벅대면 첫 단추부터 잘못됨 → **업무 중심 자기소개 완벽히 암기하지 말고, 키워드로 자연스럽게**
- 핵심 키워드(이벤트 아키텍처, 정합성, 모니터링, 자동화)를 자기소개에 녹여서 **그 방향으로 질문이 오게 유도**
- 서류가 왜 통과했는지 고민: 1차 기술면접도 합격 → **기술은 검증됨, 2차는 "같이 일할 수 있는 사람인가?"에 집중**

### 메타인지 — 가장 먼저 훈련해야 할 것
> [[Interview-Soft-Skills#메타인지 — 가장 먼저 훈련해야 할 것|메타인지]]

**내가 한 일 vs 팀이 한 일을 명확히 구분할 것**
- 노련한 면접관(대표자 포함)은 꼬리 질문 2~3단계만 내려가도 부풀린 부분을 안다
- 내가 **설계**한 것 / 내가 **구현**한 것 / 팀이 **함께한** 것 / 내가 **관찰**만 한 것을 분리해서 말하기
- 예: "이벤트 아키텍처는 제가 설계와 구현을 주도했고, 모니터링 대시보드 구축은 팀원과 함께 진행했고, 인프라 비용 산정은 시니어가 결정한 부분을 제가 검증한 것입니다"

**내가 추구하는 개발 방식 vs 올바르지 않다고 생각하는 것**
- 방법론은 취향이 아니라 **맥락**으로 설명 — "이 상황에서는 이게 맞고 저 상황에서는 저게 맞다"
- 예: "Pessimistic Lock이 항상 옳은 건 아니지만, IoT 환경의 충돌 빈도가 높은 상황에서는 Optimistic 재시도 비용이 너무 컸다"

### 면접 심리학 — 절실함은 통제권 상실로 보인다
> [[Interview-Soft-Skills#면접 심리학|면접 심리학]]

- **너무 절실하면 오히려 떨어진다**. 절실함은 면접관에게 "이 사람은 통제권이 없다"는 신호로 읽힘
- **여유는 자신감의 신호** — 같은 답변도 톤이 다르면 다르게 평가됨
- 처우 협상 단계에서도 여유 없는 태도는 감점
- 마인드셋: "이 회사가 나에게 맞는지도 내가 본다" — 일방적 평가가 아니라 **상호 검증의 자리**

### 답변 프레임워크: STAR
> 과거의 행동이 미래의 행동을 예측하는 가장 좋은 지표 — 조직 적응력·팀워크·컬처핏을 보는 표준 기법

| 단계 | 내용 | 예시 (Prisma 성능 개선) |
|------|------|----------------------|
| **Situation** | 상황/배경 | API 응답이 1000ms까지 저하, 사용자 불만 증가 |
| **Task** | 해결해야 할 과제 | 원인 파악 후 성능을 정상 수준으로 복구 |
| **Action** | 실행한 행동 | 로그 분석 → 4개 개별쿼리 발견 → 공식 문서 검토 → relationLoadStrategy 적용 |
| **Result** | 정량적 결과 | 82~90% 성능 개선, 하코 3000명 커뮤니티 발표로 지식 공유 |

문제 해결 기법 키워드: **카이젠**(점진 개선), **포카요케**(실수 방지 장치), **5W1H**(누가 무엇을 언제 어디서 왜 어떻게)

### 정답 없는 질문 대처법
- **문제 정의** → 자료수집·분석 → **해결책 모색** → 평가·실행
- 핵심: 완벽한 답이 아니라 **문제를 정의하고 해결책을 찾아가는 과정**을 보여주는 것
- 보편 타당한 가치인지, 나와 회사의 입장이 다른지, 근본적 해결이 가능한지, 대안은 있는지
- **2차 면접도 회사의 테두리 안에서** 질문한다는 것을 잊지 말 것 — 개인 가치관만 강조하지 말고 회사·팀·동료 맥락에서 답할 것

### 마지막 질의응답의 중요성
- 애매한 경우 **질의응답에서 당락이 결정**되는 변수
- 대표자 면접이므로 비전/방향성/문화에 대한 깊이 있는 질문이 좋은 인상
- **오답 정정 기회로 활용 가능** — "오늘 답변 중 추가로 설명드리면 좋을 부분이 있을까요?"라는 질문은 마지막 만회 카드

## 1. 핵심가치 매칭 (2차 면접의 본질)

> 2차 면접은 액션파워의 **3가지 핵심가치(존중·질문·실행)**에 대한 적합도를 확인하는 과정

### 존중 (We Respect) — 서로의 인격과 시간의 가치를 지켜줌

**내 매칭 에피소드**

1. **기획자와 결제 환불 의견 충돌**: 기획자가 "환불 정책은 나중에" 제안 → 일방적으로 반대하지 않고 리스크를 비개발 언어로 설명(사용자 민원, 회계 처리, 기술 부채) → 대안 제시(환불 정책 병렬 진행 + 결제 구조를 환불 고려해 미리 설계) → 일정 1주일 추가됐지만 오픈 후 안정 운영 → **이후 기획자가 먼저 "기술적으로 가능해요?" 물어보는 문화로 변화**
   - 핵심: "기획자가 틀렸다"가 아니라 "함께 더 나은 방법을 찾자"

2. **FIDO 서버 팀원 의견 수용**: 팀원이 오픈소스 라이브러리(SimpleWebAuthn)의 규격 준수 여부 의문 제기 → 리더로서 일정 압박 속에서도 "검증 먼저" 결정 → WebAuthn Spec과 대조해 실제 규격 상이 발견 → GitHub 이슈 생성 → 패치 → 인증 통과
   - 핵심: 팀원의 의문을 "시간 낭비"로 치부하지 않고 존중. **"누가 맞았냐"보다 "함께 해결했다"**

3. **동료 피드백 수용**: "조금 더 너의 주장을 강하게 해도 좋겠다"는 프론트엔드 동료 피드백 → "겸손"과 "자신감 없음"은 다르다는 깨달음 → 데이터 기반 명확한 주장 연습

- 꼬리:
  - "반대 의견을 들었을 때 감정적으로 힘들지 않나?" → 감정적 반응보다 상황 분석 우선. 비난 내용을 사실/의견/공격으로 구분하는 습관. 불편해도 근거가 있으면 수용
  - "팀원과 심각하게 갈등한 경험은?" → 갈등 자체를 피하는 게 아니라, 갈등을 "논리 vs 논리"로 전환하는 게 중요. FIDO 사례가 그 예시

### 질문 (We Question) — 미래를 향해 과감하게 질문하는 문화

**내 매칭 에피소드**

1. **CloudWatch → GPL 전환**: "모니터링이 이게 최선인가?" 질문에서 시작. 기존 CloudWatch+SNS+Lambda 구조의 한계(커스텀 비즈니스 메트릭 비용·고카디널리티 제약, PromQL 수준의 다차원 쿼리 부재, Logs Insights UX 한계, SNS+Lambda로 알림 라우팅·디듀프 수동 구현 부담)를 직접 분석 → 가중치 비교(GPL 4.65 / ELK 3.85 / Datadog 3.35 / CloudWatch 3.10) → GPL 자체 호스팅 결정 → 직접 구축
   - 핵심: "원래 쓰던 거니까"를 그냥 넘기지 않음

2. **Prisma 성능 문제 발견**: "왜 API가 느리지?" → 로그에서 4개 개별쿼리 직접 발견 → Prisma가 app-level join을 한다는 사실 확인 → 공식 문서에서 relationLoadStrategy: 'join' 발견 → 82~90% 성능 개선 → 하코 3000명 커뮤니티에서 발표
   - 핵심: ORM이 만들어주는 쿼리를 "당연한 것"으로 받아들이지 않고 의문을 품음

3. **REST→tRPC 전환 (사이드 프로젝트)**: "프론트/백 간 타입 불일치가 런타임에서야 발견되는 게 정상인가?" → tRPC로 전환하여 E2E 타입 안전성 확보 → 30개 단체 실서비스에서 타입 관련 버그 제로

- 꼬리:
  - "질문만 하고 실행을 안 하면 의미 없지 않나?" → 맞다. 질문이 의미 있으려면 검증→실행→결과까지 가야 함. 위 사례 모두 "질문 → 분석 → 실행 → 정량적 결과"로 이어짐
  - "잘못된 질문, 삽질한 경험은?" → 모든 질문이 성공하진 않음. 중요한 건 **빠르게 실패하고 빠르게 되돌리는 것**. 프로토타입 단계에서 걸러내면 프로덕션 리스크 최소화

### 실행 (We Act) — "작게 여러 번" 실행하는 철학

**내 매칭 에피소드**

1. **아키텍처 전환 3단계 점진적 실행**: 단일 EC2 → 스케일링 아키텍처 전환을 한 번에 하지 않고 3단계로 분리
   - 1단계: 컨테이너화 + LB 분리 (안정화 확인)
   - 2단계: 이벤트 기반 아키텍처 전환 (안정화 확인)
   - 3단계: DB 읽기/쓰기 분리
   - 핵심: "작게 여러 번" — 각 단계에서 안정화 확인 후 다음 단계 진행. 빅뱅 전환 대신 점진적 실행

2. **테스트 0→70% 도입**: 시솔지주에서 테스트 코드가 전혀 없는 상태 → 한 번에 100% 목표가 아니라, SonarQube 대시보드 도입 → PR 연동 60% 게이트 → 최종 70% 달성. 단계적으로 팀 문화를 바꿈

3. **출석부 프로젝트 발전**: 2022년 시작 → JS→TS 마이그레이션 → REST→tRPC 전환 → React 19 업그레이드 → 모노레포 구조화. 4년에 걸쳐 작은 개선을 반복하며 30개 단체 실서비스로 성장

- 꼬리:
  - "실패한 실행은?" → 이썸테크 6개월. 빠르게 판단하고 움직인 건 좋았지만, 입사 전 더 깊이 확인했어야 했음. 이후 "내가 원하는 환경"을 명확히 정의하는 계기가 됨
  - "급한 일과 중요한 일이 동시에 오면?" → 비즈니스 임팩트 기준 우선순위. 사용자 직접 영향 > 빈도 > 복구 시간. 혼자 해결 어려우면 바로 도움 요청

## 2. 인성/소프트스킬 예상 질문

### 자기소개

> 구조: 이름/역할 → 현재 회사 업무 → 핵심 성과 → 일하는 방식 → 왜 액션파워

안녕하세요, 4년차 백엔드 개발자 최동철입니다.

현재 트라이포드랩에서 NestJS 기반 VMI 서비스의 백엔드를 설계하고 운영하고 있습니다. 수천 대 IoT 기기가 동시에 재고 데이터를 전송하는 환경에서 DB Lock으로 데이터 정합성을 확보하고, EventBridge와 SQS 기반 이벤트 아키텍처를 설계해 수기 발주 프로세스를 완전 자동화했습니다. 단일 서버 구조를 CloudFront+ECS 아키텍처로 전환하고, Grafana 기반 모니터링 인프라도 직접 구축했습니다. 이 과정에서 슬로우 쿼리 99.3% 개선, API 성능 90% 향상, 재고관리 시간 95.8% 절감의 성과를 만들었습니다.

저는 문제를 발견하면 "원래 그런 거"로 넘기지 못하는 성격입니다. 동시에 혼자 해결하는 것보다 팀과 함께 해결해 나가는 것을 중요하게 생각합니다. 액션파워의 "존중하면서 질문하고, 작게 여러 번 실행하는" 문화가 제가 일해온 방식과 잘 맞다고 느꼈습니다.

### 왜 액션파워인가? (2차 버전)

- 1차 면접을 통해 팀의 분위기와 기술적 수준을 확인할 수 있었음
- 단순히 AI 서비스를 만드는 회사가 아니라, **"꿈꿔왔던 편리함을 가능하게"**라는 미션 아래 사용자 경험을 진지하게 고민하는 팀이라는 인상
- 특히 "작게 여러 번 실행하는" 철학이 내가 아키텍처 전환을 3단계로 나눠 점진적으로 진행한 경험과 일치
- "동료와 함께 성장하는 문화" — 하코 커뮤니티 발표, 카카오테크 멘토링 등 지식 공유를 자발적으로 해온 사람으로서 이 문화에 자연스럽게 기여할 수 있음

### 이직 사유 (2차 버전 — 방어적이지 않게)

- 트라이포드랩에서 0→1 구축, PoC 성공, 아키텍처 전환, 모니터링 구축까지 충분히 성장
- 다음 성장 단계: B2B PoC의 수개월 단위 피드백 사이클 → B2C의 빠른 피드백으로 개선 역량을 키우고 싶음
- 다글로는 B2B2C — 내가 가진 B2B 정합성 경험을 살리면서 B2C의 빠른 피드백도 경험할 수 있는 드문 환경
- 꼬리:
  - "피드백 느려지면 또 이직?" → 이직 기준은 속도가 아니라 **성장 기회**. 다글로에서 B2B2C 양쪽을 경험하는 것 자체가 장기적 성장
  - "B2B는 더 이상 싫은 건가?" → 값진 경험이지만 지금 커리어 단계에서는 빠른 개선 경험이 필요. 다글로는 B2B도 함께하니 경험을 버리는 게 아님

### 이전 회사 퇴사 사유

**이썸테크 (6개월)**: 프로젝트 단위 개발이라 한 서비스를 장기적으로 개선하는 경험을 쌓기 어려워 제품 중심 조직으로 이직 결심
- 꼬리: "6개월이면 너무 빠른 결정 아닌가?" → 3개월간 열심히 배웠지만 방향이 아니라는 확신 → 이후 시솔지주 1년, 트라이포드랩 1년 8개월 안정 근무

**시솔지주 (1년, 권고사직)**: FIDO 인증 성공시키고 백엔드 팀 리드했지만, 시장 반응 부진으로 회사가 사업 종료 → 팀 전체 정리
- 꼬리: "개인 역량 부족?" → 팀 전체 정리. 오히려 FIDO 인증 성공이 가장 큰 성과

### 장단점

- **장점: 실행력**. "이거 해야겠다" → 바로 움직임. Prisma 문제 발견 즉시 공식 문서 찾아 해결, 테스트 없는 환경에서 주저 없이 0→1 도입, 모니터링 필요성 제기→직접 구축
- **단점: 실행 판단이 빠른 만큼 성급할 때가 있음**. 충분히 검토하기 전에 움직이는 경향
- 개선: 중요한 결정 전 "이게 정말 최선인가?" 한 번 더 생각. 급할수록 우선순위 먼저 정리. 실행 전 검증 단계를 의식적으로 추가

### 동료 갈등 경험 / 같이 일하기 힘든 동료

- 기술만 잘하는데 협업 안 되는 사람, 커뮤니케이션은 좋은데 실력이 따라오지 않는 사람, 책임을 회피하는 사람 — **기술·커뮤니케이션·책임감의 균형**이 중요
- 액션파워 백엔드 팀 인터뷰에서 최환석님이 "실력이나 역량보다 **진실되고 상대방을 존중하는 배려의 마음**"을 강조한 것에 깊이 공감
- 꼬리:
  - "그런 동료와 일하게 되면?" → 직접 대화로 해결 시도. 구체적 사실 기반으로 피드백 ("이 부분이 이렇게 되면 팀에 이런 영향이 있다"). 개인 성향 비난 아님
  - "그래도 안 바뀌면?" → 팀의 그라운드 룰로 해결. 일하는 방식에 대한 명확한 기준이 있으면 개인 감정이 아니라 팀 합의로 문제를 풀 수 있음. 트라이포드랩에서도 코드 리뷰 규칙, PR 머지 기준을 팀 합의로 정해서 운영

### 소통 방식

- **기획자**: 기획 의도를 먼저 물어보고 이해 확인. 기술 용어는 풀어서 설명. "이런 게 가능하다"를 알려줌
- **프론트엔드**: API 문서를 자세하게 작성 (Request/Response 예시, 에러 케이스). "문서가 곧 소통". 에러 처리 방식 통일
- **백엔드 동료**: 의견 충돌은 감정이 아닌 논리로. 근거 기반 검증 후 함께 해결
- 비개발자에게 기술 설명: 개념 먼저 → 예시 → 비유 순서. 실제 면접에서 스택(Stack)을 인사담당자에게 설명하여 집중시킨 경험

### 성장 목표

**5년 후**: 개발 리드. 코드를 잘 짜는 것을 넘어 팀의 기술 방향을 제시하고 주니어 성장을 돕는 역할
- Staff Engineer 지향 — 코드 직접 짜면서 팀 기술 방향 제시 + 멘토링

**최종 목표**: CTO (10~15년 후). 기술적 깊이(3~5년) → 리드(5~8년) → 큰 조직(8~10년)
- 꼬리: "회사 이용만 하고 나가겠네?" → CTO가 될 만한 사람으로 성장하고 싶은 것. 함께 성장 가능하면 오래 함께할 것
- **"권한"이 아니라 "영향력"과 "기여의 크기"로 표현**

### 함께 성장하는 개발 문화를 어떻게 만드는가

- "먼저 공유하는 사람"이 되기: 하코 3,500명 커뮤니티 발표, 카카오테크 캠퍼스 멘토, 사내 트러블슈팅 노트 정리
- 코드 리뷰: "이게 틀렸다"가 아니라 "왜 이렇게 했는지" 먼저 물어보고 "이런 방법도 있다" 제안
- 액션파워 문화의 **"깨달은 점이나 도움 될 만한 것들을 정리하여 공유"**와 직접 일치
- 액션파워의 매월/매분기 회고 체계 — 트라이포드랩에서도 스프린트 회고를 진행했고, **기록으로 남기는 습관**을 통해 같은 실수 반복 방지

### 어떻게 회고하고 어떻게 성장하시나요? (액션파워 회고 문화 매칭)

> [[Retrospective-Methods|회고 기법]] — 액션파워의 매월/매분기 회고와 직접 연결되는 핵심 질문

- **회고 없는 깨달음은 자기만족** — "오늘 당장 뭘 할 건가"로 구체화되지 않으면 같은 실수가 반복됨
- 상황별로 다른 프레임을 사용:
  - **주간 단위**: KPT (Keep / Problem / Try) — 문제 정의 단계가 명시되어 있어 원인 분석에 강함
  - **스프린트/릴리즈 직후**: CSS (Continue / Stop / Start) — 액션 아이템 즉시 도출
  - **프로젝트 중간 점검**: 4L (Liked / Learned / Lacked / Longed For) — 감정·니즈까지 포함
  - **분기/연간 커리어 재설정**: WIW (What I want — 한 일/배운 것/해야 할 것)
  - **매일 학습 기록**: TIL (Event / Emotion / Learning / Action)
- 실제 적용 사례:
  - 트라이포드랩 스프린트 회고에서 "Prisma 성능 이슈가 반복되는 이유"를 KPT로 정의 → ORM 동작 원리를 깊이 학습 → 같은 패턴 재발 방지
  - 분기 회고(WIW)에서 "B2B 피드백 사이클이 너무 길다"는 인식 → 다음 성장 단계로 B2B2C 환경 탐색 → 액션파워 지원으로 이어짐
- **회고의 5가지 공통 원칙**: 구체적 행동으로 끝낼 것 / 정직할 것 / 기록으로 남길 것 / 타인 피드백 포함 / 행동→피드백→조정 루프
- 액션파워의 매월/매분기 회고에 자연스럽게 합류 가능 — 그라운드룰 안에서 협업하는 데 익숙함

- 꼬리:
  - "회고가 형식적으로 끝나는 경우는 없었나?" → 있었다. 그래서 회고 끝에 반드시 **"이번 주에 누가 무엇을 한다"**까지 적게 함. 액션 아이템에 담당자와 데드라인이 없으면 회고는 의미가 없음
  - "혼자 회고할 때는 어떻게?" → 주간 KPT를 노트에 기록. 분기말에는 WIW로 커리어 방향 점검. 면접 직전에도 WIW로 자기 객관화

### 리더십 발휘 경험

> 공식 리더가 아니어도 가능 — 의사결정 과정 + 팀원 설득 + 결과 책임의 3박자

- **시솔지주 FIDO 백엔드 리드 (1년)**:
  - 인증 통과 일정 압박 + 팀원이 SimpleWebAuthn 라이브러리 규격 의문 제기 → "검증 먼저" 결정 → WebAuthn Spec 대조 → GitHub 이슈 → 패치 → 인증 통과
  - 리더십 포인트: **일정 압박 속에서도 팀원의 의문을 존중**, 결정 → 책임 → 결과까지 끌고 감
  - 사후: "팀원의 의문이 결정적이었다"고 명시적으로 공유 — 다음에도 의문 제기가 환영받는 분위기 형성
- **트라이포드랩 코드 리뷰 문화 정착**: 공식 리더 아니지만 **PR 머지 기준, 코드 리뷰 그라운드룰을 팀 합의로 정착**시킴. 개인 감정이 아니라 팀 합의로 충돌을 푸는 구조 마련
- 리더십 철학: **서번트 리더십** — 팀원의 장애물을 제거하고 방향을 제시하는 역할. 권한이 아니라 영향력으로 표현
- 꼬리:
  - "공식 리더가 아닌데 어떻게 팀원을 설득했나?" → 권한이 없을 때일수록 **데이터와 근거**가 무기. PR 머지 기준 도입 시에도 "이번 분기 핫픽스 비율 N%, 그 중 절반이 리뷰 미흡"을 데이터로 제시
  - "갈등이 있을 때 리더로서 어떻게 결정했나?" → 결정권자가 결정하되, 결정 전까지는 모든 의견이 동등하게 들리는 자리를 만드는 것이 리더의 역할

### 5~6명 소규모 팀 리드를 가정한다면, 팀 성장을 어떻게 도모하시겠습니까?

> 시니어로 가는 길에 대한 고민을 보는 질문 — 액션파워 다음 단계 성장 비전과 연결

1. **기술 성장**: 코드 리뷰 문화, 정기 스터디(예: Node.js Design Patterns), RFC 문화로 큰 결정의 맥락을 문서화, 기술 부채 우선순위화
2. **업무 성장**: 1on1으로 개인 목표와 팀 목표 정렬, 회고를 정기 리듬으로 (주간 KPT + 분기 WIW), 성과를 정량으로 측정
3. **심리적 안전감**: 실패를 비난 없이 기록·공유(blameless postmortem), 질문을 환영하는 분위기, "모르겠다"를 말할 수 있는 문화
4. **자율성과 책임**: 작은 의사결정은 팀원에게 위임, 작은 성공 경험을 누적, **자기 조직화** 지향
5. **리더의 역할**: 외부 노이즈 차단(unblock), 방향 제시, 팀의 시간을 지켜주는 것
- 트라이포드랩에서 공식 리더는 아니지만 **PR 머지 기준·코드 리뷰 그라운드룰을 팀 합의로 정착**시킨 경험이 가장 가까운 사례
- 꼬리:
  - "5~6명이면 1on1이 부담스럽지 않나?" → 격주 30분이면 충분. 길이보다 **꾸준함**이 중요
  - "성과가 안 나오는 팀원은?" → 비난이 아니라 **시스템 점검**이 먼저. 작업 분배가 잘못됐는지, 컨텍스트 공유가 부족했는지, 학습 여건이 부족한지 — 회고와 1on1으로 원인부터 분리

### 가장 큰 보람을 느낀 순간 / 고객 만족 사례

- **트라이포드랩 수기 발주 자동화**: 운영팀이 매일 한 시간씩 수기로 작성하던 발주 프로세스를 EventBridge+SQS 기반으로 완전 자동화 → 운영팀에게 "이제 다른 일을 할 시간이 생겼다"는 피드백
  - 보람 포인트: **B2B에서 사용자(운영팀)가 명시적으로 감사**한 드문 케이스
- **하코 커뮤니티 발표**: Prisma 성능 개선 발표 후 "같은 문제를 겪고 있었는데 해결책을 알려줘서 고맙다"는 DM을 여러 명에게 받음
  - 보람 포인트: 내가 해결한 것이 **나만의 문제가 아니었다**는 검증
- 가치관 연결: **내 작업이 다른 사람의 시간을 아껴주는 것**이 가장 큰 동기. 액션파워 미션 "꿈꿔왔던 편리함을 가능하게"와 같은 결의 가치
- 꼬리:
  - "B2C는 경험 없는데?" → B2C 고객 직접 피드백은 없지만, 사내 사용자(운영팀)·커뮤니티 사용자에게 피드백을 받은 경험은 있음. 다글로의 B2C 환경에서 **사용자 직접 피드백을 받는 경험을 키우고 싶음**

### 커리어 목표를 달성하지 못한 경험 / 아쉬웠던 결정

- **이썸테크 6개월 퇴사**: "프로젝트 단위가 아닌 제품 단위 성장"이라는 목표를 가지고 입사했지만, **입사 전 충분히 환경을 검증하지 못함**. 빠르게 판단하고 움직인 건 좋았지만 사전 조사가 부족
- 그 사이에 채운 것:
  - **회사 검증 체크리스트** 정립 — 채용 공고/팀 분위기/성장 환경/도메인 규모를 사전에 정량적으로 점검
  - 시솔지주 1년, 트라이포드랩 1년 8개월의 안정 근무로 패턴 교정
  - 액션파워 지원도 **JD·포트폴리오·1차 면접 분위기를 종합 검증한 결과**
- 변하지 않은 목표: "제품 단위로 장기 성장하고 싶다" — 액션파워 다글로는 이 목표에 정확히 부합
- 꼬리:
  - "지금이라면 같은 실수 안 하나?" → 100% 보장은 못 함. 다만 **사전 검증 단계가 훨씬 조밀해짐**. 1차 면접에서 팀 분위기를 직접 확인한 것도 같은 맥락

### 메타인지 — 내가 잘 모르는 것 / 부족한 영역

> 솔직한 자기 객관화를 보는 질문. 거짓말로 회피하면 신뢰가 무너짐

- **GCP 운영 경험 없음** — AWS 중심으로 일해왔음. 다만 클라우드의 핵심 개념(IAM, VPC, 오브젝트 스토리지, 메시징, 컨테이너 오케스트레이션)은 이전 가능. 입사 후 1~2주 안에 GCP 콘솔/문서로 매핑 학습할 계획
- **대규모 ML/AI 파이프라인 직접 운영 경험 없음** — STT/번역 모델 자체를 다뤄본 적은 없음. 다만 비동기 Job 패턴, 큐 기반 워크플로우, 모니터링은 트라이포드랩에서 동일한 설계 원칙으로 적용해본 경험
- **프론트엔드 심도** — TypeScript/React를 사이드 프로젝트로 다루지만 본업은 백엔드. 백엔드와의 접점(API 계약, 타입 안전성, 인증)은 깊지만 UI/UX 깊이는 부족
- **부족함을 인정하는 방식**: 회피가 아니라 **"무엇을 어떻게 채울지"까지 함께 말하기**. 액션파워의 "바르게 실패하는 경험" 문화와 정렬

### 번아웃 관리

- 신호 캐치: 코드 짜는 게 재미없어짐, 작은 일에 짜증, 집중력 저하
- 원인 구분: 육체적 피로(→ 휴가, 운동) vs 정신적 피로(→ 업무 프로세스 개선, 새로운 도전)
- "통제 불가능한 것(면접 결과) 집착 → 번아웃" vs "통제 가능한 것(실력 쌓기) 집중 → 회복"

### 이상적인 회사 / 조직문화

- 회사는 공동의 목적을 가진 단체. 그 목적을 향해 달려가되, 소통으로 해결해야 함
- 수평적인 조직에서 일했고, 직급 없이 "-님" 호칭 사용 → 액션파워도 동일
- 과거에는 윗사람 앞에서 주저한 적이 있었지만, 동료 피드백 후 근거가 있으면 당당하게 말하는 연습을 해옴

### 자기계발

- **기술**: 실무 문제 해결 과정에서 깊이 파기 (Prisma 성능 → ORM 내부 동작 이해, 슬로우 쿼리 → 인덱스·실행 계획 심화)
- **커뮤니티**: 하코 3,500명 발표, 카카오테크 캠퍼스 멘토 — 발표 준비 과정에서 기술을 더 깊이 파고, 질의응답에서 다른 관점 획득
- **AI 도구 활용**: Claude Code를 일상적으로 사용. 출석부 프로젝트에서 AI 기반 Spec-Driven Development 워크플로우 도입
- 꼬리: "최근 관심 있는 기술?" → AI Harness Engineering — AI 도구를 단순히 사용하는 것을 넘어, AI가 일관된 품질로 동작하도록 **워크플로우를 설계하고 자동화하는 것**에 관심. 실제로 Claude Code에 커스텀 스킬(면접 준비 문서 자동 생성, 메모 정리 등), 훅(코드 변경 시 자동 검증), MCP 서버(외부 도구 연동)를 구축해서 개발 생산성을 높이고 있음. AI 모델 자체의 성능보다 **AI를 감싸는 하네스(프롬프트 설계, 도구 연동, 품질 관리)**가 실제 프로덕션에서 더 중요하다고 생각. 다글로처럼 AI 기능을 서비스로 제공하는 회사에서 이 관점이 백엔드 안정성과 직접 연결됨

### 좋은 코드, 견고한 소프트웨어가 무엇인가?

> 개발자로서 성장을 지속하는지, 스스로 고민을 해봤는지 확인하는 질문

- **좋은 코드**: 다음 사람이 읽었을 때 의도가 명확한 코드. 과도한 추상화보다 **"지금 필요한 만큼의 복잡도"**가 핵심
- **견고한 소프트웨어**: 실패를 전제로 설계된 시스템. 100% 장애 예방은 불가능하므로 **빠른 감지 + 빠른 복구**가 더 현실적
- 실무 사례: EventBridge+SQS 이벤트 아키텍처에서 DLQ + 멱등성 키 + 상태 머신으로 "실패해도 복구 가능한 구조" 설계. GPL 모니터링으로 SLO 기반 감지 체계 구축
- 꼬리: "기술부채는 어떻게 관리?" → 일정과 품질의 트레이드오프를 인정. 액션파워의 "3가지 버전(동작하는/안정적인/최적화된)" 접근법에 공감. 먼저 동작하게 만들고, 점진적으로 개선. 단, 히스토리 관리와 기록이 핵심 — 같은 실수를 반복하지 않기 위해

### 동료 코드에서 반복적으로 버그가 발생하면?

> 실수를 방지하기 위한 제도적 장치에 대한 고민을 보는 질문

- 개인을 탓하지 않고 **시스템으로 해결**하는 접근:
  1. **코드 리뷰 강화**: 리뷰 체크리스트 도입, 비즈니스 크리티컬 경로는 2인 이상 리뷰
  2. **테스트 게이트**: PR 머지 시 커버리지 기준 미달이면 차단 (시솔지주에서 SonarQube+60% 게이트 도입한 경험)
  3. **CI/CD 자동 검증**: 빌드+테스트+린트를 배포 전 자동 실행
  4. **포스트모템**: 비난 없이(blameless) 원인 분석 → 재발 방지 액션 아이템 도출
- 액션파워의 "바르게 실패하는 경험" 문화와 연결 — 실패에 대한 책임을 묻기보다 경험을 바탕으로 문제를 재정의

### 새 기술 도입 vs 일정 부족, 어떻게 설득?

> 기술·품질·일정 간의 트레이드오프 조정 능력을 보는 질문

- **원칙**: 기술 자체를 쫓는 게 아니라 "이 단계의 서비스에 필요한가?"로 판단
- **설득 방법**: 감이 아니라 **정량적 비교**로 설득
  - 실무 사례: MSK vs EventBridge+SQS 도입 시 비용($574 vs $0~18), 운영 복잡도, 현재 규모를 정량 비교하여 팀에 공유 → SQS 선택
  - 또 다른 사례: CloudWatch vs GPL 전환 시 가중치 비교표(TCO, 메트릭 생태계, 벤더 종속 등 5개 축)로 의사결정
- **일정이 부족하면**: 프로토타입(1~2일)으로 빠르게 검증 → 효과가 입증되면 다음 스프린트에 정식 도입 제안. 입증 안 되면 빠르게 포기
- 액션파워의 "작게 여러 번 실행" 철학과 일치

### 프로젝트에서 기여 역량과 동료에게 바라는 역량

> 자기 객관화와 업무에서 중요하게 생각하는 가치 파악

- **내 기여 역량**: 문제를 발견하면 정의→분석→실행→정량화까지 끌고 가는 실행력. Prisma 성능 문제, 모니터링 인프라, 아키텍처 전환 모두 이 패턴
- **동료에게 바라는 역량**: 완벽한 기술력보다 **솔직한 소통**. "모르겠다", "이거 이상한데?" 라고 말할 수 있는 용기. FIDO 사례에서 팀원의 의문 제기가 인증 통과의 결정적 계기였음
- **부족한 역량**: 때로 실행이 빨라서 팀원과 충분히 맥락을 공유하기 전에 움직이는 경향. 의식적으로 "왜 이렇게 하려는지" 먼저 설명하는 연습 중

### 마지막 한마디

- 1차 면접을 통해 느낀 점을 짧게 언급 (기술 면접의 질문 깊이, 팀 분위기 등)
- "주어진 역할에만 머무르기보다, 팀이 더 안전하게 성장할 수 있는 구조를 함께 고민하는 동료가 되고 싶다"
- 액션파워의 미션 "꿈꿔왔던 편리함을 가능하게"에 공감하며, 그 편리함을 **안정적으로 지탱하는 백엔드**를 만들고 싶음

## 3. 그라운드룰·리더십 철학 (액션파워 "자율과 책임" 매칭)

> 컬처핏 면접의 핵심 — 이 회사의 일하는 방식과 내가 동기화될 수 있는가
> [[Interview-Soft-Skills#컬처핏·그라운드룰·리더십|컬처핏·그라운드룰·리더십]]

### 그라운드룰을 어떻게 생각하는가

- **그라운드룰 = 팀 안에서 협력·소통하는 일하는 방식의 명확한 기준**
- 개인 감정이 아니라 팀 합의로 충돌을 푸는 장치 — 누가 옳고 그르냐가 아니라 "우리는 이렇게 합의했다"가 기준
- **구체적인 사례가 없으면 그라운드룰은 의미가 없음** — 추상적 원칙보다 실제 운영 사례가 더 중요
- 내 사례:
  - 트라이포드랩에서 **PR 머지 기준** (리뷰어 1인 이상 + 테스트 통과 + 컨벤션 체크)을 팀 합의로 도입
  - **코드 리뷰 톤** 그라운드룰: "이게 틀렸다" 금지, "왜 이렇게 했는지 먼저 묻기" → 감정 충돌 감소
  - 시솔지주에서 **테스트 커버리지 60% 게이트** 도입 — SonarQube + PR 차단으로 자동 강제
- 액션파워 매칭: 매월/매분기 회고 + "자율과 책임" 문화는 그라운드룰이 잘 작동하는 조직의 특징. 입사 후 기존 그라운드룰을 빠르게 학습하고, 부족한 부분이 있다면 회고를 통해 함께 개선 제안

### 어떤 리더십을 추구하는가

- **서번트 리더십** — 권한이 아니라 영향력. 팀원의 장애물을 제거하고 방향을 제시. 본인이 직접 코드를 짜는 것이 아니라, 팀이 더 잘할 수 있게 만드는 것
- **자기 조직화** — 조직이 스스로 구조와 프로세스를 만들고 관리. 자율, 유연, 창의, 책임, 연대의 5가지 축
  - 액션파워의 "자율과 책임" 문화와 정확히 일치
- **그로스 마인드** — 긍정적이고 발전적인 마인드, 도전 정신, 피드백을 통한 지속적 성장. 정해진 답이 아니라 **함께 답을 만드는 자세**
- 내가 추구하는 모습: **Staff Engineer 지향** — 코드를 직접 짜면서도 팀의 기술 방향을 제시하고 멘토링하는 역할. 5년 후 모습이자 액션파워에서 만들고 싶은 영향력의 형태

### 같이 일하기 좋은 동료의 기준

- **기술·커뮤니케이션·책임감의 균형** — 한 가지만 잘하는 사람보다 셋이 균형 잡힌 사람
- **솔직한 소통** — "모르겠다", "이거 이상한데?"를 말할 수 있는 용기. FIDO 사례에서 팀원의 의문 제기가 인증 통과의 결정적 계기였음
- **태도** — 액션파워 백엔드 팀 인터뷰(최환석님)의 **"실력보다 진실되고 상대방을 존중하는 배려의 마음"**과 깊이 공감
- 꼬리:
  - "그런 동료가 아니면 같이 일하기 어렵나?" → 처음부터 모든 사람이 그럴 수는 없음. 함께 그라운드룰을 만들고, 회고로 점검하고, 1on1으로 맞춰가는 과정이 필요. **"좋은 동료"는 발견하는 것이 아니라 함께 만드는 것**

## 4. 기술 리마인드 (2차에서 기술 질문이 섞일 수 있음)

> 1차 면접 내용을 빠르게 복기하는 용도. 상세 내용은 [[Interview-Prep-ActionPower|1차 문서]] 참고

### 핵심 성과 요약 (숫자로 기억)

| 성과              | 수치                     |
| --------------- | ---------------------- |
| 슬로우 쿼리 개선       | 15.4ms → 0.1ms (99.3%) |
| Prisma API 성능   | 100ms → 10ms (82~90%)  |
| Docker 이미지 경량화  | 909MB → 513MB (43.6%)  |
| 배포 시간 단축        | 3분10초 → 2분20초 (26.3%)  |
| 수기 재고관리 절감      | 4시간 → 10분 (95.8%)      |
| 수기 발주           | 1시간 → 완전 자동화           |
| Read Replica 효과 | 조회 40%↑, DB CPU 30%↓   |
| 테스트 커버리지        | 0% → 70%               |
| MSK vs SQS 비용   | $574/월 → $0~18/월       |

### 기술 의사결정 핵심 (왜 그 선택?)

| 의사결정 | 선택 | 이유 |
|---------|------|------|
| 동시성 제어 | Pessimistic Lock (FOR UPDATE NO WAIT) | IoT 충돌 빈도 높음 → Optimistic 재시도 비용 과도 |
| 메시지 큐 | EventBridge+SQS (Kafka 아님) | 월 10만 발주 규모에서 MSK $574 vs SQS $0~18. 실시간 불필요 |
| 모니터링 | GPL 자체 호스팅 (Datadog 아님) | 가중치 비교 4.65점 최고. TCO+벤더 독립+메트릭 생태계 |
| 아키텍처 전환 | 3단계 점진적 (빅뱅 아님) | 서비스 운영 중 — 각 단계 안정화 확인 후 다음 진행 |
| ORM | Prisma (TypeORM 아님) | 스키마 중심 설계+마이그레이션이 강점. 복잡 쿼리는 Raw/Kysely 보완 |

### 다글로 연결 포인트

- **대규모 트래픽**: IoT 수천 대 동시 요청 → 다글로 180만 사용자
- **비동기 처리**: 발주 이벤트 아키텍처 → STT/번역 비동기 Job 패턴
- **자동화**: 수기 발주 완전 자동화 → "엔지니어 개입 최소화" JD 연결
- **모니터링**: GPL 스택 직접 구축 → 대규모 AI 서비스 관측성

## 5. 역질문 (나 → 면접관) — 2차용

> 대표자 + 인사담당자 대상이므로 **조직·문화·비전** 중심

### 조직/문화
1. **"액션파워의 핵심가치 중 '질문'이 인상적이었는데, 실제로 팀원이 기존 방식에 의문을 제기했을 때 어떻게 의사결정이 이루어지나요?"** — 핵심가치의 실제 작동 방식 확인
2. **"매월/매분기 회고를 하신다고 들었는데, 회고 결과가 실제로 프로세스 변화로 이어진 사례가 있나요?"** — 회고가 형식인지 실질적인지 확인
3. **"'자율과 책임' 문화에서 개발 일정의 3가지 버전(동작하는/안정적인/최적화된) 중 어떤 버전을 주로 선택하게 되나요?"** — 현실적인 일정 운영 방식

### 성장/기대치
4. **"주니어 백엔드 개발자에게 초기 3~6개월간 기대하는 역할과 성과는?"** — 현실적 기대치
5. **"백엔드 팀 내에서 기술 공유나 학습 문화는 어떤 형태로 운영되나요?"** — 함께 성장하는 문화 실체 확인
6. **"다글로의 B2B 엔터프라이즈 확장이 올해 팀에 어떤 변화를 가져올 예정인가요?"** — 비즈니스 방향 이해

### 비전
7. **"대표님이 생각하시는 액션파워의 3년 후 모습은?"** — 장기 비전 + 내가 기여할 포인트 파악
8. **"수습 3개월 기간 중 평가 기준은?"** — 반드시 확인

### 마무리 카드 (오답 정정 / 마지막 어필)
> [[Common-Interview-Questions#Q&A 시간 활용하기|Q&A 시간 활용하기]]

9. **"오늘 제 답변 중 추가로 설명드리면 좋을 부분이 있을까요?"** — **오답 정정 마지막 카드**. 1차에서 못 풀었거나 모호하게 답했다고 느낀 질문이 있으면 마지막에 만회 가능
10. **"합격한다면 입사 전에 미리 공부하거나 준비할 것이 있을까요?"** — 적극성 어필 + 기대치 사전 정렬

### 피해야 할 역질문
- 홈페이지/Notion만 봐도 나오는 질문 ("회사 창업이 언제인가요?")
- 연봉/복지만 반복 (마지막 단계로 분리)
- 부정적 프레임 ("야근이 많다던데 사실인가요?")

### 주의사항
> [[FIT#면접 현장 주의사항|면접 현장 주의사항]] 참고

- **"성급함" 단점 → 기술적 실행 판단 예시로 한정** (이썸테크 6개월 퇴사와 연결하지 않기)
- **CTO 목표 → "권한"이 아니라 "영향력"과 "기여의 크기"로 표현**
- **GCP 경험 없음을 방어적으로 말하지 않기** → "AWS에서 동일한 패턴을 경험했고, 클라우드 개념은 이전 가능"
- **2차는 컬처핏 면접** — 기술 답변도 "팀과 함께", "소통", "공유"의 맥락으로 풀기
- **대표자가 면접관** — 비전/방향성 질문에 대비. 회사의 미션("꿈꿔왔던 편리함을 가능하게")을 자연스럽게 언급
- **STAR 프레임워크 활용** — 에피소드를 말할 때 상황→과제→행동→결과 순서로 구조화. 특히 **결과는 정량적으로**
- **외워서 말하지 않기** — 키워드 중심으로 자연스럽게. 동작 방식, 문제 해결 경험을 정리해두고 말로 설명하는 훈련
- **질문 유도** — 자기소개에 핵심 키워드를 녹여서 면접관이 그 방향으로 질문하게 유도. 이력서 기반 토론이 가능한 수준으로 준비
- **액션파워 백엔드 팀 인터뷰(최환석님) 내용 참고** — "자유로우면서도 따뜻한" 분위기, "실력보다 진실되고 존중하는 배려의 마음" 강조
- 1차 면접에서 나온 질문/분위기를 바탕으로 2차 답변을 조율할 것 (1차 경험 메모 필요)
- **부풀리기 절대 금지** — 노련한 면접관(특히 대표자)은 꼬리 질문 2~3단계만 내려가도 안다. **내가 한 일 / 팀 일 / 관찰만 한 일**을 항상 분리
- **너무 절실해 보이지 말 것** — 절실함은 통제권 상실로 보임. 여유는 자신감으로 보임. **상호 검증의 자리**라는 마인드 유지
- **메타인지 답변 준비** — "잘 모르는 것"을 솔직히 말할 것. 단, **"무엇을 어떻게 채울지"까지 함께 말하기**
- **회사의 테두리 안에서 답하기** — 개인 가치관만 강조하지 말고 "회사·팀·동료의 맥락에서" 답할 것
- **마지막 카드(Q&A 오답 정정)를 준비할 것** — 면접 중 못 풀었다고 느낀 질문이 있으면 "오늘 답변 중 추가로 설명드릴 부분이 있을까요?"로 만회

### 면접 직전·직후 루틴
> [[Retrospective-Methods|회고 기법]]

**직전 (당일 아침)**
- WIW 회고 1장 (10분): "내가 뭘 했는가 / 배운 것 / 해야 할 것" — 자기 객관화
- 핵심 성과 숫자 10개, 기술 의사결정 5개 "왜?" 한 줄 답변 빠르게 한 번 읽기
- 호흡 정리 — 절실함 모드 → 여유 모드로 스위치

**직후 (당일 안에 무조건)**
- KPT 회고 (Keep / Problem / Try) — 잘 답한 질문 / 막힌 질문 / 다음에 보강할 것
- 받은 질문을 그대로 기록 — 다음 면접에서 같은 질문이 나오면 더 잘 답할 수 있도록 정리
- 답변하면서 처음 알게 된 사실(나에 대한, 회사에 대한)도 기록

### 처우 협상 사전 준비
> [[Salary-Negotiation-Guide|연봉 협상 가이드]], [[Common-Interview-Questions#처우 협상 접근|처우 협상 접근]]

- 2차 합격 시 곧바로 처우 협상 단계로 진입 가능 → 미리 준비
- **협상이 성립하기 위한 전제**: 무턱대고 "희망액 안 주면 안 간다"는 결렬 신호. 시장 데이터·JD 매칭도·1차+2차 통과 사실을 카드로
- **여유 모드 유지** — 협상 단계에서 절실함은 가장 큰 감점. "이 회사가 나에게 맞는지도 내가 본다"는 마인드
