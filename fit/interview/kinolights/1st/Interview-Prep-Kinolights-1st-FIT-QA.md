## 1. 자기소개 (45~60초)

> 안녕하세요, 4년차 백엔드 개발자 [이름]입니다.
>
> 트라이포드랩에서 NestJS 기반 IoT VMI 서비스의 백엔드를 사실상 처음부터 설계하고 운영해왔습니다. 식품에서 제약과 부자재까지 도메인을 넓히며, 디바이스 수천 대가 동시에 보내는 재고 데이터를 **DB Lock으로 정합성**을 잡고, 복잡한 발주 로직을 **EventBridge와 SQS 이벤트 아키텍처로 디커플링**했습니다. 운영에선 복합 인덱스로 **슬로우 쿼리를 99.3% 줄이고**, Grafana, Prometheus, Loki로 관측 인프라를 직접 구축했습니다.
>
> 저는 빠르게 만드는 것보다 **운영에서 깨지지 않는 구조**, 실시간보다 **정합성과 복구 가능성**을 중시합니다. 작은 팀이 큰 트래픽을 감당하게 만드는 자동화가 강점입니다.
>
> **(키노라이츠 매핑)** 키노라이츠는 백엔드 팀이 서버, 인프라, CI/CD, 관측, 장애를 한 팀이 책임지고 MSA를 GraphQL Federation으로 통합하는 단계라는 점에서, 제가 작은 팀으로 인프라, 운영, 기술 의사결정을 직접 진 경험과 본질이 같다고 봤습니다. **백엔드 자리**에서 운영 안정성과 다음 구간을 함께 만들고 싶어 지원했습니다.

## 2. 지원 동기 / 왜 키노라이츠 (= 사전질문지 폼 제출본)

> 이 답변이 Google Form 제출본이자 본 미팅 구두 지원동기다. 둘의 톤이 어긋나면 진정성이 깨진다. 읽는 사람은 이미 이력서를 봤으므로 제품과 단계, 팀이 주제이고 경험은 근거로만 쓴다. 정량 수치는 뺀다. 연결고리 넷: ① OTT 메타데이터를 통합하고 정합성과 최신성을 지키는 일이 VMI IoT 데이터 수집과 집계의 동형이라는 점 ② B2C 대중 트래픽은 IoT 디바이스 트래픽과 결이 다른 기술 도전이고 기술 결정이 사용자에게 바로 닿는다는 점 ③ 흑자 Series A는 운영 안정성 구간이라 본인 강점과 맞는 점 ④ 검색과 추천은 안 다뤄본 학습 동기라는 점.

좋은 작품을 발견하고도 어디서 볼 수 있는지 매번 따로 검색하던 불편을, 키노라이츠는 흩어진 OTT의 콘텐츠와 가용성, 평점을 한곳에 통합해 풀어줍니다. 결국 여러 소스의 데이터를 정합성 있게 모아 최신으로 유지하고 검색과 추천으로 연결하는 일이라, 제가 수천 대 IoT 기기의 데이터를 정합성 있게 집계해 리포트로 연결해온 일과 본질이 같다고 느꼈습니다. 다만 IoT가 예측 가능한 디바이스 트래픽이었다면 B2C는 사람의 트래픽이라, 스파이크와 대규모 동시 조회 같은 결이 다른 부하를 정면으로 풀어보고 싶었고, 기술 결정이 사용자에게 바로 닿는다는 점도 끌렸습니다.

흑자 전환한 Series A에서 작은 백엔드 팀이 서버와 인프라, CI/CD, 관측, 장애 대응을 함께 책임지며 MSA를 GraphQL Federation으로 통합해가는 구조가, 제가 작은 팀에서 백엔드 전 영역을 직접 설계하고 운영해온 경험과 정확히 맞닿아 있었습니다. 검색과 추천은 아직 깊게 다뤄보지 못한 만큼 가장 파고들고 싶은 영역이고, 단독으로 검증한 주도성을 이제 동료들과 아키텍처를 함께 다듬으며 더 깊게 가져가고 싶어 지원했습니다.

**구두로 더 풀 때 (본 미팅, 재지원 언급되면)**: 작년에 한 번 지원했다 서류에서 멈췄는데, 그땐 강점을 단독으로 많이 해봤다는 넓이로만 어필했던 것 같습니다. 이번엔 기술 의사결정과 운영 안정성이라는 축으로 다시 정리해 지원했고, 그게 더 정확한 저라고 봅니다. 이어서 §6 역질문 4번으로 평가 기준 피드백을 요청한다.

> ⚠️ 첫 문장 일화는 실제여야 함. 안 써봤으면 제출 전 한 번 써보고 구체적 작품 하나를 넣으면 제품 이해 신호가 올라간다. GraphQL Federation은 통합해가는 구조까지만 표현하고, 잘 안다는 뉘앙스는 피한다. 따옴표 강조나 가운뎃점은 넣지 않는다(AI 흔적).

## 3. 이직 사유 (사수 있음 → 본래 버전 사용 OK)

> 트라이포드랩에서 백엔드를 **사실상 단독으로** 0→1, 운영 자동화, 대형 PoC까지 폭넓게 잡았습니다. 주도성과 기술 의사결정 경험은 크게 늘었고, 다음 구간은 그 위에 **깊이** — ① PoC 너머 본 계약과 다고객 운영, ② **더 깊은 기술 환경에서 동료, 리더와 아키텍처를 함께 다듬으며 합을 맞추는 경험**이라고 봤습니다.
>
> **(주의)** 키노라이츠는 사수, 백엔드 팀이 있으니 ②를 정직하게 써도 됨. 단 "배울 시니어가 필요해서"가 아니라 **"단독 주도성은 검증됐고, 이제 팀과 합을 맞추며 깊이를 더하고 싶다"**로. 외부 귀책 톤 금지("회사 성장 느려서" X).

**꼬리 — "왜 또 이직? 우리도 금방 나가는 거 아니냐"**:
> 이전 결정들은 각 자리에서 부족한 한 축씩 채운 궤적입니다(이썸=도메인 시야, 시솔=운영 품질, 트라이포드=0→1과 자동화). 시솔 1년과 트라이포드 2년+를 거치며 **쉽게 결정하지 않는 법도 배웠고**, 지금은 최소 2~3년 깊게 파고들 자리를 찾고 있습니다.

## 4. 핵심 FIT 답변 (fork)

### 왜 백엔드 (정체성)
> 백엔드는 **물리적 사물, 데이터, 시간을 디지털로 연결하고, 그게 어떤 상황에서도 깨지지 않게 떠받치는 일**이라 봅니다. IoT 재고 데이터를 정합성 있게 받아 발주와 알림으로 연결하면서 이 연결이 사업을 떠받친다는 걸 체감했고, 이 책임을 5~10년 깊게 가져가고 싶습니다.

### 장단점 (한 흐름)
- **장점**: 문제를 끝까지 파고들어 구조로 해결. (Prisma JOIN 저하 → 공식 문서 → relationLoadStrategy → 실행계획 검증 → 90% 개선. **AI나 추측 의존 X, 끝까지 검증**.)
- **단점**: 규모 있는 시스템과 표준 인프라 직접 운영 경험이 얇음(EventBridge로 판단은 했으나 Kafka 직접 운영은 X). **보완**: 도구를 '언제, 왜'로 문서화, 이벤트 아키텍처 본질(전달 보장과 멱등성) 체화 → 도구 전이 빠름.
- 꼬리: "GraphQL Federation이나 OpenSearch 없는데?" → 본질 이해 + 학습 로드맵 + 팀 구조 빠르게 따라가기 (메인 문서 갭 전략).

### 갈등, 소통 (데이터로 우선순위 가시화)
> 환불 정책을 기획자가 다음 주까지 요청했는데 기존 정산과 충돌해 환불 오차 위험이 있었습니다. **위험을 데이터로 정리(오차 시 고객 보상과 역추적 비용)해 공유**하고 일주일 미루기로 합의 → 오픈 후 민원과 회계 문제 0. 갈등의 본질은 우선순위 차이지 사람 문제가 아니라, **데이터를 같은 화면에 올리면 감정 빠지고 결정만 남습니다.**

### 팀 협업 (제품 위기 크로스펑셔널)
> 합류 직후 프로덕트 핵심 기능이 깨져 박람회 데모도 어려운 상황이라, 한동안 **팀 전체가 매달렸습니다**. PM이 올바른 비즈니스 규칙 기준을 잡고, 프론트가 화면을, 저는 **백엔드에서 그 규칙을 테스트 코드로 못 박았습니다**. 합의를 테스트라는 검증 가능한 형태로 고정하니 인식 차가 사라졌고, 그렇게 안정화한 덕에 박람회에서 제품을 보여주고 대형 PoC까지 따냈습니다. **단독 주도성은 검증됐고, 이 협업 경험을 키노라이츠 팀에서 더 깊게 가져가고 싶습니다.** (대본 §3와 동일 사건 — FIT에선 협업 각도로)

### 5년 후 / 목표
> 8년차 백엔드 리드 / **Staff Engineer 트랙** — 권한이 아니라 영향력, 직접 코드 짜며 팀 기술 방향 제시. 최종은 기술과 사업의 연결을 책임지는 자리. (관리자형 EM보다 **기술 결정 깊이**가 강점.)

### AI 도구 (AI 툴 지원 회사라 plus)
> 단순 코드 생성을 넘어 **파이프라인으로 재설계** — 계층형 CLAUDE.md, MCP, Subagent 3종, PostToolUse Hook, Stop Hook 자가 리뷰. 원칙: 설계와 의사결정 직접, 반복은 AI 협업.

### 워라밸, 강도 (물으면)
> 의미 있는 일이면 시간 투자 괜찮습니다(트라이포드 PoC 밤샘 사례). 다만 궁금한 건 **지속 가능성** — 흑자 단계에서 팀이 이 페이스를 어떻게 유지하는지. → 역질문으로 마이크 넘김.

## 5. 예상 Q&A (면접관 → 나)

### 발표, 이력서 기반 기술
| 질문 | 준비 포인트 |
|---|---|
| EventBridge vs Kafka, 언제 Kafka? | 리플레이, 순서 보장, 초당 수만+. 월 10만 규모는 SQS가 비용, 운영 압도 |
| Dual Write(DB 저장 OK, 발행 실패)? | Transactional Outbox — 같은 트랜잭션 INSERT + Relay 폴링, at-least-once |
| ECS Fargate vs EKS? | 운영 인력, 트래픽 규모에서 K8s 오버헤드 > 가치. 멀티 워크로드 늘면 재검토 |
| ALB와 NLB를 왜 둘 다? | IoT 펌웨어 IP 하드코딩 → NLB 고정 IP, 웹은 ALB 경로 라우팅 |
| SSL 인증서 관리? | 기존 Certbot 수동에서 ACM 자동 발급과 갱신으로(Route53 DNS 검증), CloudFront는 us-east-1, ALB는 리스너에 연결 |
| 슬로우 쿼리 — 인덱스 컬럼 순서 근거? | 동등(=) 앞, 범위 뒤, 카디널리티 높은 컬럼 앞 → 스캔 범위 빨리 축소 |
| 인덱스 쓰기 페널티? | INSERT/UPDATE 시 인덱스 갱신, 쿼리 패턴 기반 설계 + 모니터링 |
| Prisma vs TypeORM? | TypeORM Active Record/Data Mapper, QueryBuilder, 복잡 쿼리 불안정 / Prisma 스키마 중심, 마이그레이션 강점. **본질(추상화 비용, 실행계획 검증) 동일** |
| DB Lock — 데드락 대응? | 완전 예방 불가 → InnoDB 자동 탐지+rollback+재시도, NO WAIT로 상호대기 회피, 락 순서 통일 |
| Pessimistic 말고 Optimistic? | 읽기 중심, 충돌 낮을 때. IoT는 충돌 빈도 높아 Pessimistic |
| 관측 — Datadog 안 쓰고 GPL? | 작은 팀 비용 ROI. **키노라이츠 Datadog은 흑자 단계엔 합리적, 전이 가능** |
| 캐시 스탬피드? | TTL jitter + 만료 전 백그라운드 갱신 + mutex lock |
| Graceful Shutdown? | SIGTERM→진행 요청 완료, NestJS enableShutdownHooks, 워커는 visibility timeout 재전달 |

### JD/도메인 기반
| 질문 | 준비 포인트 |
|---|---|
