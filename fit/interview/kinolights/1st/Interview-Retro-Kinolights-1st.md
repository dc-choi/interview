## 진행 흐름 (실제 받은 질문, 답한 핵심)

### A. 발표 본론 — 메인 서사는 잘 전달됨

- 자기소개에서 아키텍처까지: 단일서버 → CloudFront/ALB/NLB + Fargate 스케일아웃 + 읽기쓰기분산 + 이벤트 아키텍처. **0에서 1 단독 설계와 운영 서사** 그대로 전달.
- 정상화: 외주 레거시(RDB 키매핑 깨짐, 기능 거의 동작 X) → PM과 비즈니스 규칙을 **실DB 통합테스트로 못박음**(트랜잭션 스크립트) → 박람회 성공 → FMB 2천억 PoC 수주, 정식 도입.
- **Q. EKS vs ECS 왜 비교?** → 신입이 남긴 도커, GitHub Actions 최소 변경하며 확장성. 쿠버 오버엔지니어링 회피 → ECS. (잘 방어)
- **Q. Kafka는 왜 고려?** → 발주 도메인 복잡 → 디커플링 위해 이벤트 아키텍처. "학부생이 한 거라기엔 과한데?" → "제가 고민한 겁니다" (오너십 명확히).
- **Q. 트래픽이 헐떡이지도 않는데 미리 다 고민한 느낌** → 모니터링으로 관측하며 판단. (선제 설계 의심을 관측으로 받아침, 무난)

### B. 관측 인프라 — 강점 구간

- CloudWatch에 SNS, Lambda 웹훅 → **가시성 부족** → GPL 자체호스팅. 콜드 데이터 S3, 프로메테우스 보관한계 → 타노스. requestId **카디널리티 폭발 → 라벨에서 빼고 로그 본문으로** (직접 터뜨려보고 배운 것).
- **Q. CloudWatch 한계가 로직 에러냐 트래픽 볼륨이냐?** → 가시성(프레젠테이션) 문제. 다른 직군(프론트, PO, 하드웨어 엔지니어)도 보게 하려고.
- 키노라이츠가 Datadog 잘 쓰더라 → GPL 직접 굴린 경험이 전이된다는 톤. (좋음)

### C. 동시성, 정합성

- 레이스컨디션(한 품목 여러 디바이스 동시 입출고) → **비관적 락**. Redis 분산락은 단일장애지점, 아키텍처 간소화 위해 보류.
- **⚠️ Q. 락과 재시도가 thundering herd로 폭발하는 구조 아닌가?** → "단일 DB라 발생 안 함"으로 회피. **NOWAIT fail-fast, 지수 백오프와 지터, 비스파이크 트래픽(1~2h 주기)** 논리로 정면 반박했어야.
- **Q. 분산락 언제 적절?** → 외부 API(결제 등) 트랜잭션 분리 시. (무난)
- Decimal.js, 통화별 반올림 → 발주 금액 신뢰 (잘 전달).

### D. 성능, ORM

- 복합인덱스 15.4ms에서 0.12ms. relationLoadStrategy(app join 82~90%↑) → 한계(DB JSON 파싱) → **Kysely**(Prisma 스키마 기반 타입 쿼리빌더).
- **⚠️ Q. 2400ms 되도록 왜 인지 못했나?** → "인지는 했다, 단독 운영이라 완벽한 구조 불가능". 약간 변명 톤. **슬로우쿼리 알림 임계가 그 시점엔 리포트 경로에 없었다**는 식의 구조적 답이 더 깔끔.
- **Q. 왜 raw나 서브쿼리 직접 튜닝 안 하고 쿼리빌더?** → 타입 안정성과 유지보수(DB 교체 시 raw 전부 수정), 크리티컬만 Kysely. (좋은 트레이드오프)
- Kysely와 Prisma는 신입이 남긴 스택임을 솔직히 인정.

### E. 발주 자동화 (이벤트 아키텍처) — ★ 가장 크게 흔들린 구간

- EventBridge 하나로 발주 이벤트 묶고 채널별 SQS에 DLQ로 디커플링.
- **Q. 트랜잭셔널 아웃박스 설명** → DB insert는 되는데 큐 발행이 네트워크로 유실되는 듀얼라이트 문제, 같은 트랜잭션에 아웃박스 insert. (개념은 정확)
- **⚠️⚠️ Q. EventBridge가 폴링하고 다시 이벤트드리븐으로 바꾼 구조가 어색하다. 아웃박스도 큐, SQS도 큐인데 왜 SQS 없이 워커 바로 안 붙이나?** → **명쾌히 못 풀고 빙빙 돔.** 면접관 핵심: 폴링이면 폴링, 인터럽트면 인터럽트인데 둘이 섞였다. 정답 축은 **멀티 인스턴스 환경에서 인메모리 이벤트 에미터로는 인스턴스 간 전파 불가 → 외부 브로커 필요**인데, 이걸 한 문장으로 못 박음.
- **Q. Lambda는 왜 고려 안 했나?** → 혼자라 람다 코드 관리비용. "더 고민했으면 좋았다" 인정(좋은 솔직함, 단 비용 논리는 약함).
- **Q. EKS, Kafka로 스택 넓히는데 비용 균형감은?** → 혼자 빠르게 하던 시기 미스 인정, MSK 월 570달러 안 씀, 트랜잭셔널 아웃박스도 결국 미적용(아쉬움) 솔직히.
- **★ Q. 기술스택에 강한 열망 있나?** → "아니요, 회사 필요하면 씁니다. 필요 판단은 도메인 이해와 히스토리." **시니어 시그널, 이번 면접 최고의 답.**

### F. CS, 기초 단답 라운드 — 기본기 구멍 노출

- Promise(콜백지옥, resolve와 reject, 상태 3개는 기억 안 남), protected(상속 자식 접근 — OK), 대칭과 비대칭키(PKI, FIDO — OK).
- **⚠️ 타입 가드** → "인터프리터가 타입을 그때그때 결정"이라 답해 면접관이 타입가드 레이어와 노드 런타임 레이어는 다른 얘기라 지적. **타입은 컴파일 시 소거, 타입가드는 런타임 JS 검사(typeof, instanceof, in)로 좁히기** 구분 못 함.
- **⚠️ NestJS 요청 라이프사이클** → 미들웨어에서 인터셉터, 가드, 파이프 순서로 답해 부정확. 실제는 **미들웨어 → 가드 → 인터셉터 → 파이프 → 컨트롤러 → 인터셉터 → 예외필터**.
- **⚠️ DI 스코프** → DEFAULT, REQUEST만. **TRANSIENT 누락.**
- **⚠️ 소프트 딜리트와 MySQL 유니크 충돌** → 모른다 인정. (deletedAt이 NULL이라 유니크 회피되는 문제를 몰랐음)
- **⚠️ git rebase vs merge, fast-forward** → rebase 설명 부정확, **fast-forward는 블랭크**("말이 안 나옵니다"). 외우면 끝나는 거라 제일 아까움.
- **⚠️ RxJS** → 인터셉터에 쓰지만 깊게 모름 인정. (GraphQL과는 무관한데 질문 순서가 엮여 나옴)
- 환경변수 → GitHub Secrets 파일 복붙(현재도). KMS 도입 고려. **Secrets Manager나 Parameter Store 언급 없어 약함.**
- **⚠️ 500 에러 인지** → SLO 99% 기준. 면접관: 퍼센트는 단 한 건짜리 크리티컬 에러를 감춘다, 결제 같은 건 단 한 건도 슬랙 웹훅. → 구글챗(사람용)만 쓰고 **기계 사이 알림(크리티컬 건별 즉시 알림) 구조 없음** 노출.

### G. 컬처, AI 논쟁, 역질문

- **★ AI 시대 인력 논쟁(가장 길고 압박)**: AI가 구현하면 왜 사람이 많이 필요한가, CTO 목표가 유효한가? → 의사결정과 맥락은 AI 못 따라옴, 책임은 사람, 집단지성, 내적동기 인재. 규모는 작아지되 고퀄 검수 위해 사람 필요. **버텨냈지만 방어적이고 반복적**이었음. 핵심 한 방(AI는 책임을 안 진다, 판단의 값이 오른다)이 늦게 나옴.
- 풀고싶은 문제: **B2C 피드백루프**(B2B는 느림)에 **검색과 추천**(데이터 정합성 경험 연결). 도메인 관심 진정성 전달됨.
- 역질문: 일하는 방식 → 대표 이니시에이션 → PM 분배 → **실무자 주도권 높음**, DB 스키마 등 디테일은 엔지니어 결정. GraphQL 필드레벨도 엔지니어.

## ✅ 잘 작동한 카드 (다음에도 재현)

1. **기술 열망 없다, 회사 필요하면 쓴다, 판단은 도메인과 히스토리** — 시니어 판단력 시그널. 면접관 톤이 풀렸음.
2. **관측 인프라 0에서 1** — GPL 자체호스팅, 카디널리티 직접 터뜨려본 경험. 키노 Datadog과 자연 연결.
3. **오너십과 솔직함** — 신입이 남긴 스택, 미적용한 아웃박스, 람다 미고려를 변명 없이 인정. Developer-Interview-Signals Q4(아쉬운 점 먼저)와 정렬.
4. **B2C와 검색추천 지원 동기** — 도메인 진정성. JD 핵심과 직격.

## ⚠️ 다듬을 점 (감점 가설)

1. **폴링 vs 인터럽트(E구간)** — 프렙엔 "폴링 말고 CDC?"는 있었지만 면접관의 "왜 SQS 없이 워커 직접?" 프레임은 못 받아침. **멀티 인스턴스라 외부 브로커 필연** 한 줄을 마스터에 박아야.
2. **기초 단답(F구간)** — 타입가드 레이어, NestJS 라이프사이클 순서, DI 스코프, soft delete 유니크, git rebase와 fast-forward. **하루 암기로 메울 수 있는데 깐깐한 면접관에겐 누적 감점.**
3. **크리티컬 에러 알림 부재** — SLO 퍼센트만으로 답해 건별 즉시 알림 없음 노출. 관측 카드에 SLO 통계 알림과 크리티컬 건별 웹훅 **이원화** 추가.
4. **방어적 톤** — 2400ms, thundering herd, AI 논쟁에서 변명이나 회피 톤. 약점은 인정 후 **구조적 반박 한 줄**로 닫는 연습.

## 결정 (현재 시점)

- **회사 fit**: 강. 검색과 추천 도메인 본인 관심 직격, NestJS와 AWS, 관측 강 매치. 갭은 GraphQL Federation과 OpenSearch(학습 의지로 정직하게 처리됨).
- **재무 fit**: 안정 쪽. 첫 흑자 전환은 긴축기 외부 자본 없이 버틸 체력([[Career-Company-Judgment]] 합격선). 백엔드 팀 존재(윤회, 스페이스맵 1인 자리와 다름).
- **본인 정서**: 채팅 맥락상 키노라이츠를 윤회보다 낫다고 본 긍정 기조다. 면접 강도가 셌어도 키노 자체에 대한 거부감은 아니었다. (입사 전 온보딩을 시킨 처사에 정이 털린 건 윤회 쪽이고 이 회고와는 별개다.) 깐깐한 검증 문화는 [[Self-Analysis|ISTP]]가 잘 맞는 논리와 근거 중심 피드백 환경의 신호라, 부정이 아니라 오히려 매치 쪽으로 읽힌다.
- **결론**: **2차 진행할 가치 충분.** 부족한 건 정보가 아니라 확신이다([[Self-Analysis]] 확신 vs 정보). 여기서 검색과 추천이라는 명확한 기술 문제를 풀 수 있나가 합격선이면 나머지는 확신의 문제. 2차 전 역반영 7개 중 **폴링과 인터럽트, AI 인력, 기초 암기** 3개는 반드시 닫고 들어간다.
