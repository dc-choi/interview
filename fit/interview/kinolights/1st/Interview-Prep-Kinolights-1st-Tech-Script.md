# 키노라이츠 1차 — 30분 기술 발표 대본

> **사용법**: 통으로 외우지 말 것. 흐름과 핵심 문장만 익히고 현장에선 자기 말로. 다이어그램을 짚으며 천천히. [시간]은 누적 목표. Q&A로 끊기면 끊긴 만큼 뒷부분을 압축한다.
> **톤**: 결정과 트레이드오프 중심. 각 결정마다 아쉬웠던 점을 내가 먼저 꺼낸다.
> **콘텐츠 밀도 주의**: 11블록. 마커상 28분(2분 버퍼). 더 줄이려면 9번(AI) → 7번 k6 순으로 압축. 관측(8번)은 featured라 유지.
> **슬라이드 자산**: 기존 아키텍처(포트폴리오 7p), 현재 아키텍처(8p), 비교표 MSK vs SQS(4p), 관측 5계층(9p).

## 0. 오프닝 [0:00–0:20]

안녕하세요, 4년차 백엔드 개발자 [이름]입니다. 오늘은 먼저 제가 어떤 길을 걸어왔는지 짧게 소개한 다음, 트라이포드랩에서 IoT 기반 VMI 서비스의 백엔드를 거의 처음부터 설계하고 운영하면서 내렸던 기술 의사결정들을 중심으로 말씀드리겠습니다.

## 1. 약력 / 커리어 소개 [0:20–1:50]

간단히 소개하면, 각 자리에서 부족했던 한 축씩 채워온 4년입니다. 최근부터 말씀드리겠습니다.

지금은 트라이포드랩에서 IoT VMI 서비스의 백엔드를 사실상 단독으로 0부터 설계하고 운영하고 있습니다. 오늘 발표 내용 대부분이 이 자리에서 나온 경험이라, 자세한 건 뒤에서 풀고 여기서는 짧게 넘어가겠습니다.

그 전 시솔지주에서는 국제 배송 서비스를 만들며 운영 품질을 처음으로 직접 책임졌습니다. 환율 배치 안정화와 테스트 게이트 정착, FIDO 인증 서버 개발을 리드하면서, 빠르게 만드는 것과 운영에서 깨지지 않게 만드는 것은 다른 일이라는 걸 배웠습니다. 가장 처음 이썸테크에서는 Java와 Spring 기반 SI 환경에서 백엔드 기본기를 다졌고, 그 사이사이 42서울과 멋쟁이사자처럼 부트캠프를 수강생 100명 중 최우수로 수료하며 기본기를 다시 다졌습니다. 정리하면 SI에서 기본기, 시솔에서 운영 품질, 그리고 지금 트라이포드랩에서 0에서 1을 직접 만드는 주도성까지 쌓아온 셈입니다.

## 2. 컨텍스트와 아키텍처 — 기존에서 현재까지 [1:50–4:50]

먼저 서비스 배경입니다. 트라이포드랩의 VMI 서비스 Earlivery는 공급자가 고객사 재고를 직접 관리해주는 서비스입니다. 현장에 설치된 IoT 디바이스 수천 대가 실시간으로 재고 데이터를 보내면, 그걸 받아 발주와 알림, 리포트로 연결하는 게 핵심입니다. 식품에서 시작해 제약과 부자재까지 도메인을 넓혔고, 그 위에서 제약바이오 280억, 식음료 2000억 규모의 대형 PoC도 진행했습니다.

제가 합류했을 때 인프라는 이런 모습이었습니다. [기존 아키텍처 화면, 포트폴리오 7p] 단일 EC2 한 대 위에 nginx 리버스 프록시와 NestJS 앱 서버를 Docker 컨테이너로 함께 띄우고, 한 서버가 프론트 정적 자원과 API 요청을 모두 처리하는 구조였습니다. SSL은 Certbot을 호스트에서 수동으로 돌려 관리했고요. 작게 시작하는 단계에서는 합리적인 구조였습니다.

문제는 서비스가 크면서 드러났습니다. 모든 요청이 한 서버로 몰리니 트래픽이 튀면 CPU와 메모리가 같이 치솟아 전체 서비스가 멈출 위험이 있었고, nginx에서 정적 리소스와 API가 뒤섞여 응답이 밀렸습니다. 특히 피크 시간엔 IoT 디바이스의 실시간 요청이 지연되거나 타임아웃됐습니다.

그래서 지금은 이렇게 바뀌었습니다. [현재 아키텍처 화면, 포트폴리오 8p] IoT 디바이스는 펌웨어에 IP가 고정으로 박혀 있어서 L4 NLB로 받고, 웹과 관리자 트래픽은 CloudFront를 거쳐 L7 ALB로 받습니다. 둘 다 ECS Fargate 위의 NestJS API로 모이고, 발주 같은 도메인 이벤트가 생기면 EventBridge가 받아 채널별 SQS로 흘려보내고 별도 워커가 외부 연동을 처리합니다. 데이터는 RDS를 Primary와 Replica로 나눠 읽기와 쓰기를 분리했습니다. 핵심은 이 그림을 한 번에 만든 게 아니라 단계적으로 키웠다는 점이고, 매 결정에서 회사의 매출 규모와 운영 인력을 같이 봤다는 점입니다.

## 3. 프로덕트 정상화 — 비즈니스 규칙을 담은 테스트 코드 [4:50–6:50]

본론의 첫 사례는 제가 합류 직후에 마주한 상황입니다. 그때 프로덕트의 핵심 기능들이 제대로 동작하지 않아서, 회사 입장에선 제품을 제대로 보여줄 수도 없는 위험한 상태였습니다. 박람회에서도 데모를 온전히 돌리기 어려운 상황이었고요.

제가 가장 먼저 한 건, 비즈니스 규칙을 테스트 코드로 담아내는 것이었습니다. 엣지 케이스까지 하나씩 잡으면서, 재고 산정이나 디코딩 같은 핵심 로직이 규칙대로 동작하는지 검증하는 테스트를 채웠습니다. 단위 테스트만으로는 실제 동작을 보장하기 어려워서, 컨테이너를 띄워 실제 DB에 붙는 통합 테스트 환경을 만들었습니다. 모킹이 아니라 진짜 DB에서 검증하니, 숨어 있던 버그와 정합성 문제가 드러났습니다.

이렇게 프로덕트가 안정적으로 동작하게 되면서, 회사가 박람회와 영업 현장에서 제품을 제대로 보여줄 수 있게 됐습니다. 그리고 운영을 하다 보니 오류가 반복적으로 나와서 모니터링과 로깅을 강화하게 됐고, 그렇게 잡은 안정적인 기반 위에서 발주 자동화 같은 다양한 비즈니스 기능으로 기여를 넓혀갔습니다. 오늘 나머지는 그 위에서 쌓은 이야기입니다.

## 4. 이벤트 아키텍처 — 발주 자동화 [6:50–10:50]

다음은 발주 자동화입니다. VMI 특성상 발주 도메인 자체가 복잡합니다. 자동발주 조건 분기만 열 가지가 넘고, 예약발주, 장바구니, 거래명세서 연동까지 얽혀 있습니다. 처음엔 재고가 임계치에 닿으면 담당자가 수기로 발주를 넣었는데, 발주 한 건에 한 시간, 재고 관리에 하루 네 시간씩 들었습니다. 이걸 이벤트 기반으로 자동화했습니다.

흐름은 이렇습니다. 재고 임계치라는 도메인 이벤트가 발생하면 EventBridge 규칙이 받아서, 알림톡과 이메일, 내부 알림이라는 채널별 SQS 큐로 보냅니다. 각 큐를 ECS 워커가 소비해서 외부 API를 호출합니다. 결과적으로 네 시간 걸리던 재고 관리가 10분으로, 95.8퍼센트 줄었습니다.

여기서 제가 가장 고민한 결정은 메시징 인프라 선택이었습니다. 보통 이벤트 아키텍처라고 하면 Kafka를 떠올리는데, 저는 정량 비교 끝에 EventBridge와 SQS를 골랐습니다. [비교표 화면, 4p] 관리형 Kafka인 MSK는 한 달에 574달러가 고정으로 나갑니다. 반면 저희 트래픽인 월 10만 발주에 액션 다섯 개를 곱하면 월 50만 건 정도인데, 이건 SQS 프리 티어 범위라 한 달에 0달러에서 18달러 수준이었습니다. PMF를 찾아가는 스타트업에 Kafka는 과한 비용이었고, 발주라는 도메인은 실시간보다 최종 일관성이 더 중요했습니다. 기술 결정을 사업 단계와 같이 본 사례입니다.

신뢰성은 채널별로 다르게 설계했습니다. 알림톡은 번호가 틀리면 점진적으로 재시도하다 포기하고, 이메일은 일시 실패면 계속 재시도합니다. 최종 실패는 DLQ로 보내고 긴급 알림을 띄워 수동 처리 큐로 넘깁니다. 멱등성은 발주 상태 머신과, 알림 로그 테이블에 발주 아이디와 채널을 묶은 유니크 제약으로 이중 발송을 원천 차단했습니다.

지금이라면 다르게 했을 부분도 있습니다. DB에는 저장됐는데 이벤트 발행이 실패하는 듀얼 라이트 구간을 초기엔 재시도로만 막았는데, 지금이라면 Transactional Outbox 패턴으로 처음부터 갔을 겁니다. 저희 규모에서는 CDC까지 갈 필요 없이 폴링만으로 충분했을 거라고 봅니다.

## 5. 인프라 전환 — ECS Fargate와 이중 로드밸런서 [10:50–14:00]

앞서 보여드린 단일 EC2 구조에서 지금 구조로 어떻게 옮겼는지입니다. 한 번에 갈아엎지 않고 세 단계로 나눴습니다. 컨테이너와 로드밸런서 도입, 이벤트 분리, DB 읽기 쓰기 분리. 둘째는 방금 다뤘으니 첫째와 셋째를 말씀드리겠습니다.

가장 흥미로운 결정이 로드밸런서를 두 개 둔 겁니다. IoT 디바이스는 펌웨어에 서버 IP가 하드코딩돼 있어서 고정 IP가 필요했습니다. 그래서 고정 IP를 줄 수 있는 NLB로 디바이스 트래픽을 받고, 웹 트래픽은 경로 라우팅이 되는 ALB로 받았습니다. 성격이 다른 두 트래픽을 하나로 묶을 수 없었습니다.

오케스트레이션은 ECS Fargate를 골랐습니다. EKS도 검토했지만 운영 인력과 트래픽 규모를 보면 쿠버네티스의 오버헤드가 가치보다 컸습니다. Fargate는 기존 Docker 환경을 그대로 확장할 수 있어 마이그레이션 비용과 운영 부담이 둘 다 낮았고요. 배포는 비용을 고려해 블루그린 대신 롤링으로 갔습니다. 정량적으로는 Docker 멀티스테이지 빌드와 빌드 캐싱으로 이미지를 909메가에서 513메가로 43퍼센트 줄이고 배포 시간을 26퍼센트 단축했습니다. 셋째 단계인 Read Replica 도입으로 조회 API는 40퍼센트 빨라지고 DB CPU는 30퍼센트 내려갔습니다.

솔직한 부분은, 쿠버네티스를 안 간 게 지금도 맞지만 영원히 맞는 건 아니라는 겁니다. 배치와 API, 모델 서빙 같은 여러 워크로드를 동시에 굴리는 단계가 오면 운영 인력과 트래픽 임계를 다시 보고 전환 시점을 판단해야 합니다.

## 6. 데이터 정합성 — 동시성과 정밀도 [14:00–16:30]

데이터 정합성은 두 축으로 잡았습니다.

첫째 동시성입니다. 같은 품목에 입출고 이벤트가 동시에 들어오면 재고가 깨지는데, 분석해보니 한 품목에 동시에 두세 대, 최대 다섯 대까지 요청이 몰리고 있었습니다. SELECT FOR UPDATE에 NO WAIT를 붙여 배타 락을 잡고, 충돌하면 100밀리초 간격으로 최대 세 번 재시도했습니다. 디바이스 타임아웃이 1초라 그 안에 수락과 거절이 결정됩니다. 낙관적 락은 충돌 빈도가 높을 때 트랜잭션 전체를 다시 실행해야 해서 비쌌고, Redis 분산 락은 단일 DB 환경에서 불필요한 인프라 의존성이라 안 썼습니다.

둘째 정밀도입니다. 재고와 금액 계산을 자바스크립트 number로 하면 부동소수점 오류가 납니다. 그래서 Decimal 연산과 통화별 반올림을 적용해서, 재고 수량과 금액이 어긋나지 않게 했습니다. 작아 보여도 발주 금액과 거래명세서가 1원이라도 틀리면 신뢰가 깨지는 도메인이라 중요했습니다.

## 7. 성능 — 인덱스 설계, ORM 한계, 성능 테스트 [16:30–20:30]

운영 중 성능은 세 가지로 말씀드리겠습니다.

첫째 인덱스 설계입니다. 디바이스 최신 상태 조회가 2초 넘게 걸렸습니다. 테이블 100만 건에서 생성시각 역순으로 전체 행을 파일소트하고 있었는데, 카디널리티를 분석해 디바이스 번호와 생성시각, 아이디를 묶은 복합 인덱스를 설계했습니다. 쿼리당 15.4밀리초에서 0.1밀리초로, 154배 빨라졌습니다. 핵심은 디바이스가 3000대로 늘어도 인덱스 탐색 한 번으로 끝나, 데이터 양과 무관하게 일정한 성능이 보장되는 구조가 됐다는 점입니다.

둘째 ORM 한계인데, 이건 두 단계로 풀었습니다. 처음엔 Prisma의 include가 관계를 가져올 때 SQL JOIN이 아니라 관계마다 별도 쿼리를 날려서, 특정 API가 100밀리초에서 1초까지 느려졌습니다. 공식 문서를 파고들어 relationLoadStrategy를 join으로 바꾸면 DB 레벨 조인으로 전환된다는 걸 찾았고, 실행계획까지 비교해 검증한 뒤 적용해 82에서 90퍼센트 개선했습니다. 그런데 더 복잡한 쿼리에서는 relationLoadStrategy로도 한계가 왔습니다. 그래서 prisma-kysely를 도입했습니다. Kysely는 타입 안전한 쿼리 빌더라, ORM이 못 푸는 복잡한 쿼리를 raw에 가깝게 직접 작성하면서도 타입 안정성은 유지할 수 있습니다. 엑셀 다운로드 리포트 쿼리를 Kysely로 재작성해 2400밀리초를 40밀리초로, 60배 줄였습니다. 정리하면 단순 CRUD는 Prisma로 두고, 성능이 크리티컬한 복잡 쿼리만 실행계획을 보고 Kysely로 내려가는 2단계 전략입니다. 키노라이츠가 TypeORM을 쓰셔도, 추상화 비용을 실행계획까지 내려가 검증하는 이 사고는 그대로 적용된다고 봅니다.

셋째 성능 테스트입니다. 이런 최적화가 회귀하지 않도록 k6로 성능 테스트 프레임워크를 구축했습니다. device-item이나 발주 같은 주요 엔드포인트별로 부하 테스트 스위트를 만들고, Ramping arrival-rate로 실제 트래픽 패턴을 흉내 냈습니다. Little's Law 기반으로 동시성을 자동 산정해서 SLA 임계값을 생성하는, 재사용 가능한 테스트 인프라로 정착시켰습니다. 발주 플로우는 장바구니 추가부터 공급업체 설정, 수량 설정, 발주까지 전체를 트랜잭션 단위로 추적하는 E2E 성능 테스트로 만들었습니다.

## 8. 관측 인프라 — Grafana, Prometheus, Loki [20:30–24:30]

앞서 말씀드렸듯 운영 중 오류가 반복되면서, 원인을 빨리 짚으려면 관측을 강화해야 했습니다. 처음엔 제가 CloudWatch와 SNS, Lambda로 기본 지표 수집과 알림을 직접 구축했는데, 서비스가 크면서 팀이 지표를 한눈에 보는 시각화를 필요로 했고, 세부 메트릭 추적과 로그 장기 보관도 한계였습니다. 그래서 Prometheus와 Loki, Grafana 오픈소스 스택을 자체 호스팅으로 직접 구축했습니다.

먼저 왜 오픈소스를 직접 띄웠냐인데, Datadog은 호스트와 메트릭, 로그 단위로 비용이 트래픽 따라 빠르게 늘어 작은 팀에는 ROI가 약했고, ELK는 라이선스 변경과 인덱스 운영 부담이 컸습니다. 오픈소스 스택은 비용이 예측 가능하고 자체 호스팅이 가능했습니다.

구조는 세 계층입니다. 메트릭은 Prometheus로 수집하고 Thanos Sidecar로 S3에 올려 장기 보관합니다. 단기는 Prometheus가, 일정 기간 지난 데이터는 S3가 맡아 비용 효율적으로 오래 보관하는 구조죠. 로그는 Promtail과 FireLens로 컨테이너 로그를 Loki로 모으고, Compactor가 오래된 로그를 S3로 압축해서 30일은 빠른 핫 스토리지, 이후는 저비용 콜드 스토리지로 운영합니다. 그리고 앱 계층에서는 요청별 메트릭을 노출하고, AsyncLocalStorage 기반 미들웨어로 요청 단위 트레이싱을 붙여서, 하나의 TraceId로 메트릭과 로그를 연결했습니다. 장애가 나면 요청 하나를 기준으로 메트릭과 로그를 같이 따라갈 수 있게 된 겁니다.

한 가지 신경 쓴 디테일은 메트릭 카디널리티입니다. userId나 traceId를 라벨에 넣으면 조합이 폭발해서 Prometheus가 메모리로 죽습니다. 그래서 그런 고유 값은 라벨이 아니라 로그 본문에 넣고, route 같은 건 정규화해서 카디널리티를 관리했습니다. 알림은 SLO 기반으로, 에러율 1퍼센트가 5분 지속되거나, 슬로우 쿼리가 500밀리초를 세 번 넘기거나, 이벤트 루프 랙이 100밀리초를 넘기거나, RDS CPU와 커넥션 풀이 임계에 닿으면 자동으로 팀에 알리게 했습니다. 이 알림 체계가 팀에 SLO 기반 운영 문화를 정착시키는 계기도 됐고요. 다만 이 자체 호스팅 결정은 회사 단계에 종속적이라, 키노라이츠가 Datadog을 쓰시는 건 흑자 단계에선 합리적이고, 오픈소스를 직접 굴려본 경험이 Datadog 설계로도 그대로 전이된다고 봅니다.

## 9. AI 워크플로우 [24:30–26:30]

최근에 집중하는 건 AI 워크플로우입니다. Claude Code를 단순 코드 생성이 아니라 개발 파이프라인 자체로 재설계해서, 계층형 컨텍스트 문서와 MCP, 서브에이전트, 작업 종료 시 자가 리뷰하는 훅까지 묶어 코드 품질 회귀를 통제합니다. 설계와 의사결정은 제가 하고 반복 작업은 AI와 협업하는 원칙이고요. 개인 프로젝트인 주일학교 출석 플랫폼을 이 방식으로 0부터 운영하고 있는데, 단체 약 100곳, 학생 약 2800명 규모이고 실DB 기반 사업 지표 분석까지 자동화했습니다.
