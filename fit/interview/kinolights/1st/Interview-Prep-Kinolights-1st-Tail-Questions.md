# 키노라이츠 1차 — 섹션별 꼬리질문 드릴 (3뎁스)

> **목적**: 발표 중 면접관이 끊고 들어오는 질문 대비. 섹션마다 체인 3~4개, 각 체인은 Q(1뎁스) → D2 → D3으로 파고든다.
> **사용법**: `→` 뒤 답변 포인트를 가리고 소리 내어 답해보기. D3까지 막힘없이 가면 그 섹션은 통과. 막히는 D3가 그날의 약점.
> **순서**: 대본의 새 시간 순 흐름(정상화 → 관측 → 인프라+동시성 → 정합성 → 성능 → 발주 자동화 클라이맥스)에 맞춰 정렬. 대본 본문은 [[Interview-Prep-Kinolights-1st-Tech-Script]]. 답변 톤은 모르면 "모릅니다 + 인접 지식 + 어떻게 알아낼지".

## 1. 약력

**Q. SI에서 제품 개발로 왜 옮겼나?** → 실 트래픽을 직접 운영하고 트러블슈팅하는 사이클을 경험하고 싶어서
- D2. SI의 어떤 점이 부족했나? → 프로젝트 단위라 운영 책임 사이클이 짧음, 언어보다 사이클을 골랐다
- D3. 그럼 또 안 맞으면 떠나나? → 이직 기준은 환경 불만이 아니라 성장 축, 시솔 1년과 트라이포드 2년+로 쉽게 안 떠나는 것도 배웠다

**Q. 부트캠프 최우수가 실무로 어떻게 이어졌나?** → CI/CD, 테스트, 코드 리뷰 문화를 거기서 체화
- D2. 트라이포드에서 구체적으로? → jest와 supertest PR 게이트, SonarCloud Quality Gate 정착
- D3. 단독 백엔드인데 리뷰는 누가? → 셀프 리뷰 한계를 ADR 문서화와 외부 발표로 보완, 그래서 팀 환경을 원함

## 2. 컨텍스트 / 아키텍처 (출발점, 목적지)

**Q. 단일 EC2에서 왜 ALB와 NLB를 둘 다?** → IoT 펌웨어에 IP가 하드코딩이라 고정 IP가 필요, ALB는 고정 IP 미보장이라 NLB(Elastic IP) 분리
- D2. ALB 하나로 IoT까지 받으면? → 고정 IP를 못 줘서 디바이스가 엔드포인트를 못 찾음
- D3. 펌웨어를 도메인 기반으로 바꾸면 NLB 불필요하지 않나? → 맞다, 다만 배포된 디바이스 OTA 비용과 리스크가 커서 인프라로 흡수하는 게 합리적이었다

**Q. SSL 인증서는 어떻게 관리하나?** → 기존엔 호스트에서 Certbot 수동 → ACM으로 옮겨 발급과 갱신을 자동화, CloudFront와 ALB에 연결
- D2. 갱신 자동화 방식은? → ACM은 Route53 DNS 검증이면 만료 전 자동 갱신, 수동 갱신 운영 부담이 0이 됨
- D3. CloudFront 인증서 특이점은? → CloudFront에 붙이는 인증서는 us-east-1 리전 발급이 필수, ALB는 같은 리전 인증서를 리스너에 연결

**Q. CloudFront는 왜 뒀나?** → 정적 자원 글로벌 엣지 캐싱과 SPA 전송
- D2. SPA 라우팅 404는? → CloudFront 커스텀 에러 응답으로 403, 404를 index.html로 fallback
- D3. API도 CF 뒤에 두면? → 동적이고 인증이라 캐싱 이점이 적고 ALB 직접이 단순, CF는 정적에 집중

**Q. 마이그레이션 중 무중단은 어떻게?** → 헬스체크 통과 후 점진 전환, 구 EC2를 살려둔 채 트래픽 컷오버
- D2. 컷오버 방식은? → Route53 가중치로 단계 전환하며 신규 ECS 모니터링
- D3. 롤백 시나리오는? → 가중치 되돌리고, DB는 두 버전 호환 스키마를 유지해 무손실 복귀

## 3. 프로덕트 정상화 (테스트 + 팀 협업)

**Q. 정상화에 팀 전체가 매달렸다는데, 본인 역할은?** → 백엔드에서 비즈니스 규칙을 테스트 코드로 못 박는 일을 맡음. PM은 올바른 규칙 기준, 프론트는 화면 쪽
- D2. PM, 프론트와 어떻게 합을 맞췄나? → 규칙 해석은 PM과 주고받고, 화면과 API 계약은 프론트와 맞춤. 합의를 테스트라는 검증 가능한 형태로 고정해 인식 차를 없앰
- D3. 단독 백엔드인데 협업 경험은 충분한가? → 제품 위기일수록 크로스펑셔널 협업이 필수임을 체감. 단독 주도성은 검증됐고, 이제 팀과 합을 맞추며 더 깊게 가고 싶음 (키노라이츠 연결)

**Q. 컨테이너로 실제 DB 붙는 통합 테스트, 어떻게 구성했나?** → 실 DB 컨테이너를 띄워 모킹 없이 실제 동작 검증
- D2. CI에서 느리지 않나? → 병렬 실행과 이미지 캐싱으로 시간 관리, 핵심 경로 위주
- D3. 테스트 간 데이터 격리는? → 테스트별 트랜잭션 롤백이나 truncate, 픽스처 시드로 독립성 확보

**Q. 비즈니스 규칙을 테스트로 담으면 규칙 변경 때 다 깨지지 않나?** → 변경의 영향 범위를 테스트가 드러내는 게 가치
- D2. 그럼 변경을 막는 족쇄 아닌가? → 의도된 변경은 테스트도 같이 수정, 의도치 않은 회귀만 잡힘
- D3. 단위와 통합 경계는? → 순수 도메인 규칙은 단위, DB와 정합성이 얽히면 통합, 피라미드 유지

**Q. 오류율은 어떻게 측정했고 정상화는 어떻게 검증?** → 디코딩 값과 실측 무게를 비교
- D2. 정확도 기준은? → 센서 raw 디코딩 결과가 실제 측정값과 일치하는지
- D3. 회귀 방지는? → 디코딩 핵심 경로 테스트와 모니터링 알림으로 이상 측정값 조기 탐지

## 4. 관측 인프라

**Q. 왜 Datadog 안 쓰고 GPL 자체 호스팅?** → 트래픽 따라 비용이 급증해 작은 팀엔 비용 예측성이 더 중요
- D2. 자체 호스팅 운영 부담은? → 그 부담보다 비용 예측성이 컸던 단계 종속적 결정
- D3. 부담이 커지면? → 흑자와 인력 여유 단계엔 Datadog이 합리적, 직접 굴려본 경험이 그대로 전이

**Q. 메트릭 카디널리티 폭발은 어떻게 막았나?** → 고유 값을 라벨에서 빼고 로그 본문으로
- D2. 구체적으로? → userId나 traceId는 라벨 금지, route는 정규화해서 조합 수를 제한
- D3. 그래도 터지면? → 라벨 드롭 스테이지, recording rule로 사전 집계, 고카디널리티는 트레이스로

**Q. 메트릭과 로그, 트레이스를 어떻게 연결?** → 하나의 TraceId로 묶음
- D2. TraceId는 어떻게 전파? → AsyncLocalStorage 미들웨어로 요청 단위 컨텍스트에 심어 로그와 메트릭에 주입
- D3. 비동기 워커까지 전파는? → SQS 메시지 속성에 TraceId를 실어 워커가 이어받아 트레이스 연속성 유지

**Q. SLO 임계(에러율 1퍼센트, 5분)의 근거는?** → SLO 역산과 단발 스파이크 필터
- D2. 왜 1퍼센트, 왜 5분? → 99퍼센트 정상이라는 목표에서 역산, for 5m으로 일시 스파이크 제외
- D3. 알림 피로는? → 지속 조건과 팀별 라우팅, 심각도 분리로 알림 폭주를 막음

## 5. 인프라 전환 (+ 동시성)

**Q. ECS Fargate 대 EKS, 지금도 같은 선택?** → 운영 인력과 트래픽 규모에서 K8s 오버헤드가 가치보다 컸다
- D2. 언제 EKS로 가나? → 멀티 워크로드(배치, API, ML), 복잡 라우팅, 멀티 클러스터 단계
- D3. Fargate의 한계는? → 노드 세밀 튜닝 불가, 태스크 기동 지연, GPU 제약, 그 한계에 닿으면 전환

**Q. 오토스케일링 기준은?** → API와 워커가 다르다
- D2. 각각 무엇으로? → API는 CPU 70퍼센트나 요청 수, 워커는 SQS 큐 depth
- D3. 스케일 인 때 처리 중 작업 유실은? → graceful shutdown으로 진행 요청 완료, 워커는 visibility timeout으로 재전달

**Q. Read Replica의 복제 지연 문제는?** → 강한 일관성 필요 조회는 Primary, 지연 허용은 Replica로 분기
- D2. 쓰기 직후 조회가 옛 데이터면? → 그 경로만 Primary 강제, 대시보드나 리포트는 Replica
- D3. lag 모니터링과 대응은? → ReplicaLag 5초 주의, 10초 경보, 심하면 스케일업이나 failover

**Q. 롤링 배포 중 구버전과 신버전이 동시에 도는데 호환성은?** → 확장 호환 마이그레이션으로 두 버전 모두 동작
- D2. 스키마 변경과 겹치면? → expand-contract, 컬럼 추가는 먼저 호환 배포 후 제거는 나중에
- D3. 실패 감지와 자동 롤백은? → 헬스체크와 ECS deployment circuit breaker로 이전 태스크 정의 복귀

**Q. 인프라 전환하면서 동시성도 같이 고민했다고?** → 한 대에서 여러 인스턴스로 스케일 아웃하면 인메모리 락이 무력화돼서, 정합성을 같은 묶음으로 설계해야 했다
- D2. 왜 한 묶음인가? → 스케일 아웃이 같은 재고 행 동시 쓰기를 실제 문제로 키운 트리거라, 인프라 결정과 분리할 수 없었다
- D3. 그래서 어떻게 풀었나? → DB 레벨 비관 락으로 (상세는 정합성 섹션). 락 대상이 DB 행이라 SELECT FOR UPDATE가 갱신과 한 트랜잭션으로 원자적, Redis 락은 트랜잭션 밖이라 desync 위험만

## 6. 데이터 정합성

**Q. 왜 비관적 락? 낙관적은?** → IoT 자동 트래픽은 충돌 빈도가 높아 낙관적은 재실행 비용이 과도
- D2. 충돌이 낮아지면? → 그땐 낙관적이 유리, 읽기 중심이나 충돌 적은 도메인
- D3. NOWAIT 대신 SKIP LOCKED는? → SKIP LOCKED는 큐 패턴(다음 행), 재고는 특정 행을 반드시 처리해야 해서 NOWAIT

**Q. 데드락은 안 나나?** → 락 순서 통일과 트랜잭션 범위 최소화로 확률을 낮춤
- D2. 순서를 어떻게 통일? → 품목 아이디 오름차순으로 락 획득
- D3. 그래도 나면? → InnoDB가 자동 탐지하고 rollback, 앱에서 잡아 재시도

**Q. 멀티 인스턴스인데 DB 락으로 충분한가? Redis 분산 락은 왜 안 썼나?** → 잠그는 대상이 DB 행이라 SELECT FOR UPDATE가 락과 갱신을 한 트랜잭션에서 원자적으로 처리, 모든 인스턴스가 같은 행에서 직렬화
- D2. Redis 락이면 안 되나? → 트랜잭션 밖이라 TTL 만료나 크래시 때 DB 상태와 desync 위험 + 인프라 추가. Redis는 DB 밖 자원 잠금이나 DB 락 경합이 병목일 때 카드
- D3. DB를 샤딩하면? → 그땐 분산 락이나 샤드 키로 같은 품목을 한 샤드에 고정

**Q. Decimal 연산의 오버헤드는?** → 정합성 크리티컬한 금액과 재고에만 적용
- D2. 모든 계산에? → 아니다, 일반 계산은 number, 금액은 Decimal과 통화별 반올림
- D3. DB 컬럼 타입은? → DECIMAL 컬럼에 스케일 고정, 앱과 DB 양쪽에서 정밀도 유지

## 7. 성능

**Q. 인덱스 컬럼 순서를 왜 그렇게 잡았나?** → 동등 조건을 앞, 범위와 정렬을 뒤에 두는 원칙
- D2. 순서가 바뀌면? → 정렬을 인덱스로 못 타서 filesort가 다시 발생
- D3. 인덱스 쓰기 페널티는? → INSERT마다 인덱스 갱신, 쿼리 패턴 기반 최소 인덱스, 핫스팟이면 파티셔닝 검토

**Q. relationLoadStrategy로 됐는데 왜 Kysely까지?** → 다중 JOIN과 집계 같은 복잡 쿼리는 그걸로도 한계
- D2. Prisma를 아예 버리면? → 단순 CRUD는 Prisma의 타입과 마이그레이션 생산성이 가치, 크리티컬만 Kysely
- D3. Kysely는 raw랑 뭐가 다른가? → 쿼리 빌더라 raw에 가깝게 쓰되 타입 안정성과 조립성을 유지

**Q. 2400ms를 40ms로, 60배가 정말 쿼리만으로?** → 리포트 쿼리 구조 자체를 Kysely 단일 최적화 쿼리로 재작성
- D2. 캐싱은? → 리포트는 실시간성이 필요해 쿼리를 고쳤고 캐싱은 별개 레이어
- D3. 검증은? → EXPLAIN으로 실행계획 before와 after 비교, k6로 P99 회귀 감시

**Q. k6 부하 테스트의 SLA 임계는 어떻게 정했나?** → 리틀의 법칙으로 필요 동시성을 역산
- D2. 산정 방식은? → 처리량 곱하기 응답시간으로 동시성 산출, 실제 트래픽이 오르내리는 패턴으로 시뮬레이션
- D3. 테스트 환경과 프로덕션 차이는? → 동일 스펙 스테이징 지향, 상대 회귀 감시로 환경 차이를 보정

## 8. 발주 자동화 (이벤트 아키텍처) — 클라이맥스

**Q. 왜 Kafka가 아니라 EventBridge와 SQS?** → MSK 월 574달러 고정 대 SQS 0에서 18달러, 발주는 최종 일관성 우선
- D2. Kafka가 맞는 순간은? → 이벤트 리플레이, 파티션 내 순서 보장, 초당 수만 건 이상
- D3. 나중에 순서 보장이 필요해지면? → SQS FIFO(MessageGroupId 기반. 일반 FIFO는 파티션당 비배치 300 API TPS, 최대 10개 배치 시 초당 3,000개 메시지다. 고처리량은 리전별 서비스 할당량과 MessageGroupId 분산을 확인한다), 또는 그 도메인만 Kafka로 분리

**Q. DB 저장은 됐는데 이벤트 발행이 실패하면?** → 듀얼 라이트 문제, Transactional Outbox로 해결
- D2. Outbox 구조는? → INSERT와 outbox INSERT를 같은 트랜잭션, 릴레이가 폴링해서 발행
- D3. 폴링 말고 CDC는? → 규모상 폴링이 단순하고 충분, 초당 수천 건 이상이면 Debezium 검토

**Q. 워커가 같은 메시지를 두 번 받으면?** → 멱등성, 발주 상태 머신과 알림 로그 유니크 제약
- D2. 유니크 제약은 어디에? → 알림 로그 테이블에 발주 아이디와 채널을 묶어 이중 발송 차단
- D3. 워커가 처리 중 죽으면 중복 처리는? → processing_started_at과 visibility timeout 배수로 crash를 판정해 재처리

## 9. AI 워크플로우

**Q. AI가 만든 코드의 품질과 회귀는 어떻게 통제?** → 자가 리뷰 훅과 실DB 통합테스트로 차단
- D2. 구체 장치는? → Stop Hook 자가 리뷰, PostToolUse lint와 prettier, 통합테스트가 회귀를 잡음
- D3. AI가 틀린 결정을 내리면? → 설계와 의사결정은 사람 몫, 예를 들어 EventBridge 대 Kafka 정량 비교는 직접 판단

**Q. 컨텍스트 토큰 폭주는 어떻게 관리?** → 계층형 컨텍스트 문서로 분할
- D2. 어떻게 분할? → 루트와 도메인별로 나눠 작업 시점에 관련만 로드
- D3. 큰 변경이면? → Plan Mode로 범위를 강제하고 도메인이나 DB, 다파일 변경은 게이트로 막음
