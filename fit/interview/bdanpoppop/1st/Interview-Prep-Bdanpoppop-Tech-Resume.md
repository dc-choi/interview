# 비단팝팝 1차 — 이력서 기반 기술 질문

> 상위 TOC: [[Interview-Prep-Bdanpoppop|비단팝팝 1차 면접 준비]]

상품권 교환 도메인은 **동시성·정합성·외부 API 연동**이 핵심이라 트라이포드랩에서 풀던 문제와 매칭이 매우 좋다. 액션파워 면접 때 다듬은 답변을 "상품권 교환" 도메인 맥락으로 연결해 어필하는 게 핵심.

> **우선순위 — 메일이 콕 집은 사례는 deep dive 검증 대상**
> 메일에서 채용팀이 명시적으로 언급한 1번(자동화 4시간→10분)·2번(안정 운영·사전 탐지) 사례는 면접관이 가장 깊게 들어올 가능성이 높다. 이 문서의 **#3 EventBridge+SQS 이벤트 아키텍처**와 **#5 Grafana/Prometheus/Loki 모니터링**을 우선순위로 준비하고, 메일에 없던 #1(DB Lock)·#2(인덱스)·#4(Docker)는 새 정보로 어필.

## 1. DB Lock으로 동시성 제어 — 상품권 교환·결제에 그대로 매핑

> 관련: [[Transaction-Lock-Contention|트랜잭션·락]], [[Lock|DB Lock]], [[Distributed-Lock|분산락]]

**경험 요약**
- 수천 대 IoT 디바이스가 동시에 같은 품목 재고를 갱신 → Lost Update 발생
- `SELECT FOR UPDATE NO WAIT`로 품목 단위 Exclusive Row Lock + 100ms 간격 3회 재시도
- Lock 순서를 품목 ID 오름차순으로 통일 → 데드락 회피
- 트랜잭션 범위 최소화 (검증은 트랜잭션 밖, 갱신만 안)

**상품권 교환 도메인 연결**
- 한 상품권을 두 사용자가 동시에 교환 신청·결제하는 시나리오 — 동일한 Lost Update 패턴
- 멱등성 키(상품권 시리얼) 기준 row lock으로 **이중 사용·이중 교환 차단**
- 교환 이벤트 = (사용자 A의 상품권 X, 사용자 B의 상품권 Y) 두 개를 한 트랜잭션에서 X Lock → 양쪽 모두 잠겨 있어야 atomic swap 가능
- 데드락 방지: 항상 **상품권 ID 작은 쪽 먼저** 잠그도록 정렬 (재고 ID 정렬과 동일 패턴)

**꼬리 질문 대비**
- "Optimistic vs Pessimistic 어느 쪽?" → 교환은 **충돌 빈도 낮음(특정 상품권에 동시 신청자 적음)** → Optimistic도 가능. 다만 결제 정산이 들어가면 재시도 비용이 커서 Pessimistic이 안전
- "분산락(Redis Redlock)은?" → 단일 DB면 DB lock으로 충분. ECS 멀티 인스턴스라도 동일 RDS면 OK. **샤딩이 들어가면 그때 분산락 검토**
- "FOR UPDATE NO WAIT vs SKIP LOCKED?" → 특정 상품권을 **반드시 처리해야 하면 NO WAIT**, 아무 상품권이나 작업 큐처럼 분배할 때 SKIP LOCKED

## 2. 슬로우 쿼리 99.3% 개선 — 2000종 상품권 조회·정렬에 직결

> 관련: [[Index|인덱스]], [[Execution-Plan|실행계획]]

**경험 요약**
- 디바이스 최신 상태 조회 서브쿼리 2000ms+ → 100만 건 테이블·850 디바이스
- EXPLAIN ANALYZE로 `ORDER BY created_at DESC` 후 filesort 확인
- 카디널리티 분석 → 복합 인덱스 `(device_number, created_at DESC, id DESC)`
- 결과: 15.4ms → 0.1ms

**상품권 교환 도메인 연결**
- 팝팝은 40개 브랜드 × 2000+ 상품권 종류 + 사용자별 보유·만기·교환 이력
- 조회 패턴 예시: "내 상품권 목록(만기 임박 우선 정렬)", "교환 가능 매물(브랜드·금액 필터·등록일 정렬)" → 복합 인덱스 설계 그대로 적용
- 만기 임박 정렬은 `(user_id, expires_at)` 복합 인덱스 + 커버링 인덱스로 테이블 접근 회피
- 교환 매물 검색은 카디널리티 높은 컬럼(brand_id, amount) 앞으로

**꼬리 질문 대비**
- "복합 인덱스 컬럼 순서 기준?" → 동등 조건(=) 앞, 범위(>, BETWEEN) 뒤. 카디널리티 높은 컬럼 앞
- "인덱스를 많이 만들면?" → SELECT 빠르지만 INSERT/UPDATE 시 인덱스 갱신 비용. 실제 쿼리 패턴 기반 설계
- "커버링 인덱스란?" → 쿼리 모든 컬럼이 인덱스에 포함 → 테이블 랜덤 I/O 회피

## 3. EventBridge+SQS 이벤트 아키텍처 — 결제·교환·알림 분리에 직결

> 관련: [[MQ-Kafka|MQ·Kafka]], [[Messaging-Patterns|메시징패턴]], [[Delivery-Semantics|전달보장]]

**경험 요약**
- 발주(핵심 도메인) → SQS → 후속 처리(카톡·이메일·발주서) 병렬 분리
- 채널별 DLQ 설정, 멱등성 키(발주 ID) + status 머신(`PENDING/PROCESSING/COMPLETED/FAILED`) + `processing_started_at` 으로 워커 crash 복구
- 비용 비교: MSK $574/월 vs EventBridge+SQS $0~18/월 → 도메인 특성(실시간 불요, 최종 일관성 OK)으로 SQS 선택

**상품권 교환 도메인 연결**
- 교환 이벤트 = 핵심 트랜잭션 + 부수 효과(알림·정산·블록체인 기록)
- 핵심: 교환 트랜잭션(상품권 소유권 변경)은 동기 + DB Lock으로 정합성 확보
- 부수 효과: SQS로 분리 → 알림(카톡·푸시), 정산(수수료·정산 큐), Web3 기록(체인 기록은 비동기 + 재시도)
- **블록체인 기록을 비동기로 빼는 게 핵심** — 체인 트랜잭션은 수~수십초 걸리고 실패할 수 있어 핵심 흐름에 동기로 묶으면 위험

**꼬리 질문 대비**
- "Kafka vs SQS — 어떤 상황이면 Kafka?" → 이벤트 리플레이, 순서 보장, 초당 수만건 이상 처리. 팝팝 초기 단계는 SQS로 충분 + 트래픽이 늘면 Kafka 검토
- "메시지 유실은?" → SQS at-least-once → 멱등성 키 필수. 워커는 발주 ID(여기서는 교환 ID)로 status 확인 후 처리
- "블록체인 트랜잭션이 실패하면?" → 체인 RPC 호출은 별도 워커에서 nonce 관리·gas 조정으로 재시도. DLQ로 빠지면 운영자 알림. **체인 실패가 사용자 교환 자체를 롤백하지 않도록 분리**가 핵심
- "SQS FIFO는 언제?" → 교환 알림 순서가 중요한 케이스(결제 → 교환 완료 → 정산 순). 다만 FIFO는 300 TPS 제한이라 표준 SQS + 시퀀스 ID로 처리하는 패턴이 더 일반적

## 4. Docker 이미지 43% 경량화 — 빌드·배포 효율

> 관련: [[Multi-Stage-Build|멀티스테이지빌드]]

**경험 요약**
- NestJS 이미지 909MB → 멀티스테이지 빌드 + .dockerignore → 513MB(43.6% ↓)
- 배포 시간 3분10초 → 2분20초(26.3% ↓), ECR 저장 비용 절감

**비단팝팝 맥락 연결**
- 출시 6~7개월차 + 1명 채용 → DevOps 비용·속도 직접 책임. 빌드/배포 사이클 효율화는 즉시 가치
- 메일에 "비용 효율" 언급 → 인프라 비용 최적화 어필 포인트

**꼬리 질문 대비**
- "alpine 이미지로 더 줄이면?" → native 모듈(bcrypt·sharp) musl libc 호환 이슈. 안정성 vs 경량화 트레이드오프
- "distroless는?" → 셸 없어 보안 강화·디버깅 어려움. 프로덕션 적합
- "더 최적화 방법?" → 번들러(esbuild) 사용, devDependencies 정리, 자주 변경되는 레이어 뒤로 배치

## 5-A. AI 업무 파이프라인 — 메일이 콕 집은 3가지 중 1개

> 메일 원문: "AI를 업무 파이프라인에 도입해 팀의 검수 작업을 자동화하신 시도는 실제로 실행해보신 분이 많지 않은 매우 앞서가는 시도"
> **부장급 면접관이 deep dive로 들어올 가능성 매우 높음. 본인 이력서의 AI 도입 사례 정량 수치(검수 시간 단축률·정확도·휴먼 인 더 루프 경계)를 사전에 정리.**

**deep dive 예상 질문 → 답변 준비 골격**

| 질문 | 답변 골격 |
|------|---------|
| "어떤 검수 작업이었나?" | 도메인·작업 종류·규모(건수·시간) — 본인 이력서 사례 |
| "AI는 어떤 모델? 자체 학습? 외부 API?" | 사용한 모델·서비스(예: GPT-4·Claude·자체 fine-tune·OCR 등). 선택 이유와 비용·정확도 트레이드오프 |
| "정확도는 어떻게 검증?" | 라벨링된 골든 데이터셋·사람 검수 비교·정밀도/재현율 |
| "사람이 봐야 하는 경계는?" | confidence threshold·예외 케이스 분류·휴먼 인 더 루프 설계. **AI를 "자동화의 끝"이 아니라 "사람의 1차 필터"로 둔 의사결정** |
| "잘못 자동화하면? 롤백·실패 안전망?" | shadow mode 운영·점진 비율 증가·이상치 알림 |
| "팝팝에 어떻게 적용 가능?" | 상품권 이미지 검수(위조·만료·브랜드 인식), 사용자 신고 1차 분류, 이상 거래 패턴 탐지(이상 교환 금액·반복 패턴) |

**팝팝 도메인 응용 — 면접관이 가장 듣고 싶어 할 답변**
- 상품권 이미지 등록 시 위조·중복·만료 1차 검증 → 의심 케이스만 사람 검수
- 교환 신청 패턴 분석으로 어뷰징(자전거래·사재기) 1차 탐지
- 고객 문의 1차 분류·답변 초안 생성

**꼬리 질문 대비**
- "AI 비용이 뛰어오를 텐데?" → 캐싱·로컬 모델·티어드 처리(저난이도는 small model). 비용 모니터링 메트릭
- "프라이버시는?" → 개인정보 마스킹 후 호출. 자체 모델 운영 vs 외부 API 비용·보안 트레이드오프
- "정확도 100%는 안 되는데 책임 경계?" → 사람의 최종 결정권 보장 + AI는 "1차 필터"로 포지셔닝. 결과 로그 보관

> ⚠️ **본인 실제 이력서 사례 수치는 면접 전에 별도 정리 필요** — 본 문서는 deep dive 골격만, 구체 수치는 본인 이력서·포트폴리오 참조

## 5. Grafana/Prometheus/Loki 모니터링 — "사전 탐지" 어필 직결

> 관련: [[Observability|관측성]], [[Logging|로깅]]

**경험 요약**
- Grafana + Prometheus(메트릭) + Loki(로그) 직접 구축
- 슬로우쿼리·5xx·응답 지연 알림 → 병목 조기 탐지
- 채용 메일이 콕 집은 "안정적 서비스 운영, 문제 상황 미리 파악할 수 있는 체계" 정확히 매핑

**비단팝팝 맥락 연결**
- 결제·교환 도메인은 장애 = 금전 손실. 모니터링이 단순 "있으면 좋은" 게 아니라 필수
- 핵심 지표: (1) 교환 성공률·체인 트랜잭션 성공률 (2) 결제 외부 API 레이턴시·실패율 (3) DLQ 적재량 (4) DB Lock 대기·데드락 발생률
- 알림 채널: Slack 임계치 알림 + 페이지(긴급)

**꼬리 질문 대비**
- "왜 직접 구축? Datadog 안 쓴 이유?" → 비용. 초기 단계 트래픽에서 Datadog Pro는 과다. ECS 환경에서 Prometheus exporter + Grafana로 충분히 커버
- "트레이싱은?" → OpenTelemetry로 분산 트레이싱. 결제·교환·체인 이벤트 흐름 추적 시 필수 검토
- "RUM은?" → 백엔드는 메트릭·로그·트레이스 3축. 프론트는 RUM(Real User Monitoring)으로 별도
