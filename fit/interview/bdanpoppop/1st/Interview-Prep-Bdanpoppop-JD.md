### 핵심 키워드 (면접에서 반복 등장할 것)

- **상품권 교환 정합성** — 동시성, 트랜잭션, 중복 사용 방지
- **Web3, 블록체인 연동** — 자체 체인 서비스, 토큰화, 전자증권
- **안정 운영, 자동화, 비용 효율, 보안** (메일/공고에서 핵심 단어 그대로 등장)

- **확장성** — 대용량 트래픽, K8s, 메시지 큐

### 우대사항 vs 내 경험 매칭

| 우대사항 | 매칭도 | 내 근거 |
|---------|------|--------|
| 블록체인(Web3) 연동 | **약** | 직접 경험 없음. 개념 학습(Web3.js, Ethers.js, 트랜잭션 상태 관리, nonce, gas) 면접 전 보강 필요 |
| DB 설계(ERD) 및 쿼리 최적화 | **강** | MongoDB→MySQL 마이그레이션 + 도메인 모듈 분리, 복합 인덱스로 슬로우 쿼리 99.3% 개선 (15.4ms→0.1ms) |
| Redis 등 캐시 시스템 | **중** | 스킬 보유, 개념 숙지. 실서비스 캐시 설계는 깊지 않음 → 패턴(Cache-Aside, Write-Through, TTL/LRU) 정리 필요 |
| 메시지 큐 (Kafka, RabbitMQ 등) | **중** | EventBridge+SQS 기반 이벤트 아키텍처 설계, 운영 (발주 자동화). Kafka는 개념. → "유사 패턴 + 도구 차이"로 어필 |
| 대용량 트래픽 처리, 튜닝 | **중** | 단일 인스턴스→ECS 스케일업, CloudFront 캐싱 전환 경험. 진성 대용량(수만 RPS)은 미경험 → 학습 |
| Docker, Kubernetes(K8s) | **중** | Docker 멀티스테이지 빌드 43.6% 경량화(909MB→513MB). K8s는 개념 — ECS 운영 경험 위주 |

→ **6개 중 5개 약~중**. 강점은 DB 최적화 1개. **블록체인 도메인은 약점이지만 메일, 공고 모두 "우대"이지 "필수"가 아님**. 자격요건 풀 매치를 메인으로 어필.

## 2. 회사 맞춤 FIT 답변

### 1분 자기소개 (40~50분 단독 면접 → 30~45초 압축본 권장)
> 구조: 이름/역할 → 도메인 경험 → 핵심 성과(수치 2~3개) → 일하는 방식 한 줄
> **메일에 적은 3가지(출시 서비스, 엔지니어링 방향, 스택 매치)는 면접관이 이미 본 전제. 자기소개에서 환기조차 하지 말 것 — 본인 경험, 수치, 일하는 방식이라는 메일에 없는 정보로만 채운다.**
> **부장급 단독 40~50분 면접: 자기소개 짧게 끝내고 답변, deep dive로 시간 확보가 우세.**

**압축본 (35초)**
안녕하세요, 4년차 백엔드 개발자 [이름]입니다.
트라이포드랩에서 NestJS 기반 B2B IoT 재고관리 서비스를 설계, 운영하면서, 수천 대 디바이스 동시 갱신 환경에서 **DB Lock으로 정합성을 잡고**, **EventBridge+SQS 기반 이벤트 아키텍처로 수기 발주를 4시간에서 10분으로 자동화**했습니다. 복합 인덱스 최적화로 슬로우 쿼리를 99.3% 줄였고, 모니터링도 직접 구축해 사용자 증가 전에 병목을 미리 잡는 체계를 만들었습니다.
일하는 방식은 직관보다 **메트릭, 로그, EXPLAIN으로 정량 근거를 먼저 만든 뒤 움직이는** 쪽입니다.

**확장본 (60초, 자기소개 시간 더 달라고 하면)**
... (위에 추가) Prisma의 N+1 패턴을 진단해 relationLoadStrategy 전환으로 82~90% 추가 개선한 사례, 단일 EC2 → ECS Fargate + Auto-scaling 아키텍처 전환까지 한 사이클을 돌렸습니다. 오늘 그 안에서 어떤 의사결정을 했는지 구체적으로 말씀드릴 수 있습니다.

### 지원 동기
> 구조: **트라이포드랩 패턴 ↔ 팝팝 도메인 1:1 매핑** + **받은 메일이 본인을 본 시각에 직접 응답**
> **"STO 좋다, 시장 시점 좋다" 같은 메일 결의 매력 표현 금지** — 메일 1번(STO/RWA 접목)의 변주가 됨.
> **메일에서 본인을 "팀의 업무 방식 자체를 효율적으로 고민할 사람", "기술 문화를 직접 설계하는 빌더"로 봤음** — 이 시각에 직접 응답하는 표현이 들어가야 한다.

가장 결정적이었던 건 **트라이포드랩에서 풀던 백엔드 패턴이 팝팝 도메인에 그대로 매핑된다**는 점이었습니다.

- **상품권 동시 교환, 결제 정합성** = IoT 재고 동시 갱신과 같은 row lock 패턴. 트라이포드랩에서 `SELECT FOR UPDATE NO WAIT` + lock 순서 통일 + 트랜잭션 범위 최소화로 풀었던 패턴이 그대로 적용됩니다.
- **결제, 블록체인 기록, 외부 발행사 API 분리** = 발주 자동화에서 카톡, 이메일, 발주서를 SQS로 분리하고 채널별 DLQ + 멱등성 키 + status 머신으로 풀었던 패턴과 동일 구조. 체인 트랜잭션이 핵심 거래를 막지 않게 분리해야 한다는 것까지 같습니다.
- **출시 6~7개월차 안정화 구간** = 단일 EC2 → ECS Fargate + Auto-scaling 전환, 모니터링 구축까지 한 사이클을 돌려본 단계라 가장 자신 있게 기여할 수 있는 시점입니다.

또 메일에서 저를 "**팀의 업무 방식 자체를 효율적으로 고민할 사람**", "**기술 문화를 직접 설계하는 빌더**"로 봐주셨다는 점이 인상 깊었습니다. 트라이포드랩에서 단순히 코드만 짜는 게 아니라 **수기 발주 프로세스를 이벤트 아키텍처로 바꾸고**, **PR 머지 게이트(테스트 커버리지 60%)와 모니터링 대시보드를 직접 만들어 팀 전체가 같은 기준으로 일하게** 한 경험이 그 기대치와 맞물린다고 봤습니다.

Web3는 도메인 학습이 필요하지만, 백엔드 인프라(트랜잭션 상태 관리, 재시도, 멱등성, 외부 RPC 호출)는 패턴이 같아 옮겨 쓸 수 있다고 봤습니다.

### 이직 사유
> 구조: 현 회사 성장 → 한계 → 다음 도전 → 비단팝팝에서만 가능한 이유

- 트라이포드랩에서 0→1 구축, B2B 대형 고객사 PoC, 단일 서버→ECS 전환, 이벤트 자동화, 모니터링 구축까지 백엔드 한 사이클을 돌렸음
- 한계: B2B PoC는 피드백이 수개월 단위. 또 도메인이 IoT 재고관리로 좁아 **새로운 도메인, 시장에서의 학습 사이클**이 부족
- 다음 도전: (1) 사용자 사이클이 빠른 C 서비스 (2) 새로운 시장이 열리는 단계의 초기 플레이어 (3) 백엔드가 단순 기능 구현이 아니라 비즈니스 안정성, 신뢰성에 직결되는 도메인
- **비단팝팝에서만 가능한 이유**: 상품권 교환(C2C, 일 단위 사이클) + STO/RWA(법제화 직후 초기 진입) + 정합성, 신뢰성 핵심 도메인 — 세 조건이 동시에 맞물림
- 꼬리 대비:
  - "신생 회사 리스크는?" → 출시 서비스 + KDX 컨소시엄(BNK금융, 한국거래소) 백킹, BDAN 1Q 영업이익 100배 성장으로 단순 페이퍼 스타트업이 아님을 확인했음
  - "Web3는 처음인데?" → 백엔드 패턴(트랜잭션 상태 관리, 재시도, 멱등성, 외부 API 연동)은 동일. 도메인 지식(체인 RPC, gas, nonce)은 학습으로 메우는 게 합리적이라고 판단
  - "또 이직하지 않을까?" → 이직 기준은 속도가 아니라 **새 시장에서의 학습 곡선**. 비단팝팝은 도메인 자체가 막 열리는 단계라 단기간에 끝나지 않을 학습 트랙
