## 1. 면접 시작 전 확인

> **확정 정보 (메일 답장 26.05.07 기준)**
> - 일시: **26.05.14(목) 16:00**, 단독 면접 **40~50분**
> - 장소: 강남구 언주로 725 보전빌딩 동관 10층 (논현동)
> - 면접관: **개발본부 부장님 1명** — 기술 검증 + 직무와 문화 적합성 동시 진행
> - 도착 시 연락: 채용담당자(과장)

> **부장급 단독 40~50분 면접 — 시간과 톤 재조정**
> - 디테일 코드 deep dive보다 **"왜 그 선택? 다른 옵션은? 한계는? 팀에 미친 영향은?"** 결 우세. 의사결정, 트레이드오프, 시스템 디자인 위주
> - 1분 자기소개도 길 수 있음 → **30~45초 압축본** 준비
> - 면접관이 부장급이라 **엑시노 기술팀 규모와 거버넌스 역질문에 답할 수 있는 사람**

- [ ] **장소 사전 답사와 동선 계획** — 보전빌딩 동관 10층(논현동, 강남구 언주로 725) 첫 방문. 지하철, 버스, 도보 시간 미리 점검. 26.05.14(목) 16:00 면접 = 평일 오후 강남 교통 변수 큼. **15:30 도착, 15:45 채용담당자(과장) 연락** 목표
- [ ] 회사 도착 후 채용담당자(과장)에게 도착 연락
- [ ] **사전 질문 권리 활용 검토** — 일정 확정 메일에 "면접 전에 추가로 궁금하신 사항 등은 언제든 편히 연락 주시기를 바랍니다"라고 명시. 면접 직전 메일 한 통으로 다음 중 필요한 것 요청 가능:
  - 면접관(개발본부 부장님)의 사전 자료 선호 — 이력서 외 포트폴리오, 블로그, GitHub 링크 추가 전달 필요한지
  - 면접 형식 — 화이트보드 코딩, 시스템 디자인 다이어그램 그리기가 있는지 (40~50분 단독이라 가능성은 낮지만)
  - 부장님 직함과 소속 외 추가 정보(예: 백엔드 출신인지, 블록체인 출신인지) — 답변 톤 조정용
  - **단, 너무 많이 묻지 말 것**. 1~2개로 압축, 또는 묻지 않는 것도 옵션
- [ ] 채용 절차 재확인: **서류 → 1차(직무, 문화) → 처우 협의 → 최종**. 2차가 별도로 없음 → 1차에서 기술과 컬처핏 모두 검증
- [ ] **회신 메일 본문 다시 읽기** — 면접관이 본 첫인상 맥락(출시 서비스, 엔지니어링 방향, 스택 매치) 환기. 이 세 가지는 면접에서 다시 풀지 않는다
- [ ] 회사, 서비스, STO 흐름 30분 내로 다시 훑기 — [[Interview-Prep-Bdanpoppop-Company|회사 조사]] 1회독
- [ ] 핵심 역질문 3~4개 외워서 준비, 나머지는 노트 — **부장이 답할 수 있는 영역(기술팀 구성, 거버넌스, 기술 부채, 기술 문화)** 우선
- [ ] **30~45초 압축 자기소개** 음성 연습 (3회) — 메일 환기 없이 본인 경험, 수치, 일하는 방식. 단독 40~50분에서 1분도 길게 느껴질 수 있음
- [ ] 지원 동기 답변 점검 — **트라이포드랩 패턴 ↔ 팝팝 도메인 1:1 매핑 3개**(동시 교환과 결제, 체인 분리, 안정화 구간) + **"팀 업무 방식 효율화, 기술 문화 빌더"** 시각 명시적 응답 (받은 메일이 본인을 그렇게 봄)
- [ ] 장단점 답변 점검 — 장에서 "운영, 자동화, DevOps" 단어 반복 안 하고 **관찰 기반 의사결정** 또는 **도메인 모듈 분리** 각도로 가는지
- [ ] 질문에 답할 때마다 **선택 → 다른 옵션 → 트레이드오프 → 한계** 4단 구조 의식 (부장급 면접 톤)

## 3. 어필 매트릭스 — 검증 대비 vs 추가 어필 분리

> **메일이 콕 집은 3가지 = 면접관이 검증할 3가지**. 어필이 아니라 **deep dive 답변 준비** 대상.
> **추가 어필은 메일이 다루지 않은 강점**으로 — 면접에서 새 정보가 되어야 한다.

### A. 검증 대비 — 메일에서 콕 집힌 3가지 (면접관이 deep dive로 들어올 가능성 높음)

| 메일이 콕 집은 강점 | 예상 deep dive 질문 | 답변 준비 |
|------------------|------------------|---------|
| **비즈니스 효율 (4시간→10분)** | "어떻게 자동화? 자동화 전후 책임 경계는? 실패 시 운영자 개입은?" | EventBridge+SQS 발주 자동화 — 채널별 DLQ, 멱등성 키, status 머신, visibility timeout 6배 룰. [[Interview-Prep-Bdanpoppop-Tech-Resume#3. EventBridge+SQS 이벤트 아키텍처 — 결제, 교환, 알림 분리에 직결|Tech-Resume #3]] |
| **안정적 운영 (사용자 증가 대비 + 사전 탐지)** | "어떤 메트릭? 임계치는? 사전 탐지가 실제 사고 막은 사례?" | Grafana, Prometheus, Loki 직접 구축. 4계층 메트릭(비즈니스, 앱, 인프라, 외부 의존성). [[Interview-Prep-Bdanpoppop-Tech-Resume#5. Grafana/Prometheus/Loki 모니터링 — "사전 탐지" 어필 직결|Tech-Resume #5]] |
| **AI 업무 파이프라인** | "구체적으로 어떤 검수? 정확도와 비용은? 사람 검토가 필요한 경계는?" | 본인 이력서 사례 deep dive — 정확도, 검수 시간 단축률, 휴먼 인 더 루프 설계 |

### B. 추가 어필 — 메일이 다루지 않은 강점 (면접에서 새 정보로 어필)

> 메일과 공고에 명시되지 않은 영역. "이런 것까지 해보셨군요"라는 반응을 끌어낼 수 있는 포인트.

- **DB 정합성과 동시성 제어 깊이**: Pessimistic Lock + NOWAIT, lock 순서 통일, 트랜잭션 범위 최소화. 상품권 교환과 결제 도메인의 핵심 — 메일에 없던 영역
- **인덱스 설계 정량 분석**: 카디널리티와 선택도 분석 후 복합 인덱스 설계, 15.4ms → 0.1ms (99.3%)
- **ORM 한계 진단과 대응**: Prisma include의 N+1 패턴 발견 → relationLoadStrategy 'join' 전환, 100ms → 10ms~18ms (82~90% 개선)
- **아키텍처 전환 경험**: 단일 EC2 → ECS Fargate + ALB + Auto-scaling. 무중단 배포, 롤링, 모니터링 통합
- **테스트 커버리지 게이트**: 시솔지주에서 0 → 70% + PR 머지 게이트 60% 미만 차단. 코드 품질 의사결정 권한 있는 위치에서 일해본 경험
- **관찰 기반 의사결정 방식**: 직관 대신 메트릭, 로그, EXPLAIN으로 정량 근거 먼저 — Service의 컬처핏 답변과 일관

## 4. 주의사항

### 회사별 추가 주의점

- **메일 단어 반복 금지 (최우선 룰)** — 회신 메일에 본인이 직접 적은 키워드(**운영, 자동화, 비용 효율, 보안, 출시된 서비스, 메인 스택 매치**)를 자기소개, 지원 동기, 장단점에서 또 풀지 말 것. 면접관이 메일 본 직후라 새 정보가 0이 됨. 메일 콕 집은 3가지(자동화, 안정성, AI)는 **deep dive 답변에서만** 풀고, 자기소개와 장단점은 메일에 없던 각도(관찰 기반 의사결정, 도메인 모듈 분리, 동시성 깊이, 시장 시점)로
- **"잘 맞는 사람을 오래 찾는다"** — 단기 성과보다 장기 정착과 도메인 학습 자세 어필. 빠른 이직 시그널 주지 말 것
- **신생 회사 + 1명 백엔드 채용** — 권한과 책임의 균형에 솔직하게. "혼자 다 떠안는 게 부담된다"는 뉘앙스 금지, 동시에 시니어 리드 부재 리스크는 역질문으로 명확히 확인
- **블록체인 모르는 척 안 하기**: 약점 인정 + 학습 의지 + 백엔드 패턴은 옮겨 쓸 수 있다는 자신감. 아는 척하면 더 위험
- **2회 탈락 이력은 다른 회사**: 비단팝팝은 첫 시도 — 자신감 가지되, 사용자 페르소나의 보류 패턴(신생, 소규모) 우려는 1차 면접에서 직접 검증
- **STO/RWA 법제화는 알고 가기**: 2026.01.15 자본시장법 개정 통과 — 단순 시범사업이 아니라 정식 시장 진입 단계라는 사실은 면접관이 본인 사업 자긍심으로 언급할 가능성 있음

### 면접 현장 일반 주의

- 외부 귀책 화법 금지 — "회사가 X였다"로 끝내지 말고 자기 귀책 + 학습 + 적용 흐름으로
- 모르는 건 솔직히 + 어떻게 학습할지 설계로 보완
- 답변은 결론 → 근거 → 사례 → 한계 순. 사례에 정량 수치
- 역질문에서 사용자 결정 4가지(엑시노 팀, 합류 팀, 처우, 거버넌스) 모두 회수
