## 1. 서비스 맥락 질문 (예상 + 답변 방향)

면접관이 "우리 서비스 상황에서는…" 패턴으로 들어올 가능성 높은 질문들. 트라이포드랩 경험과 매핑해 답하는 게 핵심.

### Q1. 두 사용자가 같은 상품권을 동시에 교환 신청하면 어떻게 막나요?

**설계 방향**
- 상품권 시리얼을 멱등성 키로, 교환 신청 시 `SELECT FOR UPDATE NO WAIT`로 row lock 획득
- 잠금 실패 시 즉시 "이미 다른 거래 진행 중" 에러 반환
- 트랜잭션 안: 상품권 status `AVAILABLE` 검증 → `PENDING_EXCHANGE`로 전환 → 거래 ID 발급 → 트랜잭션 안에서 두 상품권 모두 X Lock(작은 ID 먼저)

**실무 연결**: 트라이포드랩 IoT 재고 갱신에서 동일 패턴 — 수천 대 동시 갱신을 row lock으로 직렬화. 데드락은 lock 순서 통일로 회피
**꼬리 대비**: "Optimistic도 가능하지 않나?" → 결제·정산 후 롤백 비용이 크면 Pessimistic이 안전. 단순 status 변경이면 Optimistic도 OK

### Q2. 결제는 성공했는데 블록체인 기록이 실패하면?

**설계 방향**
- 핵심 트랜잭션(소유권 이전·결제)은 동기 + DB 트랜잭션
- 블록체인 기록은 SQS로 분리 → 별도 워커가 비동기 발행 + 재시도 + DLQ
- 체인 실패가 사용자 거래를 롤백하지 않음 — 거래는 완료 상태, 체인 기록만 보류 상태 유지(`chain_status: PENDING/CONFIRMED/FAILED`)
- 일정 시간 미확정 시 운영자 알림, 수동 개입

**실무 연결**: 트라이포드랩 발주 자동화에서 카톡·이메일·발주서 채널별 DLQ + 멱등성 키 패턴 그대로 옮김
**꼬리 대비**: "체인 기록 안 되면 사용자에게 어떻게 보이나?" → 거래 완료 + "체인 등록 진행 중" 표시. UX와 시스템 일관성을 분리해 사용자 경험을 깨지 않음

### Q3. 신상품권 발매 이벤트 트래픽 폭증 — 어떻게 대응?

**설계 방향**
- 사전: Auto-scaling 정책(target tracking) 미리 워밍, Redis 캐시·CDN으로 정적 자산 분리
- 시작: 선착순 쿠폰처럼 Rate Limit + 대기열(token bucket)로 흡수, 트래픽이 DB까지 닿지 않게 큐 단계에서 차단
- 결제·체인 기록은 비동기 분리, 첫 동기 응답은 "신청 접수" 수준으로 가볍게
- 모니터링: ECS CPU·요청 수·DLQ 길이·DB Lock 대기·Redis hit rate

**실무 연결**: 트라이포드랩 단일 EC2 → ECS Fargate 전환 + Auto-scaling 적용 경험. 직접 수만 RPS 받아본 건 아니지만 **단계별 옵션을 알고 있고**, 메트릭 기반 의사결정 가능
**꼬리 대비**: "수만 RPS 직접 받아봤나?" → 진성 대용량은 미경험. 다만 트래픽 증가 시 옵션과 트레이드오프(수직 vs 수평·캐시·샤딩 시점)는 알고 있음. 실제 도입은 메트릭 기반 단계 결정

### Q4. 외부 API(발행사·결제사·체인 RPC) 응답이 느려지면?

**설계 방향**
- Circuit Breaker (Hystrix·Resilience4j 패턴) — 일정 실패율 초과 시 빠른 실패로 시스템 보호
- Timeout + Retry with exponential backoff + jitter
- Bulkhead — 외부 API별 격리된 thread/connection pool로 한 API 장애가 전체 마비로 번지지 않게
- Fallback — 캐시된 직전 값 또는 degraded 응답
- 모니터링: 외부 API별 레이턴시·에러율 별도 대시보드

**꼬리 대비**: "Circuit Breaker 직접 구현해봤나?" → 직접 풀 구현은 안 함. 라이브러리(opossum 등) 개념 + 트라이포드랩에서 timeout·retry는 적용해봤음. **Bulkhead 패턴은 ECS 워커 분리로 구현**

### Q5. 모니터링 설계 — 팝팝에서 무엇을 보겠나?

**4계층 메트릭**
1. **비즈니스**: 교환 성공률·결제 성공률·체인 기록 확정률·사용자 활성도
2. **애플리케이션**: 5xx·4xx 비율·응답 P50/P99·핸들러별 레이턴시
3. **인프라**: ECS CPU·메모리·DB 커넥션·Redis hit rate·SQS 큐 길이·DLQ 적재량
4. **외부 의존성**: 발행사 API·결제사 API·체인 RPC 노드 레이턴시·에러율

**알림 임계치 예시**
- 결제 성공률 < 99% → 페이지(긴급)
- 체인 기록 확정 지연 P95 > 60초 → Slack
- DLQ 적재 > 10건 → Slack
- DB Lock 대기 > 100ms 평균 → Slack

## 2. 컬처핏 / 소프트스킬 질문 (체크리스트)

> [[Common-Interview-Questions|FIT]], [[Self-Analysis|나찾기]] 기반 — 비단팝팝에 특히 중요할 것들
> **메일에서 이미 "운영·자동화·보안 매치"를 어필했으므로, 장점 답변에서 같은 단어 반복 금지.** 메일과 다른 각도로 본인 강점 표현.

- [ ] **장단점**
  - 장 옵션 A — **관찰 기반 의사결정**: 문제를 직관으로 풀지 않고 메트릭·로그·EXPLAIN으로 정량 근거를 먼저 만든다. 트라이포드랩 슬로우쿼리(EXPLAIN ANALYZE → 인덱스 카디널리티 분석 → 99.3% 개선), Prisma N+1(쿼리 로그로 4쿼리 확인 후 relationLoadStrategy 변경), Docker 경량화(레이어별 사이즈 분석 후 멀티스테이지 적용) 모두 측정 → 가설 → 검증 흐름
  - 장 옵션 B — **도메인 모듈 분리·의존성 방향 통제**: 발주·재고·디바이스 도메인별 모듈 분리, 외부 의존(SQS·외부 API)은 어댑터로 격리. 상품권·교환·결제·체인 도메인 분리에 그대로 적용 가능
  - 장 옵션 C (메일 시각 직접 응답) — **팀 단위 표준화·기술 문화 빌딩**: 시솔지주에서 PR 머지 게이트(테스트 커버리지 60% 미만 차단) + SonarQube 게이트 도입해 코드 품질 의사결정을 팀 전체 기준으로 만들었고, 트라이포드랩에서는 모니터링 대시보드를 팀 공통 도구로 자리잡게 했음. **받은 메일이 본인을 "팀 업무 방식 효율화·기술 문화 빌더"로 본 시각과 직접 매칭** — 부장급 면접관에게는 이 옵션이 가장 임팩트 있을 가능성
  - 단: 진성 대용량(수만 RPS) 미경험 + 블록체인 도메인 학습 중. **학습 트랙 설계**(Ethers.js·EIP·ERC-1400)로 보완 진행
- [ ] **동료 갈등** — 트라이포드랩 일화로
- [ ] **기획자 충돌** — "왜 그렇게 만들어야 하나" 근거 기반 협의 → 합의 후 구현
- [ ] **긴급 이슈 판단** — 모니터링 알림 → 영향도(MAU·금액·복구 가능성) 평가 → 우선순위
- [ ] **번아웃 관리** — 본인 페이스 관리법
- [ ] **성장 목표** — 현 단계: 백엔드 코어 + 시스템 디자인 + 새 도메인(블록체인). 비단팝팝과 매핑
- [ ] **조직문화** — 1명 채용·"빌더"라는 위치를 어떻게 받아들이는지
- [ ] **"잘 맞는 사람을 오래 찾는다"는 메시지에 어떻게 반응하나** — 컬처핏 깊게 보겠다는 신호. 답변 방향: 단기 성과보다 장기 정착·도메인 학습 자세 어필
- [ ] **신생 회사·1명 백엔드 책임** — 권한과 책임의 균형, 의사결정 자율성에 대한 기대치 정직하게
- [ ] **첫 90일 그림 (메일 시각 직접 응답)** — 받은 메일이 본인을 "**팀의 업무 방식 자체를 효율적으로 고민할 사람**", "**기술 문화를 직접 설계하는 빌더**"로 봤음. 부장급 면접관이 "들어오시면 첫 90일 어떻게 그릴 건가요" 질문 가능성 매우 높음. 답변 골격:
  - **0~30일**: 현재 코드베이스·인프라·운영 사이클·외부 의존성·온콜 부담 측정. 메트릭·로그·EXPLAIN으로 "팀이 지금 어디에 시간을 쓰고 있는지" 정량화
  - **30~60일**: 가장 비용 큰 한두 영역 표준화 — 트라이포드랩에서 PR 머지 게이트(테스트 커버리지 60%)·SonarQube·모니터링 대시보드를 팀 공통 도구로 만든 패턴 적용
  - **60~90일**: Web3 도메인 학습 + 백엔드 인프라 매핑 (트랜잭션 상태·재시도·멱등성·외부 RPC). 팀 단위 코드 리뷰 기준 합의
- [ ] **"기술적 성장에 목마른 인재" 시각 응답** — 메일 원문 표현. 답변 방향: 학습 트랙 설계 사례(현재 진행 중인 Web3·시스템 디자인·CS 영역) + 트라이포드랩에서 학습 → 적용 → 팀 전파 사이클 (Prisma 한계 진단 → relationLoadStrategy 도입 → 팀 공유 같은 패턴). **"성장 = 본인만 잘하는 게 아니라 팀이 같은 기준으로 성장하는 것"**으로 프레이밍하면 빌더 시각과 매칭

## 3. 역질문 (나 → 면접관)

> **사용자 결정 4가지를 반드시 모두 묻기.** 면접 끝부분이 아니라 적절한 흐름에서 분산 배치하면 자연스러움.

### 기술 / 아키텍처

1. **자체 블록체인 web3 서비스**라고 공고에 적혀 있는데, 어떤 체인 인프라를 쓰시나요? EVM 호환 체인인지, 자체 컨센서스인지, RPC 노드는 자체 운영인지 외부(Infura·Alchemy)인지 궁금합니다.
2. 팝팝 출시 6~7개월차에서 현재 가장 큰 기술 부채 또는 다음 분기 우선순위는 무엇인가요?
3. 결제·교환·체인 기록의 정합성은 지금 어떤 패턴으로 풀고 계시나요? (트랜잭셔널 아웃박스, 사가, 단순 SQS+멱등성 키 등)
4. STO 연계가 본격화되면 백엔드 측 변경 범위는 어디까지 예상하시나요? (전자증권법 발행인 계좌관리기관 연동 등)

### 팀 / 조직 (사용자 결정 1·2번 매핑) — **부장님이 직접 답해줄 수 있는 영역. 우선순위 ↑**

5. **현재 엑시노 기술팀 규모와 구성**(백엔드·프론트·DevOps·블록체인)이 어떻게 되나요?
6. **제가 합류하면 백엔드는 몇 명, 어떤 분과 함께 일하게** 되나요? 코드 리뷰·기술 의사결정은 어떤 흐름으로 도나요?
7. 비단팝팝(주)과 엑시노, 그리고 부산디지털자산거래소(비단)와의 **거버넌스가 궁금합니다**. 제품·기술 의사결정의 최종 권한은 어디에 있나요?
8. 채용 메일에서 저를 "**팀의 업무 방식 자체를 효율적으로 고민할 사람**", "**기술 문화를 직접 설계하는 빌더**"로 봐주셨는데, 부장님이 보시는 현재 팀의 가장 큰 업무 방식·기술 문화 개선 포인트는 어떤 것인가요?
9. "잘 맞는 사람을 오래 찾는다"는 채용 메시지를 보았는데, 비단팝팝에서 정의하는 "잘 맞는 사람"은 구체적으로 어떤 사람인가요?

### 성장 / 기대치

9. 1명 백엔드 채용 + "빌더로 모신다"고 하셨는데, **합류 후 첫 3~6개월 기대치**는 어떻게 그리고 계신가요?
10. 평가·승진·연봉 조정 사이클은 어떻게 운영되나요? (사용자 결정 3번 — **처우 레인지는 처우 협의 단계에서 별도로 묻기**)
11. 외부 컨퍼런스·블로그·오픈소스 활동을 회사 차원에서 지원하시나요? (단순 복지 차원이 아니라 도메인 학습 트랙으로)

### 처우 (직접 묻기 부담스러우면 처우 협의 단계로)

> 1차 면접에서 베이스 숫자 직접 묻기는 어색. 다음 절차가 "처우 협의"라 명시돼 있으므로 자연스러움. 다만 1차에서 **"정규직 전환 후와 수습 기간 보상 차이가 있나요?"** 정도는 무난.
