## 현장기술스택

### 언어&프레임워크
- Java, Kotlin, Spring이 채용 공고에서 압도적으로 우세
- 기회를 더 얻기 위해서라도 이 스택에 집중할 것

### 데이터베이스
- Oracle, MySQL, PostgreSQL, MongoDB, Redis
- DB 이론과 ANSI SQL 학습이 기본
- 면접 대비로 DB별 장단점과 프로젝트에서 선택한 이유를 설명할 수 있어야 함
- MySQL vs PostgreSQL: MySQL은 UPDATE에 유리, PostgreSQL은 INSERT에 유리

### 인프라&배포
- Docker + Kubernetes를 온프레미스에 직접 구축
  - 장점: 유연한 스케일아웃, 도메인 간 느슨한 연결, 빠른 배포
  - 주의: 트랜잭션 관리 어려움, 배포 순서와 버저닝
- CI/CD: Bamboo(주), Jenkins(보조). 무중단 배포 지향
- 모니터링: Prometheus + Grafana

### 메시지브로커
- Kafka를 주로 사용 (이벤트 드리븐 아키텍처의 핵심)
- 서비스 간 낮은 결합도·의존성을 달성
- 트랜잭션에 대한 고려 필수

### 협업
- Jira/Confluence 활용
- 위키 정리, 히스토리 관리, 적극적 공유가 중요

## 면접에서요구하는수준

### Java
- 시간복잡도를 고려한 코드 작성
- 인스턴스, 메모리 관리 등 내부 구조 이해
- StringBuffer vs StringBuilder 차이
- Stream, Optional 활용

### Spring
- 요청이 Controller에 도달하기까지의 과정
- 의존성 주입 등 내부 구조/원리

### DB
- 실행계획을 보고 간단한 튜닝 가능

### JDBC/JPA
- 쿼리가 DB에 전달되기까지의 단계
- 영속성 컨텍스트 이해

### HTTP
- 구성과 의미 이해
- RESTful 설계 가능

## 커리어조언

### 태도
- 당장의 실력보다 **태도**가 더 중요. 태도가 좋은 사람이 결국 더 성장함
- 주도적으로 일을 추진해보는 경험이 상당히 중요

### 포트폴리오
- 백엔드는 보여지는 결과보다 **과정**을 보여줘야 함
- GitHub 잔디는 충분히 의미 있음
- 블로그는 단순 정리가 아닌 **오답노트**여야 함 (겪은 문제와 해결 과정 기록)

### 설계원칙
- 단일 책임 원칙을 철저히 따름
- DTO는 API별로 분리, 패키지는 도메인별로 구성
- 트레이드오프에서 최선을 선택하고, 선경험자의 베스트 프랙티스를 적극 참고

### 스타트업vs대기업
- 스타트업: 올라운더로 성장하기 좋음
- 대기업: 스페셜리스트로 성장하기 좋음
- 어느 쪽이든 운이 작용하며, 원하는 성장 방향에 따라 선택

### 추천도서
- 혼자 공부하는 컴퓨터구조, 운영체제
- 자바 ORM 표준 JPA 프로그래밍
- 헤드퍼스트 디자인패턴
- 클린 코드
- 모던 자바 인 액션
- 가상 면접 사례로 배우는 대규모 시스템 설계 기초
