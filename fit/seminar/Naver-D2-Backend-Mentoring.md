# 네이버 D2 밋업 — 백엔드 개발자를 꿈꾸는 학생에게

네이버 D2 대학생 테크 밋업에서 현장 질문에 답한 네이버 백엔드 개발자의 경험 기반 응답. 백엔드 진로·필요 지식·실무 고민·네이버 개발 프로세스가 망라된다. 주제별 상세 정리는 하단 "주제별 상세" 링크.

## 백엔드 개발자의 진로

### 일의 범위

백엔드 개발자는 범위가 넓다 — 웹 앱 개발뿐 아니라 데이터 엔지니어링·분산파일시스템·DBMS 제품 개발까지 모두 "백엔드"로 불린다. FE·모바일에 비해 **하는 일이 상대적으로 모호**.

- 프로젝트에 따라 서버관리·DB관리·프런트 개발까지 다 맡기도 함
- 선발/중간/마무리 투수처럼 분업화된 영역도 존재
- SE·FE 등 인접 분야와 소통 기회 多 → 한 분야 전문성 키울 기회도
- 예: 서비스 통계 모듈 개발 → 대용량 데이터 엔지니어로 성장

### 보람과 고충

- **비유**: 백엔드·인프라는 드럼·베이스, FE는 보컬·퍼스트 기타. 보컬이 주목받지만 드럼·베이스가 안정적이지 않으면 공연이 안 됨
- 보람: 사용자 폭주에도 무사, 성능 획기적 개선, 모듈 구조 개선으로 코드 감소
- 고충: **24시간 운영**. 노트북 상시 휴대, 회식 중 장애 대응. 대형 서비스는 oncall 로테이션

### 향후 전망

- 새 도구·프레임워크가 개발자를 대체할까? — **과제 증가 속도 > 도구의 생산성 증가 속도**. 수요 계속 증가 전망
- 기존 기술을 깊게 이해한 사람이 새 도구에도 빨리 적응 → **기회**
- Flash·Symbian처럼 저무는 분야는 있음 → 새 분야 적응 역량 필수
- 백엔드는 여러 분야와 연결되어 **무난한 선택**. RDB·TCP/IP·OS·통계 등 학교에서 쌓을 지식 많음

### ML·빅데이터의 영향

- AI 분야는 학교 이론가 유입. 백엔드 출신이 서비스 필요에 의해 해당 분야 전문가로 성장하는 경로도
- 백엔드 개발자가 **데이터 분석·모델링**에 참여하는 경우 증가
- 직접 모델링 안 해도 **로그·데이터 파이프라인·저장소 설계**가 AI/빅데이터의 기반

### 다른 분야(경영·기획)로 확장

- 기획 단계에 개발자 의견 많이 반영. 특히 **데이터 구조 고려한 백엔드 개발자**가 초기 참여하면 효율적
- '경영' 영역은 조직장 역할 — 기술 + 사람 사이의 일에 깊은 관심 필요. 모든 개발자 적성은 아님

## 백엔드 개발에 필요한 지식

JVM/Linux 기반 웹 서버 개발 관점 필수 요소들.

- **웹 생태계 스펙**: HTML, HTTP/1.1, HTTP/2
- **기본 SDK·라이브러리·프레임워크** 이해·활용
- **클라이언트를 위한 API 설계**
- **서버·컴퍼넌트·객체 간 역할 분담·의존성·통신 방법**
- **저장소 활용**: DBMS 설계, Cache(Global/Local) 적용, 파일 저장 솔루션, 검색엔진 연동
- **빌드 도구**: Maven/Gradle
- **배포 전략**
- **성능 테스트·프로파일링·튜닝**: JVM(GC 옵션), Nginx/Tomcat, OS 설정
- **인접 기술**: DBMS, FE
- **공통**: 테스트 코드 작성·리팩토링, 버전 관리(branch 정책)

한 명이 모두 깊이 알 필요는 없고 **팀 지식의 합**으로 결정.

## 데이터베이스

- 다양한 저장소 활용 시대에도 **RDB는 가장 우선**. RDB를 잘 다루는 능력은 백엔드 핵심 역량
- 개발 중에도 **쿼리 호출 수·실행 계획** 확인 습관
- 대용량 서비스에서 복잡한 JOIN 기피 경향. **단순 SQL = 캐시·분산 용이**
- **Stored Procedure 기피** — 배포·버전 관리·테스트·DB CPU 부담 문제
- 분산 수단: Master-Slave 복제, 샤딩(Spider·MySQL Fabric·Nbase-T). 샤드 키 선정 중요, 후변경 비싸다

## 개발 도구 숙련도 레벨 0~4

- Level 0: 문서 줘도 못 씀
- Level 1: 알려준 만큼만
- Level 2: 공식 레퍼런스로 스스로 익힘 + 팀 내 문서화 (**신입사원 기대 수준**)
- Level 3: 도구 비교·선택, 공식 문서 기여
- Level 4: 소스 수정·Fork·패치

자세한 내용: [[Tool-Mastery-Levels|개발 도구 숙련도 5단계]]

## 병렬처리

- Servlet 기반 Java 서버는 기본 병렬. **멤버변수는 Thread-safe만**
- Cache 대상 객체는 **Immutable** 유지
- 별도 스레드 풀 작업에는 `ExecutorService`·`ThreadPoolExecutor` (JDK5+ 권장). `new Thread()` 직접 생성 비권장

## 보안

- XSS·CSRF·SQL Injection 대처는 모든 개발자 필수
- 의존 라이브러리의 취약점 대응이 실무 많음
- Struts2 등 취약점 잦은 프레임워크는 **메일링 그룹 가입**해 모니터링

## 테스트

- "시간 없어 못 만들었다" = "테스트 작성에 시간이 오래 걸린다"
- 언어 숙련도가 올라가면 테스트 작성 시간도 줄어듦
- 점진적 도입: 유틸리티 클래스 → Spring 통합 테스트(DB 롤백) → Mockito → TDD
- 테스트 작성에 동기: **오늘의 디버깅 시간 단축** 같은 단기 이득도

## 자료구조·알고리즘

- 실무에서 기본 알고리즘 직접 구현은 거의 안 함 — **SDK·라이브러리에 이미 있음**
- 그러나 **선택·활용**을 위해 바탕 지식 필수. HashMap 내부 구조·B-Tree/B+Tree 활용 등

## 개발 프레임워크

- JVM 생태계: **Spring/Netty** 꾸준한 발전
- 비동기 I/O 서버 개발 관심↑ — 비동기 JDBC 부재로 보통 웹앱까지 확산 제한. Spring 5 WebFlux 주목
- 고부하 플랫폼은 Netty/Vert.x로 비동기 구현

## Serverless

- BaaS + FaaS 의존 아키텍처. AWS Lambda·네이버 Cloud Functions
- 의미가 더 넓어져 "서버 관리 신경 쓰지 않는 구조"로도 쓰임
- 추상화된 환경에서도 **인프라 기반 지식**이 여전히 유용

## 실무에서 하는 고민

### 용어 엄밀성

같은 용어가 다르게 쓰이는 혼란.

- **REST** — HATEOAS 없는 것은 엄격한 REST 아님. HTTP API·Web API가 무난
- **TDD** — Red/Green/Refactor 사이클. 사후 테스트 추가는 "테스트 코드 작성"
- **단위 테스트** — JUnit으로 쓴 테스트라고 다 단위 테스트 아님. 통합 테스트와 구분
- **VO(Value Object)** — getter/setter만 있는 객체는 DTO/TO. VO는 값에 의한 동등성

자세히: [[Technical-Terminology-Precision|기술 용어의 엄밀성]]

### 클라이언트·백엔드 역할분담

- 과거 JSP로 서버 렌더링 → React.js·Vue.js로 클라이언트 렌더링 증가
- Universal Rendering(초기 서버 + 이후 클라이언트) 시도 증가
- 100% 클라이언트 렌더링 = 백엔드는 HTTP API에 집중. 대신 FE 부담↑

### 클라이언트 API 설계

- CRUD는 POST/GET/PUT/DELETE 매핑, 그 외는 제각각
- 페이징·검색 조건·부분 반환 속성 표현이 프로젝트마다 다름
- **GraphQL** — Facebook에서 시작, Github 도입 등 확산

### 시스템 자르기 (MSA)

- 구성요소 간 **역할·책임 분담**이 모든 아키텍처 고민의 본질
- MSA — 작은 배포 단위 = 협업 장점, 과거 비용이 컸으나 인프라 발전으로 감소
- 그래도 **경계 설정**은 여전히 어렵고 **후변경 비용** 큼

## 네이버의 백엔드 개발

### 개발·배포 방식 (참여 프로젝트 사례)

- **주 1회 배포** — 월요일 회의로 범위 확정, 목/금 배포. Hot fix는 즉시. 최대한 무정지
- **Github feature branch + PR + master**. 마일스톤 = 버전
- **DB 스키마**: 개발은 Liquibase, 운영은 사내 스키마 관리 시스템
- **테스트**: JUnit + Rest-assured + Spring MVC Test/Embedded Tomcat. 단위 테스트 + Mockito. **라인 커버리지 74%**
- **Docker 이미지**: PR에 "Docker build" 라벨 → 이미지 생성·배포용 클러스터에 올려 확인
- **주 1회 QA**: 변경 기능과 영향 범위 중심 시나리오

### 인프라 활용

- VM 관리 = IaaS. 개발망 누구나 VM 생성 가능, 운영망은 SE 경유
- Docker + Kubernetes 시도
- 사내 PaaS: Nelo(로그), Pinpoint(APM), nGrinder(성능), NbaseARC(분산 메모리)
- 배포 솔루션 사내 자체. 모니터링 다층 복합
- 전담 'IT operation팀'은 없음. 설정 변경은 개발팀, 인프라 구성은 SE, 운영 DB는 DBA

## 마치며

개발과 요리의 비유 — 몇 명 위한 요리는 혼자 가능하지만 **500인분은 차원이 다름**. 실행만 되는 백엔드는 쉬움. 그러나 협업·성능·안정성·모니터링·데이터 수집까지 고려한 백엔드는 어렵다. 현실 문제를 해결하고 상상을 실현시키는 개발자로 나아가자.
