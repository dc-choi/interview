# TS 백엔드 밋업 10회 (2026-03-27)

AWS 역삼 센터필드 EAST 18F, 100명 규모

## 세션 1: AI 시대의 테스팅 (이세호, @D.Circle)

구현 은닉 테스팅, 그리고 LLM-as-a-judge

### Part 1: 전통적인 테스팅과 AI의 활용성

**왜 테스팅하는가 — 불가능을 가능으로:**
- ALS 기반 @Transactional 검증 → 손으로 몇 시간
- 하이브리드 검색 직접 구현 (RRF 외 2종) → 며칠
- HWPX/HWP/PDF 표 추출 알고리즘 → 2~3일
- Cursor 같은 LLM 기반 에디터로 엣지케이스 포착 → 2~3일

**왜 테스팅하는가 — 개발 시간 단축:**
- 전: 이해관계자 결정 바뀔때마다 손으로 다시 검증
- 후: 기존 버그가 두더지처럼 반복 안 함, 망치는 두더지 수만큼 자동

**왜 테스팅하는가 — 리팩토링이 쉽다:**
- 전: 1년간 "나중에 고치지" → v2 갈아엎기
- 후: 매일 리팩토링. 테케 깨지는 것만 고치면 그만

**발표자의 테스팅 서사:**
- 대학: JUnit 커버리지 100% 자동 채점
- 디써클: 기획 매달 바뀜 → 손으로 재검증 반복
- Playwright E2E 실패 → Sequelize PR 머징 경험으로 깨달음
- DB별 통합 테스팅 로컬 도커 수천건 → 사내 이식 → 월간 통합 80%+
- Testcontainers 도입 병렬처리 → 아키텍처 책 → Testable 코드 개편 → 유닛 비율 대폭 증가

**AI 시대의 테스팅:**
- 테스트 = 기계에게 의도를 전달하기 가장 쉬운 수단 → 4~10배 멀티플라이어
- 과거: TDD, 꽤 비쌈
- 현재: 인간 개입 시대. 테스트 = 자연어로 기계가 말하는 수단. 구조에 아직 인간 개입.
- 미래: 하네스만 검토. 1명이 수십 명 분량. 아키텍처·비용 효율 초점.

**흔한 함정 — 런던파 + LLM 화이트박스:**
- "코드 먼저 → 유닛 테스트 짜줘" → 사실상 함수 스펙 베낌 → 토큰 낭비
- 사람은 한 줄씩 짜면서 "왜?" 고민. LLM은 시키는대로 함.

**구현 은닉 — 핵심 원리:**
- 명세 ←(최소화)→ 테케 ←(최대화)→ 구현
- 우리가 보는 명세와 LLM이 보는 코드 명세의 괴리를 줄이려면 구현이 은닉된 구조가 좋다
- TDD가 첫 번째 방법. 그린필드에서는 눈이 가려짐. 브라운필드는 아키텍처에 따라 다름.

**TDD 학파: Classist(고전파) vs Mockist(런던파):**
- 런던파: 모듈을 격리해서 유닛테스팅. 명세 드러나는 테스트가 많아짐.
- 고전파: 여러 모듈을 모킹 없이 동시에 테스팅. 의미 있는 소수의 유닛 테스트로 커버.
- LLM은 지시 없으면 런던파로 짬. 런던파는 리팩토링 시 토큰 폭식 + 알탈(검수 어려움) + 테케 폭증

**구현 은닉 팁 1 — E2E·통합 테스트:**
- 스펙과 1대1 대응. 블랙박스. 명세↔테케 괴리 최소화, 테케↔구현 괴리 최대화.
- Docker로 직접 돌림. testcontainers로 보완. 핵심 한두 개 위험/중요 로직에만.
- 게시글 CRUD → E2E 몇 개 / 재고·결제·포인트 엮인 주문 → E2E/통합/유닛 모두 해비하게

**구현 은닉 팁 2 — 도메인 엔티티로 리팩토링:**
- Before: 서비스가 직접 계산. After: 엔티티에 eat(), sleep() 위임.
- LLM한테 구현이 바로 안 보임. 버그 더 잘 잡힘. 명세↔테케 거리 좁힘.

**구현 은닉 팁 3 — 값 객체/엔티티를 모킹하지 마세요:**
- 실제 인스턴스 사용 (고전파). 테케 수 최소화 → 검토할 코드 최소화.

**추가 팁 — 메모리에 올리기:**
- DB 쿼리 → 메모리 컬렉션. 수십 초 통합 → 0.1초 유닛.
- 어그리게트의 최대 장점: 비즈니스 응집 + 테스터빌리티

**블랙박스화 팁 정리:**
- E2E 비율 높이기 (AI 시대에 효율적)
- 엔티티 응집도로 명세↔코드 괴리 최소화
- Sociable Unit: IO만 모킹, 엔티티+서비스 묶어서 테스팅
- 메모리에 올려서 통합→유닛 전환
- Rule of Thumb: 유닛만으로 불안하면 → 통합 추가하거나 코드를 Unit Testable하게

**테스트 피라미드 재해석:**
- 피라미드(어그리게트, CQRS) / 역 피라미드(레거시, prototype) / 네모(초기 스타트업) / 허니콤(MSA, pact)
- AI 시대: 역 피라미드와 네모가 예전보다 훨씬 유효. 초기 개발 속도 좋음.

**완벽한 테스트는 없다:**
- 핵심만 검증. PMF 안정 후 커버리지 비중 올리기.
- Observability·DevOps가 마지막 보루.

**추천 도서:** Unit Testing(Khorikov), Effective Software Testing(Aniche), A Practitioner's Guide(Copeland)

### Part 2: 비결정적 테스팅과 AI — LLM-as-a-judge

**LLM-as-a-judge:** 키워드 매칭이 아닌 기술적 정확성·보안 가이드라인 준수 등 맥락 기반 평가

**사례 1 — 에이전트 평가:** SQL 시간 조건 누락을 LLM이 감지 (키워드 매칭으로는 발견 불가)
**사례 2 — 검색/추천 품질:** 실시간 검색 결과의 적합성을 LLM이 평가
**사례 3 — gold dataset 구축:** 사람 50세트 하루 → LLM 수 분, 사람은 검수만
**사례 4 — 알고리즘 튜닝:** Levenshtein threshold를 LLM 레이블링으로 자동 탐색
**사례 5 — hwpx2md/pdf2md:** LLM → gold dataset → TDD → 인간 개입 없이 2시간, 99% 정확도
**사례 6 — md2json:** 결정적 아웃풋이면 assertEquals로 충분. LLM-as-a-judge 불필요.

**정리:** 비결정적인 모든 곳에 LLM-as-a-judge. 알고리즘 튜닝과 PoC 가속.

**정확도 높이기:** few-shot, rubric, reasoning 먼저 출력(CoT), 모델 앙상블 / 인간 gold dataset 비교, Bias 최소화

### Part 3: 추가 팁

- **Live test 분리:** `LIVE_TEST=true npx jest`로 외부 API 호출 테스트 분리
- **핵심 경로만:** use-case 옆에 spec 파일 배치. CRUD는 spec 없음. 핵심만.
- **CI/CD에 넣기:** 언제부터 깨지는지 추적. Merge Block.

### 발표 정리
1. AI 시대에는 블랙박스화된 테스팅(구현 은닉)이 더 중요. 코드 아키텍처가 생산성을 좌우.
2. LLM-as-a-judge는 비결정적 평가의 강력한 무기. 활용 시야를 넓혀야.
3. 과거 이론은 그대로 유효 — 새로운 도구가 추가되었을 뿐. "나의 아이디어가 이 전쟁의 최전선."

## 세션 2: AI로 서비스 만들어 1년 이상 수익 내는 방법 (이승민, 티켓타코)

- (현) 티켓타코 1인기업 대표 / 에어프레미아 프론트 파트장
- (전) 뱅크샐러드 프론트 매니저 / 리멤버 안드로이드
- 이벤트 티켓팅/운영 플랫폼(SaaS), 2025.2 시작, 누적 거래액 2.5억

### AI 개발 방법

- 제로 하네스: 코드 베이스가 Context, 모델이 곧 AGENT
- 기획 방향 전달 → 설계 의견 → Plan Mode로 개발. Context Window를 믿고.
- 코드 다 읽기. 히스토리 지식 이점 > 코드 읽는 시간 비용.
- 생산성 차이: AI 사용 여부(⭐5) > 좋은 모델(⭐3) > 하네스(⭐1)
- AI 엔지니어링 시간 최소화, 제품 의사결정 집중. 병렬작업도 안 함.

### 수익화

- 일본 추첨 티켓팅에서 영감 → Festa Close + AI 등장 → 기회 포착
- 핵심: 기술 커뮤니티 도메인 전문가 (10년 활동, 70번 발표, GDE)
- 니즈 이해 + 인맥으로 시작. 리텐션은 제품력. VOC 1주일 내 개발 with AI.

### 도메인 쌓기

- 개발(2014) + 커뮤니티(2015) + 티켓팅(2016) = 10년 합작
- 꾸준히 오래: 1) 돈을 번다 2) 좋아하는 것을 한다
- "순수하게 즐기다보면 도메인이 되어있고, 기회가 오면 브랜드/제품이 됩니다"
- 한 자리에서 시대의 흐름을 이용하면 두려움 없이 변화를 즐길 수 있다

## 세션 3: UI 없는 소프트웨어 (김우진, 리캐치 대표)

버튼의 시대가 가고 '의도(Intent)'의 해자가 온다

- 비즈니스캔버스 대표 / 전 딜로이트컨설팅 / 전 AI 스타트업 알고리아 CSO
- Forbes Korea 2030 Power Leader / 리캐치: CRM 최다 투자 132.5억, 5,000대 기업

### Saaspocalypse

- 룰 기반 SaaS(서류 작업 소프트웨어): 3주 만에 시총 1조 달러 증발
- 수백 개 버튼 + if-then 조건문의 시대는 끝남
- 시장은 데이터 입력용 도구를 무자비하게 교체 중

### 제본스의 역설 (Jevons Paradox)

- AI 지능 비용(토큰 비용)이 0에 수렴 → 인지적 노동 수요는 폭발적 증가
- 단순한 '지능의 재판매'로는 수익 독점 불가

### 디스패처 문제 (Dispatcher Problem)

- 거대 모델(OpenAI, Anthropic) ↔ 얇은 AI 래퍼(디스패처) ↔ 엔드 유저
- 우버가 택시 배차원을 대체했듯, 단순 래퍼는 붕괴
- "더 빠르고 싸게" 포장만으로는 생존 불가

### 인터페이스의 소멸: CLI → Zero UI

- CLI(명령어) → GUI(클릭) → NUI(터치) → Zero UI(의도)
- 인지적 마찰 감소 ↔ 기계 지능 증가
- 가장 완벽한 인터페이스는 인터페이스가 없는 것

### 기록 시스템 → 실행 시스템

- 기록: 인간이 기계의 문법(UI)을 학습 (미로)
- 실행: 기계가 인간의 맥락을 학습 (의도 → 목표 직행)
- 소프트웨어 = 저장소 → 의도를 자율 실행하는 에이전트

### 외부 AI vs 내부 AI

- 외부: API 플러그인, 일반 지식, 며칠 내 대체 가능
- 내부: 기업 워크플로우 장악, 독점 맥락(Proprietary Context), 높은 전환 비용
- 해자 = 지능 수준이 아니라 시스템에 얼마나 깊이 뿌리내렸는가

### 구조적 내재화 (Durable Surplus)

- 기술 비용 0 추락 → 기술적 우위로는 방어 불가
- 고객의 일상 운영 시스템에 '제거하기 어려운 생태(Embeddedness)' 구축

### 버티컬 AI 생태계 매핑

| | Wedge (단일 기능) | Platform (다중 제품) |
|---|---|---|
| **Internal** | Precarious (불안정) | **Durable (지속 가능)** |
| **External** | Commodity Risk (범용화 위험) | Rare (드문) |

→ 내부화 깊이 + 제품 범위가 넓을수록 생존
