## DDD (Domain-Driven Design)

### MVC의한계
- 서비스 레이어가 무의미해지고 레포지토리가 비대해짐
- 도메인 모델이 아무것도 하지 않음 (빈약한 도메인 모델)

### Aggregate
- 데이터 집합체 (여러 엔티티나 VO), 함께 변경되어야 하는 것의 경계
- 소유와 참조가 구분되는 경계
- 너무 크게 설계하면 안 됨. 소유가 아니라면 id만 가질 수 있음

### DAO vs Repository
- **DAO**: 데이터 중심적 사고, 데이터의 경계 없이 자유롭게 CRUD
- **Repository**: 도메인 중심 사고, Aggregate를 조회/변경

### CQRS
- 명령과 조회의 분리, 논리부터 물리까지 분리 가능
- 서비스의 메서드를 클래스 핸들러로 분리
- 비즈니스가 복잡하지 않은 경우 컨트롤러에서 바로 핸들러로 조회해도 됨

### 트랜잭션스크립트문제
- 어플리케이션 서비스에 너무 많은 책임이 들어감
- 기존의 복잡한 서비스를 **커맨드 핸들러**로 나눠서 분리

### 도메인서비스
- 둘 이상의 Aggregate를 동시에 다루는 경우
- 도메인 규칙이지만 상태를 가지지 않는 경우
- 외부 API와 연동이 필요한 경우
- 주의: 상태를 저장하거나 사이드이펙트가 발생하면 도메인 서비스가 아님

### 엔티티매핑
- 인프라 엔티티와 도메인 엔티티를 그대로 쓰면 도메인과 인프라 관심사가 섞임
- 수동 매핑을 통해 관심사를 분리

### 비즈니스모델링우선
- 잘못된 접근: 기술부터 시작
- 올바른 접근: "주문이란 정확히 무엇인지?", "고객이 주문을 어떻게 취소할 수 있는지?"
- 기술 결정을 미루고 비즈니스 로직을 검증하는 것이 먼저
- 정확한 비즈니스 이해가 견고한 시스템을 만든다

## MSA (아임웹 사례)

### 전환배경과문제
- 기존: 거대한 PHP 모놀리식 구조
- MSA 도입 후 발생한 문제들:
  - 인프라 복잡성 급증
  - 인증 구현의 파편화
  - 공통 기능의 중복 구현

### 해결전략
- 수기 작업 제거: 인프라 담당자는 서비스 신뢰성, 백엔드는 비즈니스에 집중
- 자동화, 인증 표준화, 게이트웨이 도입

### 모노레포와인프라자동화
- 분산된 코드 관리를 위해 **모노레포** 도입
- **테라폼 모듈**: 하나의 인프라 모듈로 복붙 방지
- **ArgoCD 자동화**: 코드 푸시 후 1주일 -> 20분 배포

### 인증시스템표준화
- JWT 기반 인증 서버와 SDK/모듈
- 클라이언트 SDK와 서버 모듈로 제공
- npm 패키지로 배포

### 게이트웨이선택과정
- **Istio**: 학습 곡선이 가파르고 쿠버네티스 외부 연결 오류
- **Nginx**: 설정 유지보수 복잡, 백엔드가 다른 언어를 배워야 함
- **Kong Gateway 선택**:
  - DB 없이 YAML 기반 선언형 설정
  - 설정을 파일로 관리, Git으로 형상 관리 가능
  - 마이크로서비스의 인증, 로깅, 라우팅을 모두 처리

### 성과
- 현재 27개 서비스 운영
- 매주 새로운 어플리케이션 배포
- 기능 배포: 2주 -> 1일

## NestJS DI Deep Dive

### 의존성주입동작방식
1. 자신이 어떤 의존성을 주입 받아야 하는지 기록
2. 프로바이더 간 의존성 관계 기록
3. 실제 의존성 주입

### 메타데이터
- 보통 데코레이터와 연관지어 사용
- 트랜스파일링 과정에서도 메타데이터를 주입
- 타입, 파라미터, 반환값에 대한 메타데이터 기록 가능
- 주로 타입과 파라미터를 활용

### 프로바이더토큰
- 프로바이더는 토큰으로 구분: 문자열, 심볼, 생성자 그 자체
- NestJS는 파라미터를 통해 어떤 클래스의 객체/토큰을 원하는지 판단
- `@Inject()` 데코레이터는 셀프 파라미터를 사용

### 의존성처리과정
1. 모든 모듈 순회 후 모듈을 컨테이너에 등록
2. 모듈의 의존관계에 대한 트리 생성
3. 각 프로바이더에 대한 인스턴스 래퍼 사용
